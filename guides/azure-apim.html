Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de configuration Azure API Management
📋
Ce document est un référentiel technique couvrant Azure API Management (APIM) de bout en bout : architecture, sécurité des APIs, policy engine, AI Gateway, networking, monitoring et CI/CD. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'une instance APIM.

Analogie générale

📬
APIM = Bureau de poste. Il reçoit les requêtes (lettres), vérifie l'identité de l'expéditeur et le contenu, puis les route vers le bon destinataire (backend API). Les Policies sont le contrôle de sécurité aéroportuaire — chaque requête est inspectée avant, pendant et après le transit. Le Developer Portal est le catalogue de services — la vitrine où les développeurs découvrent et s'abonnent aux APIs.

Solutions écartées

⚠️
Alternatives évaluées et non retenues :
  • Kong Gateway / Apigee — Solutions tierces performantes mais nécessitent une infrastructure dédiée (Kubernetes ou VM). Pas d'intégration native avec Microsoft Entra ID, Azure Monitor, Key Vault. Coût total supérieur dans un écosystème Azure.
  • AWS API Gateway — Équivalent cloud Amazon. Non pertinent dans un contexte Azure (pas de VNet integration, pas de policies XML, pas de Developer Portal intégré).
  • Azure Front Door seul — WAF et CDN performants mais pas de gestion d'APIs (pas de subscription keys, pas de policies de transformation, pas de Developer Portal). Complémentaire à APIM, pas un remplacement.
  • Azure Application Gateway seul — Load balancer L7 avec WAF, mais aucune fonctionnalité de gateway API (pas de throttling, pas de JWT validation, pas de transformation). Utilisé en frontal d'APIM en mode Internal VNet.
  • YARP / Ocelot (.NET reverse proxies) — Solutions open-source légères. Manquent de Developer Portal, analytics intégrés, policies déclaratives, et support AI Gateway. Adaptées pour du routing simple, pas pour la gouvernance d'APIs.

Pourquoi chaque paramètre compte :
  • Tier sélectionné — Détermine les capacités disponibles (réseau, cache, multi-region, workspaces, self-hosted gateway). Un mauvais choix de tier est la cause n°1 de limitations en production. Consumption = pas de VNet, pas de Developer Portal, pas de cache ; workspaces = Premium / Premium v2 ; self-hosted gateway = tiers classiques compatibles uniquement.
  • Gateway managed vs self-hosted — Le self-hosted gateway (self-hosted-gateway container image) est utile pour l'hybride et le multicloud, mais ce n'est ni un tier ni une option universelle des tiers v2. Pour les workspaces, Microsoft introduit des workspace gateways managés distincts du self-hosted gateway.
  • Management plane — L'API REST de management (port 3443) permet de modifier la configuration APIM programmatiquement. Si exposée publiquement sans restriction IP ni RBAC : un attaquant avec des credentials volées peut modifier les policies, ouvrir des backends, ou supprimer des APIs.
  • Availability Zones — Distribuent les unités APIM sur plusieurs zones physiques d'une même région et améliorent la résilience régionale. Ne pas les confondre avec le multi-region, qui reste une capacité séparée et n'est pas alignée entre tiers classic et v2.
  • Managed Identity — Permet à APIM d'accéder à Key Vault, Storage, Event Hub sans stocker de secrets. Sans managed identity : les credentials backend sont stockées en clair dans les named values ou les policies — risque de fuite.
  • TLS minimum — Bon par défaut (TLS 1.2). TLS 1.0/1.1 sont vulnérables (CIS Microsoft Azure Foundations v6.0.0 — APIM non couvert nominativement, voir Microsoft Cloud Security Benchmark DP-3)(POODLE, BEAST). Réf. ANSSI-BP-028, CIS Azure Foundations v2.1.0 §8.5.
⚠️
Pièges courants :
  • Consumption = compromis fort — Le tier Consumption est un mode serverless facturé à l'usage. Il peut convenir à des charges faibles ou irrégulières, mais n'offre ni VNet, ni Developer Portal, ni cache intégré, ni domaines personnalisés multiples. De nombreuses équipes le choisissent trop vite puis découvrent ces limitations en phase de croissance.
  • Migration classic → v2 non triviale — Les v2 ne sont pas une simple bascule de SKU. La documentation live indique qu'il n'existe pas de migration automatisée d'une instance classic vers une instance v2 ; il faut planifier une nouvelle instance et la migration associée.
  • Self-hosted gateway = gestion Kubernetes — Le container self-hosted gateway nécessite un cluster Kubernetes (AKS, ARC, ou on-premises). Il faut gérer le cycle de vie du container, la connectivité au management plane (port 443 sortant), et les mises à jour de l'image.
  • Managed identity oubliée — Sans managed identity activée, impossible d'utiliser les named values depuis Key Vault. Les secrets doivent alors être stockés en texte (chiffré, mais moins sécurisé).

🚨
Scénario : Une API exposée publiquement via APIM n'a qu'une subscription key en clair dans l'URL (?subscription-key=xxx). Un développeur partage l'URL dans un ticket JIRA — la clé est indexée dans les logs, accessible à toute l'équipe. Analogie : Les mécanismes de sécurité APIM sont les contrôles aéroportuaires : la subscription key est le billet d'avion (preuve d'abonnement), le JWT est le passeport biométrique (identité vérifiable), le rate limiting est le contrôle de flux des passagers (pas plus de N par minute), et le CORS est la liste des compagnies autorisées sur ce terminal.

💡
Pourquoi chaque paramètre compte :
  • Subscription key obligatoire — Première barrière : sans clé, n'importe quel consommateur peut appeler l'API. La clé doit être dans le header (jamais en query string) pour éviter l'exposition dans les logs HTTP, proxy, et CDN.
  • JWT Validation — La subscription key identifie l'abonnement, pas l'utilisateur. Le JWT (via validate-jwt + Microsoft Entra ID) vérifie l'identité de l'appelant, ses rôles et ses scopes. Sans JWT : impossible de différencier les utilisateurs, impossible d'appliquer le moindre contrôle d'accès granulaire.
  • Rate Limiting — Protège le backend contre les pics de charge (DDoS applicatif, boucle infinie côté consommateur). Sans rate limiting, un seul consommateur peut monopoliser le backend. rate-limit-by-key opère par clé (subscription, IP, JWT claim).
  • CORS — Requis pour les SPA qui appellent l'API depuis le navigateur. Un <cors allow-credentials="true"> avec <origin>*</origin> est vulnérable : toujours lister les origines explicitement.
  • Backend auth via Managed Identity — Évite de stocker des secrets pour appeler le backend (App Service, Azure Functions, Azure OpenAI). L'identité managée obtient un token Microsoft Entra ID automatiquement.
⚠️
Pièges courants :
  • allow-credentials="true" + origin="*" — Combinaison vulnérable ! Le navigateur refuse cette combinaison (spec CORS), mais certains proxies/tests la laissent passer. Toujours spécifier les origines explicitement.
  • Subscription key en query string — Apparaît dans les logs du serveur, dans l'historique du navigateur, dans les outils de monitoring, et dans les CDN. Toujours forcer le header.
  • validate-jwt sans required-claims — Valide que le token est signé par le bon issuer, mais ne vérifie PAS les rôles ni les scopes. Tout utilisateur authentifié a accès à tout.
  • Rate limiting sur Consumption tier — Le tier Consumption ne supporte PAS rate-limit (uniquement rate-limit-by-key). Tester la policy utilisée.
  • Oublier on-error dans les policies — Si la validation JWT échoue sans section on-error, APIM renvoie une erreur 500 générique au lieu d'un 401 clair. Toujours configurer la gestion d'erreur.

2.2 — Configuration — Policies XML (inbound)

XML — Policies de sécurité complètes (inbound)
<policies>
  <inbound>
    <base />

    <!-- 1. CORS — origines explicites -->
    <cors allow-credentials="false">
      <allowed-origins>
        <origin>https://app.contoso.com</origin>
        <origin>https://portal.contoso.com</origin>
      </allowed-origins>
      <allowed-methods>
        <method>GET</method>
        <method>POST</method>
      </allowed-methods>
      <allowed-headers>
        <header>Authorization</header>
        <header>Content-Type</header>
      </allowed-headers>
    </cors>

    <!-- 2. IP Filtering — uniquement IPs autorisées -->
    <ip-filter action="allow">
      <address-range from="10.0.0.0" to="10.0.255.255" />
      <address>203.0.113.42</address>
    </ip-filter>

    <!-- 3. JWT Validation — Microsoft Entra ID -->
    <validate-jwt
      header-name="Authorization"
      failed-validation-httpcode="401"
      failed-validation-error-message="Unauthorized. Token is invalid or missing."
      require-expiration-time="true"
      require-signed-tokens="true">
      <openid-config url="https://login.microsoftonline.com/{tenant-id}/v2.0/.well-known/openid-configuration" />
      <audiences>
        <audience>api://apim-contoso-backend</audience>
      </audiences>
      <required-claims>
        <claim name="roles" match="any">
          <value>API.Read</value>
          <value>API.ReadWrite</value>
        </claim>
      </required-claims>
    </validate-jwt>

    <!-- 4. Rate Limiting — par subscription key -->
    <rate-limit-by-key
      calls="100"
      renewal-period="60"
      counter-key="@(context.Subscription?.Key ?? context.Request.IpAddress)"
      increment-condition="@(context.Response.StatusCode >= 200 && context.Response.StatusCode < 300)" />

    <!-- 5. Subscription key : supprimer avant de forwarder au backend -->
    <set-header name="Ocp-Apim-Subscription-Key" exists-action="delete" />
  </inbound>

  <backend>
    <!-- 6. Auth vers le backend via Managed Identity -->
    <authentication-managed-identity
      resource="api://backend-app-id" />
  </backend>

  <outbound>
    <base />
    <!-- Supprimer les headers internes -->
    <set-header name="X-Powered-By" exists-action="delete" />
    <set-header name="X-AspNet-Version" exists-action="delete" />
  </outbound>

  <on-error>
    <base />
    <set-header name="X-Error-Source" exists-action="override">
      <value>APIM-Gateway</value>
    </set-header>
    <return-response>
      <set-status code="@(context.Response.StatusCode)"
        reason="@(context.LastError?.Message ?? "Gateway error")" />
      <set-body>@{
        return new JObject(
          new JProperty("error", context.LastError?.Message ?? "Unknown error"),
          new JProperty("source", "APIM Gateway"),
          new JProperty("requestId", context.RequestId.ToString())
        ).ToString();
      }</set-body>
    </return-response>
  </on-error>
</policies>

3.2 — Scopes & héritage des policies

Scope Priorité Défaut Azure 🔒 Sécurisé Cas d'usage
Global (All APIs) 1 (le plus large) Vide (<base /> uniquement) CORS, rate-limit, set-header (suppression internes), on-error standardisé Appliqué à toutes les APIs sans exception
Product 2 Vide quota-by-key spécifique au produit Définir des quotas/limites par offre (Free, Starter, Enterprise)
API 3 Vide validate-jwt, authentication-managed-identity, rewrite-uri Policies spécifiques à une API (auth, transformation)

💡
Ordre d'exécution : Les policies s'exécutent du scope le plus large au plus spécifique. La balise <base /> dans chaque scope indique où insérer les policies du scope parent. Placer <base /> en début = le parent s'exécute en premier. En fin = le parent s'exécute après. Omettre <base /> = les policies parentes sont ignorées — dangereux si le global contient du rate limiting ou du JWT.
⚠️
Pièges courants :
  • Oublier <base /> — Si une policy au scope API ne contient pas <base />, les policies globales (rate-limit, CORS, on-error) sont ignorées. Bug silencieux en production.
  • Policy expressions coûteuses — Les expressions C# dans les policies (@(context.Request.Body.As<JObject>())) sont évaluées à chaque requête. Une expression lourde (parsing JSON complet, regex complexe) peut dégrader la latence. Utiliser IsCachingAllowed et limiter la taille du body parsé.
  • Policy fragments non utilisés — Le copier-coller de blocs XML entre APIs crée de la dette technique. Les policy fragments (réutilisables, versionés) sont disponibles depuis 2023. Les utiliser pour les blocs communs (JWT, error handling, logging).
  • Tester avec le mauvais scope — Le Trace (API Inspector) teste avec les policies de TOUS les scopes combinés. Un test unitaire d'une policy au scope Operation peut masquer un conflit avec une policy globale.
  • on-error mal configuré — Sans section on-error, une exception dans une expression policy renvoie un 500 générique sans détails. Configurer return-response avec un format d'erreur standardisé (RFC 7807).

  • Breaking changes non détectées — Supprimer un champ ou modifier un type dans l'OpenAPI sans prévenir les consommateurs. Intégrer un outil de breaking change detection.

🚨
Scénario : L'équipe a déployé APIM en tier Premium pour un POC avec 500 requêtes/jour. Le coût mensuel réel n'est découvert que plusieurs semaines plus tard par la finance. Un tier Consumption, Basic v2 ou Standard v2 aurait peut-être suffi selon le volume, le réseau et les exigences d'exploitation. Analogie : Choisir le tier APIM c'est comme choisir la taille du bureau de poste : un guichet mobile (Consumption) pour un petit village, un bureau standard (Standard v2) pour une ville moyenne, un centre de tri régional (Premium) pour les métropoles avec plusieurs points de distribution (multi-région).
