🚫 Solutions écartées

💡
Pourquoi PnP.PowerShell plutôt que SPO Management Shell seul ?
  • PnP.PowerShell (community, open-source) : couverture fonctionnelle plus large (sites, listes, content types, permissions granulaires), maintenu activement, compatible cross-platform. Retenu comme outil principal.
  • Microsoft.Online.SharePoint.PowerShell (SPO officiel) : cmdlets tenant-level fiables (Set-SPOTenant, Set-SPOSite), support Microsoft officiel. Retenu en complément pour les settings tenant-level.
  • Microsoft Graph API (REST) : accès programmatique aux sites et fichiers, intégration applicative. Mentionné pour référence mais non utilisé comme outil principal d'administration.
  • ShareGate (tiers payant) : migration et reporting avancé, mais coût de licence supplémentaire et dépendance à un éditeur tiers. Écarté.

Sharing Policies SPO P1
Niveaux de partage tenant et site, types de liens, expiration — 18 pts
Réf. CIS Microsoft 365 Benchmark v3.1 §3.1, §3.2
🚨
Analogie — Niveaux de verrouillage d'un bâtiment
Le paramètre SharingCapability revient à choisir entre 4 niveaux de verrouillage pour l'entrée de votre immeuble :
  • Anyone = la porte est ouverte, n'importe qui avec le lien entre sans badge
  • New & existing guests = on demande une pièce d'identité puis on délivre un badge visiteur
  • Existing guests = seuls les visiteurs déjà enregistrés au registre entrent
  • Only people in your org = seuls les employés avec badge permanent
Scénario critique : Un employé partage un lien « Anyone » vers un dossier de contrats sur SharePoint. Le lien est relayé sur un forum public — n'importe qui sur Internet peut télécharger vos documents confidentiels sans aucune authentification. Le partage externe est activé par défaut dans SharePoint et OneDrive, mais le niveau effectif (« Anyone », invités existants, etc.) doit être vérifié au tenant et au site.
💡
Pourquoi chaque paramètre compte
  • SharingCapability — Définit le niveau maximal de partage autorisé au niveau tenant. C'est le plafond : les sites ne peuvent pas être plus permissifs que le tenant.
  • DefaultSharingLinkType — Le type de lien proposé par défaut aux utilisateurs. Si c'est « Anyone », la majorité ne le change jamais et créera des liens anonymes.
  • DefaultLinkPermission — View ou Edit par défaut. « Edit » par défaut signifie que chaque lien créé donne un accès en écriture.
  • RequireAnonymousLinksExpireInDays — Impose une durée de vie aux liens anonymes. Sans expiration, les liens persistent indéfiniment.
  • FileAnonymousLinkType / FolderAnonymousLinkType — Restreint les permissions des liens anonymes (View seulement vs Edit).
  • PreventExternalUsersFromResharing — Empêche un invité externe de re-partager le contenu à des tiers.
  • EmailAttestationRequired / EmailAttestationReAuthDays — Oblige les invités à vérifier leur email périodiquement.
⚠️
Piège fréquent — Le paramètre SharingCapability du tenant est un plafond. Exemple : si le tenant est sur ExistingExternalUserSharingOnly, un site ne pourra jamais être configuré sur ExternalUserAndGuestSharing (Anyone). Réciproquement, restreindre un site ne change pas le tenant.

Autre piège : le paramètre DefaultSharingLinkType affecte l'UX mais pas la capacité. Même si vous mettez le défaut sur « Internal », si le tenant autorise « Anyone », un utilisateur peut manuellement changer le lien en « Anyone » dans le picker.

Configuration — Appliquer le Quick Win

PowerShell — PnP.PowerShell + SPO Management Shell 
# ── S1 : Sharing Policies — Quick Win ────────────────────────────────
# Prérequis : Connect-SPOService -Url https://<tenant>-admin.sharepoint.com

# 1. Restreindre le partage tenant à "Existing Guests Only"
Set-SPOTenant -SharingCapability "ExistingExternalUserSharingOnly"

# 2. Forcer le type de lien par défaut à "Internal" (org. only)
Set-SPOTenant -DefaultSharingLinkType "Internal"

# 3. Forcer la permission par défaut à View (lecture seule)
Set-SPOTenant -DefaultLinkPermission "View"

# 4. Expiration des liens anonymes à 30 jours (si encore autorisés sur certains sites)
Set-SPOTenant -RequireAnonymousLinksExpireInDays 30

# 5. Liens anonymes en lecture seule
Set-SPOTenant -FileAnonymousLinkType "View"
Set-SPOTenant -FolderAnonymousLinkType "View"

# 6. Empêcher le re-partage par les externes
Set-SPOTenant -PreventExternalUsersFromResharing $true

# 7. Activer l'attestation email pour les invités
Set-SPOTenant -EmailAttestationRequired $true
Set-SPOTenant -EmailAttestationReAuthDays 30

# 8. Notification des propriétaires lors du re-partage
Set-SPOTenant -NotifyOwnersWhenItemsReshared $true

External Sharing & Guest Access SPO P1
Guest access, B2B, domaines autorisés/bloqués, expiration invités — 14 pts
Depuis mai 2026, Microsoft Entra B2B devient le mode de partage externe standard pour SharePoint et OneDrive
Réf. CIS Microsoft 365 Benchmark v3.1 §3.3, §3.7
🚨
Analogie — Liste d'invités à un événement
Le partage externe revient à gérer l'accès d'invités à un événement privé :
  • Allow-list de domaines = invitations nominatives uniquement aux entreprises partenaires
  • Block-list de domaines = tout le monde sauf les concurrents et domaines gratuits
  • Sans filtrage = n'importe quel email peut recevoir un accès — y compris @gmail.com
Scénario critique : Un partenaire quitte l'entreprise mais conserve un accès invité à SharePoint sans date d'expiration. L'ancien prestataire continue à consulter les documents du projet pendant 18 mois — sans que personne ne le sache car les access reviews ne couvrent pas les invités SPO.
💡
Pourquoi chaque paramètre compte
  • SharingDomainRestrictionMode — Contrôle si le partage externe est limité à une allow-list ou block-list de domaines. Sans restriction, n'importe quel domaine email peut recevoir un partage.
  • SharingAllowedDomainList — Liste blanche des domaines autorisés. Le plus restrictif : seuls les partenaires vérifiés reçoivent des invitations.
  • ExternalUserExpireInDays — Révocation automatique des accès invités après N jours. Sans cette valeur, les invités ont un accès perpétuel.
  • ExternalUserExpirationRequired — Active l'expiration obligatoire des invités. Nécessaire pour que ExternalUserExpireInDays soit effectif.
  • ShowPeoplePickerSuggestionsForGuestUsers — Affiche les invités existants dans le picker de personnes. Pratique mais peut révéler des invités à d'autres utilisateurs.
  • RequireAcceptingAccountMatchInvitedAccount — Oblige l'invité à se connecter avec l'email exact de l'invitation. Empêche le forwarding d'invitations.
⚠️
Piège fréquent — Depuis mai 2026, SharePoint et OneDrive basculent sur Microsoft Entra B2B pour tous les tenants ; le paramètre EnableAzureADB2BIntegration n'a plus d'effet sur le comportement de partage. En parallèle, les invitations envoyées via le legacy SharePoint Invitation Manager n'accordent plus d'accès depuis juin 2024 : les contenus concernés doivent être repartagés.

Les paramètres d'invités restent distribués entre 3 consoles différentes :
  • SharePoint Admin Center → Sharing → domain restrictions, link expiration
  • Entra ID → External Identities → B2B collaboration settings, guest restrictions
  • Microsoft 365 Admin Center → Settings → Org settings → SharePoint sharing
Si Entra ID bloque les invitations B2B, les settings SPO sont sans effet. Toujours vérifier la cohérence entre les 3 portails.
Voir entra-id.html pour les B2B collaboration settings et entra-governance.html pour les access reviews des invités.

Storage Management SPO P1
Quotas, pooled storage, versioning, alertes de stockage — 12 pts
Réf. Microsoft Learn — Manage site storage limits
🚨
Analogie — Brouillons empilés sur un bureau
Le versioning SharePoint fonctionne comme des brouillons empilés : chaque modification crée une copie complète du fichier. Sans limite, un document Word de 50 Mo avec 500 versions occupe 25 Go — et le storage pool du tenant fond comme neige au soleil.
  • Par défaut : versioning illimité (« automatic » / 500 versions majeures)
  • Storage pool : 1 TB + 10 Go/utilisateur, partagé entre tous les sites
  • Storage mode : Automatic (Microsoft gère) ou Manual (quotas par site)
Scénario critique : Un site d'équipe contenant des fichiers CAO (500 Mo chacun) avec versioning illimité consomme 200 Go en 3 mois. Le pool tenant atteint 95% — les utilisateurs ne peuvent plus uploader de fichiers sur aucun site.
💡
Pourquoi chaque paramètre compte
  • EnableAutoExpirationVersionTrim — Active le nettoyage automatique (new versioning policy). Les versions de >60 jours et >N versions sont automatiquement supprimées. Réduit la consommation storage de 30-60%.
  • MajorVersionLimit — Nombre maximum de versions majeures conservées (applicable avec le new versioning). 100 est un bon compromis entre historique et espace.
  • ExpireVersionsAfterDays — Durée de rétention des anciennes versions en jours. 60-90 jours couvre la plupart des besoins de rollback.
  • StorageQuota / StorageQuotaWarningLevel — Quota par site et seuil d'alerte. Sans quota individuel en mode Manual, un seul site peut consommer tout le pool.
  • DenyAddAndCustomizePages — Empêche l'ajout de pages/scripts custom qui pourraient créer des données non gérées.
⚠️
Piège fréquent — Le « New Version History Limits » (annoncé en 2023, déploiement progressif) change le comportement par défaut : les nouveaux sites utilisent un trim automatique au lieu du versioning illimité. Mais les sites existants conservent leur ancien paramétrage illimité.

Le mode Automatic (par défaut pour les nouveaux tenants) ne fixe aucun quota par site. Cela signifie qu'un seul site peut consommer tout le storage tenant sans déclencher d'alerte.

En mode Manual, le quota par défaut d'un nouveau site est souvent 25 Go — trop élevé pour des sites de communication et trop bas pour des sites équipe avec des fichiers lourds.

Site Lifecycle & Governance SPO P1 SAM
Création de sites, templates, inactive site policy, Hub Sites — 10 pts
🚨
Analogie — Bail immobilier
Chaque site SharePoint est comme un local loué : sans bail (politique de cycle de vie), le locataire reste indéfiniment même s'il n'utilise plus le local. Les sites inactifs s'accumulent — « site sprawl » — et consomment du storage, des licences et de la surface d'attaque.
  • Création libre = n'importe quel employé peut créer un site/équipe Teams sans approbation
  • Pas de lifecycle = les sites projet terminés restent actifs avec des permissions orphelines
  • Pas de Hub = les sites poussent sans hiérarchie — navigation et recherche deviennent chaotiques
Scénario critique : 3 000 sites Teams créés en 2 ans, dont 60% inactifs depuis 6+ mois. Chaque site a un groupe M365, un site SPO, un canal Teams, et une boîte Exchange — des milliers d'objets Entra à nettoyer manuellement.
💡
Pourquoi chaque paramètre compte
  • SelfServiceSiteCreationDisabled (M365 Group creation) — Contrôle qui peut créer des sites équipe / Teams. Sans restriction, tout utilisateur crée des sites à volonté.
  • Inactive Site Policies (SAM) — Détecte automatiquement les sites inactifs et envoie des notifications aux propriétaires. Nécessite SharePoint Advanced Management ou un environnement où les fonctionnalités SAM sont incluses via Microsoft 365 Copilot.
  • Hub Sites — Organisent les sites en hiérarchie logique. Permettent la navigation transverse, la recherche fédérée et l'héritage de thème/navigation. Analogie : le Hub est le « siège social » des sites.
  • Site Creation Default — Template par défaut, classification, langue — assure la cohérence de la gouvernance dès la création.
⚠️
Piège fréquent — Restreindre la création de sites SharePoint via SelfServiceSiteCreationDisabled ne contrôle pas la création d'équipes Teams (qui créent un site SPO automatiquement). Pour restreindre la création de groupes M365 (et donc Teams + SPO), il faut configurer Entra ID → Group Settings via GroupCreationAllowedGroupId.

Les Inactive Site Policies s'appuient sur SAM. Ne pas figer un tarif ici : selon le contexte, les capacités SAM peuvent venir d'un add-on dédié ou des droits inclus quand le tenant dispose d'au moins une licence Microsoft 365 Copilot. Sans SAM, il faut scripter manuellement la détection des sites inactifs via Get-SPOSite -Limit All et LastContentModifiedDate.

OneDrive Administration OneDrive
Sync client, Known Folder Move, rétention, Files On-Demand — 10 pts
🚨
Analogie — Déménagement automatique du bureau
Known Folder Move (KFM) revient à déménager automatiquement les dossiers Bureau, Documents et Images vers le cloud. Au lieu de perdre des fichiers quand le PC tombe en panne, tout est synchronisé en arrière-plan.
  • Sans KFM = les fichiers critiques sont uniquement en local, perdus si le disque meurt
  • Avec KFM forcé = Bureau/Documents/Images redirigés vers OneDrive, sync transparente
  • Files On-Demand = les fichiers apparaissent dans l'explorateur mais ne sont téléchargés qu'à l'ouverture
Scénario critique : Un employé perd son laptop avec 15 Go de documents projet non synchronisés. Sans KFM, aucune récupération possible. Avec KFM, l'employé se connecte sur un nouveau PC et retrouve tous ses fichiers en 2 minutes.
💡
Pourquoi chaque paramètre compte
  • KFMOptInWithWizard / KFMSilentOptIn (GPO) — Force la redirection des dossiers connus vers OneDrive. SilentOptIn est invisible pour l'utilisateur (recommandé).
  • FilesOnDemandEnabled — Active les fichiers à la demande. Essentiel pour les PC avec peu de stockage — seuls les fichiers utilisés sont téléchargés.
  • OneDriveStorageQuota — Quota OneDrive par utilisateur (défaut 1 TB, extensible jusqu'à 5 TB selon l'éligibilité et les procédures Microsoft). Attention : 5 TB n'est pas automatique.
  • OrphanedPersonalSitesRetentionPeriod — Durée de rétention du OneDrive après départ d'un employé. Défaut 30 jours — critique pour la conformité.
  • BlockMacSync — Bloque la synchronisation depuis les appareils macOS non gérés. Pertinent si des employés utilisent des Mac personnels.
  • TenantRestrictionEnabled / AllowedDomainList (sync) — Empêche de synchroniser vers un compte OneDrive personnel ou un autre tenant.
⚠️
Piège fréquent — Le déploiement de KFM SilentOptIn via GPO/Intune nécessite le Tenant ID (pas le domain name). Si le Tenant ID est incorrect, KFM silencieux échoue sans erreur visible pour l'utilisateur.

Le quota OneDrive de 5 TB n'est pas automatique même avec E5 — Microsoft documente une extension possible jusqu'à 5 TB, mais elle dépend du contexte client et de la procédure Microsoft. Planifiez un script de monitoring pour les utilisateurs proches du quota.

Retention 30 jours par défaut pour les comptes supprimés est souvent insuffisant — si les RH mettent 3 mois à signaler un départ, les données sont déjà purgées.

Permissions & Access Control SPO P1
Modèle de permissions, héritage, groupes, Conditional Access — 12 pts
Réf. CIS Microsoft 365 Benchmark v3.1 §3.6
🚨
Analogie — Organigramme des clés d'un bâtiment
Le modèle de permissions SharePoint fonctionne comme la distribution de clés : le site hérite des permissions du parent (comme les étages du bâtiment), sauf si on « casse » l'héritage pour donner un accès spécifique à un bureau.
  • Héritage intact = tout le monde qui a accès au site a accès aux sous-dossiers
  • Héritage cassé = permissions gérées indépendamment (complexe à auditer)
  • Permissions directes = clés données en main propre (difficile à révoquer en masse)
Scénario critique : Un site projet casse l'héritage sur 200 sous-dossiers avec des permissions individuelles. Un employé change de service mais conserve l'accès à 50 dossiers — impossible à auditer sans script dédié.
💡
Pourquoi chaque paramètre compte
  • ConditionalAccessPolicy (site-level) — Applique une restriction d'accès au niveau du site : accès complet, accès limité (web only), ou bloqué pour les appareils non gérés.
  • SharingCapability (site-level override) — Permet de restreindre le partage par site en dessous du plafond tenant. Essentiel pour les sites confidentiels.
  • LockState — Permet de verrouiller un site en lecture seule (ReadOnly) ou de le bloquer complètement (NoAccess). Utile pour les projets terminés.
  • DisableCompanyWideSharingLinks — Désactive les liens « Everyone except external users » sur un site. Empêche le partage trop large en interne.
  • MembersCanShare — Si $false, seuls les propriétaires peuvent partager. Les membres ne peuvent qu'accéder au contenu.
⚠️
Piège fréquent — Le ConditionalAccessPolicy au niveau SPO site n'est pas une Conditional Access Policy Entra AU SENS STRICT. C'est un « Access Control » SPO qui restreint l'accès des appareils non gérés. Pour une CA policy avec conditions riches (location, risk, app), il faut configurer une vraie Entra Conditional Access Policy ciblant l'app « Office 365 SharePoint Online ».

Le paramètre RestrictedAccessControl (SAM) est différent : il restreint l'accès à un site aux seuls membres d'un groupe de sécurité Entra, même si d'autres utilisateurs ont des permissions SPO.

Content Types & Metadata SPO P1
Content Type Hub, colonnes de site, Term Store, architecture de l'information — 6 pts
🚨
Analogie — Système de classement d'archives
Les Content Types sont les « fiches de classement » de vos documents : ils définissent quelles métadonnées chaque type de document doit porter (contrat, facture, procédure, etc.). Le Term Store est le « vocabulaire contrôlé » — les étiquettes standardisées utilisées dans toute l'organisation.
  • Sans content types = chaque équipe invente ses colonnes — recherche et reporting impossibles
  • Avec Content Type Hub = les types de documents sont définis centralement et publiés sur tous les sites
Scénario : L'entreprise veut générer un rapport de tous les contrats signés en 2024. Sans content type « Contrat » avec colonne « Date de signature », il faut chercher manuellement dans 500 sites.
💡
Pourquoi chaque paramètre compte
  • Content Type Hub — Site SPO dédié qui publie les content types vers tous les sites. Permet la gouvernance centralisée des métadonnées.
  • Term Store (Managed Metadata) — Vocabulaire contrôlé (taxonomie) utilisé pour les colonnes de type « Managed Metadata ». Assure la cohérence des étiquettes.
  • Site Columns — Colonnes réutilisables définies au niveau du site ou du tenant. Évitent la recréation de colonnes identiques sur chaque bibliothèque.
  • Default Document Libraries settings — Permettent de forcer les content types et colonnes par défaut lors de la création de bibliothèques.
⚠️
Piège fréquent — Le Content Type Hub « classique » (site dédié + publication) est encore fonctionnel, mais Microsoft déploie progressivement la publication de content types depuis le SharePoint Admin Center (Content Services → Content type gallery). Les deux mécanismes coexistent mais peuvent créer des conflits si le même content type est publié des deux côtés.

Le Term Store n'est pas le même que les « labels » de sensitivity/retention de Purview. Ne pas confondre les termes (taxonomie) avec les sensitivity labels (protection).

🚨
Analogie — Moteur de recherche interne
Microsoft Search est le « Google de votre tenant » : il indexe SharePoint, OneDrive, Teams, Exchange et renvoie des résultats personnalisés. Les propriétés gérées sont les « filtres de recherche » (par département, type de document, date). Sans configuration, les utilisateurs trouvent tout… ou rien.

  • Avec Search bien configuré = filtres pertinents, bookmarks, acronymes, réponses instantanées
Scénario : Un collaborateur recherche « politique voyage ». Sans bookmarks ni propriétés gérées, il obtient 200 résultats. Avec un bookmark configuré, le document officiel apparaît en premier.
💡
Pourquoi chaque paramètre compte
  • Bookmarks & Q&A — Réponses administrées qui apparaissent en haut des résultats de recherche. Utile pour les questions fréquentes et documents de référence.
  • Acronyms — Définitions d'acronymes internes (ex: « DSI » = Direction des Systèmes d'Information). Apparaît en tooltip dans Search.
  • Managed Properties / Search Schema — Mapping entre les colonnes SharePoint et les propriétés de recherche. Permet le filtrage et le tri par métadonnées.
  • Result Sources — Scopes de recherche prédéfinis (ex: « Documents Finance uniquement »). Permet une recherche ciblée.
  • SharePoint Premium — Les services avancés de classification et d'extraction de métadonnées remplacent la terminologie historique Syntex. Selon le service, la facturation peut relever d'un add-on ou d'un modèle pay-as-you-go.
⚠️
Piège fréquent — Les propriétés gérées (ex: RefinableString00) doivent être mappées manuellement aux colonnes de site. Une re-indexation du site est nécessaire après modification du schéma Search (Library Settings → Advanced → « Reindex »). Le résultat n'est visible qu'après 24-48h.

SharePoint Premium (anciennement Syntex) ne doit pas être ramené à un seul prix par utilisateur : certains services de traitement documentaire sont désormais documentés en pay-as-you-go. Ne pas le confondre avec les capacités de recherche incluses dans SPO P1.

Compliance & Information Governance SPO P2
Labels sur sites, retention, DLP, audit logs, classification — 8 pts (renvois compliance)
🚨
Analogie — Armoire à archives certifiées
La compliance SharePoint est l'équivalent d'une armoire ignifugée avec clé : les sensitivity labels verrouillent l'accès aux sites, les retention policies définissent combien de temps garder les documents, et DLP empêche la fuite de données sensibles.
  • Sans compliance = les utilisateurs suppriment ou partagent librement des documents réglementés (RGPD, contractuels)
  • Avec compliance = protection automatique, rétention forcée, alertes DLP, audit trail complet
Scénario : Un employé partage un fichier contenant des données personnelles (RGPD) par lien anonyme. Sans DLP ni sensitivity label, aucune alerte ne se déclenche. Avec la compliance configurée, le partage est bloqué et l'admin notifié.
💡
Pourquoi chaque paramètre compte
  • Sensitivity Labels sur sites — Appliquent automatiquement le niveau de partage, l'accès externe et la confidentialité d'un site en fonction de son label. Requiert configuration dans purview-mip.html.
  • Retention Policies — Définissent combien de temps les documents sont conservés et quand ils sont supprimés. Appliquées au niveau site, bibliothèque ou tenant. Voir purview-compliance.html.
  • DLP Policies (Data Loss Prevention) — Détectent et bloquent le partage de données sensibles (n° CB, n° Sécu, IBAN). Voir purview-compliance.html.
  • Unified Audit Log — Enregistre toutes les actions (ouverture, partage, suppression) sur les documents SPO/ODB. Nécessaire pour investigations et conformité.
  • Information Barriers — Empêchent la communication et le partage entre segments d'utilisateurs (ex: Trading vs Research).
⚠️
Piège fréquent — Les sensitivity labels appliqués aux sites (privacy, external sharing, CA) sont différents des sensitivity labels appliqués aux fichiers (chiffrement, marquage). Les deux se configurent dans le Purview Compliance Portal, mais avec des options différentes.

Les retention policies et les sensitivity labels sont des fonctionnalités Purview, pas SharePoint. Ce guide couvre uniquement leur application aux sites/bibliothèques SPO. Pour la configuration complète, se référer aux guides dédiés :
  • purview-mip.html — Configuration complète MIP / Sensitivity Labels
  • purview-compliance.html — DLP, Retention, Audit
  • sentinel.html — Intégration Sentinel pour alertes SPO

Monitoring & Reporting SPO P1
Rapports Admin Center, usage analytics, PowerShell reporting — 5 pts
🚨
Analogie — Tableau de bord d'un immeuble
Le monitoring SharePoint est le tableau de bord du gardien : il montre qui entre, quels étages sont les plus fréquentés, quels espaces sont vides, et où les incidents se produisent. Sans monitoring, vous gérez à l'aveugle.
  • Sans monitoring = découvert des problèmes par tickets utilisateurs
  • Avec monitoring = anticipation proactive, justification budget, adoption mesurée
Scénario : Le COMEX demande si la migration SharePoint est adoptée. Sans rapports d'usage configurés, impossible de répondre factuellement.
💡
Pourquoi chaque paramètre compte
  • SharePoint Admin Center Reports — Rapports intégrés : utilisation des sites, fichiers partagés, stockage consommé, activité par site. Accessibles sans PowerShell.
  • Microsoft 365 Usage Reports — Rapports cross-workload (SPO + ODB + Teams) dans le M365 Admin Center. Historique 7/30/90/180 jours.
  • Site Analytics — Rapports au niveau de chaque site : pages vues, visiteurs uniques, fichiers populaires. Activé par défaut.
  • PowerShell Custom Reports — Scripts d'audit personnalisés pour extraire des données non disponibles dans les rapports standard (ex: sites inactifs, quotas, permissions).
⚠️
Piège fréquent — Les rapports d'usage du M365 Admin Center masquent les noms d'utilisateurs par défaut depuis 2021 (privacy). Pour les voir, un Global Admin doit aller dans Settings → Org settings → Reports → décocher « Display concealed user, group, and site names ».

Les rapports SPO Admin Center ont un délai de 24-48h. Ne pas s'alarmer si les données du jour ne sont pas visibles. Pour du temps réel, utiliser l'Unified Audit Log (S9).
