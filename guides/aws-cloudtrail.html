Analogie : CloudTrail = « registre du notaire » — chaque action AWS est enregistrée, horodatée et signée. Un Trail = « greffier permanent » — copie tout ce qui se passe dans le tribunal AWS. La Log Validation = « sceau d'authenticité » — impossible de falsifier les preuves sans que ça se voie.

Solutions écartées

❌
Approches étudiées mais non retenues dans ce guide :
  • CloudFormation plutôt que Terraform : Terraform a été retenu car il offre un state management explicite, un plan d'exécution prévisible (terraform plan), et une compatibilité multi-cloud. CloudFormation reste valid, mais génère plus de drift et est limitg aux environnements AWS purs.
  • Un trail par compte plutôt qu'Organization Trail : La multiplication des trails indépendants augmente les coûts, rend la centralisation des logs complexe et crée des risques de configuration divergente. L'Organization Trail (S7) est la solution recommandée dès que plusieurs comptes AWS sont gérés.
  • CloudTrail Lake comme unique solution de stockage (remplacement total de S3) : CloudTrail Lake est adapté pour les requêtes SQL et la rétention long terme, mais son coût à l'ingestion le rend inadapté pour remplacer un trail classique sur des volumes élevés de Data Events. L'approche hybride (trail classique S3 + Lake pour investigation) est privilégiée.
  • Event History uniquement (sans trail persistant) : Event History conserve seulement 90 jours de Management Events sans Data Events et n'est pas intégrable à un SIEM. Non retenu car ne satisfait aucune exigence PCI DSS 10.5.1 (12 mois) ni CIS 4.1.
  • Fluentd / Logstash pour la centralisation des logs : Non retenu — AWS Kinesis Data Firehose est natif AWS, plus simple à configurer dans ce contexte, et ne nécessite pas d'infrastructure supplémentaire. Les outils tiers restent valides pour les environnements hybrides mais sortent du périmètre de ce guide.

Configuration des Trails Gratuit 1er trail Payant trails+
Trail multi-région, management events Read+Write, log file validation SHA-256, CW Logs, SNS — le greffier permanent de votre compte AWS
⚠️
Scénario d'attaque : Un attaquant compromet un rôle IAM en région eu-west-2 et crée des ressources EC2 pour du crypto-mining. Votre trail, configuré uniquement sur us-east-1, n'enregistre rien. L'incident est découvert 6 semaines plus tard par une facture anormale — sans aucune trace forensique. Sans un trail multi-région avec log file validation, vous n'avez ni témoins ni preuves.

Analogie : Un trail mono-région = un greffier qui n'assiste qu'aux audiences de la salle A. Tout ce qui se passe dans les salles B, C et D est invisible. Un trail multi-région avec log validation = le registre notarié de l'intégralité du palais de justice, avec chaque feuillet signé et daté.

1.1 — Tableau de configuration complet

Paramètre Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Best Practice Source
IsMultiRegionTrail false true true true CIS 4.1
EnableLogFileValidation false true true true CIS 3.2
ReadWriteType (management events) WriteOnly All All All (Read + Write) CIS 3.5
IncludeGlobalServiceEvents false (si mono-région) true true true (capture IAM, STS, CloudFront) AWS Docs
CloudWatchLogsLogGroupArn non configuré Log group dédié Log group + metric filters Log group + 15 alarmes CIS (voir S6) CIS 3.4
SnsTopicName non configuré Optionnel SNS dédié SNS + abonnement email/SIEM AWS Docs
KMSKeyId (chiffrement) non chiffré (SSE-S3) Optionnel KMS CMK dédié aws/cloudtrail ou CMK custom CIS 3.7
IsLogging true true true true + alerte si false AWS Docs
💡
Pourquoi chaque paramètre compte :
  • IsMultiRegionTrail : Un trail mono-région crée des angles morts dans toutes les autres régions AWS. Les attaquants l'exploitent en pivotant vers des régions secondaires. CIS AWS v7.0.0 contrôle 3.1 — obligatoire.
  • EnableLogFileValidation : Sans validation (digest SHA-256), un attaquant qui accède au bucket S3 peut modifier ou supprimer les logs sans que l'intégrité puisse être vérifiée. C'est l'équivalent d'un sceau notarial — la preuve judiciaire ne tient qu'avec lui. CIS AWS v7.0.0 contrôle 3.2.
  • ReadWriteType = All : La valeur par défaut WriteOnly ne capture pas les read events (GetObject, DescribeInstances, etc.). Toute exfiltration de données via des appels de lecture (GetSecretValue, DescribeDBInstances) sera invisible. CIS 3.5.
  • IncludeGlobalServiceEvents : IAM, STS, CloudFront et Route 53 sont des services globaux qui émettent leurs événements dans us-east-1. Sans ce flag, toutes les opérations IAM (création de rôles, AssumeRole) sont invisibles dans les autres régions.
  • CloudWatchLogsLogGroupArn : Sans intégration CW Logs, aucune alarme en temps réel n'est possible. Les logs S3 sont disponibles après ~15 min de délai. CIS 3.4 — requis pour les 15 alarmes de S6.
  • KMSKeyId : Le chiffrement SSE-S3 par défaut utilise une clé gérée par AWS sans rotation configurable ni audit d'accès. Un CMK KMS permet de contrôler qui peut déchiffrer les logs (forensique réglementée). CIS 3.7.
⚠️
Piège frquent : Activer un trail multi-région via la console désactive silencieusement les trails manuels en dédoublon dans chaque région. Vérifier avec aws cloudtrail describe-trails --include-shadow-trails qu'il n'existe pas de trail "shadow" sans log validation.
Second piège : IsMultiRegionTrail=true ne déclenche pas automatiquement les Data Events (S3, Lambda) sur toutes les régions — ceux-ci doivent être configurés explicitement (voir S3).

1.2 — Configuration via AWS CLI

AWS CLI — Création d'un trail multi-région sécurisé
# Variables — adapter à votre environnement
TRAIL_NAME="cloudtrail-main"
BUCKET_NAME="my-cloudtrail-logs-$(aws sts get-caller-identity --query Account --output text)"
LOG_GROUP_ARN="arn:aws:logs:eu-west-1:ACCOUNT_ID:log-group:/aws/cloudtrail/main"
CW_ROLE_ARN="arn:aws:iam::ACCOUNT_ID:role/CloudTrail-CloudWatchLogs-Role"
KMS_KEY_ID="arn:aws:kms:eu-west-1:ACCOUNT_ID:key/KEY_ID"
REGION="eu-west-1"
PROFILE="audit-admin"

# [1/4] Créer le trail multi-région avec toutes les options de sécurité
aws cloudtrail create-trail \
    --name "$TRAIL_NAME" \
    --s3-bucket-name "$BUCKET_NAME" \
    --is-multi-region-trail \
    --enable-log-file-validation \
    --include-global-service-events \
    --cloud-watch-logs-log-group-arn "$LOG_GROUP_ARN" \
    --cloud-watch-logs-role-arn "$CW_ROLE_ARN" \
    --kms-key-id "$KMS_KEY_ID" \
    --profile "$PROFILE" \
    --region "$REGION" \
    --output json

# [2/4] Activer le logging (obligatoire après create)
aws cloudtrail start-logging \
    --name "$TRAIL_NAME" \
    --profile "$PROFILE" \
    --region "$REGION"

# [3/4] Configurer les management events Read+Write
aws cloudtrail put-event-selectors \
    --trail-name "$TRAIL_NAME" \
    --event-selectors '[{"ReadWriteType":"All","IncludeManagementEvents":true,"DataResources":[]}]' \
    --profile "$PROFILE" \
    --region "$REGION"

# [4/4] Taguer le trail
aws cloudtrail add-tags \
    --resource-id "$TRAIL_NAME" \
    --tags-list '[{"Key":"Environment","Value":"production"},{"Key":"ManagedBy","Value":"manual"}]' \
    --profile "$PROFILE" \
    --region "$REGION"

Intégrité & Protection des Logs Gratuit
Bucket policy restrictive, MFA Delete, SCP anti-tampering, Glacier, rôle IAM dédié — protéger les preuves contre toute falsification
🚨
Scénario d'attaque : Un attaquant interne ayant les droits AdministratorAccess exécute des opérations non autorisées, puis appelle aws cloudtrail stop-logging et supprime les fichiers logs du bucket S3 pour couvrir ses traces. Sans une SCP bloquant StopLogging, une bucket policy avec Deny Delete et MFA Delete, il peut éliminer toutes les preuves en moins de 2 minutes.

Analogie : Un bucket CloudTrail sans protection = des archives judiciaires stockées dans un placard ouvert. N'importe quel employé peut les brûler. La SCP + bucket policy + MFA Delete = le coffre-fort du greffe, accessible uniquement avec deux clés simultanées et interdit d'accès même au Procureur sans procédure formelle.

Lifecycle S3 → Glacier Aucun lifecycle Optionnel S3-IA après 90j S3-IA 90j → Glacier 365j PCI DSS req. 10.5.1
S3 Object Lock (WORM) Désactivé Non requis en QW Compliance mode 90j Governance mode 365j SOC 2, NIST AU-9
💡
Pourquoi chaque mesure compte :
  • Deny s3:DeleteObject : Sans ce deny, un administrateur compromise peut supprimer les logs et couvrir ses traces. La condition NotPrincipal de type "break-glass" permet de conserver une porte de sortie pour les urgences forensiques.
  • Block Public Access : Un bucket de logs CloudTrail public expose les actions de tous les utilisateurs (noms de rôles, ARN de ressources, patterns d'usage) — une carte complète de votre infrastructure pour un attaquant. CIS 3.3.
  • MFA Delete : Exige une second facteur physique pour supprimer un objet versioné. Même un token IAM volé ne suffit plus. Base légale : ISO 27001 A.12.4.2 (protection des informations de log). Rétention minimum 12 mois recommandée par PCI DSS req. 10.7.
  • SCP anti-tampering : Un Service Control Policy (SCP) s'applique au niveau de l'organisation AWS et ne peut être contourna par aucun administrateur du compte membre, même avec les droits AdministratorAccess. C'est le seul mécanisme qui résiste à une compromission totale d'un compte.
  • Lifecycle Glacier : Les logs après 90 jours ont une valeur forensique mais pas opérationnelle. Glacier réduit le coût de 95% tout en conservant les preuves. PCI DSS req. 10.5.1 exige 12 mois de rétention minimum. NIST 800-53 AU-11.
⚠️
Piège frquent : La SCP seule ne suffit pas — elle bloque les appels API mais pas la suppression directe des objets S3 par un administrateur du compte. Les deux couches (SCP + bucket policy) sont obligatoires. De même, activer MFA Delete sur un bucket existant avec des objets en cours nécessite un accès root temporaire via la console AWS (la seule entité autorisée à activer MFA Delete via CLI).

{

  "Statement": [
    {

    --policy-id "p-XXXXXXXXXX" \

Data Events 0,10 $/100k events
S3, Lambda, DynamoDB data events, Advanced Event Selectors — visibilité sur les accès aux données critiques
📦
Scénario d'attaque : Un attaquant accède à un bucket S3 contenant des données médicales et télécharge 50 000 fichiers via des appels GetObject. Les management events (trail par défaut) n'enregistrent que la création et suppression du bucket — pas les accès aux objets. Sans Data Events activés, l'exfiltration est totalement invisible. Votre DPO demande les logs pendant la notification RGPD : ils n'existent pas.

Analogie : Les management events = le registre des locataires d'un immeuble (qui a la clé). Les data events = la caméra qui enregistre chaque entrée dans chaque appartement. Sans la caméra, vous savez que quelqu'un avait la clé — mais pas qu'il a vidé l'appartement.

Type de Data Event Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Best Practice Source

💡
Pourquoi chaque paramètre compte :
  • S3 GetObject : 80% des fuites de données S3 passent par des accès GetObject en masse. Sans ce data event, une exfiltration RGPD est intraçable. CIS 3.8 — obligatoire pour les comptes traitées données sensibles.
  • S3 DeleteObject : Permet de détecter les suppressions massives de données (ransomware, sabotage interne). Sans ce log, impossible de déterminer quels objets ont été supprimés.
  • Lambda Invoke : Les fonctions Lambda sont souvent utilisées comme vecteurs de pivot (injection de code, exfiltration de secrets via GetSecretValue). Tracer les invocations permet de corréler les exécutions suspectes. CIS 3.9.
  • Advanced Event Selectors (AES) : Plus granulaires que les event selectors basiques — permettent de filtrer par tag S3, nom de bucket, préfixe d'objet. Réduisent le coût en excluant les accès read-only sur les buckets de logs. Recommandé pour les comptes à fort volume S3.
  • Exclusion buckets de logs : Sans exclusion du bucket CloudTrail lui-même, chaque écriture de log génère un Data Event, qui génère un log, qui génère un Data Event — boucle récursive exponentiellement coûteuse.
⚠️
Piège coût : Activer les Data Events sur tous les buckets S3 d'un compte à fort volume peut multiplier la facture CloudTrail par 10 à 50. Utiliser les Advanced Event Selectors pour cibler les buckets contenant des données sensibles (tag DataClassification=Sensitive) et exclure les buckets de logs, de cache CDN et de backups. Estimer le coût avant d'activer via CloudWatch Metric DataEventsDelivered.

5.3 — 10 requêtes Athena essentielles pour l'investigation

Athena SQL — 10 requêtes d'investigation IR
-- 1. Qui a supprimé la ressource X ? (chercher par nom de ressource)
SELECT eventTime, userIdentity.arn, eventName, awsRegion, sourceIPAddress
FROM cloudtrail_logs
WHERE requestParameters LIKE '%my-deleted-resource%'
  AND eventName LIKE 'Delete%'
  AND eventTime BETWEEN '2026-04-01' AND '2026-04-30'
ORDER BY eventTime DESC;

-- 2. Connexions console sans MFA
SELECT eventTime, userIdentity.userName, sourceIPAddress, awsRegion
FROM cloudtrail_logs
WHERE eventName = 'ConsoleLogin'
  AND additionalEventData LIKE '%"MFAUsed":"No"%'
  AND errorCode IS NULL
ORDER BY eventTime DESC;

-- 3. Toutes les AssumeRole (qui usurpe quel rôle ?)
SELECT eventTime,
       userIdentity.arn                                    AS caller,
       json_extract_scalar(requestParameters, '$.roleArn') AS assumed_role,
       sourceIPAddress, awsRegion
FROM cloudtrail_logs
WHERE eventName = 'AssumeRole'
ORDER BY eventTime DESC LIMIT 200;

-- 4. Créations de clés d'accès IAM (AccessKeyId créé)
SELECT eventTime, userIdentity.arn, responseElements,
       json_extract_scalar(requestParameters, '$.userName') AS target_user
FROM cloudtrail_logs
WHERE eventName = 'CreateAccessKey'
ORDER BY eventTime DESC;

-- 5. Appels API depuis des IPs non connues (adapter la liste)
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, awsRegion
FROM cloudtrail_logs
WHERE sourceIPAddress NOT LIKE '10.%'
  AND sourceIPAddress NOT LIKE '192.168.%'
  AND sourceIPAddress NOT IN ('203.0.113.10', '198.51.100.5')
  AND userIdentity.type = 'IAMUser'
ORDER BY eventTime DESC LIMIT 500;

-- 6. Ouvertures de Security Group vers 0.0.0.0/0
SELECT eventTime, userIdentity.arn, sourceIPAddress,
       json_extract_scalar(requestParameters, '$.groupId') AS sg_id
FROM cloudtrail_logs
WHERE eventName = 'AuthorizeSecurityGroupIngress'
  AND requestParameters LIKE '%0.0.0.0/0%'
ORDER BY eventTime DESC;

-- 7. Utilisation du compte root
SELECT eventTime, eventName, sourceIPAddress, awsRegion, userAgent
FROM cloudtrail_logs
WHERE userIdentity.type = 'Root'
  AND eventType != 'AwsServiceEvent'
ORDER BY eventTime DESC;

-- 8. Modifications de bucket policy S3
SELECT eventTime, userIdentity.arn, eventName, sourceIPAddress,
       json_extract_scalar(requestParameters, '$.bucketName') AS bucket
FROM cloudtrail_logs
WHERE eventName IN ('PutBucketPolicy', 'DeleteBucketPolicy',
                      'PutBucketAcl', 'PutBucketPublicAccessBlock')
ORDER BY eventTime DESC;

-- 9. Vague de refus d'accès (brute-force / escalade de privilèges)
SELECT userIdentity.arn, sourceIPAddress,
       COUNT(*) AS access_denied_count,
       MIN(eventTime) AS first_seen,
       MAX(eventTime) AS last_seen
FROM cloudtrail_logs
WHERE errorCode = 'AccessDenied'
  AND eventTime > date_format(date_add('hour', -24, now()), '%Y-%m-%dT%H:%i:%sZ')
GROUP BY 1, 2
HAVING COUNT(*) > 10
ORDER BY access_denied_count DESC;

-- 10. Appels API depuis des régions non autorisées
SELECT eventTime, eventName, userIdentity.arn, awsRegion, sourceIPAddress
FROM cloudtrail_logs
WHERE awsRegion NOT IN ('eu-west-1', 'eu-central-1', 'us-east-1')
  AND eventType = 'AwsApiCall'
ORDER BY eventTime DESC LIMIT 200;

Pourquoi chaque alarme CIS compte :
  • CIS 2.1 Root : Le compte root ne devrait jamais être utilisé — toute utilisation est une anomalie majeure à investiguer immédiatement.
  • CIS 3.2 Console sans MFA : Indique soit une session console ouverte sans MFA activé (déficit de politique IAM), soit une compromission de compte.
  • CIS 3.4 StopLogging : Action typique d'un attaquant cherchant à éviter la détection — alerte immédiate requise.
  • CIS 3.6 CMK : La suppression/désactivation d'une clé KMS rend les données chiffrées définitivement inaccessibles — impact très élevé.
  • CIS 3.9-3.13 Réseau : Toute modification du réseau (SG, NACL, routes, VPC) peut ouvrir des vecteurs d'exfiltration ou de pivoting.
⚠️
Piège bruit d'alerte : CIS 4.1 (Unauthorized API calls) peut générer des centaines d'alertes par jour dans un compte actif (scripts qui testent les permissions, rotate credentials, etc.). Configurer le threshold CloudWatch Alarm à une valeur supérieure à 1 (ex: >10 en 5 minutes) et utiliser un filtre de métrique précis. CIS 3.7 (S3 bucket policy) peut aussi être bruité si Terraform applique des changements fréquemment.

8.1 — Trail classique vs CloudTrail Lake

Critère Trail classique + S3/Athena CloudTrail Lake (EDS) Recommandation
Rétention maximum Illimitée (S3) 3653 jours max (option 1 an extensible) ou 2557 jours (option 7 ans) Lake pour SQL natif et conformité si l'option de rétention correspond au besoin
Coût ingestion CloudTrail (trails) + S3 storage Ingestion facturée par Go avec option 1 an extensible ou option 7 ans Évaluer selon le volume mensuel et la durée de rétention avant de migrer
Vitesse de requête Athena (5$/To scanné) SQL natif intégré Lake pour les investigations fréquentes
Data Events Configurés dans le trail Configurés dans l'Event Data Store Identique — AES disponibles dans les deux
Organization support Organization Trail Organization EDS natif EDS Org couvre tous les comptes membres
💡
Pourquoi activer CloudTrail Lake :
  • Rétention configurable : Lake permet de viser 2557 jours (option 7 ans) ou jusqu'à 3653 jours avec l'option 1 an extensible. Le trail classique vers S3 reste plus flexible si l'organisation veut un archivage au-delà ou différent par bucket.
  • SQL natif : Pas besoin de créer des tables Glue ni de gérer des partitions. Les requêtes SQL s'exécutent directement sur l'Event Data Store.
  • EDS Activity : Option "non-AWS events" — permet d'ingestion des events d'applications tierces (Okta, GitHub, Salesforce) dans le même store pour corrélation.
  • Immuabilité : Les EDS sont immuables — impossible de supprimer un event individuel, contrairement aux objects S3 (même avec Object Lock, l'accès au bucket est complexe).
⚠️
Lake n'est pas un remplacement total : CloudTrail Lake et les trails classiques sont complémentaires. Lake = requêtes et conformité long terme. Trail classique + CloudWatch Logs = alertes temps réel (S6). Pour les alarmes CIS de S6, le trail vers CloudWatch Logs reste indispensable.

Configuration complète de CloudTrail en une seule exécution
⚠️
Variables à définir avant exécution : Remplacer les variables en tête de script (ACCOUNT_ID, REGION, PROFILE, etc.) avant de l'exécuter. Le script est idémpotent — il vérifie l'existence avant de créer.
Bash / AWS CLI — Script All-in-One CloudTrail
#!/usr/bin/env bash
# ══════════════════════════════════════════════════════════════════
# AWS CloudTrail — Configuration All-in-One
# Phases : Trail → Intégrité → Data Events → Insights → Alarmes CIS
# Idempotent : vérifie l'existence avant création
# ══════════════════════════════════════════════════════════════════

# ── Variables ────────────────────────────────────────────────────
ACCOUNT_ID="123456789012"
REGION="eu-west-1"
PROFILE="audit-profile"
TRAIL_NAME="main-cloudtrail"
BUCKET="cloudtrail-logs-${ACCOUNT_ID}"
KMS_KEY_ALIAS="cloudtrail-key"
LOG_GROUP="/aws/cloudtrail/main"
SNS_TOPIC="cloudtrail-security-alerts"
AWS_CMD="aws --profile $PROFILE --region $REGION"

echo "[1/7] Vérification/création du bucket S3..."
if ! $AWS_CMD s3api head-bucket --bucket $BUCKET 2>/dev/null; then
    $AWS_CMD s3api create-bucket \
        --bucket $BUCKET \
        --create-bucket-configuration LocationConstraint=$REGION
    $AWS_CMD s3api put-public-access-block \
        --bucket $BUCKET \
        --public-access-block-configuration \
        BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
    $AWS_CMD s3api put-bucket-versioning \
        --bucket $BUCKET \
        --versioning-configuration Status=Enabled
    echo "  ✓ Bucket créé : $BUCKET"
else
    echo "  ✓ Bucket existe déjà"
fi

echo "[2/7] Vérification/création de la clé KMS..."
KMS_KEY_ID=$($AWS_CMD kms describe-key \
    --key-id alias/$KMS_KEY_ALIAS \
    --query 'KeyMetadata.KeyId' \
    --output text 2>/dev/null || echo "")

if [ -z "$KMS_KEY_ID" ]; then
    KMS_KEY_ID=$($AWS_CMD kms create-key \
        --description "CloudTrail encryption key" \
        --query 'KeyMetadata.KeyId' \
        --output text)
    $AWS_CMD kms create-alias \
        --alias-name alias/$KMS_KEY_ALIAS \
        --target-key-id $KMS_KEY_ID
    echo "  ✓ Clé KMS créée : $KMS_KEY_ID"
else
    echo "  ✓ Clé KMS existe : $KMS_KEY_ID"
fi

echo "[3/7] Vérification/création du trail CloudTrail..."
TRAIL_EXISTS=$($AWS_CMD cloudtrail get-trail \
    --name $TRAIL_NAME \
    --query 'Trail.Name' \
    --output text 2>/dev/null || echo "")

if [ -z "$TRAIL_EXISTS" ]; then
    $AWS_CMD cloudtrail create-trail \
        --name $TRAIL_NAME \
        --s3-bucket-name $BUCKET \
        --is-multi-region-trail \
        --enable-log-file-validation \
        --include-global-service-events \
        --kms-key-id $KMS_KEY_ID \
        --cloud-watch-logs-log-group-arn \
            "arn:aws:logs:${REGION}:${ACCOUNT_ID}:log-group:${LOG_GROUP}:*" \
        --cloud-watch-logs-role-arn \
            "arn:aws:iam::${ACCOUNT_ID}:role/CloudTrailToCloudWatchLogs"
    $AWS_CMD cloudtrail start-logging --name $TRAIL_NAME
    echo "  ✓ Trail créé et activé"
else
    echo "  ✓ Trail existe : $TRAIL_EXISTS"
fi

echo "[4/7] Configuration Data Events (Advanced Event Selectors)..."
$AWS_CMD cloudtrail put-event-selectors \
    --trail-name $TRAIL_NAME \
    --advanced-event-selectors '[
  {"Name":"S3 Write+Delete","FieldSelectors":[
    {"Field":"eventCategory","Equals":["Data"]},
    {"Field":"resources.type","Equals":["AWS::S3::Object"]},
    {"Field":"eventName","StartsWith":["PutObject","DeleteObject","CopyObject"]}
  ]},
  {"Name":"Lambda Invoke","FieldSelectors":[
    {"Field":"eventCategory","Equals":["Data"]},
    {"Field":"resources.type","Equals":["AWS::Lambda::Function"]}
  ]}
]'
echo "  ✓ Data Events configurés"

echo "[5/7] Activation CloudTrail Insights..."
$AWS_CMD cloudtrail put-insight-selectors \
    --trail-name $TRAIL_NAME \
    --insight-selectors \
    '[{"InsightType":"ApiCallRateInsight"},{"InsightType":"ApiErrorRateInsight"}]'
echo "  ✓ Insights activés (baseline ~7 jours)"

echo "[6/7] Création topic SNS pour les alarmes..."
SNS_ARN=$($AWS_CMD sns list-topics \
    --query "Topics[?ends_with(TopicArn, ':${SNS_TOPIC}')].TopicArn" \
    --output text)
if [ -z "$SNS_ARN" ]; then
    SNS_ARN=$($AWS_CMD sns create-topic \
        --name $SNS_TOPIC \
        --query 'TopicArn' \
        --output text)
    echo "  ✓ Topic SNS créé : $SNS_ARN"
else
    echo "  ✓ Topic SNS existe : $SNS_ARN"
fi

echo "[7/7] Création filtre métrique + alarme CIS (Root Activity)..."
$AWS_CMD logs put-metric-filter \
    --log-group-name $LOG_GROUP \
    --filter-name "CIS-3.3-RootActivity" \
    --filter-pattern \

Glossaire
Définition des termes clés CloudTrail
TermeDéfinition
TrailConfiguration CloudTrail qui spécifie quels événements capturer et où les livrer (S3, CloudWatch Logs). Analogie : le greffier permanent qui note toutes les actions.
Management EventsÉvénements de gestion des ressources AWS (CreateBucket, DescribeInstances, PutRolePolicy...). Inclus gratuitement pour le premier trail par région. Couverture : 99%+ des API AWS.
Data EventsÉvénements sur les données (S3 GetObject/PutObject, Lambda Invoke, DynamoDB GetItem...). Payants (0,10$/100k). Essentiels pour les audits de conformité PCI DSS / HIPAA.
Digest FileFichier hashé généré toutes les heures par CloudTrail, listant tous les fichiers de logs de l'heure écoulée et leur hash SHA-256. Permet de détecter toute modification ou suppression.
InsightsFonctionnalité CloudTrail qui détecte les activités inhabituelles en comparant les appels API courants à une baseline établie sur 7 jours. Types : ApiCallRateInsight et ApiErrorRateInsight.
Event Data Store (EDS)Composant de CloudTrail Lake — un stockage immuable, requêtable en SQL, avec rétention configurable jusqu'à 7 ans (2557 jours). Remplace S3+Athena pour les requêtes.
CloudTrail LakeService de journalisation et d'analyse avancé basé sur des Event Data Stores. SQL natif, rétention 7 ans, support multi-source (AWS events + non-AWS events). Analogie : bibliothèque nationale numérisée.
Event HistoryHistorique des 90 derniers jours d'événements de management, accessible directement dans la console CloudTrail. Gratuit, limité aux management events.
Advanced Event Selectors (AES)Filtres précis pour les Data Events — permettent de cibler des buckets, des préfixes S3, des fonctions Lambda ou des opérations spécifiques. Réduisent les coûts vs selectors classiques.
Organization TrailTrail CloudTrail créé dans le management account (ou delegated admin) qui collecte automatiquement les événements de tous les comptes membres de l'AWS Organization.
API Call Rate InsightType d'Insight qui détecte quand le volume d'appels API dépasse significativement la baseline normale. Utile pour détecter les attaques par énumération ou les runaway scripts Terraform.
API Error Rate InsightType d'Insight qui détecte les pics d'erreurs API (AccessDenied, ThrottlingException...). Signal d'une reconnaissance, d'une attaque par force brute ou d'un problème de permissions.
Log File ValidationOption du trail qui génére des fichiers digest signés numériquement (RSA-2048). Permet de prouver qu'un fichier de log n'a pas été modifié post-délivraison. CIS 3.2, NIST AU-9.
