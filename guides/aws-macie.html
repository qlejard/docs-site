📋
Ce document est un référentiel de sécurité des données AWS couvrant la découverte, la classification et la protection des données sensibles via Amazon Macie. Il s'adresse aux équipes sécurité, compliance et cloud qui doivent inventorier et protéger les données personnelles (PII), financières et confidentielles stockées dans Amazon S3. Chaque section fournit un double chemin AWS CLI + Terraform.

Analogie générale

🔍
Amazon Macie = détective spécialisé en données sensibles. Il parcourt méthodiquement vos fichiers S3 pour trouver ce qui est confidentiel (numéros de carte bancaire, identifiants personnels, secrets). Sans Macie, vos données sensibles sont là — mais personne ne sait où, ni combien, ni quelles règles elles violent.

Pourquoi chaque paramètre compte :
  • Macie Status : Macie n'est jamais activé par défaut. Sans activation explicite, aucune découverte de données sensibles n'a lieu — vos buckets S3 sont des boîtes noires.
  • Automated Discovery : l'échantillonnage continu (« patrouille régulière ») détecte les nouvelles données sensibles déposées après le scan initial. Sans lui, un fichier sensible ajouté le lendemain du scan n'est détecté que lors du prochain job ciblé.
  • Classification Scope : sans scope, Macie scanne tout — y compris les buckets de logs et de backup (CIS AWS Foundations v7.0.0 §3.1.4 Block Public Access)ts de logs/temp qui gonflent la facture sans valeur de découverte. CIS AWS 2.1.4 recommande le ciblage.
  • Service-Linked Role : AWSServiceRoleForAmazonMacie est créé automatiquement mais doit être protégé par SCP pour empêcher sa suppression accidentelle (arrêt immédiat de Macie).
  • Reveal Samples : permet de voir un extrait des données détectées dans la console pour valider les findings sans télécharger les fichiers — accélère le triage.
  • Discovery Result Encryption : les résultats de découverte contiennent des métadonnées sur les données sensibles. RGPD Art. 32 et PCI DSS Req. 3.4 exigent leur chiffrement. CMK KMS = contrôle total des accès.
  • Publication Destination : sans publication, les findings restent uniquement dans la console Macie — pas d'agrégation, pas d'alerting, pas de workflow de remediation.
  • Multi-Region : Macie est régional. Un bucket dans us-east-1 n'est pas scanné par une activation dans eu-west-1. Les données sensibles dans des régions non couvertes sont invisibles.
⚠️
Piège : Macie est régional — il faut l'activer dans chaque région où des buckets S3 existent. Un oubli de région = zéro visibilité sur les données de cette région. De plus, l'automated discovery est facturé après 30 jours — surveiller les métriques EstimatedCost dans CloudWatch.

1.3 — Configuration — AWS CLI

Bash — Activer Macie et configurer l'automated discovery
# Activer Amazon Macie dans la région courante
aws macie2 enable-macie \
  --profile audit-admin \
  --region eu-west-1

# Activer l'automated sensitive data discovery
aws macie2 update-automated-discovery-configuration \
  --profile audit-admin \
  --region eu-west-1 \
  --status ENABLED

# Configurer la publication des findings vers Security Hub
aws macie2 put-findings-publication-configuration \
  --profile audit-admin \
  --region eu-west-1 \
  --security-hub-configuration '{STR0:true,STR1:true}'

# Configurer le bucket de destination des discovery results
aws macie2 put-classification-export-configuration \
  --profile audit-admin \
  --region eu-west-1 \
  --configuration '{
    "s3Destination": {
      "bucketName": "my-macie-results-bucket",
      "keyPrefix": "macie-results/",
      "kmsKeyArn": "arn:aws:kms:eu-west-1:123456789012:key/my-macie-key-id"
    }
  }'

# Activer la prévisualisation des échantillons sensibles
aws macie2 update-reveal-configuration \
  --profile audit-admin \
  --region eu-west-1 \
  --configuration '{STR0:STR1}'

1.4 — Configuration — Terraform

Terraform — Activation Macie + automated discovery
# Provider AWS avec région et tags par défaut
provider "aws" {
  region = "eu-west-1"
  default_tags {
    tags = {
      Environment = "production"
      ManagedBy   = "terraform"
      Project     = "data-protection"
    }
  }
}

# Activer Amazon Macie
resource "aws_macie2_account" "main" {}

# Activer l'automated sensitive data discovery
resource "aws_macie2_automated_discovery_configuration" "main" {
  status = "ENABLED"

  depends_on = [aws_macie2_account.main]
}

# Bucket de destination pour les discovery results
resource "aws_s3_bucket" "macie_results" {
  bucket = "my-macie-results-${data.aws_caller_identity.current.account_id}"
}

resource "aws_s3_bucket_server_side_encryption_configuration" "macie_results" {
  bucket = aws_s3_bucket.macie_results.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.macie.arn
    }
  }
}

resource "aws_s3_bucket_public_access_block" "macie_results" {
  bucket                  = aws_s3_bucket.macie_results.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Clé KMS dédiée pour les résultats Macie
resource "aws_kms_key" "macie" {
  description             = "KMS key for Macie discovery results"
  deletion_window_in_days = 30
  enable_key_rotation     = true
}

# Configuration de l'export des résultats
resource "aws_macie2_classification_export_configuration" "main" {
  s3_destination {
    bucket_name = aws_s3_bucket.macie_results.bucket
    key_prefix  = "macie-results/"
    kms_key_arn = aws_kms_key.macie.arn
  }

  depends_on = [aws_macie2_account.main]
}

data "aws_caller_identity" "current" {}

Pourquoi chaque contrôle compte :
  • Buckets publics : cause n°1 des fuites AWS médiatisées. Macie détecte le « SharedWithEveryone » même si AWS Config ne le signale pas (différence : Macie vérifie ACLs + bucket policy + access point policy).
  • Chiffrement manquant : depuis jan. 2023 S3 applique SSE-S3 par défaut, mais les buckets créés avant ou avec chiffrement désactivé manuellement restent vulnérables. PCI DSS Req 3.4 exige un chiffrement fort (KMS, pas SSE-S3).
  • Accès cross-account : un bucket partagé avec un compte externe peut être légitime (partenaire) ou non (erreur de policy). Macie identifie précisément les comptes externes ayant accès.
  • Versioning : sans versioning, un ransomware (ou erreur humaine) détruit définitivement les objets (CIS AWS Foundations v7.0.0 §3.1.3)initivement les données. CIS AWS 2.1.3 le recommande fortement pour les buckets contenant des données sensibles.
  • Réplication : une réplication cross-région non contrôlée peut violer les exigences de résidence des données (RGPD Art. 44 — transfert hors UE).
  • Taille & croissance : un bucket qui double de taille en une semaine peut indiquer un dump non autorisé ou un data hoarding — signal précoce d'incident.

💡
Pourquoi chaque paramètre compte :
  • Discovery Jobs : sans job, Macie n'analyse que les métadonnées S3 (posture). Pour découvrir les PII dans les fichiers, il faut un job explicite ou l'automated discovery. Le job ciblé (« targeted ») scanne en profondeur mais coûte 1 $/Go.
  • Managed Data Identifiers : 200+ patterns prédéfinis (CB, passeport, NIR, IBAN, email, clés API...). Les activer tous donne une couverture maximale, mais génère du bruit sur les fichiers de test. Cibler par compliance (PCI → CB uniquement) réduit les faux positifs.
  • Allow Lists : les données de test/démo (ex: CB 4111 1111 1111 1111) génèrent des findings. Les allow lists éviten le bruit sans désactiver la détection.
  • Sampling : sur un bucket de 10 To de logs, scanner 100% coûte 10 000 $. Un sampling à 20% détecte la majorité des problèmes pour 2 000 $ — équilibre coût/couverture.
  • File Extensions : les fichiers .log, images (.png, .jpg), et binaires compilés n'ont généralement pas de PII. Les exclure réduit le coût sans perte de couverture.
  • Prefix/Tag Scoping : focaliser le scan sur les préfixes contenant des exports métier (data/, exports/) plutôt que les dossiers techniques (logs/, tmp/).

  • Step Functions : pour les actions destructives (suppression, quarantaine), un workflow avec étape d'approbation humaine évite les faux positifs coûteux.

6.5 — Lambda de Remédiation — Code Python

Python — Lambda handler pour remédiation Macie (Boto3)
import json
import boto3
import logging
from datetime import datetime

logger = logging.getLogger()
logger.setLevel(logging.INFO)

s3 = boto3.client('s3')
sns = boto3.client('sns')

SNS_TOPIC_ARN = 'arn:aws:sns:eu-west-1:123456789012:macie-remediation-alerts'
QUARANTINE_BUCKET = 'security-quarantine-bucket'

def lambda_handler(event, context):
    """Remédiation automatique des findings Macie High severity."""
    finding = event.get('detail', {})
    finding_type = finding.get('type', '')
    severity = finding.get('severity', {}).get('description', 'Unknown')
    bucket = finding.get('resourcesAffected', {}).get('s3Bucket', {}).get('name', '')
    object_key = finding.get('resourcesAffected', {}).get('s3Object', {}).get('key', '')

    logger.info(f"Finding: {finding_type} | Severity: {severity} | Bucket: {bucket}")

    actions_taken = []

    # 1. Si bucket public → bloquer l'accès public
    if 'Policy:IAMUser/S3BucketPublic' in finding_type:
        try:
            s3.put_public_access_block(
                Bucket=bucket,
                PublicAccessBlockConfiguration={
                    'BlockPublicAcls': True,
                    'IgnorePublicAcls': True,
                    'BlockPublicPolicy': True,
                    'RestrictPublicBuckets': True
                }
            )
            actions_taken.append(f"✅ Public access blocked on {bucket}")
        except Exception as e:
            actions_taken.append(f"❌ Failed to block public access: {str(e)}")

    # 2. Si données sensibles → tagger l'objet
    if 'SensitiveData' in finding_type and object_key:
        try:
            s3.put_object_tagging(
                Bucket=bucket, Key=object_key,
                Tagging={'TagSet': [
                    {'Key': 'Classification', 'Value': 'Sensitive'},
                    {'Key': 'MacieFinding', 'Value': finding_type.split('/')[-1]},
                    {'Key': 'DetectedAt', 'Value': datetime.utcnow().isoformat()}
                ]}
            )
            actions_taken.append(f"✅ Object tagged: s3://{bucket}/{object_key}")
        except Exception as e:
            actions_taken.append(f"❌ Failed to tag object: {str(e)}")

    # 3. Si credentials en clair → copier en quarantaine + notifier
    if 'Credentials' in finding_type and object_key:
        try:
            s3.copy_object(
                Bucket=QUARANTINE_BUCKET,
                Key=f"quarantine/{bucket}/{object_key}",
                CopySource={'Bucket': bucket, 'Key': object_key}
            )
            actions_taken.append(f"✅ Object copied to quarantine: {object_key}")
        except Exception as e:
            actions_taken.append(f"❌ Failed to quarantine: {str(e)}")

    # 4. Notification SNS
    message = {
        'finding_type': finding_type,
        'severity': severity,
        'bucket': bucket,
        'object_key': object_key,
        'actions_taken': actions_taken,
        'timestamp': datetime.utcnow().isoformat()
    }
    sns.publish(
        TopicArn=SNS_TOPIC_ARN,
        Subject=f"🔒 Macie Remediation [{severity}] - {finding_type}",
        Message=json.dumps(message, indent=2)
    )

    return {'statusCode': 200, 'actions': actions_taken}

Script All-in-One — Déploiement Macie Complet
Script Bash/AWS CLI exécutable pour appliquer toutes les best practices en une seule passe
⚠️ Prérequis : AWS CLI v2 configuré avec le profil audit-admin, permissions macie2:*, s3:*, kms:*, events:*, sns:*. 
#!/bin/bash
# ╔══════════════════════════════════════════════════════════════╗
# ║  Macie All-in-One — Déploiement complet best practices     ║
# ║  Profil : audit-admin | Région : eu-west-1                 ║
# ╚══════════════════════════════════════════════════════════════╝

set -euo pipefail

PROFILE="audit-admin"
REGION="eu-west-1"
ACCOUNT_ID=$(aws sts get-caller-identity --profile "$PROFILE" --query 'Account' --output text)
EXPORT_BUCKET="macie-results-${ACCOUNT_ID}-${REGION}"
KMS_ALIAS="alias/macie-export-key"
SNS_TOPIC="macie-high-findings"
ALERT_EMAIL="security-team@example.com"

echo "═══ [S1] Activation de Macie ═══"
aws macie2 enable-macie \
    --profile "$PROFILE" --region "$REGION" \
    --finding-publishing-frequency FIFTEEN_MINUTES \
    --status ENABLED 2>/dev/null || echo "Macie déjà activé"

echo "═══ [S1] Activation Automated Sensitive Data Discovery ═══"
aws macie2 update-automated-discovery-configuration \
    --profile "$PROFILE" --region "$REGION" \
    --status ENABLED \
    --auto-enable-organization-members ALL

echo "═══ [S2] Création bucket export (chiffré SSE-KMS) ═══"
aws s3api create-bucket \
    --profile "$PROFILE" --region "$REGION" \
    --bucket "$EXPORT_BUCKET" \
    --create-bucket-configuration LocationConstraint="$REGION" 2>/dev/null || true

aws s3api put-public-access-block \
    --profile "$PROFILE" --region "$REGION" \
    --bucket "$EXPORT_BUCKET" \
    --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

echo "═══ [S3] Configuration export des résultats ═══"
KMS_KEY_ARN=$(aws kms describe-key --profile "$PROFILE" --region "$REGION" \
    --key-id "$KMS_ALIAS" --query 'KeyMetadata.Arn' --output text 2>/dev/null || echo "NONE")

if [ "$KMS_KEY_ARN" != "NONE" ]; then
    aws macie2 put-classification-export-configuration \
        --profile "$PROFILE" --region "$REGION" \
        --configuration "{
            \"s3Destination\": {
                \"bucketName\": \"${EXPORT_BUCKET}\",
                \"keyPrefix\": \"discovery-results/\",
                \"kmsKeyArn\": \"${KMS_KEY_ARN}\"
            }
        }"
    echo "Export configuré vers s3://${EXPORT_BUCKET}/discovery-results/"
else
    echo "⚠ Clé KMS ${KMS_ALIAS} non trouvée — configurer manuellement"
fi

echo "═══ [S6] Création SNS Topic + alerte EventBridge ═══"
TOPIC_ARN=$(aws sns create-topic \
    --profile "$PROFILE" --region "$REGION" \
    --name "$SNS_TOPIC" --query 'TopicArn' --output text)

aws sns subscribe \
    --profile "$PROFILE" --region "$REGION" \
    --topic-arn "$TOPIC_ARN" \
    --protocol email --notification-endpoint "$ALERT_EMAIL"

aws events put-rule \
    --profile "$PROFILE" --region "$REGION" \
    --name "macie-high-findings-rule" \
    --event-pattern '{
        "source": ["aws.macie"],
        "detail-type": ["Macie Finding"],
        "detail": {
            "severity": { "description": ["High", "Critical"] }
        }
    }'

aws events put-targets \
    --profile "$PROFILE" --region "$REGION" \
    --rule "macie-high-findings-rule" \
    --targets "Id=sns-target,Arn=${TOPIC_ARN}"

echo "═══ [S9] Intégration Security Hub ═══"
aws securityhub enable-import-findings-for-product \
    --profile "$PROFILE" --region "$REGION" \
    --product-arn "arn:aws:securityhub:${REGION}::product/aws/macie" 2>/dev/null || \
    echo "Intégration Security Hub déjà active"

echo ""
echo "╔═══════════════════════════════════════════════════════════╗"
echo "║  ✅ Déploiement Macie terminé                            ║"
echo "║  - Macie activé + Automated Discovery ON                ║"
echo "║  - Export S3 : s3://${EXPORT_BUCKET}                      ║"
echo "║  - Alertes HIGH/CRITICAL → ${ALERT_EMAIL}                ║"
echo "║  - Security Hub intégré                                 ║"
echo "║                                                           ║"
echo "║  ⚠ Actions manuelles restantes :                         ║"
echo "║  - Confirmer l'abonnement email SNS                      ║"
echo "║  - Créer les Custom Data Identifiers (S7)               ║"
echo "║  - Configurer les comptes membres (S5)                   ║"
echo "╚═══════════════════════════════════════════════════════════╝"
✅
