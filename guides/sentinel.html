Argumentaire client — impact Business Premium sur Sentinel :
• Visibilité Defender XDR : Business Premium inclut Defender for Business (version simplifiée de MDE). Les tables DeviceEvents sont disponibles mais sans advanced hunting complet, sans custom detections, ni timeline complète. E5 Security donne MDE P2 + MDO P2 + MDCA + MDI.

Architecture & Workspace Design Sentinel Log Analytics
Conception du workspace Log Analytics, rétention, data residency et décisions architecturales fondatrices
Réf. Microsoft Learn — Design your workspace architecture
🚨
Scénario : Un RSSI déploie Sentinel dans un workspace existant utilisé par l'équipe IT pour du monitoring applicatif. Résultat : les analystes SOC voient des milliers d'entrées non-sécurité (perf, heartbeat, custom logs) — bruit qui noie les alertes. L'espace de rétention explose. Les coûts deviennent imprévisibles car les données ops inondent le commitment tier.
Analogie : Le workspace est le bâtiment de la tour de contrôle. Si les écrans de la tour de contrôle aérien affichent aussi le trafic routier et les horaires de trains, le contrôleur manquera l'avion en détresse. Un workspace dédié sécurité = une tour de contrôle focalisée.

ParamètreDéfaut⚡ Quick Win🎯 Best PracticeImpact

Pricing tier Pay-per-GB Pay-per-GB (si < 100 GB/jour) Commitment tier 100+ GB/jour si volume justifié (−50% dès 100 GB) Coût d'ingestion
💡
Pourquoi chaque paramètre compte :
  • Workspace dédié : un workspace partagé ops/sécu mélange les données, complique le RBAC, et gonfle artificiellement le volume facturé à Sentinel. Microsoft recommande un workspace séparé pour Sentinel (Best practices).
  • Région : les données de logs contiennent des données personnelles (UPN, IP). Le RGPD Art. 44-49 impose que ces données restent dans l'EEE sauf adéquation. Choisir West Europe ou France Central pour un tenant EU.
  • Rétention 90j : la rétention interactive gratuite est de 30 jours. Un attaquant APT a un dwell time moyen de 16 jours (Mandiant M-Trends 2024) mais peut aller à 6+ mois. 90 jours interactive = capacité de hunting rétroactif sur les 3 derniers mois. Au-delà, l'archive (à coût réduit) couvre les obligations légales (ANSSI : 1 an minimum pour les OIV/OSE, Code de commerce L123-22 : 10 ans pour les écritures comptables).
  • Single vs Multi : le multi-workspace est justifié pour l'isolation RBAC stricte (MSSP avec clients multiples), la data residency (EU + US), ou la séparation réglementaire. Sinon, un seul workspace simplifie les analytics rules cross-source.
⚠️
Piège classique : Activer Sentinel sur un workspace Log Analytics existant utilisé par Azure Monitor (VM perf, app diagnostics). Les données ops (Perf, Heartbeat, ContainerLog) s'ajoutent au volume facturé Sentinel. Créer un workspace dédié pour Sentinel et utiliser Lighthouse ou cross-workspace queries si nécessaire.

A.2 — Création du workspace & onboarding Sentinel

Az CLI — Créer le workspace et activer Sentinel
# Variables à adapter
$rgName      = "rg-sentinel-prod"
$location    = "francecentral"
$wsName      = "law-sentinel-prod"
$retention   = 90

# Créer le resource group
az group create --name $rgName --location $location

# Créer le workspace Log Analytics (rétention 90 jours)
az monitor log-analytics workspace create `
    --resource-group $rgName `
    --workspace-name $wsName `
    --location $location `
    --retention-in-days $retention `
    --sku PerGB2018

# Activer Sentinel (onboarding) sur le workspace
az sentinel onboarding-state create `
    --resource-group $rgName `
    --workspace-name $wsName `
    --name "default"

# Configurer l'archive sur les tables critiques (2 ans)
az monitor log-analytics workspace table update `
    --resource-group $rgName `
    --workspace-name $wsName `
    --name "SigninLogs" `
    --total-retention-in-days 730

az monitor log-analytics workspace table update `
    --resource-group $rgName `
    --workspace-name $wsName `
    --name "SecurityEvent" `
    --total-retention-in-days 730

Pourquoi chaque paramètre Data Connector compte :
  • Connecteur Entra ID (SigninLogs, AuditLogs) : sans ces logs, un brute-force sur un compte admin, une assignation de rôle Global Admin ou une modification de Conditional Access passent totalement inaperçus. C'est la source n°1 en termes de valeur détective.
  • Azure Activity : trace toutes les opérations de plan de contrôle Azure (création/suppression de ressources, changements RBAC, modifications NSG). Sans lui, un attaquant qui supprime vos backups ou ouvre un port RDP public agit en invisibilité totale.
  • Office 365 (OfficeActivity) : détecte l'exfiltration via mail forwarding rules, le téléchargement massif SharePoint, et les connexions Teams suspectes. Un employé qui copie 5 Go de fichiers clients la veille de sa démission est invisible sans ce connecteur.
  • Defender XDR (SecurityAlert) : les alertes/incidents Defender sont gratuites à l'ingestion. Désactiver ce connecteur = dupliquer le travail de triage entre deux consoles. Les raw tables (DeviceEvents, EmailEvents) sont payantes et volumineux — activer sélectivement.
  • Rétention 90j interactif / 365j archive : le délai moyen de détection d'une compromission est de 70 jours (IBM CODB 2024). Avec 30j de rétention par défaut, les preuves sont détruites avant découverte. 90j interactif couvre l'investigation, 365j archive couvre la conformité (RGPD art. 5, ANSSI).
  • DCR (Data Collection Rules) : sans filtrage à la source, vous payez l'ingestion de millions d'événements Windows inutiles (logon Machine Account, EventID 4624 type 3). Les DCR réduisent le volume de 30-50% sur SecurityEvent — économie directe et immédiate.
  • AMA vs MMA : le Log Analytics agent (MMA/OMS) est retiré depuis le 31/08/2024. Microsoft indique qu'après le 02/03/2026 l'upload cloud MMA peut s'arrêter à tout moment. AMA + DCR est le chemin supporté.

1.3 — Data Collection Rules (DCR) — Filtrage à la source

ℹ️
ROI immédiat : Les DCR filtrent les événements avant l'ingestion dans Log Analytics. Typiquement, filtrer les Event ID Windows inutiles (logon Machine Account, EventID 4624 type 3 en masse) réduit le volume de 30-50% sur SecurityEvent, soit une économie directe de coût.
az CLI — Créer une DCR avec filtrage Windows Security Events
# Créer une DCR qui filtre les Event ID à haute valeur uniquement
az monitor data-collection rule create \
  --resource-group $rgName \
  --name "dcr-windows-security-filtered" \
  --location "westeurope" \
  --data-flows '[{
    "streams": ["Microsoft-SecurityEvent"],
    "destinations": ["la-sentinel"],
    "transformKql": "source | where EventID in (4624, 4625, 4648, 4672, 4688, 4698, 4720, 4726, 4732, 4756, 4768, 4769, 4771, 4776) | where not(EventID == 4624 and LogonType == 3 and TargetUserName endswith \"$\")"
  }]' \
  --destinations '{
    "logAnalytics": [{
      "workspaceResourceId": "/subscriptions/'$subscriptionId'/resourceGroups/'$rgName'/providers/Microsoft.OperationalInsights/workspaces/'$workspaceName'",
      "name": "la-sentinel"
    }]
  }' \
  --data-sources '{
    "windowsEventLogs": [{
      "streams": ["Microsoft-SecurityEvent"],
      "xPathQueries": ["Security!*"],
      "name": "security-events"
    }]
  }'

let failThreshold = 10;
let timeWindow = 10m;

| where TimeGenerated > ago(1h)
| where ResultType != "0"  // échecs uniquement

| where LastFail - FirstFail <= timeWindow

| join kind=inner (
    SigninLogs
    | where TimeGenerated > ago(1h)
    | where ResultType == "0"
    | project SuccessTime = TimeGenerated, IPAddress, SuccessUser = UserPrincipalName

| where SuccessTime > LastFail
| project IPAddress, Location, FailCount, TargetAccounts, CompromisedUser = SuccessUser, SuccessTime

Python — Exemple minimal MSTICPy (Sentinel Notebook)

import msticpy as mp

# Connexion au workspace Sentinel

# Hunting : failed sign-ins des 7 derniers jours
df_signin = qry_prov.exec_query("""
SigninLogs
| where TimeGenerated > ago(7d)
| where ResultType != "0"
| summarize FailCount=count() by UserPrincipalName, IPAddress
| sort by FailCount desc
| take 20
""")

  • Sentinel Reader : SOC L1 voit sans modifier. Un L1 compromis ne peut pas désactiver les rules de détection.
  • Sentinel Responder : SOC L2 gère les incidents (assign, close, comment) sans toucher à la configuration. Séparation opérations / ingénierie.
  • Sentinel Contributor : réservé SOC L3 / ingénieurs. Crée/modifie rules, workbooks, watchlists. Compte critique à protéger par MFA + PIM.
  • Log Analytics Reader : complément indispensable pour que les rôles Sentinel puissent exécuter les requêtes KQL. Souvent oublié → l'analyste a le rôle Sentinel mais ne voit aucune donnée.
  • Resource-context RBAC : mode avancé où l'accès aux logs dépend du rôle sur la ressource Azure source. Idéal pour les environnements multi-équipes.
  • Table-level RBAC : restreindre l'accès à des tables spécifiques (ex : SigninLogs = PII RGPD). Couche supplémentaire de protection des données personnelles.

10.3 — RBAC avancé : resource-context & table-level

ℹ️
Workspace-context (défaut) vs Resource-context :
  • Workspace-context : l'utilisateur a un rôle sur le workspace → il voit toutes les données de toutes les tables. C'est le mode par défaut.
  • Resource-context : l'utilisateur voit uniquement les logs des ressources Azure sur lesquelles il a un rôle RBAC. Activé via Settings → Access control mode = Resource-specific. Idéal pour donner aux propriétaires d'app l'accès à leurs logs sans exposer les logs des autres équipes.
ℹ️
Table-level RBAC : Restreindre l'accès à des tables spécifiques dans le workspace. Ex : permettre au SOC L1 de lire SecurityAlert et SecurityIncident mais pas SigninLogs (données personnelles RGPD).
Configuration : az role assignment create --role "Log Analytics Reader" --scope "/subscriptions/$subscriptionId/resourceGroups/$rgName/providers/Microsoft.OperationalInsights/workspaces/$workspaceName/tables/SecurityAlert"

Gain de temps moyen estimé : 40% sur le triage (source : Microsoft Unified SOC TEI study).
  • Incidents bi-directionnels : un incident créé dans Defender XDR (ex: malware sur endpoint) remonte automatiquement dans Sentinel avec toute la corrélation.   • Advanced Hunting cross-product : dans le portail Defender, une requête KQL peut interroger simultanément EmailEvents (Defender for Office), DeviceProcessEvents (Defender for Endpoint), SigninLogs (Sentinel/Entra ID) et AzureActivity (Sentinel). Impossible dans le portail Azure seul.
  • AIR + SOAR : AIR est l'automation native de Defender XDR (investigation automatique des alertes endpoint/email). SOAR (Logic Apps) est l'automation de Sentinel. En Unified SOC, les deux coexistent : AIR pour les alertes Defender (gratuit avec la licence), SOAR pour les sources custom.
⚠️
Piège classique : Activer le connecteur Microsoft Defender XDR dans Sentinel sans comprendre la duplication d'incidents. Par défaut, le connecteur synchronise les incidents Defender vers Sentinel. Si un analyste clôture l'incident dans Defender, il se clôture dans Sentinel (et inversement). Mais si l'équipe utilise les deux portails en parallèle sans process clair, les incidents sont traités deux fois ou pas du tout.
Règle d'or : Choisir un portail principal pour le triage (recommandé : Defender) et désactiver la création d'incidents dans l'autre portail. Ou : migrer intégralement vers le Unified SOC.

KQL — Corrélation email → endpoint → sign-in (Advanced Hunting)
// Chasse cross-product : email de phishing → exécution sur endpoint → sign-in anormal
// Requête uniquement disponible dans le portail Defender (Advanced Hunting)
let phishingEmails = EmailEvents
    | where TimeGenerated > ago(24h)
    | where ThreatTypes has "Phish"
    | project RecipientEmailAddress, NetworkMessageId, Subject, SenderFromAddress;
let recipientDevices = phishingEmails
    | join kind=inner (
        DeviceProcessEvents
        | where TimeGenerated > ago(24h)
        | where FileName in~ ("powershell.exe", "cmd.exe", "wscript.exe", "mshta.exe")
    ) on $left.RecipientEmailAddress == $right.AccountUpn
    | project AccountUpn, DeviceName, FileName, ProcessCommandLine, NetworkMessageId;
recipientDevices
| join kind=leftouter (
    SigninLogs
    | where TimeGenerated > ago(24h)
    | where ResultType == "0"
    | project UserPrincipalName, IPAddress, Location, AppDisplayName
) on $left.AccountUpn == $right.UserPrincipalName
| project AccountUpn, DeviceName, SuspiciousProcess = FileName, ProcessCommandLine, SignInIP = IPAddress, SignInLocation = Location
