Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'administration Exchange Online
📋
Ce guide couvre l'administration Exchange Online : mail flow, transport rules, connecteurs, gestion des boîtes aux lettres, groupes, migrations, accès client et monitoring. Il ne couvre pas la sécurité email (EOP, MDO, DMARC) → voir email_security_guide, ni la conformité (retention, DLP, eDiscovery) → voir purview_compliance_guide.
⚠️
Choix structurant — PowerShell first : Exchange Online s'administre essentiellement en PowerShell via le module ExchangeOnlineManagement V3 (REST-based). Chaque opération est documentée en PowerShell en priorité + chemin EAC (portail) en secondaire.
Alternative écartée : les anciens scénarios RPS / Basic Auth ne sont plus supportés ; les connexions REST du module EXO V3 les remplacent depuis octobre 2023. Ce guide utilise exclusivement les cmdlets REST V3.

Mail Flow & Transport Rules EXO P1
Règles de transport, priorités, disclaimers, redirect/BCC, modération & message trace
Réf. CIS Microsoft 365 Foundations Benchmark v4.0 §2.1
🚨
Scénario concret : Sans transport rules, un employé peut configurer un auto-forward vers une adresse Gmail personnelle et exfiltrer silencieusement tous les emails de l'entreprise — pendant des mois, sans aucune alerte. En 2024, 28 % des fuites de données email impliquérent un forwarding non détecté (Verizon DBIR 2024).
Analogie : les transport rules sont le « tri postal automatique » de votre messagerie — elles aiguillent, tamponnent, bloquent ou copient chaque courrier selon des critères précis, exactement comme un centre de tri postal qui oriente les lettres avant distribution.

💡
Pourquoi chaque paramètre compte :
  • Block auto-forward externe : sans cette règle, un compte compromis ou un employé malveillant peut rediriger tous ses emails vers l'extérieur. CIS Benchmark M365 v4.0 recommande explicitement de bloquer le forwarding externe (contrôle 2.1.1). L'action RejectMessage est préférée à Reject car elle envoie un NDR clair à l'expéditeur.
  • Disclaimer légal : obligation légale dans de nombreuses juridictions (RGPD Art. 13 — information de l'interlocuteur, Code de commerce Art. L123-22 — mentions légales). L'absence de disclaimer sur les emails sortants peut entraîner des sanctions réglementaires.
  • Tag external emails : aide les utilisateurs à identifier immédiatement un email externe. Réduit de 30 % le taux de clic sur les phishing internes (source : Microsoft Security Blog, 2023). Le tag natif EXO (activé depuis fin 2023) couvre Outlook, mais une transport rule en complément couvre les clients tiers.
  • SenderAddressLocation = HeaderOrEnvelope : la valeur par défaut Header ne vérifie que le champ From:. Un attaquant peut usurper l'envelope sender (SMTP MAIL FROM) pour contourner les règles. HeaderOrEnvelope couvre les deux vecteurs.
  • Priorité documentée : sans convention, les règles de sécurité (blocage forwarding) risquent d'être évaluées après des règles business avec StopRuleProcessing, créant un bypass involontaire.
⚠️
Piège classique : Le tag natif « External » d'Exchange Online (activé via Set-ExternalInOutlook) ne fonctionne que dans Outlook desktop/mobile. Les clients tiers (Thunderbird, Apple Mail) ne le voient pas. Pour une couverture complète, ajouter une transport rule PrependSubject en complément.

Autre piège : si une règle contient StopRuleProcessing $true avec une priorité haute, les règles de sécurité (blocage forwarding) placées plus bas ne seront jamais évaluées. Règle d'or : les règles de sécurité doivent toujours avoir les priorités les plus hautes (0-9).

1.2 — Configuration

PowerShell — Transport Rules essentielles (ExchangeOnlineManagement V3)
# Connexion
Connect-ExchangeOnline -UserPrincipalName $adminEmail

# --- Règle 1 : Bloquer le forwarding automatique vers l'extérieur ---
New-TransportRule -Name "SEC-001 Block Auto-Forward External" `
    -Priority 0 `
    -MessageTypeMatches AutoForward `
    -SentToScope NotInOrganization `
    -RejectMessageReasonText "Le transfert automatique vers des destinataires externes est bloqué par la politique de sécurité." `
    -GenerateIncidentReport $adminEmail `
    -IncidentReportContent Sender,Recipients,Subject,MessageId `
    -Comments "CIS M365 v4.0 - 2.1.1 - Block external auto-forwarding"

# --- Règle 2 : Disclaimer légal sur les emails sortants ---
$disclaimerHtml = "<p style='font-size:10px;color:#666;'>Ce message et ses pièces jointes sont confidentiels. " +
    "Si vous n'êtes pas le destinataire, merci de le supprimer. " +
    "Mentions légales : https://$domain/legal</p>"

New-TransportRule -Name "BIZ-020 Disclaimer Légal Sortant" `
    -Priority 20 `
    -SentToScope NotInOrganization `
    -ApplyHtmlDisclaimerText $disclaimerHtml `
    -ApplyHtmlDisclaimerLocation Append `
    -ApplyHtmlDisclaimerFallbackAction Wrap `
    -Comments "RGPD Art.13 - Mentions légales obligatoires"

# --- Règle 3 : Tag [EXTERNE] dans le sujet (complément au tag natif) ---
New-TransportRule -Name "SEC-002 Tag External Emails Subject" `
    -Priority 1 `
    -FromScope NotInOrganization `
    -PrependSubject "[EXTERNE] " `
    -Comments "Complément tag natif EXO pour clients non-Outlook"

# --- Activer le tag natif « External » dans Outlook ---
Set-ExternalInOutlook -Enabled $true

# --- Forcer SenderAddressLocation sur HeaderOrEnvelope ---
Get-TransportRule | ForEach-Object {
    Set-TransportRule -Identity $_.Identity -SenderAddressLocation HeaderOrEnvelope
}
ℹ️
Chemin EAC (portail) : Exchange Admin Center → Mail flow → Rules → + Add a rule. Chaque condition/action du PowerShell correspond à un menu déroulant dans le wizard de création de règle. La priorité se gère via les flèches « Move up / Move down » dans la liste.

Connecteurs (Inbound / Outbound) EXO P1 Hybride
Connecteurs Partner (TLS org-to-org), Application (relai SMTP), Enhanced Filtering, connecteurs hybrides
🚨
Scénario concret : Une entreprise utilise un ERP on-premises qui envoie des factures par email via le port 25 sans authentification. Sans connecteur sécurisé, ces emails sont bloqués ou atterrissent en spam chez les destinataires — voire pire, un attaquant exploite le relai SMTP ouvert pour envoyer des campagnes de phishing au nom de l'entreprise.
Analogie : un connecteur Exchange Online est un « tuyau dédié et sécurisé » entre deux systèmes de messagerie. Le « Partner connector » est un tunnel chiffré TLS entre deux organisations, le « connector Application » est un accès contrôlé pour les machines et services.

💡
Pourquoi chaque paramètre compte :
  • Connecteur Partner TLS : sans TLS forcé, les emails entre organisations partenaires transitent en clair. Un attaquant en position MitM peut lire ou modifier les messages. Le mode CertificateValidation vérifie l'identité du serveur distant via son certificat — le nécessaire minimum pour des échanges contractuels sensibles.
  • Connecteur Application (relai SMTP) : les ERP, scanners, imprimantes doivent envoyer des emails. L'authentification SMTP basique (SmtpClientAuthentication) est dépréciée par Microsoft depuis 2023 et vulnérable aux attaques par brute-force. Un connecteur par IP ou certificat client est la seule méthode supportée à long terme.
  • Enhanced Filtering : si le MX pointe vers une passerelle tierce (Proofpoint, Mimecast, Barracuda), EXO voit l'IP de la passerelle, pas l'IP de l'expéditeur réel. SPF, DKIM et DMARC échouent. EFSkipLastIP ou EFSkipIPs permet à EOP de remonter à l'IP d'origine — essentiel pour la détection anti-spam et anti-phishing.
  • RestrictDomainsToCertificate : sans ce paramètre, un attaquant peut spoofer le domaine partenaire et l'email sera accepté tant que le TLS est établi avec n'importe quel certificat valide. Cette option lie le domaine au certificat spécifique — une couche d'authentification supplémentaire.
⚠️
Piège classique : Ne jamais modifier ou supprimer les connecteurs créés par le Hybrid Configuration Wizard (noms contenant « Inbound from » ou « Outbound to » + votre org). Toute modification manuelle casse la coexistence hybride et peut provoquer des boucles de routage.

Autre piège : un InboundConnector avec -SenderDomains * accepte tous les domaines comme partenaires de confiance, contournant les filtres anti-spam. Toujours restreindre à des domaines spécifiques.

2.2 — Configuration

PowerShell — Connecteurs (ExchangeOnlineManagement V3)
# --- Partner Inbound : TLS forcé avec validation certificat ---
New-InboundConnector -Name "Partner-Inbound-ContosoPharma" `
    -ConnectorType Partner `
    -SenderDomains "contosopharma.com" `
    -RequireTls $true `
    -TlsSenderCertificateName "mail.contosopharma.com" `
    -RestrictDomainsToCertificate $true `
    -Enabled $true

# --- Partner Outbound : TLS + validation domaine ---
New-OutboundConnector -Name "Partner-Outbound-ContosoPharma" `
    -ConnectorType Partner `
    -RecipientDomains "contosopharma.com" `
    -TlsSettings CertificateValidation `
    -TlsDomain "mail.contosopharma.com" `
    -Enabled $true

# --- Connecteur Application pour ERP (relai SMTP par IP) ---
New-InboundConnector -Name "App-ERP-Relay" `
    -ConnectorType OnPremises `
    -SenderIPAddresses "203.0.113.10" `
    -RequireTls $true `
    -RestrictDomainsToIPAddresses $true `
    -SenderDomains "votre-domaine.com" `
    -Enabled $true

# --- Enhanced Filtering (si passerelle tierce devant EXO) ---
Set-InboundConnector -Identity "Via-Proofpoint" `
    -EFSkipLastIP $true `
    -EFSkipIPs "198.51.100.1","198.51.100.2"
ℹ️
Chemin EAC (portail) : Exchange Admin Center → Mail flow → Connectors → + Add a connector. Choisir « Connection from: Partner organization » ou « Your organization's email server » selon le scénario. Le Enhanced Filtering se configure dans Security & Compliance Center → Anti-spam → Enhanced Filtering.

Accepted Domains & Routing EXO P1
Authoritative vs Internal Relay vs External Relay, MX records, remote domains, shared address space
🚨
Scénario concret : Après une acquisition, l'entreprise ajoute le domaine de la filiale comme « Internal Relay » au lieu de « Authoritative ». Résultat : EXO accepte tous les emails pour ce domaine puis les reroute vers Internet si la mailbox n'existe pas — créant un open relay involontaire exploité par les spammeurs. De plus, le remote domain « Default (*) » laissé en configuration par défaut autorise l'auto-forward et les OOF vers l'extérieur.
Analogie : les accepted domains sont votre « panneau d'adresse » — la liste des noms de rue que votre bureau de poste reconnât et accepte de distribuer. Un domaine Authoritative = « je suis la dernière destination » ; Internal Relay = « je fais suivre si je ne connais pas ».

💡
Pourquoi chaque paramètre compte :
  • AutoForwardEnabled = $false : CIS Benchmark M365 v4.0 §2.1.4 recommande de désactiver le forwarding automatique via remote domain en complément des transport rules. La transport rule bloque au niveau du mail flow ; le remote domain bloque au niveau du protocole client (OWA, Outlook rules).
  • AllowedOOFType : les réponses automatiques (OOF) peuvent révéler l'absence d'un collaborateur, sa date de retour, et un contact alternatif — informations exploitables en ingénierie sociale (attaques BEC/CEO fraud).
  • NDREnabled = $false : les Non-Delivery Reports vers l'extérieur permettent à un attaquant de deviner les adresses email valides (« directory harvesting attack »). En désactivant les NDR externes, EXO « silently drop » les emails vers des adresses inexistantes.
  • InternalRelay : un domaine InternalRelay accepte les emails ET les fait suivre vers un connecteur si la mailbox n'existe pas localement. Si le connecteur est mal configuré ou supprimé, EXO reroute vers le MX public → risque de boucle ou d'open relay.
⚠️
Piège classique : Le RemoteDomain Default (*) s'applique à tous les domaines non explicitement configurés. Si vous créez un remote domain spécifique pour un partenaire avec AutoForwardEnabled = $true, cela N'affecte PAS le défaut. Mais l'inverse est vrai : si le défaut est à $true, tous les domaines non listés autorisent le forwarding.

Autre piège : supprimer un accepted domain sans avoir migré les mailboxes qui l'utilisent comme adresse primaire rend ces mailboxes inaccessibles.

3.2 — Configuration

PowerShell — Accepted Domains & Remote Domains
# --- Auditer les accepted domains ---
Get-AcceptedDomain | Format-Table DomainName, DomainType, Default -AutoSize

# --- Sécuriser le Remote Domain par défaut (*) ---
Set-RemoteDomain -Identity Default `
    -AutoForwardEnabled $false `
    -AllowedOOFType ExternalLegacy `
    -NDREnabled $false `
    -DeliveryReportEnabled $false `
    -TNEFEnabled $false

# --- Créer un remote domain spécifique pour un partenaire ---
New-RemoteDomain -Name "Partner-ContosoPharma" `
    -DomainName "contosopharma.com"

Set-RemoteDomain -Identity "Partner-ContosoPharma" `
    -AutoForwardEnabled $false `
    -AllowedOOFType InternalLegacy `
    -NDREnabled $true `
    -TNEFEnabled $true

# --- Convertir un InternalRelay en Authoritative (post-migration) ---
Set-AcceptedDomain -Identity "filiale.com" `
    -DomainType Authoritative
ℹ️
Chemin EAC (portail) : Exchange Admin Center → Mail flow → Accepted domains. Pour les remote domains : Mail flow → Remote domains. Vérifier que chaque domaine a un enregistrement MX correct dans le DNS public.

8
Address Book & GAL EXO P1
Global Address List, Offline Address Book, Address Book Policies (ségrégation multi-entité), Hierarchical AB
🚨
Scénario concret : Un groupe avec 3 filiales (cabinet d'avocats, cabinet comptable, société immobilière) partage un tenant M365. Sans Address Book Policy, les clients du cabinet d'avocats apparaissent comme contacts dans la GAL consultée par le cabinet comptable — violation du secret professionnel et du cloisonnement réglementaire.
Analogie : la GAL est l'« annuaire téléphonique interne » de l'organisation. Les Address Book Policies sont des cloisons permettant de créer des annuaires séparés par filiale ou département, comme si chaque entité avait son propre annuaire.

Pourquoi chaque paramètre compte :
  • Address Book Policy : dans un tenant multi-entité, la GAL par défaut expose TOUS les destinataires à TOUS les utilisateurs. Une ABP crée une « virtual organization » où chaque utilisateur ne voit que les membres de sa filiale. Essentiel pour la conformité réglementaire (secret professionnel, RGPD Art. 25 — protection par défaut).
  • OAB custom : Outlook desktop télécharge l'OAB pour la résolution de noms hors-ligne. Si l'OAB contient toute la GAL alors que l'ABP ne montre qu'une filiale, l'utilisateur peut chercher des adresses « cachées » dans l'OAB local — contournant la ségrégation.
  • Masquer les comptes de service : les comptes techniques (scanners, applications) polluent l'auto-complétion et sont des cibles de phishing (les noms génériques comme « noreply@ » sont facilement usurpés).
⚠️
Piège classique : Une ABP ne bloque PAS l'envoi d'email entre entités. Un utilisateur de la filiale A peut toujours envoyer à un utilisateur de la filiale B s'il connaît l'adresse. L'ABP ne fait que masquer la découverte, pas bloquer la communication. Pour un vrai cloisonnement, combiner avec des transport rules (S1) ou des Information Barriers (Purview Compliance Guide).

Autre piège : activer l'ABP après que les utilisateurs aient déjà des contacts inter-filiales dans leur cache Outlook. L'OAB ne se met à jour que toutes les 24h. Planifier l'activation un vendredi soir.

Journaling & Compliance EXO P1 EXO P2
Journal rules, Litigation Hold, Mailbox Audit Logging — renvoi Purview pour retention/eDiscovery
🚨
Scénario concret : Lors d'un litige avec un ancien employé, le service juridique demande tous les emails échangés sur 2 ans. Sans Litigation Hold ni journal rule, les emails supprimés sont purgés après 14 jours (Recoverable Items default). L'entreprise ne peut pas fournir les preuves — conséquence : présomption défavorable devant le tribunal.
Analogie : le journaling est un « coffre-fort notarié » — une copie conforme de chaque email est archivée dans un endroit inaltérable, comme un acte notarié est conservé chez le notaire indépendamment des parties.
➡️ Note : la retention avancée et l'eDiscovery sont détaillés dans le Guide Purview Compliance. Cette section couvre les fonctionnalités Exchange-natives.

  • Litigation Hold : obligation légale dans les secteurs réglementés (finance : MiFID II 5 ans, santé : 20 ans, commerce : 10 ans). Le hold préserve les éléments supprimés et les versions originales des éléments modifiés dans le dossier Recoverable Items.

⚠️
Piège classique : Le Litigation Hold ne supprime rien — il accumule des données dans le Recoverable Items (quota : 100 Go en E3/E5). Sur des mailboxes très actives, ce quota peut être atteint, provoquant des erreurs de réplication. Surveiller avec Get-MailboxStatistics → TotalDeletedItemSize.

Autre piège : le journal rule scope « Global » capture TOUS les emails (internes + externes). Le volume peut être énorme. Estimer le volume avant activation. La mailbox journal ne doit JAMAIS être une mailbox ordinaire — utiliser une mailbox dédiée ou un service tiers de journaling.

🚨

Script All-in-One — Hardening Exchange Online
Script exécutable pour appliquer toutes les best practices d'administration EXO
⚠️
Prérequis : Module ExchangeOnlineManagement V3 installé. Connexion établie via Connect-ExchangeOnline avec un compte Global Admin ou Exchange Admin. Tester d'abord en mode audit ($WhatIf = $true) avant d'appliquer en production.
PowerShell — Script All-in-One EXO Hardening
#Requires -Modules ExchangeOnlineManagement
# ============================================================
# Exchange Online Hardening — All-in-One Audit & Remediation
# Version 1.1 | Mai 2026
# ============================================================

param(
    [switch]$AuditOnly,        # Mode audit uniquement (pas de modification)
    [switch]$SkipConnection    # Passer si déjà connecté
)

# --- Connexion ---
if (-not $SkipConnection) {
    Connect-ExchangeOnline -ShowBanner:$false
}

$report = @()

# ===================== S1 : MAIL FLOW =====================
Write-Host "[S1] Vérification Mail Flow & Transport Rules..." -ForegroundColor Cyan

# Bloquer auto-forward externe
$fwdRule = Get-TransportRule | Where-Object { $_.Name -like "*Block*Forward*External*" }
if (-not $fwdRule) {
    $report += "[S1] MANQUANT : Règle anti-forward externe"
    if (-not $AuditOnly) {
        New-TransportRule -Name "Block Auto-Forward External" `
            -FromScope InOrganization -SentToScope NotInOrganization `
            -MessageTypeMatches AutoForward -RejectMessageReasonText "Auto-forward externe interdit par politique" `
            -Priority 0
    }
}

# Tag External (Outlook)
$extTag = (Get-ExternalInOutlook).Enabled
if (-not $extTag) {
    $report += "[S1] MANQUANT : Tag [External] sur emails externes"
    if (-not $AuditOnly) { Set-ExternalInOutlook -Enabled $true }
}

# ===================== S3 : REMOTE DOMAINS =====================
Write-Host "[S3] Vérification Remote Domains..." -ForegroundColor Cyan

$rd = Get-RemoteDomain Default
if ($rd.AutoForwardEnabled) {
    $report += "[S3] RISQUE : Remote Domain Default AutoForwardEnabled = True"
    if (-not $AuditOnly) { Set-RemoteDomain Default -AutoForwardEnabled $false }
}
if ($rd.NDREnabled) {
    $report += "[S3] RISQUE : Remote Domain Default NDREnabled = True (info leak)"
    if (-not $AuditOnly) { Set-RemoteDomain Default -NDREnabled $false }
}

# ===================== S4 : RETENTION =====================
Write-Host "[S4] Vérification RetainDeletedItemsFor..." -ForegroundColor Cyan

$lowRetention = Get-Mailbox -ResultSize Unlimited |
    Where-Object { $_.RetainDeletedItemsFor.Days -lt 30 }
if ($lowRetention) {
    $report += "[S4] $($lowRetention.Count) mailboxes avec retention < 30 jours"
    if (-not $AuditOnly) {
        $lowRetention | Set-Mailbox -RetainDeletedItemsFor 30
    }
}

# ===================== S7 : PROTOCOLES LEGACY =====================
Write-Host "[S7] Vérification protocoles legacy..." -ForegroundColor Cyan

$legacyPop = Get-CASMailbox -ResultSize Unlimited |
    Where-Object { $_.PopEnabled -or $_.ImapEnabled }
if ($legacyPop) {
    $report += "[S7] $($legacyPop.Count) mailboxes avec POP3/IMAP4 actif"
    if (-not $AuditOnly) {
        $legacyPop | Set-CASMailbox -PopEnabled $false -ImapEnabled $false
    }
}

# ===================== S9 : LITIGATION HOLD =====================
Write-Host "[S9] Vérification Litigation Hold..." -ForegroundColor Cyan

$noHold = Get-Mailbox -ResultSize Unlimited -Filter "LitigationHoldEnabled -eq 'False' -and RecipientTypeDetails -eq 'UserMailbox'"
if ($noHold) {
    $report += "[S9] $($noHold.Count) mailboxes sans Litigation Hold"
}

# ===================== RAPPORT =====================
Write-Host "`n========== RAPPORT ==========" -ForegroundColor Yellow
if ($report.Count -eq 0) {
    Write-Host "Toutes les vérifications sont conformes." -ForegroundColor Green
} else {
    $report | ForEach-Object { Write-Host $_ -ForegroundColor Red }
}

# Export CSV
$report | Out-File "EXO-Hardening-Report-$(Get-Date -f yyyyMMdd).txt" -Encoding UTF8
Write-Host "`nRapport exporté : EXO-Hardening-Report-$(Get-Date -f yyyyMMdd).txt"
