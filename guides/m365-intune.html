S1
Enrollment & Autopilot Intune P1
Enrollment des devices, Windows Autopilot, restrictions et ESP — Critique — 15 pts
🚨
Scénario d'attaque — Sans restrictions d'enrollment, un attaquant possédant des credentials volés peut enroller un device personnel dans le tenant, obtenir un token PRT (Primary Refresh Token) et accéder aux ressources comme s'il était sur un poste corporate. C'est l'équivalent de laisser quelqu'un entrer dans le bâtiment avec un badge cloned : il a accès à tout.
🏠
Analogie — L'enrollment Autopilot, c'est comme l'accueil sécurisé d'un immeuble : le gardien (Autopilot) vérifie l'identité du visiteur (user), vérifie que le badge (device) est dans la liste autorisée (hardware hash), puis attribue un étage et un bureau (profil de configuration). L'ESP (Enrollment Status Page) est le sas de sécurité — le visiteur ne peut pas avancer tant que toutes les vérifications ne sont pas terminées.
💡
Pourquoi chaque paramètre compte — Les restrictions d'enrollment et la configuration Autopilot contrôlent quels appareils accèdent à votre tenant et dans quelles conditions. Un device limit trop permissif ou l'absence d'ESP permettent à des appareils non sécurisés d'obtenir un PRT et d'accéder aux ressources M365. Chaque paramètre non verrouillé est un point d'entrée potentiel.

🤔 Pourquoi c'est critique

1
PRT abuse — Un device enrollé reçoit un Primary Refresh Token (PRT) qui agit comme un SSO silencieux vers toutes les ressources M365. Sans restriction d'enrollment, un attaquant peut générer son propre PRT.
2
Shadow IT devices — Sans limite de devices par utilisateur et sans blocage des devices personnels, un collaborateur peut enroller 15 appareils y compris des machines partagées non sécurisées.
3
ESP = zéro-gap — Sans ESP, l'utilisateur accède au desktop avant que BitLocker, l'antivirus et les policies de sécurité ne soient appliqués. C'est une fenêtre de vulnérabilité de 10-30 minutes.
⚠️
Piège Autopilot — Le profil Autopilot Self-Deploying ne supporte pas l'affinité utilisateur (user affinity). Si vous l'appliquez à des postes utilisateurs standards, personne ne sera associé au device dans Intune → les policies ciblant des groupes utilisateurs ne s'appliqueront pas. Réservez Self-Deploying aux kiosks et aux salles de réunion.

⚙️ Configuration — Enrollment Restriction via Graph

PowerShell — Microsoft.Graph
# Connexion avec les scopes nécessaires
Connect-MgGraph -Scopes "DeviceManagementServiceConfig.ReadWrite.All"

# Récupérer la restriction d'enrollment par défaut (platform)
$restrictions = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceEnrollmentConfigurations"

# Créer une restriction bloquant les devices personnels (Windows)
$body = @{
    "@odata.type"  = "#microsoft.graph.deviceEnrollmentPlatformRestrictionConfiguration"
    displayName    = "Block Personal Windows Devices"
    platformType   = "windows"
    platformRestriction = @{
        personalDeviceEnrollmentBlocked = $true
        osMinimumVersion                = "10.0.19045"  # Windows 10 22H2 minimum
    }
} | ConvertTo-Json -Depth 5

Invoke-MgGraphRequest -Method POST `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceEnrollmentConfigurations" `
    -Body $body `
    -ContentType "application/json"

# Configurer la limite de devices par utilisateur à 5
$limitBody = @{
    "@odata.type" = "#microsoft.graph.deviceEnrollmentLimitConfiguration"
    displayName = "Limit 5 Devices Per User"
    limit       = 5
} | ConvertTo-Json

Invoke-MgGraphRequest -Method POST `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceEnrollmentConfigurations" `
    -Body $limitBody `
    -ContentType "application/json"

S2
Device Compliance Policies Intune P1
Policies de conformité par plateforme, actions de non-conformité et intégration CA — Critique — 15 pts
🚨
Scénario d'attaque — Un laptop dont le disque n'est pas chiffré (BitLocker off) et dont l'antivirus n'est pas à jour est volé dans un train. Sans compliance policy associée à un Conditional Access « Require compliant device », ce laptop reste marqué conforme et le voleur accède à tous les services M365 via le PRT caché. C'est comme verrouiller la porte d'entrée mais laisser la fenêtre grande ouverte.
🏠
Analogie — La compliance policy est le contrôle technique de votre voiture : sans plaquettes de frein (encryption), sans phares (antivirus) ou avec un moteur polluant (OS obsolète), le véhicule ne peut pas rouler. Le Conditional Access est la barrière de péage : pas de vignette technique (compliant device) = pas d'accès à l'autoroute (M365).
💡
Pourquoi chaque paramètre compte — Les policies de conformité définissent le seuil minimum de sécurité pour chaque device. Sans exigence de chiffrement, d'OS à jour ou d'antivirus, un poste compromis conserve un accès complet aux données d'entreprise. Couplé au Conditional Access, chaque paramètre de compliance agit comme un contrôle d'accès continu.

🤔 Pourquoi c'est critique

1
Zéro compliance = zéro barrière — Par défaut, tous les devices enrollés sont marqués conformes s'il n'y a pas de policy. Conditional Access ne peut pas bloquer ce qu'Intune ne mesure pas.
2
Signal MDE intégré — Le champ « Machine Risk Level » dans la compliance policy est alimenté en temps réel par MDE. Un device compromis est automatiquement marqué non-conforme → accès coupé via CA en quelques minutes.
3
Actions échelonnées — Les « Actions for noncompliance » permettent d'escalader : notification (J+0) → marquage non-conforme (J+1) → retrait des ressources (J+7) → wipe (J+30 si nécessaire). Cela évite les blocages brutaux.
⚠️
Piège « Mark devices with no compliance policy as » — Dans Tenant administration > Compliance policy settings, le paramètre par défaut est « Compliant ». Cela signifie que tout device SANS policy est automatiquement considéré conforme par CA. Changez-le immédiatement à « Not compliant » pour que le doute profite à la sécurité.

# Créer une compliance policy Windows
Connect-MgGraph -Scopes "DeviceManagementConfiguration.ReadWrite.All"

$complianceBody = @{
    "@odata.type" = "#microsoft.graph.windows10CompliancePolicy"
    displayName = "WIN-Compliance-Baseline"
    description = "Policy de conformité Windows - CIS aligned"
    bitLockerEnabled              = $true
    secureBootEnabled             = $true
    codeIntegrityEnabled          = $true
    osMinimumVersion              = "10.0.19045"
    firewallEnabled               = $true
    antivirusRequired             = $true
    antiSpywareRequired           = $true
    defenderEnabled               = $true
    passwordRequired              = $true
    passwordMinimumLength         = 6
    passwordRequiredType          = "alphanumeric"
    deviceThreatProtectionEnabled = $true
    deviceThreatProtectionRequiredSecurityLevel = "medium"
    scheduledActionsForRule = @(
        @{
            ruleName = "PasswordRequired"
            scheduledActionConfigurations = @(
                @{ actionType = "notification";  gracePeriodHours = 0 }
                @{ actionType = "block";         gracePeriodHours = 24 }
                @{ actionType = "retire";        gracePeriodHours = 720 }
            )
        }
    )
} | ConvertTo-Json -Depth 10

    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceCompliancePolicies" `

Configuration Profiles Intune P1
Settings Catalog, templates, ADMX — configurer les devices à l'échelle
🚨
Scénario de risque : un user désactive le pare-feu Windows pour « résoudre un problème réseau ». Sans configuration profile forçant le firewall, Intune ne le reconfigure pas automatiquement. Analogie : les configuration profiles sont le règlement intérieur — des règles permanentes appliquées à chaque device sans intervention manuelle.
ℹ️
Intune propose 3 mécanismes : Settings Catalog (recommandé — granulaire, typé, recherche native), Templates (VPN, Wi-Fi, SCEP, PKCS, Device Restrictions), Administrative Templates ADMX (migration GPO, dette technique à terme).
💡
Pourquoi chaque paramètre compte — Les profils de configuration uniformisent les paramètres de sécurité sur l'ensemble du parc. Un BitLocker non déployé, un firewall non configuré ou un USB non bloqué créent des vulnérabilités que les attaquants exploitent systématiquement. Le Settings Catalog permet un contrôle granulaire de chaque paramètre.

  • Doublon Settings Catalog + Template — Même paramètre dans les deux = conflit, état « Error ».
  • All devices sans exclusion — Bloque les kiosques/salles de conf. Utiliser des groupes dynamiques.
  • ADMX ingestion — Pas d'erreur si le CSP n'existe pas sur l'édition Windows (param Enterprise sur Win Pro).

Antivirus, Disk Encryption, Firewall, EDR onboarding, ASR deployment, LAPS, Windows Hello — Important — 10 pts
🚨

💡

⚙️ Configuration — CA Policy «Require Compliant Device»

PowerShell — Microsoft.Graph
# Créer une Conditional Access Policy exigeant un device compliant
Connect-MgGraph -Scopes "Policy.ReadWrite.ConditionalAccess"

# Pré-requis : créer un groupe break-glass exclu de la policy
$breakGlassGroupId = "<ID-du-groupe-break-glass>"

$caBody = @{
    displayName = "CA-REQUIRE-COMPLIANT-DEVICE"
    state       = "enabledForReportingButNotEnforced"  # Report-only d'abord !
    conditions  = @{
        users = @{
            includeUsers  = @("All")
            excludeGroups = @($breakGlassGroupId)
        }
        applications   = @{ includeApplications = @("All") }
        clientAppTypes = @("browser", "mobileAppsAndDesktopClients")
    }
    grantControls = @{
        operator        = "OR"
        builtInControls = @("compliantDevice", "domainJoinedDevice")
    }
} | ConvertTo-Json -Depth 10

Invoke-MgGraphRequest -Method POST `
    -Uri "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies" `
    -Body $caBody -ContentType "application/json"

Write-Host "[OK] CA Policy créée en mode Report-Only. Activer après 7 jours de monitoring." -ForegroundColor Green

S10
RBAC & Scope Tags Intune P1
Rôles custom, Scope Tags, délégation, moindre privilège, audit des permissions admin — Informatif — 5 pts
🚨
Scénario d'attaque — Un technicien helpdesk dispose du rôle « Intune Administrator » (Global Admin Intune). Il peut créer des Compliance Policies, modifier les baselines de sécurité, ou même wipe un device exécutif. Un attaquant compromettant ce compte a le contrôle total de la gestion des devices. Principe du moindre privilège : chaque admin ne doit voir et gérer que ce qui est nécessaire à son rôle.
💡
Pourquoi chaque paramètre compte — Les permissions RBAC déterminent qui peut modifier vos policies de sécurité. Un technicien helpdesk avec le rôle Intune Administrator peut wipe un device exécutif ou désactiver une baseline. Chaque rôle custom et chaque Scope Tag réduit le blast radius en cas de compromission de compte admin.

🤔 Pourquoi c'est important

1
Scope Tags = multi-tenant léger — Dans un groupe avec plusieurs filiales, les Scope Tags permettent de segmenter les policies, apps et devices par entité. L'admin de la filiale A ne voit que ses objets. C'est la délégation sans la complexité multi-tenant.
2
Rôles custom > rôles built-in — Les rôles built-in sont trop larges. Un « Helpdesk Operator » peut par défaut wipe un device. Créez un rôle custom « L1-Support » avec uniquement : Remote Lock, Reset Passcode, View Device. Pas de wipe, pas de retire.
3
Audit trail obligatoire — Chaque action admin est loggée dans l'Audit Log Intune. Mais si tout le monde est « Intune Admin », vous ne pouvez pas distinguer qui a modifié quoi. Des rôles granulaires = un audit log exploitable.
⚠️
Piège : Scope Tags oubliés sur les policies — Quand vous créez une Compliance Policy sans Scope Tag explicite, elle reçoit le tag « Default » visible par tous les admins. Pire : un admin avec Scope Tag « Filiale-A » ne verra pas cette policy (pas dans son scope) mais ses users seront quand même ciblés. Règle : toujours assigner un Scope Tag explicite à chaque objet.

💻 Script All-in-One — Audit Intune

Script PowerShell en 7 phases couvrant l'audit complet du tenant Intune. Utilise exclusivement Microsoft.Graph et Invoke-MgGraphRequest.

⚠️
Prérequis — Install-Module Microsoft.Graph.Authentication. Permissions requises : DeviceManagementConfiguration.Read.All, DeviceManagementManagedDevices.Read.All, DeviceManagementRBAC.Read.All, Policy.Read.All, DeviceAppManagement.Read.All.
PowerShell — Audit Intune All-in-One (7 phases)
#Requires -Modules Microsoft.Graph.Authentication
# ============================================================
# Script All-in-One : Audit Microsoft Intune
# Utilise Invoke-MgGraphRequest (PAS Microsoft.Graph.Intune déprécié)
# ============================================================

param(
    [string]$OutputPath = ".\IntuneAudit_$(Get-Date -Format 'yyyyMMdd_HHmmss')"
)

$ErrorActionPreference = "Stop"
New-Item -Path $OutputPath -ItemType Directory -Force | Out-Null

# --- Connexion ---
Connect-MgGraph -Scopes @(
    "DeviceManagementConfiguration.Read.All"
    "DeviceManagementManagedDevices.Read.All"
    "DeviceManagementRBAC.Read.All"
    "Policy.Read.All"
    "DeviceAppManagement.Read.All"
)

$results = [ordered]@{}

# ============================================================
# PHASE 1 : Enrollment & Autopilot (S1 - 15 pts)
# ============================================================
Write-Host "`n=== PHASE 1 : Enrollment & Autopilot ===" -ForegroundColor Cyan

$enrollRestrictions = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceEnrollmentConfigurations"
$autopilotProfiles = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/windowsAutopilotDeploymentProfiles"

$s1Score = 5 # base
if ($autopilotProfiles.value.Count -gt 0) { $s1Score += 4 }
$espConfigs = $enrollRestrictions.value | Where-Object { $_."@odata.type" -like "*EnrollmentStatusPage*" }
if ($espConfigs.Count -gt 0) { $s1Score += 3 }
$deviceLimit = $enrollRestrictions.value | Where-Object { $_."@odata.type" -like "*DeviceLimit*" }
if ($deviceLimit -and $deviceLimit.limit -le 5) { $s1Score += 3 }

$results["S1_Enrollment"] = $s1Score
Write-Host "  S1 Score: $s1Score/15"

# ============================================================
# PHASE 2 : Device Compliance (S2 - 15 pts)
# ============================================================
Write-Host "`n=== PHASE 2 : Device Compliance ===" -ForegroundColor Cyan

$compliancePolicies = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceCompliancePolicies"

$s2Score = 3
if ($compliancePolicies.value.Count -gt 0) { $s2Score += 3 }
$winCompliance = $compliancePolicies.value | Where-Object { $_."@odata.type" -like "*windows*" }
foreach ($cp in $winCompliance) {
    if ($cp.bitLockerEnabled) { $s2Score += 3; break }
}
foreach ($cp in $winCompliance) {
    if ($cp.defenderEnabled) { $s2Score += 3; break }
}
$markNoPolicy = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/settings"
if ($markNoPolicy.deviceComplianceCheckinThresholdDays -le 30) { $s2Score += 3 }

$s2Score = [Math]::Min($s2Score, 15)
$results["S2_Compliance"] = $s2Score
Write-Host "  S2 Score: $s2Score/15"

# ============================================================
# PHASE 3 : Configuration Profiles & Baselines (S3+S4 - 24 pts)
# ============================================================
Write-Host "`n=== PHASE 3 : Config Profiles & Baselines ===" -ForegroundColor Cyan

$configPolicies = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/configurationPolicies"
$baselines = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/intents"

$s3Score = 2
if ($configPolicies.value.Count -gt 0) { $s3Score += 4 }
if ($configPolicies.value.Count -gt 5) { $s3Score += 3 }
if ($configPolicies.value | Where-Object { $_.name -match "BitLocker|FileVault|USB" }) { $s3Score += 3 }
$s3Score = [Math]::Min($s3Score, 12)

$s4Score = 2
if ($baselines.value.Count -gt 0) { $s4Score += 5 }
if ($baselines.value.Count -ge 3) { $s4Score += 5 }
$s4Score = [Math]::Min($s4Score, 12)

$results["S3_ConfigProfiles"] = $s3Score
$results["S4_Baselines"]      = $s4Score
Write-Host "  S3 Score: $s3Score/12 | S4 Score: $s4Score/12"

# ============================================================
# PHASE 4 : Endpoint Security (S5 - 10 pts)
# ============================================================
Write-Host "`n=== PHASE 4 : Endpoint Security ===" -ForegroundColor Cyan

$epIntents = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/intents?\`$filter=templateId ne null"
$mdeConnector = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/mobileThreatDefenseConnectors"

$s5Score = 3
if ($epIntents.value.Count -gt 0) { $s5Score += 2 }
$mdeEnabled = $mdeConnector.value | Where-Object { $_.state -eq "enabled" }
if ($mdeEnabled) { $s5Score += 3 }
$asrPolicy = $epIntents.value | Where-Object { $_.displayName -match "ASR|Attack Surface" }
if ($asrPolicy) { $s5Score += 2 }
$s5Score = [Math]::Min($s5Score, 10)

$results["S5_EndpointSecurity"] = $s5Score
Write-Host "  S5 Score: $s5Score/10"

# ============================================================
# PHASE 5 : MAM & Conditional Access (S6+S7 - 18 pts)
# ============================================================
Write-Host "`n=== PHASE 5 : MAM & Conditional Access ===" -ForegroundColor Cyan

$appPolicies = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceAppManagement/managedAppPolicies"
$caPolicies = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies"

# S6 MAM (10 pts)
$s6Score = 2
$iosApps = $appPolicies.value | Where-Object { $_."@odata.type" -like "*iosManagedAppProtection*" }
$androidApps = $appPolicies.value | Where-Object { $_."@odata.type" -like "*androidManagedAppProtection*" }
if ($iosApps.Count -gt 0) { $s6Score += 3 }
if ($androidApps.Count -gt 0) { $s6Score += 3 }
if ($iosApps | Where-Object { $_.pinRequired -eq $true }) { $s6Score += 2 }
$s6Score = [Math]::Min($s6Score, 10)

# S7 Conditional Access (8 pts)
$s7Score = 3
$enabledCA = $caPolicies.value | Where-Object { $_.state -eq "enabled" }
if ($enabledCA.Count -gt 0) { $s7Score += 2 }
$requireCompliant = $enabledCA | Where-Object {
    $_.grantControls.builtInControls -contains "compliantDevice"
}
if ($requireCompliant) { $s7Score += 3 }
$s7Score = [Math]::Min($s7Score, 8)

$results["S6_MAM"] = $s6Score
$results["S7_ConditionalAccess"] = $s7Score
Write-Host "  S6 Score: $s6Score/10 | S7 Score: $s7Score/8"

# ============================================================
# PHASE 6 : Windows Update & Reporting (S8+S9 - 13 pts)
# ============================================================
Write-Host "`n=== PHASE 6 : Windows Update & Reporting ===" -ForegroundColor Cyan

$updateRings = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/deviceConfigurations?\`$filter=isof('microsoft.graph.windowsUpdateForBusinessConfiguration')"
$featureUpdates = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/windowsFeatureUpdateProfiles"

# S8 Windows Update (8 pts)
$s8Score = 4
if ($updateRings.value.Count -ge 2) { $s8Score += 2 }
if ($featureUpdates.value.Count -gt 0) { $s8Score += 2 }
$s8Score = [Math]::Min($s8Score, 8)

# S9 Reporting (5 pts)
$s9Score = 2
$deviceOverview = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/managedDeviceOverview"
if ($deviceOverview.enrolledDeviceCount -gt 0) { $s9Score += 1 }
$diagSettings = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/reports"
if ($diagSettings) { $s9Score += 2 }
$s9Score = [Math]::Min($s9Score, 5)

$results["S8_WindowsUpdate"] = $s8Score
$results["S9_Reporting"]     = $s9Score
Write-Host "  S8 Score: $s8Score/8 | S9 Score: $s9Score/5"

# ============================================================
# PHASE 7 : RBAC & Summary (S10 - 5 pts)
# ============================================================
Write-Host "`n=== PHASE 7 : RBAC & Summary ===" -ForegroundColor Cyan

$roleDefinitions = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/roleDefinitions"
$roleAssignments = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/roleAssignments"

$s10Score = 2
$customRoles = $roleDefinitions.value | Where-Object { $_.isBuiltIn -eq $false }
if ($customRoles.Count -gt 0) { $s10Score += 2 }
$scopeTags = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/roleScopeTags"
if ($scopeTags.value.Count -gt 1) { $s10Score += 1 }
$s10Score = [Math]::Min($s10Score, 5)

$results["S10_RBAC"] = $s10Score
Write-Host "  S10 Score: $s10Score/5"

# ============================================================
# SUMMARY
# ============================================================
$totalScore = ($results.Values | Measure-Object -Sum).Sum
Write-Host "`n============================================" -ForegroundColor Yellow
Write-Host "  SCORE TOTAL : $totalScore / 100" -ForegroundColor Yellow
Write-Host "============================================" -ForegroundColor Yellow

Write-Host "`nDétail par section :"
foreach ($key in $results.Keys) {
    $max = switch ($key) {
        "S1_Enrollment"      { 15 }
        "S2_Compliance"      { 15 }
        "S3_ConfigProfiles"  { 12 }
        "S4_Baselines"       { 12 }
        "S5_EndpointSecurity"{ 10 }
        "S6_MAM"             { 10 }
        "S7_ConditionalAccess"{ 8 }
        "S8_WindowsUpdate"   { 8 }
        "S9_Reporting"       { 5 }
        "S10_RBAC"           { 5 }
    }
    $pct = [Math]::Round(($results[$key] / $max) * 100)
    $bar = ("#" * [Math]::Round($pct / 5)).PadRight(20)
    Write-Host "  $($key.PadRight(22)) $($results[$key])/$max  [$bar] $pct%"
}

# --- Export JSON ---
$report = @{
    Date       = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
    TotalScore = $totalScore
    MaxScore   = 100
    Details    = $results
}
$report | ConvertTo-Json -Depth 3 | Out-File "$OutputPath\IntuneAudit_Score.json"
Write-Host "`nRapport exporté : $OutputPath\IntuneAudit_Score.json" -ForegroundColor Green

Disconnect-MgGraph
