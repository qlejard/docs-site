💡
Pourquoi chaque capacité est-elle critique ?
  • Visibilité temps réel : sans métriques CloudWatch Shield, vous ne savez pas si vous êtes sous attaque jusqu'à ce que vos utilisateurs vous appellent. Le délai de détection passe de heures à secondes avec Shield Advanced.
  • SRT 24/7 : lors d'une attaque, chaque minute d'indisponibilité coûte. Le SRT connaît les patterns d'attaque actuels (ils voient les attaques sur tous les clients AWS) et peut créer des règles WAF d'urgence en quelques minutes. Source : AWS Shield Advanced SRT Documentation.
  • Cost Protection : une attaque DDoS de 4h peut générer des milliers de dollars de data transfer soudain. Ce crédit Shield Advanced couvre ces coûts — il transforme Shield Advanced en assurance DDoS rentable.
  • Protection L7 automatique : Shield Standard ne voit pas les attaques HTTP (couche application). Une attaque à 1 000 req/s ciblant un endpoint lent peut être invisible pour Shield Standard mais paralyser votre ALB.
  • Engagement 1 an : ne souscrire que lors d'une attaque est impossible — l'activation prend du temps et les coûts de l'attaque ne sont remboursés que si vous étiez déjà souscrit. À souscrire avant, pas pendant.
⚠️
Pièges courants :
  • "Shield Standard nous suffit" : Standard couvre les attaques volumétriques L3/L4 (SYN floods), mais toutes les attaques modernes ciblées incluent une composante L7. Sans Advanced, vous êtes aveugle et sans réponse L7.
  • Souscrire Advanced trop tard : l'activation d'Advanced ne prend effet pour la Cost Protection qu'après validation. Si vous souscrivez pendant une attaque, les coûts antérieurs ne sont pas couverts.
  • Scope "organization" oublié : Shield Advanced s'active par compte. Dans une organisation AWS multi-comptes, chaque compte doit souscrire ou utiliser AWS Firewall Manager pour une gestion centralisée.
  • Support Plan insuffisant : le SRT n'est accessible que si vous avez AWS Support Business ou Enterprise. Shield Advanced sans ce plan = 3 000$/mois sans l'équipe de réponse.
  • Ne pas associer de Web ACL WAF : Shield Advanced active la protection L7 automatique uniquement si une Web ACL WAF est associée à la ressource protégée. Sans WAF = pas de mitigation L7 même avec Advanced.

1.2 — Configuration — Souscrire à Shield Advanced

1.2.1 — Console AWS (GUI)

1
Ouvrir AWS Shield Console → Get started → Subscribe to Shield Advanced
2
Lire et accepter les termes — Engagement de 1 an, 3 000$/mois (tarif indicatif, avril 2026) → cocher la case de confirmation
3
Cliquer Subscribe → la souscription est immédiate. Vérifier le statut dans Shield → Overview : l'indicateur devient ACTIVE
4
Étape critique : Naviguer vers Protected resources → Add resources to protect pour activer la protection sur vos ressources (voir Section S2). La souscription seule ne protège pas encore vos ressources.

1.2.2 — AWS CLI

Bash — Souscrire à Shield Advanced
# Variables à adapter
PROFILE="security-admin"
REGION="us-east-1"  # Shield est global mais l'API est en us-east-1

# Souscrire à Shield Advanced (engagement 1 an)
aws shield subscribe \
    --profile "$PROFILE"

# Vérifier le statut de la souscription
aws shield describe-subscription \
    --profile "$PROFILE" \
    --output json

# Activer la protection L7 automatique au niveau organisation (optionnel)
# Nécessite AWS Firewall Manager pour la centralisation
aws shield update-subscription \
    --auto-renew ENABLED \
    --profile "$PROFILE"

1.2.3 — Terraform

HCL — Souscription Shield Advanced via Terraform
# Note : la ressource aws_shield_subscription n'existe pas dans le provider Terraform officiel.

resource "aws_shield_protection" "cloudfront" {

  resource_arn = aws_cloudfront_distribution.main.arn

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

resource "aws_shield_protection" "alb" {
  name         = "shield-alb-prod"
  resource_arn = aws_lb.main.arn
}

resource "aws_shield_protection_group" "all_critical" {
  protection_group_id = "production-critical"
  aggregation         = "SUM"
  pattern             = "ARBITRARY"
  members             = [
    aws_shield_protection.cloudfront.id,
    aws_shield_protection.alb.id
  ]
}

2.2 — Protection Groups — Détection corrélée

Paramètre Valeur par défaut ⚡ Quick Win 🎯 Optimal Explication
Protection Group défini Aucun par défaut 1 group "production-critical" Group par type de ressource + environnement Corrèle les alertes pour détecter une attaque distribuée sur plusieurs endpoints
Aggregation N/A SUM SUM pour CloudFront, MAX pour EIP SUM = somme du trafic du groupe. MAX = ressource la plus ciblée. MEAN = moyenne.
Pattern N/A ARBITRARY (liste manuelle) BY_RESOURCE_TYPE ou ALL pour couverture auto ALL inclut automatiquement toutes les ressources protégées du type.
Health-based detection Activé automatiquement si health check Route 53 associé Associer health checks à chaque protection Health check sur chaque endpoint critique, avec failover Route 53 Shield utilise la dégradation du health check pour corréler avec le trafic DDoS et réduire les faux positifs.
💡
Pourquoi chaque paramètre est-il important ?
  • CloudFront en 1er : placez CloudFront devant votre stack (ALB, API Gateway, S3) — c'est le point d'entrée le plus efficace pour absorber le DDoS (CDN global, edge nodes AWS). En protégeant CloudFront, vous protégez toute l'architecture derrière. Source : AWS Best Practices for DDoS Resiliency.
  • Route 53 systématiquement : une attaque DNS peut mettre hors ligne tous vos endpoints même si vos serveurs sont fonctionnels. Route 53 est la cible la plus fréquente des attaques par amplification DNS. Shield Advanced sur Route 53 est gratuit en termes de ressource à protéger (inclus dans l'abonnement).
  • Health-based detection : sans health check associé, Shield Advanced utilise uniquement les métriques de trafic pour détecter une attaque. Avec un health check, il corrèle trafic anormal + dégradation de service — réduisant drastiquement les faux positifs et permettant au SRT de comprendre l'impact réel. Réf. : Shield Advanced — Health-based Detection.
  • EIP vs ALB : si votre architecture expose des IP publiques EC2 directement (pas de load balancer), chaque EIP doit être protégée individuellement. Une EIP oubliée peut être le vecteur d'attaque.
  • Protection Groups : une attaque distribuée sur 10 CloudFront distributions peut passer sous le seuil d'alerte individuel mais être détectée en tant qu'attaque coordonnée si les distributions sont dans un groupe commun.
⚠️
Pièges courants :
  • Penser que la souscription protège tout : Shield Advanced n'auto-protège aucune ressource. Chaque ressource doit être explicitement ajoutée via create-protection. Un compte AWS avec 20 distributions CloudFront doit ajouter chaque distribution indépendamment.
  • Oublier les EIP de NAT Gateways : les NAT Gateways ont des EIP publiques. Si votre architecture utilise une NAT Gateway exposée, son EIP doit être protégée. Sinon, le trafic de retour peut être perturbé.
  • ALB interne oublié : Shield Advanced ne protège que les ressources avec une IP publique. Un ALB internal (scheme = internal) n'est pas exposé internet et n'a pas besoin de protection Shield.
  • Limite de 1 000 protections par compte : Shield Advanced permet jusqu'à 1 000 protections par compte AWS. Pour les organisations massives, utiliser Firewall Manager pour gérer les protections centralement.
  • Health check sur l'URL de production : associez le health check sur un endpoint de monitoring dédié (ex: /health), pas sur l'URL racine (à risque de déclencher des protections lors de maintenances planifiées).

2.3.1 — Console AWS (GUI)

1

2
Sélectionner le type de ressource → cocher les ressources à protéger (CloudFront, Route 53, ALB, EIP) → cliquer Protect with Shield Advanced

2.3.2 — AWS CLI

Bash — Ajouter des ressources à Shield Advanced
# Variables à adapter
PROFILE="security-admin"
REGION="eu-west-1"
ACCOUNT_ID="123456789012"

# Protéger une distribution CloudFront
CF_ARN="arn:aws:cloudfront::${ACCOUNT_ID}:distribution/E1ABCDEF12345"
aws shield create-protection \
    --name         "shield-cloudfront-prod" \
    --resource-arn "$CF_ARN" \
    --profile      "$PROFILE"

# Protéger une hosted zone Route 53
R53_ARN="arn:aws:route53:::hostedzone/Z1ABCDEF12345"
aws shield create-protection \
    --name         "shield-route53-main" \
    --resource-arn "$R53_ARN" \
    --profile      "$PROFILE"

# Protéger un Application Load Balancer
ALB_ARN="arn:aws:elasticloadbalancing:${REGION}:${ACCOUNT_ID}:loadbalancer/app/prod-alb/1234567890abcdef"
aws shield create-protection \
    --name         "shield-alb-prod" \
    --resource-arn "$ALB_ARN" \
    --profile      "$PROFILE"

# Associer un Health Check Route 53 à une protection
PROTECTION_ID="$(aws shield describe-protection --resource-arn "$CF_ARN" --profile "$PROFILE" --query "Protection.Id" --output text)"
HEALTH_CHECK_ARN="arn:aws:route53:::healthcheck/abc12345-def6-7890-ghij-klmno1234567"
aws shield associate-health-check \
    --protection-id    "$PROTECTION_ID" \
    --health-check-arn "$HEALTH_CHECK_ARN" \
    --profile          "$PROFILE"

# Créer un Protection Group pour les ressources critiques
aws shield create-protection-group \
    --protection-group-id "production-critical" \
    --aggregation         "SUM" \
    --pattern             "ARBITRARY" \
    --members             "$CF_ARN" "$R53_ARN" "$ALB_ARN" \
    --profile             "$PROFILE"

Métrique Namespace Description Alarme recommandée Priorité

💡
Pourquoi configurer ces alarmes ?
  • DDoSDetected est binaire : la métrique DDoSDetected passe à 1 dès qu'une attaque est classifiée par Shield. C'est le signal le plus fiable car il est généré par le moteur de détection AWS, pas par une métrique de trafic indirecte. Délai de détection typ. : 5-15 minutes. Réf. : AWS Shield CloudWatch Metrics.
  • BitsPerSecond vs RequestsPerSecond : les attaques volumétriques L3/L4 (flood UDP/SYN) se mesurent en DDoSAttackBitsPerSecond. Les attaques applicatives L7 (HTTP flood, Slowloris) se mesurent en DDoSAttackRequestsPerSecond. Les deux alarmes sont nécessaires pour une couverture complète.
  • Corrélation avec backend : DDoSDetected = 1 mais tous les hosts sains (UnHealthyHostCount = 0) ? Shield mitigue correctement. DDoSDetected = 1 et des hosts tombent ? L'attaque impact le service — escalade SRT immédiate.
  • Dimension ResourceArn : les métriques Shield sont dimensionnées par ResourceArn. Créer des alarmes par ressource critique, pas une alarme globale (pour cibler les alertes).
⚠️
Pièges courants :
  • Métriques disponibles seulement avec Shield Advanced : les métriques AWS/DDoSProtection n'existent pas avec Shield Standard. Elles apparaissent dans CloudWatch uniquement après souscription Advanced et après qu'une ressource soit protégée.
  • Période d'alarme trop longue : une alarme sur 5 minutes lisse les pics courts (attaque de 60 secondes = non déclenchée). Utiliser Period: 60 (1 minute) pour les alarmes DDoSDetected.
  • SNS non abonné : créer une alarme CloudWatch sans abonnement SNS actif revient à créer une alarme silencieuse. Vérifier que le topic SNS a au moins un abonnement email ou Lambda confirmé.
  • EventBridge vs CloudWatch Alarms : pour des workflows d'automatisation (ex: activer le rate limiting WAF automatiquement lors d'une attaque), utiliser EventBridge avec la source aws.shield et l'événement DDoS Attack. Les CloudWatch Alarms sont pour les notifications humaines.

Pourquoi chaque paramètre est-il important ?
  • Rôle IAM SRT : la SRT a besoin d'un rôle IAM cross-account pour accéder à votre compte. Ce rôle est assumé par le principal drt.shield.amazonaws.com.   • Numéro de téléphone obligatoire pour Proactive Engagement : AWS valide les contacts avant d'activer Proactive Engagement. Un contact sans numéro de téléphone ne peut pas recevoir d'appel — la validation AWS échoue. Minimum : un contact avec téléphone valide.
  • Proactive Engagement : sans PE, la SRT attend que vous ouvriez un ticket support. Avec PE, si Shield détecte une attaque et que votre health check Route 53 se dégrade, la SRT vous appelle automatiquement — même à 3h du matin. Réf. : Proactive Engagement.
  • Health Checks pour Proactive Engagement : PE ne se déclenche que si un health check Route 53 associé à une protection se dégrade. Sans health check associé, PE reste inerte même lors d'attaques sévères.
⚠️
Pièges courants :
  • Basic Support Plan : avec le plan Basic, vous ne pouvez pas contacter la SRT ni activer Proactive Engagement. La SRT est exclusivement disponible pour les clients Business et Enterprise Support. Vérifier avec aws support describe-severity-levels.
  • Rôle IAM SRT créé mais permissions insuffisantes : utiliser la policy AWS managed AWSShieldDRTAccessPolicy telle quelle — ne pas la réduire. La SRT a besoin d'accéder aux logs WAF, CloudFront, de modifier les WebACLs.
  • Proactive Engagement activé sans health check : PE est actif mais si aucun health check n'est associé aux protections, PE ne se déclenchera jamais. Vérifier que HealthCheckIds n'est pas vide sur les protections critiques.
  • Contact d'urgence sans disponibilité 24/7 : définir un contact de garde à permanence ou un numéro d'astreinte — pas la messagerie d'un responsable en vacances.

Pourquoi Shield Advanced + WAF est la combinaison gagnante ?
  • Shield Advanced seul ne bloque pas le L7 : Shield Advanced protège au niveau réseau (L3/L4) automatiquement. Pour la couche applicative (L7 — HTTP/S), il a besoin d'AWS WAF pour créer des règles de blocage. Sans WAF, l'attaque HTTP atteint vos serveurs. Réf. : AWS Shield Advanced — Application Layer Protections.
  • AALDM = réponse automatique en secondes : quand une attaque HTTP flood est détectée, AALDM crée automatiquement des règles WAF temporaires qui bloquent les IP sources de l'attaque. Ce processus prend quelques minutes, contrairement à une intervention SRT manuelle (15-30 min).
  • Mode COUNT d'abord, BLOCK ensuite : si vous activez AALDM en mode BLOCK directement sur une application critique non testée, vous risquez des faux positifs (blocage de trafic légitime). Commencer par COUNT (observation) pour valider la baseline, puis passer à BLOCK.
  • Cost Protection s'applique aux coûts WAF : lors d'une attaque DDoS où WAF génère un sur-coût (milliards de règles évaluées), Shield Advanced Cost Protection couvre ces coûts supplémentaires.
⚠️
Pièges courants :
  • WebACL associée à la mauvaise ressource : une WebACL associée à un ALB ne protège pas un CloudFront et vice versa. Chaque ressource nécessite sa propre WebACL associée. Vérifier toutes les associations avec aws wafv2 list-web-acls.
  • Règles WAF Shield vs règles manuelles : Shield crée ses propres règles WAF temporaires dans votre WebACL pendant une attaque (préfixe ShieldMitigationRuleGroup_). Ne pas supprimer ces règles manuellement pendant l'incident — elles disparaissent automatiquement après la mitigation.
  • Priorité de règles WAF : les règles créées par Shield ont une priorité spécifique. Si vos règles manuelles ont une priorité plus basse (numéro plus élevé), elles s'exécutent après et peuvent interférer. Vérifier la priorité des règles.
  • AALDM non disponible pour NLB/EIP : si votre architecture expose des NLB ou des EIP publiques, AALDM ne s'applique pas. La protection L7 n'est disponible que pour CloudFront et ALB.

    --resource-arn "$CF_ARN" \
    --action       '{"Block": {}}' \

Analogie : Un incident DDoS sans runbook, c'est combattre un incendie sans plan d'évacuation. Vous sortez tout le monde — mais dans le désordre, en oubliant des étapes critiques.

Points clés du runbook DDoS :
  • Cost Protection Claim : Shield Advanced inclut une protection contre les coûts supplémentaires engendrés par une attaque DDoS (AutoScaling, WAF requêtes, Route 53 queries).   • Attack Report de Shield : Shield Advanced génère un rapport détaillé pour chaque attaque détectée (vecteurs, durée, pics). Ce rapport est accessible via describe-attack et peut être utilisé pour la conformité (PCI DSS, ISO 27001) et le post-mortem.

  • Lessons Learned : après chaque incident, répondre aux questions : l'attaque était-elle prévisible ? Quelles règles WAF manquaient ? Le runbook a-t-il été suivi ? L'architecture doit-elle évoluer (CloudFront devant l'ALB, GeoBlocking, etc.) ?

  • Cost Protection Claim délai 30 jours : la demande de Cost Protection doit être faite dans les 30 jours suivant l'attaque. Passé ce délai, AWS ne remboursera pas les coûts. Définir un processus post-incident systématique.
  • Supprimer les règles WAF SRT trop vite : la SRT crée des règles WAF temporaires pendant l'incident. Certaines clients les suppriment dès que l'attaque semble mitigée. Attendre la confirmation SRT avant de retirer ces règles (les attaques DDoS ont des phases d'intensité variable).
  • Manque de métriques de baseline : pour identifier l'impact d'une attaque, vous avez besoin d'une baseline de trafic normal. Configurer des dashboards CloudWatch avec des métriques de référence avant l'incident.
  • Pas de test du runbook : réaliser un exercice de simulation DDoS annuellement avec la SRT (via les Gamedays AWS) pour valider que le runbook fonctionne.

7.1 — Patterns d'Architecture DDoS-Resilient

Pattern Composants Protection contre Coût Recommandé pour
Anti-pattern : ALB Direct ALB avec IP publique exposée directement Protection partielle L3/L4 via Shield Standard uniquement Gratuit (Standard) Non recommandé pour trafic public
Pattern Basique CloudFront + Shield Standard DDoS L3/L4 volumétriques, obfuscation IP ALB, cache DDoS HTTP GET flood Shield Standard gratuit + coûts CloudFront Petites et moyennes applications web publiques

💡
AWS Best Practices DDoS Resiliency (BP1-BP7) :
  • BP1 — CloudFront devant tout : CloudFront dispose d'une capacité d'absorption massive via son réseau mondial (600+ PoP). Il absorbe les attaques au plus près de la source. C'est la première couche de défense. Gratuit avec Shield Standard.
  • BP2 — Obfusquer l'IP de l'origine : si l'IP de votre ALB est connue, les attaquants peuvent la cibler directement en contournant CloudFront. Restreindre les Security Groups de l'ALB aux seuls plages d'IPs CloudFront. Vérifier avec aws ec2 describe-security-groups.
  • BP3 — Global Accelerator pour les APIs : CloudFront est optimisé pour le HTTP/S. Pour les protocoles TCP/UDP (jeux, VoIP, apps natives), Global Accelerator offre les mêmes avantages d'absorption et d'anycast BGP. Protégé par Shield Advanced.
  • BP4 — Auto Scaling + Health Checks : même avec Shield, certaines attaques applicatives sophistiquées génèrent une charge CPU/RAM. Auto Scaling absorbe la charge pendant la mitigation Shield. Coupler les health checks Route 53 pour un failover automatique.
  • BP5 — Route 53 Shuffle Sharding : protection DNS contre les attaques DDoS sur les résolveurs DNS. Voir Section 8.
  • BP6 — Protection Groups Shield Advanced : protéger des groupes de ressources similaires pour une détection plus précise. Voir Section 2.
  • BP7 — WAF Managed Rules : activer AWS Managed Rules (Core Rule Set, Known Bad Inputs) pour bloquer les attaques applicatives connues sans configuration manuelle. Voir aws-waf.html.
⚠️
Pièges d'architecture :
  • IP d'origine non obfusquée : le piège le plus courant. CloudFront est en place, mais l'ALB accepte du trafic direct Internet depuis n'importe quelle IP. Le Security Group de l'ALB doit uniquement accepter les plages d'IPs CloudFront (télécharger depuis ip-ranges.amazonaws.com service "CLOUDFRONT").
  • EC2 avec IP publique directe : les instances EC2 avec des EIPs publiques sont exposées directement. Toujours placer les EC2 derrière un ALB, lui-même derrière CloudFront.
  • Auto Scaling trop lent : si l'Auto Scaling prend 5 min à déclencher et que l'attaque sature les instances en 30 secondes, les nouvelles instances arrivent trop tard. Configurer des politiques predictive scaling ou même du pre-scaling avant les événements prévisibles.

Scénario : Une attaque DDoS DNS flood vise vos serveurs de noms Route 53. Des milliards de requêtes DNS aléatoires saturent les résolveurs. Analogie : Shuffle Sharding, c'est comme avoir 10 serveurs de livraison pour différents quartiers. Si le quartier Est est attaqué, les 9 autres quartiers continuent d'être livrés normalement.

8.1 — Politiques de routage Route 53 & résilience DDoS

Politique de routage Fonctionnement Utile pour DDoS ? Health Check requis Cas d'usage
Simple Un seul endpoint, aucune logique Non — SPOF Non Développement, tests uniquement
Failover Primaire actif, secondaire passif ; bascule si health check échoue Oui — Failover automatique Obligatoire Applications critiques avec DR multi-région
Weighted Répartition du trafic selon des poids (ex: 80/20) Oui — distribution de charge Recommandé Blue/Green deployments, canary releases
Latency-based Route vers la région la plus proche du client Partiel — distribution géographique Recommandé Audiences mondiales, performances globales
Geolocation Route selon la localisation géographique du client Partiel — blocage géographique Recommandé Conformité régionale, géoblocking
Multi-Value Retourne plusieurs IPs valides (health-checkées) Oui — résilience client-side Obligatoire Distribution simple sans load balancer dédié
💡
Shuffle Sharding et protections DNS Route 53 :
  • Shuffle Sharding : Route 53 attribue à chaque zone DNS un sous-ensemble unique de 4 serveurs de noms parmi des centaines. Si un attaquant cible les serveurs de noms d'un client, seul ce client est affecté — les autres clients ont des serveurs de noms différents. Protection automatique, aucune configuration requise.
  • DNS Anycast : Route 53 utilise BGP Anycast via ses 100+ PoP mondiaux. Une attaque DDoS DNS est absorbée et distribuée à travers l'infrastructure mondiale d'AWS. Route 53 dispose d'une capacité absorbée massivement supérieure à celle d'un DNS auto-hébergé.
  • Health Checks + Failover : configurez des Health Checks Route 53 sur vos endpoints. En cas d'attaque réussie sur la région primaire, Route 53 bascule automatiquement vers la région secondaire. Coupler avec Shield Advanced pour protéger les Hosted Zones.
  • DNSSEC : empêche les attaques de type DNS spoofing et cache poisoning. Active la validation cryptographique des réponses DNS. Active par défaut dans Route 53 pour les nouveaux domaines.
⚠️
Pièges courants :
  • TTL trop élevé sur les enregistrements Failover : si le TTL de votre enregistrement DNS est de 3600 secondes (1h), un failover prend jusqu'à 1h pour être propagé aux clients. Pour les enregistrements critiques avec Failover, utiliser un TTL de 60 secondes.
  • Health Check vers l'IP ALB directe : si votre Health Check teste l'IP publique de l'ALB (et non pas le domaine CloudFront), il peut réussir même quand CloudFront est indéponible. Tester l'endpoint applicatif final.
  • Route 53 Resolver non protégé : si vous utilisez Route 53 Resolver (DNS privé VPC), les requêtes DNS internes peuvent révéler des informations sur votre infrastructure. Configurer les Firewall Rules Route 53 Resolver pour bloquer les domaines malveillants.

WEBACL_ARN="arn:aws:wafv2:${REGION}:${ACCOUNT_ID}:regional/webacl/prod-webacl/XXXX-XXXX-XXXX-XXXX"
WEBACL_ARN_CF="arn:aws:wafv2:us-east-1:${ACCOUNT_ID}:global/webacl/prod-cloudfront-webacl/XXXX-XXXX-XXXX-XXXX"

# SRT et alertes
SRT_ROLE_NAME="AWSShieldDRTAccessRole"
EMERGENCY_EMAIL="security-oncall@example.com"
EMERGENCY_PHONE="+33600000000"
SNS_EMAIL="security-alerts@example.com"
HEALTH_CHECK_ID="XXXX-XXXX-XXXX-XXXX"  # Route 53 Health Check ID (créer en S8 d'abord)

echo "╔═══════════════════════════════════════════════╗"
echo "║  AWS Shield Advanced — Configuration All-in-One ║"
echo "╚═══════════════════════════════════════════════╝"

# ── PHASE [1/12] Subscribe Shield Advanced ───────────────────────────────────
echo "[1/12] Souscription Shield Advanced..."
aws shield subscribe \
    --profile "$PROFILE" 2>&1 | grep -v "AlreadySubscribedException" || true
echo "[OK] Shield Advanced actif"

# ── PHASE [2/12] Protections des ressources ──────────────────────────────────
echo "[2/12] Création des protections Shield..."

for RESOURCE in \
    "cloudfront-prod:${CF_ARN}" \
    "alb-prod:${ALB_ARN}" \
    "route53-hosted-zone:${R53_ZONE_ARN}" \
    "eip-prod:${EIP_ARN}"; do
    NAME=${RESOURCE%%:*}
    ARN=${RESOURCE#*:}
    aws shield create-protection \
        --name         "$NAME" \
        --resource-arn "$ARN" \
        --profile      "$PROFILE" 2>&1 | grep -v "ResourceAlreadyExistsException" || true
    echo "  [OK] Protection : $NAME"
done

# ── PHASE [3/12] Protection Group ────────────────────────────────────────────
echo "[3/12] Création du Protection Group (tous web assets)..."
aws shield create-protection-group \
    --protection-group-id "web-assets-group" \
    --aggregation         "MAX" \
    --pattern             "BY_RESOURCE_TYPE" \
    --resource-type       "CLOUDFRONT_DISTRIBUTION" \
    --profile             "$PROFILE" 2>&1 | grep -v "OptimisticLockException" || true
echo "[OK] Protection Group créé"

# ── PHASE [4/12] SRT IAM Role ─────────────────────────────────────────────────
echo "[4/12] Création du rôle IAM SRT..."
aws iam create-role \
    --role-name                   "$SRT_ROLE_NAME" \
    --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"drt.shield.amazonaws.com"},"Action":"sts:AssumeRole"}]}' \
    --profile                     "$PROFILE" 2>&1 | grep -v "EntityAlreadyExists" || true
aws iam attach-role-policy \
    --role-name  "$SRT_ROLE_NAME" \
    --policy-arn "arn:aws:iam::aws:policy/service-role/AWSShieldDRTAccessPolicy" \
    --profile    "$PROFILE" 2>&1 | grep -v "NoSuchEntityException" || true
SRT_ROLE_ARN="arn:aws:iam::${ACCOUNT_ID}:role/${SRT_ROLE_NAME}"
aws shield associate-drt-role \
    --role-arn "$SRT_ROLE_ARN" \
    --profile  "$PROFILE" 2>&1 | grep -v "InvalidOperationException" || true
echo "[OK] Rôle SRT associé : $SRT_ROLE_ARN"

# ── PHASE [5/12] SRT Contacts d'urgence ──────────────────────────────────────
echo "[5/12] Configuration des contacts SRT d'urgence..."
aws shield update-emergency-contact-settings \
    --emergency-contact-list \
    "EmailAddress=${EMERGENCY_EMAIL},PhoneNumber=${EMERGENCY_PHONE},ContactNotes=SecurityOncall24x7" \
    --profile "$PROFILE"
echo "[OK] Contact d'urgence : $EMERGENCY_EMAIL"

# ── PHASE [6/12] Proactive Engagement ────────────────────────────────────────
echo "[6/12] Activation Proactive Engagement..."
aws shield enable-proactive-engagement \
    --profile "$PROFILE" 2>&1 | grep -v "InvalidOperationException" || true
echo "[OK] Proactive Engagement activé"

# ── PHASE [7/12] WAF Association (CloudFront) ────────────────────────────────
echo "[7/12] Association WAF WebACL → CloudFront..."
aws wafv2 associate-web-acl \
    --web-acl-arn  "$WEBACL_ARN_CF" \
    --resource-arn "$CF_ARN" \
    --scope        "CLOUDFRONT" \
    --region       "us-east-1" \
    --profile      "$PROFILE" 2>&1 | grep -v "WAFAssociatedItemException" || true
echo "[OK] WebACL associée à CloudFront"

# ── PHASE [8/12] WAF Association (ALB) ──────────────────────────────────────
echo "[8/12] Association WAF WebACL → ALB..."
aws wafv2 associate-web-acl \
    --web-acl-arn  "$WEBACL_ARN" \
    --resource-arn "$ALB_ARN" \
    --scope        "REGIONAL" \
    --region       "$REGION" \
    --profile      "$PROFILE" 2>&1 | grep -v "WAFAssociatedItemException" || true
echo "[OK] WebACL associée à l'ALB"

# ── PHASE [9/12] AALDM (Automatic L7 Mitigation) ────────────────────────────
echo "[9/12] Activation AALDM sur CloudFront et ALB..."
for RESOURCE_ARN in "$CF_ARN" "$ALB_ARN"; do
    aws shield enable-application-layer-automatic-response \
        --resource-arn     "$RESOURCE_ARN" \
        --action           '{"Block":{}}' \
        --profile          "$PROFILE" 2>&1 | grep -v "InvalidOperationException" || true
    echo "  [OK] AALDM activé sur : $RESOURCE_ARN"
done

# ── PHASE [10/12] SNS Topic + Alarme CloudWatch DDoSDetected ─────────────────
echo "[10/12] Création SNS Topic et alarmes CloudWatch..."
SNS_TOPIC_ARN=$(aws sns create-topic \
    --name "shield-ddos-alerts" \
    --profile "$PROFILE" \
    --region "$REGION" \
    --query 'TopicArn' --output text)
aws sns subscribe \
    --topic-arn "$SNS_TOPIC_ARN" \
    --protocol  "email" \
    --notification-endpoint "$SNS_EMAIL" \
    --profile   "$PROFILE" \
    --region    "$REGION" > /dev/null
aws cloudwatch put-metric-alarm \
    --alarm-name        "Shield-DDoSDetected" \
    --namespace         "AWS/DDoSProtection" \
    --metric-name       "DDoSDetected" \
    --dimensions        "Name=ResourceArn,Value=${CF_ARN}" \
    --statistic         "Maximum" \
    --period            "60" \
    --threshold         "1" \
    --comparison-operator "GreaterThanOrEqualToThreshold" \
    --evaluation-periods "1" \
    --alarm-actions     "$SNS_TOPIC_ARN" \
    --treat-missing-data "notBreaching" \
    --profile           "$PROFILE" \
    --region            "$REGION"
echo "[OK] SNS : $SNS_TOPIC_ARN — Alarme : Shield-DDoSDetected"

# ── PHASE [11/12] EventBridge Rule ────────────────────────────────────────────
echo "[11/12] Création règle EventBridge pour attaques Shield..."
aws events put-rule \
    --name          "shield-attack-detected" \
    --event-pattern '{"source":["aws.shield"],"detail-type":["AWS Shield Advanced - DDoS Attack Detected"]}' \
    --state         "ENABLED" \
    --profile       "$PROFILE" \
    --region        "$REGION" > /dev/null
aws events put-targets \
    --rule    "shield-attack-detected" \
    --targets "Id=sns-target,Arn=${SNS_TOPIC_ARN}" \
    --profile "$PROFILE" \
    --region  "$REGION" > /dev/null
echo "[OK] EventBridge rule : shield-attack-detected"

# ── PHASE [12/12] Vérification finale ────────────────────────────────────────
echo "[12/12] Vérification finale..."
aws shield describe-subscription \
    --profile "$PROFILE" \
    --query   "Subscription.{Status:'ACTIVE',AutoRenew:AutoRenew}"
aws shield list-protections \
    --profile "$PROFILE" \
    --query   "Protections[*].{Name:Name,ResourceArn:ResourceArn}" \
    --output  table
aws shield describe-drt-access \
    --profile "$PROFILE"

echo "  RAPPEL : Confirmer l'abonnement SNS email : $SNS_EMAIL"

✅
