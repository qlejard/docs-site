Backup Plans & Policies Pay-per-storage
Plans de sauvegarde, règles de fréquence, lifecycle, retention & assignment par tags — 🔴 Criticité maximale
Réf. CIS AWS Foundations Benchmark v7.0.0, AWS Well-Architected — Reliability Pillar
🚨
Scénario concret : Sans backup plan automatisé, un ransomware chiffre votre base RDS de production un vendredi soir. Lundi matin, l'équipe découvre que le dernier snapshot manuel date de 3 semaines. Résultat : 3 semaines de données perdues, avec un impact opérationnel, financier et réglementaire potentiellement majeur.
Analogie : un backup plan, c'est une police d'assurance automatique — vous définissez la couverture (quels services), la fréquence (combien de sauvegardes), et la durée de conservation. Sans police, vous payez de votre poche après chaque sinistre.

💡
Pourquoi chaque paramètre compte :
  • Backup plan existant : sans plan, aucune sauvegarde n'est exécutée automatiquement. La majorité des comptes AWS en production n'ont aucun plan configuré — le premier ransomware sera le dernier avertissement.
  • Fréquence (ScheduleExpression) : définit votre RPO (Recovery Point Objective). Un backup daily = RPO 24h max. Si votre business tolère seulement 1h de perte, il faut du PITR ou du horaire.
  • Backup window : fenêtre pendant laquelle le job doit démarrer. Trop large = backups imprévisibles. Trop serrée = jobs échoués si la charge est forte. 60 min est un bon compromis production.
  • Retention 35 jours (daily) : coût minimal, couvre 5 semaines de récupération. Pour la conformité réglementaire, les backups compliance (yearly) doivent atteindre 7 ans (Code de commerce art. L123-22, Livre des procédures fiscales art. L102 B).
  • Cold storage transition : pour les ressources qui le supportent, AWS Backup permet de passer d'un tier warm à un tier cold/archive à moindre coût. Le recovery point doit alors rester au moins 90 jours en cold ; pour les backups incrémentaux, AWS recommande au moins 8 jours en warm. Vérifiez la matrice de support et le pricing live par ressource.
  • Tag-based assignment : sans tags, les nouvelles ressources ne sont pas sauvegardées automatiquement. Un développeur lance un RDS de production, oublie le tag → aucun backup pendant des mois.
  • PITR (Continuous backup) : permet la restauration à 1 seconde de précision dans une fenêtre maximale de 35 jours. Indispensable pour les bases de données critiques. Supporté par AWS Backup pour Amazon S3, Amazon RDS, Amazon Aurora et SAP HANA on EC2.
⚠️
Pièges à connaître :
  • Opt-in par service : les paramètres de service opt-in s'appliquent surtout aux sélections basées sur tags. Si un type de ressource est explicitement assigné au backup plan (par exemple S3, EC2 ou RDS), il est inclus même si l'opt-in n'est pas activé pour ce service dans la région. Utiliser describe-region-settings / update-region-settings pour vérifier l'état réel.
  • PITR ≠ backup standard : le PITR consomme du stockage en continu. Pour S3, il nécessite le versioning activé. Coût supplémentaire non négligeable sur de gros volumes.
  • Cold storage minimum 90 jours : une fois en cold, le recovery point doit y rester au minimum 90 jours. Si vous mettez DeleteAfterDays=60 avec MoveToColdStorageAfterDays=30, AWS ajustera automatiquement la rétention à 120 jours.
  • Tags non hérités : les EC2 launchées par un Auto Scaling Group n'héritent pas forcément des tags de backup. Vérifier la propagation des tags dans le launch template.

1.3 — Configuration — AWS CLI

Bash — Création d'un Backup Plan "Production Daily"
# Créer un backup plan "Production Daily" — retention 35 jours, daily à 05:00 UTC
aws backup create-backup-plan \
    --profile $AWS_PROFILE \
    --region eu-west-1 \
    --backup-plan '{
        "BackupPlanName": "Production-Daily-35d",
        "Rules": [
            {
                "RuleName": "DailyBackup",
                "TargetBackupVaultName": "production-vault",
                "ScheduleExpression": "cron(0 5 ? * * *)",
                "StartWindowMinutes": 60,
                "CompletionWindowMinutes": 360,
                "Lifecycle": {
                    "DeleteAfterDays": 35
                }
            }
        ]
    }'

# Associer les ressources par tag (Backup=true)
aws backup create-backup-selection \
    --profile $AWS_PROFILE \
    --region eu-west-1 \
    --backup-plan-id $BACKUP_PLAN_ID \
    --backup-selection '{
        "SelectionName": "TagBasedSelection",
        "IamRoleArn": "arn:aws:iam::ACCOUNT_ID:role/AWSBackupDefaultServiceRole",
        "ListOfTags": [
            {
                "ConditionType": "STRINGEQUALS",
                "ConditionKey": "Backup",
                "ConditionValue": "true"
            }
        ]
    }'

1.4 — Configuration — Terraform

Terraform — Backup Plan Production Daily
# --- Provider AWS avec region et tags par défaut ---
provider "aws" {
  region = "eu-west-1"
  default_tags {
    tags = {
      Environment = "production"
      ManagedBy   = "terraform"
    }
  }
}

# --- Backup Plan : Production Daily 35 jours ---
resource "aws_backup_plan" "production_daily" {
  name = "Production-Daily-35d"

  rule {
    rule_name         = "DailyBackup"
    target_vault_name = aws_backup_vault.production.name
    schedule          = "cron(0 5 ? * * *)"
    start_window      = 60
    completion_window  = 360

    lifecycle {
      delete_after = 35
    }
  }
}

# --- Sélection par tag : Backup=true ---
resource "aws_backup_selection" "tag_based" {
  name         = "TagBasedSelection"
  plan_id      = aws_backup_plan.production_daily.id
  iam_role_arn = "arn:aws:iam::${data.aws_caller_identity.current.account_id}:role/AWSBackupDefaultServiceRole"

  selection_tag {
    type  = "STRINGEQUALS"
    key   = "Backup"
    value = "true"
  }
}

# --- Backup Plan : Compliance 7 ans (multi-règle) ---
resource "aws_backup_plan" "compliance_7y" {
  name = "Compliance-7years"

  rule {
    rule_name         = "DailyBackup"
    target_vault_name = aws_backup_vault.compliance.name
    schedule          = "cron(0 5 ? * * *)"
    start_window      = 60
    completion_window  = 360
    lifecycle {
      delete_after = 35
    }
  }

  rule {
    rule_name         = "MonthlyBackup"
    target_vault_name = aws_backup_vault.compliance.name
    schedule          = "cron(0 5 1 * ? *)"
    start_window      = 60
    completion_window  = 720
    lifecycle {
      cold_storage_after = 30
      delete_after       = 365
    }
  }

  rule {
    rule_name         = "YearlyBackup"
    target_vault_name = aws_backup_vault.compliance.name
    schedule          = "cron(0 5 1 1 ? *)"
    start_window      = 60
    completion_window  = 720
    lifecycle {
      cold_storage_after = 30
      delete_after       = 2555  # 7 ans = 2555 jours
    }
  }
}

Services Supportés & Opt-in Pay-per-storage
Activation par service, limitations, PITR, couverture complète de l'infrastructure — 🔴 Criticité maximale
Réf. AWS Backup Developer Guide — Supported Resources
🚨
Scénario concret : Votre équipe a configuré un backup plan parfait, mais personne n'a activé le service opt-in pour DynamoDB. 6 mois plus tard, la table principale est corrompue par un bug applicatif. Aucun recovery point n'existe. Le plan était là, mais il ne couvrait pas le bon service.
Analogie : l'opt-in, c'est comme inscrire chaque bâtiment auprès de l'assureur — la police existe, mais elle ne couvre que les bâtiments déclarés. Un nouvel entrepôt non déclaré n'est pas assuré.

💡
Pourquoi l'opt-in est critique :
  • Seuls EBS et RDS sont activés par défaut. Tous les autres services nécessitent une activation explicite dans AWS Backup Settings > Service opt-in.
  • DynamoDB : si l'opt-in n'est pas activé, AWS Backup ne peut pas créer de recovery points même si la table est taggée Backup=true.
  • S3 : nécessite le versioning activé sur le bucket pour le PITR. Sans versioning, seul le backup périodique fonctionne.
  • EC2 AMI : sauvegarde la configuration complète (instance type, security groups, user data) + tous les volumes attachés. Plus complet qu'un simple snapshot EBS.
  • VMware on-premises : via AWS Backup Gateway (appliance virtuelle), permet de sauvegarder les VM VMware directement dans un vault AWS.
⚠️
Pièges à connaître :
  • Opt-in régional : l'activation est par région. Activer S3 backup en eu-west-1 ne l'active pas en us-east-1. Si vous utilisez plusieurs régions, vérifier l'opt-in dans chacune.
  • RDS multi-AZ ≠ backup cross-region : le multi-AZ assure la haute disponibilité (failover), pas la reprise après sinistre régional. Seul le cross-region copy protège contre une défaillance régionale complète.
  • DynamoDB : backup natif vs AWS Backup : DynamoDB a son propre système de backup on-demand et PITR. L'opt-in AWS Backup permet de centraliser la gestion mais ne désactive pas les backups natifs — les deux coexistent et facturent séparément.
  • Redshift cross-account : non supporté. Pour le DR cross-account Redshift, utiliser la copie de snapshot manuelle avec partage de clé KMS.

Scénario concret : Un attaquant compromet un compte admin et exécute aws backup delete-recovery-point sur tous les backups avant de lancer le ransomware. Pourquoi chaque paramètre compte :
  • Vault dédié : le vault par défaut mélange tous les backups. Séparer par environnement permet des policies d'accès granulaires et évite qu'un développeur supprime accidentellement un backup de production.
  • CMK (Customer-Managed Key) : la clé AWS-managed ne permet pas de contrôler la rotation, l'accès ou la révocation. Une CMK dédiée permet de restreindre qui peut déchiffrer les backups. → Documentation KMS
  • Vault Lock compliance vs governance : governance = un admin peut retirer le lock (utile en phase de test). Compliance = irréversible, même AWS Support ne peut pas le retirer. Après la période de grâce (ChangeableForDays), c'est définitif.
  • ChangeableForDays = 3 : période de 72h pour tester la configuration Vault Lock avant qu'elle devienne irréversible. Profitez-en pour vérifier que les rétentions sont correctes.
⚠️
Pièges à connaître :
  • Vault Lock compliance = irréversible : une fois la période de grâce expirée, vous ne pouvez plus changer les rétentions min/max ni supprimer le lock. Si MinRetentionDays=365, tous les backups resteront au moins 1 an, même en dev.
  • KMS cross-region : les clés KMS sont régionales. Un cross-region copy nécessite une clé KMS dans la région de destination. Ne pas oublier de créer la clé dans la région DR.
  • Vault access policy vs IAM policy : les deux sont évaluées ensemble (intersection). Un Deny dans la vault policy prime sur un Allow dans la IAM policy.
  • Suppression de vault non vide impossible : un vault contenant des recovery points ne peut pas être supprimé. Il faut d'abord supprimer tous les recovery points (ou attendre leur expiration).

Restore & Recovery Pay-per-storage
Procédures de restauration, restore testing automatisé, runbooks par service — 🟠 Criticité haute
Réf. NIST 800-53 CP-10, ISO 27031
🚨
Scénario concret : Votre base RDS de production est corrompue. L'équipe lance une restauration — mais le restore échoue car le security group référencé n'existe plus, le subnet a changé, et personne n'a jamais testé un restore. Pendant ce temps, le service est down depuis 4 heures. Le coût de downtime pour une application critique : 5 600 $/minute en moyenne (Gartner 2023).
Analogie : un restore test, c'est un exercice incendie — vous ne découvrez pas le jour de l'incendie que la porte de secours est bloquée. Testez régulièrement que vos backups se restaurent vraiment.

4.1 — Runbook de restauration par service

Service Méthode de restore Temps typique Prérequis Points d'attention
Amazon EC2 Restore AMI → nouvelle instance 5-15 min IAM role, subnet, security group, key pair L'instance restore reçoit une nouvelle IP. Mettre à jour DNS/LB. Volumes EBS supplémentaires à rattacher manuellement si non inclus dans l'AMI.
Amazon EBS Restore snapshot → nouveau volume → attacher 1-5 min Même AZ que l'instance cible Le volume restauré peut être lent au début (lazy loading). Activer Fast Snapshot Restore (FSR) pour les volumes critiques.
Amazon RDS Restore → nouvelle instance DB 10-30 min (selon taille) DB subnet group, parameter group, security group La nouvelle instance a un nouvel endpoint. Le mot de passe master est conservé. Les read replicas ne sont pas restaurées. Reconfigurer le monitoring.
Amazon Aurora Restore cluster → nouveau cluster 10-20 min Cluster subnet group, parameter group Le nouveau cluster a un nouvel endpoint. Les Global Databases ne sont pas restaurées automatiquement. Recréer les read replicas si nécessaire.
Amazon DynamoDB Restore → nouvelle table Variable (selon taille, quelques min à heures) N/A (restauration dans la même région/compte) La table restaurée a un nouveau nom. Les index GSI/LSI sont recréés. Le mode capacité (provisioned/on-demand) est préservé. Les triggers Lambda ne sont pas restaurés.
Amazon EFS Restore complet ou sélectif (fichiers/dossiers) Variable (selon taille du filesystem) Nouveau filesystem ou écrasement de l'existant Le restore sélectif est possible (dossier spécifique). Le filesystem restauré est en mode General Purpose par défaut.
Amazon S3 Restore → même bucket ou nouveau bucket Variable Versioning activé pour PITR Restore possible d'objets spécifiques par préfixe. Les ACL et tags d'objets sont restaurés. Les lifecycle rules ne sont pas restaurées.
💡
Pourquoi le restore testing est critique :
  • Un backup non testé n'est pas un backup : selon la règle classique de la sécurité, un backup qui n'a jamais été restauré n'est qu'une promesse non vérifiée.
  • Dérive de configuration : les security groups, subnets, parameter groups évoluent. Un backup de 3 mois peut référencer des ressources qui n'existent plus.
  • AWS Backup Restore Testing : service automatisé qui restaure périodiquement des backups, vérifie leur intégrité, puis nettoie automatiquement les ressources temporaires.
  • RTO réaliste : sans test, vous estimez un RTO de 30 min. En réalité, c'est souvent 2-4h (résolution des dépendances, reconfiguration réseau, validation applicative).
⚠️
Pièges à connaître :
  • Nouvel endpoint/IP : la restauration RDS/Aurora crée toujours une nouvelle instance avec un nouvel endpoint. L'application doit être mise à jour (ou utiliser un CNAME DNS).
  • DynamoDB : restore = nouvelle table : vous ne pouvez pas restaurer par-dessus la table existante. Il faut restaurer avec un nouveau nom, puis renommer (ou mettre à jour l'application).
  • Coût du restore testing : les ressources restaurées sont facturées tant qu'elles existent. AWS Backup Restore Testing supprime automatiquement après validation, mais vérifiez la configuration ValidationWindowHours.
  • KMS cross-region : si le backup est chiffré avec une CMK régionale, le restore dans une autre région nécessite l'accès à la clé d'origine (ou un re-encrypt avec la clé locale lors du copy job).

Cross-Region & Cross-Account Pay-per-storage Organizations
Copie DR inter-région, isolation cross-account, copy jobs — 🟠 Criticité haute
Réf. AWS Well-Architected — Reliability Pillar, ANSSI Guide Sauvegarde
🚨
Scénario concret : Un incendie dans le datacenter eu-west-1 (Dublin) détruit l'intégralité de la région. Ou un attaquant compromet votre compte AWS principal et supprime tout, y compris les backups. Si vos sauvegardes sont uniquement dans la même région et le même compte, vous avez tout perdu.
Analogie : le cross-region copy, c'est une copie dans un autre pays. Le cross-account, c'est un coffre-fort chez un tiers de confiance — même si votre maison brûle et que votre coffre personnel est compromis, les copies sont intactes.

5.1 — Stratégies de copie

Stratégie Quoi Protection contre Coût supplémentaire Complexité Recommandation
Aucune copie Backup local uniquement ❌ Panne régionale, compromission compte 0 Aucune Non acceptable pour production
Cross-Region
CopyActions		 Copie dans une autre région AWS ✅ Panne régionale complète Stockage région 2 + transfert inter-région Faible Minimum recommandé
Cross-Account
DestinationBackupVaultArn		 Copie dans un compte AWS dédié DR ✅ Compromission du compte source Stockage compte DR + transfert Moyenne Fortement recommandé
Cross-Region + Cross-Account Copie dans un compte DR dans une autre région ✅ Panne régionale + compromission Stockage + 2x transfert Haute Optimal pour production critique
Multi-vault même compte Vaults différents dans la même région ❌ Panne régionale, ✅ suppression accidentelle Stockage uniquement Faible Utile mais insuffisant seul
💡
Pourquoi chaque stratégie compte :
  • Cross-Region : protège contre les catastrophes régionales (panne massive, problème de datacenter). Le RPO dépend du délai de copie (généralement < 1h pour des snapshots < 500 Go).
  • Cross-Account : protège contre la compromission du compte source. L'attaquant qui obtient un accès root au compte source ne peut pas supprimer les copies dans le compte DR (comptes séparés = IAM séparé).
  • Combinaison des deux : la règle 3-2-1 appliquée au cloud — 3 copies, 2 supports/régions différents, 1 copie hors du compte principal. → Documentation Organizations
⚠️
Pièges à connaître :
  • KMS régional : chaque région de destination nécessite sa propre clé KMS. Préparez les clés avant de configurer les copy rules.
  • Cross-account : vault access policy requise : le vault de destination doit explicitement autoriser le compte source à effectuer backup:CopyIntoBackupVault.
  • Transfert de données inter-région facturé : la facturation dépend du groupe de ressource et du trajet source/destination. Le compte source paie le transfert sortant ; le compte de destination paie le stockage de la copie. Vérifier la page AWS Backup Pricing avant d'activer des copies massives.
  • Certains services ne supportent pas toutes les variantes de copy : vérifiez la matrice S2 par ressource et par région (same-region, cross-region, cross-account, air-gapped vault).
  • Rétention séparée pour les copies : la rétention dans CopyActions.Lifecycle est indépendante du backup source. Vous pouvez garder les copies DR plus longtemps.

Disaster Recovery Patterns Variable
Stratégies DR : Backup & Restore, Pilot Light, Warm Standby, Multi-site, RPO/RTO — 🟡 Criticité moyenne
Réf. AWS Disaster Recovery Whitepaper, ISO 22301, ANSSI Guide PCA
🚨
Scénario concret : Une panne régionale AWS dure 8 heures (cf. eu-west-1 incident de 2023). Votre application critique est complètement inaccessible. Le choix de la stratégie DR dépend du coût du downtime vs. le coût de l'infrastructure DR.
Analogie : Backup & Restore = reconstruire la maison après l'incendie. Pilot Light = garder les fondations prêtes. Warm Standby = avoir une maison secondaire meublée. Multi-site = vivre dans deux maisons en même temps.

8.1 — Comparaison des stratégies DR

Stratégie RPO RTO Coût DR Composants en région DR Cas d'usage
Backup & Restore Heures 4-24h $ (stockage uniquement) Copies de backups dans un vault DR Dev/staging, applications non critiques, coût minimal
Pilot Light Minutes 1-4h $$ (DB + core infra minimale) DB répliquée (read replica cross-region), AMI pré-buildées, IaC prête Applications internes, RPO serré mais RTO tolérant
Warm Standby Secondes-Minutes 10-30 min $$$ (infra réduite active en DR) Stack applicative complète à échelle réduite, auto-scaling prêt Applications critiques, e-commerce, SaaS
Multi-Site Active-Active ~0 (temps réel) ~0 (failover DNS) $$$$ (double infrastructure) Stack complète à pleine échelle, Route53 health checks, Global Accelerator Fintech, santé, zéro downtime obligatoire
Backup & Restore + Pilot Light (hybride) Minutes 1-2h $$ (DB active + backups complémentaires) Read replica active, backups cross-region pour les autres services, IaC prête Compromis coût/RTO pour la majorité des workloads prod
💡
Pourquoi chaque stratégie compte :
  • RPO (Recovery Point Objective) = combien de données pouvez-vous perdre ? Si RPO = 1h, vous acceptez de perdre jusqu'à 1h de données.
  • RTO (Recovery Time Objective) = combien de temps pouvez-vous être hors service ? Si RTO = 30 min, le service doit être restauré en 30 min max.
  • Règle du coût : chaque divisée par 10 du RTO multiplie le coût DR par ~3-5x. Backup & Restore (RTO 24h) coûte ~5 % du budget infra. Multi-site (RTO ~0) coûte ~100 %.
  • AWS Backup = fondation : quelle que soit la stratégie DR, les backups cross-region via AWS Backup sont la fondation. Même en Multi-site, les backups sont nécessaires pour les corruptions logiques.
⚠️
Pièges à connaître :
  • RTO théorique vs. réel : le RTO mesuré en exercice est toujours 2-3x plus long que le RTO estimé. Testez régulièrement.
  • Corruption logique vs. panne régionale : la réplication temps réel (Multi-site) propage aussi les corruptions. Les backups ponctuels (AWS Backup) sont le seul moyen de revenir à un état antérieur sain.
  • DR ≠ HA : High Availability (Multi-AZ) protège contre les pannes d'AZ. DR (cross-region) protège contre les pannes régionales. Les deux sont nécessaires.
  • Coût oublié du Warm Standby : les instances réduites en DR consomment du budget même sans trafic. Prévoyez ce coût dans le budget mensuel.

Organization-Wide Backup Organizations
Backup policies Organizations, delegated administrator, gestion centralisée multi-comptes — 🔵 Complémentaire
Réf. AWS Organizations User Guide — Backup Policies
🚨
Scénario concret : Votre entreprise gère 50 comptes AWS (dev, staging, prod par BU). Chaque équipe configure ses propres backup plans — ou pas. Lors d'un audit, vous découvrez que 12 comptes n'ont aucun backup et que 8 comptes utilisent des rétentions de 7 jours au lieu de 90. Les backup policies Organizations permettent de définir un plan centralisé hérité par tous les comptes — impossible à contourner.
Analogie : les backup policies Organizations, c'est comme une politique d'assurance groupe — chaque employé est couvert automatiquement, sans avoir à souscrire individuellement.

9.1 — Composants de la gestion multi-comptes

Composant Fonction Portée Prérequis
Backup Policy (Organizations) Définit plans de backup + sélection de ressources au niveau OU/Account Organization, OU, ou Account AWS Organizations + Backup policies activées
Delegated Administrator Compte dédié (ex: «security-account») qui administre les backups pour toute l'org Organization Management account doit déléguer
Cross-Account Management Copier les backups des comptes membres vers un vault centralisé Organization Vault policy + KMS cross-account (voir S5)

💡
Pourquoi centraliser avec Organizations :
  • Conformité forcée : les comptes membres ne peuvent pas supprimer ou modifier les plans hérités. Le management account contrôle la politique.
  • Merge automatique : les policies se combinent par héritage (OU → sous-OU → compte). Un compte peut avoir sa propre policy en plus de celle héritée.
  • Delegated admin : évite d'utiliser le management account pour les opérations quotidiennes (best practice sécurité).
⚠️
Pièges à connaître :
  • Activation préalable : les backup policies doivent être activées au niveau de l'Organization (enable-policy-type BACKUP_POLICY) avant de pouvoir les attacher.
  • Policy merge conflicts : si une OU et un compte définissent des plans avec le même nom, le compte «gagne» (override). Utilisez des noms différents pour éviter la confusion.
  • IAM role dans chaque compte : la policy Organizations référence un IAM role (ex: AWSBackupDefaultServiceRole) qui doit exister dans chaque compte membre.
  • Tags manquants : si la policy sélectionne par tag Backup=true, les ressources sans ce tag ne seront pas protégées. Combinez avec une tag policy Organizations.

Cost Optimization Pay-per-storage
Tiering warm/cold, rétention optimisée, scheduling, réduction du cross-region — 🔵 Complémentaire
Réf. AWS Well-Architected — Cost Optimization Pillar
🚨
Scénario concret : Votre facture AWS Backup dérive fortement en quelques mois. En analysant, vous trouvez des backups warm conservés trop longtemps, des copies inter-région sur des environnements non critiques, des plans appliqués à du dev/staging inutile et des options annexes activées sans revue. Les leviers d'optimisation sont le tiering quand il est supporté, la rétention adaptée par environnement et la limitation des copies DR aux données critiques.
Analogie : garder tous vos backups en warm storage, c'est comme laisser toutes les lumières allumées 24/7 — ça fonctionne mais ça coûte cher pour rien.

10.1 — Leviers d'optimisation des coûts

💡
Comprendre la tarification AWS Backup :
  • Warm storage : facturé selon le type de ressource, la région et la volumétrie moyenne.
  • Cold storage : disponible uniquement pour certaines ressources ; minimum 90 jours de stockage une fois la transition effectuée.
  • Cross-region copy : transfert sortant + stockage destination ; le tarif dépend du groupe de ressource et des régions.
  • Restore : facturé selon le type de ressource et le volume restauré ; le restore testing suit un modèle de prix distinct.
⚠️
Pièges à connaître :
  • Cold storage minimum 90 jours : si vous supprimez un backup cold avant 90 jours, vous payez quand même les 90 jours. Ne mettez en cold que les backups avec rétention ≥ 120 jours.
  • RDS/Aurora : pas de cold storage : seuls EBS, EFS, et DynamoDB supportent le cold storage. Les snapshots RDS restent en warm.
  • Incremental vs. Full : la plupart des backups AWS sont incrémentaux (seuls les blocs modifiés). Le coût réel est souvent bien inférieur au volume total.
  • Vault Lock empêche la suppression : si Vault Lock est configuré avec MinRetentionDays: 365, vous ne pouvez pas réduire la rétention en dessous — même pour optimiser les coûts.

# ============================================================
# AWS Backup — Configuration complète Best Practices
# Usage: ./setup-aws-backup.sh
# Prérequis : AWS CLI v2, profil configuré, permissions admin
# ============================================================
set -euo pipefail

# --- Variables ---
PROFILE="${AWS_PROFILE:-default}"

VAULT_DR_NAME="dr-vault-frankfurt"
PLAN_NAME="production-daily-backup"

# === S2 : OPT-IN SERVICES ===
echo "[S2] Activation des services opt-in..."
aws backup update-region-settings \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --resource-type-opt-in-preference '{
        "Aurora":true,"DocumentDB":true,"DynamoDB":true,"EBS":true,
        "EC2":true,"EFS":true,"FSx":true,"Neptune":true,
        "RDS":true,"S3":true,"SAP HANA on Amazon EC2":true,
        "Storage Gateway":true,"Timestream":true,"VirtualMachine":true
    }'

# === S3 : CMK + VAULT ===
echo "[S3] Création CMK et vault..."
KMS_KEY_ID=$(aws kms create-key \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --description "AWS Backup vault encryption key" \
    --query 'KeyMetadata.KeyId' --output text)

aws kms create-alias \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --alias-name "alias/backup-vault-key" \
    --target-key-id "$KMS_KEY_ID"

aws kms enable-key-rotation \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --key-id "$KMS_KEY_ID"

aws backup create-backup-vault \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --backup-vault-name "$VAULT_NAME" \
    --encryption-key-arn "arn:aws:kms:$REGION_PRIMARY:$ACCOUNT_ID:key/$KMS_KEY_ID"

# Vault DR
KMS_DR_KEY_ID=$(aws kms create-key \
    --profile "$PROFILE" --region "$REGION_DR" \
    --description "AWS Backup DR vault encryption key" \
    --query 'KeyMetadata.KeyId' --output text)

aws backup create-backup-vault \
    --profile "$PROFILE" --region "$REGION_DR" \
    --backup-vault-name "$VAULT_DR_NAME" \
    --encryption-key-arn "arn:aws:kms:$REGION_DR:$ACCOUNT_ID:key/$KMS_DR_KEY_ID"

# === S3 : VAULT LOCK ===
echo "[S3] Activation Vault Lock..."
aws backup put-backup-vault-lock-configuration \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --backup-vault-name "$VAULT_NAME" \
    --min-retention-days 30 \
    --max-retention-days 365 \
    --changeable-for-days 3

# === S1 : BACKUP PLAN + SELECTION ===
echo "[S1] Création backup plan avec cross-region copy..."
aws backup create-backup-plan \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --backup-plan "{
        \"BackupPlanName\": \"$PLAN_NAME\",
        \"Rules\": [{
            \"RuleName\": \"daily-90d-cold30\",
            \"TargetBackupVaultName\": \"$VAULT_NAME\",
            \"ScheduleExpression\": \"cron(0 3 * * ? *)\",
            \"StartWindowMinutes\": 60,
            \"CompletionWindowMinutes\": 480,
            \"Lifecycle\": {
                \"MoveToColdStorageAfterDays\": 30,
                \"DeleteAfterDays\": 365
            },
            \"CopyActions\": [{
                \"DestinationBackupVaultArn\": \"arn:aws:backup:$REGION_DR:$ACCOUNT_ID:backup-vault:$VAULT_DR_NAME\",
                \"Lifecycle\": { \"DeleteAfterDays\": 90 }
            }]
        }]
    }"

PLAN_ID=$(aws backup list-backup-plans \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --query "BackupPlansList[?BackupPlanName=='$PLAN_NAME'].BackupPlanId" --output text)

aws backup create-backup-selection \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --backup-plan-id "$PLAN_ID" \
    --backup-selection "{
        \"SelectionName\": \"tagged-resources\",
        \"IamRoleArn\": \"arn:aws:iam::$ACCOUNT_ID:role/AWSBackupDefaultServiceRole\",
        \"ListOfTags\": [{
            \"ConditionType\": \"STRINGEQUALS\",
            \"ConditionKey\": \"Backup\",
            \"ConditionValue\": \"true\"
        }]
    }"

# === S7 : MONITORING ===
echo "[S7] Configuration EventBridge + SNS alerting..."
SNS_ARN=$(aws sns create-topic \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --name "backup-alerts" --query 'TopicArn' --output text)

aws events put-rule \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --name "backup-job-failures" \
    --event-pattern '{
        "source":["aws.backup"],
        "detail-type":["Backup Job State Change","Copy Job State Change","Restore Job State Change"],
        "detail":{"state":["FAILED","EXPIRED","ABORTED"]}
    }'

aws events put-targets \
    --profile "$PROFILE" --region "$REGION_PRIMARY" \
    --rule "backup-job-failures" \
    --targets "[{\"Id\":\"sns-alerts\",\"Arn\":\"$SNS_ARN\"}]"

echo "   aws backup list-backup-plans --profile $PROFILE --region $REGION_PRIMARY"
