Solutions écartées et choix structurants

💡
Pourquoi les outils natifs GitHub / Azure DevOps ?
  • CodeQL vs Semgrep/SonarQube : CodeQL est intégré nativement à GitHub ; il reste gratuit sur les dépôts publics et, pour les dépôts privés, passe par GitHub Code Security / GitHub Advanced Security selon le modèle acheté. Semgrep et SonarQube restent valides, mais ajoutent une pile tierce à gouverner.
  • Dependabot vs Renovate : Dependabot est natif GitHub avec zéro config pour les alerts. Renovate offre plus de flexibilité (monorepo, grouping) mais nécessite une maintenance supplémentaire. Dependabot est le choix par défaut dans ce guide.
  • GitHub Actions vs Jenkins/GitLab CI : ce guide couvre GitHub Actions + Azure Pipelines comme plateformes natives. Jenkins et GitLab CI sont mentionnés dans le contexte Defender for DevOps (qui supporte les deux) mais ne sont pas détaillés.
  • Checkov/PSRule vs tfsec/Terrascan : Microsoft Security DevOps orchestre un ensemble de scanners selon la task et la configuration. Checkov et PSRule sont des choix courants dans le parcours Microsoft ; tfsec, Terrascan et Trivy restent des alternatives valides à piloter explicitement si vous les retenez.

💡
Pourquoi chaque paramètre compte :
  • Require pull request : sans PR obligatoire, un push direct sur main bypass toute revue de code, tout status check et toute traçabilité. C'est un vecteur majeur de contournement des contrôles de supply chain internes.
  • Required reviewers (≥1, idéal 2) : un seul développeur compromis ne peut pas merger son propre code malveillant. La règle des "4 yeux" est une exigence SOC 2 Type II (CC6.1) et ISO 27001 A.12.1.4.
  • Dismiss stale reviews : sans cette option, un développeur peut obtenir une approbation, puis modifier son code après la review et merger la version modifiée.
  • Require status checks : empêche le merge de code qui ne compile pas, qui casse les tests ou qui contient des vulnérabilités détectées par SAST. Ce contrôle s'aligne bien avec les pratiques de supply chain et de gouvernance des builds.
  • Signed commits : la signature GPG/SSH prouve l'identité de l'auteur du commit. Sans elle, un attaquant peut usurper un git config user.email et créer des commits au nom d'un autre développeur.
  • Include administrators : sans cette option, les admins peuvent bypass toutes les protections — ce qui crée un risque d'escalade de privilège et viole le principe du moindre privilège (NIST SP 800-53 AC-6).
  • CODEOWNERS : assure que les modifications sur des fichiers critiques (CI/CD, IaC, configs sécurité) sont revues par les équipes compétentes, pas seulement par n'importe quel reviewer.
  • .gitignore + pre-commit hooks : dernière ligne de défense locale avant que des secrets (.env, clés privées, tfvars) n'atteignent le dépôt distant. Une fois poussés, ils restent dans l'historique Git même après suppression.
⚠️
Pièges classiques :
  • Include administrators = Disabled par défaut sur GitHub → les org admins et repo admins peuvent bypass toutes les branch protections. C'est un piège fréquent.
  • Les branch protection rules GitHub classiques s'appliquent par branche. Les repository rulesets (plus récents) permettent d'appliquer des règles à refs/heads/* en une seule config — préférer les rulesets.
  • Sur Azure DevOps, les build validation policies ne sont pas activées par défaut — il faut les ajouter manuellement pour chaque branche protégée.
  • Un CODEOWNERS sans Require review from Code Owners activé est purement informatif — il n'impose rien.

1.2 — Configuration

GitHub CLI — Activer branch protection sur main
# Créer un ruleset pour la branche main (préféré aux branch protection rules)
gh api --method POST "/repos/{owner}/{repo}/rulesets" `
  -f name="main-protection" `
  -f target="branch" `
  -f enforcement="active" `
  --input - <<'EOF'
{
  "conditions": {
    "ref_name": { "include": ["refs/heads/main"], "exclude": [] }
  },
  "rules": [
    { "type": "pull_request", "parameters": {
        "required_approving_review_count": 2,
        "dismiss_stale_reviews_on_push": true,
        "require_code_owner_review": true,
        "require_last_push_approval": true
    }},
    { "type": "required_status_checks", "parameters": {
        "required_status_checks": [{"context": "ci/build"}],
        "strict_required_status_checks_policy": true
    }},
    { "type": "commit_message_pattern", "parameters": {
        "operator": "starts_with", "pattern": ""
    }}
  ],
  "bypass_actors": []
}
EOF
Azure DevOps CLI — Activer branch policy sur main
# Variables à adapter
$org     = "https://dev.azure.com/VOTRE_ORG"
$project = "VOTRE_PROJET"
$repoId  = "REPO_ID"  # az repos show --repository NOM --query id

# Activer la politique de reviewers minimum (2 reviewers, pas de self-approval)
az repos policy approver-count create `
  --org $org --project $project `
  --repository-id $repoId `
  --branch "main" `
  --minimum-approver-count 2 `
  --creator-vote-counts false `
  --allow-downvotes false `
  --reset-on-source-push true `
  --blocking true `
  --enabled true

# Activer la politique de build validation
az repos policy build create `
  --org $org --project $project `
  --repository-id $repoId `
  --branch "main" `
  --build-definition-id $buildDefId `
  --queue-on-source-update-only true `
  --manual-queue-only false `
  --valid-duration 720 `
  --blocking true `
  --enabled true

Scénario : Un développeur commit accidentellement un client_secret Entra ID (app registration) dans le code. 💡
Pourquoi chaque paramètre compte :
  • Secret scanning + push protection : les secrets committés restent un risque majeur sur les plateformes Git. Push protection bloque le commit avant qu'il n'atteigne le dépôt distant.
  • OIDC (Workload Identity Federation) : remplace les secrets statiques (client_secret, PAT) par des jetons à courte durée pour l'authentification pipeline→Azure. Cela réduit fortement le besoin de stocker, distribuer et faire tourner des secrets persistants. Analogie : reconnaissance faciale au lieu d'un badge clonable.
  • GITHUB_TOKEN = read-only : par défaut, le GITHUB_TOKEN a des permissions read/write sur tous les scopes — un workflow compromis peut modifier le code, créer des releases, pousser sur des branches. Réduire à read-only et spécifier explicitement les permissions nécessaires par job.
  • Service connections ADO restrictées : par défaut, une service connection est accessible par tous les pipelines du projet. Un pipeline compromis (via PR malveillante) peut utiliser la connexion pour déployer du code arbitraire en production.
  • Key Vault-linked variables : les secrets dans les variable groups ADO classiques sont chiffrés au repos mais accessibles par tous les pipelines ayant accès au groupe. Key Vault ajoute RBAC, rotation, audit trail et politique de réseau.
⚠️
Pièges classiques :
  • Supprimer un secret du code ≠ le supprimer de l'historique Git. Il faut git filter-repo ou BFG Repo-Cleaner ET révoquer le secret côté provider (Microsoft Entra ID, AWS, etc.).
  • Push protection bypass : par défaut, un développeur peut bypass le push protection en cochant "I'll fix it later". Configurer who_can_bypass = nobody en production.
  • GitHub Actions pull_request_target : ce trigger s'exécute dans le contexte de la branche cible (main) avec ses secrets — une PR malveillante peut exfiltrer tous les secrets du repo. Ne jamais checkout le code de la PR dans un workflow pull_request_target sans sandbox.
  • ADO : "Allow access to all pipelines" est coché par défaut sur les service connections — le décocher immédiatement.
  • Rotation des secrets : configurer OIDC n'est utile que si vous désactivez ensuite les anciens secrets statiques. Sinon les deux voies coexistent et le secret statique reste un vecteur d'attaque.

name: Deploy to Azure (OIDC)
on:
  push:
    branches: [main]
permissions:
  id-token: write   # Requis pour OIDC

    runs-on: ubuntu-latest
    environment: production  # Lie les secrets à l'environment

          client-id: ${{ secrets.AZURE_CLIENT_ID }}

  • Environments + protection rules : créent une porte de validation humaine avant le déploiement en production. Le wait timer empêche le déploiement impulsif.
  • ADO classic pipelines : les classic build/release pipelines ne bénéficient pas du YAML versionné dans le repo, ce qui affaiblit la revue de code et la traçabilité. Migrer vers YAML pipelines aligne mieux Azure DevOps avec les pratiques modernes de gouvernance et de supply chain.

⚠️

  • Pipeline poisoning via PR : un attaquant peut modifier azure-pipelines.yml ou .github/workflows/*.yml dans une PR pour ajouter des steps malveillants. Protéger les fichiers CI via CODEOWNERS + required review.
  • actions/checkout@v4 dans pull_request_target : checkout le code de la PR dans un workflow qui a accès aux secrets de main = exfiltration de tous les secrets.
  • ADO : "Protect access to repositories in YAML pipelines" est un paramètre org-level désactivé par défaut. Sans lui, tout pipeline YAML peut accéder à tous les repos du projet.
  • Matrix/dynamic strategies : les stratégies dynamiques peuvent créer des jobs avec des permissions inattendues si les permissions ne sont pas définies au niveau de chaque job.

# devops-hardening-github.sh — Script All-in-One GitHub
# Usage : ./devops-hardening-github.sh <ORG> <REPO> [--dry-run]

ORG="${1:?Usage: $0 <ORG> <REPO> [--dry-run]}"
REPO="${2:?Usage: $0 <ORG> <REPO> [--dry-run]}"
DRY_RUN="${3:-}"

run() {
  if [[ "$DRY_RUN" == "--dry-run" ]]; then
    echo "[DRY-RUN] $*"
  else
    eval "$@"
  fi
}

echo "=== [S1] Repository Security — Branch Protection ==="
run gh api --method PUT "/repos/$ORG/$REPO/rulesets" \
  -f name="production-protection" \
  -f target="branch" -f enforcement="active" \
  --input - <<'EOF'
{"conditions":{"ref_name":{"include":["refs/heads/main"]}},"rules":[
  {"type":"pull_request","parameters":{"required_approving_review_count":2,"dismiss_stale_reviews_on_push":true,"require_code_owner_review":true}},
  {"type":"required_status_checks","parameters":{"strict_status_check_policy":true}},
  {"type":"deletion"},{"type":"non_fast_forward"},
  {"type":"required_signatures"}
]}
EOF

echo "=== [S2] Secret Management — Push Protection ==="
run gh api --method PUT "/repos/$ORG/$REPO/secret-scanning/push-protection" \
  -f status="enabled"

echo "=== [S3] Pipeline Security — Default Permissions ==="
run gh api --method PUT "/repos/$ORG/$REPO/actions/permissions/workflow" \
  -f default_workflow_permissions="read" \
  -F can_approve_pull_request_reviews=false

echo "=== [S5] Code Scanning — CodeQL Default Setup ==="
run gh api --method PATCH "/repos/$ORG/$REPO/code-scanning/default-setup" \
  -f state="configured" -f query_suite="extended" \
  -f languages='[STR0,STR1,STR2]'

  -f fine_grained_personal_access_tokens_enabled="true" \
  -f fine_grained_personal_access_tokens_require_approval="true" \
  -F max_lifetime_days=90

echo "=== [S10] Copilot — Code Referencing Filter ==="

  -f public_code_suggestions="block"

GHASGitHub Advanced Security — suite de fonctionnalités payantes incluant CodeQL, secret scanning avancé, dependency review. Depuis 2025, scindé en Secret Protection ($19/committer/mois) + Code Security ($30/committer/mois) = $49 total (tarif indicatif, avril 2026).
SLSASupply-chain Levels for Software Artifacts — framework de Google/OpenSSF définissant 4 niveaux de sécurité supply chain (L0 à L3).
SPDXSoftware Package Data Exchange — standard ISO/IEC 5962:2021 pour décrire les composants logiciels et leurs licences.
IaCInfrastructure as Code — gestion de l'infrastructure via des fichiers déclaratifs (Bicep, Terraform, ARM) versioned dans Git.
PATPersonal Access Token — jeton d'authentification pour les API GitHub/ADO. Préférer les fine-grained PATs aux classic PATs.
EMUEnterprise Managed Users — comptes GitHub entièrement gérés par Entra ID via SCIM provisioning. L'offboarding Entra = offboarding GitHub automatique.
MSDOMicrosoft Security DevOps — extension Azure DevOps / GitHub Action intégrant plusieurs outils de scan (Checkov, PSRule, Terrascan, etc.).
CodeQLMoteur d'analyse sémantique de code de GitHub. Traite le code comme des données interrogeables en QL (langage de requête spécifique).
CODEOWNERSFichier GitHub définissant les propriétaires de code par chemin. Déclenche automatiquement les reviews par les équipes concernées.
RulesetRemplacement moderne des branch protection rules GitHub. Permet des règles au niveau org, héritables et avec bypass granulaire.
DependabotService GitHub qui crée automatiquement des PRs pour mettre à jour les dépendances vulnérables ou obsolètes.
