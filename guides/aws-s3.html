Mode d’emploi de ce guide
Comment utiliser ce document comme référentiel d’audit et de configuration S3
📋
Ce document est un référentiel technique complet couvrant toutes les configurations Amazon S3 avec les best practices AWS, CIS AWS Foundations Benchmark v7.0.0 et AWS Well-Architected Framework — Security Pillar. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d’un compte AWS.

Analogies générales — Amazon S3 comme système de stockage sécurisé

🏭
Amazon S3 = l’entrepôt universel. Il stocke tout, pour tous, n’importe où — mais sans sécurisation, c’est un hangar grand ouvert.
Bucket = une salle sécurisée dans l’entrepôt, avec sa propre serrure et ses propres visiteurs autorisés.
Block Public Access = la grille de sécurité. Personne n’entre sans invitation explicite, même si une clé existe quelque part.
Object Lock = la chambre forte WORM. Les documents ne peuvent plus être modifiés ni supprimés — même par un administrateur.
Versioning = le carnet d’archive. Chaque version est conservée ; rien n’est vraiment perdu, même après une suppression.
Access Point = un guichet dédié. Chaque application a son propre point d’entrée avec ses propres règles, sans toucher la policy centrale.
Bucket Policy = le règlement intérieur affiché à l’entrée. Il définit précisément qui peut faire quoi et sous quelles conditions.

Block Public Access & ACLs Gratuit
Première ligne de défense contre l’exposition accidentelle des données — 18 points
🚨
Scénario réel : en 2023, une entreprise de santé américaine a exposé 3 millions de dossiers patients parce qu’un développeur a créé un bucket S3 avec une ACL public-read pour « tester rapidement ». Les ACLs étant activées par défaut, une seule permission erronée suffit à rendre tous les objets lisibles par n’importe qui sur Internet.

Analogie : Block Public Access = la grille de sécurité du bâtiment. Même si un employé laisse une porte ouverte, la grille extérieure empgêche les intrus d’entrer. Sans elle, la moindre erreur de configuration suffit.

S1.1 — Block Public Access (BPA) — Niveau Compte

Le BPA au niveau compte s’applique à tous les buckets du compte, même ceux créés ulter&rrueurement. C’est le filet de sécurité global.

Paramètre Défaut AWS ⚡ Quick Win 🔒 Hardened 🎯 Best Practice Coût
BlockPublicAcls
Bloque les requêtes qui activeraient des ACLs publiques		 false (anciens comptes)
true depuis 2023 (nouveaux comptes)		 true true true Gratuit
IgnorePublicAcls
Ignore les ACLs publiques existantes sur les objets		 false (anciens comptes) true true true Gratuit
BlockPublicPolicy
Bloque les bucket policies qui accordent un accès public		 false (anciens comptes) true true true Gratuit
RestrictPublicBuckets
Restreint l’accès aux buckets avec policy publique aux seuls services AWS et utilisateurs autorisés		 false (anciens comptes) true true true Gratuit

Override possible si BPA compte activé ? Non — le niveau compte prime		 Hérité du compte Hérité Hérité + appliquer également Appliquer au niveau bucket en plus du compte Gratuit
💡
Pourquoi chaque paramètre compte :
• BlockPublicAcls : empêche les futures erreurs humaines (un aws s3api put-object-acl --acl public-read accidentel).
• IgnorePublicAcls : neutralise les ACLs publiques déjà en place — essentiel pour les comptes existants avec historique d’erreurs.
• BlockPublicPolicy : empêche les bucket policies avec "Principal": "*" sans condition — vecteur d’exfiltration fréquent dans les fuites de données S3.
• RestrictPublicBuckets : même si une policy publique existe, seuls les services AWS et les identités IAM autorisées peuvent accéder — verrou d’urgence.
Source : CIS AWS Foundations Benchmark v7.0.0 — Section 3.1.4 (Block Public Access) • AWS FSBP S3.8

⚠️
Piège fréquent : activer BucketOwnerEnforced sur un bucket qui reçoit des uploads cross-account (ex: logs CloudTrail, exports Athena) brise immédiatement ces flux car l’uploader cross-account perd l’ownership de ses objets. Tester sur un bucket de test avant de généraliser.
De même, le BPA au niveau compte bloque les sites web statiques S3 — prévoir une exception via CloudFront (Origin Access Control) plutôt qu’une exposition directe.

S1.3 — Configuration — AWS CLI

AWS CLI — Activer Block Public Access au niveau compte
# ⚡ QUICK WIN — Appliquer le BPA au niveau du compte AWS entier
# Ceci couvre TOUS les buckets présents et futurs du compte
aws s3control put-public-access-block \
    --account-id $(aws sts get-caller-identity --query Account --output text) \
    --public-access-block-configuration 'BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true' \
    --profile audit-admin --region eu-west-1

# Appliquer également au niveau bucket individuel (défense en profondeur)
# Remplacer my-bucket par le nom réel du bucket
aws s3api put-public-access-block \
    --bucket my-bucket \
    --public-access-block-configuration 'BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true' \
    --profile audit-admin --region eu-west-1

# Désactiver les ACLs (BucketOwnerEnforced)
aws s3api put-bucket-ownership-controls \
    --bucket my-bucket \
    --ownership-controls 'Rules=[{ObjectOwnership=BucketOwnerEnforced}]' \
    --profile audit-admin --region eu-west-1

S1.5 — Vérification

AWS CLI — Vérification BPA (compte + tous les buckets)
# Vérifier le BPA au niveau compte
aws s3control get-public-access-block \
    --account-id $(aws sts get-caller-identity --query Account --output text) \
    --profile audit-admin --region eu-west-1 \
    --output table

# Auditer tous les buckets — lister ceux qui n'ont pas le BPA activé
for bucket in $(aws s3api list-buckets --query "Buckets[].Name" --output text \
    --profile audit-admin); do
  result=$(aws s3api get-public-access-block --bucket $bucket \
    --profile audit-admin --region eu-west-1 2>/dev/null)
  if echo "$result" | grep -q '"false"'; then
    echo "⚠️  BPA incomplet sur: $bucket"
  fi
done

    --config-rule-name s3-bucket-public-read-prohibited \
    --compliance-types NON_COMPLIANT \
    --profile audit-admin --region eu-west-1

Scénario réel : en 2020, des chercheurs de Grayhat ont découvert 2 700 buckets S3 avec une policy contenant "Principal": "*" sans condition, permettant à n’importe quelle identité internet de lire, écrire ou supprimer des données. 💡
Pourquoi le deny explicite est préférable au allow restreint :
En IAM/S3, un allow peut être contrecarré par une identity policy plus permissive. Un deny explicite dans la bucket policy est absolu et ne peut pas être outrepassé par une identity policy IAM — même par un administrateur.
Règle : Deny > Allow dans le modèle d’évaluation des permissions AWS.

S2.8 — Terraform

Terraform — Bucket Policy Enforce TLS
data "aws_iam_policy_document" "deny_insecure" {
  statement {
    sid     = "DenyInsecureTransport"
    effect  = "Deny"
    principals { type = "*"; identifiers = ["*"] }
    actions   = ["s3:*"]
    resources = [
      aws_s3_bucket.example.arn,
      "${aws_s3_bucket.example.arn}/*",
    ]
    condition {
      test     = "Bool"
      variable = "aws:SecureTransport"
      values   = ["false"]
    }
  }
}

resource "aws_s3_bucket_policy" "example" {
  bucket = aws_s3_bucket.example.id
  policy = data.aws_iam_policy_document.deny_insecure.json

  # S'assurer que le BPA est actif avant d'appliquer la policy
  depends_on = [aws_s3_bucket_public_access_block.example]
}
⚠️
Piège fréquent — Confused Deputy : quand un service AWS (CloudTrail, Config, Load Balancer) écrit dans votre bucket, il utilise sa propre identité de service, pas la vôtre. Une condition aws:SourceVpce bloquera ces services qui n’ont pas de VPC endpoint.
Solution : combiner StringNotEquals aws:SourceVpce avec une exception ArnLike aws:SourceArn arn:aws:cloudtrail:*:*:trail/*.

Piège — Policy vide : un bucket sans policy n’est pas nécessairement sécurisé. La sécurité dépend aussi du BPA (S1) et des identity policies IAM.

Chiffrement Gratuit (SSE-S3) KMS (SSE-KMS)
Protection des données au repos — SSE-S3, SSE-KMS CMK, DSSE-KMS, enforce via bucket policy — 15 points
🚨
Scénario réel : en 2022, des données de 100 millions de clients Twitch ont été exfiltrées depuis des buckets S3 internes. Le chiffrement SSE-S3 était présent, mais AWS gère la clé — un attaquant avec accès IAM peut quand même déchiffrer. SSE-KMS avec CMK permet une séparation des droits : accès S3 ≠ accès KMS.

Analogie : SSE-S3 = cadenas fourni par l’entrepôt, dont l’entrepôt garde un double. SSE-KMS avec CMK = cadenas dont vous seul détenez la clé, au séquestre chez un tiers (AWS KMS) sous votre contrôle.

S3.1 — Comparaison des algorithmes de chiffrement S3

Versioning & Protection des Données Gratuit Réplication
WORM, réplication DR, cycle de vie des versions — Object Lock, MFA Delete, CRR/SRR — 10 points
🚨
Scénario réel : en 2021, une entreprise pharmaceutique a subi une attaque ransomware qui a supprimé et chiffré ses fichiers S3. Access Points & Multi-Tenant Payant
Isolation par application — Access Points, Object Lambda, Multi-Region Access Points — 7 points
🚨
Scénario réel : une équipe data partage un bucket S3 entre 5 applications avec des permissions différentes. La bucket policy devient un monstre de 200 lignes, une erreur de logique expose des données de facturation à l’équipe marketing. S3 Access Points permettent d’avoir une policy par application, isolée et versionnée indépendamment.

Analogie : bucket unique = guichet unique avec un règlement complexe pour chaque visiteur. Access Points = guichets dédiés par service, chacun avec son propre règlement simple et clair.

S7.1 — Comparaison des types d’Access Points

Type Accès depuis Policy dédiée Cas d’usage Coût
Access Point Standard
Alias DNS unique par application		 Internet ou VPC Oui — indépendante de la bucket policy Isolation par équipe / app dans un compte Gratuit
VPC Access Point
Accessible uniquement depuis un VPC spécifique		 VPC uniquement (NetworkOrigin: VPC) Oui + restriction VPC hardcodée App interne isolée dans un VPC privé Gratuit
Multi-Region Access Point (MRAP)
Routage automatique vers la région la plus proche		 Internet (ARN global) Oui (MRAP policy) Applications multi-régions, failover S3 automatique 1,25x le prix transfert S3

Transformation à la volée via Lambda sur GetObject		 Internet ou VPC Via Lambda execution role Masquage PII dynamique, conversion de format Lambda + 0,005$/Go

💡
Pourquoi Access Points > bucket policy monolithique :
• Simplicité : chaque Access Point a sa propre policy — lisible, testable, versionnée séparément.
• Isolation : un Access Point VPC ne peut être appelé que depuis ce VPC — impossible depuis internet même avec les bonnes credentials IAM.
• Limite : un bucket peut avoir jusqu’à 10 000 Access Points par région.
• Condition clé : la bucket policy doit déléguer les permissions à l’Access Point avec s3:DataAccessPointAccount ou une condition s3:AccessPointNetworkOrigin.

Optimisation des Coûts & Lifecycle Storage Lens
Storage classes, transitions automatiques, Intelligent-Tiering — 4 points
💸
Scénario réel : une startup SaaS stocke 10 To de logs applicatifs dans S3 Standard depuis 3 ans. Coût mensuel : 230 $. Après activation d’Intelligent-Tiering + règle de transition vers Glacier Deep Archive après 90 jours : coût réduit à 18 $/mois (−92%).

Analogie : S3 Standard = stock en rayon de magasin (à portée immédiate mais coûteux). Intelligent-Tiering = système automatique qui déplace les articles rares vers la réserve. Glacier Deep Archive = archives dans un entrepôt distant — moins cher, accès en quelques heures.

S9.1 — Comparaison des Storage Classes S3

S3 Standard 99,99% — 3 AZ Aucune Immédiat 0,023 $/Go/mois
S3 Intelligent-Tiering 99,9% — 3 AZ Aucune Immédiat → Archive 0,023 $/Go + 0,0025 $/1000 obj.
S3 Standard-IA 99,9% — 3 AZ 30 jours Immédiat 0,0125 $/Go/mois
S3 One Zone-IA 99,5% — 1 AZ 30 jours Immédiat 0,01 $/Go/mois

Pourquoi c’est important — le versioning multiplie les coûts sans Lifecycle :
• Chaque version non-current est facturée comme un objet normal — après 1 an de versioning sans règle d’expiration, le stockage peut tripler.
• Intelligent-Tiering est la solution zéro-risque pour les données à accès imprévisible : il bascule automatiquement vers le tier adapté sans intervention manuelle.
• AbortIncompleteMultipartUpload est le Quick Win le plus sous-estimé : les uploads interrompus accumulent du stockage invisible et facturable indéfiniment sans cette règle.

Incident Response & Remédiation Gratuit
Playbook bucket public détecté, isolation, analyse post-incident — 3 points
🚨
Playbook d’urgence : un bucket S3 est signalé comme public par Security Hub ou Config.
Vous disposez d’une fenêtre critique de <15 minutes pour isoler, analyser et remédier avant exfiltration massive. Ce playbook détaille les commandes exactes à exécuter, dans l’ordre, sans chercher la documentation en urgence.

S10.1 — Playbook — Bucket Public Détecté

AWS CLI — Playbook Incident Response S3 (4 phases)
# ═══════════════════════════════════════════════════
# PHASE 1/4 — ISOLER (0-2 min) : bloquer immédiatement l'accès public
# ═══════════════════════════════════════════════════
BUCKET="nom-du-bucket-compromis"
PROFILE="incident-responder"
REGION="eu-west-1"

# Activer le Block Public Access en urgence
aws s3api put-public-access-block \
    --bucket $BUCKET \
    --public-access-block-configuration \
        "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true" \
    --profile $PROFILE --region $REGION

# Supprimer toute ACL publique sur le bucket
aws s3api put-bucket-acl \
    --bucket $BUCKET \
    --acl private \
    --profile $PROFILE --region $REGION

# ═══════════════════════════════════════════════════
# PHASE 2/4 — ANALYSER (2-10 min) : étendue de l'exposition
# ═══════════════════════════════════════════════════
# Vérifier la policy du bucket (chercher Principal:"*" sans condition)
aws s3api get-bucket-policy \
    --bucket $BUCKET --profile $PROFILE --region $REGION \
    --query Policy --output text | python -m json.tool

# Interroger CloudTrail : qui a accédé pendant les 24 dernières heures ?
aws cloudtrail lookup-events \
    --lookup-attributes AttributeKey=ResourceName,AttributeValue=$BUCKET \
    --start-time $(date -u -d '-24 hours' '+%Y-%m-%dT%H:%M:%SZ') \
    --profile $PROFILE --region $REGION \
    --query "Events[?EventName=='GetObject'].{User:Username,Time:EventTime,IP:CloudTrailEvent}" \
    --output table

# ═══════════════════════════════════════════════════
# PHASE 3/4 — REMÉDIER (10-15 min) : corriger la cause racine
# ═══════════════════════════════════════════════════
# Supprimer la policy publique (ATTENTION : vérifier avant suppression)
aws s3api delete-bucket-policy \
    --bucket $BUCKET --profile $PROFILE --region $REGION

# Remettre une policy TLS minimale sécurisée
aws s3api put-bucket-policy \
    --bucket $BUCKET \
    --policy '{"Version":"2012-10-17","Statement":[{"Sid":"DenyInsecureTransport","Effect":"Deny","Principal":"*","Action":"s3:*","Resource":["arn:aws:s3:::BUCKET","arn:aws:s3:::BUCKET/*"],"Condition":{"Bool":{"aws:SecureTransport":"false"}}}]}' \
    --profile $PROFILE --region $REGION

# ═══════════════════════════════════════════════════
# PHASE 4/4 — NOTIFIER & DOCUMENTER
# ═══════════════════════════════════════════════════
# Vérifier la conformité après remédiation
aws configservice start-config-rules-evaluation \
    --config-rule-names s3-bucket-public-read-prohibited s3-bucket-level-public-access-prohibited \
    --profile $PROFILE --region $REGION

Post-incident — questions obligatoires :
• Quelle modification a rendu le bucket public ? (CloudTrail PutBucketPolicy / PutBucketAcl)
• Quels objets ont été lus ? (GetObject dans CloudTrail Data Events ou Server Access Logs)
• Les données exposées contiennent-elles des PII → obligation de notification RGPD dans les 72h ?
• SCP/preventive control manquant à ajouter pour empêcher la récurrence ?

Script All-in-One — Hardening Complet S3
Script AWS CLI exécutable en une seule fois pour appliquer toutes les best practices
⚠️
Prérequis : définir les variables BUCKET, LOG_BUCKET, KMS_KEY_ID et ACCOUNT_ID avant exécution. Ce script applique les Quick Wins + Hardened sans Object Lock ni MFA Delete (nécessitent une validation manuelle).
Bash — Script All-in-One Amazon S3 Hardening (phases 1/10 → 10/10)
#!/usr/bin/env bash
# Amazon S3 Security Hardening — All-in-One Script
# Prérequis : AWS CLI v2, profil IAM avec droits S3 + KMS + Config + EC2
set -euo pipefail

BUCKET="mon-bucket-a-securiser"     # <-- MODIFIER
LOG_BUCKET="mon-bucket-de-logs"       # <-- MODIFIER (bucket dédié aux logs)
KMS_KEY_ID="arn:aws:kms:eu-west-1:123456789012:key/xxxx"  # <-- MODIFIER
ACCOUNT_ID="123456789012"             # <-- MODIFIER
PROFILE="default"
REGION="eu-west-1"

echo "[1/10] Activation Block Public Access au niveau du compte AWS..."
aws s3control put-public-access-block \
    --account-id $ACCOUNT_ID \
    --public-access-block-configuration \
        "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true" \
    --profile $PROFILE --region $REGION

echo "[2/10] Activation Block Public Access sur le bucket $BUCKET..."
aws s3api put-public-access-block \
    --bucket $BUCKET \
    --public-access-block-configuration \
        "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true" \
    --profile $PROFILE --region $REGION

echo "[3/10] Désactivation des ACLs (BucketOwnerEnforced)..."
aws s3api put-bucket-ownership-controls \
    --bucket $BUCKET \
    --ownership-controls 'Rules=[{ObjectOwnership=BucketOwnerEnforced}]' \
    --profile $PROFILE --region $REGION

echo "[4/10] Application de la politique TLS obligatoire..."
aws s3api put-bucket-policy \
    --bucket $BUCKET \
    --policy '{"Version":"2012-10-17","Statement":[{"Sid":"DenyInsecureTransport","Effect":"Deny","Principal":"*","Action":"s3:*","Resource":["arn:aws:s3:::'$BUCKET'","arn:aws:s3:::'$BUCKET'/*"],"Condition":{"Bool":{"aws:SecureTransport":"false"}}}]}' \
    --profile $PROFILE --region $REGION

echo "[5/10] Chiffrement SSE-KMS par défaut..."
aws s3api put-bucket-encryption \
    --bucket $BUCKET \
    --server-side-encryption-configuration \
        '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"aws:kms","KMSMasterKeyID":"'$KMS_KEY_ID'"},"BucketKeyEnabled":true}]}' \
    --profile $PROFILE --region $REGION

echo "[6/10] Activation du versioning..."
aws s3api put-bucket-versioning \
    --bucket $BUCKET \
    --versioning-configuration Status=Enabled \
    --profile $PROFILE --region $REGION

echo "[7/10] Activation du Server Access Logging..."
aws s3api put-bucket-logging \
    --bucket $BUCKET \
    --bucket-logging-status \
        '{"LoggingEnabled":{"TargetBucket":"'$LOG_BUCKET'","TargetPrefix":"s3-access-logs/'$BUCKET'/"}}' \
    --profile $PROFILE --region $REGION

echo "[8/10] Application d'une règle Lifecycle (Standard → IA → Glacier)..."
aws s3api put-bucket-lifecycle-configuration \
    --bucket $BUCKET \
    --lifecycle-configuration '{"Rules":[{"ID":"auto-tiering","Status":"Enabled","Filter":{"Prefix":""},"Transitions":[{"Days":30,"StorageClass":"STANDARD_IA"},{"Days":90,"StorageClass":"GLACIER"}],"NoncurrentVersionTransitions":[{"NoncurrentDays":30,"StorageClass":"GLACIER"}],"NoncurrentVersionExpiration":{"NoncurrentDays":365},"AbortIncompleteMultipartUpload":{"DaysAfterInitiation":7}}]}' \
    --profile $PROFILE --region $REGION

echo "[9/10] Vérification des Config Rules S3 essentielles..."
aws configservice describe-compliance-by-config-rule \
    --config-rule-names \
        s3-bucket-public-read-prohibited \
        s3-bucket-ssl-requests-only \
        s3-default-encryption-kms \
        s3-bucket-logging-enabled \
        s3-bucket-versioning-enabled \
    --profile $PROFILE --region $REGION \
    --query "ComplianceByConfigRules[].{Rule:ConfigRuleName,Status:Compliance.ComplianceType}" \
    --output table

echo "[10/10] Rapport final : état de sécurité du bucket $BUCKET"
aws s3api get-bucket-encryption     --bucket $BUCKET --profile $PROFILE --region $REGION
aws s3api get-public-access-block   --bucket $BUCKET --profile $PROFILE --region $REGION
aws s3api get-bucket-versioning     --bucket $BUCKET --profile $PROFILE --region $REGION
aws s3api get-bucket-logging        --bucket $BUCKET --profile $PROFILE --region $REGION

echo "Hardening terminé. Vérifiez les étapes manuelles restantes :"
echo "  - Object Lock : activer AVANT les premiers uploads (S4)"
echo "  - MFA Delete : nécessite les root credentials (S4)"
echo "  - VPC Endpoint : aws ec2 create-vpc-endpoint (S6)"
echo "  - CloudTrail Data Events : aws cloudtrail put-event-selectors (S5)"
✅
