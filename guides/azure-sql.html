Authentication & Access Control Inclus Azure SQL
Microsoft Entra ID auth, Entra-only mode, contained users, Managed Identity, least privilege — 🔴 Critique (18 pts)
🔐
Analogie : L'authentification SQL, c'est comme une clé physique qu'on copie à l'infini sans registre — n'importe qui peut entrer sans traçabilité. Microsoft Entra ID, c'est un badge électronique nominatif avec MFA, logs et révocation instantanée.

Pourquoi c'est critique : L'authentification est la première ligne de défense. Par défaut, Azure SQL autorise l'authentification SQL (login/password) stockée localement, sans MFA, sans Conditional Access, sans contrôle centralisé. Le passage à Entra-only supprime 90 % de la surface d'attaque d'authentification.

Admin serveur SQL admin (login/password) Entra admin désigné (groupe) Entra admin (groupe PIM) Entra admin PIM + JIT + breakglass
Utilisateurs de base SQL logins (CREATE LOGIN) Contained users Entra Contained Entra + groupes Contained Entra + groupes dynamiques

Pourquoi c'est important

D
Décideur : L'authentification SQL expose à des fuites de credentials (pas de MFA, pas de logs Entra). En cas de compromission, aucune révocation centralisée possible. Les normes ISO 27001, SOC 2 et RGPD exigent une authentification forte centralisée.
A
Admin : Avec Entra-only, plus besoin de gérer des mots de passe SQL. Les Managed Identities éliminent tout secret dans les connection strings. Les logs d'authentification remontent directement dans Entra ID Sign-in Logs + Azure SQL Auditing.
Ar
Architecte : Entra-only simplifie le modèle : un seul Identity Provider, un seul plan de Conditional Access. Les Managed Identities (user-assigned) sont portables entre environnements (dev/staging/prod) et compatibles avec Key Vault, Storage, Event Hub.
🚨
Piège classique : Activer Entra admin sans désactiver SQL auth. Le serveur reste vulnérable car les anciens logins SQL continuent de fonctionner sans MFA. Il faut activer le Microsoft Entra-only authentication pour que seuls les comptes Entra soient acceptés. À vérifier aussi : les connection strings applicatives qui utilisent encore User ID=sa;Password=... — elles casseront après activation d'Entra-only.

-- S1 : Créer des contained users Entra ID dans la base
-- Exécuter connecté en tant qu'Entra admin sur la base cible

-- Utilisateur individuel Entra
CREATE USER [john.doe@contoso.com] FROM EXTERNAL PROVIDER;

-- Groupe Entra (recommandé)
CREATE USER [sg-sql-readers] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [sg-sql-readers];

-- Managed Identity (pour accès applicatif sans secret)
CREATE USER [mi-app-backend-prod] FROM EXTERNAL PROVIDER;

-- Rôle custom granulaire (Best Practice)
CREATE ROLE role_api_readwrite;
GRANT SELECT, INSERT, UPDATE ON SCHEMA::dbo TO role_api_readwrite;
DENY DELETE ON SCHEMA::dbo TO role_api_readwrite;
ALTER ROLE role_api_readwrite ADD MEMBER [mi-app-backend-prod];

-- Row-Level Security (RLS) — exemple filtrage par tenant
CREATE SCHEMA Security;
GO
CREATE FUNCTION Security.fn_tenant_filter(@TenantId INT)
RETURNS TABLE
WITH SCHEMABINDING
AS
    RETURN SELECT 1 AS result
    WHERE @TenantId = CAST(SESSION_CONTEXT(N'TenantId') AS INT);

// S1 : Bicep — Azure SQL Server avec Entra-only authentication

param location string = resourceGroup().location
param sqlServerName string = 'sql-prod-001'
param entraAdminGroupName string = 'sg-sql-admins'
param entraAdminGroupObjectId string

resource sqlServer 'Microsoft.Sql/servers@2023-08-01-preview' = {
  name: sqlServerName
  location: location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    // Pas de administratorLogin/administratorLoginPassword = Entra-only
    minimalTlsVersion: '1.2'
    administrators: {
      administratorType: 'ActiveDirectory'
      principalType: 'Group'
      login: entraAdminGroupName
      sid: entraAdminGroupObjectId
      tenantId: tenant().tenantId
      azureADOnlyAuthentication: true   // ← Entra-only
    }
  }
}

output sqlServerFqdn string = sqlServer.properties.fullyQualifiedDomainName

T-SQL — Vérifier les utilisateurs et permissions
-- Vérification S1 : Lister les utilisateurs Entra et leurs rôles

-- 1. Lister tous les database principals (contained users)
SELECT name, type_desc, authentication_type_desc, create_date
FROM sys.database_principals
WHERE type IN ('E', 'X')   -- E=Entra user, X=Entra group
ORDER BY type_desc, name;

-- 2. Vérifier les membres des rôles fixes
SELECT r.name AS role_name, m.name AS member_name, m.type_desc
FROM sys.database_role_members drm
JOIN sys.database_principals r ON drm.role_principal_id = r.principal_id
JOIN sys.database_principals m ON drm.member_principal_id = m.principal_id
ORDER BY r.name;

-- 3. Détecter les utilisateurs avec db_owner (à réduire)
SELECT m.name AS user_name, m.type_desc
FROM sys.database_role_members drm
JOIN sys.database_principals r ON drm.role_principal_id = r.principal_id
JOIN sys.database_principals m ON drm.member_principal_id = m.principal_id
WHERE r.name = 'db_owner';
-- Attendu : uniquement l'Entra admin ou les comptes breakglass

Network Security Inclus Azure SQL
Firewall rules, private endpoints, VNet rules, connection policy (Redirect/Proxy/Default) — 🔴 Critique (15 pts)
🌐
Analogie : Un private endpoint, c'est comme un tunnel privé entre votre bureau et la banque — les données ne transitent jamais par la rue (internet public). Sans private endpoint, votre serveur SQL est exposé sur internet avec un simple firewall IP comme seule protection.

Pourquoi c'est critique : Par défaut, Azure SQL Database est accessible via un endpoint public (*.database.windows.net). Même avec un firewall, une règle 0.0.0.0-255.255.255.255 ou « Allow Azure services » ouvre l'accès à toute IP Azure. Le private endpoint élimine l'exposition publique.

Pourquoi c'est important

D
Décideur : Un serveur SQL accessible publiquement est une cible pour les scanners automatiques et les brute-force. L'amende RGPD pour exposition de données personnelles peut atteindre 4 % du CA. Le private endpoint réduit la surface d'attaque réseau à zéro.
A
Admin : Private endpoint = une IP privée dans votre VNet (ex. 10.0.1.5). Plus besoin de gérer des listes d'IP publiques dans le firewall. La résolution DNS se fait via Private DNS Zone (privatelink.database.windows.net).
Ar
Architecte : La connection policy Redirect (port 1433 + ports 11000-11999) offre de meilleures performances et permet le routage direct vers le nœud backend. Avec un private endpoint, le traffic ne quitte jamais le backbone Azure.
🚨
Piège classique : Activer « Allow Azure services and resources to access this server ». Cette règle ajoute 0.0.0.0 au firewall et autorise toute ressource Azure (y compris d'autres tenants !) à se connecter. Ce n'est pas limité à votre subscription. Toujours désactiver et utiliser des private endpoints ou des VNet rules spécifiques.

Configuration

Az CLI — Private endpoint & firewall hardening
# --- S2 : Network Security (Az CLI) ---

RG="rg-sql-prod"
SERVER="sql-prod-001"
VNET="vnet-prod"
SUBNET_PE="snet-privateendpoints"
PE_NAME="pe-sql-prod-001"

# 1. Désactiver "Allow Azure services"
az sql server firewall-rule delete \
  --resource-group $RG \
  --server $SERVER \
  --name "AllowAllWindowsAzureIps" 2>/dev/null

# 2. Supprimer les règles firewall trop larges
az sql server firewall-rule list \
  --resource-group $RG \
  --server $SERVER \
  --query "[?startIpAddress=='0.0.0.0'].name" -o tsv | \
  xargs -I {} az sql server firewall-rule delete \
    --resource-group $RG --server $SERVER --name {}

# 3. Créer le private endpoint
SQL_ID=$(az sql server show --resource-group $RG --name $SERVER --query id -o tsv)
az network private-endpoint create \
  --name $PE_NAME \
  --resource-group $RG \
  --vnet-name $VNET \
  --subnet $SUBNET_PE \
  --private-connection-resource-id $SQL_ID \
  --group-ids sqlServer \
  --connection-name "plsc-sql-prod-001"

# 4. Configurer la DNS zone privée
az network private-dns zone create \
  --resource-group $RG \
  --name "privatelink.database.windows.net"

az network private-dns link vnet create \
  --resource-group $RG \
  --zone-name "privatelink.database.windows.net" \
  --name "dnslink-vnet-prod" \
  --virtual-network $VNET \
  --registration-enabled false

az network private-endpoint dns-zone-group create \
  --resource-group $RG \
  --endpoint-name $PE_NAME \
  --name "default" \
  --private-dns-zone "privatelink.database.windows.net" \
  --zone-name "privatelink.database.windows.net"

# 5. Désactiver l'accès public
az sql server update \
  --resource-group $RG \
  --name $SERVER \
  --set publicNetworkAccess="Disabled"

# 6. Forcer connection policy Redirect
az sql server conn-policy update \
  --resource-group $RG \
  --server $SERVER \
  --connection-type "Redirect"

# 7. TLS 1.2 minimum
az sql server update \
  --resource-group $RG \
  --name $SERVER \
  --minimal-tls-version "1.2"

  parent: vnet

Transparent Data Encryption (TDE) Inclus Azure SQL
TDE service-managed vs CMK (Key Vault), BYOK, auto-rotation, infrastructure encryption — 🔴 Critique (12 pts)
🔒
Analogie : TDE, c'est un coffre-fort pour le disque dur. Toutes les données sont chiffrées au repos (fichiers data, logs, backups) de manière transparente pour l'application.   --name $KV_NAME \
  --resource-group $RG \
  --enable-soft-delete true \
  --enable-purge-protection true \
  --sku premium   # premium = HSM-backed (recommandé)

# 2. Créer la clé RSA 2048 pour TDE
az keyvault key create \
  --vault-name $KV_NAME \
  --name $KEY_NAME \
  --kty RSA \
  --size 2048 \
  --ops wrapKey unwrapKey

# 3. Assigner la Managed Identity du serveur SQL
SQL_MI_ID=$(az sql server show --resource-group $RG --name $SERVER \
  --query identity.principalId -o tsv)

  --name $KV_NAME \
  --object-id $SQL_MI_ID \
  --key-permissions get wrapKey unwrapKey

  --query key.kid -o tsv)

  --resource-group $RG \
  --server $SERVER \
  --server-key-type AzureKeyVault \
  --kid $KEY_URI

  --vault-name $KV_NAME \
  --name $KEY_NAME \

Deterministic vs randomized encryption, CMK/CEK via Key Vault, secure enclaves — 🟠 Important (10 pts)
✉️
Analogie : Always Encrypted, c'est une enveloppe scellée. Les données sont chiffrées côté client avant d'arriver au serveur SQL. D

A

Ar
Architecte : Deux types de clés : CMK (Column Master Key, dans Key Vault) et CEK (Column Encryption Key, dans la DB, chiffrée par la CMK). Le pilote client déchiffre à la volée. 🚨
Piège classique : Chiffrer une colonne utilisée dans un WHERE ou un JOIN en mode Randomized. La même valeur produit un chiffré différent à chaque fois → impossible de comparer, filtrer ou joindre. Backup Security & Geo-Replication Inclus Azure SQL
PITR, Long-Term Retention (LTR), backup redundancy, geo-replication, failover groups — 🔵 Faible (6 pts)
💾
Analogie : Les backups Azure SQL, c'est votre assurance incendie. Même avec toutes les protections, un incident grave (corruption, ransomware, erreur humaine) peut survenir. Pourquoi c'est important : Les backups sont automatiques dans Azure SQL, mais la configuration par défaut (7 jours PITR, pas de LTR, geo-redundant storage) n'est pas toujours suffisante. La résilience complète inclut geo-replication active et failover groups pour le DR.

# ═══════════════════════════════════════════════════

set -e

# ── Variables à adapter ──
RG="rg-sql-prod"
SERVER="sql-prod-001"
DB="db-prod"
LOCATION="francecentral"
ADMIN_EMAIL="security@contoso.com"
ENTRA_ADMIN_SID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
ENTRA_ADMIN_LOGIN="entra-sql-admin@contoso.com"
LA_WORKSPACE_ID="/subscriptions/xxx/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/la-sql-audit"
KV_NAME="kv-sql-cmk-prod"
VNET_RG="rg-network"
VNET_NAME="vnet-prod"
SUBNET_NAME="snet-sql"

echo "══════ S1 : Authentication ══════"
# Activer Entra-only authentication
az sql server ad-admin create --resource-group $RG --server $SERVER \
  --display-name $ENTRA_ADMIN_LOGIN --object-id $ENTRA_ADMIN_SID
az sql server update --resource-group $RG --server $SERVER \
  --enable-ad-only-auth true

echo "══════ S2 : Network Security ══════"
# Private Endpoint
az network private-endpoint create \
  --resource-group $VNET_RG --name "pe-sql-prod" \
  --vnet-name $VNET_NAME --subnet $SUBNET_NAME \
  --private-connection-resource-id $(az sql server show -g $RG -n $SERVER --query id -o tsv) \
  --group-ids sqlServer --connection-name "pec-sql-prod"
# Désactiver l'accès public
az sql server update --resource-group $RG --server $SERVER \
  --set publicNetworkAccess=Disabled
# TLS 1.2 minimum
az sql server update --resource-group $RG --server $SERVER --minimal-tls-version 1.2

echo "══════ S3 : TDE CMK ══════"
# Key Vault + CMK
az keyvault create --name $KV_NAME -g $RG -l $LOCATION \
  --enable-purge-protection --enable-soft-delete
az keyvault key create --vault-name $KV_NAME --name "tde-sql-key" --kty RSA --size 2048
# Assigner au serveur SQL (MI requise)
MI_PRINCIPAL_ID=$(az sql server show -g $RG -n $SERVER --query identity.principalId -o tsv)
az keyvault set-policy --name $KV_NAME --object-id $MI_PRINCIPAL_ID \
  --key-permissions get wrapKey unwrapKey
KEY_URI=$(az keyvault key show --vault-name $KV_NAME --name "tde-sql-key" --query key.kid -o tsv)
az sql server tde-key set --resource-group $RG --server $SERVER \
  --server-key-type AzureKeyVault --kid $KEY_URI --auto-rotation-enabled

echo "══════ S6 : Defender for SQL ══════"
az sql server advanced-threat-protection-setting update \
  --resource-group $RG --server $SERVER --state Enabled
az sql server va-setting update \
  --resource-group $RG --server $SERVER --is-enabled true --type express

echo "══════ S7 : Auditing ══════"
az sql server audit-policy update \
  --resource-group $RG --server $SERVER --state Enabled \
  --log-analytics-target-state Enabled \
  --log-analytics-workspace-resource-id $LA_WORKSPACE_ID \
  --retention-days 365 \
  --actions "BATCH_COMPLETED_GROUP" "FAILED_DATABASE_AUTHENTICATION_GROUP" \
            "SCHEMA_OBJECT_CHANGE_GROUP" "DATABASE_PERMISSION_CHANGE_GROUP"

echo "══════ S9 : Backup LTR ══════"
az sql db update -g $RG -s $SERVER -n $DB \
  --backup-storage-redundancy Geo --short-term-retention-days 35
az sql db ltr-policy set -g $RG -s $SERVER --database $DB \
  --weekly-retention P4W --monthly-retention P12M --yearly-retention P7Y --week-of-year 1

echo "✅ Hardening terminé. Vérifier les résultats avec les commandes de vérification de chaque section."

💡
Interactions clés à retenir :
• S3 ↔ S4 : TDE et Always Encrypted sont complémentaires, pas redondants. TDE protège le stockage, AE protège la donnée en mémoire/transit.
• S4 ↔ S5 : DDM et AE ne s'appliquent pas à la même colonne. Choisir l'un ou l'autre selon la sensibilité.
• S3 ↔ S9 : Si TDE CMK, le serveur DR doit avoir accès au Key Vault. Pré-configurer impérativement.
• S8 → S4/S5 : La classification guide le choix : colonnes HC → Always Encrypted, colonnes modérées → DDM.
• S6 ↔ S7 : Defender et Auditing se complètent — Defender détecte, l'audit fournit les preuves.
