Authentification & Intégration Microsoft Entra ID 🔴 Critique
Entra ID obligatoire, PAT, SSH Keys, Conditional Access, MFA, comptes de service
🚨
Scénario d'attaque — PAT compromis = accès total à votre code source
Un développeur stocke son PAT Azure DevOps (sans expiration, scope "Full access") dans un fichier .env commité par erreur. Un attaquant récupère ce PAT, clone tous les repos, accède aux pipelines et exfiltre les secrets des Variable Groups. Résultat : violation de code source, secrets d'infrastructure exposés, pipeline détourné pour injecter du code malveillant.

Analogie : un PAT sans expiration = un passe-partout maître de votre bureau, glissé sous la porte pour n'importe qui.

Pourquoi chaque paramètre compte
Désactiver l'auth Microsoft alternative : les comptes MSA/LiveID ne sont pas gérés par votre tenant Entra ID — ils échappent à vos Conditional Access policies, à vos revues d'accès, et à votre offboarding. Un ex-employé peut conserver l'accès avec son compte personnel.

PAT 30 jours max : Microsoft recommande de réduire l'usage des PATs et de leur appliquer une durée de vie courte via la policy Microsoft Entra de lifespan maximum. Un PAT expiré ne peut pas être réutilisé par un attaquant, même s'il a été exposé dans un log ou un email.

MFA via Entra ID CA : Azure DevOps ne gère pas lui-même le MFA — il délègue à Entra ID. Sans policy Conditional Access ciblant dev.azure.com, le MFA tenant ne s'applique pas nécessairement aux sessions ADO.

Scope minimal PAT : un PAT "Full access" donne lecture ET écriture sur tous les repos, pipelines, variable groups et service connections. Le scope minimal limite le rayon d'explosion en cas de vol, et la tenant policy de restriction des full-scoped PATs évite d'en recréer sans justification.
⚠️
Pièges courants
• La policy de restriction de création des PATs ne supprime pas les PATs existants : ils restent valides jusqu'à expiration, donc il faut planifier leur révocation ou leur rotation.
• Désactiver l'authentification alternative peut bloquer des comptes de service qui utilisent encore des MSA — auditer avant de désactiver.
• Une Conditional Access Policy sur Azure DevOps doit cibler l'application "Azure DevOps" (App ID : 499b84ac-1321-427f-aa17-267ca6975798) — pas "Microsoft Azure Management".

Configuration — Portail Azure DevOps

1
Désactiver l'auth alternative (MSA)
dev.azure.com/[ORG] → Organization Settings → Policies → "Allow public projects" OFF → "Allow third-party application access via OAuth" → désactiver → "Allow Microsoft account credentials in addition to Microsoft Entra ID credentials" → OFF
2
Limiter la durée des PATs
Organization Settings → Microsoft Entra → "Enforce maximum personal access token lifespan" → activer, puis définir 30 days
En complément, utiliser Organization Settings → Policies → "Restrict personal access token (PAT) creation" pour réduire qui peut encore créer ou régénérer des PATs.
3
Conditional Access via Entra ID
entra.microsoft.com → Protection → Conditional Access → Nouvelle policy → Application cible : "Azure DevOps" → Conditions : Tous les utilisateurs → Contrôle : Exiger MFA

PowerShell — Audit complet des PATs actifs (token Entra requis)
# Les PAT Lifecycle APIs n'acceptent pas un PAT en entrée : utiliser un jeton Microsoft Entra utilisateur
param(
    [string]$OrgName = "[VOTRE-ORG]",
    [string]$EntraAccessToken = "[JETON-ENTRA-UTILISATEUR]"
)

$headers = @{
    Authorization = "Bearer $EntraAccessToken"
    "Content-Type" = "application/json"
}

# Liste des PATs via les PAT Lifecycle APIs (token Entra utilisateur + permission adaptée)
$uri = "https://vssps.dev.azure.com/$OrgName/_apis/tokens/pats?api-version=7.1-preview.1"
$result = Invoke-RestMethod -Uri $uri -Headers $headers -Method Get

$result.patTokens | ForEach-Object {
    $exp = if ($_.validTo) { [datetime]$_.validTo } else { "AUCUNE" }
    $daysLeft = if ($exp -is [datetime]) { ([datetime]$exp - (Get-Date)).Days } else { -999 }
    [PSCustomObject]@{
        DisplayName  = $_.displayName
        Scope        = $_.scope
        ValidTo      = $exp
        DaysLeft     = $daysLeft
        Risk         = if ($daysLeft -eq -999) { "🔴 SANS EXPIRATION" }
                       elseif ($daysLeft -gt 30) { "🟡 > 30 jours" }
                       else { "🟢 OK" }
    }
} | Sort-Object DaysLeft | Format-Table -AutoSize

🚨
Scénario d'attaque — Service Principal compromis = accès illimité à votre infrastructure Azure
Un pipeline utilise une Service Connection de type "Service Principal (automatic)" avec un client_secret valable 2 ans. Ce secret est stocké dans les variables ADO (non liées à Key Vault). Un attaquant accède à un agent self-hosted, lit les variables d'environnement, récupère le secret, et obtient un accès Contributor sur toute la subscription Azure.

Analogie : une Service Connection = "badge d'accès au datacenter" donné à votre pipeline pour agir sur vos ressources Azure. Workload Identity Federation = "reconnaissance biométrique" — le pipeline prouve son identité sans jamais avoir de mot de passe.

Pourquoi chaque paramètre compte
WIF vs Service Principal secret : avec WIF, ADO obtient un token JWT signé par Entra ID pour chaque run de pipeline, sans aucun secret statique. Ce token a une durée de vie de quelques minutes. Même si un attaquant intercepte le token, il est inutilisable après expiration. CIS recommande WIF comme seule méthode acceptable (CIS Azure DevOps Benchmark v1.0, recommandation 5.1).

Scope Resource Group : une connexion avec scope Subscription permet d'agir sur toutes les ressources Azure de la subscription. Un scope Resource Group limite le rayon d'explosion à un seul groupe — principe de least privilege (NIST SP 800-53 AC-6).

Pipeline-level permissions : si la Service Connection est accessible à tous les pipelines du projet, un pipeline compromis peut l'utiliser pour déployer du code malveillant. Les permissions pipeline-level garantissent que seul le pipeline autorisé peut utiliser la connexion.
⚠️
Pièges — Migration WIF
• WIF nécessite que l'application Entra ID ait une Federated Credential configurée avec l'issuer ADO exact (https://vstoken.dev.azure.com/[ORG-ID]).
• Les connexions "automatic" (créées par ADO) peuvent être converties via l'UI — les connexions "manual" nécessitent une reconfiguration manuelle dans Entra ID.
• Certains tasks tiers (Terraform Cloud, HashiCorp Vault) ne supportent pas encore WIF — vérifier la compatibilité avant migration.

Azure CLI — Créer une Service Connection WIF
$orgName     = "[VOTRE-ORG]"
$projectName = "[VOTRE-PROJECT]"
$subId       = "[SUBSCRIPTION-ID]"
$rgName      = "[RESOURCE-GROUP]"  # scope au RG, pas à la subscription

# Créer la Service Connection Azure RM avec Workload Identity Federation
az devops service-endpoint azurerm create `
    --azure-rm-service-principal-authentication WorkloadIdentityFederation `
    --azure-rm-subscription-id $subId `
    --azure-rm-subscription-name "[NOM-SUBSCRIPTION]" `
    --azure-rm-tenant-id "[TENANT-ID]" `
    --name "sc-wif-$rgName" `
    --organization https://dev.azure.com/$orgName `
    --project $projectName

# Restreindre le RBAC au Resource Group uniquement
$spId = az devops service-endpoint list `
    --organization https://dev.azure.com/$orgName `
    --project $projectName `
    --query "[?name=='sc-wif-$rgName'].authorization.parameters.serviceprincipalid" `
    --output tsv

az role assignment create `
    --assignee $spId `
    --role Contributor `
    --scope /subscriptions/$subId/resourceGroups/$rgName

Scénario d'attaque — Script injection dans un pipeline YAML
Un pipeline Classic Pipeline utilise une variable d'environnement issue d'une PR externe (fork). L'attaquant soumet une PR malveillante qui contient $(System.TeamProject) dans un champ de formulaire — ce champ est injecté tel quel dans un script bash echo $TITLE. Pourquoi chaque paramètre compte
Désactiver les Classic Pipelines : les pipelines Classic (GUI) n'ont pas de code review possible — un admin peut modifier un step sans laisser de trace dans git. YAML = les modifications passent par une PR, sont reviewées, et sont auditables dans l'historique git.

Microsoft-hosted agents : chaque job démarre dans une VM fraîche, sans persistance du système de fichiers. Les agents self-hosted partagent le disque entre pipelines — un pipeline peut lire les secrets mis en cache par un pipeline précédent.

Secrets accessibles aux forks : les PRs venant de forks peuvent exécuter du YAML qui accède aux secrets si cette option est activée. OWASP CI/CD Security Risks recommande de toujours isoler les contextes de sécurité des forks (CICD-SEC-04).
⚠️

• Désactiver la création des Classic Pipelines n'élimine pas les pipelines classiques déjà présents : il faut planifier leur migration vers YAML puis leur retrait séparément.

Scénario d'attaque — Secret en clair dans les variables YAML
Un pipeline YAML contient directement la chaîne de connexion SQL dans une variable $(DB_CONNECTION_STRING) non marquée comme "secret". Pourquoi chaque paramètre compte
Key Vault-linked Variable Group : avec un VG lié à Key Vault, ADO récupère la valeur du secret directement depuis Key Vault au moment de l'exécution du pipeline, sans jamais la stocker côté ADO. Si le secret est rotaté dans Key Vault, le pipeline utilise automatiquement la nouvelle valeur sans reconfiguration.

Variables "Secret" masquées dans logs : les variables marquées "Secret" sont automatiquement masquées dans tous les outputs de pipeline (affichées comme ***). Sans ce marquage, même un echo $VAR accidentel expose le secret dans les logs.

Pipeline-level permissions : par défaut, un Variable Group est accessible à tous les pipelines du projet. Un pipeline compromis peut donc lire les secrets d'un autre pipeline. Le pipeline-level permission garantit que seuls les pipelines listés peuvent accéder au VG.
⚠️
Pièges courants
• Un secret Key Vault-linked est tout de même masqué dans les logs ADO — mais il reste accessible au code exécuté dans le pipeline. La protection est contre l'exfiltration accidentelle, pas contre le code malveillant dans le pipeline.
• La Service Connection utilisée pour lire Key Vault doit avoir le rôle Key Vault Secrets User — pas Key Vault Administrator.
• Les Variable Groups avec Key Vault nécessitent une connexion Active pour chaque refresh — si la SC est supprimée, le pipeline échoue silencieusement.

🚨

Pourquoi chaque paramètre compte
Secret Scanning vs Code Scanning vs Dependency Scanning : ces trois types couvrent des risques différents. Secret Scanning détecte les données sensibles statiques (clés API, tokens, mots de passe). Code Scanning (CodeQL) détecte les vulnérabilités de logique dans votre code (SQL injection, XSS, path traversal). Dependency Scanning (SCA) détecte les vulnérabilités connues dans vos bibliothèques tierces (CVEs dans les dépendances npm, NuGet, Maven).

Push Protection : contrairement au Secret Scanning standard (qui crée une alerte après détection), la Push Protection bloque le push avant que le secret n'atteigne le repo. Le développeur voit un message d'erreur explicatif et doit corriger ou bypasser de manière exceptionnelle selon la procédure prévue.

Coût GHAS-ADO : Microsoft distingue désormais Secret Protection et Code Security, chacun facturé par committer actif sur les repos couverts. Le guide ne doit plus présenter un forfait unique figé. Un "committer actif" reste un utilisateur ayant contribué récemment à un repo couvert par le produit activé.
⚠️

• Activer GHAS sur un repo ancien peut générer des centaines d'alertes existantes — prévoir une phase de triage avant de bloquer les PRs.
• Le Secret Scanning scanne l'historique complet — des secrets supprimés dans le dernier commit peuvent toujours créer des alertes.
• Secret Protection et Code Security s'activent et se facturent séparément — commencer par les repos les plus critiques et par le bon produit pour contrôler les coûts.

Pourquoi chaque paramètre compte
Branch Control Check : sans ce check, un pipeline peut déployer en production depuis n'importe quelle branch — y compris une branch de test ou de développement non reviewée. Exclusive Lock : sans ce lock, deux pipelines peuvent déployer simultanément en production, créant des race conditions, des configurations incohérentes, ou écrasant un déploiement en cours par un rollback non souhaité.

Approval timeout : un timeout explicite (ex : 4h ou 24h) évite qu'une PR de déploiement reste "en attente d'approbation" indéfiniment et bloque d'autres déploiements via l'Exclusive Lock. Après le timeout, le run est automatiquement rejeté.

Glossaire
Définitions des acronymes et concepts clés Azure DevOps
TermeDéfinitionSection
ADOAzure DevOps — plateforme Microsoft de dev collaboratif (repos, pipelines, boards, artifacts)Tout
PATPersonal Access Token — jeton d'authentification ADO lié à un compte utilisateur. Alternative aux mots de passe pour l'API REST.S1
WIFWorkload Identity Federation — mécanisme permettant à un pipeline ADO de s'authentifier sur Azure sans secret statique, via des tokens OIDC éphémères.S2
SPService Principal — identité applicative dans Entra ID. Utilisée par les Service Connections ADO (ancienne méthode — remplacer par WIF).S2
GHAS-ADOGitHub Advanced Security for Azure DevOps — fonctionnalité payante ajoutant le Secret Scanning, le Code Scanning (CodeQL) et le Dependency Scanning à ADO.S6
YAMLYAML Ain't Markup Language — format texte utilisé pour définir les pipelines ADO. Les pipelines YAML sont versionés dans le repo (vs Classic qui sont dans l'interface UI).S3
SASTStatic Application Security Testing — analyse du code source à la recherche de vulnérabilités sans exécuter le programme. Implémenté par CodeQL dans GHAS-ADO.S6
SCASoftware Composition Analysis — analyse des dépendances tierces (npm, NuGet, Maven) pour détecter les CVE connues. Implémenté par Dependency Scanning GHAS-ADO.S6, S7
CodeQLMoteur d'analyse sémantique de code développé par GitHub. Utilisé par GHAS-ADO pour le Code Scanning (SAST). Supporte : C/C++, C#, Java, JavaScript, Python, Go, Ruby, Swift.S6
RBACRole-Based Access Control — modèle de contrôle d'accès par rôles. Dans Azure : Owner / Contributor / Reader. Dans ADO : Project Administrator, Build Administrator, etc.S2, S10
CAConditional Access — politique Entra ID qui conditionne l'accès aux applications (dont ADO) à des critères : MFA, localisation IP, conformité de l'appareil.S1
CI/CDContinuous Integration / Continuous Deployment — pratique d'automatisation du build, des tests et du déploiement applicatif. Les pipelines ADO implémentent le CI/CD.S3, S9
PRPull Request — demande de merge d'une branch vers une autre. Dans ADO : mécanisme de révision du code. Les branch policies peuvent rendre les PR obligatoires sur main.S4
CODEOWNERSFichier de configuration Git définissant les propriétaires responsables de chaque partie du code. Dans ADO : utilisé avec la policy "Required reviewers" pour notifications automatiques.S4
CVECommon Vulnerabilities and Exposures — identifiant standardisé pour les vulnérabilités sécurité (ex : CVE-2021-44228 = Log4Shell). Référencé par le Dependency Scanning GHAS-ADO.S6, S7
SBOMSoftware Bill of Materials — inventaire exhaustif des composants logiciels (librairies, dépendances, versions) d'une application. Concept utile pour la supply chain logicielle, mais non documenté ici comme sortie native de GHAS-ADO.S6, S7
MSAMicrosoft Account — compte Microsoft personnel (ex : user@outlook.com). À ne pas confondre avec un compte Entra ID professionnel. Les MSA ne supportent pas toutes les politiques de sécurité ADO.S1
KQLKusto Query Language — langage de requête Microsoft pour interroger Log Analytics, Azure Data Explorer, et Sentinel. Peut être utilisé si les audit streams ADO sont envoyés vers une plateforme compatible.S8
WIWorkload Identity — identité non humaine (managed identity, service principal, federated identity) utilisée par des applications et pipelines pour s'authentifier sur Azure sans mots de passe.S2
