Incidents Unifiés & Corrélation MDE MDO MDI MDCA
Corrélation automatique des alertes cross-workload en incidents, classification, assignment et tags
ℹ️
Concept : le moteur XDR fusionne automatiquement les alertes de MDE, MDO, MDI et MDCA en incidents unifiés quand il détecte des liens (même user, même device, même fichier, timing corrélé). Un email de phishing (MDO) → clic sur un lien (MDO) → exécution d'un payload (MDE) → lateral movement (MDI) = 1 seul incident au lieu de 4 alertes dispersées.

Analogie : imaginez 4 caméras de surveillance (MDO, MDE, MDI, MDCA) filmant un cambrioleur. Sans XDR, chaque caméra génère sa propre alerte. Avec XDR, le centre de commandement reconstitue automatiquement le scénario complet du cambriolage.
🚨
Risque : sans gestion active de la file d'incidents, les vrais positifs (TP) se noient dans le bruit des faux positifs (FP). Un SOC qui ne classifie pas systématiquement ses incidents passe à côté d'attaques réelles. Selon le DBIR 2024, le délai médian de détection d'une brèche est de 204 jours — un triage rigoureux via XDR réduit ce délai à quelques heures.

Pourquoi chaque paramètre compte

💡
  • Classification TP/FP/Info : un incident non classifié ne génère aucune métrique. Impossible de mesurer le taux de FP, de tuner les détections, ou de prouver l'efficacité du SOC sans classification systématique.
  • Assignment : un incident sans propriétaire = un incident que personne ne traite. C'est le problème n°1 des SOC immatures (Ponemon 2024 : 43% des alertes ignorées faute d'ownership).
  • Tags : permettent le reporting par catégorie d'attaque. « Combien d'incidents BEC ce trimestre ? » est impossible sans tags structurés.
  • Attack Story : le graphe visuel montre la kill chain complète en un clin d'œil. Un analyste SOC qui ne consulte pas l'Attack Story passe à côté de la vision d'ensemble.
  • Linking : deux incidents apparemment distincts peuvent être les deux phases d'une même attaque. Le merge évite de traiter deux fois le même dossier.

1.2 — Procédure portail — Triage d'un incident

1
Ouvrir la file d'incidents → security.microsoft.com/incidents. Filtrer par Status = Active et Severity = High, Medium.
2
Ouvrir le premier incident non assigné. Consulter l'onglet Attack Story pour voir le graphe visuel : quels users, devices, emails et apps sont impliqués.
3
Classifier l'incident :
  • True Positive : activité malveillante confirmée → investiguer + remédier
  • Informational, expected activity : activité légitime mais suspecte (pentest, admin) → documenter
  • False Positive : fausse alerte → fermer + tuner la détection
4
Assigner l'incident à un analyste. Ajouter les tags pertinents (ex: Phishing, BEC, Ransomware, Lateral-Movement).
5
Ajouter un commentaire résumant l'analyse initiale. Indiquer les actions prises ou à prendre. Cela crée un historique audit-ready.

1.3 — Vérification via API — État de la file d'incidents

KQL — Advanced Hunting — Incidents non classifiés depuis 24h
// Trouver les incidents actifs non classifiés depuis plus de 24h
// Indicateur clé : le taux d'incidents non classifiés ne doit pas dépasser 10%
AlertInfo
| where Timestamp > ago(24h)
| summarize
    TotalAlerts = count(),
    Unclassified = countif(AttackTechniques == "")
| extend PctUnclassified = round(100.0 * Unclassified / TotalAlerts, 1)

Automated Attack Disruption MDE MDI MDCA
Containment automatique des attaques en cours : isolation device, désactivation user, blocage app OAuth
ℹ️
Concept : l'Attack Disruption est un mécanisme entièrement automatique qui détecte les attaques en cours (ransomware, BEC, adversary-in-the-middle) et prend des actions de containment sans intervention humaine. Le moteur XDR analyse la kill chain en temps réel : si un user est compromis et commence du lateral movement, XDR désactive automatiquement le compte et isole le device.

Analogie : c'est le sprinkler automatique du bâtiment. Il détecte l'incendie et agit immédiatement, avant même que les pompiers (le SOC) n'arrivent. L'incendie est contenu, pas éteint — les pompiers finissent le travail.
🚨
Risque sans disruption : un ransomware se propage en moins de 45 minutes après le premier accès (CrowdStrike 2024 : breakout time moyen = 62 min). Sans disruption automatique, le délai de réponse humain (triage + escalation + action) dépasse souvent 4h. La disruption réduit le blast radius en agissant en secondes.

2.1 — Tableau — Actions de disruption automatique

Scénario détecté Action automatique Défaut XDR Réversible ? Source Licence
Ransomware (human-operated) Contain device (réseau isolé, seul Defender communique) Actif ✅ Release depuis Action Center MDE MDE P2
BEC (Business Email Compromise) Disable user dans Entra ID (suspend sign-in) Actif ✅ Re-enable depuis Action Center MDI MDI
Adversary-in-the-middle (AiTM) Disable user + révoquer sessions Actif ✅ Re-enable depuis Action Center MDI MDI
App OAuth malveillante Disable app + révoquer consentements Actif ✅ Re-enable app MDCA MDCA
Lateral movement detecté Contain device source + disable user compromis Actif ✅ Actions indépendantes MDE + MDI E5
✅
Bonne nouvelle : l'Attack Disruption est actif par défaut dès que les produits sources sont déployés. Aucune configuration n'est nécessaire. Le score par défaut de cette section est élevé car c'est du « fire and forget ».
⚠️
Pièges à connaître :
  • Pas d'exclusion native par groupe : un compte admin stratégique peut être désactivé. Avoir un compte break-glass NON onboardé dans MDI est indispensable.
  • Action irréversible si le device est hors-ligne : un contain sur un laptop hors-réseau sera appliqué dès qu'il se reconnecte — attention au timing.
  • Disruption ≠ remédiation : le containment arrête la propagation, mais ne nettoie pas l'infection. L'investigation et le nettoyage restent manuels.

Pourquoi chaque paramètre compte

💡
  • Ransomware containment : l'isolation automatique du device empêche le chiffrement latéral. Sans disruption, un ransomware human-operated peut chiffrer l'ensemble du réseau en moins de 45 minutes (CrowdStrike 2024 : breakout time moyen = 62 min).
  • BEC & AiTM — disable user : la désactivation immédiate du compte compromis empêche l'attaquant de pivoter vers les boîtes mail, SharePoint ou Teams. Un BEC non contenu peut générer des virements frauduleux en quelques heures.
  • OAuth app blocking : une app OAuth malveillante avec consentement admin peut exfiltrer des données de toutes les boîtes mail du tenant. La révocation automatique des consentements limite le blast radius.
  • Compte break-glass : si la disruption désactive un compte Global Admin, vous perdez l'accès au tenant. Un compte break-glass non onboardé dans MDI est la seule parade.
  • Réversibilité : chaque action de disruption est annulable via l'Action Center. Connaître la procédure d'undo avant qu'un incident survienne est critique — ce n'est pas le moment d'apprendre pendant une crise.

2.3 — AIR — Automated Investigation & Response

💡
AIR ≠ Attack Disruption. L'Attack Disruption (2.1) est une réponse immédiate et préventive qui contient une menace en cours (disable user, isolate device). L'AIR est une investigation post-alerte qui analyse les preuves et propose ou exécute des remédiations (quarantaine email, suppression fichier, etc.).

Advanced Hunting (KQL) MDE MDO MDI MDCA
Requêtes KQL cross-workload pour l'investigation, le threat hunting et la création de détections
ℹ️
Concept : Advanced Hunting permet d'interroger en KQL toutes les tables de tous les produits Defender en une seule requête. Un phishing (MDO) suivi d'une exécution de payload (MDE) suivi d'un vol de credentials (MDI) : une seule requête KQL avec des join reconstitue toute la chaîne.

Analogie : Advanced Hunting est un interrogatoire structuré de tous les témoins en même temps. Au lieu d'interroger chaque caméra séparément, vous posez une question unique à toutes les sources simultanément.
⚠️
Retention des données : Advanced Hunting conserve les données 30 jours par défaut. Au-delà, les données ne sont plus interrogeables. Pour une rétention plus longue, utiliser Sentinel (90 jours à 2 ans selon la table) ou streamer vers un Storage Account via les Streaming API.

Pourquoi chaque paramètre compte

💡
  • Requêtes sauvées : sans bibliothèque, chaque investigation part de zéro. Avec 10 requêtes prêtes, un analyste passe de 30 min à 5 min pour les vérifications de base.
  • Cross-workload joins : c'est LA valeur unique d'XDR par rapport à chaque produit isolé. Joindre EmailEvents → DeviceFileEvents → IdentityLogonEvents reconstitue une kill chain complète.
  • Performance KQL : une requête mal écrite sur 30 jours peut timeout. Toujours filtrer par Timestamp en premier, utiliser project tôt pour réduire les colonnes.
  • Sessions planifiées : le hunting réactif (uniquement après incident) rate les menaces dormantes. Le hunting proactif planifié découvre les attaques avant qu'elles ne deviennent des incidents.

3.3 — Requêtes KQL prêtes à l'emploi

KQL — #1 Détecter une campagne de phishing suivie d'une exécution
// Scénario : email de phishing reçu → clic sur URL → exécution de processus suspect
// Jointure MDO → MDE sur le même user dans une fenêtre de 1h
EmailEvents
| where Timestamp > ago(7d)
| where ThreatTypes has "Phish"
| where DeliveryAction == "Delivered"
| project EmailTime = Timestamp, RecipientEmailAddress, Subject, SenderFromAddress
| join kind=inner (
    DeviceProcessEvents
    | where Timestamp > ago(7d)
    | where InitiatingProcessFileName in~ ("outlook.exe", "msedge.exe", "chrome.exe", "firefox.exe")
    | where FileName endswith ".exe" or FileName endswith ".ps1"
    | project ProcTime = Timestamp, AccountUpn, DeviceName, FileName, ProcessCommandLine
) on $left.RecipientEmailAddress == $right.AccountUpn
| where ProcTime between (EmailTime .. (EmailTime + 1h))
| project EmailTime, RecipientEmailAddress, Subject, ProcTime, DeviceName, FileName, ProcessCommandLine
KQL — #2 Lateral movement : logon anormal suivi d'accès fichier
// Scénario : authentification inhabituelle (MDI) suivie d'accès fichier sur un device différent (MDE)
IdentityLogonEvents
| where Timestamp > ago(7d)
| where LogonType == "Interactive"
| where Application != "Active Directory"
| project LogonTime = Timestamp, AccountUpn, SourceDevice = DeviceName, DestinationDeviceName, LogonType
| join kind=inner (
    DeviceFileEvents
    | where Timestamp > ago(7d)
    | where ActionType == "FileCreated"
    | where FileName endswith ".exe" or FileName endswith ".dll"
    | project FileTime = Timestamp, AccountUpn = InitiatingProcessAccountUpn, DeviceName, FileName, FolderPath
) on AccountUpn
| where FileTime between (LogonTime .. (LogonTime + 30m))
| where DeviceName != SourceDevice
| project LogonTime, AccountUpn, SourceDevice, DeviceName, FileName, FolderPath
KQL — #3 Exfiltration de données : volume anormal d'upload
// Scénario : détecter un user qui uploade un volume anormalement élevé vers un cloud externe
CloudAppEvents
| where Timestamp > ago(7d)
| where ActionType == "FileUploaded"
| summarize
    UploadCount = count(),
    TotalSizeMB = sum(tolong(RawEventData.FileSize)) / 1048576
    by AccountId, AccountDisplayName, bin(Timestamp, 1d)
| where TotalSizeMB > 500
| sort by TotalSizeMB desc
KQL — #4 Compromission de compte : connexion impossible (distance / temps)
// Scénario : même user connecté depuis 2 pays différents en moins de 2h (impossible travel)
IdentityLogonEvents
| where Timestamp > ago(7d)
| where ActionType == "LogonSuccess"
| project Timestamp, AccountUpn, Location, IPAddress
| partition hint.strategy=shuffle by AccountUpn(
    sort by Timestamp asc
    | serialize
    | extend PrevLocation = prev(Location, 1), PrevTime = prev(Timestamp, 1)
)
| where isnotempty(PrevLocation)
| where Location != PrevLocation
| where Timestamp - PrevTime < 2h
| project Timestamp, AccountUpn, PrevLocation, Location, TimeDelta = Timestamp - PrevTime
KQL — #5 Ransomware indicators : chiffrement massif de fichiers
// Scénario : un processus renomme/modifie des centaines de fichiers en quelques minutes
DeviceFileEvents
| where Timestamp > ago(1d)
| where ActionType in ("FileRenamed", "FileModified")
| summarize
    FileCount = dcount(FileName),

Custom Detection Rules MDE MDO MDI E5
Transformer les requêtes KQL en détections automatisées avec actions de réponse intégrées
ℹ️
Concept : Les Custom Detection Rules transforment une requête KQL d'Advanced Hunting en détection planifiée qui s'exécute automatiquement à intervalle régulier (1h, 3h, 12h, 24h). Quand la requête retourne des résultats, XDR crée une alerte et peut déclencher des actions de réponse automatiques (isolation, blocage, collecte).

Analogie : Advanced Hunting est un interrogatoire ponctuel ; une Custom Detection est une ronde de sécurité automatique qui patrouille toutes les X heures avec les mêmes critères.
🚨
Risque sans détections custom : les détections intégrées de Microsoft couvrent les menaces génériques, mais pas les spécificités de votre environnement. Les attaques ciblées utilisant des outils légitimes (LOLBins), des noms de domaine internes, ou des patterns propres à votre organisation ne seront pas détectées.

Pièges fréquents :
  • Lookback < fréquence : si la règle tourne toutes les 24h avec un lookback de 12h, vous perdez 12h de données à chaque cycle. Le lookback doit être ≥ à la fréquence.
  • Requête non testée : toujours exécuter la requête en Advanced Hunting AVANT de créer la règle. Vérifier le volume de résultats et le taux de faux positifs.
  • Actions sans rollback : une règle qui isole automatiquement peut isoler un serveur de production. Commencer par "Generate alert" seul, puis ajouter les actions quand la règle est stable.
  • Limite : maximum 100 règles custom par tenant. Prioriser la couverture des phases critiques de la kill chain.

Pourquoi chaque paramètre compte

💡
  • Règles actives : sans détections custom, vous dépendez uniquement des signatures Microsoft. Les attaques ciblées passent entre les mailles.
  • Fréquence : une détection à 24h signifie un délai moyen de 12h avant détection. Pour un ransomware, c'est 12h de trop. Les règles critiques doivent tourner toutes les heures.
  • Actions automatiques : la valeur d'une détection est multipliée quand elle déclenche une réponse. Isoler un device compromis en 1h au lieu d'attendre 8h qu'un analyste le fasse manuellement.
  • Mapping MITRE : permet de visualiser la couverture de détection dans la matrice ATT&CK et d'identifier les trous.

4.2 — Procédure : créer une règle custom

1
Tester la requête : security.microsoft.com → Hunting → Advanced Hunting. Coller la requête KQL, exécuter sur 7 jours. Vérifier que les résultats sont pertinents et le volume acceptable (<50 résultats/jour pour éviter l'alert fatigue).
2
Créer la règle : Cliquer Create detection rule. Nommer clairement : [CUSTOM] Phishing-to-Execution Chain. Renseigner Severity, MITRE technique, Description avec le scénario visé.
3
Mapper les entités : associer les colonnes de la requête aux entités XDR : DeviceName → Device, AccountUpn → User, FileName → File, IPAddress → IP. Ceci permet l'enrichissement automatique dans l'alerte.
4
Configurer la fréquence : Fréquence = 1h pour les menaces critiques (ransomware, lateral movement). Lookback = 2h minimum pour cette fréquence. Règles moins critiques : 12h/24h.
5
Actions (prudence) : commencer avec "Generate alert" uniquement pendant 2 semaines. Si le taux FP est <5%, ajouter progressivement les actions : Isolate device, Collect investigation package, Block file.

  • Playbooks : la valeur de Sentinel est dans l'orchestration. Ouvrir un ticket ServiceNow, enrichir une IP dans VirusTotal, isoler un device via XDR — le tout déclenché automatiquement.

Script All-in-One — Audit XDR
Toutes les requêtes de vérification consolidées — exécuter dans Advanced Hunting + PowerShell
ℹ️
Mode d'emploi : exécuter ces requêtes dans l'ordre pour auditer la posture XDR. Les requêtes KQL s'exécutent dans Advanced Hunting (security.microsoft.com). Les commandes PowerShell nécessitent le module Microsoft.Graph avec les permissions SecurityEvents.Read.All.

Phase 1 — Incidents & Disruption (S1-S2)

KQL — 1.1 Incidents récents et distribution
// Phase 1.1 : Distribution des incidents par sévérité et statut
AlertInfo
| where Timestamp > ago(30d)
| summarize
    AlertCount = count(),
    Sources = dcount(DetectionSource)
    by Severity
| sort by AlertCount desc
KQL — 1.2 Attack Disruption : actions déclenchées
// Phase 1.2 : Vérifier les disruptions automatiques
AlertInfo
| where Timestamp > ago(30d)
| where DetectionSource == "AutomaticAttackDisruption"
| summarize Count = count() by Title, Severity
| sort by Count desc

Phase 2 — Hunting & Détections (S3-S4)

KQL — 2.1 Tables Advanced Hunting utilisées
// Phase 2.1 : Vérifier que les tables cross-workload ont des données
union
    (DeviceEvents | where Timestamp > ago(1d) | summarize Rows=count() | extend Table="DeviceEvents"),
    (EmailEvents | where Timestamp > ago(1d) | summarize Rows=count() | extend Table="EmailEvents"),
    (IdentityLogonEvents | where Timestamp > ago(1d) | summarize Rows=count() | extend Table="IdentityLogonEvents"),
    (CloudAppEvents | where Timestamp > ago(1d) | summarize Rows=count() | extend Table="CloudAppEvents")
| project Table, Rows
KQL — 2.2 Custom Detections : alertes générées
// Phase 2.2 : Vérifier les alertes générées par les Custom Detections
AlertInfo
| where Timestamp > ago(30d)
| where DetectionSource == "CustomDetection"
| summarize
    AlertCount = count(),
    LastFired = max(Timestamp)
    by Title, Severity
| sort by AlertCount desc

Phase 3 — Couverture & Threat Intel (S5-S6)

KQL — 3.1 Actions de remédiation récentes
// Phase 3.1 : Historique des actions Action Center
AlertEvidence
| where Timestamp > ago(30d)
| where EntityType in ("Machine", "User", "File")
| summarize Count = count() by EntityType, RemediationStatus
| sort by Count desc
KQL — 3.2 Couverture MITRE ATT&CK des alertes
// Phase 3.2 : Répartition des alertes par technique MITRE
AlertInfo
| where Timestamp > ago(30d)
| where AttackTechniques != ""
| mv-expand Technique = split(AttackTechniques, ",")
| summarize AlertCount = count() by tostring(Technique)
| sort by AlertCount desc
| take 20
KQL — 3.3 Vulnérabilités avec exploits connus
// Phase 3.3 : Vulnérabilités exploitées activement
DeviceTvmSoftwareVulnerabilities
| where IsExploitAvailable == true
| summarize DeviceCount = dcount(DeviceName)
    by CveId, SoftwareName, VulnerabilitySeverityLevel
| sort by DeviceCount desc
| take 20

PowerShell — 4.1 Vérifier les settings XDR
# Phase 4 : Audit des paramètres XDR via Graph Security API
# Prérequis : Connect-MgGraph -Scopes "SecurityEvents.Read.All"

# 4.1 Incidents récents
$incidents = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/security/incidents?`$top=10&`$orderby=createdDateTime desc"

Write-Host "=== 10 derniers incidents ==="
$incidents.value | Select-Object displayName, severity, status, createdDateTime |
    Format-Table -AutoSize

# 4.2 Alertes par source de détection
$alerts = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/security/alerts_v2?`$top=100"

Write-Host "=== Sources de détection ==="
$alerts.value | Group-Object detectionSource |
    Select-Object Name, Count |
    Sort-Object Count -Descending |
    Format-Table -AutoSize

# Note : endpoint beta (pas encore disponible en v1.0)
$rules = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/security/rules/detectionRules"

Write-Host "=== Custom Detection Rules ==="
Write-Host "Nombre de règles : $($rules.value.Count) / 100 max"
$rules.value | Select-Object displayName, isEnabled, lastRunDateTime |
