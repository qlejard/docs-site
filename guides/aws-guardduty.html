💡
Pourquoi chaque paramètre compte :
  • Activation multi-région : un attaquant cible spécifiquement les régions où GuardDuty n'est pas actif pour échapper à la détection. CIS AWS Foundations Benchmark v7.0.0, contrôle 4.1 : « Ensure GuardDuty is enabled in all regions ».
  • Organizations Delegated Admin : sans admin délégué, chaque compte doit être géré individuellement — les nouveaux comptes ne sont pas protégés automatiquement. AWS Well-Architected Framework, Security Pillar §SEC 1.
  • Auto-Enable Members = ALL : sans auto-enable, tout nouveau compte ajouté à l'Organization est un angle mort — pas de détection jusqu'à activation manuelle.
  • CloudTrail Management Events : source fondamentale — détecte les appels API non autorisés (ex: CreateUser, AttachPolicy depuis une IP suspecte).
  • VPC Flow Logs : détecte le trafic réseau anormal (communication avec des IPs de C&C connues, minage crypto, exfiltration). Source la plus volumineuse = la plus coûteuse, mais aussi la plus riche en détections.
  • DNS Query Logs : détecte les requêtes DNS vers des domaines malveillants (C&C, DGA) sans nécessiter de VPC Flow Logs.
  • Publishing Frequency = 15 min : le défaut de 6 heures crée un délai inacceptable pour la réponse aux incidents. Un attaquant peut exfiltrer des données ou pivoter dans l'infrastructure en moins de 15 minutes (MITRE ATT&CK: moyenne 30 min pour le mouvement latéral).
⚠️
Pièges classiques :
  • Régions non utilisées : GuardDuty doit être actif même dans les régions que vous n'utilisez pas — un attaquant y créera des ressources spécifiquement pour échapper à la détection. Le coût dans une région inactive est quasi nul (pas de trafic).
  • Publishing Frequency par défaut = 6h : ce paramètre n'est jamais changé automatiquement — il faut une action explicite via CLI ou console pour le réduire à 15 min.
  • GuardDuty ne dépend PAS de l'activation de CloudTrail ou VPC Flow Logs : il utilise ses propres copies indépendantes des logs. Pas besoin de configurer CloudTrail séparément pour GuardDuty.
  • Suspension vs désactivation : StopMonitoringMembers suspend la surveillance mais conserve les findings. DeleteDetector supprime tout irréversiblement — ne jamais utiliser sans raison valable.

AWS CLI — Activer GuardDuty dans toutes les régions
# Variables à adapter
PROFILE="security-admin"

# Lister toutes les régions disponibles
aws ec2 describe-regions --query "Regions[].RegionName" --output text --profile $PROFILE --region eu-west-1 | tr '\t' '\n' | while read region; do
    echo "[+] Activation GuardDuty dans $region..."
    aws guardduty create-detector \
        --enable \
        --finding-publishing-frequency FIFTEEN_MINUTES \
        --profile $PROFILE \
        --region $region \
        --output json 2>/dev/null || echo "  Déjà activé dans $region"
done

S3 Protection, EKS, Lambda, RDS, Malware, EC2 Runtime Monitoring — 🔴 Critique (18 pts)
Réf. AWS Foundational Security Best Practices (FSBP)
🚨
Scénario concret : Un bucket S3 public non surveillé est utilisé comme vecteur d'exfiltration — un attaquant utilise des appels GetObject massifs depuis une IP anonyme pour télécharger des données sensibles. Sans S3 Protection activé, GuardDuty ne détecte pas ces accès anormaux aux données.

💡
Pourquoi chaque Protection Plan compte :
  • S3 Protection : détecte les accès anomaux aux buckets S3 (exfiltration, accès depuis des IPs suspectes, désactivation de chiffrement). FSBP contrôle GuardDuty.1 : « GuardDuty should be enabled » inclut S3 Protection.
  • EKS Audit Logs : détecte les activités suspectes Kubernetes (pods privilégiés, accès API Kubernetes depuis TOR, conteneurs avec accès host network). Essentiel si vous utilisez Amazon EKS.
  • EKS/EC2 Runtime Monitoring : détection au niveau OS des process suspects (reverse shell, crypto-mining, mouvement latéral). Nécessite l'installation d'un agent (auto-managed ou manuel via SSM).
  • Lambda Network Activity : détecte si une fonction Lambda compromise communique avec des serveurs C&C ou effectue du DNS tunneling.
  • RDS Login Activity : détecte les tentatives de brute force et les connexions suspectes sur Amazon Aurora.
  • Malware Protection : scan antimalware des volumes EBS attachés aux instances suspectes — déclenché automatiquement quand GuardDuty détecte un finding EC2.
⚠️
Pièges classiques :
  • S3 Protection ≠ CloudTrail S3 Data Events : GuardDuty analyse ses propres copies des S3 Data Events. Il n'est pas nécessaire d'activer CloudTrail S3 Data Event Logging séparément pour GuardDuty (mais recommandé pour l'audit indépendant).
  • Runtime Monitoring nécessite un agent : contrairement aux autres Protection Plans qui sont sans agent, le Runtime Monitoring EKS/EC2 déploie un agent via SSM. Vérifiez la couverture dans Runtime Monitoring > Coverage.
  • Malware Protection crée des snapshots EBS : chaque scan copie le volume EBS. Les snapshots sont automatiquement supprimés après le scan, mais des règles SCP restrictives peuvent bloquer la création de snapshots.
  • RDS Login Activity limité à Aurora : ne couvre pas RDS MySQL/PostgreSQL classique (non Aurora). Vérifiez la compatibilité de votre moteur.

Pièges classiques :
  • Trusted IP Lists ne supprime PAS les findings : elles empêchent la génération de nouveaux findings pour ces IPs. Les findings existants restent. Maximum 1 trusted IP list, 2000 IPs max.
  • Threat Intel Lists sont ADDITIVES : elles complètent (ne remplacent pas) les feeds AWS intégrés. Max 6 listes, format STIX (un IP/CIDR/domaine par ligne dans un fichier texte hébergé sur S3).
  • Finding updates vs nouvelles findings : GuardDuty agrège les occurrences répétées. Un finding « Updated » avec un updatedAt récent peut signifier une attaque en cours, pas un ancien problème.
  • Archiver ≠ Résoudre : archiver un finding le masque de la vue console mais ne le supprime pas. Il reste accessible via l'API et réapparaîtra si l'activité reprend.

5.3 — Playbook EC2 Compromise (Python Lambda)

Python — Lambda EC2 quarantaine automatique
import json, boto3, os

ec2 = boto3.client('ec2')
sns = boto3.client('sns')
QUARANTINE_SG = os.environ['QUARANTINE_SG_ID']
SNS_TOPIC     = os.environ['SNS_TOPIC_ARN']

def handler(event, context):
    detail    = event['detail']
    finding   = detail['type']
    severity  = detail['severity']
    instance  = detail['resource']['instanceDetails']['instanceId']

    # 1. Snapshot EBS pour forensics
    volumes = ec2.describe_instances(
        InstanceIds=[instance]
    )['Reservations'][0]['Instances'][0]['BlockDeviceMappings']
    for vol in volumes:
        vol_id = vol['Ebs']['VolumeId']
        ec2.create_snapshot(
            VolumeId=vol_id,
            Description=f'Forensic snapshot - {finding} - {instance}',
            TagSpecifications=[{'ResourceType': 'snapshot',
                'Tags': [{'Key': 'forensic', 'Value': 'true'}]}]
        )

    # 2. Quarantaine — remplacer tous les SG par le SG quarantaine
    ec2.modify_instance_attribute(
        InstanceId=instance,
        Groups=[QUARANTINE_SG]
    )

    # 3. Tag de quarantaine
    ec2.create_tags(
        Resources=[instance],
        Tags=[
            {'Key': 'quarantine', 'Value': 'true'},
            {'Key': 'quarantine-reason', 'Value': finding}
        ]
    )

    # 4. Notification SOC
    sns.publish(
        TopicArn=SNS_TOPIC,
        Subject=f'[GuardDuty] EC2 {instance} quarantiné — {finding}',
        Message=json.dumps(detail, indent=2, default=str)
    )
    return {'statusCode': 200, 'instance': instance, 'action': 'quarantined'}

SNS_TOPIC_ARN="arn:aws:sns:eu-west-1:123456789012:security-guardduty"
EXPORT_BUCKET="my-guardduty-findings-export"
KMS_KEY_ARN="arn:aws:kms:eu-west-1:123456789012:key/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
TRUSTED_IPS_URL="s3://my-security-config/trusted-ips.txt"

echo "══════ [S1] Activation GuardDuty ══════"
DETECTOR_ID=$(aws guardduty create-detector \
    --enable \
    --finding-publishing-frequency FIFTEEN_MINUTES \
    --query DetectorId --output text \
    --profile "$PROFILE" --region "$REGION" 2>/dev/null || \
    aws guardduty list-detectors --query 'DetectorIds[0]' --output text \
    --profile "$PROFILE" --region "$REGION")
echo "Detector ID: $DETECTOR_ID"

echo "══════ [S2] Activation de toutes les Protection Plans ══════"
aws guardduty update-detector \
    --detector-id "$DETECTOR_ID" \
    --enable \
    --finding-publishing-frequency FIFTEEN_MINUTES \
    --features '[
        {"Name":"S3_DATA_EVENTS","Status":"ENABLED"},
        {"Name":"EKS_AUDIT_LOGS","Status":"ENABLED"},
        {"Name":"EBS_MALWARE_PROTECTION","Status":"ENABLED"},
        {"Name":"RDS_LOGIN_EVENTS","Status":"ENABLED"},
        {"Name":"LAMBDA_NETWORK_LOGS","Status":"ENABLED"},
        {"Name":"EKS_RUNTIME_MONITORING","Status":"ENABLED","AdditionalConfiguration":[{"Name":"EKS_ADDON_MANAGEMENT","Status":"ENABLED"}]},
        {"Name":"EC2_RUNTIME_MONITORING","Status":"ENABLED","AdditionalConfiguration":[{"Name":"EC2_AGENT_MANAGEMENT","Status":"ENABLED"}]},
        {"Name":"RUNTIME_MONITORING","Status":"ENABLED","AdditionalConfiguration":[{"Name":"EKS_ADDON_MANAGEMENT","Status":"ENABLED"},{"Name":"ECS_FARGATE_AGENT_MANAGEMENT","Status":"ENABLED"},{"Name":"EC2_AGENT_MANAGEMENT","Status":"ENABLED"}]}
    ]' \
    --profile "$PROFILE" --region "$REGION"

echo "══════ [S3] Trusted IP List ══════"
aws guardduty create-ip-set \
    --detector-id "$DETECTOR_ID" \
    --name "Corporate-IPs" \
    --format TXT \
    --location "$TRUSTED_IPS_URL" \
    --activate \
    --profile "$PROFILE" --region "$REGION" || echo "IP Set déjà existant"

echo "══════ [S4/S5] EventBridge → SNS (notifications) ══════"
aws events put-rule \
    --name "GuardDuty-HighSeverity" \
    --event-pattern '{"source":["aws.guardduty"],"detail-type":["GuardDuty Finding"],"detail":{"severity":[{"numeric":[">=",7]}]}}' \
    --profile "$PROFILE" --region "$REGION"

aws events put-targets \
    --rule "GuardDuty-HighSeverity" \
    --targets '[{"Id":"sns-target","Arn":"'"$SNS_TOPIC_ARN"'"}]' \
    --profile "$PROFILE" --region "$REGION"

echo "══════ [S6] Export Findings → S3 ══════"
aws guardduty create-publishing-destination \
    --detector-id "$DETECTOR_ID" \
    --destination-type S3 \
    --destination-properties '{"DestinationArn":"arn:aws:s3:::'"$EXPORT_BUCKET"'","KmsKeyArn":"'"$KMS_KEY_ARN"'"}' \
    --profile "$PROFILE" --region "$REGION" || echo "Publishing destination déjà configurée"

echo "══════ [S9] CloudWatch Alarm anti-désactivation ══════"
aws logs put-metric-filter \
    --log-group-name "CloudTrail/DefaultLogGroup" \
    --filter-name "GuardDutyDisableAttempt" \
    --filter-pattern '{ ($.eventName = "DeleteDetector") || ($.eventName = "UpdateDetector") || ($.eventName = "DisableOrganizationAdminAccount") }' \
    --metric-transformations '[{"metricName":"GuardDutyDisableAttempt","metricNamespace":"SecurityMonitoring","metricValue":"1"}]' \
    --profile "$PROFILE" --region "$REGION"

aws cloudwatch put-metric-alarm \
    --alarm-name "GuardDuty-Disable-Attempt" \
    --metric-name GuardDutyDisableAttempt \
    --namespace SecurityMonitoring \
    --statistic Sum --period 300 --threshold 1 \
    --comparison-operator GreaterThanOrEqualToThreshold \
    --evaluation-periods 1 \
    --alarm-actions "$SNS_TOPIC_ARN" \
    --profile "$PROFILE" --region "$REGION"

echo "══════ Vérification finale ══════"
aws guardduty get-detector --detector-id "$DETECTOR_ID" \
    --profile "$PROFILE" --region "$REGION" | \
    python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Status: {d[\"Status\"]}'); [print(f'  {f[\"Name\"]}: {f[\"Status\"]}') for f in d.get('Features',[])]"

echo "══════ ✅ Hardening terminé ══════"
