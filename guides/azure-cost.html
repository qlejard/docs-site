Solutions écartées

💡
Pourquoi Azure Cost Management natif et pas un outil tiers ?
  • Infracost (IaC cost estimation) → scope différent : pré-déploiement, pas post-déploiement. Complémentaire, pas substitut.
  • CloudHealth / Apptio → outils tiers multi-cloud souvent coûteux. Azure Cost Management est gratuit et intégré nativement.
  • OpenCost / Kubecost → spécifiques Kubernetes. Azure Cost Management couvre l'intégralité du scope Azure.
  • FinOps Hub (open source) → complémentaire, mentionné en S8 comme extension Power BI.
Source : FinOps Foundation, State of FinOps 2024 — 78 % des organisations utilisent l'outil natif du cloud provider comme première ligne de visibilité.

Pourquoi chaque paramètre compte :
  • Actual vs Amortized : « Actual » montre la charge comptable brute. « Amortized » répartit le coût des Reservations/Savings Plans sur la durée — indispensable pour comparer le coût réel par workload. Sans Amortized, une subscription supportant une RI apparaît artificiellement chère au mois d'achat.
  • Granularité Daily : en Monthly, un pic de consommation à J+5 est invisible jusqu'à la fin du mois. En Daily, on détecte les anomalies en 24-48 h.
  • Saved Views : sans vues sauvées, chaque consultation du portail nécessite de reconfigurer les filtres — perte de temps (2-5 min) et risque d'oublier un scope.
  • Dimensions : sans regroupement par Service ou Tag, vous voyez un total plat inutilisable pour identifier la source d'un surcoût. C'est la différence entre « vous avez un dépassement global » et « AKS-prod domine le spend, SQL-dev suit, et un RG de test dérive ».
  • Exports planifiés : sans export, les données détailleées ne sont disponibles que via le portail. L'export quotidien vers un Storage Account permet l'historisation longue durée (>13 mois) et l'analyse Power BI/Excel.
⚠️
Piège classique — Les 3 erreurs les plus fréquentes :
  • Scope trop large : analyser au niveau Billing Account sans filtre → données noyées, temps de chargement >30 s. Toujours commencer par subscription puis drill-down.
  • Ignorer les taxes/support : Cost Analysis n'inclut pas les frais de support ni les taxes par défaut. La facture finale peut différer de 8-20 % selon le pays et le plan support.
  • Confondre « Cost » et « Usage » : une VM éteinte (« deallocated ») n'a plus de coût compute, mais les disques continuent d'être facturés. Vérifier les composantes : Compute, Storage, Network séparément.

1.2 — Configuration

Chemin portail

Azure Portal → Cost Management → Cost analysis

  1. Sélectionner le Scope (subscription ou resource group)
  2. Passer en vue « Daily costs » (menu déroulant en haut)
  3. Cliquer Group by → choisir ServiceName
  4. Cliquer 💾 Save → nommer la vue (ex : « Daily-by-Service »)
  5. Répéter pour une vue par ResourceGroup et une vue par Tag

Az CLI

Az CLI — Requête Cost Analysis
# Coûts des 30 derniers jours, groupés par Service
az costmanagement query --type ActualCost \
    --scope "subscriptions/$SUBSCRIPTION_ID" \
    --timeframe MonthToDate \
    --dataset-grouping '[{"type":"Dimension","name":"ServiceName"}]' \
    --dataset-aggregation '{"totalCost":{"name":"Cost","function":"Sum"}}'

# Top 10 des Resource Groups les plus coûteux
az costmanagement query --type ActualCost \
    --scope "subscriptions/$SUBSCRIPTION_ID" \
    --timeframe MonthToDate \
    --dataset-grouping '[{"type":"Dimension","name":"ResourceGroup"}]' \
    --dataset-aggregation '{"totalCost":{"name":"Cost","function":"Sum"}}' \
    --query "sort_by(@, &[1]) | reverse(@) | [:10]"

# Créer un export planifié (quotidien) vers un Storage Account
az costmanagement export create \
    --name "daily-cost-export" \
    --scope "subscriptions/$SUBSCRIPTION_ID" \
    --type ActualCost \
    --timeframe MonthToDate \
    --schedule-recurrence Daily \
    --schedule-status Active \
    --storage-account-id "$STORAGE_ACCOUNT_ID" \
    --storage-container "cost-exports"

Pourquoi chaque paramètre compte :
  • Budget à 0 : Azure ne crée aucun budget par défaut. Tant qu'aucun budget n'est créé, aucune alerte ne sera envoyée, même si la consommation explose.
  • 3 seuils (50/80/100 %) : le seuil à 50 % permet un premier checkpoint mi-mois. Le 80 % donne 4-6 jours pour réagir. Le 100 % confirme le dépassement. Sans le seuil à 50 %, la première alerte arrive souvent trop tard.
  • Forecast alert : il détection un dépassement avant qu'il ne se produise en extrapolant la tendance des 30 derniers jours. C'est la différence entre éteindre l'incendie et l'empêcher.
  • Action Group : sans Action Group, l'alerte passe uniquement par email. Avec un Action Group, on peut notifier un canal Teams (webhook), déclencher un Logic App, ou appeler une Azure Function qui arrête automatiquement les ressources non-production.
  • Budget filtré : un budget global de 10 000 €/mois ne dit pas qui dépense. Un budget par RG ou par tag CostCenter permet l'accountability par équipe.
⚠️
Pièges classiques :
  • Budget ≠ limite dure : un budget Azure est informatif uniquement. Il ne bloque pas la consommation. Même à 200 % du budget, les ressources continuent de tourner. Pour une action corrective, il faut configurer une automation (Logic App ou Azure Function) déclenchée par l'alerte.
  • Email dans les spams : les alertes viennent de azure-noreply@microsoft.com. Beaucoup de filtres anti-spam les bloquent. Vérifier la whitelist ou utiliser un Action Group avec webhook Teams.
  • Délai de 8-24 h : les données de coût Azure ont un délai de publication de 8 à 24 h. Un budget à 100 % peut se déclencher alors que la consommation réelle est déjà à 110 %. Prévoir une marge de 10-20 %.

Az CLI — Création d'un budget avec alertes

SUBSCRIPTION_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
BUDGET_AMOUNT=5000
ALERT_EMAIL="finops-team@contoso.com"

    --budget-name "budget-sub-prod-monthly" \

    --notifications '{
        "alert50": {"enabled":true, "operator":"GreaterThanOrEqualTo", "threshold":50, "contactEmails":["'$ALERT_EMAIL'"]},

    --name "ag-cost-alerts" \

Pourquoi chaque paramètre compte :

  • Advisor Score : une métrique /100 qui reflète le pourcentage de recommandations appliquées. Si vous êtes à 30 %, cela signifie que 70 % des économies identifiées ne sont pas appliquées. KPI FinOps cible : ≥ 80 %.
  • Advisor Alerts : sans alert rule, il faut consulter le portail manuellement. Avec une alert rule, chaque nouvelle recommandation coût déclenche une notification push — plus de recommandations « perdues ».

⚠️
Pièges classiques :
  • « Postpone » ≠ « Dismiss » : Postpone reporte la recommandation pour une durée choisie (utile si vous avez un plan). Dismiss la supprime définitivement. Beaucoup d'admins « Dismiss » sans réaliser que la recommandation ne reviendra plus.
  • Rightsize ≠ Shutdown : une VM à 5 % de CPU peut être légitime (load balancer, DNS, jump server). Advisor recommande d'abord le rightsizing (réduire la taille SKU), pas le shutdown. Vérifier le contexte avant d'appliquer.
  • Recommandations de Reservation : Advisor suggère des achats de RI basés sur 30 jours d'historique. Si vos workloads sont volatils (batch, seasonal), une RI 3 ans pourrait ne pas être rentable. Valider avec ≥ 3 mois d'historique.

4.2 — Arbre de décision : Reservation vs Savings Plan vs PAYG

🎯
Règle simplifiée :
  1. Workload stable, même SKU > 12 mois ? → Reservation (RI) — remise potentiellement la plus profonde, mais moins flexible
  2. Workload stable, mais SKU/région susceptible de changer ? → Savings Plan — remise variable selon le produit, avec plus de flexibilité
  3. Workload volatile, batch, seasonal, dev/test ? → PAYG + auto-shutdown + Spot VMs (voir S6)
  4. Workload mixte ? → RI pour la base + Savings Plan pour le variable — les deux se cumulent
💡
Pourquoi chaque paramètre compte :
  • Scope « Shared » : une RI en scope « single subscription » ne s'applique qu'à cette subscription. Si la VM est migrée vers une autre subscription, la RI est gaspillée. En « shared », l'application est automatique sur toutes les subscriptions du billing account.
  • Instance size flexibility : avec cette option, une RI D4s_v5 s'applique aussi à une D2s_v5 (au prorata). Cela réduit le risque de sous-utilisation si vous downsizez une VM.
  • Utilization < 80 % = gaspillage : une RI non utilisée coûte quand même. Si l'utilization tombe en dessous de 80 %, il faut soit réassigner, soit échanger la RI (exchange policy), soit la retourner (dans les 7 jours si eligible).
  • Paiement mensuel : le paiement upfront donne une légère remise, mais le paiement mensuel est identique en coût total pour la plupart des services. Commencer par mensuel réduit le risque financier.
⚠️

  • Services non couverts : les RI ne couvrent pas le réseau (bandwidth, VPN Gateway), ni les services consommés (Logic Apps, Functions Consumption). Vérifier la liste des services éligibles.

# --- Audit FinOps KPIs ---
# Prérequis : modules Az.CostManagement, Az.ResourceGraph

# 1. Tag coverage (Resource Graph)
$query = @'
resources
| extend hasTag = isnotnull(tags['CostCenter']) and tags['CostCenter'] != ''
| summarize Total=count(), Tagged=countif(hasTag)
| extend CoveragePct=round(100.0*Tagged/Total,1)
'@
$tagCoverage = Search-AzGraph -Query $query
Write-Host "Tag coverage: $($tagCoverage.CoveragePct) % (cible ≥ 90 %)"

# 2. Advisor Score (Cost category)
$advisor = Get-AzAdvisorRecommendation | Where-Object { $_.Category -eq 'Cost' }
Write-Host "Recommandations Advisor Cost ouvertes: $($advisor.Count)"

# 3. Budget variance (mois en cours)
$budgets = Get-AzConsumptionBudget
foreach ($b in $budgets) {
    $variance = [math]::Round(($b.CurrentSpend.Amount / $b.Amount - 1) * 100, 1)
    Write-Host "Budget '$($b.Name)': $($b.CurrentSpend.Amount) / $($b.Amount) ($variance %)"
}

# 4. Orphaned disks (waste)
$orphaned = Search-AzGraph -Query "resources | where type == 'microsoft.compute/disks' | where properties.diskState == 'Unattached'"
Write-Host "Disques orphelins: $($orphaned.Count)"

Script All-in-One — Audit & Optimisation Coûts
Script exécutable pour auditer et appliquer les best practices de gestion des coûts
⚠️
Prérequis : Az CLI installé, az login exécuté, extensions costmanagement et resource-graph installées.
Bash — Script d'audit FinOps complet
#!/bin/bash
# =============================================================================
# AUDIT FINOPS AZURE — Script All-in-One
# Vérifie la mise en place des best practices de chaque section du guide
# =============================================================================

set -euo pipefail
SUBSCRIPTION_ID=$(az account show --query id -o tsv)
echo "=== AUDIT FINOPS — Subscription: $SUBSCRIPTION_ID ==="
SCORE=0

# --- S1 : Cost Analysis (15 pts) ---
echo "\n[S1] Cost Analysis"
EXPORTS=$(az costmanagement export list --scope "subscriptions/$SUBSCRIPTION_ID" --query "length([])" -o tsv 2>/dev/null || echo 0)
if [ "$EXPORTS" -gt 0 ]; then
    echo "  ✅ $EXPORTS export(s) configuré(s)"
    SCORE=$((SCORE + 8))
else
    echo "  ❌ Aucun export planifié"
fi

# --- S2 : Budgets (15 pts) ---
echo "\n[S2] Budgets & Alertes"
BUDGETS=$(az consumption budget list --query "length([])" -o tsv 2>/dev/null || echo 0)
if [ "$BUDGETS" -gt 0 ]; then
    echo "  ✅ $BUDGETS budget(s) défini(s)"
    SCORE=$((SCORE + 10))
else
    echo "  ❌ Aucun budget configuré"
fi

# --- S3 : Advisor Cost (12 pts) ---
echo "\n[S3] Azure Advisor Cost"
ADVISOR_RECS=$(az advisor recommendation list --category Cost --query "length([])" -o tsv 2>/dev/null || echo 0)
echo "  ℹ️  $ADVISOR_RECS recommandation(s) Cost en attente"
if [ "$ADVISOR_RECS" -eq 0 ]; then
    SCORE=$((SCORE + 12))
    echo "  ✅ Toutes les recommandations traitées"
else
    SCORE=$((SCORE + 5))
    echo "  ⚠️  Recommandations à traiter"
fi

# --- S6 : Orphaned Resources ---
echo "\n[S6] Resource Optimization"
ORPHAN_DISKS=$(az disk list --query "length([?diskState=='Unattached'])" -o tsv 2>/dev/null || echo 0)
ORPHAN_IPS=$(az network public-ip list --query "length([?ipConfiguration==null])" -o tsv 2>/dev/null || echo 0)
echo "  Disques orphelins: $ORPHAN_DISKS | IPs orphelines: $ORPHAN_IPS"
if [ "$ORPHAN_DISKS" -eq 0 ] && [ "$ORPHAN_IPS" -eq 0 ]; then
    SCORE=$((SCORE + 8))
    echo "  ✅ Aucune ressource orpheline"
else
    SCORE=$((SCORE + 3))
    echo "  ⚠️  Nettoyage recommandé"
fi

# --- S7 : Tagging ---
echo "\n[S7] Tagging FinOps"
UNTAGGED=$(az graph query -q "resources | where isnull(tags['CostCenter']) or tags['CostCenter'] == '' | count" --query "data[0].Count" -o tsv 2>/dev/null || echo "N/A")
TOTAL=$(az graph query -q "resources | count" --query "data[0].Count" -o tsv 2>/dev/null || echo "N/A")
echo "  Ressources sans CostCenter: $UNTAGGED / $TOTAL"

# --- Résultat ---
echo "\n========================================"
echo "SCORE FINOPS ESTIMÉ : $SCORE /100"
echo "========================================"
echo "Cible Quick Win : 50 /100"
echo "Cible Avancé  : 82 /100"
✅
