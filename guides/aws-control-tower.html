📋
Ce guide est un référentiel complet AWS Control Tower couvrant le déploiement d'une Landing Zone, la gouvernance multi-compte, les guardrails et l'Account Factory. Il peut être utilisé tel quel pour auditer un environnement existant ou planifier un déploiement greenfield. Analogie clé : Control Tower = "promoteur immobilier sécurisé" qui construit des logements standardisés (comptes AWS) selon des normes de sécurité strictes.

🚨
Scénario : Un DevOps crée un compte AWS de test avec des droits admin. Il oublie de configurer CloudTrail, MFA et les SCP. 6 mois plus tard, une clé IAM est leakée sur GitHub. L'attaquant crée 50 instances EC2 dans 10 régions pour du crypto-mining : facture de 80 000 $ en 48h, aucune alerte, aucune trace.
Analogie : La Landing Zone = "résidence sécurisée" : le complexe immobilier avec ses règles de copropriété. Sans elle, chaque logement (compte AWS) est construit par un artisan différent, sans normes.

1.1 — Prérequis avant déploiement

⚠️
Prérequis bloquants : (1) AWS Control Tower peut être lancé dans une organisation AWS Organizations nouvelle ou existante, mais les quotas doivent permettre la création d'au moins deux comptes supplémentaires. (2) Prévoir deux adresses e-mail uniques pour Log Archive et Audit, sauf si vous réutilisez des comptes existants compatibles. (3) Si IAM Identity Center est déjà configuré, la région home de Control Tower doit correspondre à la région IAM Identity Center ; si IAM Identity Center est en us-east-1, Control Tower réutilise cette instance. (4) Les endpoints STS doivent être activés dans toutes les régions gouvernées avant le lancement.
Prérequis Défaut AWS ⚡ Quick Win 🔒 Requis pour CT 🎯 Best Practice Source
Management Account MFA root Non activé MFA hardware Obligatoire MFA hardware (FIDO2) + backup IAM Guide S2
AWS Organizations activé Non activé Activer All Features All Features obligatoire All Features (pas Consolidated Billing seul) AWS docs, CT prerequisite
Région home Control Tower Non définie Choisir 1 région principale Immuable après setup us-east-1 ou eu-west-1 selon géographie AWS CT User Guide
KMS Customer Managed Key AWS-managed key CMK dédiée Control Tower Recommandé CMK dans chaque région gérée CIS AWS 3.x, AWS Security Best Practices

Limites de service (Service Quotas) Par défaut AWS Vérifier quotas Organizations Max accounts quota ≥ nb comptes ciblés Demander augmentation à l'avance (AWS Support) AWS Service Quotas
💡
Pourquoi chaque prérequis compte :
• MFA root obligatoire : Le Management Account contrôle toute l'organisation. Compromis = toute votre infrastructure AWS compromise. AWS exige MFA root depuis 2024 pour tout nouveau compte.
• All Features Organizations : Le mode "Consolidated Billing" ne permet pas les SCPs — sans SCPs, les guardrails préventifs de Control Tower ne peuvent pas fonctionner.
• Région home immuable : Changer la région home après déploiement nécessite de tout détruire et recréer. Choisir soigneusement avant le premier déploiement.
• CMK Control Tower : Chiffrement des buckets S3 du Log Archive et des trails CloudTrail. Sans CMK, les logs utilisent la clé AWS-managed, non révocable en cas d'incident.

# Console > Services > Management & Governance > AWS Control Tower
# URL directe : https://console.aws.amazon.com/controltower/

# ÉTAPE 2 — Set up landing zone
# Cliquer "Set up landing zone"
# Choisir la Home Region (immuable après déploiement)

# ÉTAPE 3 — Configure OUs
# Security OU : contient Log Archive + Audit (nommé "Security" par défaut)
# Sandbox OU : contient les comptes de développement (renommable)

# ÉTAPE 4 — Shared accounts
# Log Archive Account email : aws-log-archive@company.com
# Audit Account email       : aws-audit@company.com

# ÉTAPE 5 — Configure CloudTrail + KMS
# Choisir si AWS Control Tower gère le trail d'organisation CloudTrail
# ou si vous conservez un modèle CloudTrail hors CT ; vérifier ensuite le chiffrement

# ÉTAPE 6 — IAM Identity Center
# Control Tower active ou réutilise IAM Identity Center selon l'existant

# Durée du déploiement : 30-60 minutes
☀️ AWS CLI — Vérification post-déploiement Landing Zone 
# Vérifier le statut de la Landing Zone
LANDING_ZONE_ARN=$(aws controltower list-landing-zones \
  --query 'landingZones[0].arn' \
  --output text)

aws controltower get-landing-zone \
  --landing-zone-identifier $LANDING_ZONE_ARN \
  --query 'landingZone.{Status:status,Version:version,DriftStatus:driftStatus.status}'

# Résultat attendu :
# { "Status": "ACTIVE", "Version": "<landing-zone-version>", "DriftStatus": "NOT_DRIFTED" }

# Lister les OUs gouvernées
aws organizations list-organizational-units-for-parent \
  --parent-id $(aws organizations list-roots --query 'Roots[0].Id' --output text) \
  --query 'OrganizationalUnits[].{Name:Name,Id:Id}'

# Vérifier que CloudTrail organization trail est actif
aws cloudtrail describe-trails \
  --include-shadow-trails \
  --query 'trailList[?IsOrganizationTrail==`true`].{Name:Name,S3Bucket:S3BucketName,KMSKey:KMSKeyId}'

# Vérifier la présence éventuelle d'un VPC par défaut dans les régions supportées / gouvernées
for REGION in us-east-1 eu-west-1 ap-southeast-1; do
  VPC_ID=$(aws ec2 describe-vpcs \
    --region $REGION \
    --filters "Name=is-default,Values=true" \
    --query 'Vpcs[0].VpcId' \
    --output text)
  if [ "$VPC_ID" != "None" ]; then
        echo "VPC défaut encore présent dans $REGION : $VPC_ID — vérifier si le compte a été provisionné hors blueprint VPC CT ou hors région supportée"
    else
        echo "Pas de VPC défaut détecté dans $REGION"
  fi
done

Guardrails (Controls) Inclus
Préventifs (SCP), détectifs (Config), proactifs (CloudFormation hooks) — Mandatory, Strongly Recommended, Elective
🚨
Scénario : Un développeur crée un bucket S3 public dans un compte sandbox. Il y stocke des fichiers de configuration contenant des clés API. 3 jours plus tard, un scanner automatique trouve le bucket et exfiltre 50 000 enregistrements clients. Les guardrails détectifs auraient alerté en 15 minutes — les guardrails préventifs l'auraient bloqué immédiatement.
Analogie : Guardrail préventif = "verrou électronique" (SCP qui empêche physiquement l'action). Guardrail détectif = "caméra de surveillance" (Config rule qui enregistre et alerte, mais n'empêche pas).

2.1 — Catalogue des Guardrails Strongly Recommended

💡
Pourquoi activer les Strongly Recommended : Les guardrails Mandatory sont activés automatiquement et ne peuvent pas être désactivés. Les Strongly Recommended couvrent 80% des incidents de sécurité réels mais nécessitent une activation manuelle. Les Elective sont spécifiques à certains secteurs (HIPAA, PCI). Sans les Strongly Recommended, votre environnement est exposé aux vecteurs d'attaque les plus fréquents (S3 public, root non restreint, ports ouverts).

2.2 — Activer un Guardrail (double chemin)

🖥️ Console AWS — Activation d'un guardrail sur une OU 
# Console > AWS Control Tower > Controls (ex-Guardrails)
# 1. Sélectionner le contrôle (ex: "Detect public S3 buckets")
# 2. Onglet "Organizational units"
# 3. Cliquer "Enable control on OU"
# 4. Sélectionner la ou les OUs cibles
# 5. Cliquer "Enable control"
# Délai d'activation : 2-5 minutes par OU
☀️ AWS CLI — Activation / Vérification Guardrails 
# Lister tous les guardrails disponibles dans Control Tower
aws controltower list-controls \
  --query 'controls[?implementationDetails.type==`AWS::ControlTower::Control`].{Name:name,Guidance:implementationDetails.type}' \
  --output table

# Activer un guardrail sur une OU
OU_ARN="arn:aws:organizations::123456789012:ou/o-xxxx/ou-xxxx-xxxxxxxx"
CONTROL_ARN="arn:aws:controltower:us-east-1::control/AWS-GR_RESTRICT_ROOT_USER"

aws controltower enable-control \
  --control-identifier $CONTROL_ARN \
  --target-identifier $OU_ARN

# Vérifier le statut des contrôles actifs sur une OU
aws controltower list-enabled-controls \
  --target-identifier $OU_ARN \
  --query 'enabledControls[].{Control:controlIdentifier,Status:statusSummary.status}' \
  --output table

# Identifier les non-conformités détectées par les guardrails détectifs
aws configservice describe-compliance-by-config-rule \
  --compliance-types NON_COMPLIANT \
  --query 'ComplianceByConfigRules[].{Rule:ConfigRuleName,Compliance:Compliance.ComplianceType}'
⚠️
Piège : Guardrails préventifs dans le Management Account : Les SCPs ne s'appliquent pas au Management Account lui-même. Activer les guardrails sur les OUs enfants, mais le Management Account reste non couvert par les SCP. Il faut surveiller le Management Account via d'autres moyens (CloudTrail alerting, GuardDuty, AWS Security Hub).

🚨
Scénario : Une entreprise de 200 comptes AWS crée chaque nouveau compte manuellement : CloudTrail oublié sur 20% des comptes, VPC par défaut actif partout, GuardDuty non activé dans 40% des comptes. 6 mois de dette technique accumulée, impossible d'auditer. 3.2 — Créer un compte via Account Factory (double chemin)

🖥️ Console AWS — Création compte via Account Factory 
# Console > AWS Control Tower > Account Factory
# Cliquer "Enroll account" (compte existant) ou "Create account" (nouveau)

# Champs requis :
# - Account email      : aws-dev-projectX@company.com (unique dans Organizations)
# - Display name       : ProjectX-Dev
# - IAM Identity Center email : Propriétaire du compte
# - IAM Identity Center user  : Prénom Nom
# - Organizational unit : Workloads/Dev (sélectionner dans la hiérarchie)
# - AWS SSO user : Attribuer le permission set approprié

# Durée : 15-30 minutes. Baselines appliquées automatiquement :
# - CloudTrail activé
# - AWS Config activé
# - IAM Identity Center assignment créé
# - Guardrails hérités de l'OU parent
☀️ AWS CLI — Account Factory via Service Catalog 
# Account Factory utilise Service Catalog en interne
# Identifier le product et l'artifact Account Factory
PRODUCT_ID=$(aws servicecatalog search-products \
  --filters 'FullTextSearch=["AWS Control Tower Account Factory"]' \
  --query 'ProductViewSummaries[0].ProductId' \
  --output text)

ARTIFACT_ID=$(aws servicecatalog list-provisioning-artifacts \
  --product-id $PRODUCT_ID \
  --query 'ProvisioningArtifactDetails[-1].Id' \
  --output text)

# Créer un nouveau compte via Account Factory
aws servicecatalog provision-product \
  --product-id $PRODUCT_ID \
  --provisioning-artifact-id $ARTIFACT_ID \
  --provisioned-product-name "ProjectX-Dev" \
  --provisioning-parameters \
    '[
      {"Key":"AccountName","Value":"ProjectX-Dev"},
      {"Key":"AccountEmail","Value":"aws-dev-projectx@company.com"},
      {"Key":"SSOUserEmail","Value":"john.doe@company.com"},
      {"Key":"SSOUserFirstName","Value":"John"},
      {"Key":"SSOUserLastName","Value":"Doe"},
      {"Key":"ManagedOrganizationalUnit","Value":"Workloads/Dev"},
      {"Key":"VPCOptions","Value":"No VPC"}
    ]'

# Surveiller l'état de provisioning
aws servicecatalog describe-provisioned-product \
  --name "ProjectX-Dev" \
  --query 'ProvisionedProductDetail.{Status:Status,StatusMessage:StatusMessage}'
💡
Enrollment de comptes existants : Pour intégrer un compte AWS déjà existant dans Control Tower, le compte doit : (1) ne pas avoir d'AWS Config dans des régions gérées avec un enregistreur déjà actif, (2) ne pas avoir de CloudTrail organization trail en conflit, (3) être dans l'organisation AWS. Utiliser "Enroll account" dans Account Factory plutôt que "Create account".

OU Structure & Gouvernance AWS Organizations
Structure OU recommandée, nested OUs, SCP inheritance, Quarantine OU, moving accounts
🚨
Scénario : Une entreprise a tous ses comptes AWS dans une seule OU "root". Les SCPs de quarantaine ne peuvent pas être appliquées à un compte compromis sans affecter tous les autres. Un compte de test a le même niveau d'accès qu'un compte de production. Sans hiérarchie OU structurée, la gestion des accès et l'isolation sont impossibles.

4.1 — Structure OU recommandée

💡
Principe d'héritage SCP : Une SCP attachée à une OU s'applique à tous les comptes enfants ET à toutes les sous-OUs. La racine Organizations ne doit porter que des politiques minimales communes. Plus vous descendez dans la hiérarchie, plus les restrictions peuvent être spécifiques. 
Root (Organization)
├── Security OU (créé par CT — Log Archive + Audit)
│   ├── [Guardrails CT obligatoires]
│   └── [SCP: pas de workloads, accès lecture seule]
│
├── Infrastructure OU
│   ├── Network Account (Transit Gateway, VPC partagés)
│   ├── DNS Account (Route 53 Resolver)
│   └── Shared Services Account
│
├── Workloads OU
│   ├── Production OU
│   │   ├── App1-Prod Account
│   │   └── App2-Prod Account
│   ├── Staging OU
│   │   └── App1-Staging Account
│   └── Dev OU
│       └── App1-Dev Account
│
├── Sandbox OU (créé par CT — expérimentation libre)
│   └── [SCP: pas d'accès Internet en sortie vers production]
│
├── Suspended OU
│   └── [SCP "Quarantaine" : deny tout sauf lecture logs]
│
└── Graveyard OU
    └── [Comptes à fermer — aucun accès actif]

4.2 — Tableau des SCPs par OU

OU SCPs recommandées Guardrails CT Cas d'usage
Security Deny EC2/S3 non-audit, deny resource création Tous mandatory + recommended Log Archive, Audit — isolation totale
Production Deny disable CloudTrail/Config, deny IAM Admin hors brise-glace Tous mandatory + recommended + elective PCI si applicable Workloads de production
Dev Deny accès ressources production, limité à régions autorisées Mandatory + quelques recommended Développement actif
Sandbox Deny peering prod, deny IGW hors subnet sandbox Mandatory uniquement Expérimentation, POC, formation
Suspended Deny * sauf CloudTrail:Get*, S3:Get* (lecture logs uniquement) Tous (compte en quarantaine) Compte suspect — investigation en cours
⚠️
Piège : OU Sandbox non isolée de la production : Par défaut, les comptes Sandbox peuvent créer des VPC peerings vers d'autres comptes de l'organisation. Ajouter une SCP explicite "Deny": "ec2:CreateVpcPeeringConnection" vers les CIDR de production, et "Deny": "sts:AssumeRole" vers les comptes de production dans l'OU Sandbox.
☀️ AWS CLI — Gestion structure OU et SCPs 
# Créer une OU sous la root
ROOT_ID=$(aws organizations list-roots \
  --query 'Roots[0].Id' --output text)

aws organizations create-organizational-unit \
  --parent-id $ROOT_ID \
  --name "Workloads"

# Créer une sous-OU (ex: Production sous Workloads)
WORKLOADS_OU_ID=$(aws organizations list-organizational-units-for-parent \
  --parent-id $ROOT_ID \
  --query "OrganizationalUnits[?Name=='Workloads'].Id" \
  --output text)

aws organizations create-organizational-unit \
  --parent-id $WORKLOADS_OU_ID \
  --name "Production"

# Créer et attacher une SCP de quarantaine (Suspended OU)
QUARANTINE_SCP='{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyAllExceptAudit",
    "Effect": "Deny",
    "NotAction": [
      "cloudtrail:Get*", "cloudtrail:Describe*",
      "s3:GetObject", "s3:ListBucket",
      "logs:Get*", "logs:Describe*"
    ],
    "Resource": "*"
  }]
}'

SCP_ID=$(aws organizations create-policy \
  --content "$QUARANTINE_SCP" \
  --name "QuarantinePolicy" \
  --type SERVICE_CONTROL_POLICY \
  --query 'Policy.PolicySummary.Id' \
  --output text)

# Déplacer un compte compromis vers la Suspended OU
aws organizations move-account \
  --account-id "COMPROMISED_ACCOUNT_ID" \
  --source-parent-id $WORKLOADS_OU_ID \
  --destination-parent-id "SUSPENDED_OU_ID"

Drift Detection & Remediation Inclus
Types de drift, détection automatique, re-registration, re-enrollment, notifications
🚨
Scénario : Un administrateur modifie directement une SCP depuis la console Organizations pour "débloquer temporairement" une action. Il oublie de la rétablir. 3 mois plus tard, lors d'un audit, on découvre que le guardrail "Restrict root user" était en drift depuis 90 jours — et qu'un acteur malveillant a utilisé root dans 5 comptes pendant cette période.

5.1 — Types de dérives et détection

Type de dérive Cause fréquente Détecté par Correction Criticité
SCP Guard­rail modifiée Modification manuelle dans Organizations CT dashboard (DRIFT_DETECTED) Re-register OU dans CT Critique
Compte sorti de CT Move account hors OU gérée par CT CT dashboard (Account drifted) Re-enroll le compte Critique
CloudTrail org trail supprimé CLI/console hors CT CT drift + Config rule Re-register Landing Zone Critique
Config Recorder arrêté Optimisation coûts non autorisée Config rule (CT guardrail) Re-activer + re-register Haut

Permission set modifié hors CT Modification manuelle IAM IC CT dashboard (soft drift) Resync depuis CT Moyen
💡
Correction de dérive : Pour la plupart des dérives, la correction consiste à re-register l'OU ou le compte dans Control Tower (sans recréer). Dans CT Console > Organization > sélectionner l'OU ou le compte en drift > "Re-register". CT reapplique les configurations attendues sans toucher aux workloads. Durée : 5-15 minutes.

Customizations for Control Tower (CfCT) CodePipeline
StackSets + SCPs, pipeline CodePipeline, manifest.yaml, comparaison CfCT vs AFT
🚨
Scénario : Une organisation de 100 comptes déploie des ressources de baseline (Security Hub, GuardDuty, budgets) en cliquant manuellement dans chaque compte. 3 personnes y passent 2 semaines. Lors d'une réorganisation des OUs, toutes les customizations sont perdues et doivent être réappliquées manuellement. CfCT (Customizations for Control Tower) automatise ce processus via un pipeline GitOps.

6.1 — Architecture CfCT

💡
CfCT vs Account Factory Customizations : CfCT est un pipeline CodePipeline déclenché par des événements CT (nouveau compte, mise à jour OU). Il déploie des StackSets CloudFormation dans les comptes cibles. Différence clé : CfCT est déclaratif et GitOps — toute la configuration est dans un dépôt Git. Pour les nouveaux projets >20 comptes, préférer AFT (S7) qui est plus complet.
Composant CfCT Rôle Technologie Quand utiliser
manifest.yaml Définit quelles StackSets déployer sur quelles OUs/comptes YAML CfCT Toujours (fichier de config central)
CodeCommit / S3 bucket Source des templates CloudFormation Git / S3 Toujours
CodePipeline Orchestre le déploiement des StackSets AWS CodePipeline Automatique à chaque commit / événement CT
CloudFormation StackSets Déploie les ressources dans les comptes cibles CloudFormation Déploiements multi-comptes
Service Control Policies (SCPs custom) Déploie des SCPs additionnelles via CfCT CfCT + Organizations SCPs métier non gérées par CT natif
📄 CfCT manifest.yaml — Exemple de déploiement baseline 
# manifest.yaml — CfCT configuration
# Déployer Security Hub sur tous les comptes de l'OU Workloads
region: us-east-1
version: 2021-03-15

resources:
  - name: SecurityHubBaseline
    description: Active Security Hub sur tous les comptes workloads
    resource_file: templates/security-hub-baseline.yaml
    deploy_method: stack_set
    deployment_targets:
      organizational_units:
        - Workloads/Production
        - Workloads/Dev
    regions:
      - us-east-1
      - eu-west-1
    export_outputs:
      - name: SecurityHubArn
        value: $[output_SecurityHubArn]

  - name: BudgetAlerts
    description: Budgets et alertes par compte
    resource_file: templates/budget-alerts.yaml
    deploy_method: stack_set
    deployment_targets:
      accounts:
        - ALL_ACCOUNTS_IN_OU:Workloads
    parameters:
      MonthlyBudgetUSD: "500"
      AlertEmail: aws-billing@company.com
⚠️
Piège : CfCT ne déploie pas en temps réel — CfCT est déclenché par des événements CT (nouveau compte créé via Account Factory). Si un compte est créé manuellement hors Account Factory, CfCT n'est pas déclenché. Toujours utiliser Account Factory pour créer des comptes afin que CfCT reçoive le trigger d'automatisation.

6.2 — Checklist CfCT

Pipeline testé sur un compte sandbox avant déploiement production
Tous les nouveaux comptes créés via Account Factory (pour trigger CfCT)

Account Factory for Terraform (AFT) Terraform
Architecture AFT, account request Terraform, customizations, GitOps, account data
🚨
Scénario : Une banque crée 50 comptes AWS par an. Avec Account Factory natif, chaque compte nécessite une customization manuelle post-création (TGW attachment, DNS, Security Hub, budget, tags). L'équipe Cloud passe 4h par compte. AFT réduit ce temps à 0h (tout est automatique, GitOps, versionné, auditabl).

7.1 — Architecture AFT

💡
Prérequis AFT : Un compte AFT dédié ("AFT Management Account") est nécessaire, distinct du CT Management Account. AFT supporte Terraform Community Edition, Terraform Cloud et Terraform Enterprise en version 1.6.1 ou ultérieure. Un fournisseur VCS est requis ; AWS indique CodeCommit par défaut, mais une première installation sans dépôt CodeCommit existant doit utiliser un VCS externe tel que GitHub ou Bitbucket. AFT déploie ensuite ses propres services (pipeline, orchestration, stockage, notifications) dans le compte AFT.
Composant AFT Rôle Déclencheur
Account Request Fichier Terraform .tf définissant un compte à créer Pull Request Git approuvée
Account Provisioning Pipeline Crée le compte via Account Factory CT Merge de la PR
Global Customizations Customizations appliquées à TOUS les nouveaux comptes Post-création automatique
Account Customizations Customizations spécifiques à un compte ou groupe Post-création selon tags du compte
Account Provisioning Customizations Hooks pre/post création de compte Avant/après Account Factory
📄 AFT — Exemple account request Terraform

  source = "./modules/aft-account-request"

    AccountEmail              = "aws-dev-projectx@company.com"
    AccountName               = "ProjectX-Dev"
    ManagedOrganizationalUnit = "Workloads/Dev"
    SSOUserEmail              = "john.doe@company.com"
    SSOUserFirstName          = "John"
    SSOUserLastName           = "Doe"
  }

  account_tags = {
    Environment = "dev"
    Project     = "ProjectX"
    CostCenter  = "CC-1234"
    Owner       = "team-projectx"
  }

  change_management_parameters = {
    change_requested_by = "John Doe"
    change_reason       = "Nouveau compte dev pour ProjectX"
  }

  # Customizations à appliquer en plus des global customizations

⚠️
Piège : Coût AFT : AFT n'a pas de licence dédiée, mais il déploie plusieurs services AWS facturés à l'usage dans le compte AFT. Le coût dépend surtout de l'activité des pipelines, du stockage, des notifications et de l'usage réseau ; AWS précise aussi que désactiver le VPC AFT via aft_enable_vpc=false peut réduire le coût pour certains patterns. Pour des besoins simples, Account Factory natif + CfCT reste souvent plus léger.

🚨
Scénario : Une entreprise a 40 comptes AWS créés manuellement sur 3 ans, sans structure. L'audit révèle : 15 comptes sans CloudTrail, 30 comptes avec VPC par défaut actif, 8 comptes orphelins. La migration vers Control Tower doit se faire sans interruption des workloads de production.

10.1 — Plan de migration vers Control Tower

Étape Action Pré-requis Risque Durée
1 Inventaire Lister tous les comptes, OUs, CloudTrails, Config existants AWS Organizations activé Faible 1-2 jours
2 Nettoyage pré-migration Supprimer CloudTrails en conflit, arrêter Config Recorders multi-régions Liste des ressources en conflit Moyen (perte temporaire de logs) 1-3 jours
3 Déploiement Landing Zone Déployer CT dans un Management Account vierge (ou existant si compatible) MFA root, emails LZ, All Features Organizations Moyen (30-60 min indisponibilité CT setup) 1 jour
4 Création OUs Recréer la hiérarchie OU dans CT Landing Zone active Faible 0.5 jour
5 Enrollment comptes Enroller les comptes existants via Account Factory (Enroll account) Comptes nettoyés (step 2) Moyen (15-30 min/compte) 1-5 jours selon volume
6 Activation guardrails Activer les guardrails Strongly Recommended sur toutes les OUs Comptes enrollés Faible si comptes étaient conformes 0.5 jour

Piège majeur — Enrollment d'un compte avec des ressources existantes : L'enrollment CT applique des restrictions. Si un compte a des ressources utilisant des actions que les guardrails bloquent (ex: root user actif, S3 buckets publics), l'enrollment peut échouer ou provoquer des alertes. Script All-in-One AWS CLI
Script complet de déploiement et audit AWS Control Tower — idempotent

Script bash idempotent — exécuter depuis le Management Account (ou Audit Account pour certaines sections).

📜 Script All-in-One — Vérification complète Control Tower 
#!/usr/bin/env bash
# ============================================================
# AWS Control Tower — Script de vérification all-in-one
# Exécuter depuis le Management Account
# Prérequis : awscli v2, jq, permissions Organizations + CT
# ============================================================

# ---- VARIABLES (à adapter) --------------------------------
AUDIT_ACCOUNT_ID=""           # ID du compte Audit CT
HOME_REGION="us-east-1"        # Région home Control Tower
GOVERNED_REGIONS="us-east-1 eu-west-1"
CHECK_S1="Y"    # [Y/N] Landing Zone
CHECK_S2="Y"    # [Y/N] Guardrails
CHECK_S4="Y"    # [Y/N] OU Structure
CHECK_S5="Y"    # [Y/N] Drift Detection
CHECK_S8="Y"    # [Y/N] Security Baseline

# ---- FONCTIONS -------------------------------------------- 
ok()   { echo "[OK]  $1"; }
warn() { echo "[WARN] $1"; }
fail() { echo "[FAIL] $1"; }

# ---- S1 : LANDING ZONE ----------------------------------- 
if [ "$CHECK_S1" = "Y" ]; then
  echo ""
  echo "=== S1 — Landing Zone ==="
  LZ_STATUS=$(aws controltower list-landing-zones \
    --query 'landingZones[0].status' --output text 2>/dev/null)
  [ "$LZ_STATUS" = "ACTIVE" ] && ok "Landing Zone ACTIVE" || fail "Landing Zone non ACTIVE (statut: $LZ_STATUS)"

  LZ_DRIFT=$(aws controltower list-landing-zones \
    --query 'landingZones[0].driftStatus' --output text 2>/dev/null)
  [ "$LZ_DRIFT" = "NOT_DRIFTED" ] && ok "Landing Zone pas en drift" || fail "Landing Zone en drift: $LZ_DRIFT"

  CT=$(aws cloudtrail describe-trails --include-shadow-trails \
    --query 'trailList[?IsOrganizationTrail==`true`] | length(@)' --output text)
  [ "$CT" -ge 1 ] && ok "CloudTrail organization trail actif" || fail "Pas de CloudTrail organization trail"
fi

# ---- S2 : GUARDRAILS ------------------------------------- 
if [ "$CHECK_S2" = "Y" ]; then
  echo ""
  echo "=== S2 — Guardrails ==="
  ROOT_ID=$(aws organizations list-roots --query 'Roots[0].Id' --output text)
  OUS=$(aws organizations list-organizational-units-for-parent \
    --parent-id $ROOT_ID --query 'OrganizationalUnits[].Id' --output text)
  for OU in $OUS; do
    OU_ARN="arn:aws:organizations::$(aws sts get-caller-identity --query Account --output text):ou/$(aws organizations list-roots --query 'Roots[0].Id' --output text)/$OU"
    CTRL_COUNT=$(aws controltower list-enabled-controls \
      --target-identifier $OU_ARN \
      --query 'length(enabledControls)' --output text 2>/dev/null || echo "0")
    [ "$CTRL_COUNT" -ge 5 ] && ok "OU $OU : $CTRL_COUNT guardrails actifs" || warn "OU $OU : seulement $CTRL_COUNT guardrails"
  done
fi

# ---- S5 : DRIFT ------------------------------------------ 
if [ "$CHECK_S5" = "Y" ]; then
  echo ""
  echo "=== S5 — Drift Detection ==="
  LZ_ARN=$(aws controltower list-landing-zones \
    --query 'landingZones[0].arn' --output text)
  DRIFT=$(aws controltower get-landing-zone \
    --landing-zone-identifier $LZ_ARN \
    --query 'landingZone.driftStatus.status' --output text)
  [ "$DRIFT" = "NOT_DRIFTED" ] && ok "Pas de drift détecté" || fail "DRIFT DÉTECTÉ : $DRIFT — Exécuter reset-landing-zone"
fi

# ---- S8 : SECURITY BASELINE ------------------------------ 
if [ "$CHECK_S8" = "Y" ] && [ -n "$AUDIT_ACCOUNT_ID" ]; then
  echo ""
  echo "=== S8 — Security Baseline ==="
  GD_ADMIN=$(aws guardduty list-organization-admin-accounts \
    --query 'AdminAccounts[0].AdminAccountId' --output text 2>/dev/null)
  [ "$GD_ADMIN" = "$AUDIT_ACCOUNT_ID" ] && ok "GuardDuty Delegated Admin = compte Audit" || warn "GuardDuty Delegated Admin non configuré"

  SH_ADMIN=$(aws securityhub list-organization-admin-accounts \
    --query 'AdminAccounts[0].AccountId' --output text 2>/dev/null)
  [ "$SH_ADMIN" = "$AUDIT_ACCOUNT_ID" ] && ok "Security Hub Delegated Admin = compte Audit" || warn "Security Hub Delegated Admin non configuré"
fi

echo ""
echo "=== Vérification terminée ==="

Checklist de Déploiement
Checklist complète pré/pendant/post déploiement AWS Control Tower — 50+ items

Checklist complète de déploiement — 50+ items organisés par phase.

📋 Pré-déploiement

Management Account créé et dédié (aucun workload)
MFA activé sur le compte root du Management Account
Adresses email préparées : Log Archive, Audit (boîtes fonctionnelles)
AWS Organizations en mode All Features (pas Consolidated Billing uniquement)
IAM Identity Center activé ou accèpté d'activer
Région Home sélectionnée (cohérente avec RGPD si besoin)
Liste des régions à gouverner validée avec l'équipe sécurité

🏗️ S1 — Landing Zone

Landing Zone deployée avec statut ACTIVE
Compte Log Archive créé et accessible
Compte Audit créé et accessible
S3 bucket CloudTrail chiffré avec CMK dans Log Archive
Organization CloudTrail actif et multi-régions
Config Aggregator actif dans Audit
IAM Identity Center configuré avec au moins un Permission Set Admin
VPC par défaut supprimé dans Management Account
Accès au Management Account restreint (<5 admins)

🛡️ S2 — Guardrails

Guardrails Mandatory actifs (vérifiés dans dashboard CT)
Guardrails Strongly Recommended activés sur toutes les OUs de production
Guardrail "Disallow root user" actif sur toutes les OUs membres
Guardrail "Require MFA for IAM users" actif
Guardrail "Deny public S3 buckets" actif sur OUs production
Aucun contrôle en statut UNKNOWN ou FAILED

🏭 S3 — Account Factory

Account Factory configuré avec Network Customization (VPC sans IGW par défaut)
VPC par défaut désactivé pour les nouveaux comptes
Processus de demande de compte documenté et connu des équipes
OU de destination définie pour chaque type de workload
Tous les comptes créés via Account Factory (plus de création manuelle)

🗂️ S4 — OU Structure

OU Security créée (Log Archive + Audit dédiés)
OU Infrastructure créée (réseau, shared services)
OU Workloads créée avec sous-OUs (Prod/Dev/Test)
OU Sandbox isolée avec SCP bloquant les accès production
OU Suspended avec SCP de quarantaine sur les comptes désactivés
Hiérarchie OU documentée et validée en gouvernance

🔍 S5 — Drift Detection

Procédure de vérification mensuelle du drift définie
Alerte EventBridge configurée sur les événements CT Drift
Responsable désigné pour la correction du drift
Procédure de re-registration de compte documentée
Aucun contrôle en DRIFTED dans le dashboard CT

📋 S6 — CfCT (si applicable)

Pipeline CfCT déployé dans Management Account
manifest.yaml versionné dans Git (main branch = prod)
Tests de non-régression sur les StackSets

🧰 S7 — AFT (si applicable)

Module Terraform AFT déployé dans le compte AFT Management dédié
Répertoire account-requests configuré dans Git
Global customizations (rôles, tags, CloudTrail) définis et testés

🛡️ S8 — Security Baseline

Amazon GuardDuty activé sur tous les comptes avec Delegated Admin = Audit
AWS Security Hub activé avec standards CIS AWS Foundations
Amazon Macie activé sur le compte Log Archive
AWS Config Aggregator actif couvrant toutes les régions gouvernées
Procédure de traitement des findings Security Hub définie (SLA)
Alertes sur les findings CRITICAL configurées

📊 S9 — Reporting

Config Aggregator couvrant tous les comptes et toutes les régions
Rapport mensuel de conformité automatisé (Security Hub)
KPIs de conformité définis et partagés avec le management

🔄 S10 — Migration

Tous les comptes existants enrollés (statut ENROLLED)
Aucun compte en statut DRIFTED post-enrollment
Plan de migration validé et communiqué
Clôture de la migration : documentation mise à jour
