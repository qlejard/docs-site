Mode d'emploi de ce guide
Comment utiliser ce document pour auditer et durcir Microsoft Defender for Endpoint
📋
Ce guide couvre la configuration complète de Microsoft Defender for Endpoint (MDE) : onboarding des devices, règles ASR, EDR, protection réseau, TVM, indicators custom et automation. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un tenant. Le déploiement via Intune est référencé (chemins portail) mais non détaillé — ce guide documente le quoi configurer, pas le véhicule de distribution.

Stratégie de déploiement recommandée

1
Onboarding & protections de base — ~15 min, +20 pts. Vérifier l'onboarding, activer tamper protection, cloud protection High+, block at first sight, real-time protection.
2
ASR Rules en Audit — ~15 min, +8 pts. Déployer toutes les règles ASR en mode Audit pour observer l'impact avant le blocage.
3
Hardening EDR + Network — ~1h. Passer les ASR rules critiques en Block, activer network protection en Block, configurer web content filtering.
4
Segmentation & automation — ~2h. Créer les device groups, configurer le RBAC, activer AIR Full automation, déployer les custom detections.
5
Intégration & forensics — Variable. Activer Live Response (P2), connecter à Sentinel, configurer les indicators IoC, tuner les alertes.

1
Onboarding Devices MDE P1
Enrôlement des endpoints dans Defender for Endpoint — la fondation de toute la chaîne de protection
🚨
Vecteur de risque : un device non onboardé est invisible pour MDE. Pas de télémétrie, pas d'alerte, pas de réponse. C'est comme avoir une caméra de surveillance éteinte : l'intrus entre sans être détecté. Statistique terrain : dans 40% des audits, des machines existent dans Entra ID mais ne sont pas onboardées dans MDE.
ℹ️
L'onboarding MDE installe un sensor sur le device qui remonte la télémétrie vers le cloud Microsoft (process, network, file events). Plusieurs méthodes existent : Intune (recommandé), GPO, script local, SCCM/ConfigMgr. Le sensor communique via HTTPS vers les URLs Microsoft requises — un proxy mal configuré est la cause n°1 d'échec d'onboarding.

Piège courant : l'onboarding via Intune ne fonctionne pas si le device n'est pas Entra ID Joined ou Hybrid Joined. 💡
Pourquoi chaque paramètre compte :
  • Onboarding status — Sans sensor actif, aucune protection MDE ne fonctionne. Le device est un angle mort total.
  • Méthode de déploiement — Intune est préféré car il garantit le re-onboarding automatique en cas de reset. GPO ne protège pas contre un wipe/re-image.
  • Connectivity test — Sans connectivité cloud, le sensor stocke localement (limite 30 Mo) puis perd les données. L'attaquant a une fenêtre d'invisibilité.
  • Sample submission (Send all) — Requis pour Block at First Sight. Sans cela, un fichier inconnu est autorisé 10 secondes pendant l'analyse cloud — assez pour exécuter un payload. Réf. CIS Microsoft 365 Benchmark 3.0, section 5.3.2.
  • Proxy/URL allowlist — Cause n°1 d'échec d'onboarding. La liste officielle Microsoft contient ~10 patterns d'URLs à autoriser. Ref: Configure proxy.

1.4 — État des lieux

💡
Device Discovery : cette fonctionnalité MDE détecte les devices non managés présents sur le réseau (imprimantes, IoT, serveurs legacy, machines BYOD). Activée par défaut en mode Basic, elle utilise le trafic réseau passif des devices onboardés. Le mode Standard envoie des probes actives pour enrichir l'inventaire. Chemin : Settings > Endpoints > Device discovery. Chaque device découvert peut ensuite être onboardé directement — c'est la clé pour éliminer les angles morts réseau.
2
Attack Surface Reduction (ASR) Rules MDE P1
Règles de réduction de la surface d'attaque — le bouclier préventif le plus impactant de MDE
🚨
Scénario d'attaque : un employé ouvre un fichier Word contenant une macro VBA malveillante. Sans ASR rules, la macro crée un processus enfant (powershell.exe), télécharge un payload et l'exécute. Avec la règle « Block Office applications from creating child processes » en mode Block, l'attaque est stoppée instantanément. Analogie : les ASR rules sont les barreaux aux fenêtres de votre système — elles empêchent l'intrusion avant même que l'alarme (EDR) ne se déclenche.
⚠️
Piège critique : par défaut, aucune règle ASR n'est activée (toutes en Not configured). Même avec un preset Intune Endpoint Security, seules quelques règles passent en Audit. Le CIS Benchmark M365 (section 5.3.x) recommande le mode Block pour les règles critiques. Déploiement recommandé : Audit 2-4 semaines → analyser les faux positifs → Block.

Pourquoi chaque règle ASR compte :
  • Office → child processes — Bloque la chaîne macro VBA → PowerShell → payload. Vecteur n°1 du phishing ciblé. MITRE ATT&CK T1204.002.
  • Credential stealing (LSASS) — Empêche Mimikatz et outils similaires d'extraire les credentials en mémoire. MITRE T1003.001. Complémentaire à Credential Guard (hardware).
  • Potentially obfuscated scripts — Détecte l'obfuscation PowerShell (base64, string concatenation) utilisée par 85% des malwares fileless. MITRE T1059.001.
  • USB untrusted processes — Bloque l'exécution de binaires non signés depuis clés USB. Empêche les attaques de type « clé USB trouvée sur le parking ». MITRE T1091.
  • Ransomware protection — Heuristique spécifique anti-ransomware (pattern de chiffrement de fichiers en masse). Dernière ligne de défense avant le chiffrement.
  • WMI persistence — Bloque l'inscription d'events WMI permanents utilisés pour la persistance avrès reboot. MITRE T1546.003.
  • PSExec/WMI lateral movement — Bloque le mouvement latéral via PsExec et WMI remote — techniques classiques post-compromission. MITRE T1570.

Cloud protection, real-time protection, tamper protection — le moteur antivirus nouvelle génération
🚨
Scénario d'attaque : un ransomware « zero-day » arrive par email. L'antivirus local avec signatures à J-1 ne le détecte pas. Avec Cloud protection level = High+ et Block at first sight activé, le fichier est envoyé au cloud Microsoft en <1 seconde et bloqué par le machine learning avant exécution. Analogie : sans cloud protection, votre antivirus est un dictionnaire périmé. Avec, c'est un analyste connecté en temps réel à l'intelligence collective de 10 milliards de signaux/jour.
⚠️
Piège critique : Tamper Protection empêche la désactivation de l'antivirus par un attaquant (ou un admin distrait). Si elle n'est pas activée, un malware avec privilèges admin peut exécuter Set-MpPreference -DisableRealtimeMonitoring $true et désactiver toute la protection. Par défaut, Tamper Protection est activée pour les tenants MDE, mais peut être désactivée — vérifiez !

3.1 — Paramètres EDR / Antivirus

💡 Pourquoi chaque paramètre compte

Cloud protection, Block at First Sight et Tamper Protection forment le triptyque fondamental de la détection MDE. Sans cloud protection High+, les menaces zero-day échappent aux signatures locales pendant des heures. Un attaquant qui désactive Tamper Protection rend l'ensemble de la chaîne EDR inopérante — chaque paramètre de cette section est un verrou critique de votre posture endpoint.

💡
Pourquoi ces paramètres comptent :
  • Cloud protection High+ — Active le machine learning agressif côté cloud. Peut générer plus de faux positifs mais détecte les menaces zero-day 10x plus vite que le mode Default.
  • Block at first sight — Un fichier inconnu est mis en quarantaine pendant que le cloud l'analyse (< 10 sec). Sans BAFS, le fichier s'exécute pendant l'analyse. CIS M365 5.1.2.2.
  • PUA Protection — Bloque les adware, toolbar, cryptominer. Souvent désactivé par défaut. CIS M365 5.1.2.3.
  • Tamper Protection — Empêche la modification de Defender par scripts/malware même avec des droits admin. Essentiel contre les ransomware qui désactivent l'AV en premier.
  • EDR in block mode (P2) — Permet à EDR de bloquer des menaces post-breach détectées par le comportement, même si l'AV classique ne les a pas vues. Agit comme un « second filet ».
  • Sample submission = All samples — Nécessaire pour que BAFS et la protection cloud fonctionnent sur les fichiers non-PE (scripts, documents). Contrôlez les exclusions de contenu sensible dans les règles Intune.
Réf. : Configure Defender Antivirus features — Microsoft Learn | CIS M365 Benchmark v3.0 section 5.1.

Le tuning des alertes et l'intégration Sentinel/XDR déterminent la capacité du SOC à distinguer les vraies menaces du bruit. Sans suppression rules ni classification systématique, l'alert fatigue conduit inévitablement à ignorer des incidents critiques — un SOC noyé sous 500 alertes/jour rate l'attaque réelle.

Pourquoi le tuning est critique :
  • Alert suppression — Réduit le bruit de 60-80%. Exemples : outils sysadmin (PsExec sur machines d'admin), scanners de vuln légitimes, machines de lab/test.
  • Classification — Permet de mesurer le taux de faux positifs. Si FP > 70%, vos détections ont un problème. Objectif : FP < 30%.
  • Sentinel — Centralise les alertes MDE avec les logs réseau, AAD, Office 365. Permet les détections cross-produit. Voir sentinel.html pour la config Sentinel.
  • XDR — Defender XDR corrèle automatiquement les alertes MDE + MDO (email) + MDI (identité) + MDCA (cloud apps) en incidents. Un phishing (MDO) suivi d'un malware (MDE) = 1 incident, pas 2 alertes séparées.

10.2 — Intégration Conditional Access (MDE ↔ Entra ID)

💡
Signal de risque device → Conditional Access : MDE évalue le niveau de risque de chaque device (Low / Medium / High / Critical). Ce signal est transmis à Entra ID via le connecteur MDE ↔ Intune ↔ Entra ID, permettant de créer des Conditional Access policies basées sur le risque machine.
  • Pré-requis : connecteur Intune ↔ MDE actif (Settings > Endpoints > Advanced features > Microsoft Intune connection)
  • Flux : MDE détecte une menace → device marqué « High risk » → Entra ID CA policy bloque l'accès aux apps sensibles → requête de remédiation envoyée à l'utilisateur
  • Policy recommandée : Grant access > Require device to be marked as compliant + Require device risk level ≤ Medium
  • Impact : un device compromis est automatiquement isolé des ressources Microsoft 365 — Zero Trust réactif
Voir entra-id.html — S1 Conditional Access pour la configuration complète des CA policies.

Script All-in-One — Audit MDE
⚠️
Deux API distinctes :
  • Microsoft Graph Security API (graph.microsoft.com/v1.0/security/...) — utilisée via Connect-MgGraph + Invoke-MgGraphRequest. Couvre alertes, incidents, Secure Score.
  • MDE API (api.securitycenter.microsoft.com/api/...) — deprecated, migrer vers graph.microsoft.com/beta/security/microsoft/windowsDefenderATP/.... Le script ci-dessous utilise déjà Graph pour la majorité des appels.
Le script ci-dessous unifie les deux via une fonction helper Invoke-MdeApi. Pas de commandes Intune (le déploiement des policies se fait via Intune, pas via script).
Prérequis : module Microsoft.Graph, app registration avec permissions SecurityEvents.Read.All, Machine.Read.All (WindowsDefenderATP), SecurityActions.Read.All.

# Utilise MSAL pour obtenir un token WindowsDefenderATP

    param([string]$Path)

        -Uri "https://graph.microsoft.com/beta/security/microsoft/windowsDefenderATP$Path"

    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/managedDevices"

$active = ($machines.value | Where-Object { $_.managedDeviceOwnerType -eq "company" }).Count
Write-Host "Machines gérées : $total | Corporate : $active | Taux : $([math]::Round($active/$total*100))%"
