Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de configuration
📋
Ce document est un référentiel technique couvrant la gouvernance multi-compte AWS avec Organizations et Control Tower. Il peut être utilisé tel quel en audit ou adapté au contexte spécifique d'un environnement AWS.

Analogie : AWS Organizations est le groupe d'entreprises (holding) qui supervise les filiales (comptes AWS). Les SCPs sont le règlement intérieur que personne ne peut contourner. Control Tower est la tour de contrôle qui pilote l'ensemble depuis un point central.

S1
Organizations Setup Gratuit
Création de l'Organization, mode all features, structure OU & bonnes pratiques du management account — 20 pts
🚨
Scénario d'attaque : Sans Organization, chaque compte AWS est une île isolée. Un développeur crée un compte personnel, y déploie des workloads avec des clés d'accès root, expose un bucket S3 public, et personne ne le sait. Aucune SCP ne l'empêche, aucun trail ne le trace, aucun budget ne l'alerte. Résultat : fuite de données, facture surprise de 50 000 $, aucune traçabilité.

Analogie : AWS Organizations est le groupe d'entreprises (holding) qui supervise ses filiales. Sans holding, chaque filiale fait ce qu'elle veut — comptabilité séparée, règles différentes, aucune vue consolidée. Avec la holding, vous avez un règlement intérieur (SCPs), une comptabilité consolidée (billing), et une tour de contrôle (Control Tower).

💡
Pourquoi chaque paramètre compte :
  • ALL_FEATURES mode : le mode « consolidated billing only » ne permet PAS d'appliquer des SCPs. Sans SCPs, aucune barrière de sécurité ne peut être imposée aux comptes membres. Le mode ALL_FEATURES est irréversible — une fois activé, impossible de revenir en arrière (source : AWS Organizations docs).
  • Structure OU à 5 niveaux : une OU flat (tous les comptes sous Root) empêche d'appliquer des SCPs différenciées. La structure Security / Infrastructure / Workloads / Sandbox / Suspended est recommandée par AWS Well-Architected Framework (Security Pillar, SEC01-BP01). L'OU Suspended permet de « geler » un compte compromis sans le supprimer.
  • Zéro workload dans le management account : le management account a des privilèges spéciaux (bypass des SCPs, accès billing). Tout workload déployé ici augmente la surface d'attaque du compte le plus sensible. Source : CIS AWS Foundations Benchmark v7.0.0, recommandation 1.1.
  • Root user sécurisé : le root user du management account peut dissoudre l'Organization entière et supprimer tous les SCPs. MFA matériel obligatoire + suppression des access keys = réduction drastique du risque de takeover. Source : CIS AWS Foundations Benchmark v7.0.0, 1.5 & 1.6.
  • Delegated administrators : sans délégation, les équipes sécurité doivent se connecter au management account pour gérer Security Hub ou Amazon GuardDuty — augmentant le risque d'erreur sur le compte le plus critique. La délégation déplace ces opérations vers un compte dédié (Security Tooling).
  • Email management : chaque compte AWS possède un email root unique. Si cet email est une boîte personnelle, la récupération de compte devient impossible quand l'employé part. Distribution lists dédiées = continuité d'accès.
⚠️
Pièges :
  • Le management account bypass les SCPs : aucune SCP ne s'applique au management account lui-même. C'est pourquoi il ne doit contenir aucun workload et son root doit être ultra-sécurisé.
  • Limite de 5 SCPs par entité : chaque OU ou compte ne peut avoir que 5 SCPs attachées. Planifier le regroupement des deny statements.
  • Limite de 5 120 caractères par SCP : les SCPs verbeuses sont vite tronquées. Utiliser des deny-list (plus compactes) plutôt que des allow-list.
  • ALL_FEATURES est irréversible : tester sur une Organization de test avant de migrer la production.
  • Le compte de management ne peut PAS être déplacé : il reste toujours à la racine. Ne pas confondre avec les comptes membres.

1.3 — Configuration

1.3.1 — Créer l'Organization (AWS CLI)

Bash — Création de l'Organization et de la structure OU
# Créer l'Organization en mode ALL_FEATURES
aws organizations create-organization \
    --feature-set ALL_FEATURES \
    --profile management-admin \
    --region eu-west-1 \
    --output json

# Récupérer l'ID de la racine
ROOT_ID=$(aws organizations list-roots \
    --profile management-admin \
    --region eu-west-1 \
    --query 'Roots[0].Id' --output text)

# Créer les OUs de niveau 1
for OU_NAME in "Security" "Infrastructure" "Workloads" "Sandbox" "Suspended" "Exceptions"; do
    aws organizations create-organizational-unit \
        --parent-id "$ROOT_ID" \
        --name "$OU_NAME" \
        --profile management-admin \
        --region eu-west-1
    echo "✅ OU créée : $OU_NAME"
done

# Récupérer l'ID de l'OU Workloads pour créer les sous-OUs
WORKLOADS_OU_ID=$(aws organizations list-organizational-units-for-parent \
    --parent-id "$ROOT_ID" \
    --profile management-admin \
    --region eu-west-1 \
    --query "OrganizationalUnits[?Name=='Workloads'].Id" --output text)

# Créer les sous-OUs Workloads
for SUB_OU in "Production" "Staging" "Development"; do
    aws organizations create-organizational-unit \
        --parent-id "$WORKLOADS_OU_ID" \
        --name "$SUB_OU" \
        --profile management-admin \
        --region eu-west-1
    echo "✅ Sous-OU créée : Workloads/$SUB_OU"
done

# Créer les sous-OUs Security
SECURITY_OU_ID=$(aws organizations list-organizational-units-for-parent \
    --parent-id "$ROOT_ID" \
    --profile management-admin \
    --region eu-west-1 \
    --query "OrganizationalUnits[?Name=='Security'].Id" --output text)

for SUB_OU in "Log Archive" "Security Tooling" "Forensics"; do
    aws organizations create-organizational-unit \
        --parent-id "$SECURITY_OU_ID" \
        --name "$SUB_OU" \
        --profile management-admin \
        --region eu-west-1
    echo "✅ Sous-OU créée : Security/$SUB_OU"
done

S2
Service Control Policies (SCPs) Gratuit
Deny-list strategy, SCPs critiques, héritage & limites — le règlement intérieur de l'organization — 15 pts
🚨
Scénario d'attaque : Un administrateur d'un compte membre décide de désactiver CloudTrail pour « réduire les coûts ». Sans SCP, rien ne l'en empêche. Les logs sont supprimés, un attaquant pénètre le compte via une clé exposée, mine du crypto pendant 3 semaines. Facture : 180 000 $. Aucune trace d'audit.

Analogie : Les SCPs sont le règlement intérieur de l'entreprise. Même le directeur d'une filiale (admin d'un compte) ne peut pas contourner le règlement. Mais le PDG (management account) est au-dessus du règlement — raison de plus pour sécuriser ce compte en priorité.
ℹ️
Concepts clés :
  • Deny-list vs Allow-list : AWS recommande la stratégie deny-list (garder la FullAWSAccess par défaut et ajouter des deny explicites). Plus maintenable et moins de risque de se bloquer soi-même.
  • Héritage : les SCPs se cumulent de Root → OU → sous-OU → Compte. L'intersection des permissions est appliquée. Un deny à n'importe quel niveau l'emporte toujours.
  • Limites : max 5 SCPs par entité, max 5 120 caractères par SCP (sans espaces blancs). Planifier le regroupement des règles.
  • Management account exempté : les SCPs ne s'appliquent JAMAIS au management account. Utilisez des IAM policies et des guardrails de monitoring pour celui-ci.

Paramètre Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût

aws organizations list-roots → PolicyTypes		 SCPs désactivées (billing only) SCPs activées (ALL_FEATURES requis) Activées Activées + TAG_POLICY + BACKUP_POLICY Gratuit
Deny root actions
SCP #1 : Deny root user access		 Root user actif sur tous les comptes Deny root:* sur tous les comptes membres + Exception pour les rares actions root-only + Alerte immédiate si root login détecté Gratuit
Deny region
SCP #2 : Deny non-approved regions		 Toutes les régions accessibles Deny sauf eu-west-1, eu-west-3, us-east-1 (global services) + Conditions spécifiques par OU (sandbox = 1 région) + Tag-based exceptions pour cas légitimes Gratuit
Protect CloudTrail
SCP #3 : Deny StopLogging / DeleteTrail		 Aucune protection Deny cloudtrail:StopLogging, cloudtrail:DeleteTrail + Deny cloudtrail:UpdateTrail, PutEventSelectors + Deny s3:Delete* sur le bucket de logs Gratuit
Protect Config
SCP #4 : Deny StopConfigurationRecorder / DeleteConfigRule		 Aucune protection Deny config:StopConfigurationRecorder, config:DeleteConfigurationRecorder + Deny config:DeleteConfigRule, DeleteRetentionConfiguration + Deny modification des conformance packs Gratuit
Deny quitter l'Organization
SCP #5 : Deny organizations:LeaveOrganization		 Les comptes peuvent quitter librement Deny organizations:LeaveOrganization + Monitoring des tentatives + Intégré dans le pipeline de décommissionnement Gratuit
Deny services non autorisés
SCP #6 : Deny services sensibles		 Tous les services accessibles — Deny GameLift, Ground Station, etc. + Allow-list par OU/workload Gratuit
Deny accès public S3/Lambda
SCP #7 : Deny public access		 Accès public possible — Deny s3:PutBucketPublicAccessBlock (enforce block), Deny lambda:AddPermission (public) + Enforcement sur tous les services exposés Gratuit
SCP Suspended OU
SCP #8 : Deny All (sauf lecture)		 — Deny * sauf read-only sur Suspended OU Deny * strict (aucune exception) Deny * + tag « quarantine » automatisé Gratuit
⚠️
Pièges critiques :
  • Ne PAS retirer la policy FullAWSAccess : si vous retirez cette policy par défaut sans avoir de SCP allow-list en place, TOUS les accès sont bloqués sur tous les comptes membres. Récupération uniquement via le management account.
  • Tester sur Sandbox AVANT Production : une SCP mal formée peut bloquer des services critiques. Toujours déployer d'abord sur une OU Sandbox, valider pendant 24h, puis propager.
  • Les SCPs n'accordent PAS de permissions : une SCP est un filtre qui restreint ce que les IAM policies peuvent accorder. Même avec FullAWSAccess SCP, un utilisateur a besoin d'une IAM policy pour agir.
  • Attention aux conditions globales : aws:RequestedRegion ne filtre pas les services globaux (IAM, STS, Route 53, CloudFront). Toujours inclure une exclusion pour les global services.
  • Impact sur les service-linked roles : les SCPs affectent également les service-linked roles. Un deny sur ec2:* empêcherait Auto Scaling de fonctionner.

2.2 — Héritage des SCPs

Héritage des SCPs — Intersection (AND logic) Root ── FullAWSAccess (par défaut) │ + DenyRootAccess │ + DenyUnapprovedRegions │ + ProtectCloudTrail │ + DenyLeaveOrganization │ ├── Security OU │ Hérite : toutes les SCPs Root │ + ProtectConfigRecorder │ ├── Workloads / Production OU │ Hérite : toutes les SCPs Root │ + DenyPublicAccess │ + DenyUnauthorizedServices │ ├── Sandbox OU │ Hérite : toutes les SCPs Root │ + DenyExpensiveServices │ └── Suspended OU Hérite : toutes les SCPs Root + DenyAll (sauf read-only) Résultat effectif = FullAWSAccess ∩ SCPs Root ∩ SCPs OU ∩ SCPs sous-OU ∩ SCPs Compte Un Deny à n'importe quel niveau l'emporte TOUJOURS sur un Allow

3.2 — Landing Zone Checklist

📋
Prérequis avant de déployer Control Tower :
  • ☐ Organization en mode ALL_FEATURES (voir S1)
  • ☐ Le compte management n'est PAS un compte AWS créé pour un événement temporaire (Activate, Workshop)
  • ☐ Aucun AWS Config recorder actif dans la home region sur le management account (CT le crée)
  • ☐ Email unique disponible pour le compte Log Archive
  • ☐ Email unique disponible pour le compte Audit (Security Tooling)
  • ☐ Aucune OU nommée Core ou Custom dans l'Organization (CT utilise ces noms)
  • ☐ Quota de comptes AWS suffisant (vérifier via Service Quotas)
  • ☐ IAM Identity Center doit être dans la même région que la home region de Control Tower
⚠️
Pièges :
  • Control Tower active AWS Config dans chaque compte et région gouvernée : c'est la première source de coût indirect. Pour 10 comptes × 3 régions = 30 Config recorders × ~30 $/mois = ~900 $/mois. Limiter les régions gouvernées au strict nécessaire.
  • Ne PAS désactiver les guardrails mandatory : certains guardrails mandatory sont marqués « Mandatory » et ne peuvent pas être désactivés. Les guardrails « Strongly recommended » peuvent l'être mais c'est déconseillé.
  • Drift detection : si quelqu'un modifie manuellement une ressource gérée par Control Tower (OU, SCP, CloudTrail, Config), un drift est détecté. Ne pas ignorer les drifts — ils peuvent indiquer une compromission.
  • Landing Zone update : avant de mettre à jour la Landing Zone, lire les release notes. Certaines mises à jour modifient les guardrails et peuvent impacter les workloads.

🚨
Scénario d'attaque : Un attaquant compromet un compte membre et désactive CloudTrail localement. Sans organization trail, personne ne s'en aperçoit. L'attaquant opère pendant 6 mois sans trace. L'investigation forensique est impossible car les logs n'existent plus.

Concept clé : Le monitoring centralisé est la vidéosurveillance centralisée de l'immeuble. Chaque appartement (compte) a sa caméra, mais les flux sont tous stockés dans un local sécurisé (compte Log Archive) auquel les locataires n'ont pas accès. Même si un cambrioleur casse la caméra locale, l'enregistrement central persiste.

  • Organization Trail : sans Organization Trail, chaque compte doit configurer son propre CloudTrail — et certains ne le feront pas. L'Organization Trail garantit que 100% des appels API de tous les comptes sont loggés dans un bucket centralisé protégé.
  • S3 bucket immutable : un attaquant qui compromet un compte essaiera de supprimer les logs pour couvrir ses traces. Object Lock (mode Compliance) + MFA Delete rendent les logs indestructibles, même par le root user du compte Log Archive.
  • Config Rules agrégées : AWS Config évalue la conformité des ressources. Sans agrégation, il faut se connecter à chaque compte pour vérifier la conformité. L'agrégateur multi-compte centralise la vue dans le compte Security.

  • Rétention 365 jours : les investigations forensiques après un incident nécessitent souvent 6-12 mois d'historique. La rétention par défaut de 90 jours est insuffisante. 365 jours couvre la plupart des obligations réglementaires (PCI-DSS, SOC 2).

  • Checklist pré-migration : migrer un compte sans inventaire préalable provoque des surprises : IAM roles cross-account cassés, DNS qui ne résolvent plus, pipelines CI/CD en échec. La checklist de 12 points vérifie chaque dépendance avant le move.
  • Invitation vs création : créer un nouveau compte via Account Factory est toujours préférable (baseline appliquée automatiquement). L'invitation d'un compte existant hérite des SCPs mais PAS de la baseline Landing Zone — nécessite une remédiation manuelle.
  • Billing transfer : lors du déplacement d'un compte vers l'Organization, le billing bascule vers le management account. Les Reserved Instances et Savings Plans du compte source ne sont plus partagés avec l'ancienne Organisation — impact financier potentiel significatif.
  • Rollback strategy : un compte déplacé dans l'Organization peut être retiré (leave), mais il perd toutes les SCPs et la baseline Control Tower. Documenter la procédure de rollback AVANT la migration évite la panique en cas de problème.
  • Fenêtre de migration : pendant le move, il y a une période de 1-5 minutes où les SCPs ne sont pas encore appliquées. Planifier les migrations en heures creuses réduit le risque d'actions non autorisées pendant cette fenêtre.

Cost Optimization Payant
Optimisation des coûts Config, CloudTrail mutualisé, RI/SP centralisation — 5 pts
💰
Analogie : L'optimisation des coûts est le gel des comptes inutiles. Comme une entreprise qui ferme les bureaux inoccupés et négocie des tarifs de groupe, vous devez régulièrement identifier les comptes dormants, les services inutilisés et les régions sans workloads.

Impact : Une Organization de 20 comptes avec Config activé sur 3 régions × 30 rules = ~3 600 $/mois juste en Config. Réduire à 1 région + 15 rules = ~900 $/mois. Économie : 2 700 $/mois = 32 400 $/an.

Pourquoi chaque paramètre compte :
  • Billing consolidé : sans consolidated billing, chaque compte paie ses propres volume tiers. Avec le billing consolidé, l'usage de S3, EC2, Lambda de tous les comptes est agrégé — déclenchant les paliers de réduction volume automatiquement (typiquement 5-15% d'économies).
  • Réduction des régions : chaque région gouvernée par Control Tower déploie Config Rules, CloudTrail, GuardDuty. Gouverner 4 régions au lieu de 16 divise le coût de monitoring par 4. Utiliser les SCPs pour interdire les régions non nécessaires.
  • Partage des RI/SP : les Reserved Instances et Savings Plans sont partagés automatiquement entre comptes de la même Organization (sauf désactivation explicite). Un RI acheté dans le compte Shared Services bénéficie à tous les comptes utilisant le même type d'instance.
  • Budgets par OU : sans budgets granulaires, une équipe peut consommer 80% du budget cloud sans que personne ne s'en aperçoive. AWS Budgets par OU + alertes à 50%, 80%, 100% permettent une réaction proactive.
  • Cost anomaly detection : AWS Cost Anomaly Detection utilise le ML pour détecter les pics anormaux. Un développeur qui lance 100 instances GPU par erreur est détecté en heures au lieu de jours — économisant potentiellement des milliers de dollars.

Script All-in-One — Hardening Organizations & Control Tower
Script exécutable pour appliquer toutes les best practices de gouvernance
⚠️
Prérequis : AWS CLI v2 installé, profil management-admin configuré avec accès au management account, droits organizations:*, cloudtrail:*, config:*, guardduty:*, securityhub:*.
Usage : chmod +x org-hardening.sh && ./org-hardening.sh
Bash — org-hardening.sh — Hardening complet Organizations & Control Tower
#!/usr/bin/env bash
# ============================================================
# org-hardening.sh — AWS Organizations & Control Tower Hardening
# Version 1.0 — Avril 2026
# ============================================================
set -euo pipefail

PROFILE="management-admin"
REGION="eu-west-1"
ORG_TRAIL_NAME="org-management-trail"
LOG_ARCHIVE_BUCKET="org-log-archive-$(aws sts get-caller-identity --profile $PROFILE --query Account --output text)"

echo "=========================================="
echo " AWS Organizations Hardening Script"
echo "=========================================="

# ─── 1. Vérifier l'Organization ───
echo "[S1] Vérification de l'Organization..."
ORG_ID=$(aws organizations describe-organization \
    --profile $PROFILE --region $REGION \
    --query "Organization.Id" --output text 2>/dev/null || echo "NONE")

if [ "$ORG_ID" = "NONE" ]; then
    echo "  ✗ Aucune Organization détectée. Création..."
    aws organizations create-organization --feature-set ALL \
        --profile $PROFILE --region $REGION
    ORG_ID=$(aws organizations describe-organization \
        --profile $PROFILE --region $REGION \
        --query "Organization.Id" --output text)
fi
echo "  ✓ Organization: $ORG_ID"

# ─── 2. Créer la structure d'OUs ───
echo "[S1] Création de la structure d'OUs..."
ROOT_ID=$(aws organizations list-roots \
    --profile $PROFILE --region $REGION \
    --query "Roots[0].Id" --output text)

create_ou() {
    local NAME=$1
    local PARENT=$2
    EXISTING=$(aws organizations list-organizational-units-for-parent \
        --parent-id "$PARENT" --profile $PROFILE --region $REGION \
        --query "OrganizationalUnits[?Name=='$NAME'].Id" --output text)
    if [ -z "$EXISTING" ]; then
        aws organizations create-organizational-unit \
            --parent-id "$PARENT" --name "$NAME" \
            --profile $PROFILE --region $REGION \
            --query "OrganizationalUnit.Id" --output text
    else
        echo "$EXISTING"
    fi
}

OU_SECURITY=$(create_ou "Security" "$ROOT_ID")
OU_INFRA=$(create_ou "Infrastructure" "$ROOT_ID")
OU_WORKLOADS=$(create_ou "Workloads" "$ROOT_ID")
OU_SANDBOX=$(create_ou "Sandbox" "$ROOT_ID")
OU_SUSPENDED=$(create_ou "Suspended" "$ROOT_ID")
OU_QUARANTINE=$(create_ou "Quarantine" "$ROOT_ID")

# Sous-OUs Workloads
OU_WL_PROD=$(create_ou "Production" "$OU_WORKLOADS")
OU_WL_STAGING=$(create_ou "Staging" "$OU_WORKLOADS")
OU_WL_DEV=$(create_ou "Development" "$OU_WORKLOADS")

echo "  ✓ OUs créées: Security, Infrastructure, Workloads(Prod/Staging/Dev), Sandbox, Suspended, Quarantine"

# ─── 3. SCPs essentiels ───
echo "[S2] Déploiement des SCPs..."

# SCP Deny Regions
SCP_DENY_REGIONS='{
  "Version":"2012-10-17",
  "Statement":[{
    "Sid":"DenyNonEURegions",
    "Effect":"Deny",
    "Action":"*",
    "Resource":"*",
    "Condition":{"StringNotEquals":{"aws:RequestedRegion":["eu-west-1","eu-central-1"]},"ForAnyValue:StringNotLike":{"aws:PrincipalArn":["arn:aws:iam::*:role/AWSControlTowerExecution","arn:aws:iam::*:role/OrganizationAccountAccessRole"]}}
  }]
}'

SCP_ID=$(aws organizations create-policy \
    --name "deny-non-eu-regions" \
    --type SERVICE_CONTROL_POLICY \
    --content "$SCP_DENY_REGIONS" \
    --description "Restreindre aux régions EU uniquement" \
    --profile $PROFILE --region $REGION \
    --query "Policy.PolicySummary.Id" --output text 2>/dev/null || \
    aws organizations list-policies --filter SERVICE_CONTROL_POLICY \
    --profile $PROFILE --region $REGION \
    --query "Policies[?Name=='deny-non-eu-regions'].Id" --output text)

aws organizations attach-policy \
    --policy-id "$SCP_ID" --target-id "$OU_WORKLOADS" \
    --profile $PROFILE --region $REGION 2>/dev/null || true

echo "  ✓ SCP deny-non-eu-regions attaché aux Workloads"

# ─── 4. CloudTrail Organization Trail ───
echo "[S8] Configuration CloudTrail Organization Trail..."
aws cloudtrail create-trail \
    --name "$ORG_TRAIL_NAME" \
    --s3-bucket-name "$LOG_ARCHIVE_BUCKET" \
    --is-organization-trail \
    --is-multi-region-trail \
    --enable-log-file-validation \
    --profile $PROFILE --region $REGION 2>/dev/null || true

aws cloudtrail start-logging --name "$ORG_TRAIL_NAME" \
    --profile $PROFILE --region $REGION 2>/dev/null || true
echo "  ✓ CloudTrail: $ORG_TRAIL_NAME"

# ─── 5. Budgets ───
echo "[S6] Configuration Budget avec alerte..."
MGMT_ACCOUNT=$(aws sts get-caller-identity \
    --profile $PROFILE --query Account --output text)

aws budgets create-budget \
    --account-id "$MGMT_ACCOUNT" \
    --budget '{
        "BudgetName":"org-monthly-total",
        "BudgetLimit":{"Amount":"5000","Unit":"USD"},
        "TimeUnit":"MONTHLY",
        "BudgetType":"COST"
    }' \
    --notifications-with-subscribers '[{
        "Notification":{"NotificationType":"ACTUAL","ComparisonOperator":"GREATER_THAN","Threshold":80,"ThresholdType":"PERCENTAGE"},
        "Subscribers":[{"SubscriptionType":"EMAIL","Address":"cloud-team@votre-domaine.com"}]
    }]' \
    --profile $PROFILE --region $REGION 2>/dev/null || true
echo "  ✓ Budget 5000 $/mois avec alerte à 80%"

echo ""
echo "=========================================="
echo " ✓ Hardening terminé"
echo "=========================================="
echo "Prochaines étapes manuelles :"
echo "  1. Activer Control Tower via la console"
echo "  2. Configurer IAM Identity Center"
echo "  3. Activer GuardDuty & Security Hub via delegated admin"
echo "  4. Appliquer les tag policies & backup policies"
