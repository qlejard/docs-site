Stratégie de déploiement recommandée

1
Quick Win — ~15 min, +25 pts. Valider le soft delete secure by default sur tous les vaults, revoir la rétention si nécessaire, vérifier les policies de backup VM par défaut et activer les alertes built-in Azure Monitor pour les échecs.
2
Protection de base — ~30 min. Configurer les policies de rétention (daily/weekly/monthly/yearly), activer le backup pour Azure Files et SQL, configurer Resiliency comme vue unifiée.
3
Sécurisation — ~1-2h. Activer immutability sur les vaults critiques, configurer MUA (Resource Guard), private endpoints, Azure Policy pour enforce backup.
4
BCDR complet — Variable. Configurer ASR (réplication VM), planifier les DR drills trimestriels, diagnostics vers Log Analytics, optimiser les coûts (archive tier, cross-region restore).

🚨
Priorité critique — Le Recovery Services Vault est le coffre-fort blindé de toute stratégie de sauvegarde Azure. Un vault mal configuré (LRS au lieu de GRS, rétention soft delete minimale non revue, pas d'immutabilité) expose l'entreprise à une perte totale des sauvegardes en cas de ransomware ou de sinistre régional. Sans vault correctement configuré, toutes les autres protections sont inutiles.
🏦
Analogie : Le Recovery Services Vault est un coffre-fort blindé. Vous y déposez vos sauvegardes comme des objets de valeur. La réplication de stockage (GRS) est la copie dans un coffre distant, le soft delete est la corbeille de 14 jours, et l'immutabilité est le scellé notarial qui empêche quiconque (même un admin) de détruire les sauvegardes.
💡 Pourquoi chaque paramètre compte

Chaque paramètre du vault conditionne la résilience de l'ensemble de la stratégie de sauvegarde. Le type de réplication (GRS vs LRS) détermine la survie des données en cas de sinistre régional, le soft delete protège contre les suppressions malveillantes (ransomware), et l'immutabilité empêche toute altération des points de restauration — même par un administrateur compromis.

Pourquoi c'est critique

⚠️
Ransomware : Sans soft delete secure by default correctement piloté et sans immutabilité, un attaquant qui compromet un compte admin peut supprimer toutes les sauvegardes en une commande. Les restaurations deviennent impossibles, la négociation de rançon est la seule option.
⚠️
Sinistre régional : Un vault en LRS avec des sauvegardes dans une seule région est perdu si la région tombe. La combinaison GRS + Cross-Region Restore permet de restaurer depuis la région pairée.

Pièges fréquents

💣
Piège n°1 — Storage replication non modifiable après premier backup : Le type de réplication (LRS/GRS/ZRS) ne peut plus être changé une fois qu'un backup item est enregistré dans le vault. Il faut choisir GRS avant de configurer le moindre backup.
💣
Piège n°2 — Immutability locked est irréversible : Passer en mode « locked » est irréversible. Si les règles de rétention sont trop courtes, vous ne pourrez jamais les réduire. Validez soigneusement les policies de rétention avant de verrouiller.
💣
Piège n°3 — RSV ≠ Backup Vault : Le Recovery Services Vault (RSV) gère VM/SQL/Files. Le Backup Vault gère Blob/Disk/PostgreSQL. Ils ne sont pas interchangeables. Créez le bon type dès le départ.

Configuration — Créer et sécuriser un Recovery Services Vault

Az CLI — Création et configuration du vault
# ── 1. Créer le Recovery Services Vault en GRS ──
az backup vault create \
  --resource-group "rg-backup-prod" \
  --name "rsv-prod-westeurope" \
  --location "westeurope"

# ── 2. Vérifier et confirmer la réplication GRS (défaut) ──
az backup vault backup-properties show \
  --resource-group "rg-backup-prod" \
  --name "rsv-prod-westeurope" \
  --query "storageModelType"

# ── 3. Activer Cross-Region Restore (nécessite GRS) ──
az backup vault backup-properties set \
  --resource-group "rg-backup-prod" \
  --name "rsv-prod-westeurope" \
  --cross-region-restore-flag "true"

# ── 4. Si l'état reste administrable dans votre région/API : normaliser le soft delete et régler la rétention ──
az backup vault backup-properties set \
  --resource-group "rg-backup-prod" \
  --name "rsv-prod-westeurope" \
  --soft-delete-feature-state "AlwaysOn" \
  --soft-delete-duration "30"

# ── 5. Activer l'immutabilité (mode unlocked) ──
az backup vault backup-properties set \
  --resource-group "rg-backup-prod" \
  --name "rsv-prod-westeurope" \
  --immutability-state "Unlocked"

# ── 6. Configurer les diagnostics vers Log Analytics ──
az monitor diagnostic-settings create \
  --name "diag-rsv-prod" \
  --resource "/subscriptions/{sub-id}/resourceGroups/rg-backup-prod/providers/Microsoft.RecoveryServices/vaults/rsv-prod-westeurope" \
  --workspace "/subscriptions/{sub-id}/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-backup-prod" \
  --logs '[{"category":"AzureBackupReport","enabled":true},{"category":"CoreAzureBackup","enabled":true},{"category":"AddonAzureBackupJobs","enabled":true},{"category":"AddonAzureBackupAlerts","enabled":true},{"category":"AddonAzureBackupPolicy","enabled":true},{"category":"AddonAzureBackupStorage","enabled":true},{"category":"AddonAzureBackupProtectedInstance","enabled":true}]'
Bicep — Recovery Services Vault avec best practices
// ── Recovery Services Vault ── GRS + soft delete durci + immutabilité ──
resource rsv 'Microsoft.RecoveryServices/vaults@2024-04-01' = {
  name: 'rsv-prod-westeurope'
  location: 'westeurope'
  sku: {
    name: 'RS0'
    tier: 'Standard'
  }
  properties: {
    publicNetworkAccess: 'Disabled'      // Private endpoints uniquement
    securitySettings: {
      immutabilitySettings: {
        state: 'Unlocked'               // Passer en Locked après validation

      softDeleteSettings: {
                softDeleteState: 'AlwaysOn'     // Normalisation explicite si l'état n'est pas déjà imposé

    }
  }
}

🚨
Priorité critique — La sauvegarde des VMs Azure est la première ligne de défense. Par défaut, aucune VM n'est sauvegardée automatiquement. Sans policy de backup explicitée, la VM et ses données peuvent être perdues irrémédiablement en cas de suppression accidentelle, corruption ou ransomware.
📸
Analogie : Le backup VM fonctionne comme un appareil photo automatique. L'instant restore snapshot est une photo récente gardée sur place (restauration en minutes), le vault tier est l'album photo archivé au coffre-fort (restauration en heures mais sûre). L'Enhanced policy est l'appareil haut de gamme : plusieurs photos par jour, plus de contrôle.
💡 Pourquoi chaque paramètre compte

La protection des VMs repose sur le bon calibrage de la fréquence de backup (RPO), de la rétention (durée de récupération) et du type de policy (Standard vs Enhanced). Un RPO de 24h signifie accepter de perdre une journée de travail ; une rétention trop courte expose à l'impossibilité de restaurer après détection tardive d'un ransomware. Le choix du consistency level impacte directement la fiabilité des restaurations de bases de données.

Pourquoi c'est critique

⚠️
Pas de backup = pas de restauration. Azure ne sauvegarde rien par défaut. La suppression accidentelle d'un resource group, un az vm delete --force mal ciblé, ou un ransomware qui chiffre les disques — sans backup, ces événements sont irréversibles.
📊
Enhanced vs Standard Policy :
  • Standard : 1 backup/jour, 2 jours d'instant restore, pas de multi-disk consistency garanti
  • Enhanced : jusqu'à 1 backup toutes les 4h, 30 jours d'instant restore max, support Trusted Launch VMs, multi-disk crash consistent
Pour les VMs critiques, Enhanced est le choix à privilégier.

Pièges fréquents

💣
Piège n°1 — VM déployée sans backup : La création d'une VM via CLI/Bicep ne configure aucun backup automatiquement. Utilisez Azure Policy Deploy Azure Backup on VMs pour garantir que chaque VM est protégée.
💣
Piège n°2 — Instant restore snapshots = coût invisible : Les snapshots d'instant restore sont stockés dans le resource group de la VM (pas dans le vault). Plus la rétention est longue, plus le coût snapshot augmente. 5 jours est le Sweet spot coût/rapidité.
💣
Piège n°3 — Linux file-system consistent ≠ application-consistent : Sans scripts pré/post, le backup Linux est file-system consistent (crash-consistent). Pour les bases de données (MySQL, PostgreSQL), il faut des scripts custom pour garantir la cohérence applicative.

az backup policy create \
  --resource-group "rg-backup-prod" \

  "instantRpRetentionRangeInDays": 5,
  "schedulePolicy": {
    "schedulePolicyType": "SimpleSchedulePolicyV2",

  "retentionPolicy": {
    "retentionPolicyType": "LongTermRetentionPolicy",
    "dailySchedule": { "retentionTimes": ["2024-01-01T20:00:00Z"], "retentionDuration": { "count": 30, "durationType": "Days" } },
    "weeklySchedule": { "daysOfTheWeek": ["Sunday"], "retentionTimes": ["2024-01-01T20:00:00Z"], "retentionDuration": { "count": 12, "durationType": "Weeks" } },

⚠️

Le choix entre les trois mécanismes de backup (Files snapshot, Blob operational, Blob vaulted) dépend du type de stockage et du niveau de protection requis. Une confusion entre backup opérationnel (PITR dans le même compte) et backup vaulted (copie isolée dans un vault) peut laisser des données sans protection en cas de suppression du compte de stockage source.

Azure Site Recovery (ASR) Site Recovery
Réplication VM (Azure-to-Azure, VMware, Hyper-V), failover, failback, recovery plans, network mapping
🚨
Priorité critique — Azure Site Recovery est le site miroir de votre infrastructure. Sans ASR, une panne régionale Azure signifie un arrêt complet de l'activité jusqu'à la restauration manuelle des sauvegardes — potentiellement des heures voire des jours d'indisponibilité. 🏭
Analogie : ASR est un site miroir. Imaginez une usine jumelle dans une autre ville, synchronisée en permanence. Si l'usine principale brûle (sinistre régional), la jumelle prend le relais immédiatement (failover). Après réparation, la production revient à l'original (failback). Le test failover est l'exercice incendie : on teste sans affecter la production.
💡 Pourquoi chaque paramètre compte

Chaque paramètre ASR traduit un compromis entre coût de réplication et temps de reprise. La fréquence de réplication (RPO 5-15 min) détermine la perte maximale de données, le recovery plan orchestre l'ordre de démarrage des VMs (respect des dépendances applicatives), et le test failover valide que le plan fonctionne réellement — sans test, un plan de reprise n'est qu'un document théorique.

💣

Backup Policies & Scheduling Azure Backup
Policy types (Enhanced vs Standard), scheduling, retention rules, tiering, best practices par workload
⚠️
Priorité haute — Les policies de backup définissent la fréquence, la rétention et le tiering des sauvegardes. Une policy mal calibrée = soit un RPO trop long (perte de données), soit un coût excessif (stockage inutile). Chaque workload nécessite une policy adaptée à sa criticité.
📅
Analogie : Les policies sont les règles de l'agenda de sauvegarde. La Standard policy est un agenda papier simple (1 photo/jour, conservée 30 jours). L'Enhanced policy est un agenda numérique intelligent (photos toutes les 4h, conservation différenciée, archivage automatique).
💡 Pourquoi chaque paramètre compte

Les policies définissent le contrat de protection de chaque workload. Une fréquence inadaptée (1x/jour sur une VM Tier 0) crée un RPO de 24h — inacceptable pour les systèmes critiques. Une rétention excessive (10 ans partout) génère des coûts de stockage disproportionnés. L'alignement policy ↔ criticité ↔ budget est la clé d'une stratégie durable.

Pourquoi c'est important

⚠️
Archive tier = économies sur la rétention longue : le stockage archive est nettement moins cher que le vault tier standard, mais le gain exact dépend du niveau de redondance et du volume archivé. Pour les points de restauration >6 mois, migrer vers l'archive tier peut réduire sensiblement la facture Backup, sous réserve d'accepter les contraintes de conservation minimale et de réhydratation. Attention : la restauration depuis l'archive prend 24-48h (tier réhydratation).

Pièges fréquents

💣
Piège n°1 — Default policy pour tout : La policy par défaut (DefaultPolicy : daily, 30j instant restore 2j) est un minimum vital. L'appliquer à des VMs critiques (Tier 0/1) est dangereux : RPO 24h, pas de weekly/monthly, pas d'archive tier.
💣
Piège n°2 — Rétention sans archive = explosion du coût : Conserver 10 ans de backups en vault tier standard coûte une fortune. Activer le smart tiering ou configurer manuellement le déplacement vers l'archive tier après 3-6 mois.
💣
Piège n°3 — Instant restore trop long = coût snapshot : Les snapshots d'instant restore sont stockés en managed snapshots (coût élevé). Au-delà de 7 jours, le coût des snapshots dépasse souvent celui du vault tier.

Security & Compliance Inclus Azure Add-on
MUA (Resource Guard), soft delete obligatoire, immutable vaults, CMK, private endpoints, Azure Policy, anti-ransomware
🚨
Priorité CRITIQUE — Les backups sont la dernière ligne de défense contre les ransomwares. Si un attaquant compromet un compte admin et supprime les backups, la récupération est impossible. La sécurité du vault est aussi importante que la sécurité des données elles-mêmes.
🔒
Analogie : Le vault est un coffre-fort dans une banque. Soft delete = la banque garde votre or 14 jours avant de le fondre (suppression accidentelle). Immutability = le coffre est scellé, même le directeur ne peut pas l'ouvrir avant l'échéance. MUA (Resource Guard) = il faut deux clés simultanées (deux personnes) pour ouvrir le coffre.
💡 Pourquoi chaque paramètre compte

Chaque couche de sécurité du vault bloque un vecteur d'attaque spécifique. Le soft delete protège contre la suppression accidentelle ou malveillante, l'immutabilité empêche la modification des points de restauration, et le MUA (Resource Guard) impose une deuxième approbation pour les opérations destructives — rendant impossible pour un seul compte compromis de détruire toutes les sauvegardes.

Couches de sécurité Azure Backup

Paramètre Défaut Azure Quick Win Avancé Best Practice
Soft Delete Activé (14j par défaut, mode imposé selon vault/région) Vérifier le mode secure by default Rétention 30-180j Mode imposé ou normalisé + rétention 30-180j
Immutability Désactivé Enable (unlocked) Locked immutability Locked + durée > rétention max policy
MUA (Resource Guard) Désactivé Créer Resource Guard Associer au vault (sub différente) Resource Guard + JIT PIM + subscription isolée
Encryption PMK (Platform-Managed Keys) PMK (suffisant) CMK (Customer-Managed Keys) CMK + Key Vault RBAC + rotation auto
Private Endpoints Public access Private endpoint PE + deny public access PE + deny public + DNS privé
RBAC Classic (Backup roles) Backup Operator/Reader Custom roles least-privilege PIM + JIT + Conditional Access
Azure Policy Aucune Audit: VMs sans backup DeployIfNotExists: auto-protect + Audit: écarts soft delete / immutabilité

Pourquoi c'est important

🚨
Scénario ransomware : L'attaquant compromet un compte Backup Contributor → tente de réduire la protection soft delete ou de supprimer tous les recovery points → chiffre les VMs → aucune restauration possible.
Avec MUA + immutability + soft delete secure by default, ce scénario devient nettement plus difficile : l'attaquant ne peut ni désactiver la protection ni supprimer les points de restauration.

Pièges fréquents

💣
Piège n°1 — Penser que le soft delete reste optionnel partout : Sur Recovery Services Vault, le mode secure by default rend la désactivation indisponible via le portail dans les régions onboardées. Sur Backup Vault, le comportement dépend encore de la région. Le vrai levier opérationnel à piloter est surtout la rétention 14 à 180 jours, avec éventuellement une normalisation explicite à AlwaysOn quand l'API le permet.
💣
Piège n°2 — Resource Guard dans la même subscription : Si le Resource Guard est dans la même subscription que le vault, un Owner de cette subscription peut quand même contourner la MUA. Toujours placer le Resource Guard dans une subscription isolée gérée par une équipe différente (Security team).
💣
Piège n°3 — CMK sans rotation : Le chiffrement CMK sans rotation automatique de la clé Key Vault est un faux sentiment de sécurité. Configurer la rotation automatique dans Key Vault (tous les 90 jours recommandé).

🚨
Priorité CRITIQUE — Un plan BCDR sans test régulier est un document marketing, pas un plan de reprise. 67% des entreprises qui pensent avoir un DR fonctionnel échouent lors du premier sinistre réel. Tester = garantir que la reprise fonctionne avant d'en avoir besoin.
🎯
Analogie : Le plan BCDR est un exercice incendie pour votre SI. RTO = combien de temps pour évacuer le bâtiment et reprendre le travail (durée d'interruption tolérée). RPO = combien de documents non sauvegardés vous acceptez de perdre. DR drill = l'exercice d'évacuation trimestriel — sans exercice, personne ne saura où aller le jour J.
💡 Pourquoi chaque paramètre compte

La stratégie BCDR transforme des composants techniques (backup, ASR, policies) en un plan de continuité d'activité vérifiable. Le RTO et le RPO définissent les engagements de service, les DR drills valident que ces engagements sont tenables, et la classification par tier garantit que les investissements de protection sont proportionnels à l'impact business de chaque workload.

Pourquoi c'est important

🚨
Cas réel : Une entreprise avait un « plan DR » documenté mais jamais testé. Lors d'une panne régionale, le failover ASR a échoué : les recovery plans référençaient des VMs supprimées, les scripts post-failover avaient des credentials expirés, et le RTO réel a été de 72h au lieu de 4h.

Pièges fréquents

💣
Piège n°1 — Test failover oublié : ASR réplique les données mais ne garantit pas que le failover fonctionne. Les recovery plans peuvent référencer des ressources obsolètes. Tester trimestriellement avec un test failover (isolé, sans impact prod).
💣
Piège n°2 — Restore non testé : Un backup réussi ≠ un restore réussi. Les backups peuvent être corrompus, les disques restaurés incompatibles, les applications incapables de démarrer. Tester un restore complet (VM + app + validation) chaque trimestre.
💣
Piège n°3 — Région DR sans quotas : La région de failover (ex: North Europe) doit avoir les quotas suffisants (vCPUs, public IPs, NICs) pour accueillir les workloads. Vérifier les quotas avant un sinistre.

Pièges fréquents

💣
Piège n°1 — GRS+CRR partout : Le CRR n'est pertinent que pour les workloads qui exigent une restauration inter-région. Il impose du GRS/RA-GRS sur le vault et augmente donc le coût de stockage. Pour des VMs Tier 3 (dev/test), le LRS peut suffire si l'analyse de risque n'exige pas de reprise régionale.
💣
Piège n°2 — Snapshots non nettoyés : Les managed snapshots d'instant restore persistent même quand on diminue la rétention. Vérifier que les anciens snapshots sont bien purgés après modification de policy.
💣
Piège n°3 — Coût de restauration archive : La restauration depuis l'archive tier entraîne un coût de réhydratation et de retrieval, avec un niveau de prix qui varie selon la priorité choisie et le volume restauré. Anticiper : la restauration depuis archive est lente (24-48h) et coûteuse.

Script All-in-One — Configuration Backup & BCDR
Script exécutable en une seule fois pour appliquer toutes les best practices
⚠️
Pré-requis : Modifier les variables SUB_ID, SECURITY_SUB_ID, noms de resource groups et vault avant exécution. Tester en environnement dev/test avant d'appliquer en production. Certaines opérations sont irréversibles (état AlwaysOn lorsqu'il est explicitement appliqué, locked immutability).
Az CLI & PowerShell — Script All-in-One
#!/bin/bash
# ============================================================
# Azure Backup & BCDR — Script All-in-One Best Practices
# ============================================================
# MODIFIER CES VARIABLES AVANT EXÉCUTION
SUB_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
SECURITY_SUB_ID="yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy"
RG="rg-backup-prod"
VAULT="rsv-prod-westeurope"
LOCATION="westeurope"
LAW="law-backup-prod"

az account set --subscription "$SUB_ID"

# ═══════════ S1 — Recovery Services Vault ═══════════
echo "[S1] Création du RSV..."
az backup vault create \
  --resource-group "$RG" --name "$VAULT" --location "$LOCATION"

# GRS + CRR
az backup vault backup-properties set \
  --resource-group "$RG" --name "$VAULT" \
  --backup-storage-redundancy "GeoRedundant"
az backup vault backup-properties set \
  --resource-group "$RG" --name "$VAULT" \
  --cross-region-restore-flag "True"

# ═══════════ S8 — Security Hardening ═══════════
echo "[S8] Hardening sécurité..."
# Soft delete : vérifier secure-by-default, puis normaliser si l'état reste éditable
az backup vault backup-properties set \
  --resource-group "$RG" --name "$VAULT" \
  --soft-delete-feature-state "AlwaysOn" \
  --soft-delete-duration 30

# Enhanced security
az backup vault backup-properties set \
  --resource-group "$RG" --name "$VAULT" \
  --enhanced-security-state "Enabled"

# Immutability (unlocked — passer en locked seulement après validation)
az backup vault backup-properties set \
  --resource-group "$RG" --name "$VAULT" \
  --immutability-state "Unlocked"

# ═══════════ S7 — Monitoring ═══════════
echo "[S7] Configuration monitoring..."
az monitor log-analytics workspace create \
  --resource-group "$RG" --workspace-name "$LAW" \
  --location "$LOCATION" --retention-in-days 365

VAULT_ID=$(az backup vault show --resource-group "$RG" --name "$VAULT" --query id -o tsv)
LAW_ID=$(az monitor log-analytics workspace show --resource-group "$RG" --workspace-name "$LAW" --query id -o tsv)

az monitor diagnostic-settings create \
  --name "diag-rsv-backup" \
  --resource "$VAULT_ID" \
  --workspace "$LAW_ID" \
  --logs '[{"categoryGroup":"allLogs","enabled":true}]'

# Action Group
az monitor action-group create \
  --resource-group "$RG" --name "ag-backup-alerts" \
  --short-name "BackupAlert" \
  --action email admin-backup admin-backup@contoso.com

# ═══════════ S6 — Enhanced Backup Policy (Tier 0/1) ═══════════
