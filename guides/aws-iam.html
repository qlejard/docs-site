Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de configuration IAM AWS
📋
Ce document est un référentiel technique complet couvrant toutes les configurations AWS IAM & Identity Center avec les best practices AWS, CIS Benchmark v7.0.0 et AWS Well-Architected Framework — Security Pillar. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un compte AWS.

Analogie générale — IAM comme système de contrôle d'accès d'un immeuble

🏢
AWS IAM = le registre d'accès du bâtiment. Il définit qui a le droit d'entrer et dans quelles pièces.
Rôle IAM = un badge temporaire. On vous le prête pour une mission précise, il expire automatiquement.
SCP = le règlement de l'immeuble. Des restrictions globales que personne ne peut contourner, même le propriétaire.
Permission Boundary = un plafond de verre. Même si votre policy dit oui, la boundary peut dire non.
MFA = double vérification d'identité. Badge + code : deux preuves pour entrer.
Access Analyzer = le détective des portes ouvertes. Il parcourt l'immeuble et signale les accès excessifs.

IAM Users, Groups & Policies Gratuit
Création d'utilisateurs, groupes, policies IAM — Principle of Least Privilege — 20 pts
🚨
Scénario d'attaque : Un développeur possède la policy AdministratorAccess « pour aller plus vite ». Un attaquant compromet ses access keys (commitées dans un repo Git public). En 4 minutes, il crée 200 instances EC2 pour du cryptomining, supprime les CloudTrail logs, et crée un nouvel IAM user de backdoor. Coût : ~45 000 $ en 48h avant détection.

Analogie : IAM = le registre d'accès du bâtiment. Donner AdministratorAccess à un développeur, c'est donner le pass général de l'immeuble à un livreur — il peut ouvrir toutes les portes, y compris la salle des coffres.

💡
Pourquoi chaque paramètre IAM Users & Policies compte :
  • Root access keys : le compte root a un accès illimité et ne peut pas être restreint par SCPs. Des access keys root compromises = perte totale du compte, sans recours. Source : CIS AWS Foundations Benchmark v7.0.0, contrôle 2.4.
  • Password policy (longueur, complexité, rotation) : un mot de passe de 8 caractères sans complexité se brute-force en ~2h (hashcat, GPU moderne). 14 caractères + complexité = ~millions d'années. Rotation 90j limite la fenêtre d'exploitation d'un credential volé. Sources : CIS 2.8-2.9, NIST SP 800-63B (alternative : 0 rotation + MFA obligatoire).
  • Inline policies sur users : les inline policies sont invisibles dans les audits IAM Access Analyzer et ne suivent pas le cycle de vie des managed policies. Elles créent des « permissions fantômes » difficiles à détecter. Source : AWS IAM Best Practices, Well-Architected Framework SEC02-BP02.
  • Policies directement sur users : sans groups, chaque user est un cas particulier. À 50 users, l'audit devient impossible. Les groups permettent le contrôle centralisé et la revue. Source : CIS 2.13.
  • Wildcard "Resource": "*" : permet d'agir sur TOUTES les ressources du compte. Combiné avec "Action": "ec2:*", un user peut supprimer toutes les instances. Least privilege = spécifier les ARN exacts. Source : AWS SEC02-BP04.
  • Access keys rotation ≤90j : une clé de >90 jours a statistiquement plus de chances d'avoir été exposée (logs, scripts, anciens employés). CIS 2.12 exige ≤90j. Préférer les rôles IAM quand possible.
  • Conditions IAM : sans aws:MultiFactorAuthPresent, un credential volé suffit. Sans aws:SourceIp, l'accès est possible depuis n'importe quelle IP. Les conditions sont la dernière ligne de défense. Source : AWS IAM Policy Conditions Best Practices.

1.3 — Anti-patterns IAM courants

⚠️
Les 7 anti-patterns IAM les plus fréquents :
  1. AdministratorAccess sur des humains — 60% des comptes audités ont au moins 1 user avec admin full. Utiliser des rôles à durée limitée via Identity Center.
  2. "Resource": "*" avec des actions destructives (ec2:Terminate*, s3:Delete*, iam:Delete*) — spécifier les ARN.
  3. Access keys pour le compte root — le root est pour les tâches root-only (account settings, support plan). Jamais d'access keys.
  4. Inline policies — invisibles à l'audit, non réutilisables, non versionnées. Utiliser des customer-managed policies.
  5. Policies directement sur les users — impossibles à auditer à l'échelle. Toujours passer par des groups.
  6. Pas de MFA — un mot de passe seul est insuffisant. MFA obligatoire pour tous les accès console (voir S4).
  7. Access keys non tournées depuis >1 an — fenêtre d'exploitation massive. Rotation ≤90j ou remplacement par rôles.

1.4 — Configuration — AWS CLI

Bash — Durcir la password policy IAM
# Durcir la password policy IAM (CIS 2.8-2.9)
aws iam update-account-password-policy \
    --minimum-password-length 14 \
    --require-symbols \
    --require-numbers \
    --require-uppercase-characters \
    --require-lowercase-characters \
    --max-password-age 90 \
    --password-reuse-prevention 24 \
    --allow-users-to-change-password \
    --profile admin-secure \
    --region eu-west-1

# ⚠️ Action destructive — Lister les access keys root

    --profile root-account \

# Supprimer chaque access key root identifiée

    --access-key-id AKIAIOSFODNN7EXAMPLE \
    --profile root-account
Bash — Identifier les users avec policies directes
# Lister les users ayant des policies attachées directement (CIS 2.13)
aws iam list-users \
    --profile audit-readonly \
    --region eu-west-1 \
    --query 'Users[].UserName' \

        --user-name "$user" \

IAM Roles & Trust Policies Gratuit
Rôles pour services, cross-account, trust policies, permission boundaries — 15 pts
🚨
Scénario d'attaque : Une trust policy avec "Principal": {"AWS": "*"} permet à n'importe quel compte AWS au monde d'assumer le rôle. Un attaquant scanne les rôles ouverts avec iam:ListRoles, assume le rôle depuis son propre compte, et obtient un accès complet à votre S3, DynamoDB, ou EC2. L'accès est légitime du point de vue AWS — pas d'alerte par défaut.

Analogie : Un rôle IAM = un badge temporaire. Vous le prêtez à un visiteur pour une mission précise (livraison étage 3), et il expire automatiquement. La trust policy est la liste des personnes autorisées à emprunter ce badge. "Principal": "*" = n'importe qui dans la rue peut prendre votre badge.

💡
Pourquoi chaque paramètre des IAM Roles compte :
  • Trust policy wildcard "*" : c'est l'équivalent de laisser la porte d'entrée ouverte. N'importe quel des ~1M de comptes AWS peut assumer votre rôle. Access Analyzer détecte ce cas. Source : AWS Well-Architected SEC03-BP05.
  • ExternalId : sans ExternalId, un service tiers SaaS (ex: Datadog) peut être exploité par un attaquant qui connaît votre Account ID (« confused deputy problem »). L'ExternalId agit comme un mot de passe partagé entre vous et le tiers. Source : AWS IAM External ID documentation.
  • MaxSessionDuration : une session de 12h signifie qu'un token volé est valide 12h. Avec 1h, la fenêtre d'exploitation est réduite à 1/12e. Pour l'automation, 15 min suffisent souvent.
  • Permission boundaries : le plafond de verre. Même si un développeur crée un rôle avec AdministratorAccess, la boundary limite les permissions effectives. Indispensable pour la délégation IAM sécurisée. Source : AWS SEC03-BP07.
  • Rôles vs access keys : un rôle IAM sur un EC2 (via instance profile) génère des credentials temporaires (durée 1-6h, rotation automatique). Des access keys sont permanentes et statiques — si elles fuitent, l'accès est compromis jusqu'à révocation manuelle.

2.4 — Configuration — Terraform

HCL — Terraform — Rôle EC2 avec instance profile + boundary
# Permission boundary — plafond pour les rôles délégués
resource "aws_iam_policy" "dev_boundary" {
  name   = "DevPermissionBoundary"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid      = "AllowDevServices"
        Effect   = "Allow"
        Action   = ["s3:*", "dynamodb:*", "lambda:*", "logs:*", "cloudwatch:*"]
        Resource = "*"
      },
      {
        Sid      = "DenyIAMEscalation"
        Effect   = "Deny"
        Action   = ["iam:CreateUser", "iam:CreateRole", "iam:PutRolePolicy",
                     "iam:AttachRolePolicy", "iam:DeleteRolePermissionsBoundary",
                     "organizations:*", "account:*"]
        Resource = "*"
      }
    ]
  })
}

Cross-Account & Federation Gratuit
SAML 2.0, OIDC providers, cross-account access, AWS STS regional endpoints — 10 pts
🚨
Scénario d'attaque : Un partenaire externe dispose d'un IAM role cross-account avec "Principal": {"AWS": "arn:aws:iam::999999999999:root"} — ce qui donne à tous les utilisateurs du compte partenaire un accès complet. Le contrat se termine, mais le role reste. Un employé de l'ex-partenaire assume le rôle 6 mois plus tard et exfiltre les données.

Analogie : La fédération = le badge visiteur contrôlé. Au lieu de donner un badge permanent (IAM user), vous donnez un badge temporaire validé par la sécurité de l'entreprise partenaire (IdP). Le badge expire automatiquement. Le cross-account = donner un accès contrôlé à un étage précis de votre immeuble, avec une durée limitée et un contrôle d'identité à chaque entrée.

Pourquoi la fédération et le cross-account sécurisé sont critiques :
  • Zéro IAM user humain : avec SAML/OIDC, les identités sont gérées par l'IdP corporate. Déprovisioning automatique quand l'employé quitte l'entreprise. Plus de « zombie accounts ».

  • Confused deputy : sans ExternalId, un tiers malveillant peut forcer un service AWS à assumer votre rôle cross-account. L'ExternalId est un secret partagé qui protège contre ce vecteur.
  • STS régional : le global endpoint sts.amazonaws.com résout vers us-east-1. Si cette région a un incident, toutes vos authentifications tombent. Configurer sts.eu-west-1.amazonaws.com comme endpoint principal.

Pourquoi le monitoring IAM est non négociable :
  • CloudTrail Organization trail : un seul trail capture tous les events de tous les comptes. Logs stockés dans un bucket S3 du compte Log Archive avec lifecycle policy (1 an standard, 7 ans Glacier). CIS 4.1.
  • CIS Monitoring Alarms (3.1-3.14) : 14 alarmes CloudWatch basées sur des metric filters CloudTrail. Root login, IAM policy changes, MFA console login sans MFA, VPC changes, etc. Chaque alarme coûte ~$0.10/mois.
  • EventBridge > CloudWatch pour le temps réel : EventBridge déclenche en <1 minute contre 5-15 min pour CloudWatch metric filters. Privilégier EventBridge pour les alertes critiques.

Pièges courants — monitoring IAM :
  1. CloudTrail uniquement dans une région : un attaquant lance des ressources dans ap-southeast-1 = invisible. Multi-region obligatoire.
  2. Logs sans protection : le bucket S3 des logs doit avoir versioning, MFA Delete, encryption KMS, et bucket policy deny delete.
  3. Alert fatigue : trop d'alarmes non triées = personne ne les regarde. Prioriser les 5 alarmes critiques (root, IAM changes, policy changes).
  4. Config sans remediation : AWS Config détecte la non-conformité mais ne corrige pas. Ajouter des SSM Automation remediations.

10.2 — Optimisations recommandées

💡
Réduire les coûts sans réduire la sécurité :
  • CloudTrail S3 lifecycle : 90j Standard → 1 an Infrequent Access → 7 ans Glacier Deep Archive. Réduit le stockage de ~80%.
  • CloudTrail Lake vs S3 : CloudTrail Lake (requêtes SQL) coûte $2.50/GB scanné. Pour l'analyse ponctuelle, préférer Athena sur S3 ($5/TB scanné).
  • Config rules : limiter aux rules IAM essentielles (10-15 rules) plutôt que le conformance pack complet (200+ rules). Réduit les coûts de ~90%.
  • Security Hub : désactiver les standards non nécessaires (PCI DSS si non concerné). Garder CIS + AWS Foundational.
  • Parameter Store vs Secrets Manager : pour les secrets non critiques (API keys, configs), utiliser Parameter Store SecureString (gratuit) au lieu de Secrets Manager ($0.40/mois).
  • SCPs pour le cost control : deny les services coûteux non approuvés (SageMaker, Redshift) = prévention des coûts involontaires + sécurité.

SolutionRaison de l'exclusionQuand reconsidérer
IAM Users avec access keys pour tout Anti-pattern majeur. Les access keys sont des credentials permanentes, impossibles à auditer finement, et la première cause de compromission AWS. IAM Identity Center + rôles temporaires couvrent 100% des cas. Jamais. Privilégier systématiquement les rôles et OIDC.

Entreprises avec exigences PAM (Privileged Access Management) réglementaires strictes nécessitant un vault externe.
Policy allow-list SCPs (tout bloquer sauf) Approche fragile : chaque nouveau service AWS doit être explicitement autorisé. Casse fréquemment les services dépendants (ex : Lambda a besoin de CloudWatch Logs). La stratégie deny-list est plus résiliente. Environnements ultra-restrictifs (gouvernemental, défense) où chaque service doit être approuvé individuellement.
AWS Control Tower uniquement Control Tower est un orchestrateur de guardrails, pas un remplacement d'IAM. Il simplifie le déploiement initial mais ne dispense pas de comprendre et configurer IAM, SCPs, et Identity Center manuellement. Nouvel environnement multi-compte à créer from scratch — Control Tower accélère le bootstrapping.
HashiCorp Vault pour tous les secrets AWS Vault ajoute une infrastructure à gérer (HA, unsealing, backup). Pour les secrets AWS natifs (BDD RDS, clés API), Secrets Manager avec rotation Lambda est plus simple et intégré. Script All-in-One — Audit & Hardening IAM
Script exécutable pour auditer et durcir la configuration IAM
⚠️
Mode audit (lecture seule) — Ce script ne modifie rien. Il génère un rapport JSON avec les findings. Exécuter avec un profil audit-readonly disposant de SecurityAudit + ViewOnlyAccess.
Bash — iam-audit.sh (audit complet IAM)
#!/usr/bin/env bash
# iam-audit.sh — Audit IAM complet (lecture seule)
# Usage : ./iam-audit.sh --profile audit-readonly --region eu-west-1
set -euo pipefail

PROFILE="${1:---profile audit-readonly}"
REGION="${2:---region eu-west-1}"
OUTPUT="iam-audit-$(date +%Y%m%d-%H%M%S).json"

echo "🔍 IAM Audit — $(date)"
echo "========================================"

# S1 — Users sans MFA, access keys > 90j
echo "[S1] Credential Report..."
aws iam generate-credential-report $PROFILE > /dev/null 2>&1
sleep 5
CRED_REPORT=$(aws iam get-credential-report $PROFILE --output json | jq -r '.Content' | base64 -d)
USERS_NO_MFA=$(echo "$CRED_REPORT" | awk -F',' 'NR>1 && $4=="true" && $8=="false" {print $1}')
OLD_KEYS=$(echo "$CRED_REPORT" | awk -F',' -v d=$(date -d '-90 days' +%Y-%m-%d) 'NR>1 && $9=="true" && $10<d {print $1": key1="$10}')
echo "  Users sans MFA: $(echo "$USERS_NO_MFA" | grep -c . || echo 0)"
echo "  Access keys > 90j: $(echo "$OLD_KEYS" | grep -c . || echo 0)"

# S2 — Roles avec trust policy trop permissive
echo "[S2] Trust policies..."
WILD_TRUST=$(aws iam list-roles $PROFILE --output json | \
    jq -r '.Roles[] | select(.AssumeRolePolicyDocument.Statement[].Principal == "*") | .RoleName')
echo "  Roles avec Principal \"*\": $(echo "$WILD_TRUST" | grep -c . || echo 0)"

# S4 — Root MFA
echo "[S4] Root account MFA..."
ROOT_MFA=$(aws iam get-account-summary $PROFILE --output json | jq '.SummaryMap.AccountMFAEnabled')
echo "  Root MFA enabled: $ROOT_MFA"

# S5 — SCPs actives
echo "[S5] Service Control Policies..."
SCP_COUNT=$(aws organizations list-policies --filter SERVICE_CONTROL_POLICY $PROFILE --output json 2>/dev/null | jq '.Policies | length' || echo "N/A (pas management account)")
echo "  SCPs actives: $SCP_COUNT"

# S6 — Access Analyzer findings
echo "[S6] Access Analyzer findings..."
for ANALYZER in $(aws accessanalyzer list-analyzers $PROFILE $REGION --output json | jq -r '.analyzers[].arn'); do
    FINDINGS=$(aws accessanalyzer list-findings --analyzer-arn "$ANALYZER" \
        --filter '{"status":{"eq":["ACTIVE"]}}' $PROFILE $REGION --output json | jq '.findings | length')
    echo "  Analyzer $(basename $ANALYZER): $FINDINGS findings actifs"
done

# S8 — Secrets sans rotation
echo "[S8] Secrets Manager rotation..."
NO_ROTATION=$(aws secretsmanager list-secrets $PROFILE $REGION --output json | \
    jq '[.SecretList[] | select(.RotationEnabled != true)] | length')
echo "  Secrets sans rotation: $NO_ROTATION"

# S9 — CloudTrail multi-region
echo "[S9] CloudTrail..."
TRAILS=$(aws cloudtrail describe-trails $PROFILE $REGION --output json | \
    jq '[.trailList[] | select(.IsMultiRegionTrail==true and .IsOrganizationTrail==true)] | length')
