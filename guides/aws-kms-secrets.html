KMS Keys & Configuration AWS managed CMK
Types de clés, key material origin, rotation automatique — le socle du chiffrement AWS
🚨
Scénario : Un développeur utilise une clé AWS managed par défaut pour chiffrer un bucket S3 contenant des données clients. Un ex-collaborateur ayant encore accès IAM au compte peut déchiffrer toutes les données — car aucune key policy restrictive n'existe sur les AWS managed keys. Avec une CMK dédiée et une key policy explicite, seuls les rôles autorisés peuvent déchiffrer.

Analogie : AWS KMS est un coffre-fort de clés numériques — il ne stocke pas vos données, mais les clés qui les protègent. Une CMK (Customer Managed Key) est votre clé maîtresse : c'est elle qui protège toutes les autres clés de chiffrement.

💡
Pourquoi chaque paramètre KMS Keys compte :
  • CMK vs AWS managed key : les AWS managed keys (préfixées aws/) ne permettent pas de contrôle granulaire (CIS AWS Foundations Benchmark v7.0.0 — KMS non couvert nominativement, voir AWS FSBP KMS.1-KMS.4) sont gratuites mais ne permettent aucun contrôle sur la key policy — n'importe quel principal IAM avec les droits kms:Decrypt dans le compte peut déchiffrer. Une CMK dédiée permet de restreindre l'accès via une key policy explicite (CIS AWS Foundations Benchmark 3.0, contrôle 3.8).
  • Rotation automatique : sans rotation, une clé compromise reste valide indéfiniment. La rotation annuelle crée un nouveau key material tout en gardant l'ancien pour déchiffrer les données existantes. Conforme à NIST SP 800-57 Part 1 (cryptoperiods) et PCI DSS v4.0 Requirement 3.6.4.
  • Key material origin : AWS_KMS est le défaut sécurisé (généré dans les HSM FIPS 140-2 L2 de KMS). EXTERNAL (BYOK) pour les cas où la réglementation exige un contrôle total sur la génération. AWS_CLOUDHSM pour FIPS 140-2 Level 3.
  • Key policy : la policy par défaut donne le contrôle total au compte root. C'est nécessaire mais insuffisant — il faut séparer les administrateurs de clés (qui gèrent le cycle de vie) des utilisateurs de clés (qui chiffrent/déchiffrent). Principe de séparation des privilèges (NIST SP 800-53 AC-5).
  • Alias : sans alias, les opérateurs manipulent des UUID (1234abcd-5678-...) — risque d'erreur humaine. Un alias alias/prod-rds-encryption est auto-documentant.
  • Tags : indispensables pour le suivi des coûts (Cost Explorer par tag), la gouvernance (SCP tag-based) et l'audit (AWS Config rules basées sur les tags). Conforme au AWS Well-Architected Framework, pilier Cost Optimization.
  • Multi-region : nécessaire uniquement pour le DR cross-region ou le chiffrement/déchiffrement dans plusieurs régions. Éviter par défaut (coût x N réplicas).
⚠️
Pièges fréquents :
  • Rotation ≠ re-chiffrement : activer EnableKeyRotation crée un nouveau key material annuellement, mais les données existantes restent chiffrées avec l'ancien material. KMS gère transparemment le déchiffrement. Cependant, pour un re-chiffrement complet (ex: après compromission), il faut utiliser ReEncrypt manuellement.
  • AWS managed keys : pas de rotation configurable — elles sont rotées automatiquement tous les ans par AWS, mais vous ne pouvez ni accélérer ni désactiver cette rotation.
  • Suppression de clé : ScheduleKeyDeletion impose un délai de 7-30 jours. Si la clé est supprimée, toutes les données chiffrées avec elle sont IRRECUPÉRABLES. Toujours tester avec DisableKey d'abord.

1.2 — Configuration

AWS CLI — Créer une CMK symétrique avec rotation
# Créer une CMK symétrique AES-256 dédiée
aws kms create-key \
    --description "CMK production - chiffrement RDS et S3" \
    --key-spec SYMMETRIC_DEFAULT \
    --key-usage ENCRYPT_DECRYPT \
    --tags '[{"TagKey":"Environment","TagValue":"production"},{"TagKey":"Service","TagValue":"rds-s3"},{"TagKey":"Owner","TagValue":"security-team"},{"TagKey":"ManagedBy","TagValue":"terraform"}]' \
    --profile $AWS_PROFILE \
    --region $AWS_REGION \
    --output json

# Récupérer le KeyId depuis la sortie, puis :

# Activer la rotation automatique (annuelle)
aws kms enable-key-rotation \
    --key-id $KEY_ID \
    --profile $AWS_PROFILE \
    --region $AWS_REGION

# Créer un alias descriptif
aws kms create-alias \
    --alias-name "alias/prod-rds-s3-encryption" \
    --target-key-id $KEY_ID \
    --profile $AWS_PROFILE \
    --region $AWS_REGION
Terraform — CMK avec rotation et alias
resource "aws_kms_key" "main" {
  description             = "CMK production - chiffrement RDS et S3"
  deletion_window_in_days = 30
  enable_key_rotation     = true
  key_usage               = "ENCRYPT_DECRYPT"
  customer_master_key_spec = "SYMMETRIC_DEFAULT"

  tags = {
    Environment = "production"
    Service     = "rds-s3"
    Owner       = "security-team"
    ManagedBy   = "terraform"
  }
}

resource "aws_kms_alias" "main" {
  name          = "alias/prod-rds-s3-encryption"
  target_key_id = aws_kms_key.main.key_id
}

1.3 — Vérification

AWS CLI — Vérification de la clé et de la rotation
# Vérifier les propriétés de la clé
aws kms describe-key \
    --key-id "alias/prod-rds-s3-encryption" \
    --profile $AWS_PROFILE \
    --region $AWS_REGION \
    --query "KeyMetadata.{KeyId:KeyId,KeyState:KeyState,KeySpec:KeySpec,Origin:Origin,KeyManager:KeyManager,MultiRegion:MultiRegion,Description:Description}" \
    --output table

# Vérifier que la rotation automatique est activée
aws kms get-key-rotation-status \
    --key-id "alias/prod-rds-s3-encryption" \
    --profile $AWS_PROFILE \
    --region $AWS_REGION
# Attendu : "KeyRotationEnabled": true

# Lister toutes les CMK sans rotation (audit)
aws kms list-keys --profile $AWS_PROFILE --region $AWS_REGION --output json | \
  jq -r '.Keys[].KeyId' | while read kid; do
    rot=$(aws kms get-key-rotation-status --key-id "$kid" --profile $AWS_PROFILE --region $AWS_REGION --query "KeyRotationEnabled" --output text 2>/dev/null)
    mgr=$(aws kms describe-key --key-id "$kid" --profile $AWS_PROFILE --region $AWS_REGION --query "KeyMetadata.KeyManager" --output text)
    if [ "$mgr" = "CUSTOMER" ] && [ "$rot" = "False" ]; then
      echo "⚠ CMK SANS ROTATION: $kid"
    fi
  done

Encryption Operations KMS
Envelope encryption, GenerateDataKey, Encryption SDK, chiffrement côté serveur et client
🚨
Scénario : Une application chiffre directement un fichier de 5 Go avec aws kms encrypt — l'appel échoue car KMS ne chiffre que 4 Ko de données maximum par appel direct. La bonne approche est l'envelope encryption : KMS génère une clé de données (data key), l'application chiffre localement avec cette clé, puis stocke la data key chiffrée à côté du fichier.

Analogie : L'envelope encryption fonctionne comme un coffre-fort à double clé : KMS est le gardien de la clé maîtresse (CMK) qui protège les clés des coffres individuels (data keys). Chaque coffre (fichier chiffré) a sa propre clé, mais seule la clé maîtresse peut la déverrouiller.

Pourquoi chaque paramètre Encryption Operations compte :
  • Envelope encryption : le pattern fondamental de KMS. KMS ne chiffre que 4 Ko max directement. Pour tout le reste (fichiers, bases, blobs), l'envelope encryption est obligatoire. C'est le modèle utilisé par tous les services AWS intégrés (S3 SSE-KMS, EBS, RDS).
  • Encryption Context : données supplémentaires (paires clé/valeur) liées au chiffrement. Elles sont loggées dans CloudTrail et peuvent être exigées via key policy (kms:EncryptionContext:*). C'est à la fois un mécanisme d'audit et de contrôle d'accès.
  • S3 Bucket Key : sans bucket key, chaque objet S3 déclenche un appel KMS. Avec bucket key, KMS génère une clé temporaire par bucket qui réduit les appels de 99%. Impact direct sur les coûts et la latence.
  • Client-Side Encryption : chiffre avant l'envoi vers AWS. Nécessaire quand on ne fait pas confiance au transport ou quand les données doivent rester chiffrées même dans la mémoire du service. L'AWS Encryption SDK simplifie cette implémentation.
  • ReEncrypt : permet de changer la clé de chiffrement sans jamais exposer le plaintext côté client. KMS déchiffre et re-chiffre en interne. Indispensable en cas de compromission suspecte ou de migration de clé.
⚠️
Pièges fréquents :
  • Limites d'API KMS : les comptes ont un quota de requêtes KMS par seconde (5 500-30 000 selon la région et le type d'opération). Un service à fort débit sans data key caching peut atteindre le throttling (ThrottlingException). Solution : activer le data key caching dans l'Encryption SDK ou utiliser les S3 Bucket Keys.
  • Encryption context oublié : si un chiffrement est fait avec un encryption context, le même context doit être fourni au déchiffrement. Oublier le context = données inaccessibles.
  • SSE-S3 ≠ SSE-KMS : SSE-S3 chiffre avec une clé gérée entièrement par AWS — pas d'audit CloudTrail, pas de key policy, pas de rotation configurable. SSE-KMS offre un contrôle complet.

2.3 — Configuration

AWS CLI — Envelope encryption avec GenerateDataKey
# Générer une data key (retourne le plaintext + la version chiffrée)
aws kms generate-data-key \
    --key-id "alias/prod-rds-s3-encryption" \
    --key-spec AES_256 \
    --encryption-context '{"service":"myapp","environment":"production"}' \
    --profile $AWS_PROFILE \
    --region $AWS_REGION
# Sortie : Plaintext (base64) + CiphertextBlob (base64) + KeyId
# 1. Utiliser Plaintext pour chiffrer vos données localement (AES-256-GCM)
# 2. Stocker CiphertextBlob à côté des données chiffrées
# 3. SUPPRIMER Plaintext de la mémoire immédiatement

# Activer S3 Bucket Key pour réduire les appels KMS
aws s3api put-bucket-encryption \
    --bucket $BUCKET_NAME \
    --server-side-encryption-configuration '{
      "Rules": [{
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "aws:kms",
          "KMSMasterKeyID": "alias/prod-rds-s3-encryption"
        },
        "BucketKeyEnabled": true
      }]
    }' \
    --profile $AWS_PROFILE \
    --region $AWS_REGION
Terraform — S3 SSE-KMS avec Bucket Key
resource "aws_s3_bucket_server_side_encryption_configuration" "main" {
  bucket = aws_s3_bucket.main.id

  rule {
    bucket_key_enabled = true
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.main.arn
    }
  }
}

# Policy S3 pour interdire les uploads non chiffrés KMS
resource "aws_s3_bucket_policy" "deny_non_kms" {
  bucket = aws_s3_bucket.main.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "DenyNonKMSEncryption"
      Effect    = "Deny"
      Principal = "*"
      Action    = "s3:PutObject"
      Resource  = "${aws_s3_bucket.main.arn}/*"
      Condition = {
        StringNotEquals = {
          "s3:x-amz-server-side-encryption" = "aws:kms"
        }
      }
    }]
  })
}

Secrets Manager Secrets Manager
Création de secrets, rotation automatique Lambda, réplicas multi-région, resource policies
🚨
Scénario : Un mot de passe de base de données RDS est stocké en dur dans le code source (.env, variable d'environnement, fichier de config). Un développeur quitte l'équipe — le mot de passe n'est jamais changé car "c'est compliqué de mettre à jour partout". Avec Secrets Manager, le secret est stocké centralement, roté automatiquement via Lambda, et l'application le récupère à la volée via l'API.

Analogie : Secrets Manager est un distributeur automatique de mots de passe à usage limité. Au lieu de donner un double de clé permanent, il distribue des clés temporaires qui changent régulièrement — et seules les personnes autorisées peuvent en demander une.

  • Rotation automatique : sans rotation, un secret compromis reste valide indéfiniment. La rotation Lambda de Secrets Manager change le secret et met à jour le service consommateur (ex: mot de passe RDS) en une seule opération atomique.

  • Multi-user rotation : pour les bases de données, utiliser deux utilisateurs en alternance (évite les interruptions pendant la rotation). Le premier est actif pendant que le second est roté, puis on bascule.

  • Lambda de rotation & réseau : la Lambda de rotation doit pouvoir atteindre Secrets Manager (via endpoint VPC ou Internet) et la base de données (security group). Erreur courante : Lambda dans un VPC sans endpoint Secrets Manager → timeout.

  • Coût des appels API : chaque GetSecretValue est facturé. Implémenter un cache côté application (TTL 5-15 min) avec le SDK caching de Secrets Manager.

🚨
Scénario : Un administrateur crée une clé KMS sans key policy explicite. Analogie : Une key policy KMS est le règlement intérieur du coffre-fort. La politique IAM dit qui peut entrer dans le bâtiment ; la key policy dit qui peut ouvrir chaque coffre spécifique. Les deux doivent autoriser pour que l'accès soit accordé.

5.1 — Tableau de configuration

Paramètre / Configuration Défaut AWS ⚡ Quick Win 🔒 Sécurisé 🎯 Best Practice Coût

(admin ≠ usage)		 Aucune séparation Rôles séparés admin/usage 3 rôles : admin, usage, audit + rôle rotation, rôle grant-admin Gratuit
Condition kms:ViaService Aucune condition Aucune condition ViaService pour S3, EBS, RDS ViaService + EncryptionContext obligatoire Gratuit
Grants Aucun grant Aucun grant Grants pour services AWS temporaires Grants scopés + constraints + retiring Gratuit

💡
Pourquoi chaque mécanisme d'accès compte :
  • Key policy : c'est la seule politique de ressource d'une clé KMS. Si elle n'autorise pas un principal, même un IAM admin ne peut pas utiliser la clé — contrairement aux autres services AWS où IAM suffit.
  • Séparation des devoirs : un kms:CreateKey / kms:PutKeyPolicy ne devrait jamais être combiné avec kms:Decrypt sur le même rôle. Sinon, un admin peut à la fois modifier la policy et déchiffrer les données.
  • ViaService : restreint l'usage de la clé à un service spécifique (ex: s3.eu-west-1.amazonaws.com). Empêche un attaquant d'utiliser kms:Decrypt directement via l'API KMS.
  • Grants : accès temporaires et scopés utilisés par les services AWS (EBS, RDS). Préférer les grants aux policies IAM pour les accès ponctuels car ils sont révocables immédiatement.
  • Cross-account : requiert à la fois la key policy (compte source) et la policy IAM (compte cible). Utiliser aws:PrincipalOrgID pour limiter aux comptes de votre organisation.
⚠️
Pièges fréquents :
  • Key policy lock-out : si vous supprimez le root account de la key policy sans alternative, la clé devient inaccessible. Toujours garder le root account avec kms:* comme filet de sécurité.
  • IAM seul ne suffit pas : une politique IAM Allow kms:Decrypt est ignorée si la key policy ne délègue pas à IAM via la clause "Enable IAM User Permissions".
  • Grants non révoqués : les grants créés par les services AWS persistent parfois après la suppression de la ressource. Auditer régulièrement avec list-grants.
  • kms:* dans les policies IAM : ne jamais accorder kms:* dans une policy IAM — utiliser des actions spécifiques (kms:Encrypt, kms:Decrypt, kms:GenerateDataKey).

CloudHSM CloudHSM
FIPS 140-2 Level 3, custom key store, cas d'usage réglementaire, décision KMS vs CloudHSM
🏦
Scénario : Un organisme financier ou de santé est soumis à des réglementations exigeant que les clés cryptographiques soient stockées dans un HSM certifié FIPS 140-2 Level 3 (PCI DSS, eIDAS, HDS). KMS standard est certifié FIPS 140-2 Level 2 pour les clés logicielles — insuffisant pour ces exigences. CloudHSM fournit des HSM dédiés à instance unique dans votre VPC.

Analogie : KMS est un service de coffre-fort bancaire partagé (multi-tenant, géré par la banque). CloudHSM est votre propre coffre-fort physique dans une salle blindée — vous avez les clés, personne d'autre ne peut y accéder, même pas AWS.

6.1 — Décision : KMS standard vs CloudHSM

Critère AWS KMS (standard) AWS CloudHSM KMS + Custom Key Store
Certification FIPS 140-2 Level 2 (logiciel)
Level 3 (module HSM interne)		 FIPS 140-2 Level 3 (matériel dédié) FIPS 140-2 Level 3 via CloudHSM
Contrôle des clés AWS gère le HSM (multi-tenant) Client contrôle entièrement (single-tenant) Client contrôle via CloudHSM + API KMS
Intégration services AWS Native (S3, EBS, RDS, Lambda, etc.) Limitée (SDK/PKCS#11/JCE) Native via KMS API
Coût 1 $/clé/mois + appels API ~1,50 $/heure/HSM (∼1 100 $/mois) KMS + CloudHSM combinés
Haute dispo Multi-AZ automatique Min 2 HSM dans 2 AZ (client gère) Cluster CloudHSM multi-AZ requis
Algorithmes custom AES-256, RSA, ECC fixés Libre (PKCS#11, JCE, OpenSSL) Via CloudHSM uniquement
Cas d'usage typique 99 % des workloads Conformité réglementaire stricte, PKI, signature de code Conformité + intégration KMS native
✅
Règle simple : Commencez toujours par KMS standard. Passez à CloudHSM (ou KMS + Custom Key Store) uniquement si :
1. Une réglementation exige explicitement FIPS 140-2 Level 3 matériel dédié
2. Vous avez besoin d'algorithmes cryptographiques non supportés par KMS
3. Vous gérez une PKI ou devez signer du code/documents avec un HSM dédié
⚠️
Pièges fréquents CloudHSM :
  • Coût continu : un cluster CloudHSM coûte ∼1 100 $/mois/HSM même sans utilisation. Minimum 2 HSM pour la HA = ∼2 200 $/mois de base.
  • Gestion client : vous gérez les utilisateurs HSM, les backups, la HA, les mises à jour firmware. Si le Crypto Officer perd ses credentials → clés inaccessibles.
  • Réseau : le cluster CloudHSM est dans votre VPC. Les ENI doivent être dans des subnets privés avec les security groups adéquats (port 2223-2225).
  • Custom Key Store déconnecté : si le cluster CloudHSM est arrêté ou inaccessible, les clés KMS du custom key store deviennent inutilisables.

Script All-in-One — Audit & Hardening Complet
Script exécutable pour auditer et appliquer toutes les best practices KMS & Secrets
⚠
Attention : Ce script est en mode audit-only par défaut. Les commandes de remédiation sont présentes mais commentées. Décommentez uniquement après validation en environnement de test.
Bash — audit-kms-secrets.sh
#!/usr/bin/env bash
# ═══════════════════════════════════════════════════════════════════
# audit-kms-secrets.sh — Audit & Hardening AWS KMS & Secrets Manager
# Version 1.0 | Mode: audit-only (remédiations commentées)
# Prérequis : aws cli v2, jq, profil configuré
# Usage : ./audit-kms-secrets.sh --profile PROFILE --region REGION
# ═══════════════════════════════════════════════════════════════════
set -euo pipefail

# ─── Paramètres ───
PROFILE="${AWS_PROFILE:-default}"
REGION="${AWS_REGION:-eu-west-1}"

while [[ $# -gt 0 ]]; do
  case "$1" in
    --profile) PROFILE="$2"; shift 2;;
    --region)  REGION="$2";  shift 2;;
    *) echo "Usage: $0 [--profile PROFILE] [--region REGION]"; exit 1;;
  esac
done

AWS="aws --profile $PROFILE --region $REGION"
PASS=0; WARN=0; FAIL=0; INFO=0

log_pass() { echo -e "\e[32m[PASS]\e[0m $1"; ((PASS++)); }
log_warn() { echo -e "\e[33m[WARN]\e[0m $1"; ((WARN++)); }
log_fail() { echo -e "\e[31m[FAIL]\e[0m $1"; ((FAIL++)); }
log_info() { echo -e "\e[36m[INFO]\e[0m $1"; ((INFO++)); }

echo "═══════════════════════════════════════════════════"
echo " AWS KMS & Secrets Manager — Audit Report"
echo " Profil: $PROFILE | Région: $REGION"
echo " Date: $(date -u +%Y-%m-%dT%H:%M:%SZ)"
echo "═══════════════════════════════════════════════════"
echo ""

# ─── S1 : CMK Inventory ───
echo "━━━ S1: KMS Keys ━━━"
KEYS=$($AWS kms list-keys --query "Keys[].KeyId" --output text)
CMK_COUNT=0
for key in $KEYS; do
    meta=$($AWS kms describe-key --key-id "$key" --query "KeyMetadata.{Manager:KeyManager,State:KeyState,Rotation:RotationEnabled,Origin:Origin,Desc:Description}" --output json 2>/dev/null || echo "{}")
    manager=$(echo "$meta" | jq -r '.Manager // STR1')
    state=$(echo "$meta" | jq -r '.State // STR1')

    [ "$manager" != "CUSTOMER" ] && continue
    [ "$state" != "Enabled" ] && continue
    ((CMK_COUNT++))

    # Vérifier la rotation
    rotation=$($AWS kms get-key-rotation-status --key-id "$key" --query "KeyRotationEnabled" --output text 2>/dev/null || echo "false")
    if [ "$rotation" = "True" ] || [ "$rotation" = "true" ]; then
        log_pass "CMK $key : rotation activée"
    else
        log_fail "CMK $key : rotation DÉSACTIVÉE"
        # REMÉDIATION : $AWS kms enable-key-rotation --key-id "$key"
    fi

    # Vérifier les tags
    env_tag=$($AWS kms list-resource-tags --key-id "$key" --query "Tags[?TagKey=='Environment'].TagValue" --output text)
    if [ -z "$env_tag" ]; then
        log_warn "CMK $key : tag 'Environment' manquant"
    fi
done
log_info "Total CMK customer-managed actives: $CMK_COUNT (coût: ${CMK_COUNT} $/mois)"
echo ""

# ─── S3 : Secrets Manager ───
echo "━━━ S3: Secrets Manager ━━━"
SECRETS=$($AWS secretsmanager list-secrets --query "SecretList[].{Name:Name,ARN:ARN,Rotation:RotationEnabled}" --output json 2>/dev/null || echo "[]")
SM_COUNT=$(echo "$SECRETS" | jq 'length')
log_info "Nombre de secrets: $SM_COUNT (coût estimé: $(echo "$SM_COUNT * 0.40" | bc) $/mois)"

echo "$SECRETS" | jq -c '.[]' | while read -r secret; do
    name=$(echo "$secret" | jq -r '.Name')
    rotation=$(echo "$secret" | jq -r '.Rotation // false')
    if [ "$rotation" = "true" ]; then
        log_pass "Secret '$name' : rotation activée"
    else
        log_fail "Secret '$name' : rotation DÉSACTIVÉE"
    fi
done
echo ""

# ─── S7 : EBS Default Encryption ───
echo "━━━ S7: Intégration — EBS ━━━"
EBS_DEFAULT=$($AWS ec2 get-ebs-encryption-by-default --query "EbsEncryptionByDefault" --output text)
if [ "$EBS_DEFAULT" = "True" ] || [ "$EBS_DEFAULT" = "true" ]; then
    log_pass "Chiffrement EBS par défaut activé"
else
    log_fail "Chiffrement EBS par défaut DÉSACTIVÉ"
    # REMÉDIATION : $AWS ec2 enable-ebs-encryption-by-default
fi

UNENC_VOLS=$($AWS ec2 describe-volumes --filters Name=encrypted,Values=false --query "length(Volumes)" --output text)
if [ "${UNENC_VOLS:-0}" -gt 0 ]; then
    log_warn "$UNENC_VOLS volumes EBS non chiffrés détectés"
else
    log_pass "Tous les volumes EBS sont chiffrés"
fi
echo ""

# ─── S7 : S3 Bucket Keys ───
echo "━━━ S7: Intégration — S3 ━━━"
BUCKETS=$($AWS s3api list-buckets --query "Buckets[].Name" --output text)
for bucket in $BUCKETS; do
    enc=$($AWS s3api get-bucket-encryption --bucket "$bucket" --output json 2>/dev/null || echo "")
    if [ -z "$enc" ]; then
        log_fail "Bucket '$bucket' : PAS de chiffrement par défaut"
    else
        algo=$(echo "$enc" | jq -r '.ServerSideEncryptionConfiguration.Rules[0].ApplyServerSideEncryptionByDefault.SSEAlgorithm // STR1')
        bk=$(echo "$enc" | jq -r '.ServerSideEncryptionConfiguration.Rules[0].BucketKeyEnabled // false')
        if [ "$algo" = "aws:kms" ] && [ "$bk" = "true" ]; then
            log_pass "Bucket '$bucket' : SSE-KMS + Bucket Key ✓"
        elif [ "$algo" = "aws:kms" ]; then
            log_warn "Bucket '$bucket' : SSE-KMS mais Bucket Key désactivé (surcoût API)"
        else
            log_warn "Bucket '$bucket' : utilise $algo (pas SSE-KMS)"
        fi
    fi
done
echo ""

# ─── S8 : Config Rules Compliance ───
echo "━━━ S8: Monitoring & Config ━━━"
for rule in cmk-rotation-enabled secrets-rotation-check; do
    status=$($AWS configservice get-compliance-details-by-config-rule \
        --config-rule-name "$rule" --compliance-types NON_COMPLIANT \
        --query "length(EvaluationResults)" --output text 2>/dev/null || echo "N/A")
    if [ "$status" = "0" ]; then
        log_pass "Config rule '$rule' : 100% conforme"
    elif [ "$status" = "N/A" ]; then
        log_warn "Config rule '$rule' : non déployée"
    else
        log_fail "Config rule '$rule' : $status ressources NON_COMPLIANT"
    fi
done
echo ""

# ─── Résumé ───
echo "═══════════════════════════════════════════════════"
echo " RÉSUMÉ"
echo "═══════════════════════════════════════════════════"
TOTAL=$((PASS + WARN + FAIL))

if [ "$FAIL" -eq 0 ]; then
    echo -e " \e[32m✓ Aucun point critique détecté\e[0m"
else
    echo -e " \e[31m✗ $FAIL points critiques à remédier\e[0m"
fi
echo "═══════════════════════════════════════════════════"
