S1
Cost Explorer & Analysis Gratuit API 0,01$/req
Visibilité complète des dépenses : Cost Explorer, CUR, Cost Categories, Tags d'allocation
Réf. AWS Well-Architected Framework — Cost Optimization Pillar • CIS AWS Foundations v7.0.0
🚨
Scénario concret : Sans Cost Explorer ni CUR activés, une équipe DevOps lance un lot d'instances GPU pour un test ML vendredi soir. Personne ne les éteint le week-end. Selon la région, l'OS et le volume utilisé, la dérive peut représenter plusieurs dizaines de milliers de dollars avant d'être visible le lundi.
Analogie : Cost Explorer est votre relevé bancaire détaillé — sans lui, vous découvrez vos dépenses une fois par mois quand la facture arrive. Avec lui, vous suivez chaque transaction en temps réel.

💡
Pourquoi chaque paramètre compte :
  • Cost Explorer (activation) : sans lui, zéro visibilité sur les dépenses. La console Billing ne montre qu'un total mensuel. Cost Explorer permet de décomposer par service, compte, région, tag — c'est la base de toute démarche FinOps.
  • Granularité horaire : elle aide l'investigation récente sur 14 jours, mais elle est facturée au nombre d'enregistrements et les données Cost Explorer restent rafraîchies au moins toutes les 24 heures.
  • CUR 2.0 : Cost Explorer montre des agrégats, alors qu'AWS Data Exports expose CUR 2.0 et FOCUS 1.2 with AWS columns pour l'analyse détaillée vers S3, Athena et QuickSight. C'est la base du chargeback et de l'interopérabilité FinOps.
  • Cost Categories : permettent de créer des vues métier (par BU, par projet) sans dépendre uniquement des tags. Règles de catégorisation automatiques = pas de dépendance au tagging humain.
  • Cost Allocation Tags : sans tags activés dans Billing, les tags posés sur les ressources ne remontent pas dans Cost Explorer. Étape invisible mais critique.
  • Rightsizing Recommendations : AWS analyse les métriques CloudWatch pour recommander des tailles d'instances adaptées. Non activé par défaut = opportunités d'économies non identifiées.
⚠️
Pièges :
  • Cost Explorer ne sert pas au temps réel — les données du mois courant deviennent visibles en environ 24 heures et sont rafraîchies au moins toutes les 24 heures. Utiliser Cost Anomaly Detection (S4) pour les dérives.
  • Les tags doivent être "activés" dans Billing — poser un tag sur une ressource ne suffit pas. Il faut aller dans Billing > Cost allocation tags > activer explicitement chaque clé.
  • CUR ne rétroagit pas — les données ne sont disponibles qu'à partir de la date d'activation. Activer CUR le plus tôt possible.
  • L'API Cost Explorer coûte 0,01 $/requête — un script mal conçu qui requête toutes les 5 min = 0,01 × 288/jour × 30 = 86 $/mois.

  --time-period Start=2026-04-01,End=2026-04-20 \

  --metrics BlendedCost \
  --profile $PROFILE --region us-east-1 --output json
      # Note : cet appel vérifie l'accès ; il n'active pas Cost Explorer.

      # 3. Créer le bucket S3 pour CUR (chiffré, versionné)

  --bucket $CUR_BUCKET \
  --region $REGION \
  --create-bucket-configuration LocationConstraint=$REGION \
  --profile $PROFILE

  --bucket $CUR_BUCKET \

💡
Pourquoi chaque paramètre compte :
  • Budget mensuel global : premier filet de sécurité. Sans budget, aucun mécanisme d'alerte automatique n'existe — la facture AWS n'a pas de plafond. C'est le Quick Win #1.
  • Seuils multiples (50/80/100%) : un seul seuil à 100% prévient trop tard. Trois seuils donnent le temps d'investiguer et de réagir progressivement.
  • Forecasted budget : alerte basée sur la prévision et non le coût réel. Permet de réagir en milieu de mois avant que le budget ne soit réellement dépassé.
  • Budget Actions : seul moyen de stopper automatiquement les dépenses (appliquer un SCP deny ou stopper des instances). Niveau ultime de protection.
  • RI/SP coverage budget : alerte quand le taux de couverture Savings Plans ou RI descend en dessous du seuil — signal que du compute on-demand non couvert augmente.
⚠️
Pièges :
  • Les budgets de suivi sont gratuits, mais les Budget Actions ne sont gratuites que pour les deux premiers budgets action-enabled du mois ; au-delà, chaque budget avec actions est facturé 0,10 $/jour.
  • Les Budget Actions "auto-apply" peuvent casser la production — toujours tester en mode "approval required" d'abord. Un SCP deny appliqué au mauvais OU peut bloquer toute l'organisation.
  • Les emails d'alerte vont souvent dans les spams — configurer en plus un topic SNS qui notifie Slack/Teams via Lambda ou AWS Chatbot.
  • Un budget à 100 $/mois sans contexte est inutile — baser le montant sur la moyenne des 3 derniers mois + 15% de marge.

Réductions par engagement : Compute SP vs EC2 SP vs RI, termes, paiement, recommandations
Réf. AWS Well-Architected — COST 09 • FinOps Framework — Optimize Phase
🚨
Scénario concret : Une entreprise exécute un socle compute stable en on-demand depuis plusieurs trimestres. En basculant vers le bon type de Savings Plan ou de Reserved Instance, le niveau de remise dépendra du terme, du paiement upfront, du taux de couverture et du degré de flexibilité conservé.
Analogie : Un Savings Plan est votre abonnement annuel — comme un forfait téléphone, vous vous engagez sur une durée pour payer moins cher l'unité. Plus l'engagement est long et le paiement anticipé, plus la remise est forte.
💡

Script All-in-One — Audit & Configuration FinOps
Script Bash exécutable pour auditer et configurer toutes les best practices de ce guide
⚠️
Prérequis : AWS CLI v2 configuré avec un profil ayant les permissions ce:*, budgets:*, ec2:Describe*, s3:GetBucketPolicy, compute-optimizer:*, organizations:List*.
Adapter la variable PROFILE et REGION avant exécution.
Bash — aws-finops-audit.sh
#!/usr/bin/env bash
# aws-finops-audit.sh — Audit FinOps complet
# Usage : ./aws-finops-audit.sh [PROFILE] [REGION]
set -euo pipefail

PROFILE="${1:-default}"
REGION="${2:-eu-west-3}"
OUTPUT="finops-audit-$(date +%Y%m%d).txt"

echo "=== AWS FinOps Audit — $(date) ===" | tee "$OUTPUT"
echo "Profil: $PROFILE | Région: $REGION" | tee -a "$OUTPUT"

# S1 — Cost Explorer (30 derniers jours)
echo -e "\n--- S1: Coûts 30j par service ---" | tee -a "$OUTPUT"
aws ce get-cost-and-usage \
  --time-period Start=$(date -d '-30 days' +%Y-%m-%d),End=$(date +%Y-%m-%d) \
  --granularity MONTHLY --metrics UnblendedCost \
  --group-by Type=DIMENSION,Key=SERVICE \
  --profile "$PROFILE" --region us-east-1 --output table 2>&1 | tee -a "$OUTPUT"

# S2 — Budgets existants
echo -e "\n--- S2: Budgets ---" | tee -a "$OUTPUT"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text --profile "$PROFILE")
aws budgets describe-budgets --account-id "$ACCOUNT_ID" \
  --profile "$PROFILE" --output table 2>&1 | tee -a "$OUTPUT"

# S3 — SP/RI recommandations
echo -e "\n--- S3: Savings Plan Recommendations ---" | tee -a "$OUTPUT"
aws ce get-savings-plans-purchase-recommendation \
  --savings-plans-type COMPUTE_SP --term-in-years ONE_YEAR \
  --payment-option NO_UPFRONT --lookback-period-in-days SIXTY_DAYS \
  --profile "$PROFILE" --region us-east-1 --output table 2>&1 | tee -a "$OUTPUT"

# S5 — Quick Wins: gp2 volumes, stopped EC2, unattached EIP
echo -e "\n--- S5: Volumes gp2 restants ---" | tee -a "$OUTPUT"
aws ec2 describe-volumes \
  --filters Name=volume-type,Values=gp2 \
  --query "Volumes[].{ID:VolumeId,Size:Size,State:State}" \
  --profile "$PROFILE" --region "$REGION" --output table 2>&1 | tee -a "$OUTPUT"

echo -e "\n--- S5: Instances EC2 arrêtées ---" | tee -a "$OUTPUT"
aws ec2 describe-instances \
  --filters Name=instance-state-name,Values=stopped \
  --query "Reservations[].Instances[].{ID:InstanceId,Type:InstanceType,Name:Tags[?Key=='Name'].Value|[0]}" \
  --profile "$PROFILE" --region "$REGION" --output table 2>&1 | tee -a "$OUTPUT"

echo -e "\n--- S5: EIP non associées ---" | tee -a "$OUTPUT"
aws ec2 describe-addresses \
  --query "Addresses[?AssociationId==null].{IP:PublicIp,AllocId:AllocationId}" \
  --profile "$PROFILE" --region "$REGION" --output table 2>&1 | tee -a "$OUTPUT"

# S7 — Ressources sans tags obligatoires
echo -e "\n--- S7: Instances sans tag Environment ---" | tee -a "$OUTPUT"
aws ec2 describe-instances \
  --query "Reservations[].Instances[?!Tags || !contains(Tags[].Key, 'Environment')].{ID:InstanceId,Type:InstanceType}" \
  --profile "$PROFILE" --region "$REGION" --output table 2>&1 | tee -a "$OUTPUT"

echo -e "\n=== Audit terminé — résultats dans $OUTPUT ===" | tee -a "$OUTPUT"
