1

2
Alertes fondamentales (~30 min, +15 pts). Créer les alertes Service Health + Resource Health + seuils CPU/mémoire critiques + Action Group email.
3
AMA + DCR (~2h). Migrer MMA → AMA, configurer les Data Collection Rules pour Windows Events, Syslog, Performance Counters. Déployer via Azure Policy.
4

1.1 — Tableau de configuration : fondations de l'architecture

ParamètreDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence
Workspace Log Analytics Aucun créé 1 workspace central /souscription 1 central + 1 séparé pour Sentinel Hub-spoke multi-workspace avec cross-workspace queries Log Analytics
Diagnostic Settings Désactivés Activés sur ressources critiques (KV, NSG, SQL) Activés sur toutes les ressources via Azure Policy Policy + catégories granulaires par type Inclus Azure
Azure Monitor Agent (AMA) Non installé Installé via Policy sur toutes les VMs AMA + DCR granulaires (Perf, Events, Syslog) AMA + DCR avec transformations KQL + DCE privé Inclus Azure
Activity Log → workspace Pas de redirection (90j portail seulement) Export vers workspace LA Export + alertes sur opérations critiques Export + KQL detection rules + archivage >90j Log Analytics
Service Health alerts Aucune alerte Alerte Service Health → Action Group email + Resource Health alerts + intégration ITSM (ServiceNow, PagerDuty) Inclus Azure
Legacy MMA agent Peut être présent (retiré le 31/08/2024) Migration MMA → AMA MMA supprimé de toutes les VMs Aucun agent legacy, validation par Azure Policy Inclus Azure
💡
Pourquoi chaque paramètre compte :
  • Workspace Log Analytics : sans workspace, les logs n'ont aucune destination — les Diagnostic Settings, AMA et alertes log (KQL) ne peuvent pas fonctionner. C'est le prérequis n°1 de toute la stack Azure Monitor.
  • Diagnostic Settings : les ressources Azure génèrent des logs (audit, metrics, errors) mais ne les envoient nulle part par défaut. Sans branchement explicite, ces données sont perdues définitivement.
  • AMA : seul moyen supporté pour collecter les logs OS (Windows Events, Syslog, Performance Counters) des VMs. Le MMA a été retiré le 31/08/2024, et l'ingestion cloud restante peut s'interrompre à tout moment depuis le 02/03/2026.
  • Activity Log → workspace : l'Activity Log est conservé 90 jours dans le portail, mais non interrogeable par KQL et non disponible pour les alertes log sans export vers un workspace.
  • Service Health alerts : Azure ne notifie pas automatiquement des incidents de plateforme. Sans alerte, vous découvrez les pannes Azure en même temps que vos utilisateurs.
  • Legacy MMA : en cohabitation AMA+MMA, les logs peuvent être dupliqués (double coût d'ingestion) et les DCR ne s'appliquent qu'à AMA.
⚠️
Piège classique : Beaucoup d'organisations créent un workspace Log Analytics puis considèrent l'observabilité « en place ». Or, créer un workspace ne suffit pas : sans Diagnostic Settings configurés sur chaque ressource et sans AMA sur les VMs, le workspace reste vide.

Autre piège : choisir un workspace dans une autre région sans l'avoir justifié par des contraintes de résidence, de résilience ou de coût. Microsoft recommande d'évaluer les bandwidth charges surtout pour les VMs, tout en rappelant que les Diagnostic Settings Azure n'entraînent pas d'egress charges pour cet envoi vers le workspace.

1.2 — Configuration : créer les fondations

Az CLI — Créer le workspace Log Analytics central
# Variables
RG="rg-monitoring-prod"
LOCATION="westeurope"
LAW_NAME="law-central-prod"

# Créer le Resource Group de monitoring
az group create --name $RG --location $LOCATION

# Créer le workspace Log Analytics (retention 90j par défaut)
az monitor log-analytics workspace create \
    --resource-group $RG \
    --workspace-name $LAW_NAME \
    --location $LOCATION \
    --retention-in-days 90 \
    --sku PerGB2018

# Exporter l'Activity Log de la souscription vers le workspace
LAW_ID=$(az monitor log-analytics workspace show \
    --resource-group $RG --workspace-name $LAW_NAME \
    --query id -o tsv)

az monitor diagnostic-settings subscription create \
    --name "diag-activitylog-to-law" \
    --workspace $LAW_ID \
    --logs '[{"category":"Administrative","enabled":true},{"category":"Security","enabled":true},{"category":"Alert","enabled":true},{"category":"Policy","enabled":true}]'
Bicep — Workspace Log Analytics + Diagnostic Settings
// workspace-law.bicep
param location string = resourceGroup().location
param lawName string = 'law-central-prod'
param retentionDays int = 90

resource law 'Microsoft.OperationalInsights/workspaces@2023-09-01' = {
  name: lawName
  location: location
  properties: {
    sku: { name: 'PerGB2018' }
    retentionInDays: retentionDays
    features: {
      enableLogAccessUsingOnlyResourcePermissions: true
    }
  }
}

output lawId string = law.id
output lawCustomerId string = law.properties.customerId

🚨
Scénario concret : Un workspace Log Analytics configuré en retention 30 jours voit ses logs d'audit supprimés avant qu'une investigation post-incident ne puisse les consulter. L'équipe SOC ne peut pas remonter au-delà d'un mois, rendant l'analyse forensique impossible.

Analogie : Le workspace est la boîte noire de votre infrastructure — il enregistre tout ce qui se passe. Si vous réglez l'enregistrement sur 30 jours au lieu de 90, la bande est réécrite avant que les enquêteurs n'arrivent.

2.1 — Tableau de configuration : Workspace design

ParamètreDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence

Basic Logs (ContainerLogV2, etc.) Non configuré (Analytics Logs pour tout) Tables volumineuses → Basic Logs Filtrage DCR + Basic Logs Basic Logs + transformations KQL pour réduire le volume Log Analytics
Archive tier Non activé Non requis Tables d'audit → Archive (1-12 ans) Archive + search jobs pour investigation à la demande Log Analytics
Access Control Mode Workspace-context (accès total) Resource-context Resource-context + table-level RBAC Resource-context + table-level RBAC + custom roles Log Analytics
Daily cap Aucun (illimité) Cap défini selon budget Cap + alerte à 80% du seuil Commitment tier au lieu du cap Log Analytics
Commitment tier Pay-as-you-go Pay-as-you-go si <100 GB/j Commitment tier adapté (≥100 GB/j) Tier optimal recalculé mensuellement via Workbook coûts Log Analytics
Purge / Customer-managed keys Non configuré Non requis Purge capabilities activées CMK + dedicated cluster si ≥500 GB/j Log Analytics
💡
Pourquoi chaque paramètre compte :
  • Retention 90j minimum : 31 jours d'analytics retention sont inclus pour les tables Analytics, mais plusieurs tables ont des règles particulières. Pour les investigations sécurité, viser 60 à 90 jours sur les tables critiques reste une base prudente.
  • Basic Logs : utiles pour les tables volumineuses consultées rarement. Leur intérêt économique dépend du pricing régional et du volume réel. En contrepartie, les requêtes interactives sont facturées, les Insights ne s'appuient pas dessus, et l'usage KQL reste surtout pensé pour une table unique avec extension possible via lookup vers une table Analytics.
  • Access Control = Resource-context : en mode workspace-context, tout utilisateur avec accès au workspace voit tous les logs de toutes les ressources. Le mode resource-context filtre automatiquement selon les permissions RBAC sur la ressource source.
  • Daily cap : sans cap, une application qui boucle peut générer des centaines de GB en quelques heures → facture surprise. Le cap coupe l'ingestion au seuil défini.
  • Commitment tier : disponible à partir de 100 GB/jour. Selon la région et le niveau choisi, il peut réduire le coût des Analytics Logs jusqu'à 30%. Le workspace reste sur le tier sélectionné jusqu'à modification explicite.
  • CMK / Dedicated cluster : obligatoire pour les clients avec exigences réglementaires (souveraineté des clés). Le cluster dédié offre aussi cross-workspace queries optimisées.
⚠️
Piège classique n°1 : Configurer un daily cap sans alerte à 80% du seuil. Quand le cap est atteint, l'ingestion s'arrête silencieusement — plus aucun log n'est collecté jusqu'au lendemain, y compris les logs de sécurité critiques.

Piège classique n°2 : Confondre Analytics Logs vs Basic Logs. Les Basic Logs ne sont pas un clone low-cost du plan Analytics : ils supportent les simple log alerts, mais la requête interactive est facturée, les Insights ne s'y appuient pas et les corrélations avancées doivent être revalidées avant bascule.

2.2 — Configuration

Az CLI — Configurer le workspace (retention, access mode, daily cap)
# Passer la retention à 90 jours
az monitor log-analytics workspace update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --retention-in-days 90

# Activer le mode resource-context (RBAC légitime)
az monitor log-analytics workspace update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --set features.enableLogAccessUsingOnlyResourcePermissions=true

# Définir un daily cap (exemple : 10 GB/jour)
az monitor log-analytics workspace update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --set workspaceCapping.dailyQuotaGb=10

# Configurer une table en Basic Logs (ex: ContainerLogV2)
az monitor log-analytics workspace table update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --name ContainerLogV2 \
    --plan Basic

# Configurer l'archivage sur une table (ex: SecurityEvent, 365j total)
az monitor log-analytics workspace table update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --name SecurityEvent \
    --retention-in-days 90 \
    --total-retention-in-days 365

Pourquoi chaque paramètre compte :
  • Diagnostic Settings activés : c'est le seul moyen d'envoyer les resource logs (audit, errors, metrics) depuis une ressource Azure vers un workspace. Sans eux, les données sont invisibles — ni requêtables, ni alertables.
  • Key Vault AuditEvent : permet de tracer chaque accès aux secrets, clés et certificats. Indispensable pour détecter un accès non autorisé ou pour la conformité (audit trail).
  • Flow logs réseau : en 2026, les VNet Flow Logs sont la cible par défaut pour les nouveaux déploiements. Les NSG Flow Logs restent un sujet d'inventaire et de migration pour l'existant.
  • App Service logs : les logs applicatifs restent sur le filesystem local par défaut et sont purgés à chaque redéploiement. Sans export, impossible de trier les erreurs 5xx ou de corréler avec les métriques.
  • Azure Policy (DeployIfNotExists) : la configuration manuelle ne scale pas. Chaque nouvelle ressource créée par un développeur sera automatiquement branchée sans intervention humaine.
⚠️
Piège classique n°1 : Activer « allLogs » sur toutes les ressources sans analyse préalable du volume. Certaines ressources (Application Gateway, Firewall) génèrent des dizaines de GB par jour. Préférez sélectionner les catégories pertinentes.

Piège classique n°2 : Chaque ressource peut avoir plusieurs Diagnostic Settings pointant vers différentes destinations, mais une même catégorie de log ne doit pas être envoyée à deux destinations identiques — sinon les logs sont dupliqués et facturés deux fois.

3.2 — Configuration

Az CLI — Configurer les Diagnostic Settings sur les ressources critiques
# Variables
LAW_ID="/subscriptions/$SUB_ID/resourceGroups/$RG/providers/Microsoft.OperationalInsights/workspaces/$LAW_NAME"

# Key Vault → AuditEvent + AllMetrics
az monitor diagnostic-settings create \
    --resource "/subscriptions/$SUB_ID/resourceGroups/$RG/providers/Microsoft.KeyVault/vaults/$KV_NAME" \
    --name "diag-keyvault-to-law" \
    --workspace $LAW_ID \
    --logs '[{"category":"AuditEvent","enabled":true}]' \
    --metrics '[{"category":"AllMetrics","enabled":true}]'

# App Service → HTTP Logs + Console Logs + Audit Logs
az monitor diagnostic-settings create \
    --resource "/subscriptions/$SUB_ID/resourceGroups/$RG/providers/Microsoft.Web/sites/$APP_NAME" \
    --name "diag-appservice-to-law" \
    --workspace $LAW_ID \
    --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true},{"category":"AppServiceAuditLogs","enabled":true},{"category":"AppServiceAppLogs","enabled":true}]'

# SQL Database → Security Audit + Errors
az monitor diagnostic-settings create \
    --resource "/subscriptions/$SUB_ID/resourceGroups/$RG/providers/Microsoft.Sql/servers/$SQL_SERVER/databases/$DB_NAME" \
    --name "diag-sqldb-to-law" \
    --workspace $LAW_ID \
    --logs '[{"category":"SQLSecurityAuditEvents","enabled":true},{"category":"Errors","enabled":true}]'
Azure Policy — Déployer automatiquement les Diagnostic Settings
# Assigner la built-in policy initiative « Enable Azure Monitor for all resources »
# ID: /providers/Microsoft.Authorization/policySetDefinitions/0884adba-2c38-497a-8b5d-xxx

# Exemple : policy DeployIfNotExists pour Key Vault
az policy assignment create \
    --name "deploy-diag-keyvault" \
    --display-name "Deploy Diagnostic Settings for Key Vault to Log Analytics" \
    --policy "951af2fa-529b-416e-ab6e-066fd85ac459" \
    --scope "/subscriptions/$SUB_ID" \
    --params "{\"logAnalytics\":{\"value\":\"$LAW_ID\"},\"categoryGroup\":{\"value\":\"allLogs\"}}" \

    --location $LOCATION

# Créer la remediation task pour les ressources existantes
az policy remediation create \
    --name "remediate-diag-keyvault" \
    --policy-assignment "deploy-diag-keyvault" \
    --resource-group $RG

🚨
Scénario concret : Une base de données SQL atteint 95% de DTU pendant 2 heures en production. Les métriques sont bien collectées dans Azure Monitor, mais aucune alerte n'est configurée. Analogie : Les alertes sont l'alarme incendie de votre infrastructure. Les capteurs (métriques, logs) détectent la fumée, mais sans alarme configurée, personne n'est prévenu et le feu se propage.

  • Service Health alerts : Azure ne notifie pas automatiquement des incidents de plateforme. Sans cette alerte, vous apprenez les pannes Azure en même temps que vos utilisateurs. Gratuit, 0 raison de ne pas l'activer.
  • Resource Health : détecte quand Azure considère qu'une de vos ressources est dégradée ou indisponible. Complémentaire aux métriques : couvre les problèmes côté plateforme.
  • Alertes métriques (CPU/Mem/Disk) : préviennent avant la panne. Un CPU à 95% pendant 5 min est le signal qu'il faut scale-up maintenant, pas après l'indisponibilité.
  • Alertes log KQL : détectent des patterns événementiels impossibles à capturer avec des métriques : bruteforce (10+ echecs 4625 en 5 min), accès secrets Key Vault hors heures ouvrables, erreurs application récurrentes.
  • Action Groups différenciés : Sev0 (critique) doit réveiller quelqu'un (téléphone/SMS), Sev2 (information) peut attendre un email. Sans différenciation, soit on sur-notifie (alert fatigue), soit on sous-notifie (incidents ratés).
  • Alert Processing Rules : suppriment les alertes pendant les fenêtres de maintenance. Sans elles, chaque mise à jour programmée génère des dizaines d'alertes parasites → alert fatigue → vrais incidents ignorés.
⚠️
Piège classique n°1 : Créer des alertes sans Action Group. L'alerte se déclenche, apparaît dans le portail Azure Monitor > Alerts, mais personne n'est notifié. L'équipe ne consulte le portail qu'après coup.

Cost Optimization & Best Practices Log Analytics
Estimation des coûts, commitment tiers, Basic Logs, Archive, filtrage DCR
🚨
Scénario concret : Une entreprise active toutes les fonctionnalités Azure Monitor sans stratégie de coût. En quelques semaines, la facture grimpe à cause de tables volumineuses peu consultées et d'une rétention uniforme. En rebasculant certaines tables compatibles vers Basic, en appliquant des transformations DCR et en choisissant un commitment tier aligné sur le volume réel, la facture peut baisser fortement — à condition de vérifier le pricing live et le workbook Usage and estimated costs avant toute décision.

Analogie : L'optimisation des coûts Monitor est comme la gestion énergétique d'un bâtiment. Chauffer toutes les pièces à 25°C coûte cher. Les pièces fréquentées (Analytics Logs) méritent le chauffage premium. Les archives (Archive tier) sont en mode hors-gel. Les pièces rarement visitées (Basic Logs) ont un chauffage économique.

Pourquoi chaque paramètre compte :
  • Commitment tier : disponible à partir de 100 Go/jour. Le niveau choisi engage le workspace pour 31 jours et peut réduire le coût des Analytics Logs jusqu'à 30% selon la région et le tier.
  • Basic Logs : réduisent le coût d'ingestion mais ajoutent des contraintes fortes : requêtes interactives facturées, usage surtout mono-table avec lookup vers Analytics, et absence de support pour les expériences Insights. Les simple log alerts restent cependant supportées.
  • Archive : permet une rétention longue durée jusqu'à 12 ans. Les données s'exploitent via search jobs ou restore selon le besoin, avec un pricing à vérifier sur la page Azure Monitor.
  • Filtrage DCR : réduit le volume avant l'ingestion. C'est la bonne variable d'ajustement pour retirer des signaux peu utiles sans sacrifier les tables critiques au daily cap.
  • Daily Cap : filet de sécurité contre les explosions de volume (boucle de log, attaque, mauvaise configuration). Attention : quand le cap est atteint, toute l'ingestion s'arrête (y compris sécurité). Configurer l'alerte à 80% pour intervenir avant.
  • Rétention différenciée : 31 jours d'analytics retention sont inclus pour les tables Analytics, mais certaines tables comme AzureActivity, Usage et les tables Application Insights ont une rétention par défaut plus favorable. Garder 90 jours seulement sur les tables critiques évite de payer plus longtemps des données rarement consultées.
⚠️
Piège classique n°1 : Choisir un commitment tier trop élevé. Si votre consommation réelle est 80 Go/jour et vous prenez un tier 200 Go/jour, vous payez 200 Go même les jours calmes. Toujours analyser 30 jours de volume avant de choisir.

Piège classique n°2 : Passer des tables utilisées pour des corrélations complexes, des Insights ou des workbooks avancés en Basic Logs sans revue d'impact. Les simple log alerts restent possibles, mais il faut revalider chaque usage avant bascule.

Piège classique n°3 : Le Daily Cap arrête toute ingestion sans distinction. Si le cap est atteint par des logs applicatifs verbeux, les logs de sécurité sont aussi bloqués. Préférer le filtrage DCR en amont au daily cap comme mécanisme principal.

#!/bin/bash
set -euo pipefail

# ═══════════════════════════════════════════════════════════════
# VARIABLES — À ADAPTER À VOTRE ENVIRONNEMENT
# ═══════════════════════════════════════════════════════════════
SUB_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
RG="rg-monitoring-prod"
LOCATION="westeurope"
LAW_NAME="law-central-prod"
AG_NAME="ag-ops-team"
AG_EMAIL="ops-team@contoso.com"
WEBAPP_NAME="app-contoso-prod"

echo "🎯 Démarrage du déploiement observabilité Azure Monitor"
echo "   Subscription: $SUB_ID"
echo "   Resource Group: $RG"
az account set --subscription $SUB_ID

# ═══════════════════════════════════════════════════════════════
# [1/8] LOG ANALYTICS WORKSPACE (S1 + S2)
# ═══════════════════════════════════════════════════════════════
echo "[1/8] Création du Log Analytics Workspace..."
az monitor log-analytics workspace create \
    --resource-group $RG --workspace-name $LAW_NAME \
    --location $LOCATION --retention-in-days 90 \
    --sku PerGB2018 2>/dev/null || echo "   ↳ Workspace déjà existant"

LAW_ID=$(az monitor log-analytics workspace show \
    --resource-group $RG --workspace-name $LAW_NAME \
    --query id -o tsv)

# [2/8] ACTION GROUP (S5)
echo "[2/8] Création de l'Action Group..."
az monitor action-group create \
    --resource-group $RG --name $AG_NAME \
    --short-name "OpsTeam" \
    --action email ops-lead $AG_EMAIL 2>/dev/null || echo "   ↳ AG déjà existant"

AG_ID="/subscriptions/$SUB_ID/resourceGroups/$RG/providers/Microsoft.Insights/actionGroups/$AG_NAME"

# [3/8] DIAGNOSTIC SETTINGS — Activity Log (S3)
echo "[3/8] Export Activity Log vers LAW..."
az monitor diagnostic-settings subscription create \
    --name "diag-activitylog-law" \
    --location $LOCATION \
    --workspace $LAW_ID \
    --logs '[{STR1:STR2,STR3:true},{STR4:STR5,STR6:true},{STR7:STR8,STR9:true},{STR10:STR11,STR12:true}]' \
    2>/dev/null || echo "   ↳ Diag Settings Activity Log déjà configuré"

# [4/8] SERVICE HEALTH ALERT (S5)
echo "[4/8] Alerte Service Health..."
az monitor activity-log alert create \
    --resource-group $RG --name "alert-service-health" \
    --condition category=ServiceHealth \
    --action-group $AG_ID \
    --description "Azure Service Health - tous incidents" \
    2>/dev/null || echo "   ↳ Alerte Service Health déjà existante"

# [5/8] ALERTE MÉTRIQUE CPU (S5)
echo "[5/8] Alerte CPU > 90% sur VMs..."
az monitor metrics alert create \
    --resource-group $RG --name "alert-vm-cpu-critical" \
    --scopes "/subscriptions/$SUB_ID/resourceGroups/$RG" \
    --condition "avg Percentage CPU > 90" \
    --window-size 5m --evaluation-frequency 1m \
    --severity 1 --action $AG_ID \
    --target-resource-type "Microsoft.Compute/virtualMachines" \
    --description "VM CPU > 90% pendant 5 min" \
    2>/dev/null || echo "   ↳ Alerte CPU déjà existante"

# [6/8] APPLICATION INSIGHTS (S7)
echo "[6/8] Application Insights workspace-based..."
az monitor app-insights component create \
    --app "appi-$WEBAPP_NAME" --resource-group $RG \
    --location $LOCATION --kind web \
    --workspace $LAW_ID --application-type web \
    2>/dev/null || echo "   ↳ AppInsights déjà existant"

# [7/8] DAILY CAP (S10)
echo "[7/8] Configuration du daily cap LAW..."
az monitor log-analytics workspace update \
    --resource-group $RG --workspace-name $LAW_NAME \
    --daily-quota-gb 5 2>/dev/null || true

# [8/8] VÉRIFICATION FINALE
echo "[8/8] Vérification..."
echo "--- Workspace ---"
az monitor log-analytics workspace show \
    --resource-group $RG --workspace-name $LAW_NAME \
    --query "{Name:name, Sku:sku.name, Retention:retentionInDays, DailyCap:workspaceCapping.dailyQuotaGb}" -o table

echo "--- Action Groups ---"
az monitor action-group list --resource-group $RG \
    --query "[].{Name:name, Emails:emailReceivers[].emailAddress}" -o table

echo "--- Alertes ---"
az monitor metrics alert list --resource-group $RG \
    --query "[].{Name:name, Severity:severity, Enabled:isEnabled}" -o table

echo "✅ Déploiement observabilité terminé. Score estimé : ~75/100 (Quick Win)"
✅
