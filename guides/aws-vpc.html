Solutions écartées

💡
Pourquoi AWS natif et pas une solution tierce ?
  • Palo Alto / Fortinet VM : coût de licence + maintenance, intégration moins native avec AWS services, mais plus riche en fonctionnalités IPS/IDS. Pertinent pour les grands comptes multi-cloud. Non retenu ici car le guide se concentre sur l'écosystème natif AWS.
  • HashiCorp Consul (service mesh) : pertinent pour le service discovery inter-VPC mais ajoute une couche de complexité. Transit Gateway + PrivateLink couvrent les cas d'usage réseau standard.
  • VPN open source (WireGuard, StrongSwan) : moins cher mais maintenance lourde, pas de SLA AWS. Site-to-Site VPN managed offre un meilleur compromis coût/fiabilité.

🚨
Erreur fréquente : utiliser le Default VPC (172.31.0.0/16) en production. Ce VPC a des subnets publics dans chaque AZ, un Internet Gateway attaché, et des routes publiques par défaut. Toute instance lancée dedans reçoit une IP publique automatiquement.
Analogie : le VPC est votre bâtiment privé. Le Default VPC, c'est un bâtiment avec toutes les portes ouvertes — utilisable pour tester, mais pas pour y habiter.

Pourquoi un VPC custom est-il indispensable ?

💡
5 raisons de ne jamais utiliser le Default VPC en production :
  1. Exposition automatique — MapPublicIpOnLaunch=true : chaque EC2 reçoit une IPv4 publique sans action de votre part
  2. CIDR non planifié — 172.31.0.0/16 peut entrer en conflit avec votre réseau on-premise
  3. Pas de segmentation — une seule couche, tout dans les mêmes subnets
  4. NACLs par défaut — autorisent tout le trafic entrant ET sortant
  5. Non compatible IPAM — impossible de centraliser la gestion des adresses IP multi-comptes
⚠️
Piège n°1 — Sous-dimensionner les CIDR : un /24 ne donne que ~251 IPs utilisables (AWS en réserve 5 par subnet). Avec les ENI des Lambda, ECS et endpoints, un /24 peut être saturé rapidement. Prévoir au minimum un /20 pour les subnets privés applicatifs.

Piège n°2 — Oublier le CIDR secondaire : une fois le VPC créé, le CIDR primaire ne peut pas être modifié. Toujours ajouter un CIDR secondaire en réserve dès la création pour éviter de recréer le VPC.

Configuration — AWS CLI

Bash — AWS CLI
# ─── Variables ──────────────────────────────────────────────
REGION="eu-west-1"
PROFILE="production"
VPC_CIDR="10.0.0.0/16"
VPC_NAME="prod-vpc"

# ─── Étape 1 : Supprimer le Default VPC (optionnel mais recommandé) ─
DEFAULT_VPC=$(aws ec2 describe-vpcs \
    --filters "Name=isDefault,Values=true" \
    --query "Vpcs[0].VpcId" --output text \
    --region $REGION --profile $PROFILE)

if [ "$DEFAULT_VPC" != "None" ]; then
    # Supprimer toutes les dépendances du Default VPC d'abord
    echo "Suppression des IGW, subnets et SG du Default VPC..."
    # Détacher et supprimer l'Internet Gateway
    IGW=$(aws ec2 describe-internet-gateways \
        --filters "Name=attachment.vpc-id,Values=$DEFAULT_VPC" \
        --query "InternetGateways[0].InternetGatewayId" --output text \
        --region $REGION --profile $PROFILE)
    aws ec2 detach-internet-gateway --internet-gateway-id $IGW \
        --vpc-id $DEFAULT_VPC --region $REGION --profile $PROFILE
    aws ec2 delete-internet-gateway --internet-gateway-id $IGW \
        --region $REGION --profile $PROFILE
    # Supprimer les subnets
    aws ec2 describe-subnets --filters "Name=vpc-id,Values=$DEFAULT_VPC" \
        --query "Subnets[].SubnetId" --output text \
        --region $REGION --profile $PROFILE | \
        xargs -n1 -I{} aws ec2 delete-subnet --subnet-id {} \
        --region $REGION --profile $PROFILE
    # Supprimer le VPC
    aws ec2 delete-vpc --vpc-id $DEFAULT_VPC \
        --region $REGION --profile $PROFILE
fi

# ─── Étape 2 : Créer le VPC custom ──────────────────────────
VPC_ID=$(aws ec2 create-vpc \
    --cidr-block $VPC_CIDR \
    --tag-specifications "ResourceType=vpc,Tags=[{Key=Name,Value=$VPC_NAME},{Key=Environment,Value=production}]" \
    --query "Vpc.VpcId" --output text \
    --region $REGION --profile $PROFILE)

# Activer DNS hostnames (désactivé par défaut)
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID \
    --enable-dns-hostnames "{\"Value\":true}" \
    --region $REGION --profile $PROFILE

# ─── Étape 3 : Créer les subnets multi-AZ (3 tiers × 3 AZ) ─
for i in 1 2 3; do
    AZ="${REGION}$(echo $i | tr '123' 'abc')"
    # Subnet public (ALB/NLB uniquement) — /24
    aws ec2 create-subnet --vpc-id $VPC_ID \
        --cidr-block "10.0.$((i-1)).0/24" \
        --availability-zone $AZ \
        --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=pub-$AZ},{Key=Tier,Value=public}]" \
        --region $REGION --profile $PROFILE
    # Subnet privé application — /20
    aws ec2 create-subnet --vpc-id $VPC_ID \
        --cidr-block "10.0.$((16*i)).0/20" \
        --availability-zone $AZ \
        --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=app-$AZ},{Key=Tier,Value=private}]" \
        --region $REGION --profile $PROFILE
    # Subnet isolé data (pas de route NAT) — /24
    aws ec2 create-subnet --vpc-id $VPC_ID \
        --cidr-block "10.0.$((100+i)).0/24" \
        --availability-zone $AZ \
        --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=data-$AZ},{Key=Tier,Value=isolated}]" \
        --region $REGION --profile $PROFILE
done

# ─── Étape 4 : Internet Gateway (pour subnets publics) ──────
IGW_ID=$(aws ec2 create-internet-gateway \
    --tag-specifications "ResourceType=internet-gateway,Tags=[{Key=Name,Value=${VPC_NAME}-igw}]" \
    --query "InternetGateway.InternetGatewayId" --output text \
    --region $REGION --profile $PROFILE)

aws ec2 attach-internet-gateway --internet-gateway-id $IGW_ID \
    --vpc-id $VPC_ID --region $REGION --profile $PROFILE

# Créer route table publique + route 0.0.0.0/0 → IGW
PUB_RT=$(aws ec2 create-route-table --vpc-id $VPC_ID \
    --tag-specifications "ResourceType=route-table,Tags=[{Key=Name,Value=pub-rt}]" \
    --query "RouteTable.RouteTableId" --output text \
    --region $REGION --profile $PROFILE)

aws ec2 create-route --route-table-id $PUB_RT \
    --destination-cidr-block "0.0.0.0/0" --gateway-id $IGW_ID \
    --region $REGION --profile $PROFILE

echo "✅ VPC $VPC_ID créé avec subnets multi-AZ (3 tiers × 3 AZ)"

  default = "production"
}

  default = "10.0.0.0/16"
}

  default = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
}

  default = "production"

# ─── vpc.tf ─────────────────────────────────────────────────
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true    # Quick Win : activé (défaut=false)

  tags = {
    Name        = "prod-vpc"
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# ─── Subnets publics (ALB/NLB uniquement) ───────────────────
resource "aws_subnet" "public" {
  count             = length(var.azs)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, count.index)
  availability_zone = var.azs[count.index]

  # NE PAS activer map_public_ip_on_launch = true
  map_public_ip_on_launch = false

  tags = {
    Name = "pub-${var.azs[count.index]}"
    Tier = "public"
  }
}

# ─── Subnets privés application (/20 pour ENI capacity) ─────
resource "aws_subnet" "private" {
  count             = length(var.azs)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 4, count.index + 1)
  availability_zone = var.azs[count.index]

  tags = {
    Name = "app-${var.azs[count.index]}"
    Tier = "private"
  }
}

# ─── Subnets isolés data (pas de NAT, pas d'IGW) ───────────
resource "aws_subnet" "isolated" {
  count             = length(var.azs)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 8, 100 + count.index)
  availability_zone = var.azs[count.index]

  tags = {
    Name = "data-${var.azs[count.index]}"
    Tier = "isolated"
  }
}

# ─── Internet Gateway ───────────────────────────────────────
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "prod-vpc-igw" }
}

# ─── Route Table publique ───────────────────────────────────
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = { Name = "pub-rt" }
}

resource "aws_route_table_association" "public" {
  count          = length(var.azs)
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

# ─── Route Table isolée (pas de route externe) ─────────────
resource "aws_route_table" "isolated" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "isolated-rt" }
}

resource "aws_route_table_association" "isolated" {
  count          = length(var.azs)
  subnet_id      = aws_subnet.isolated[count.index].id
  route_table_id = aws_route_table.isolated.id
}

Erreur fréquente : le Security Group par défaut autorise tout le trafic sortant (0.0.0.0/0) et tout le trafic entrant depuis lui-même (self-referencing). Si toutes les ressources sont assignées au SG par défaut, elles communiquent entre elles sans restriction.
Analogie : les Security Groups sont des badges d'accès — chaque instance porte un ou plusieurs badges qui définissent précisément qui peut entrer et vers où elle peut aller. Les NACLs sont les vigiles à l'entrée de chaque étage (subnet) — ils filtrent indépendamment, avant même que le badge soit vérifié.

Pourquoi les Security Groups seuls ne suffisent pas ?

💡
SG vs NACLs — défense en profondeur :
  • Security Groups = stateful, au niveau instance/ENI, évaluent toutes les règles (pas d'ordre). Pas de règle DENY explicite possible.
  • NACLs = stateless, au niveau subnet, évaluées dans l'ordre numérique (première règle qui match). Supportent les règles DENY.
  • Combinaison recommandée : SG pour le filtrage fin par service + NACLs pour le blocage en masse (ex : bloquer une plage IP malveillante ou un pays entier).
  • Limitation SG : max 60 règles inbound + 60 outbound par SG, 5 SG par ENI = 300 règles max évaluées. Au-delà, utiliser prefix lists.
⚠️
Piège n°1 — Référencer 0.0.0.0/0 en ingress : une règle ingress 0.0.0.0/0:443 sur un SG d'instance EC2 signifie que toute adresse IP d'internet peut accéder au port 443. Même si l'instance est en subnet privé (pas d'IP publique), un attaquant pourrait atteindre cette instance via un pivot depuis une autre instance compromise.

Piège n°2 — NACLs stateless : contrairement aux SG, les NACLs nécessitent une règle de retour explicite (ports éphémères 1024-65535). Oublier les règles sortantes = trafic légitime bloqué.

Piège n°3 — Dépendances circulaires Terraform : deux SG qui se référencent mutuellement créent un cycle. Utiliser aws_security_group_rule séparé pour résoudre.

Configuration — AWS CLI

Bash — AWS CLI
# ─── Variables ──────────────────────────────────────────────
VPC_ID="vpc-0xxxxxxxxxxxx"   # votre VPC custom
REGION="eu-west-1"
PROFILE="production"

# ─── Étape 1 : Verrouiller le SG par défaut ─────────────────
DEFAULT_SG=$(aws ec2 describe-security-groups \
    --filters "Name=vpc-id,Values=$VPC_ID" "Name=group-name,Values=default" \
    --query "SecurityGroups[0].GroupId" --output text \
    --region $REGION --profile $PROFILE)

# Supprimer la règle egress all
aws ec2 revoke-security-group-egress --group-id $DEFAULT_SG \
    --ip-permissions '[{"IpProtocol":"-1","IpRanges":[{"CidrIp":"0.0.0.0/0"}]}]' \
    --region $REGION --profile $PROFILE

# Supprimer la règle ingress self-referencing
aws ec2 revoke-security-group-ingress --group-id $DEFAULT_SG \

Pourquoi un NAT Gateway par AZ ?

💡
  • Résilience : si un NAT GW tombe (ou si l'AZ a un incident), les instances des autres AZ conservent leur accès internet sortant
  • Performance : le trafic reste dans la même AZ, pas de coût inter-AZ (0,01 $/Go économisé)
  • Bandwidth : chaque NAT GW supporte jusqu'à 100 Gbps. Un seul NAT GW peut devenir un goulot d'étranglement pour les architectures à fort trafic
  • Coût : ~32 $/mois par NAT GW supplémentaire (tarif indicatif, avril 2026). Le surcoût est largement compensé par la résilience
⚠️
Piège n°1 — NAT Gateway et Lambda/ECS : chaque invocation Lambda ou task ECS en VPC utilise le NAT GW pour ses appels sortants. Avec des milliers d'invocations, la facture NAT explose. Utiliser des VPC Endpoints (S4) pour les appels AWS (SQS, SNS, Secrets Manager, etc.).

Piège n°2 — Port exhaustion : un NAT GW supporte ~55 000 connexions simultanées vers une même destination. Au-delà, les connexions échouent avec ErrorPortAllocation. Solution : répartir le trafic sur plusieurs destinations ou plusieurs NAT GW.

# ─── Créer NAT Gateway par AZ ───────────────────────────────
for i in 1 2 3; do
    AZ="${REGION}$(echo $i | tr '123' 'abc')"

    # Allouer une Elastic IP
    EIP_ID=$(aws ec2 allocate-address --domain vpc \
        --tag-specifications "ResourceType=elastic-ip,Tags=[{Key=Name,Value=nat-eip-$AZ}]" \
        --query "AllocationId" --output text \
        --region $REGION --profile $PROFILE)

    # Récupérer le subnet public de cette AZ
    PUB_SUBNET=$(aws ec2 describe-subnets \
        --filters "Name=vpc-id,Values=$VPC_ID" \
                  "Name=availability-zone,Values=$AZ" \
                  "Name=tag:Tier,Values=public" \
        --query "Subnets[0].SubnetId" --output text \
        --region $REGION --profile $PROFILE)

    # Créer le NAT Gateway
    NAT_ID=$(aws ec2 create-nat-gateway --subnet-id $PUB_SUBNET \
        --allocation-id $EIP_ID --connectivity-type public \
        --tag-specifications "ResourceType=natgateway,Tags=[{Key=Name,Value=nat-$AZ}]" \
        --query "NatGateway.NatGatewayId" --output text \
        --region $REGION --profile $PROFILE)

    # Créer route table privée pour cette AZ
    PRIV_RT=$(aws ec2 create-route-table --vpc-id $VPC_ID \
        --tag-specifications "ResourceType=route-table,Tags=[{Key=Name,Value=priv-rt-$AZ}]" \
        --query "RouteTable.RouteTableId" --output text \
        --region $REGION --profile $PROFILE)

    # Attendre que le NAT GW soit available
    aws ec2 wait nat-gateway-available --nat-gateway-ids $NAT_ID \
        --region $REGION --profile $PROFILE

    # Route 0.0.0.0/0 → NAT Gateway
    aws ec2 create-route --route-table-id $PRIV_RT \
        --destination-cidr-block "0.0.0.0/0" --nat-gateway-id $NAT_ID \
        --region $REGION --profile $PROFILE

    echo "✅ NAT GW $NAT_ID créé dans $AZ avec RT $PRIV_RT"
done

Endpoints prioritaires selon l'architecture

💡
Règle de priorisation : créer en premier les endpoints pour les services les plus appelés.
  • Toujours : S3 (Gateway, gratuit), DynamoDB (Gateway, gratuit)
  • Si ECS/EKS : ECR (api + dkr), CloudWatch Logs, STS, Secrets Manager, KMS
  • Si Lambda en VPC : STS, Secrets Manager, SQS/SNS (selon triggers)

  • Sécurité zéro-trust : KMS, STS, monitoring, logs

Transit Gateway & Multi-VPC Payant
Hub-and-spoke, route tables TGW, VPC Peering, segmentation multi-comptes
🚨
Erreur fréquente : utiliser du VPC Peering full-mesh entre 5+ VPC. Avec N VPC, il faut N×(N-1)/2 peerings (10 VPC = 45 peerings). Un Transit Gateway centralise tout en hub-and-spoke : 1 attachment par VPC.
Analogie : le Transit Gateway est un aéroport central (hub) — au lieu que chaque ville (VPC) ait des vols directs vers toutes les autres, tout le monde passe par le hub. Cela simplifie le routage et permet la segmentation via des route tables TGW.

Tableau Transit Gateway vs VPC Peering

Critère VPC Peering Transit Gateway Recommandation
Topologie Point-to-point (full mesh) Hub-and-spoke TGW dès 3+ VPC
Coût fixe 0 $ (gratuit) ~36 $/mois par attachment (tarif indicatif, avril 2026) Peering si 2 VPC uniquement
Coût transfert 0,01 $/Go inter-AZ / inter-région 0,02 $/Go Peering si gros volume <3 VPC
Segmentation Pas de route table (tout ou rien) Route tables TGW (isolation par env) TGW pour séparation prod/dev/shared
Transitivité Non transitif Transitif (A→Hub→B) TGW si on-prem + multi-VPC
Multi-compte Cross-account possible RAM sharing + cross-account TGW partagé via AWS RAM

Architecture de segmentation TGW

💡
Modèle recommandé : 3 route tables TGW pour isolation.
  • RT-Shared : VPC shared-services (DNS, AD, logging) — accessible par tous
  • RT-Production : VPC prod — accès shared + on-prem, PAS accès dev
  • RT-Development : VPC dev/staging — accès shared, PAS accès prod ni on-prem
Chaque VPC est associé à une seule route table et propage ses routes vers les RT autorisées.
⚠️
Piège n°1 — Default route table TGW : par défaut, tous les attachments sont associés à la même route table avec propagation automatique. Résultat : tous les VPC communiquent entre eux. Désactiver DefaultRouteTableAssociation et DefaultRouteTablePropagation.

Piège n°2 — CIDR overlap : les VPC connectés via TGW ne doivent PAS avoir de CIDR qui se chevauchent. Planifier les CIDR en amont avec un IPAM (Amazon VPC IPAM).

AWS Network Firewall Payant
Règles stateless/stateful (Suricata), TLS inspection, domain filtering
🚨
Erreur fréquente : ne pas mettre en place d'inspection du trafic et compter uniquement sur les Security Groups. Les SG ne font pas d'inspection applicative (pas de détection d'exfiltration DNS, pas de filtrage de domaines, pas d'IDS/IPS). AWS Network Firewall comble ce manque avec des règles Suricata.
Analogie : si les Security Groups sont des badges d'accès, le Network Firewall est le scanner à rayons X — il inspecte le contenu des paquets, pas juste les en-têtes.

Tableau AWS Network Firewall

Fonctionnalité Sans Network Firewall Avec Network Firewall Coût
Domain filtering Aucun — tout domaine accessible Allowlist de domaines HTTPS/TLS ~328 $/mois par endpoint + 0,065 $/Go (tarif indicatif, avril 2026) Payant
IDS/IPS Aucun (sauf Amazon GuardDuty pour détection) Règles Suricata stateful
TLS inspection Impossible Déchiffrement + inspection + re-chiffrement
Stateless rules NACLs uniquement Filtrage haut débit (IP/port/protocol)
Logging VPC Flow Logs (L3/L4) Alert + Flow logs L7 vers S3/CloudWatch/Kinesis

Quand déployer AWS Network Firewall ?

💡
  • Conformité : exigences réglementaires (PCI-DSS, HIPAA) imposant un IDS/IPS
  • Exfiltration : besoin de bloquer les domaines non autorisés (allowlist DNS)
  • TLS inspection : inspecter le trafic chiffré sortant
  • Centralisation : inspection centralisée via TGW (VPC d'inspection dédié)
  • Alternative légère : pour le simple filtrage DNS, Amazon Route 53 Resolver DNS Firewall (S7) est moins coûteux
⚠️
Piège n°1 — Routage asymétrique : le trafic doit traverser le firewall dans les deux sens (aller ET retour). Un routage asymétrique (aller via le FW, retour direct) casse l'inspection stateful. Toujours utiliser des subnets dédiés au firewall.

Piège n°2 — Coût : à ~328 $/mois/endpoint + 0,065 $/Go, le coût peut être significatif. Prévoir un VPC d'inspection centralisé (via TGW) plutôt qu'un firewall par VPC.

Pourquoi 2 tunnels + BGP ?

💡
Chaque connexion Site-to-Site VPN crée 2 tunnels IPsec vers 2 endpoints AWS distincts. Configurer BGP (protocole de routage dynamique) plutôt que du routage statique permet :
  • Failover automatique : si un tunnel tombe, BGP bascule en ~30s sans intervention
  • ECMP via TGW : agréger la bande passante de plusieurs connexions VPN (jusqu'à 50 Gbps)
  • Propagation des routes : les préfixes on-prem sont annoncés dynamiquement, pas de mise à jour manuelle
⚠️
Piège n°1 — ASN identique : le Customer Gateway et le Virtual Private Gateway doivent avoir des ASN BGP différents. Utiliser l'ASN privé 65000 des deux côtés provoque un rejet de session BGP.

Piège n°2 — MTU et MSS : les tunnels IPsec réduisent le MTU effectif à 1399 octets. Sans ajustement MSS clamping côté on-prem, les paquets >1399 sont fragmentés ou droppés, causant des timeouts mystérieux.

Monitoring & Troubleshooting Gratuit / Payant
VPC Flow Logs, Amazon GuardDuty, Reachability Analyzer, Athena queries
🚨
Erreur fréquente : ne pas activer les VPC Flow Logs. Sans eux, il est impossible de diagnostiquer les problèmes de connectivité, de détecter les scans de ports, ou de satisfaire les exigences d'audit. Les Flow Logs sont le minimum absolu de visibilité réseau.
Analogie : les VPC Flow Logs sont les caméras de surveillance du réseau. Elles n'empêchent rien, mais sans elles, on ne sait pas ce qui s'est passé.

Outil Fonction Quand l'utiliser Coût

Format VPC Flow Logs recommandé

💡
Utiliser le format personnalisé (v5) pour capturer plus de métadonnées que le format par défaut :
${version} ${account-id} ${interface-id} ${srcaddr} ${dstaddr} ${srcport} ${dstport} ${protocol} ${packets} ${bytes} ${start} ${end} ${action} ${log-status} ${vpc-id} ${subnet-id} ${az-id} ${sublocation-type} ${flow-direction} ${pkt-srcaddr} ${pkt-dstaddr} ${tcp-flags}

Destination recommandée :
  • S3 pour archivage long terme + Athena queries (le moins cher)
  • CloudWatch Logs pour alertes temps réel + Metric Filters
  • Kinesis Data Firehose pour SIEM/Splunk/OpenSearch
⚠️
Piège n°1 — Coût des Flow Logs : dans un VPC à fort trafic, les Flow Logs peuvent générer des Go/jour. Toujours envoyer vers S3 (0,023 $/Go) plutôt que CloudWatch Logs (0,50 $/Go ingestion). Utiliser Amazon Athena pour requêter.

Piège n°2 — REJECT uniquement : ne capturer que les REJECT est tentant pour réduire le volume, mais on perd la visibilité sur le trafic légitime. Capturer ALL, filtrer ensuite.

Cost Management Gratuit
Optimisation NAT Gateway, data transfer, comparaison Transit Gateway vs Peering
🚨
Erreur fréquente : ne pas surveiller les coûts réseau. Le data transfer est le poste de dépense le plus sous-estimé sur AWS. Un NAT Gateway à fort trafic peut coûter >500 $/mois sans qu'on s'en aperçoive.
Analogie : le réseau AWS est comme un péage autoroutier — on ne paie pas l'infrastructure, mais chaque passage (Go transféré) a un coût. Et les tarifs varient selon la direction (intra-AZ gratuit, inter-AZ payant, Internet payant).

Tableau des coûts réseau AWS

Type de transfert Coût / Go Optimisation Impact mensuel (1 To)
Intra-AZ (même AZ) 0 $ Placer les services communicants dans la même AZ 0 $
Inter-AZ 0,01 $/Go par direction Minimiser le cross-AZ inutile ~20 $ (aller+retour)
NAT Gateway processing 0,045 $/Go + 0,045 $/h (tarif indicatif, avril 2026) VPC Endpoints pour S3/DynamoDB (gratuit) ~78 $ (45 Go/jour + fixe)
Internet sortant 0,09 $/Go (1er Go gratuit) CloudFront (0,085 $/Go, gratuit vers origine) ~92 $
Transit Gateway 0,02 $/Go + ~36 $/mois/attachment VPC Peering si <3 VPC (gratuit) ~56 $ (20 Go/jour + fixe)
VPC Endpoint Gateway (S3/DDB) 0 $ Toujours utiliser (pas de raison de ne pas le faire) 0 $ (vs ~45 $ via NAT)

Tous les tarifs sont indicatifs (avril 2026), région us-east-1.

Top 5 des optimisations coût réseau

💡
  1. VPC Endpoints Gateway pour S3/DynamoDB — gratuit, économise le NAT Gateway
  2. Amazon CloudFront devant ALB — réduit le data transfer sortant
  3. VPC Peering au lieu de TGW si <3 VPC — pas de coût par Go
  4. Flow Logs vers S3 (pas CloudWatch) — 20x moins cher en ingestion
  5. AWS Cost Explorer avec filtre « Service = VPC » — identifier les postes inattendus
⚠️
Piège n°1 — NAT Gateway fantôme : un NAT Gateway inutilisé coûte ~32 $/mois (0,045 $/h × 730h). Vérifier les NAT GW sans trafic via CloudWatch BytesOutToDestination.

Piège n°2 — Trafic S3 via NAT : sans VPC Endpoint S3, tout le trafic S3 passe par le NAT Gateway (double coût : NAT processing + Internet). Avec l'Endpoint Gateway : 0 $.

#!/usr/bin/env bash
# ═══════════════════════════════════════════════════════════
# vpc-hardening.sh — Script all-in-one hardening réseau AWS
# Pré-requis : aws-cli v2, jq, $REGION, $PROFILE définis
# ═══════════════════════════════════════════════════════════
set -euo pipefail

REGION="${AWS_DEFAULT_REGION:-eu-west-1}"
PROFILE="${AWS_PROFILE:-default}"
PROJECT="myproject"
ENV="prod"
VPC_CIDR="10.0.0.0/16"

echo "🚀 Démarrage du hardening réseau — $PROJECT-$ENV ($REGION)"

# ─── S1 : Supprimer le VPC par défaut ───────────────────────
DEFAULT_VPC=$(aws ec2 describe-vpcs --filters "Name=is-default,Values=true" \
    --query "Vpcs[0].VpcId" --output text --region $REGION --profile $PROFILE)
if [[ $DEFAULT_VPC != "None" ]]; then
    echo "⚠️  VPC par défaut détecté : $DEFAULT_VPC"
    # Décommenter pour supprimer :
    # aws ec2 delete-vpc --vpc-id $DEFAULT_VPC --region $REGION --profile $PROFILE
fi

# ─── S1 : Créer le VPC custom ───────────────────────────────
VPC_ID=$(aws ec2 create-vpc --cidr-block $VPC_CIDR \
    --tag-specifications "ResourceType=vpc,Tags=[{Key=Name,Value=$PROJECT-$ENV-vpc}]" \
    --query "Vpc.VpcId" --output text --region $REGION --profile $PROFILE)
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-hostnames --region $REGION --profile $PROFILE
echo "✅ VPC créé : $VPC_ID"

# ─── S4 : VPC Endpoints (S3 + DynamoDB gratuits) ────────────
PRIVATE_RT=$(aws ec2 describe-route-tables --filters "Name=vpc-id,Values=$VPC_ID" \
    --query "RouteTables[0].RouteTableId" --output text --region $REGION --profile $PROFILE)
for svc in s3 dynamodb; do
    aws ec2 create-vpc-endpoint --vpc-id $VPC_ID --service-name "com.amazonaws.$REGION.$svc" \
        --route-table-ids $PRIVATE_RT --vpc-endpoint-type Gateway \
        --region $REGION --profile $PROFILE 2>/dev/null || echo "Endpoint $svc déjà existant"
done
echo "✅ VPC Endpoints Gateway (S3 + DynamoDB)"

# ─── S7 : DNS Firewall avec managed lists ───────────────────
FW_GROUP=$(aws route53resolver create-firewall-rule-group --name "$PROJECT-dns-fw" \
    --query "FirewallRuleGroup.Id" --output text --region $REGION --profile $PROFILE)
aws route53resolver associate-firewall-rule-group \
    --firewall-rule-group-id $FW_GROUP --vpc-id $VPC_ID --priority 101 \
    --name "$PROJECT-dns-fw-assoc" --region $REGION --profile $PROFILE
echo "✅ DNS Firewall associé"

# ─── S9 : VPC Flow Logs vers S3 ─────────────────────────────
ACCOUNT_ID=$(aws sts get-caller-identity --query "Account" --output text --profile $PROFILE)
LOG_BUCKET="arn:aws:s3:::flowlogs-${ACCOUNT_ID}-${REGION}"
aws ec2 create-flow-logs --resource-type VPC --resource-ids $VPC_ID --traffic-type ALL \
    --log-destination-type s3 --log-destination $LOG_BUCKET --max-aggregation-interval 60 \
    --region $REGION --profile $PROFILE
echo "✅ VPC Flow Logs activés vers S3"

# ─── S9 : GuardDuty ─────────────────────────────────────────
aws guardduty create-detector --enable --region $REGION --profile $PROFILE 2>/dev/null \
    || echo "GuardDuty déjà activé"
echo "✅ GuardDuty activé"

echo ""
echo "═══════════════════════════════════════════════════════════"
echo "✅ Hardening réseau terminé pour $PROJECT-$ENV"
echo "   VPC : $VPC_ID"
echo "   Prochaines étapes manuelles :"
echo "   - Créer les subnets (3 tiers × 3 AZ)"
echo "   - Configurer les Security Groups (S2)"
echo "   - Déployer les NAT Gateways (S3)"
echo "═══════════════════════════════════════════════════════════"
