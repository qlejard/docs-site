Stratégie de déploiement recommandée

1
Quick Win (~15 min, +30 pts) — Créer une Web ACL avec les AWS Managed Rules (Core Rule Set + Known Bad Inputs) + rate limiting à 2 000 req/5 min. Attacher à vos ressources (ALB, CloudFront, API Gateway).
2
Managed Rules & Logging (~30 min) — Ajouter les rule groups SQL injection, IP Reputation, Anonymous IP. Activer le logging vers S3 ou CloudWatch Logs.
3
Bot Control & Tuning (~2h) — Activer Bot Control (Common), tuner les faux positifs via count mode, créer des IP sets automatisés.
4
Shield Advanced & Firewall Manager — Évaluer le besoin Shield Advanced (ROI), déployer les policies Firewall Manager pour la gouvernance multi-comptes.

Scénario d'attaque : Sans Web ACL, vos applications web sont exposées directement à Internet. Un attaquant peut injecter du SQL dans un formulaire de login (' OR 1=1 --), exécuter du JavaScript malveillant via XSS dans un champ de recherche, ou scanner massivement vos endpoints API à la recherche de vulnérabilités. Analogie : une Web ACL, c'est le videur de la discothèque — sans lui, n'importe qui entre, y compris ceux qui veulent casser l'intérieur. Le videur vérifie chaque personne (requête HTTP) selon des règles précises avant de laisser entrer.

💡
Pourquoi chaque paramètre Web ACL compte :
  • Web ACL (existence) : sans Web ACL, zéro filtrage HTTP — toutes les requêtes atteignent votre application. Le OWASP Top 10 (injection, XSS, SSRF) est exploité dans 40% des breaches web (Verizon DBIR 2024). Réf. : AWS Well-Architected Framework, Security Pillar — SEC06.
  • DefaultAction = Allow : c'est le modèle « deny by rule » — seules les requêtes matchées par une rule sont bloquées. L'alternative (Block par défaut) est possible mais risquée sans whitelist exhaustive. Réf. : AWS WAF Developer Guide — Default action.
  • Scope : choisir REGIONAL pour ALB, API Gateway, AppSync, App Runner. Choisir CLOUDFRONT uniquement pour les distributions CloudFront (obligatoire, déployé en us-east-1). Se tromper de scope = Web ACL invisible pour la ressource. Réf. : AWS WAF API Reference — Scope.
  • Rule priority : les rules s'évaluent dans l'ordre numérique croissant (0 = première). La première rule qui matche détermine l'action. Un mauvais ordre = bypass de protection (ex: une rule Allow trop haute peut court-circuiter une rule Block). Réf. : AWS WAF Developer Guide — Rule evaluation.
  • IP Sets : permettent de whitelister des IP internes (monitoring, CI/CD) et de blacklister des IP malveillantes. Sans IP set, vous bloquez potentiellement vos propres systèmes ou laissez passer des attaquants connus. Réf. : CIS AWS Foundations Benchmark v7.0.0.
  • Labels : mécanisme de tagging des requêtes par les managed rules. Permettent des décisions multi-critères (ex: une requête labellisée « bot » ET « rate-exceeded » = block, sinon count). Essentiels pour le tuning avancé. Réf. : AWS WAF Developer Guide — Labels.
⚠️
Pièges courants :
  • Web ACL créée mais non associée : une Web ACL sans association à une ressource ne protège rien. Vérifiez toujours list-resources-for-web-acl.
  • Scope CLOUDFRONT en région autre que us-east-1 : les Web ACL scope CLOUDFRONT doivent être créées dans us-east-1. Erreur silencieuse si vous utilisez une autre région.
  • Limite de 1 500 WCU par Web ACL : les Web Capacity Units (WCU) limitent la complexité des rules. Chaque managed rule group consomme des WCU. Dépasser la limite = erreur de déploiement.
  • Les managed rules mises à jour sans prévenir : AWS met à jour les managed rules automatiquement. Une mise à jour peut bloquer du trafic légitime. Toujours monitorer après une mise à jour de version.

2.1 — Matrice des AWS Managed Rule Groups

Rule Group Description WCU ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût
AWSManagedRulesCommonRuleSet OWASP Top 10 : XSS, SQLi, LFI, RFI, size restrictions 700 ✅ Activer ✅ + override SizeRestrictions_BODY en Count ✅ + label matching Gratuit
AWSManagedRulesKnownBadInputsRuleSet Log4j/JNDI, Host header attacks, PROPFIND method 200 ✅ Activer ✅ ✅ Gratuit
AWSManagedRulesAmazonIpReputationList IP réputation : botnets, proxies, scanners connus 25 Optionnel ✅ Activer ✅ Gratuit
AWSManagedRulesAnonymousIpList VPN, Tor, hosting providers, proxies anonymes 50 Optionnel ✅ Activer en Count d'abord ✅ Block après tuning Gratuit
AWSManagedRulesSQLiRuleSet Détection SQL injection avancée 200 Optionnel ✅ Activer ✅ Gratuit
AWSManagedRulesLinuxRuleSet LFI spécifique Linux (/etc/passwd, /proc) 200 Optionnel Si backend Linux : ✅ Activer ✅ Gratuit
AWSManagedRulesWindowsRuleSet PowerShell injection, Windows-specific paths 200 Optionnel Si backend Windows : ✅ Activer ✅ Gratuit
AWSManagedRulesPHPRuleSet PHP injection, unsafe functions, file inclusion 100 Optionnel Si backend PHP : ✅ Activer ✅ Gratuit
AWSManagedRulesWordPressRuleSet Exploits WordPress spécifiques 100 Optionnel Si WordPress : ✅ Activer ✅ Gratuit
AWSManagedRulesBotControlRuleSet Bot détection (Common ou Targeted) 50 — — ✅ Voir S3 Bot Control
AWSManagedRulesACFPRuleSet Account Creation Fraud Prevention 50 — — Si création de comptes : ✅ Bot Control
AWSManagedRulesATPRuleSet Account Takeover Prevention 50 — — Si login : ✅ Bot Control
💡
Pourquoi les Managed Rules sont indispensables :
  • Temps de réaction : quand Log4Shell a été découvert (décembre 2021), AWS a poussé une règle de protection dans les 12 heures. Sans managed rules, vous deviez écrire votre propre regex en urgence.
  • Coût nul : les AWS Managed Rules sont gratuites (pas de coût supplémentaire au-delà du prix WAF par rule). Seuls Bot Control et ATP ont un coût additionnel.
  • WCU : budget total de 1 500 WCU par Web ACL. Le Core Rule Set (700 WCU) + Known Bad Inputs (200) + IP Reputation (25) + SQLi (200) = 1 125 WCU, laissant 375 WCU pour des custom rules.
  • Override en Count : chaque rule d'un managed rule group peut être overridée individuellement en Count au lieu de Block. Essentiel pour le tuning des faux positifs. Réf. : AWS WAF Developer Guide — Rule group override.
⚠️
Pièges courants :
  • Activer toutes les managed rules d'un coup : risque de bloquer du trafic légitime immédiatement. Toujours activer en Count d'abord pendant 48-72h, analyser les sampled requests, puis passer en Block.
  • Ignorer les versions : les managed rules ont des versions (ex: Version_1.2). Par défaut, AWS utilise la dernière version. Vous pouvez fixer une version pour éviter les surprises, mais perdez les mises à jour automatiques.
  • Dépasser le budget WCU : ajouter trop de managed rule groups sans vérifier les WCU → erreur WAFInvalidParameterException. Vérifiez avec check-capacity.
  • SizeRestrictions_BODY (Core Rule Set) : cette règle bloque les body > 8 Ko. Si votre API accepte des uploads ou des JSON volumineux, overridez cette règle en Count ou augmentez la limite.

Bot Control & Fraud Prevention Bot Control
Détection de bots, CAPTCHA/Challenge, Account Takeover Prevention (ATP), Account Creation Fraud Prevention (ACFP)
🚨
Scénario d'attaque : Un botnet automatise des centaines de milliers de tentatives de login par heure (credential stuffing). Les bots représentent jusqu'à 40% du trafic web mondial (Imperva Bad Bot Report 2024), dont la moitié sont malveillants. Sans détection de bots, vous ne distinguez pas un vrai utilisateur d'un script automatisé. Analogie : Bot Control est le détecteur de déguisements — il vérifie si la « personne » à l'entrée est vraiment humaine ou un robot qui imite un navigateur.

3.1 — Tableau de configuration

Paramètre / Composant Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût

Bot Control (Targeted) Désactivé — — Pour bots sophistiqués (fingerprinting JS) 10$/ACL + 10$/M req
CAPTCHA action Non configuré — CAPTCHA sur /login, /signup CAPTCHA dynamique selon le risque (label-based) Inclus WAF
Challenge action Non configuré — Challenge silencieux (JS SDK) Challenge + CAPTCHA en escalade Inclus WAF
Token domain config Non configuré — Token domain = domaine application Multi-domain token sharing Inclus WAF

ACFP (Account Creation Fraud) Désactivé — — Si création de compte : activer ACFP 10$/ACL + 2$/1000 créations

💡
Pourquoi Bot Control est critique :
  • Bot Control Common vs Targeted : Common détecte les bots basés sur les signatures connues (user-agent, patterns HTTP). Targeted utilise du fingerprinting JavaScript côté client pour détecter les bots qui imitent de vrais navigateurs. Le coût est 10x supérieur mais nécessaire pour les attaques sophistiquées. Réf. : AWS WAF Developer Guide — Bot Control.
  • CAPTCHA vs Challenge : le Challenge est silencieux (pas de friction utilisateur) et vérifie que le navigateur exécute du JavaScript. Le CAPTCHA est visible et demande une interaction humaine. Stratégie recommandée : Challenge d'abord, CAPTCHA en escalade si le challenge échoue. Réf. : AWS WAF Developer Guide — CAPTCHA and Challenge.
  • ATP (Account Takeover) : surveille les tentatives de login et détecte le credential stuffing (mots de passe issus de fuites). Utilise les données AWS sur les compromissions connues. Nécessite de déclarer le chemin de login (ex: /api/auth/login). Réf. : AWS WAF Developer Guide — ATP.
  • Scope-down essentiel : Bot Control s'applique à TOUTES les requêtes par défaut. Sans scope-down, vous payez 1$/M req sur TOUT votre trafic. Scope-down aux URIs critiques (/login, /api, /checkout) réduit le coût de 80-90%. Réf. : AWS Well-Architected Framework — Cost Optimization.
⚠️
Pièges courants :
  • Bot Control sans scope-down = facture surprise : un site à 100M requêtes/mois sans scope-down = 100 $/mois (Common) ou 1 000 $/mois (Targeted). Avec scope-down à /login (5M req) : 5 $/mois.
  • Bloquer les bots vérifiés (Googlebot, Bingbot) : Bot Control Common les détecte et les labelle comme « verified bots ». Ne les bloquez JAMAIS — cela tue votre SEO. Override ces rules en Allow.
  • CAPTCHA sur les API : un CAPTCHA sur un endpoint API consommé par un mobile ou SPA ne fonctionne pas. Utilisez le Challenge (silencieux) pour les API et le SDK JavaScript AWS WAF pour gérer les tokens.
  • ATP déclenché sans configurer le login path : ATP ne protège rien si vous ne déclarez pas le chemin et le payload de votre formulaire de login. Configuration obligatoire.

Rate Limiting & DDoS Shield Standard Shield Advanced

🚨
Scénario d'attaque : Votre API expose un endpoint /search sans rate limiting. Un attaquant envoie 50 000 requêtes/seconde depuis un botnet — votre ALB scale mais la facture explose, ou pire, votre base de données sature et l'application crash pour tous les utilisateurs. Shield est le bouclier anti-émeute — il protège contre les attaques massives (DDoS volumetriques L3/L4).

  • Shield Standard : protection automatique et gratuite contre les attaques volumetriques L3/L4 (SYN flood, UDP reflection, DNS amplification). Actif sur tous les services AWS (CloudFront, ALB, Route 53, Global Accelerator, EC2). Réf. : AWS Shield Developer Guide — Shield Standard.
  • Rate-based rules : la fenêtre est fixe à 5 minutes, le seuil minimum est 100. AWS compte les requêtes par IP source. Après dépassement, les requêtes suivantes sont bloquées jusqu'à la fin de la fenêtre. Réf. : AWS WAF Developer Guide — Rate-based rules.

  • Rate limit trop bas sur un site à fort trafic : 100 req/5 min sur une homepage populaire = blocage d'utilisateurs légitimes (surtout derrière un NAT d'entreprise où des dizaines d'utilisateurs partagent la même IP). Calibrez selon vos métriques réelles.
  • Rate limit sans scope-down : s'applique à toutes les URLs. Un utilisateur qui charge 10 images statiques consomme 10 requêtes. Scope-down aux endpoints API uniquement.
  • Shield Advanced sans health check Route 53 : Shield Advanced utilise les health checks pour détecter l'impact réel d'une attaque. Sans health check, la réponse est plus lente.
  • Shield Advanced inutile sur des workloads < 3 000 $/mois : le ROI est rarement positif pour les petites charges. Shield Standard + WAF rate limiting suffisent généralement.

Pourquoi CloudFront + WAF :
  • Scope CLOUDFRONT obligatoire en us-east-1 : les Web ACL associées à CloudFront DOIVENT être créées en us-east-1 avec scope = CLOUDFRONT. Un Web ACL REGIONAL ne peut PAS être associé à une distribution CloudFront. Erreur fréquente #1. Réf. : AWS WAF Developer Guide — Scope.
  • Double WAF (edge + origin) : déployez un WAF CLOUDFRONT sur la distribution (filtrage DDoS/bots à l'edge) ET un WAF REGIONAL sur l'ALB d'origine (filtrage applicatif spécifique). Réf. : AWS Well-Architected — Security Pillar.

⚠️

  • WAF créé dans la mauvaise région : créer un Web ACL en eu-west-1 puis tenter de l'associer à CloudFront → erreur. CloudFront = us-east-1 uniquement.
  • ALB public sans restriction SG : même avec CloudFront, l'ALB reste accessible par son DNS. Un simple nslookup révèle les IPs. Verrouillez le SG.
  • Terraform multi-région oublié : pour déployer un WAF CLOUDFRONT + REGIONAL, il faut deux providers Terraform (un us-east-1, un régional). Oubli fréquent → tout en REGIONAL → CloudFront non protégé.
  • Geo restriction bypass via VPN : la geo restriction se base sur la GeoIP de l'IP source. Un VPN la contourne. Pour des données sensibles, combinez avec de l'authentification.

  • Pré-requis obligatoire : AWS Organizations activé, avec « All Features » (pas juste la facturation consolidée). Le compte admin FMS doit être désigné depuis le management account. Réf. : AWS FMS Developer Guide — Prerequisites.
  • Policies = règles centralisées : une policy WAF crée automatiquement un Web ACL dans chaque compte/région cible et l'associe aux ressources correspondantes (ALB, API GW, CloudFront). Supprime le besoin pour chaque équipe de gérer son propre WAF.

  • Policy sans scope = déploiement partout : sans restriction par OU ou tags, la policy s'applique à TOUS les comptes de l'organisation, y compris sandbox et dev. Scopez à Production.
  • FMS + custom rules conflictuelles : FMS gère les « first rules » et « last rules » du Web ACL. Les équipes peuvent ajouter leurs propres rules entre les deux. Documentez clairement cette séparation.
  • Coût multiplicatif : 100$/policy/région. 3 policies (WAF, Shield, SG) × 4 régions = 1 200 $/mois. Limitez les régions et policies au strict nécessaire.

  • Nom du bucket/log group obligatoire : le préfixe aws-waf-logs- est OBLIGATOIRE pour S3 et CloudWatch Logs. Sans ce préfixe, WAF refuse la configuration. Réf. : AWS WAF Developer Guide — Logging.

  • Log filtering = contrôle des coûts : un site à 100M req/mois génère ~500 Go de logs WAF/mois si tout est loggé. Filtrez pour ne logger que les BLOCK et COUNT → réduction de 90%+. Réf. : AWS WAF Developer Guide — Log filtering.

  • Préfixe manquant = erreur silencieuse : un bucket my-waf-logs au lieu de aws-waf-logs-my-project → la configuration de logging échoue. Nom obligatoire : aws-waf-logs-*.

  • Sampled requests ≠ full logs : les sampled requests ne sont conservées que 3 heures et ne contiennent qu'un échantillon. Pour l'audit, le full logging est indispensable.
  • Cross-region logging : le bucket S3 DOIT être dans la même région que le Web ACL (ou us-east-1 pour CLOUDFRONT). Un bucket cross-region → erreur.

Testing & Tuning WAF pay-per-use
Count mode, sampled requests, faux positifs, regex tuning, playbooks de correction
🚨
Scénario : Vous activez le AWSManagedRulesCommonRuleSet en mode BLOCK directement en production. Résultat : 40% des requêtes légitimes sont bloquées — la règle SizeRestrictions_BODY rejette tous les uploads de fichiers > 8 Ko. Règle d'or : toujours déployer en COUNT mode d'abord, analyser les logs pendant 7 jours, puis passer en BLOCK. Analogie : le COUNT mode est la période d'essai du videur — il note qui il aurait refoulé sans vraiment bloquer personne.

8.1 — Tableau de configuration

Paramètre / Composant Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût
Mode de déploiement BLOCK immédiat COUNT 7 jours → BLOCK COUNT + analyse logs automatisée Pipeline CI/CD : COUNT → canary → BLOCK progressif Inclus WAF
Exclusion de règles Aucune (tout bloque) Identifier les faux positifs via logs Excluded rules (count individuel) Scope-down statements (exclure par path/IP) Inclus WAF
Label matching Non utilisé — Labels pour router en COUNT sélectif Labels + custom response (403 personnalisé) Inclus WAF
Regex pattern sets Non utilisé — Regex set pour paths légitimes à whitelister Regex set + scope-down pour exceptions ciblées Inclus WAF
Rollback plan Aucun Garder la config précédente Terraform state / versioning Blue/green Web ACL + switch instantané Inclus
Testing automatisé Aucun — Script curl de test (OWASP payloads) Tests intégrés CI/CD + rapports automatiques Gratuit
💡
Pourquoi le tuning est critique :
  • Faux positifs fréquents : les règles CRS (Common Rule Set) bloquent parfois des requêtes légitimes — uploads de fichiers, APIs JSON avec des payloads larges, ou des paramètres contenant du HTML (WYSIWYG editors). Sans tuning, vous bloquez vos propres utilisateurs.
  • COUNT mode = zéro risque : en COUNT, la règle évalue et labellise la requête mais la laisse passer. Vous voyez dans les logs ce qui SERAIT bloqué, sans impact utilisateur. Réf. : AWS WAF Developer Guide — Testing and tuning.
  • Excluded rules vs Scope-down : excluded_rule met une règle spécifique en COUNT dans tout le rule group. scope_down_statement applique la règle uniquement à un sous-ensemble de requêtes (ex: seulement /api/*). Scope-down est plus précis.
  • Version pinning des managed rules : AWS met à jour les managed rules. Un update peut introduire de nouveaux faux positifs. Utilisez version pour épingler une version et tester avant d'upgrader.
⚠️
Pièges courants :
  • Déployer en BLOCK sans période COUNT : cause numéro 1 de panne applicative liée au WAF. Toujours 7 jours minimum en COUNT pour un nouveau rule group.
  • Confondre override_action et action : pour un managed rule group, on utilise override_action = none (laisser les actions individuelles de chaque règle). override_action = count force TOUT le group en count. action ne s'applique qu'aux custom rules.
  • Regex trop permissif : un regex .* dans un allow path exclut TOUT du WAF. Soyez spécifique : ^/api/v[12]/uploads$.
  • Pas de rollback plan : en cas de faux positif en production, vous devez pouvoir revenir à la configuration précédente en secondes. Gardez le Terraform state ou utilisez deux Web ACL (blue/green).

8.2.3 — Script de test OWASP

Bash — Tests de validation WAF
#!/bin/bash
# Script de test WAF — vérifier que les règles bloquent bien
# Usage : ./test_waf.sh https://your-app.example.com

TARGET="$1"
PASSED=0
FAILED=0

test_rule() {
    local desc="$1" expected_code="$2"
    shift 2
    code=$(curl -s -o /dev/null -w "%{http_code}" "$@")
    if [ "$code" = "$expected_code" ]; then
        echo "[PASS] $desc (HTTP $code)"
        ((PASSED++))
    else
        echo "[FAIL] $desc — attendu $expected_code, reçu $code"
        ((FAILED++))
    fi
}

echo "=== Tests WAF sur $TARGET ==="

# SQL Injection
test_rule "SQLi dans query string" "403" \
    "$TARGET/?id=1'+OR+'1'='1"

# XSS
test_rule "XSS dans query string" "403" \
    "$TARGET/?q=<script>alert(1)</script>"

# Path traversal
test_rule "Path traversal" "403" \
    "$TARGET/../../etc/passwd"

# Requête légitime (ne doit PAS être bloquée)
test_rule "Requête légitime" "200" \
    "$TARGET/"

echo ""
echo "=== Résultats : $PASSED passés, $FAILED échoués ==="

🚨
Scénario : Votre architecture a un ALB, une API Gateway REST, et un AppSync GraphQL. Le WAF est configuré sur l'ALB uniquement — les deux autres endpoints sont exposés sans protection. Chaque point d'entrée public doit avoir son Web ACL. Analogie : chaque porte de la discothèque (entrée principale, entrée VIP, entrée artistes) a besoin de son videur.

Pourquoi c'est important :
  • Un Web ACL par ressource : un ALB ne peut avoir qu'un seul Web ACL associé. Mais un même Web ACL peut protéger plusieurs ressources du même type dans la même région. Consolidez quand les règles sont identiques.
  • IP Sets dynamiques : au lieu de hardcoder les IPs dans les règles, utilisez des IP Sets. Ils peuvent être mis à jour par Lambda (threat intel feeds) sans modifier le Web ACL. Réf. : AWS WAF Developer Guide — IP set reference.
  • Terraform module réutilisable : créez un module Terraform WAF avec des variables pour le scope, les managed rules, et les custom rules. Chaque service consomme le même module → cohérence garantie.
  • API Gateway : attention au stage : l'association WAF est sur le stage (prod, staging), pas sur l'API. Un nouveau stage déployé sans WAF = endpoint non protégé.
⚠️
Pièges courants :
  • Oublier un endpoint : si vous avez ALB + API Gateway + CloudFront, chacun doit avoir un Web ACL. Automatisez la détection avec Config Rules ou FMS.
  • Cross-region = Web ACL séparé : un Web ACL REGIONAL ne couvre qu'une région. Multi-région = un Web ACL par région. Seul CLOUDFRONT scope est global.
  • IP Set taille maximale : 10 000 CIDR par IP Set (default). Si vous avez plus d'IPs à bloquer, utilisez plusieurs IP Sets avec OR logic.
  • AppSync introspection : les queries d'introspection GraphQL (__schema) doivent être bloquées en production. Ajoutez une règle WAF custom pour bloquer les requêtes contenant __schema dans le body.

Cost Optimization WAF pay-per-use
Consolidation des Web ACLs, rule ordering, ROI Shield Advanced, Bot Control Common vs Targeted
🚨
Scénario : Votre facture WAF est de 2 500 $/mois. Vous avez 8 Web ACLs (un par environnement + un par service), dont 3 identiques. Les managed rules sont en double. Bot Control Targeted coûte 10$/1M requêtes mais 95% de votre trafic est légitime. Optimiser ne veut pas dire supprimer la protection — mais éliminer les doublons et choisir le bon niveau. Analogie : c'est comme avoir 3 videurs identiques à la même porte — un seul suffit, les deux autres sont du gaspillage.

10.1 — Grille tarifaire WAF & Shield

Composant Tarification Exemple 10M req/mois 💰 Optimisation ⚠️ Risque si supprimé
Web ACL 5 $/mois/Web ACL 5 $ (1 ACL) à 40 $ (8 ACLs) Consolider : 1 ACL partagé par région Faible (consolidation, pas suppression)
Rules 1 $/mois/rule 10 $ (10 rules) Consolider regex patterns en 1 rule Faible
Requêtes inspectées 0.60 $/1M requêtes 6 $ Scope-down pour exclure les health checks Faible
Bot Control Common 1 $/1M requêtes 10 $ Scope-down : appliquer uniquement à /login, /api Modéré
Bot Control Targeted 10 $/1M requêtes 100 $ Scope-down agressif ou downgrade à Common Élevé (perte détection avancée)
ATP (Account Takeover) 10 $/1M requêtes 100 $ (si appliqué à tout) Scope-down : uniquement /login, /auth Élevé
ACFP (Fraud Prevention) 10 $/1M requêtes 100 $ (si appliqué à tout) Scope-down : uniquement /register, /signup Élevé
Shield Advanced 3 000 $/mois + data transfer 3 000 $/mois ROI : coût d'une heure de downtime > 3 000 $ ? Critique (DDoS L7 non protégé)
💡
Stratégies d'optimisation :
  • Consolidation des Web ACLs : un même Web ACL peut protéger plusieurs ALB dans la même région. 8 ALBs identiques = 1 Web ACL, pas 8. Économie : 35 $/mois en Web ACLs seuls.
  • Scope-down = levier principal : Bot Control Targeted à 10$/1M req est prohibitif sur tout le trafic. Scope-down sur /login et /api/sensitive uniquement → typiquement 5% du trafic → divisé par 20.
  • Rule ordering (evaluate_on_first_match) : placez les règles les plus bloquantes en premier (rate-limiting, IP blocklist). Les requêtes bloquées tôt ne sont pas évaluées par les règles suivantes → réduction du coût par requête des managed rules.
  • ROI Shield Advanced : 3 000 $/mois se justifie si : (1) une heure de downtime coûte > 36 000 $/an, (2) vous avez besoin du SRT, (3) vous voulez la protection de coût DDoS. Pour un site vitrine à faible revenu : Shield Standard suffit.
⚠️
Pièges d'optimisation :
  • Supprimer au lieu de scope-down : ne supprimez JAMAIS Bot Control ou ATP pour économiser. Utilisez scope-down pour réduire le volume facturé sans perdre la protection.
  • Consolider des règles incompatibles : ne partagez pas un Web ACL entre une API REST et un site web statique — les règles de body inspection ne s'appliquent pas au contenu statique. Séparez par type d'application.
  • Ignorer les coûts CloudWatch : chaque règle avec visibility_config génère des métriques CloudWatch. 50 rules × 5 métriques = 250 métriques. Au-delà de 10 gratuits : 0.30$/métrique/mois.
  • Oublier les WAF inactifs : des Web ACLs non associés à aucune ressource coûtent toujours 5 $/mois + rules. Nettoyez régulièrement.

Script All-in-One — Hardening WAF Complet
Script exécutable en une seule fois — AWS CLI + Terraform
🚨
Prérequis : AWS CLI v2 configuré avec --profile PROD, Terraform ≥ 1.5, jq installé. Ce script est un audit uniquement (read-only) — aucune modification n'est effectuée.
📋 Script d'audit complet — waf_audit.sh
#!/bin/bash
# ═══════════════════════════════════════════════════════
# WAF & Shield — Script d'audit complet
# Usage : chmod +x waf_audit.sh && ./waf_audit.sh
# ═══════════════════════════════════════════════════════
set -euo pipefail

PROFILE="${AWS_PROFILE:-PROD}"
REGION="${AWS_REGION:-eu-west-1}"
SCOPE="REGIONAL"
REPORT="waf_audit_$(date +%Y%m%d_%H%M%S).txt"

log() { echo "[$(date +%H:%M:%S)] $*" | tee -a "$REPORT"; }
separator() { echo "═══════════════════════════════════════════" | tee -a "$REPORT"; }

log "Début de l'audit WAF — Profil: $PROFILE, Région: $REGION"
separator

# ── S1 : Web ACLs ──
log "S1 — Inventaire des Web ACLs"
ACLS=$(aws wafv2 list-web-acls --profile "$PROFILE" --region "$REGION" \
  --scope "$SCOPE" --query 'WebACLs[].{Name:Name,Id:Id,ARN:ARN}' --output json)
echo "$ACLS" | jq -r '.[] | STR1' | tee -a "$REPORT"
ACL_COUNT=$(echo "$ACLS" | jq length)
log "  Total Web ACLs : $ACL_COUNT"
separator

# ── S2 : Managed Rules ──
log "S2 — Vérification des managed rule groups"
for ARN in $(echo "$ACLS" | jq -r '.[].ARN'); do
  NAME=$(echo "$ACLS" | jq -r --arg arn "$ARN" '.[] | select(.ARN==$arn) | .Name')
  RULES=$(aws wafv2 get-web-acl --profile "$PROFILE" --region "$REGION" \
    --scope "$SCOPE" --name "$NAME" --id "$(echo "$ACLS" | jq -r --arg arn "$ARN" '.[] | select(.ARN==$arn) | .Id')" \
    --query 'WebACL.Rules[?Statement.ManagedRuleGroupStatement].Statement.ManagedRuleGroupStatement.Name' \
    --output text 2>/dev/null || echo "ERREUR")
  log "  $NAME : $RULES"
done
separator

# ── S7 : Logging ──
log "S7 — Vérification du logging"
for ARN in $(echo "$ACLS" | jq -r '.[].ARN'); do
  LOG_CFG=$(aws wafv2 get-logging-configuration --profile "$PROFILE" --region "$REGION" \
    --resource-arn "$ARN" --query 'LoggingConfiguration.LogDestinationConfigs[0]' \
    --output text 2>/dev/null || echo "PAS DE LOGGING")
  NAME=$(echo "$ACLS" | jq -r --arg arn "$ARN" '.[] | select(.ARN==$arn) | .Name')
  if [ "$LOG_CFG" = "PAS DE LOGGING" ]; then
    log "  ❌ $NAME : AUCUN logging configuré"
  else
    log "  ✅ $NAME : $LOG_CFG"
  fi
done
separator

# ── S9 : Ressources protégées ──
log "S9 — Ressources associées aux Web ACLs"
ORPHAN_COUNT=0
for ARN in $(echo "$ACLS" | jq -r '.[].ARN'); do
  NAME=$(echo "$ACLS" | jq -r --arg arn "$ARN" '.[] | select(.ARN==$arn) | .Name')
  RESOURCES=$(aws wafv2 list-resources-for-web-acl --profile "$PROFILE" --region "$REGION" \
    --web-acl-arn "$ARN" --query 'ResourceArns' --output json 2>/dev/null || echo "[]")
  RES_COUNT=$(echo "$RESOURCES" | jq length)
  if [ "$RES_COUNT" -eq 0 ]; then
    log "  ⚠️ $NAME : ORPHELIN (0 ressources)"
    ORPHAN_COUNT=$((ORPHAN_COUNT + 1))
  else
    log "  ✅ $NAME : $RES_COUNT ressource(s)"
  fi
done
log "  WAFs orphelins : $ORPHAN_COUNT"
separator

# ── S4 : Shield ──
log "S4 — Statut Shield Advanced"
SHIELD=$(aws shield describe-subscription --profile "$PROFILE" --region us-east-1 \
  --query 'Subscription.{State:SubscriptionState,End:EndTime}' --output json 2>/dev/null \
  || echo '{STR0:STR1}')
log "  $(echo "$SHIELD" | jq -r '.State')"
separator

# ── Résumé ──
log "═══ RÉSUMÉ ═══"
log "Web ACLs trouvées : $ACL_COUNT"
log "WAFs orphelins    : $ORPHAN_COUNT"
log "Rapport sauvegardé : $REPORT"
