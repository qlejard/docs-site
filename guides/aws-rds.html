1
Encryption at Rest & in Transit Gratuit CMK
Chiffrement des données au repos (KMS) et en transit (SSL/TLS) — 🔴 Critique
Réf. CIS AWS Foundations Benchmark v7.0.0 §3.2.1 (RDS encryption-at-rest) / AWS FSBP RDS.3, RDS.27
🚨
Scénario : Sans chiffrement at rest, un attaquant ayant accès aux snapshots EBS ou aux backups peut lire l'intégralité des données de votre base (PII, credentials, données métier). Sans SSL/TLS, les requêtes SQL transitent en clair sur le réseau — une attaque man-in-the-middle permet d'intercepter mots de passe et données.
Analogie : Le chiffrement at rest est un coffre-fort pour vos données au repos — même si quelqu'un vole le disque dur, il ne peut rien lire. Le SSL/TLS est le fourgon blindé qui transporte les données entre votre application et la base.
Point critique : L'encryption at rest ne peut PAS être activée après la création d'une instance RDS. Il faut recréer l'instance à partir d'un snapshot chiffré.

💡
Pourquoi chaque paramètre Encryption compte :
  • StorageEncrypted : sans chiffrement, les données sur les volumes EBS sous-jacents sont lisibles en clair. Un accès non autorisé aux snapshots ou au stockage physique expose l'intégralité des données. RGPD Art. 32 exige des mesures techniques de protection — le chiffrement est la mesure de base.
  • KmsKeyId (CMK vs default) : la clé aws/rds par défaut est partagée au niveau du compte et ne peut pas avoir de key policy restrictive. Une CMK dédiée permet le contrôle granulaire (qui peut chiffrer/déchiffrer), la rotation automatique, et l'audit via CloudTrail.
  • require_secure_transport / rds.force_ssl : sans cette option, les clients peuvent se connecter en clair. Un attaquant sur le réseau VPC (compromission d'une instance EC2) peut intercepter les requêtes SQL et les credentials. PCI DSS v4.0 Req. 4.2.1 exige le chiffrement en transit.
  • Certificat SSL/TLS : le certificat rds-ca-2019 a expiré. Les instances utilisant encore cet ancien CA échoueront à la validation de certificat. La rotation proactive évite une interruption de service.
  • Encrypted snapshots cross-region : un snapshot chiffré avec la clé par défaut aws/rds ne peut pas être partagé cross-account ni copié vers une autre région sans re-chiffrement. Utiliser une CMK dés le départ évite cette limitation critique pour le DR.
  • Réplicas : un read replica hérite du statut encryption du master. Si le master n'est pas chiffré, le replica ne le sera pas non plus — créant un point faible dans la chaîne de protection.
⚠️
Pièges clés :
  • Encryption irréversible à la création : vous ne pouvez pas activer l'encryption sur une instance existante non chiffrée. Il faut : snapshot → copie chiffrée du snapshot → restore depuis le snapshot chiffré. Cela implique un changement d'endpoint et un downtime.
  • Clé KMS default non partageable : aws/rds ne permet pas le partage cross-account. Si vous prévoyez du multi-compte (Organizations), utilisez une CMK dès le départ.
  • SSL forcé ≠ certificat validé : forcer SSL côté serveur ne garantit pas que le client valide le certificat. Configurer sslmode=verify-full (PostgreSQL) ou --ssl-mode=VERIFY_IDENTITY (MySQL) côté client.

1.3 — Configuration

🖥️ Portail AWS Console

1
Accéder à RDS Console > Create database
2
Dans la section Encryption, cocher Enable encryption
3
Sélectionner la AWS KMS key : choisir une CMK dédiée (pas (default) aws/rds)
4
Pour forcer SSL : Parameter groups > créer/éditer un parameter group > rds.force_ssl = 1 (PostgreSQL) ou require_secure_transport = 1 (MySQL)

💻 AWS CLI

Bash — Création d'une instance RDS chiffrée avec CMK
# Variables à adapter
REGION="eu-west-1"
PROFILE="audit-readonly"
KMS_KEY_ARN="arn:aws:kms:eu-west-1:123456789012:key/abcd1234-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
DB_INSTANCE_ID="prod-app-db"

# Création d'une instance RDS PostgreSQL chiffrée
aws rds create-db-instance \
    --db-instance-identifier $DB_INSTANCE_ID \
    --db-instance-class "db.r6g.large" \
    --engine "postgres" \
    --engine-version "16.4" \
    --allocated-storage 100 \
    --storage-type "gp3" \
    --storage-encrypted \
    --kms-key-id $KMS_KEY_ARN \
    --master-username "dbadmin" \
    --manage-master-user-password \
    --master-user-secret-kms-key-id $KMS_KEY_ARN \
    --no-publicly-accessible \
    --deletion-protection \
    --region $REGION \
    --profile $PROFILE \
    --output json

# Forcer SSL via parameter group (PostgreSQL)
aws rds create-db-parameter-group \
    --db-parameter-group-name "secure-postgres16" \
    --db-parameter-group-family "postgres16" \
    --description "PostgreSQL 16 - SSL forcé" \
    --region $REGION \
    --profile $PROFILE

aws rds modify-db-parameter-group \
    --db-parameter-group-name "secure-postgres16" \
    --parameters "ParameterName=rds.force_ssl,ParameterValue=1,ApplyMethod=pending-reboot" \
    --region $REGION \
    --profile $PROFILE

📦 Terraform

HCL — Instance RDS chiffrée avec CMK et SSL forcé
# KMS Key dédiée pour RDS
resource "aws_kms_key" "rds" {
  description             = "CMK pour chiffrement RDS"
  deletion_window_in_days = 30
  enable_key_rotation     = true
}

# Parameter group forçant SSL
resource "aws_db_parameter_group" "secure_postgres" {
  family = "postgres16"
  name   = "secure-postgres16"

  parameter {
    name  = "rds.force_ssl"
    value = "1"
  }
}

# Instance RDS chiffrée
resource "aws_db_instance" "main" {
  identifier     = "prod-app-db"
  engine         = "postgres"
  engine_version = "16.4"
  instance_class = "db.r6g.large"

  allocated_storage = 100
  storage_type      = "gp3"

  # Encryption at rest
  storage_encrypted = true
  kms_key_id        = aws_kms_key.rds.arn

  # SSL forcé
  parameter_group_name = aws_db_parameter_group.secure_postgres.name

  # Accès privé uniquement
  publicly_accessible    = false
  deletion_protection    = true

  # Master password géré par Secrets Manager
  manage_master_user_password   = true
  master_user_secret_kms_key_id = aws_kms_key.rds.arn
}

Network Isolation Gratuit
Isolation réseau : DB subnet groups, security groups, accès privé uniquement — 🔴 Critique
Réf. CIS AWS Foundations v7.0.0 §3.2.3 (RDS not publicly accessible) / AWS FSBP RDS.2, RDS.23
🚨
Scénario : Une instance RDS avec PubliclyAccessible=true et un security group ouvert en 0.0.0.0/0 est directement exposée sur Internet. Des scans automatisés détectent ces instances en minutes, suivis de tentatives de brute-force sur les credentials. En 2023, plusieurs failles RDS ont été exploitées via des security groups trop permissifs.
Analogie : Le security group est une liste d'invités — seules les personnes (adresses IP / security groups) nommément invitées peuvent accéder à votre base. Le DB subnet group est le quartier privé dans lequel votre base est installée, sans accès direct depuis la rue.

2.1 — Tableau de configuration complet

Paramètre Défaut AWS ⚡ Quick Win 🔒 Renforcé 🎯 Optimal Coût
PubliclyAccessible true (via console "Easy Create") false false false Gratuit
Security Group - Source default VPC SG (all traffic intra-SG) SG applicatif (sg-app → sg-db) SG applicatif uniquement, port 3306/5432 SG app + bastion uniquement, règles révisées mensuellement Gratuit
DB Subnet Group default (subnets publics si VPC par défaut) Subnets privés dédiés (pas de route IGW) Subnets privés + NAT Gateway si outbound nécessaire Subnets isolés (pas de NAT) + VPC endpoints Gratuit
VPC Endpoints (Interface) Non configuré Non configuré VPC Endpoint pour S3 (Gateway) + KMS, Secrets Manager (Interface) Tous les endpoints nécessaires + endpoint policy restrictive ~10 $/mois/endpoint
Network ACLs (NACLs) Allow all inbound/outbound Allow all (laisser par défaut) Restreindre aux CIDR applicatifs + port DB NACLs restrictives + SG (defense in depth) Gratuit
Multi-AZ false (single-AZ) true true Multi-AZ Cluster (3 AZ, 2 read replicas) 2× instance-heure
DNS Resolution / Hostnames Activé dans VPC par défaut Vérifier activé Activé Route53 private hosted zone avec alias CNAME Gratuit
💡
Pourquoi chaque paramètre Network compte :
  • PubliclyAccessible : quand activé, une adresse IP publique est assignée à l'instance et le DNS public résout vers cette IP. Même avec un SG restrictif, l'instance est découvrable par scan DNS/IP. La désactivation supprime l'IP publique.
  • Security Group : le SG par défaut du VPC autorise tout le trafic entre les ressources du même SG — un EC2 compromis aurait accès direct à la DB. Un SG dédié avec référence SG-à-SG (pas IP) est la meilleure pratique : seuls les serveurs applicatifs peuvent se connecter.
  • DB Subnet Group : définit dans quels subnets l'instance sera placée. Les subnets privés (sans route vers IGW) empêchent tout accès direct depuis Internet, même si PubliclyAccessible=true par erreur.
  • VPC Endpoints : permettent à RDS d'accéder à KMS (encryption), S3 (exports/imports), Secrets Manager sans passer par Internet. Réduit la surface d'attaque et améliore la latence.
  • Multi-AZ : en plus de la haute disponibilité, le failover automatique réduit l'impact d'une attaque DoS ciblant une AZ spécifique.
⚠️
Pièges clés :
  • VPC par défaut : le VPC par défaut créé automatiquement par AWS contient uniquement des subnets publics. Créer un VPC dédié avec des subnets privés pour la production. 🔗 Guide VPC — Architecture réseau
  • SG trop large : n'utilisez jamais 0.0.0.0/0 en source de SG pour un port DB (3306, 5432, 1521, 1433). Utilisez une référence source_security_group_id vers le SG applicatif.
  • Modification PubliclyAccessible : changer PubliclyAccessible de true à false nécessite un reboot (bref, mais planifier).
  • Cross-ref réseau : vérifier que les NACLs, route tables et security groups sont cohérents — un allow dans le SG ne sert à rien si le NACL bloque le trafic.

2.3 — Configuration

💻 AWS CLI

Bash — Création du DB subnet group et security group
# Variables
REGION="eu-west-1"
PROFILE="audit-readonly"
VPC_ID="vpc-0123456789abcdef0"
PRIVATE_SUBNET_A="subnet-aaaaaa"
PRIVATE_SUBNET_B="subnet-bbbbbb"
APP_SG_ID="sg-app-xxxxxxxxx"

# Créer le DB subnet group (subnets privés uniquement)
aws rds create-db-subnet-group \
    --db-subnet-group-name "db-private-subnets" \
    --db-subnet-group-description "Subnets privés pour RDS" \
    --subnet-ids $PRIVATE_SUBNET_A $PRIVATE_SUBNET_B \
    --region $REGION \
    --profile $PROFILE

# Créer le security group dédié RDS
DB_SG_ID=$(aws ec2 create-security-group \
    --group-name "sg-rds-prod" \
    --description "SG dédié RDS - accès app only" \
    --vpc-id $VPC_ID \
    --query "GroupId" \
    --output text \
    --region $REGION \
    --profile $PROFILE)

# Autoriser UNIQUEMENT le SG applicatif sur le port PostgreSQL
aws ec2 authorize-security-group-ingress \
    --group-id $DB_SG_ID \
    --protocol tcp \
    --port 5432 \
    --source-group $APP_SG_ID \
    --region $REGION \
    --profile $PROFILE

# Supprimer la règle outbound par défaut (all traffic)
aws ec2 revoke-security-group-egress \
    --group-id $DB_SG_ID \
    --protocol -1 \
    --port -1 \
    --cidr "0.0.0.0/0" \
    --region $REGION \
    --profile $PROFILE

# Désactiver l'accès public sur une instance existante
aws rds modify-db-instance \
    --db-instance-identifier "prod-app-db" \
    --no-publicly-accessible \
    --apply-immediately \
    --region $REGION \
    --profile $PROFILE

  security_group_id        = aws_security_group.rds.id

resource "aws_db_instance" "main" {
  # ... (encryption etc. cf. S1)

  db_subnet_group_name   = aws_db_subnet_group.private.name

  • Master User Password : avec --manage-master-user-password, AWS stocke le master password dans Secrets Manager et le fait tourner automatiquement. Zéro risque de password oublié dans un ticket Jira ou un fichier.
  • Application credentials : Secrets Manager + SDK (cache local) élimine les variables d'environnement en clair. Le SDK récupère automatiquement le secret en cours de rotation sans interruption.
  • Rotation automatique : un mot de passe qui ne tourne jamais est un mot de passe qui sera compromis un jour. La rotation automatique réduit la fenêtre d'exposition. La Lambda de rotation RDS native gère la double-version du secret (AWSCURRENT/AWSPREVIOUS) pour zéro downtime.

Pièges clés :
  • IAM Auth non supporté pour Oracle/SQL Server : l'IAM DB Authentication n'est disponible que pour MySQL, PostgreSQL, MariaDB et Aurora (MySQL/PostgreSQL). Pour Oracle/SQL Server, utiliser Secrets Manager avec rotation.
  • 20 conn/sec limit : pour les applications serverless (Lambda) avec forte concurrence, chaque invocation crée une connexion → risque de dépassement. Utiliser RDS Proxy dans ce cas.
  • Token expiration : le token IAM expire après 15 minutes. Les connexions longue durée (>15 min) restent validées tant que la connexion TCP est ouverte, mais une reconnexion nécessite un nouveau token.
  • Secrets Manager + multi-region : la réplication de secret cross-region est nécessaire si l'application a un DR plan dans une autre région.

Pourquoi chaque paramètre Performance Insights compte :
  • PerformanceInsightsEnabled : sans PI, le seul outil de diagnostic est Enhanced Monitoring (métriques OS) et CloudWatch (métriques DB agrégées). Vous ne voyez pas quelles requêtes spécifiques consomment les ressources. PI ajoute la dimension SQL — indispensable pour un diagnostic incident précis.
  • PerformanceInsightsRetentionPeriod : la rétention de 7 jours signifie qu'un incident détecté après une semaine n'a plus de traces. 31 jours couvre les incidents de fin de mois. 731 jours est requis par SOC 2 Type II (traçabilité sur 12 mois) et recommandé pour PCI DSS Req. 10.7.
  • PerformanceInsightsKMSKeyId : sans KMS, les textes des requêtes SQL (potentiellement contenant des PII) sont stockés non chiffrés dans le datastore AWS interne. Le chiffrement est exigé par RGPD Art. 32 pour les traitements de données personnelles.
  • Wait Events (db.load.avg) : le wait event identifie la cause racine d'un problème de performance (CPU, I/O, lock, réseau). Sans lui, vous voyez que le CPU est à 100% mais pas pourquoi. Avec PI, vous voyez exactement quel wait event consomme le plus — et savez où agir.
  • Top SQL Digest : dans 80% des incidents, une seule requête rogue cause 80% du problème. PI classe les requêtes par DB load — le coupable apparaît en 10 secondes d'investigation.
  • Enhanced Monitoring : PI donne la vue SQL, Enhanced Monitoring donne la vue OS (CPU steal, swap, métriques par processus). Combinés, ils permettent de distinguer un problème applicatif d'un problème d'infrastructure.
  • Counter Metrics (lock_waits) : les lock contentions sont silencieuses — aucune erreur n'est levée, l'application ralentit progressivement. Monitorer db.Locks.row_lock_waits.avg permet de détecter les deadlocks avant incident.
⚠️
Pièges à éviter :
  • PI ≠ audit log : Performance Insights trace les performances, pas les accès sécurité (qui a lu quoi). Pour l'audit de sécurité, utiliser pgAudit (PostgreSQL) ou audit_log (MySQL) documentés en S4 — les deux sont complémentaires.
  • Rétention non rétroactive : passer de 7 j à 31 j est instantané via modify-db-instance, mais les données antérieures ne sont pas conservées rétroactivement. Configurer dès le départ sur les environnements critiques.
  • Instances burstables (T3/T4) : sur les T-series, PI peut manquer de granularité lors des burst de CPU. Préférer les M-series ou R-series en production.
  • SQL digest masque les valeurs de bind : PI affiche les requêtes sous forme de digest (? à la place des paramètres). Pour récupérer les valeurs réelles lors d'un incident de sécurité, croiser avec les audit logs de la section S4.

La protection contre la suppression (DeletionProtection) est désactivée par défaut.
Analogie : Le PITR est une machine à remonter le temps — vous pouvez revenir à n'importe quelle seconde des 35 derniers jours. Sans backups, vous n'avez pas de machine à remonter le temps — vous avez juste un instant passé perdu à jamais.

Pourquoi chaque paramètre Backup compte :
  • BackupRetentionPeriod (7 → 35 jours) : la valeur par défaut de 1 jour est dangereusement courte. Un ransomware se propage souvent en silence pendant 2-3 semaines avant le chiffrement visible. 7 jours = Quick Win. 35 jours = recommandation ANSSI BP-100 §3.2 (règle 3-2-1 : 3 copies, 2 supports, 1 hors site). Pour les données comptables, Code de commerce Art. L123-22 impose 10 ans de conservation — nécessitant des snapshots manuels complémentaires.
  • DeletionProtection : sans cette option, un opérateur mal informé ou une IaC mal configurée peut supprimer irrémédiablement l'instance de production. DeletionProtection=true force un acte volontaire en deux étapes. Combiné à une SCP (Organizations), même un administrateur compromis ne peut pas supprimer l'instance sans approbation.
  • PreferredBackupWindow : un backup pendant les heures de pointe peut dégrader les performances (I/O snapshot). Une fenêtre à 02:00 UTC évite les impacts sur les utilisateurs europopséens et américains simultanément.
  • PITR — granularité 5 minutes : le PITR est actif dès que BackupRetentionPeriod ≥ 1. Il permet de restaurer à n'importe quelle seconde (précision binlog/WAL) dans la fenêtre de rétention. C'est la protection n°1 contre les suppressions accidentelles.
  • Snapshot cross-region : les backups dans la même région ne protègent pas contre une panne régionale AWS (rare mais possible). La copie cross-region répond aux exigences de PCI DSS Req. 12.3.4 (recovery multi-site) et ISO 22301 (continuité d'activité).
  • AWS Backup Vault Lock (WORM) : empêche la modification ou suppression des backups pendant la période de rétention, même par un administrateur. Protection contre les attaques ransomware ciblant les sauvegardes. Exigé par certaines réglementations financières.
⚠️
Pièges à éviter :
  • BackupRetentionPeriod = 0 désactive PITR : mettre 0 supprime tous les backups automatiques et désactive le PITR. Cette valeur ne doit jamais être utilisée en production. Si vous voulez économiser sur les backups, baissez le coût via le type de stockage plutôt que la rétention.
  • Restore d'un snapshot = nouvel endpoint : la restauration d'un snapshot crée une nouvelle instance RDS avec un nouvel endpoint DNS. Les applications doivent pointer vers le nouvel endpoint. Documenter et tester la procédure de basculement.
  • DeletionProtection n'empêche pas la modification : DeletionProtection protège contre la suppression mais pas contre la modification destructrice (réduction de la rétention, désactivation du chiffrement via snapshot). Utiliser IAM + SCP pour restreindre aussi les modifications.
  • Multi-AZ ≠ backup : Multi-AZ est une protection contre les pannes d'AZ (haute disponibilité), pas une solution de backup. Une suppression accidentelle de table se réplique immédiatement sur le standby Multi-AZ.

RDS Proxy Payant
Connection pooling, IAM auth via Proxy, intégration Lambda, failover accéléré — 🟡 Modéré
🚨
Scénario : Sans RDS Proxy, chaque invocation Lambda crée une nouvelle connexion vers RDS. Un pic de trafic avec 5 000 Lambdas concurrentes épuise le pool de connexions de l'instance (max_connections = ~500 sur db.t3.medium) en quelques secondes — toutes les nouvelles requêtes échouent avec « Too many connections ». L'application tombe.
Analogie : RDS Proxy est le standardiste d'un hôtel — les 5 000 clients (Lambdas) appellent le standardiste (Proxy) qui gère lui-même un pool limité de lignes directes (connexions RDS). Chaque client obtient une réponse sans saturer le central téléphonique.

Pourquoi chaque paramètre RDS Proxy compte :
  • RequireTLS : sans TLS, les connexions clients → Proxy transitent en clair. Comme les connexions vont souvent depuis Lambda via Internet (si Lambda sans VPC), le chiffrement est obligatoire. TLS est également exigé par PCI DSS Req. 4.2.1 pour toute transmission de données d'authentification.
  • IAM Authentication via Proxy : le Proxy centralise l'authentification. Les Lambdas obtiennent un token IAM temporaire (15 min TTL) sans jamais connaître le mot de passe RDS. Secrets Manager stocke les credentials réels et les fournit au Proxy — les applications n'ont jamais accès direct aux credentials de la base.
  • MaxConnectionsPercent = 50% : réserver 50% des connexions pour les accès directs (administrateur, outils d'audit, sessions d'urgence). Si le Proxy utilise 100%, une urgence nécessitant une connexion directe est impossible.
  • IdleClientTimeout réduit : des connexions inutilisées maintenues 30 minutes consomment des slots de connexion sans valeur ajoutée. Avec 900 secondes, les connexions Lambda inactives sont libérées 3x plus vite.
  • SessionPinningFilters : le pinning de session force une connexion exclusive (1 Lambda = 1 connexion RDS), annulant le bénéfice du pooling. Les commandes SET (ex: timezone) déclenchent le pinning. Utiliser EXCLUDE_VARIABLE_SETS pour les variables inoffensives.
  • Multi-AZ Proxy : le Proxy lui-même doit être Multi-AZ. Un Proxy Single-AZ crée un SPOF au niveau de la couche de connexion — exactement ce que vous vouliez éviter.
⚠️
Pièges RDS Proxy :
  • Proxy = latence additive : RDS Proxy ajoute ~1-2 ms de latence par requête. Pour les workloads nécessitant <5 ms de latence (trading, gaming temps réel), évaluer si ce surcoût est acceptable.
  • Pinning non détecté par défaut : si vos connexions sont systématiquement pinées, le Proxy ne sert à rien (1:1 mapping). Activer les métriques DatabaseConnectionsCurrentlySessionPinned pour détecter le problème.
  • Proxy non disponible pour tous les engines : RDS Proxy supporte MySQL 5.6/5.7/8.0, PostgreSQL 10-15 et Aurora MySQL/PostgreSQL. Il ne supporte pas Oracle, SQL Server, MariaDB — vérifier la compatibilité avant déploiement.
  • IAM auth via Proxy ≠ IAM auth directe : activer IAM auth sur le Proxy ne signifie pas que IAM auth directe sur RDS est activée. Les deux sont indépendants. Configurer les deux (S3) pour une sécurité complète.

Compliance & Governance Config / Security Hub
AWS Config rules, Security Hub FSBP, notifications d'événements, deletion protection — 🔵 Gouvernance
Réf. AWS FSBP / CIS AWS Foundations v7.0.0
🚨
Scénario : Sans AWS Config rules, une instance RDS avec PubliclyAccessible=true ou sans chiffrement peut rester non-détectée pendant des semaines. Lors d'un audit PCI DSS ou SOC 2, cette non-conformité représente un échec de contrôle majeur pouvant entraîner la suspension de la certification. Pourquoi la gouvernance automatique est critique :
  • AWS Config rules (détection continue) : contrairement à un audit ponctuel, Config évalue la conformité en temps réel à chaque changement de configuration. Une instance non conforme créée par un pipeline Terraform défaillant est détectée en minutes, pas en semaines.
  • Security Hub FSBP : le Foundational Security Best Practices (FSBP) d'AWS Security Hub agrège automatiquement les résultats de config rules RDS en findings centralisés. Les contrôles RDS.1 à RDS.27 couvrent toutes les sections de ce guide et permettent un score de conformité unifié.
  • EventBridge alertes CloudTrail : les événements RDS critiques (suppression, modification SG, désactivation backup) sont enregistrés dans CloudTrail et peuvent déclencher des EventBridge rules vers SNS, Slack, PagerDuty. Délai d'alerte typique : <30 secondes.
  • SCP (Service Control Policies) : les Config rules détectent mais n'empêchent pas. Les SCP (via AWS Organizations) bloquent préventivement les actions non conformes — un administrateur ne peut pas créer une instance non chiffrée, même s'il dispose des permissions IAM.
  • Conformance Packs : AWS fournit des packs pré-configurés pour PCI DSS, HIPAA, NIST 800-53, SOC 2 incluant des règles RDS pré-définies. Activer le Conformance Pack PCI DSS couvre automatiquement 15+ contrôles RDS.
⚠️
Pièges Compliance & Governance :
  • AWS Config multi-région non activé par défaut : Config doit être activé dans chaque région où vous avez des ressources RDS. Un enregistreur global existe mais les règles sont régionales. Utiliser CloudFormation StackSets pour déployer Config rules uniformément.
  • Security Hub = findings, pas remédiation : Security Hub signale les problèmes mais ne les corrige pas. Configurer des actions automatisées via Systems Manager Automation runbooks pour les remédiation critiques (ex: désactiver l'accès public automatiquement).
  • Config Rules à déclenchement vs périodique : certaines règles sont évaluées à chaque changement, d'autres périodiquement (toutes les 24h). Pour RDS, préférer CONFIGURATION_CHANGE trigger sur AWS::RDS::DBInstance.
  • DeletionProtection ≠ protection contre la modification : voir S6. La gouvernance complète nécessite à la fois DeletionProtection, Config Rules, SCPs, et IAM boundaries.

Cost Optimization RDS Aurora
Reserved Instances, Graviton, gp3, Aurora Serverless v2, optimisation I/O — 🔵 Optimisation
💡
Optimisation des coûts RDS : Les instances RDS représentent souvent 30-50% de la facture AWS d'une application. Une stratégie d'optimisation en 4 axes (instances, stockage, architectures, réservations) peut réduire la facture de 40-60% sans dégrader les performances ni la sécurité. Ces optimisations sont complémentaires des sections de sécurité précédentes — elles ne les remplacent pas.

💡
Décision Aurora I/O-Optimized vs Standard :
  • Aurora Standard : coût stockage + I/O facturés séparément (0,20 $/million I/O requests). Économique pour workloads à faible volume d'I/O.
  • Aurora I/O-Optimized : prix storage +25% mais 0$ d'I/O charges. Rentable si les I/O représentent plus de 25% de la facture Aurora. Calculer avec : I/O cost / total Aurora cost > 25% → basculer vers I/O-Optimized.
  • gp3 vs gp2 : gp3 offre 3 000 IOPS gratuites (vs gp2 proportionnel au volume). Pour un volume de 100 Go, gp3 est moins cher et plus flexible (IOPS configurable indépendamment).
  • Graviton check de compatibilité : la plupart des engines RDS (MySQL 8.0, PostgreSQL 12+, Aurora MySQL/PostgreSQL) sont compatibles Graviton. Oracle et SQL Server ne sont pas disponibles sur Graviton.
  • Stop/Start RDS (7 jours max) : AWS redémarre automatiquement une instance stoppée après 7 jours. Utiliser cette stratégie pour les environnements dev/staging avec un scheduler EventBridge.
⚠️
Pièges d'optimisation des coûts :
  • RI 3 ans = risque d'engagement : n'acheter des RI 3 ans que pour des instances dont vous êtes certain de la pérennité. Les Savings Plans Compute (EC2) ne s'appliquent pas à RDS — il faut des RI RDS spécifiques.
  • Ne pas réduire la rétention de backup pour économiser : l'économie sur les backups est mineure, le risque de perte de données est majeur. Le stockage backup gratuit est égal à 100% de la taille de l'instance.
  • gp3 migration = brève interruption possible : la migration de gp2 vers gp3 est généralement à chaud (online) sur RDS mais peut nécessiter une fenêtre de maintenance selon l'engine et la version. Vérifier la documentation AWS avant.
  • Aurora Serverless v2 cold start : avec MinACU=0, il y a un délai de réchauffement. Pour les applications à SLA strict (<100ms), utiliser MinACU=0,5 ou 1 pour conserver des ressources minimal.

Matrice d'interactions entre sections
Dépendances et flux de données entre les différentes sections du guide
💡
Les sections ne sont pas indépendantes. Certains contrôles sont prérequis pour d'autres — par exemple, la CMK KMS (S1) est nécessaire pour chiffrer les snapshots cross-region (S6) et les données Performance Insights (S5). Respecter cet ordre déploiement.
Section source Section cible Type de dépendance Description Impact si absent
S1 — Chiffrement (KMS CMK) S6 — Backup & PITR Prérequis fort Les snapshots héritent du chiffrement KMS de l'instance source. Sans CMK, les snapshots cross-region utilisent la clé aws/rds non portable. Snapshots non chiffrables cross-region avec CMK client
S1 — Chiffrement (KMS CMK) S7 — Aurora Specific Prérequis fort Le stockage Aurora partagé utilise la CMK assignée au cluster. Les clones Aurora héritent de cette CMK. Clones Aurora non chiffrés avec CMK dédiée
S1 — Chiffrement (KMS CMK) S5 — Performance Insights Dépendance optionnelle Performance Insights peut être chiffré avec une CMK dédiée via PerformanceInsightsKMSKeyId. PI chiffré avec clé aws/rds (non contrôlée)
S3 — IAM Auth (Secrets Manager) S8 — RDS Proxy Prérequis fort RDS Proxy nécessite le Secret ARN Secrets Manager pour gérer le pool de connexions. Sans Secrets Manager, le Proxy ne peut pas s'authentifier à la DB. RDS Proxy non fonctionnel sans Secrets Manager
S4 — Audit & Logging S9 — Compliance & Governance Renforcement mutuel Les logs CloudWatch / pgAudit sont la source de données pour les règles AWS Config et les findings Security Hub FSBP. Findings Security Hub incomplets sans logs d'audit
S5 — Performance Insights S10 — Cost Optimization Outil d'analyse PI fournit les métriques Average Active Sessions et CPU utilisées pour le rightsizing. Sans PI, le choix d'instance class est empirique. Rightsizing basé sur estimé sans données réelles
S6 — Backup & PITR S7 — Aurora Global Database Complémentarité Aurora Global Database assure la réplication continue (RPO ~1s) tandis que les backups cross-region couvrent la durabilité longue durée. Les deux sont nécessaires pour la stratégie DR complète. Gap DR : sans backup, Global DB failover insuffisant pour RTO <1h
S8 — RDS Proxy S3 — IAM Auth Renforcement mutuel Le Proxy avec IAMAuth=REQUIRED impose l'authentification IAM pour toutes les connexions client, renforçant S3. Proxy devient le point d'entrée unifié. Authentification directe DB possible si Proxy absent
S9 — Compliance (Config) S6 — Backup & PITR Contrôle automatique La règle rds-backup-enabled vérifie automatiquement BackupRetentionPeriod >0. Complète le contrôle manuel de S6. Config détecte automatiquement les instances sans backup
S9 — Compliance (Config) S2 — Réseau Contrôle automatique La règle rds-instance-public-access-check détecte toute instance avec PubliclyAccessible=true et alerte immédiatement. Détection automatique d'exposition réseau

  2. S2 — Isolation réseau (SG + subnets privés)
  3. S3 — IAM Auth + Secrets Manager (prérequis S8)
  4. S4 — Audit Logging CloudWatch + pgAudit
  5. S5 — Performance Insights + Enhanced Monitoring
