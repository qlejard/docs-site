📋
Ce document est un référentiel d'audit et de configuration couvrant l'intégralité de la synchronisation d'identités hybrides avec Microsoft Entra Connect. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un tenant.

Analogie : Entra Connect est le pont entre deux mondes — il relie votre annuaire Active Directory local au cloud Microsoft Entra ID. Sans ce pont, vos utilisateurs doivent gérer deux identités distinctes, deux mots de passe, et votre équipe IT perd la maîtrise centralisée.

Solutions écartées

🚫
Alternatives considérées puis écartées :
  • DirSync / AAD Sync — Obsolètes depuis 2017, plus de support Microsoft. Remplacés par Entra Connect.
  • MIM (Microsoft Identity Manager) — Solution complète mais surdimensionnée pour la synchronisation cloud uniquement. Réservé aux scénarios multi-forêts complexes avec provisioning RH.
  • Scripts PowerShell custom — Non supportés, pas de delta sync, pas de monitoring intégré. Risque de dérive silencieuse.
  • LDAP direct vers Entra ID — Non supporté nativement. Entra Connect est le seul connecteur officiel AD → Entra ID.

🚨
Scénario : Entra Connect est installé sur un contrôleur de domaine avec un compte Domain Admin comme compte de service, TLS 1.0/1.1 encore actif, et aucun serveur staging. Le serveur tombe — plus de synchronisation pendant 48h, les mots de passe ne se synchronisent plus, les nouveaux employés ne peuvent pas se connecter au cloud.

Analogie : Installer Entra Connect, c'est comme construire un pont entre deux villes. Si les fondations (prérequis) sont mal posées — mauvais sol, matériaux inadaptés, un seul pilier — le pont s'effondre au premier orage.

💡
Pourquoi chaque paramètre Installation compte :
  • Serveur dédié (pas DC) : installer Entra Connect sur un DC viole le principe de séparation des rôles. Si le DC est compromis, l'attaquant accède directement au connecteur cloud. Source : Microsoft Learn — Prerequisites.
  • Version V2 : Azure AD Connect V1 est retiré depuis le 31/08/2022. En plus, Microsoft impose une montée à 2.5.79.0 minimum avant le 30/09/2026 pour éviter l'arrêt des services de synchronisation après le changement de back-end annoncé en mai 2025.
  • TLS 1.2 : TLS 1.0/1.1 sont vulnérables (POODLE, BEAST). Microsoft exige TLS 1.2 pour toute communication Entra Connect → Entra ID. Source : Microsoft Learn — TLS enforcement.
  • gMSA : un Group Managed Service Account permet la rotation automatique du mot de passe toutes les 30 jours sans intervention humaine. Un compte classique oublié = mot de passe fixe pendant des années, cible idéale pour un attaquant.
  • SQL LocalDB vs Remote : LocalDB est limité à 10 Go (~100 000 objets). Au-delà, la sync ralentit puis échoue silencieusement. Les organisations >100k objets DOIVENT utiliser SQL Server remote. Source : Microsoft Learn.
  • Staging Server : sans doublure, la panne du serveur Entra Connect = arrêt complet de la synchronisation. Les nouveaux comptes ne se créent plus côté cloud, les mots de passe ne se synchronisent plus. Analogie : un avion sans copilote.
⚠️
Pièges courants :
  • Auto-Upgrade : activé par défaut uniquement en mode Express. En mode Custom, il faut l'activer manuellement via Set-ADSyncAutoUpgrade -AutoUpgradeState Enabled.
  • Proxy : si un proxy est présent, le fichier C:\Windows\Microsoft.NET\Framework64\v4.0.30319\Config\machine.config doit contenir la configuration proxy. Le wizard ne la détecte pas toujours.
  • Antivirus : exclure C:\Program Files\Microsoft Azure AD Sync\ et la base SQL LocalDB des scans temps réel, sous peine de corruption de la base sync.
  • Connector Account : le wizard en mode Express crée un compte MSOL_xxxxxxxx avec des permissions excessives. En mode Custom, créer un compte dédié avec les permissions minimum.

Architecture de Synchronisation Entra ID Free
Topologies, Connect Sync vs Cloud Sync, moteur de synchronisation — 12 pts
🚨
Scénario : Une organisation multi-forêts déploie Entra Connect avec la topologie par défaut (single forest). Résultat : seule une forêt est synchronisée, les utilisateurs de la seconde forêt ne peuvent pas accéder aux services cloud. Un delta import tourne en boucle, le CPU explose, les alertes Connect Health inondent la boîte mail.

Analogie : L'architecture de synchronisation, c'est le plan du réseau routier entre les deux villes. Une seule route (topologie) pour un trafic massif = embouteillage. Plusieurs routes bien dimensionnées = fluidité et résilience.

2.1 — Comparaison : Connect Sync vs Cloud Sync

Critère Entra Connect Sync Entra Cloud Sync Recommandation
Architecture Agent lourd (Windows Server dédié) Agent léger (provisioning agent) Cloud Sync préféré pour les nouveaux déploiements simples
Haute disponibilité Staging server (1 actif / 1 passif) Multi-agents natif Cloud Sync = HA native
Multi-forêts Support complet Support complet Équivalent
Attributs Exchange Hybrid ✅ Support complet ✅ Supporté Comparer les dépendances Connect spécifiques avant migration
Group Writeback v1 (legacy) ✅ Supporté ✅ Supporté Équivalent pour les scénarios legacy ; vérifier si la dépendance existe encore
Device Writeback ✅ ❌ Connect Sync si Hybrid Entra Join requis
Password Writeback ✅ ✅ Équivalent
Sync Rules personnalisées ✅ Éditeur complet ❌ Limité (attribute mapping) Connect Sync si transformations complexes
Gestion Console locale (Sync Service Manager) 100% cloud (portail Entra) Cloud Sync = gestion simplifiée
Intervalle de sync 30 min (configurable, min 30 min) 2 min (near real-time) Cloud Sync = quasi temps réel
💡
Pourquoi chaque paramètre Architecture compte :
  • Connect Sync (agent lourd) = pont en pierre. Robuste, supporte tout le trafic (Exchange, devices, sync rules custom), mais nécessite maintenance (serveur dédié, mises à jour, staging). Source : Microsoft Learn — Topologies.
  • Cloud Sync (agent léger) = pont suspendu moderne. Plus léger, HA native, sync quasi temps réel et nouvelles capacités prioritairement livrées dans cette plateforme, mais il reste des écarts sur le device sync, les règles avancées et certains cas Connect spécifiques. Source : Microsoft Learn — Cloud Sync.
  • Topologie : Microsoft supporte uniquement 1 instance Entra Connect Sync active par tenant. Deux serveurs actifs = corruption de données. Le staging server est la seule façon d'avoir de la redondance avec Connect Sync.
  • Multi-forêts : un seul serveur Connect Sync peut synchroniser plusieurs forêts AD. Chaque forêt nécessite un Connector Account dédié avec les bonnes permissions.

Pièges d'architecture :
  • Deux serveurs Connect Sync actifs = corruption garantie. Jamais de « load balancing » entre deux instances Connect Sync. Un seul actif, un en staging.
  • Cloud Sync n'est plus à exclure par principe en présence d'Exchange Hybrid : vérifier les attributs Exchange réellement utilisés, puis arbitrer surtout selon les besoins restant côté Connect Sync (device sync, règles avancées, dépendances historiques).
  • SourceAnchor mal choisi = impossible de changer après la première sync sans re-provisionner tous les objets. Utiliser ms-DS-ConsistencyGuid (défaut en V2). Analogie : c'est le numéro de sécurité sociale de chaque objet — on ne le change pas.
  • Soft Matching / Hard Matching : si un utilisateur existe déjà côté cloud, Connect tentera un soft match (par SMTP/UPN). Si échec = duplication d'objet. Vérifier avant la première sync que les UPN correspondent.

2.5 — Moteur de synchronisation : les 5 étapes

1
Import (AD DS) — Lecture des objets depuis Active Directory via LDAP. Les objets sont placés dans le AD Connector Space (zone de staging locale).
Analogie : scanner les passeports à la frontière.
2
Synchronization — Application des Sync Rules (règles de traduction) pour déterminer quels objets et attributs transitent vers le Metaverse (base centrale).
Analogie : le bureau de douane qui décide qui passe et avec quels bagages.
3
Export (Entra ID) — Envoi des modifications du Metaverse vers Microsoft Entra ID via HTTPS 443.
Analogie : envoyer les documents validés au pays de destination.
4
Import (Entra ID) — Lecture de confirmation depuis Entra ID pour vérifier que l'export a réussi.
5
Export (AD DS) — Writeback : écriture des modifications cloud vers AD on-prem (passwords, groups, devices) si activé. Entra ID P1

Méthodes d'Authentification Entra ID Free AD FS infra

🚨
Scénario : Une entreprise utilise AD FS depuis 2016. La ferme AD FS (2 serveurs + 2 WAP) tombe un samedi à cause d'un certificat expiré. Aucun utilisateur ne peut se connecter à Microsoft 365. Le PCA est déclenché. Temps de rétablissement : 6 heures.

Analogie : La méthode d'authentification, c'est le système de contrôle des passeports à la frontière. PHS = copie de secours du mot de passe déjà prête dans le pays d'arrivée. PTA = vérification en direct auprès du pays d'origine. AD FS = bureau de douane complet avec infrastructure dédiée.

💡
Pourquoi chaque paramètre Authentification compte :
  • Résilience maximale : si l'infrastructure on-premises tombe, les utilisateurs continuent de s'authentifier via le hash en cloud. Source : Microsoft Learn — PHS.
  • Leaked Credentials : seul PHS permet à Entra ID Protection de détecter si un mot de passe a fuité sur le dark web (comparaison hash). Source : Microsoft Learn — Identity Protection.
  • Sécurité du hash : Microsoft ne stocke pas le mot de passe en clair. Le processus applique un SHA256 + salt + PBKDF2 avant export. Le hash stocké en cloud ne peut pas être utilisé pour s'authentifier on-prem.
  • Migration AD FS : Microsoft pousse activement à migrer les déploiements AD FS vers PHS + Seamless SSO. Les claims rules custom restent le seul cas d'usage justifiant AD FS.
  • Backup auth : même avec PTA comme méthode primaire, toujours activer PHS en backup pour la résilience.

3.2 — Arbre de décision

1
Le mot de passe peut-il quitter l'AD on-prem (même sous forme de hash) ?
✅ OUI → PHS (recommandé) + Seamless SSO
❌ NON (contrainte réglementaire stricte) → Étape 2
2
Des claims rules custom ou MFA tiers sont-ils requis ?
❌ NON → PTA + Seamless SSO + PHS en backup
✅ OUI → Étape 3
3
AD FS (seul choix restant) — Planifier la migration vers PHS dès que les claims rules pourront être portées vers Conditional Access.

6.1 — Comparaison des fonctionnalités mot de passe

Fonctionnalité Description Licence Pré-requis Recommandation

💡
Pourquoi chaque paramètre Mots de Passe compte :
  • Cohérence : sans writeback, un changement de MDP cloud ne se propage pas on-prem. L'utilisateur a deux MDP différents. Source : Microsoft Learn — Password Writeback.
  • SSPR hybride : SSPR sans writeback = inutile en hybride. L'utilisateur reset en cloud mais ne peut plus se connecter on-prem.
  • Password Protection on-prem : le DC Agent intercepte les changements de MDP avant qu'ils soient appliqués dans AD. Les mots de passe faibles sont bloqués même en on-prem. Source : Microsoft Learn — Password Protection.

  • Smart Lockout + AD Lockout : si le seuil Smart Lockout cloud est inférieur au seuil AD, les tentatives peuvent verrouiller l'utilisateur on-prem avant que le cloud ne bloque. Aligner les seuils.

Pourquoi chaque paramètre Monitoring compte :
  • Détection précoce : une erreur de sync non détectée pendant 24h peut affecter des centaines d'utilisateurs (MDP expirés, groupes manquants, licences non assignées). Source : Microsoft Learn — Connect Health.
  • SLA interne : le cycle de sync par défaut est de 30 min. Si la sync s'arrête, les changements AD ne sont plus propagés. Les nouveaux employés n'ont pas accès à M365.

  • Agent Health hors ligne : l'agent Health nécessite un accès HTTPS sortant vers *.blob.core.windows.net, *.servicebus.windows.net et management.azure.com. Un proxy mal configuré bloque silencieusement la télémétrie.

  • Erreurs de sync ignorées : les erreurs d'export « InvalidSoftMatch » et « DataValidationFailed » sont fréquentes mais critiques. Ne pas les traiter comme du bruit.

Staging Server & Disaster Recovery Entra ID Free
Staging mode, failover, backup config, swing migration — 5 pts
🚨
Scénario : Le serveur Entra Connect principal tombe un vendredi soir (disque dur défaillant). Aucun staging server n'est déployé. La sync est arrêtée — les nouveaux comptes, changements de mot de passe et modifications de groupes ne sont plus propagés. La reconstruction prend 2 jours.

Analogie : Le staging server, c'est la doublure d'un acteur principal. Elle répète en permanence les mêmes scènes mais ne monte jamais sur scène — sauf quand l'acteur principal est indisponible.

9.1 — Comparaison des stratégies DR

Stratégie RTO Complexité Coût Recommandation
Staging Server (recommandé) 15-30 min Faible 1 VM supplémentaire Best practice Microsoft
Swing Migration 30-60 min Moyenne Temporaire (2ème serveur) Pour les upgrades majeures
Reconstruction complète 2-8 heures Haute Faible (pas de 2ème serveur) Non recommandé
Cloud Sync en fallback Variable Moyenne Agents cloud Alternative si Cloud Sync compatible
Backup/Restore config 1-4 heures Moyenne Faible Complément au staging server
💡
Pourquoi chaque paramètre Staging & DR compte :
  • Continuité : si la sync s'arrête plus de 36h, Microsoft peut désactiver la synchronisation du tenant (threshold par défaut). Un staging server permet un failover en minutes. Source : Microsoft Learn — Staging Mode.
  • Tests avant production : le staging server exécute les imports et la synchronisation sans exporter vers Entra ID. Idéal pour tester des changements de config, des nouvelles sync rules ou des upgrades.
  • Swing migration : pour upgrader sans downtime, installer la nouvelle version sur le staging, vérifier, puis inverser les rôles (staging → production, ancien production → staging).
  • Coût minimal : une VM Windows Server 2016+ avec 4 Go RAM suffit. Le coût est négligeable face au risque d'interruption.
⚠️
Pièges staging server :
  • Désactiver le scheduler sur le staging : le staging server NE doit PAS avoir le scheduler actif en mode staging. Vérifier que StagingModeEnabled = True et que le scheduler ne lance pas d'exports.
  • Configs désynchronisées : si vous modifiez les sync rules sur le serveur production, vous devez appliquer les mêmes changements sur le staging. Sinon le failover introduira des régressions.
  • Deux serveurs actifs : JAMAIS deux serveurs en mode actif (non-staging) simultanément. Cela corrompt les données Entra ID (conflits d'export, suppressions intempestives).
  • Password writeback : le staging server ne gère PAS le password writeback. Lors d'un failover, le writeback est brièvement interrompu.

Migration & Modernisation Entra ID Free
V1→V2, Connect Sync→Cloud Sync, AD FS→PHS/PTA — 5 pts
🚨
Scénario : Entra Connect V1 arrive en fin de support. L'admin reporte l'upgrade « parce que ça marche ». Un jour, Microsoft désactive la sync car la version est trop ancienne. Les comptes ne se synchronisent plus. L'upgrade d'urgence introduit des régressions non testées.

Analogie : La migration, c'est le changement de moteur d'un avion en vol. Mieux vaut le planifier en escale qu'attendre la panne moteur à 10 000 mètres.

10.1 — Comparaison des scénarios de migration

Migration Complexité Downtime Urgence Recommandation
V1 → V2 (in-place upgrade) Faible 1-2 cycles (30-60 min) V1 fin de support Priorité immédiate
V1 → V2 (swing migration) Moyenne 1 cycle (30 min) Haute Recommandé si environnement complexe
Connect Sync → Cloud Sync Haute Variable Moyenne Vérifier la compatibilité des features
AD FS → PHS + Seamless SSO Moyenne Staged Rollout = 0 downtime Haute (AD FS déprécié) Utiliser Staged Rollout
AD FS → PTA + Seamless SSO Moyenne Staged Rollout = 0 downtime Haute Alternative si PHS impossible
💡
Pourquoi chaque paramètre Migration compte :
  • V1 fin de support : Entra Connect V1 n'est plus supporté. Microsoft peut désactiver la sync pour les tenants utilisant des versions trop anciennes. Source : Microsoft Learn — What's new in V2.
  • AD FS déprécié : Microsoft recommande de migrer de AD FS vers PHS ou PTA. AD FS ajoute de la complexité (serveurs, certificats, DMZ) sans valeur ajoutée pour la plupart des scénarios. Source : Microsoft Learn — Migrate from federation.
  • Cloud Sync : agent léger (pas de serveur dédié), support multi-forêts simplifié, gestion cloud-native. Les nouvelles capacités arrivent prioritairement ici, y compris le provisioning de security groups vers AD. Les vrais écarts restants concernent surtout le device sync, certaines règles avancées et des dépendances Connect historiques.
  • TLS 1.2 : V2 exige TLS 1.2 (sécurité). V1 pouvait fonctionner avec TLS 1.0/1.1 dépréciés.
⚠️
Pièges migration :
  • In-place upgrade sans staging : toujours tester l'upgrade sur le staging server d'abord (swing migration). Un échec d'upgrade in-place = reconstruction complète.
  • Cloud Sync : vérifier les écarts restants : device writeback, sync rules custom avancées, et certaines dépendances legacy Connect. Pour le writeback de security groups, la voie supportée est désormais Cloud Sync ; Group Writeback V2 n'est plus la cible à déployer.
  • AD FS → PHS : ne pas migrer en Big Bang : utiliser le Staged Rollout pour migrer groupe par groupe. Un Big Bang affecte tous les utilisateurs simultanément.
  • Auto-upgrade désactivé : si l'auto-upgrade est désactivé (custom sync rules), vous devez planifier les upgrades manuellement. Ne pas laisser la version dériver.

Script All-in-One — Audit & Hardening Entra Connect
Script exécutable pour auditer et appliquer les best practices
PowerShell — Audit-EntraConnect.ps1
#Requires -RunAsAdministrator
#Requires -Modules ADSync
########################################################################
# Audit-EntraConnect.ps1
# Audit complet de la configuration Entra Connect
# Référence : guide Entra Connect Identité Hybride v1.0
########################################################################

$results = [System.Collections.ArrayList]::new()

function Add-Check {
    param($Section, $Check, $Status, $Detail)
    [void]$results.Add([PSCustomObject]@{
        Section = $Section; Check = $Check
        Status  = $Status;  Detail = $Detail
    })
}

## ═══ S1 — Installation & Prérequis ═══
Write-Host "[S1] Installation & Prérequis..." -ForegroundColor Cyan

# Version
$ver = (Get-ADSyncGlobalSettings).Parameters |
    Where-Object { $_.Name -eq "Microsoft.Synchronize.ServerConfigurationVersion" }
Add-Check "S1" "Version Entra Connect" $(
    if ($ver.Value -ge "2.0") { "OK" } else { "WARN" }
) $ver.Value

# TLS 1.2
$tls = Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.2\Client" -ErrorAction SilentlyContinue
Add-Check "S1" "TLS 1.2 Client" $(
    if ($tls.Enabled -eq 1) { "OK" } else { "FAIL" }
) "Enabled=$($tls.Enabled)"

## ═══ S3 — Authentification ═══
Write-Host "[S3] Authentification..." -ForegroundColor Cyan

$scheduler = Get-ADSyncScheduler
Add-Check "S3" "Sync Scheduler actif" $(
    if ($scheduler.SyncCycleEnabled) { "OK" } else { "FAIL" }
) "SyncCycleEnabled=$($scheduler.SyncCycleEnabled)"

## ═══ S5 — Sync Rules ═══
Write-Host "[S5] Sync Rules..." -ForegroundColor Cyan

$rules = Get-ADSyncRule
$oobModified = $rules | Where-Object { $_.ImmutableTag -and $_.IsStandardRule -eq $false }
Add-Check "S5" "Règles OOB modifiées" $(
    if ($oobModified.Count -eq 0) { "OK" } else { "WARN" }
) "$($oobModified.Count) règle(s) OOB modifiée(s)"

$customRules = $rules | Where-Object { -not $_.ImmutableTag }
Add-Check "S5" "Sync Rules custom" "INFO" "$($customRules.Count) règle(s) custom"

## ═══ S6 — Mots de passe ═══
Write-Host "[S6] Mots de passe..." -ForegroundColor Cyan

$phsConfig = Get-ADSyncAADPasswordSyncConfiguration -SourceConnector ((Get-ADSyncConnector | Where-Object { $_.ConnectorTypeName -eq "AD" }).Name)
Add-Check "S6" "PHS activé" $(
    if ($phsConfig.Enabled) { "OK" } else { "WARN" }
) "Enabled=$($phsConfig.Enabled)"

$pwdReset = Get-ADSyncAADPasswordResetConfiguration -Connector ((Get-ADSyncConnector | Where-Object { $_.ConnectorTypeName -eq "AD" }).Name)
Add-Check "S6" "Password Writeback" $(
    if ($pwdReset.Enabled) { "OK" } else { "WARN" }
) "Enabled=$($pwdReset.Enabled)"

## ═══ S7 — Features hybrides ═══
Write-Host "[S7] Features hybrides..." -ForegroundColor Cyan

$features = Get-ADSyncAADCompanyFeature
Add-Check "S7" "Group Writeback V2 (deprecated)" $(
    if ($features.GroupWritebackV2) { "WARN" } else { "INFO" }
) "Enabled=$($features.GroupWritebackV2)"

Add-Check "S7" "Device Writeback" $(
    if ($features.DeviceWriteback) { "OK" } else { "INFO" }
) "Enabled=$($features.DeviceWriteback)"

## ═══ S8 — Monitoring ═══
Write-Host "[S8] Monitoring..." -ForegroundColor Cyan

$healthService = Get-Service "Azure AD Connect Health Sync Insights Service" -ErrorAction SilentlyContinue
Add-Check "S8" "Connect Health Agent" $(
    if ($healthService -and $healthService.Status -eq "Running") { "OK" } else { "FAIL" }
) "Status=$(if ($healthService) { $healthService.Status } else { 'NotInstalled' })"

## ═══ S9 — Staging ═══
Write-Host "[S9] Staging..." -ForegroundColor Cyan

Add-Check "S9" "Staging Mode" $(
    if ($scheduler.StagingModeEnabled) { "INFO" } else { "OK" }
) "StagingMode=$($scheduler.StagingModeEnabled) (ce serveur est $(
    if ($scheduler.StagingModeEnabled) { 'STAGING' } else { 'PRODUCTION' }
))"

## ═══ Résultats ═══
Write-Host "`n═══ RÉSULTATS AUDIT ═══" -ForegroundColor Yellow
$results | Format-Table Section, Status, Check, Detail -AutoSize

$fails = ($results | Where-Object { $_.Status -eq "FAIL" }).Count
$warns = ($results | Where-Object { $_.Status -eq "WARN" }).Count
Write-Host "Total: $($results.Count) checks | $($fails) FAIL | $($warns) WARN" -ForegroundColor $(
    if ($fails -gt 0) { "Red" } elseif ($warns -gt 0) { "Yellow" } else { "Green" }
)

Checklist de déploiement
Cocher au fur et à mesure de la mise en œuvre

S1 — Installation & Prérequis (15 pts)

Serveur dédié (pas un DC) avec 16 Go RAM
TLS 1.2 enforcé (registry + .NET)
Entra Connect V2 installé (dernière version)
Compte de service gMSA ou dédié (pas admin domaine)
SQL Server Express ou SQL dédié (>100k objets)

S2 — Architecture de Synchronisation (12 pts)

Topologie validée (single forest / multi-forest)
Connecteurs AD configurés pour chaque forêt
Cycle de sync = 30 min (défaut validé)

S3 — Méthodes d'Authentification (18 pts)

PHS activé (au minimum comme backup)
PTA : 3 agents minimum si utilisé
Seamless SSO configuré
Clé Kerberos : rotation mensuelle planifiée

S4 — Filtrage & Scoping (10 pts)

Filtrage OU activé (pas de sync de l'arbre complet)
Comptes de service exclus
Filtrage documenté (OU listées)

S5 — Flux d'Attributs & Sync Rules (10 pts)

Aucune règle OOB modifiée directement
SourceAnchor = ms-DS-ConsistencyGuid
UPN = domaine routable (pas .local)
IdFix exécuté et erreurs corrigées

S6 — Gestion des Mots de Passe (12 pts)

PHS activé
Password Writeback activé
SSPR configuré (si P1)
Password Protection DC Agent déployé (si P1)
Liste de mots interdits custom ajoutée

S7 — Writeback & Fonctionnalités Hybrides (5 pts)

Group provisioning Cloud Sync évalué / activé ; Group Writeback V2 retiré si encore présent
Device Writeback configuré si Hybrid Join
OU cible créée pour les groupes writeback

S8 — Monitoring & Health (8 pts)

Connect Health agent installé et fonctionnel
Alertes email configurées
Sync errors dashboard consulté régulièrement

S9 — Staging Server & DR (5 pts)

Staging server déployé
Configuration identique à la production
Procédure de failover documentée et testée

S10 — Migration & Modernisation (5 pts)

Entra Connect V2 déployé
AD FS migré ou plan de migration en cours
Éligibilité Cloud Sync évaluée
