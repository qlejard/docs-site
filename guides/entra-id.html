Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de configuration
📋
Ce document est un référentiel générique couvrant la sécurisation de l'identité Microsoft Entra ID : Conditional Access, MFA, Identity Protection, PIM, break-glass et Workload Identities. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un tenant.

Analogie clé : Conditional Access = « badge d'accès intelligent » qui vérifie qui se connecte, d'où, avec quel device et à quel niveau de risque avant d'autoriser l'accès.

Security Defaults vs Conditional Access

⚠️
Transition critique : les Security Defaults (gratuits) activent MFA pour tous et bloquent le legacy auth — c'est un bon début. Mais dès qu'on a besoin de granularité (exclusions, policies par groupe, device compliance, risk-based), il faut les désactiver et basculer vers Conditional Access (P1).

Attention : désactiver Security Defaults sans créer de CA policies = zéro protection. Toujours créer les policies CA avant de désactiver Security Defaults.

1

2
Break-glass — Créer les comptes d'urgence et les exclure de toutes les CA policies (~15 min).
3
Identity Protection — Activer les risk-based policies (sign-in risk + user risk) si licence P2 (~30 min, +15 pts).

Stratégie Conditional Access P1
🔴 Critique — Le socle de toute sécurisation identité moderne
📋
Conditional Access est le « badge d'accès intelligent » de votre tenant : chaque connexion est évaluée selon qui (user/group), quoi (app), d'où (location/device), avec quel risque (sign-in/user risk) avant de décider : autoriser, bloquer, ou exiger un contrôle supplémentaire (MFA, compliant device).

Pourquoi chaque paramètre compte

💡
Rationnel de chaque paramètre clé :
  • Stratégie CA (3 policies min.) — Sans CA, l'accès au tenant repose uniquement sur le mot de passe. 3 policies de base (MFA admins, MFA all, block legacy) couvrent 80% des vecteurs d'attaque identité. CIS 5.2.2.1, 5.2.2.3
  • Report-only systématique — Une policy mal calibrée peut bloquer tous les utilisateurs. 7 jours en report-only = zéro impact sur la production, visibilité totale sur l'effet.
  • Break-glass exclusion — Sans exclusion, une panne MFA = lockout total. Le compte d'urgence est la seule garantie de reprise. CIS 1.1.2
  • Convention de nommage — Avec 10+ policies, sans convention, impossible de savoir qui fait quoi. Le format [Action]-[Target]-[App]-[Control] rend chaque policy lisible en un coup d'œil.
  • What-If avant go-live — L'addition des policies CA est imprévisible (la plus restrictive gagne). What-If simule le résultat exact avant activation.

Pièges fréquents

⚠️
Piège #1 — Désactiver Security Defaults sans CA : à la création d'un tenant M365, Security Defaults fournit une protection de base gratuite. Dès qu'on souscrit P1 pour Conditional Access, il faut d'abord créer les policies CA, puis désactiver Security Defaults. L'inverse laisse le tenant sans aucune protection MFA.

Piège #2 — Oublier les break-glass : une policy « MFA for all users » sans exclusion = lockout complet si le système MFA est indisponible (panne Authenticator, perte de tous les devices admin). Toujours exclure les comptes break-glass.

Piège #3 — Trop de policies = confusion : les policies CA s'évaluent de manière additive (la plus restrictive gagne). Un empilement non maîtrisé peut bloquer des accès légitimes. Utiliser report-only + What-If systématiquement.

Via Microsoft Graph PowerShell

PowerShell — Microsoft.Graph
# Connexion avec les permissions Conditional Access
Connect-MgGraph -Scopes "Policy.Read.All","Policy.ReadWrite.ConditionalAccess","Application.Read.All"

# ── CA001 : MFA pour les rôles admin ──
$params = @{
    DisplayName = "CA001-Grant-AdminRoles-AllApps-RequireMFA"
    State       = "enabledForReportingButNotEnforced"  # Report-only
    Conditions  = @{
        Users = @{
            IncludeRoles = @(
                "62e90394-69f5-4237-9190-012177145e10"  # Global Administrator
                "fe930be7-5e62-47db-91af-98c3a49a38b1"  # User Administrator
                "29232cdf-9323-42fd-ade2-1d097af3e4de"  # Exchange Administrator
                "194ae4cb-b126-40b2-bd5b-6091b380977d"  # Security Administrator
                "f28a1f50-f6e7-4571-818b-6a12f2af6b6c"  # SharePoint Administrator
            )
            ExcludeGroups = @("<BreakGlassGroupId>")
        }
        Applications = @{ IncludeApplications = @("All") }
    }
    GrantControls = @{
        BuiltInControls = @("mfa")
        Operator        = "OR"
    }
}
New-MgIdentityConditionalAccessPolicy -BodyParameter $params

# ── CA002 : MFA pour tous les utilisateurs ──
$params2 = @{
    DisplayName = "CA002-Grant-AllUsers-AllApps-RequireMFA"
    State       = "enabledForReportingButNotEnforced"
    Conditions  = @{
        Users = @{
            IncludeUsers  = @("All")
            ExcludeGroups = @("<BreakGlassGroupId>")
        }
        Applications = @{ IncludeApplications = @("All") }
    }
    GrantControls = @{
        BuiltInControls = @("mfa")
        Operator        = "OR"
    }
}
New-MgIdentityConditionalAccessPolicy -BodyParameter $params2

# ── CA003 : Bloquer l'authentification legacy ──
$params3 = @{
    DisplayName = "CA003-Block-AllUsers-LegacyAuth"
    State       = "enabledForReportingButNotEnforced"
    Conditions  = @{
        Users = @{
            IncludeUsers  = @("All")
            ExcludeGroups = @("<BreakGlassGroupId>")
        }
        Applications = @{ IncludeApplications = @("All") }
        ClientAppTypes = @("exchangeActiveSync", "other")
    }
    GrantControls = @{
        BuiltInControls = @("block")
        Operator        = "OR"
    }
}
New-MgIdentityConditionalAccessPolicy -BodyParameter $params3

Write-Host "✅ 3 policies créées en Report-only. Vérifier dans le portail Entra."

MFA & Authentication Strengths Free P1
🔴 Critique — Première ligne de défense contre le vol d'identité
📋
MFA ajoute un deuxième facteur de vérification au-delà du mot de passe. Microsoft rapporte que le MFA bloque 99,2% des attaques par compromission de compte.

Authentication Strengths (P1) permet de définir quel type de MFA est acceptable : un simple push Authenticator, un Passwordless, ou un FIDO2 phishing-resistant. C'est le passage du « badge simple » au « badge biométrique ».

CIS M365 v6.0.1 : 5.2.2.1 (MFA tous users), 5.2.2.5 (Phishing-resistant pour admins), 5.2.3.4, 5.2.3.5 (méthodes d'auth)

💡
Pourquoi chaque paramètre compte :
  • MFA activé (99%+ des users) — Microsoft rapporte que le MFA bloque 99,2% des attaques par compromission de compte. Chaque user sans MFA est une porte ouverte. CIS 5.2.2.1
  • Méthodes autorisées (Authenticator min.) — Le SMS est vulnérable au SIM swap et à l'interception SS7. Limiter aux méthodes résistantes réduit la surface d'attaque. CIS 5.2.3.5
  • Number matching — Sans number matching, un attaquant peut spammer les push MFA (MFA fatigue) jusqu'à ce que l'utilisateur accepte. Le number matching force la vérification active.
  • Per-user MFA désactivé — Per-user MFA (legacy) et CA s'appliquent indépendamment. Le per-user MFA force le MFA même depuis une trusted location, créant des conflits et une mauvaise expérience utilisateur.
  • Auth Strength phishing-resistant (admins) — Un Global Admin compromis = contrôle total du tenant. FIDO2/WHfB empêche le phishing même si l'admin clique sur un lien malveillant (le secret ne quitte jamais le device). CIS 5.2.2.5
  • Registration campaign — Sans incitation, les users repoussent l'enregistrement MFA. La campagne avec snooze limité (≤ 7j) garantit l'adoption avant l'enforcement.
si le per-user MFA (legacy portal) est activé en parallèle de Conditional Access, les deux s'appliquent indépendamment. Le per-user MFA force le MFA même quand la CA policy n'en demande pas (ex: trusted location). Migration obligatoire : désactiver le per-user MFA pour tous les users après déploiement CA.

Piège #2 — MFA Fatigue : un attaquant qui a le mot de passe peut spammer l'utilisateur de push MFA jusqu'à ce qu'il accepte par lassitude. Number matching (actif par défaut depuis 2023) atténue ce risque. Pour les admins, exiger du phishing-resistant (FIDO2 / WHfB).

Privileged Identity Management (PIM) P2
🟠 Important — Accès privilégié Just-In-Time, pas permanent
📋
PIM = « coffre-fort avec minuterie ». Au lieu de donner un rôle admin en permanence, on rend l'utilisateur éligible : il doit activer son rôle quand il en a besoin, avec justification, MFA, et une durée limitée (ex: 4h max). Après expiration, le rôle se désactive automatiquement.

Résultat : même si un compte admin est compromis, l'attaquant n'a aucun rôle actif — il devrait activer PIM, ce qui déclenche MFA + notification.

  • Approbation multi-niveau (GA, PRA) — Sans approbateur, un seul compte compromis peut s'auto-élever en Global Admin. L'approbation par un tiers casse cette chaîne d'attaque. CIS 5.3.4
  • Durée d'activation limitée (max 8h) — Plus la fenêtre d'activation est longue, plus le risque d'exploitation augmente. 4-8h couvre les opérations courantes sans laisser le rôle actif inutilement.
  • MFA à l'activation — Confirme que c'est bien l'admin légitime qui active le rôle, pas un attaquant qui a volé sa session.
  • Alertes à chaque activation — Sans alerte, une activation PIM par un attaquant passe inaperçue. L'alerte temps réel permet l'investigation immédiate.
  • Revues d'accès trimestrielles — Les rôles « eligible » non utilisés depuis 90j indiquent un accès obsolète. Les access reviews détectent et retirent ces privilèges dormants.

Piège #3 — Durée trop longue : une activation de 24h réduit l'intérêt du JIT. Limiter à 4-8h pour les rôles critiques.

Piège #2 — Pas de test : un compte break-glass jamais testé peut avoir un mot de passe expiré, une MFA non fonctionnelle, ou une policy qui le bloque sans qu'on le sache. Test trimestriel obligatoire.

Piège #3 — Domaine custom : utiliser un domaine custom (contoso.com) crée une dépendance à la fédération/DNS. Le .onmicrosoft.com fonctionne toujours.

Workload Identities Free WI Premium
🟡 Modéré — Sécuriser les identités non humaines
📋
Les workload identities sont les identités utilisées par les applications, scripts et services — pas par des humains. Elles incluent les Service Principals, App Registrations (avec secrets/certificates), et les Managed Identities.

Analogie : ce sont les « badges des robots » — les systèmes automatisés qui accèdent aux ressources. Un secret expiré = le robot ne fonctionne plus. Un secret volé = l'attaquant est le robot.

En chiffres : Microsoft rapporte que les workload identities sont 10x plus nombreuses que les identités humaines dans un tenant moyen, mais souvent négligées.

Pourquoi chaque paramètre compte :
  • Inventaire complet des SP/App Registrations — Impossible de sécuriser ce qu'on ne connaît pas. Les service principals oubliées de démos ou POCs restent avec des permissions actives et des secrets non-expirés indéfiniment.
  • Rotation des secrets (≤ 180j) — Un secret non-roté accumule le risque d'exposition. Au-delà de 180j, la probabilité qu'il ait fuité (logs, repo, email) augmente significativement.
  • Managed Identity préférée — Les Managed Identities n'ont pas de secret à gérer, donc pas de rotation, pas de fuite possible. C'est le gold standard pour les workloads Azure. Zero Trust principle
  • Least privilege Graph API — Une app avec Directory.ReadWrite.All peut créer des Global Admins. Auditer chaque permission et réduire au strict nécessaire. CIS 5.1.5.1
  • Pas de multi-tenant sauf besoin — Une app multi-tenant autorise n'importe quel tenant à générer un service principal chez eux avec vos permissions. Limiter au single-tenant sauf besoin SaaS explicite.
⚠️
Piège #1 — Secrets qui expirent silencieusement : un client secret qui expire = l'app/script ne fonctionne plus. Sans monitoring, c'est découvert par l'incident de production.

Piège #2 — Permissions excessives : beaucoup de service principals ont des permissions Graph comme Directory.ReadWrite.All alors qu'elles n'ont besoin que de User.Read.All. Auditer et réduire.

Piège #3 — Secrets dans le code : des secrets copiés dans des scripts, des configs git, ou des variables d'environnement non sécurisées. Migrer vers Managed Identity ou Azure Key Vault.

Identity Governance (aperçu) P2 Governance / Suite
🟡 Modéré — Contrôler qui a accès à quoi et recertifier régulièrement
📋
Identity Governance répond à la question : « est-ce que les bons utilisateurs ont toujours les bons accès ? ». Deux piliers principaux :
  • Access Reviews (socle P2, capacités avancées en Governance) : demander périodiquement aux managers/owners de recertifier les accès (groupes, apps, rôles PIM). Les accès non confirmés sont automatiquement révoqués.
  • Entitlement Management (socle historiquement disponible en P2, avancé en Governance) : créer des « access packages » regroupant groupes + apps + rôles, avec demande self-service, approbation, expiration automatique.
Analogie : Access Reviews = inventaire annuel des clés d'accès. Entitlement Management = système de demande de badges avec approbation automatique et expiration.

Ce guide ne couvre qu'un aperçu — un guide dédié Identity Governance est recommandé pour les déploiements avancés.
⚠️
Piège #1 — Access Reviews sans auto-remove : si l'action « Upon completion » est configurée sur « No change », les accès non confirmés restent actifs. Toujours configurer Auto-remove access pour que la non-réponse du reviewer entraîne la révocation.

Piège #2 — Auto-review par le propriétaire : un admin qui revoit ses propres privilèges n'a aucune raison de les révoquer. Désigner un manager ou le CISO comme reviewer pour garantir une revue objective. CIS 5.3.3

Piège #3 — Gouvernance des invités sans guest billing : les scénarios Identity Governance appliqués aux comptes invités reposent sur le modèle MAU / guest billing et nécessitent une souscription Azure sélectionnée pour la facturation des guests.

Pourquoi chaque paramètre compte :
  • Access Reviews trimestrielles — Sans revue périodique, les accès accumulés ne sont jamais retirés (privilege creep). Un collaborateur qui change de poste garde ses anciens rôles indéfiniment.
  • Auto-remove à expiration — Si le reviewer ne répond pas, l'accès doit être retiré automatiquement (deny par défaut). Sans cela, l'inaction = maintien des privilèges.
  • Reviewer ≠ propriétaire de l'accès — L'auto-review (un admin revoit ses propres accès) n'apporte aucune valeur. Désigner un manager ou un owner du groupe différent garantit un regard objectif.
  • Guests included — Les comptes invités (B2B) sont souvent créés pour un projet puis oubliés. Inclure les guests dans les access reviews élimine les accès orphelins. CIS 5.3.2
  • Lifecycle Workflows (joiner/mover/leaver) — Automatiser l'attribution et le retrait d'accès selon le cycle de vie RH élimine la dépendance au ticket helpdesk et les délais humains.

Script d'audit All-in-One
Script Microsoft Graph PowerShell pour auditer l'ensemble du tenant Entra ID
⚠️
Ce script est en lecture seule (audit). Il ne modifie aucune configuration. Exécuter avec un compte ayant les rôles Global Reader ou Security Reader. Module requis : Microsoft.Graph (Install-Module Microsoft.Graph).
PowerShell — Audit Entra ID Complet
#Requires -Modules Microsoft.Graph.Authentication, Microsoft.Graph.Identity.SignIns,
#  Microsoft.Graph.Identity.Governance, Microsoft.Graph.Users, Microsoft.Graph.Groups,
#  Microsoft.Graph.Applications, Microsoft.Graph.Reports

# ══════════════════════════════════════════════════
# PHASE 0 : Connexion
# ══════════════════════════════════════════════════
$scopes = @(
    "Policy.Read.All", "Directory.Read.All", "IdentityRiskEvent.Read.All",
    "IdentityRiskyUser.Read.All", "UserAuthenticationMethod.Read.All",
    "AuditLog.Read.All", "Application.Read.All", "RoleManagement.Read.Directory",
    "AccessReview.Read.All"
)
Connect-MgGraph -Scopes $scopes
Write-Host "✅ Connecté à Microsoft Graph" -ForegroundColor Green

$results = [ordered]@{}

# ══════════════════════════════════════════════════
# PHASE 1 : S1 — Conditional Access
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S1 — Conditional Access" -ForegroundColor Cyan
$policies = Get-MgIdentityConditionalAccessPolicy -All
$results["S1_TotalPolicies"] = $policies.Count
$results["S1_EnabledPolicies"] = ($policies | Where-Object State -eq "enabled").Count
$results["S1_ReportOnly"] = ($policies | Where-Object State -eq "enabledForReportingButNotEnforced").Count
$results["S1_Disabled"] = ($policies | Where-Object State -eq "disabled").Count

# Security Defaults
$secDef = Invoke-MgGraphRequest -Method GET -Uri "https://graph.microsoft.com/v1.0/policies/identitySecurityDefaultsEnforcementPolicy"
$results["S1_SecurityDefaultsEnabled"] = $secDef.IsEnabled

$policies | Select-Object DisplayName, State, CreatedDateTime | Format-Table -AutoSize

# ══════════════════════════════════════════════════
# PHASE 2 : S2 — MFA
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S2 — MFA Registration" -ForegroundColor Cyan
$mfaReport = Get-MgReportAuthenticationMethodUserRegistrationDetail -All
$results["S2_TotalUsers"] = $mfaReport.Count
$results["S2_MfaCapable"] = ($mfaReport | Where-Object IsMfaCapable).Count
$results["S2_MfaRegistered"] = ($mfaReport | Where-Object IsMfaRegistered).Count
$results["S2_PasswordlessCapable"] = ($mfaReport | Where-Object IsPasswordlessCapable).Count
$mfaPct = if ($mfaReport.Count -gt 0) { [math]::Round($results["S2_MfaRegistered"] / $mfaReport.Count * 100, 1) } else { 0 }
$results["S2_MfaPercentage"] = "$mfaPct%"
Write-Host "   MFA enregistré : $($results['S2_MfaRegistered']) / $($results['S2_TotalUsers']) ($mfaPct%)"

# ══════════════════════════════════════════════════
# PHASE 3 : S3 — Legacy Auth
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S3 — Legacy Auth Sign-ins (7j)" -ForegroundColor Cyan
$legacyApps = @("Exchange ActiveSync","IMAP4","POP3","SMTP","Other clients","Autodiscover")
$startDate = (Get-Date).AddDays(-7).ToString("yyyy-MM-ddTHH:mm:ssZ")
$legacySignIns = Get-MgAuditLogSignIn -Filter "createdDateTime ge $startDate" -Top 500 |
    Where-Object { $_.ClientAppUsed -in $legacyApps }
$results["S3_LegacySignIns7d"] = $legacySignIns.Count
Write-Host "   Sign-ins legacy (7j) : $($legacySignIns.Count)"

# ══════════════════════════════════════════════════
# PHASE 4 : S4 — Risky Users
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S4 — Risky Users" -ForegroundColor Cyan
try {
    $riskyUsers = Get-MgRiskyUser -Filter "riskState eq 'atRisk'" -All
    $results["S4_RiskyUsersAtRisk"] = $riskyUsers.Count
    Write-Host "   Users at risk : $($riskyUsers.Count)"
} catch {
    $results["S4_RiskyUsersAtRisk"] = "N/A (P2 requis)"
    Write-Host "   Risky Users : P2 requis" -ForegroundColor Yellow
}

# ══════════════════════════════════════════════════
# PHASE 5 : S5 — Named Locations
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S5 — Named Locations" -ForegroundColor Cyan
$locations = Get-MgIdentityConditionalAccessNamedLocation -All
$results["S5_NamedLocations"] = $locations.Count
$locations | Select-Object DisplayName, @{N="Type";E={$_."@odata.type" -replace ".*\.",""}} | Format-Table

# ══════════════════════════════════════════════════
# PHASE 6 : S6 — PIM (Global Admins permanents)
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S6 — Global Admins permanents" -ForegroundColor Cyan
$gaRole = Get-MgDirectoryRole -Filter "displayName eq 'Global Administrator'" -ErrorAction SilentlyContinue
if ($gaRole) {
    $gaMembers = Get-MgDirectoryRoleMember -DirectoryRoleId $gaRole.Id -All
    $results["S6_PermanentGlobalAdmins"] = $gaMembers.Count
    Write-Host "   Global Admins permanents : $($gaMembers.Count)" -ForegroundColor $(if ($gaMembers.Count -gt 2) { "Red" } else { "Green" })
    $gaMembers | ForEach-Object {
        $u = Get-MgUser -UserId $_.Id -ErrorAction SilentlyContinue
        if ($u) { Write-Host "     - $($u.DisplayName) ($($u.UserPrincipalName))" }
    }
}

# ══════════════════════════════════════════════════
# PHASE 7 : S7 — Break-glass
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S7 — Break-glass accounts" -ForegroundColor Cyan
$bgAccounts = Get-MgUser -Filter "startswith(displayName,'BreakGlass') or startswith(displayName,'BG-')" -All
$results["S7_BreakGlassAccounts"] = $bgAccounts.Count
if ($bgAccounts.Count -ge 2) {
    Write-Host "   ✅ $($bgAccounts.Count) compte(s) break-glass trouvé(s)" -ForegroundColor Green
} else {
    Write-Host "   ❌ Seulement $($bgAccounts.Count) break-glass (recommandé : 2)" -ForegroundColor Red
}

# ══════════════════════════════════════════════════
# PHASE 8 : S8 — Workload Identities (secrets expirés)
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S8 — Workload Identities (secrets)" -ForegroundColor Cyan
$apps = Get-MgApplication -All
$expiredSecrets = 0; $expiringSecrets = 0
foreach ($app in $apps) {
    foreach ($secret in $app.PasswordCredentials) {
        if ($secret.EndDateTime -lt (Get-Date)) { $expiredSecrets++ }
        elseif ($secret.EndDateTime -lt (Get-Date).AddDays(30)) { $expiringSecrets++ }
    }
    foreach ($cert in $app.KeyCredentials) {
        if ($cert.EndDateTime -lt (Get-Date)) { $expiredSecrets++ }
        elseif ($cert.EndDateTime -lt (Get-Date).AddDays(30)) { $expiringSecrets++ }
    }
}
$results["S8_TotalApps"] = $apps.Count
$results["S8_ExpiredSecrets"] = $expiredSecrets
$results["S8_ExpiringIn30d"] = $expiringSecrets
Write-Host "   Apps : $($apps.Count) | Secrets expirés : $expiredSecrets | Expirant <30j : $expiringSecrets"

# ══════════════════════════════════════════════════
# PHASE 9 : S9 — Access Reviews
# ══════════════════════════════════════════════════
Write-Host "`n🔍 S9 — Access Reviews" -ForegroundColor Cyan
try {
    $reviews = Get-MgIdentityGovernanceAccessReviewDefinition -All
    $results["S9_AccessReviews"] = $reviews.Count
    Write-Host "   Access Reviews configurées : $($reviews.Count)"
} catch {
    $results["S9_AccessReviews"] = "N/A (P2/Gov requis)"
    Write-Host "   Access Reviews : P2/Governance requis" -ForegroundColor Yellow
}

# Guests sans review
$guests = Get-MgUser -Filter "userType eq 'Guest'" -All
$oldGuests = $guests | Where-Object { $_.CreatedDateTime -and ($_.CreatedDateTime -lt (Get-Date).AddDays(-90)) }
$results["S9_TotalGuests"] = $guests.Count
$results["S9_GuestsOver90d"] = $oldGuests.Count

# ══════════════════════════════════════════════════
# PHASE 10 : Résumé
# ══════════════════════════════════════════════════
Write-Host "`n" + ("=" * 60) -ForegroundColor Cyan
Write-Host "   RÉSUMÉ AUDIT ENTRA ID" -ForegroundColor Cyan
Write-Host ("=" * 60) -ForegroundColor Cyan
$results.GetEnumerator() | ForEach-Object {
    Write-Host "   $($_.Key) : $($_.Value)"
}
Write-Host ("=" * 60) -ForegroundColor Cyan

# Export JSON
$outputPath = ".\EntraID_Audit_$(Get-Date -Format 'yyyyMMdd_HHmmss').json"
$results | ConvertTo-Json -Depth 5 | Out-File -FilePath $outputPath -Encoding UTF8
Write-Host "`n💾 Résultats exportés : $outputPath" -ForegroundColor Green

Disconnect-MgGraph
