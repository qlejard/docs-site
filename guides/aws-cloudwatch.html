Analogie générale

🚗
CloudWatch = tableau de bord de la voiture. Les métriques sont les jauges (vitesse, température, niveau de carburant). Les alarmes sont les voyants rouges qui s'allument quand un seuil est dépassé. Les logs sont la boîte noire de l'avion — tout est enregistré pour analyse post-incident. X-Ray est le scanner médical — il visualise le trajet d'une requête à travers les services. Synthetics est le client mystère — il simule un utilisateur réel pour vérifier que tout fonctionne.

Métriques & Namespaces Free Tier Pay-per-use
Métriques par défaut, custom, CloudWatch Agent, dimensions et résolutions — le socle de l'observabilité
Réf. CIS AWS Foundations Benchmark v7.0.0 §4, AWS Well-Architected — Reliability Pillar
🚨
Scénario concret : Sans métriques custom ni CloudWatch Agent, votre instance EC2 ne remonte que 5 métriques de base (CPU, réseau, disque IOPS). Vous ne voyez ni la mémoire, ni l'espace disque, ni les processus. Un serveur peut remplir son disque à 100% et crash sans que personne ne soit alerté.
Analogie : c'est comme conduire une voiture avec uniquement le compteur de vitesse — pas de jauge de température, pas de niveau de carburant, pas de voyant d'huile. Vous roulez à l'aveugle.

Paramètre / Composant Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût

Detailed Monitoring (EC2) Désactivé Activé (1 min) Activé (1 min) Activé (1 min) Pay-per-use
CloudWatch Agent (mémoire, disque) Non installé Installé (mem + disk) Installé + processus Installé + StatsD/collectd Pay-per-use
Custom Metrics (PutMetricData) Aucune Métriques applicatives clés Métriques business + techniques High-resolution (1s) pour les critiques Pay-per-use
Résolution métriques custom Standard (60s) Standard (60s) Standard (60s) High-res (1s) pour critiques Pay-per-use
Metric Math (calculs dérivés) Non utilisé Formules de base (%, ratio) Anomaly bands, METRICS() Expressions complexes + SEARCH() Free Tier
Namespaces custom Aucun 1 namespace par application Namespace par service + env Convention complète (app/env/team) Free Tier
Métriques EC2 par défaut CPUUtilization, NetworkIn/Out, DiskReadOps/WriteOps, StatusCheckFailed Idem Idem Idem Free Tier
💡
Pourquoi chaque paramètre compte :
  • Detailed Monitoring (1 min) : avec le basic monitoring (5 min), vous avez un point de données toutes les 5 minutes. Un pic CPU de 4 minutes est invisible. Le detailed monitoring à 1 min est nécessaire pour déclencher des alarmes réactives. Source : AWS Well-Architected — Reliability Pillar, REL-06.
  • CloudWatch Agent (mémoire, disque) : EC2 ne remonte pas la mémoire ni l'espace disque par défaut (le hyperviseur ne voit pas à l'intérieur du guest OS). Sans l'agent, un disque à 100% ou un OOM killer ne déclenchent aucune alarme. C'est le trou n°1 en audit d'observabilité.
  • Custom Metrics : les métriques par défaut couvrent l'infra, pas l'applicatif. Sans custom metrics, vous ne savez pas si votre file d'attente grossit, si votre taux d'erreurs 5xx monte, ou si votre cache hit ratio chute.
  • Namespaces custom : sans convention de nommage, les métriques de 10 applications finissent dans un namespace fourre-tout. Impossible de filtrer, de créer des alarmes par application, ou de contrôler les coûts par équipe.
  • High-resolution (1s) : pour les systèmes critiques (trading, temps réel), la résolution standard (60s) masque les micro-pics. Le high-res permet de détecter un spike de latence de 10 secondes invisible en agrégation 60s.
⚠️
Pièges classiques :
  • mem_used_percent et disk_used_percent nécessitent le CloudWatch Agent — ils ne sont jamais disponibles par défaut.
  • Les métriques custom avec dimensions sont facturées par combinaison unique de dimensions. 10 métriques × 5 dimensions = 50 métriques facturées.
  • Les métriques high-resolution ont un coût 10x plus élevé — les réserver aux métriques véritablement critiques.
  • L'agent CloudWatch sur Windows nécessite des Performance Counters spécifiques (pas les mêmes noms que Linux).

1.2 — Configuration

Chemin 1 — Console AWS :

1
Ouvrir EC2 Console > Sélectionner l'instance > Actions > Monitor and troubleshoot > Manage detailed monitoring > Enable
2
Pour l'agent : Systems Manager > Run Command > AWS-ConfigureAWSPackage > AmazonCloudWatchAgent

Chemin 2 — AWS CLI :

Bash — Activer le detailed monitoring + installer l'agent
# Activer le detailed monitoring sur une instance EC2
aws ec2 monitor-instances \
    --instance-ids $INSTANCE_ID \
    --profile "audit-admin" \
    --region "eu-west-1"

# Installer le CloudWatch Agent via SSM
aws ssm send-command \
    --document-name "AWS-ConfigureAWSPackage" \
    --parameters '{"action":["Install"],"name":["AmazonCloudWatchAgent"]}' \
    --targets 'Key=tag:Environment,Values=production' \
    --profile "audit-admin" \
    --region "eu-west-1"

# Publier une métrique custom
aws cloudwatch put-metric-data \
    --namespace "MyApp/Production" \
    --metric-name "QueueDepth" \
    --value 42 \
    --unit "Count" \
    --dimensions "Service=OrderProcessor,Environment=prod" \
    --profile "audit-admin" \
    --region "eu-west-1"

1.4 — État des lieux

2
CloudWatch Logs Free Tier Pay-per-use
Log groups, log streams, rétention, subscription filters, metric filters, log classes
Réf. CIS AWS Foundations Benchmark v7.0.0 §3.1-3.14
🚨
Scénario concret : Par défaut, les log groups CloudWatch n'ont aucune rétention configurée (= « Never expire »). En 6 mois, un environnement de production peut accumuler 500 Go de logs stockés indéfiniment à 0,03 $/Go/mois = ~180 $/an de logs inutiles. De plus, sans subscription filters ni metric filters, les erreurs applicatives critiques passent inaperçues dans le flux de logs.
Analogie : c'est la boîte noire de l'avion. Si personne ne l'écoute après un incident, à quoi sert-elle ? Et si elle enregistre pendant 30 ans sans limite, le stockage explose.

2.1 — Tableau de configuration complet

Paramètre / Composant Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Coût
Rétention des logs Never expire (∞) 30 jours (prod) / 7 jours (dev) 90 jours (prod) / 14 jours (dev) Tiered : 90j standard + archive S3 Pay-per-use
Log class Standard Standard pour prod Infrequent Access pour dev/staging IA pour dev + Standard pour prod Pay-per-use
Metric Filters Aucun Filter sur ERROR / CRITICAL Filters par HTTP status (4xx, 5xx) Filters business + technique Custom metrics pricing
Subscription Filters Aucun Vers Lambda / Firehose si besoin de streaming Vers Lambda / Kinesis / Firehose / OpenSearch OAM pour la vue unifiée + centralisation si copie requise Pay-per-use
Chiffrement (KMS) Chiffrement CloudWatch Logs géré par AWS Chiffrement géré par AWS Clé KMS gérée par le client KMS par log group + policy restreinte Pay-per-use
Export vers S3 Non configuré Export manuel ponctuel Export automatisé (Lambda cron) Subscription filter > Kinesis Firehose > S3 Pay-per-use
Convention de nommage Auto-généré (/aws/lambda/...) /app/env/service /org/app/env/service Convention globale documentée Free Tier
Log format structuré Texte brut JSON structuré JSON + correlation ID JSON + traceId X-Ray Free Tier
💡
Pourquoi chaque paramètre compte :
  • Rétention : « Never expire » semble pratique mais coûte cher. Des logs de plus de 90 jours sont rarement consultés — et quand ils le sont, un export S3 est 10x moins cher. Source : AWS Cost Optimization — CloudWatch Logs best practices.
  • Log class Infrequent Access (IA) : elle réduit le coût d'ingestion, mais retire des capacités comme les subscription filters, les metric filters, le Live Tail et le field indexing. La réserver aux logs consultés à froid.
  • Metric Filters : ils transforment les logs en métriques CloudWatch. Les métriques publiées sont facturées comme des custom metrics au-delà du free tier, avec une vigilance particulière sur la cardinalité des dimensions.
  • Subscription Filters : ils permettent le streaming temps réel des logs vers d'autres destinations (Lambda, Kinesis, OpenSearch). Sans eux, les logs restent passifs.
  • Chiffrement KMS géré par le client : le chiffrement par défaut CloudWatch Logs est géré par AWS. Une clé KMS gérée par le client devient utile quand vous devez contrôler l'usage de la clé, la restreindre à un log group ou répondre à des exigences de conformité spécifiques.
  • Logs JSON structurés : les logs en texte brut nécessitent du parsing regex coûteux dans Logs Insights. Le JSON est parsé nativement — les requêtes sont 5-10x plus rapides et moins chères.
⚠️
Pièges classiques :
  • Un log group en mode Infrequent Access ne supporte pas les subscription filters, les metric filters, ni les API GetLogEvents/FilterLogEvents — conserver la classe Standard pour le monitoring temps réel.
  • La limite de 2 subscription filters par log group est un hard limit — impossible de l'augmenter via support.
  • L'export vers S3 (CreateExportTask) est asynchrone et peut prendre plusieurs heures pour de gros volumes — préférer Kinesis Firehose pour le temps réel.
  • Les logs Lambda sont créés automatiquement dans /aws/lambda/<function-name> avec rétention « Never expire » — penser à la changer immédiatement en Terraform/IaC.
  • Une clé KMS peut aussi être associée après la création d'un log group via associate-kms-key — prévoir les permissions KMS et un délai de propagation pouvant aller jusqu'à ~5 minutes.

2.2 — Configuration

Chemin 1 — AWS CLI :

Bash — Log groups, rétention, metric filters
# Créer un log group avec rétention de 30 jours
aws logs create-log-group \
    --log-group-name "/app/production/api-gateway" \
    --kms-key-id "arn:aws:kms:eu-west-1:123456789012:key/12345678-1234-1234-1234-123456789012" \
    --profile "audit-admin" \
    --region "eu-west-1"

aws logs put-retention-policy \
    --log-group-name "/app/production/api-gateway" \
    --retention-in-days 30 \
    --profile "audit-admin" \
    --region "eu-west-1"

# Créer un metric filter pour compter les erreurs
aws logs put-metric-filter \
    --log-group-name "/app/production/api-gateway" \
    --filter-name "ErrorCount" \
    --filter-pattern '{ $.level = "ERROR" }' \
    --metric-transformations \
        'metricName=ErrorCount,metricNamespace=MyApp/Production,metricValue=1,defaultValue=0' \
    --profile "audit-admin" \
    --region "eu-west-1"

# Créer un subscription filter vers Lambda
aws logs put-subscription-filter \
    --log-group-name "/app/production/api-gateway" \
    --filter-name "CriticalErrors" \
    --filter-pattern '{ $.level = "CRITICAL" }' \
    --destination-arn "arn:aws:lambda:eu-west-1:123456789012:function:log-alert-handler" \
    --profile "audit-admin" \
    --region "eu-west-1"

# Appliquer la rétention à TOUS les log groups sans rétention (audit bulk)
for lg in $(aws logs describe-log-groups \
    --query "logGroups[?!retentionInDays].logGroupName" \
    --output text \
    --profile "audit-admin" \
    --region "eu-west-1"); do
    echo "Setting 30-day retention on: $lg"
    aws logs put-retention-policy \
        --log-group-name "$lg" \
        --retention-in-days 30 \
        --profile "audit-admin" \
        --region "eu-west-1"
done

    Environment = "dev"

# Metric filter — compter les erreurs HTTP 5xx
resource "aws_cloudwatch_log_metric_filter" "error_5xx" {
  name           = "Http5xxErrors"
  pattern        = '{ $.statusCode >= 500 }'
  log_group_name = aws_cloudwatch_log_group.api_logs.name

  metric_transformation {
    name          = "Http5xxErrorCount"
    namespace     = "MyApp/Production"

resource "aws_cloudwatch_log_subscription_filter" "critical_errors" {
  name            = "CriticalErrors"

}

resource "aws_kms_key" "logs" {

    Version = "2012-10-17"

      {
        Sid    = "AllowCloudWatchLogs"
        Effect = "Allow"
        Principal = { Service = "logs.eu-west-1.amazonaws.com" }
        Action = ["kms:Encrypt*", "kms:Decrypt*", "kms:GenerateDataKey*", "kms:DescribeKey"]
        Resource = "*"
      },
      {
        Sid    = "AllowKeyAdmin"
        Effect = "Allow"
        Principal = { AWS = "arn:aws:iam::123456789012:root" }
        Action = "kms:*"
        Resource = "*"
      }
    ]
  })
}

💡
Pourquoi chaque paramètre compte :
  • OAM (Observability Access Manager) : solution native AWS pour partager métriques, logs et traces entre comptes sans copier les données. Le compte monitoring crée un sink, chaque compte source crée un link. Il n'y a pas de surcoût direct pour les métriques, logs et Application Signals ; la première copie de traces est gratuite, puis la tarification CloudWatch s'applique aux traces au-delà.
  • Cross-account dashboards : avec OAM, un dashboard dans le compte central peut afficher des métriques de n'importe quel compte lié. Plus besoin de switcher entre les comptes.
  • EventBridge cross-account : permet de router les événements de sécurité (GuardDuty, Config) de tous les comptes vers un bus central pour une réaction unifiée.
⚠️
Pièges classiques :
  • OAM vs. centralisation : privilégier OAM pour la vue unifiée. Ajouter une centralisation de logs ou des subscription filters seulement si vous avez besoin de copies persistantes ou de pipelines en aval.
  • Les subscription filters sont limités à 2 par log group. Si vous avez déjà un filtre (ex: vers Lambda), il n'en reste plus qu'un pour le cross-account.
  • Le délai de propagation des métriques cross-account peut atteindre quelques minutes — ne pas s'alarmer si le dashboard central a un léger décalage.
  • L'onboarding automatique des comptes source suppose AWS Organizations ; sur des comptes standalone, les liens OAM doivent être créés compte par compte.

#!/usr/bin/env bash
# deploy-observability.sh — Quick Win Observability Stack
# Usage : chmod +x deploy-observability.sh && ./deploy-observability.sh
set -euo pipefail

# ═══════ VARIABLES À ADAPTER ═══════
PROFILE="audit-admin"
REGION="eu-west-1"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text --profile "$PROFILE")
SNS_TOPIC="arn:aws:sns:${REGION}:${ACCOUNT_ID}:prod-alerts-p1"
BILLING_SNS="arn:aws:sns:us-east-1:${ACCOUNT_ID}:billing-alerts"
LOG_RETENTION=90
BUDGET_AMOUNT="600"
BILLING_THRESHOLD=500

echo "🚀 Déploiement observabilité — Compte $ACCOUNT_ID — Région $REGION"

# ═══════ S2 : RETENTION DES LOGS ═══════
echo "\n📋 S2 — Configuration rétention des log groups..."
for lg in $(aws logs describe-log-groups \
    --query "logGroups[?retentionInDays==null].logGroupName" \
    --output text --profile "$PROFILE" --region "$REGION"); do
    echo "  → $lg → ${LOG_RETENTION} jours"
    aws logs put-retention-policy \
        --log-group-name "$lg" \
        --retention-in-days $LOG_RETENTION \
        --profile "$PROFILE" --region "$REGION"
done

# ═══════ S3 : ALARMES CRITIQUES ═══════
echo "\n🚨 S3 — Création des alarmes critiques..."

# EC2 CPU > 90%
aws cloudwatch put-metric-alarm \
    --alarm-name "ec2-cpu-high-all" \
    --namespace "AWS/EC2" \
    --metric-name "CPUUtilization" \
    --statistic Average --period 300 \
    --evaluation-periods 3 --threshold 90 \
    --comparison-operator GreaterThanThreshold \
    --alarm-actions "$SNS_TOPIC" \
    --profile "$PROFILE" --region "$REGION"

# Lambda Errors
aws cloudwatch put-metric-alarm \
    --alarm-name "lambda-errors-all" \
    --namespace "AWS/Lambda" \
    --metric-name "Errors" \
    --statistic Sum --period 300 \
    --evaluation-periods 2 --threshold 5 \
    --comparison-operator GreaterThanThreshold \
    --alarm-actions "$SNS_TOPIC" \
    --profile "$PROFILE" --region "$REGION"

# ALB 5xx > 10
aws cloudwatch put-metric-alarm \
    --alarm-name "alb-5xx-high" \
    --namespace "AWS/ApplicationELB" \
    --metric-name "HTTPCode_ELB_5XX_Count" \
    --statistic Sum --period 300 \
    --evaluation-periods 2 --threshold 10 \
    --comparison-operator GreaterThanThreshold \
    --alarm-actions "$SNS_TOPIC" \
    --profile "$PROFILE" --region "$REGION"

# ═══════ S9 : BILLING ALARM ═══════
echo "\n💰 S9 — Alarme billing..."
aws cloudwatch put-metric-alarm \
    --alarm-name "billing-estimated-charges-${BILLING_THRESHOLD}" \
    --namespace "AWS/Billing" \
    --metric-name "EstimatedCharges" \
    --dimensions "Name=Currency,Value=USD" \
    --statistic Maximum --period 21600 \
    --evaluation-periods 1 --threshold $BILLING_THRESHOLD \
    --comparison-operator GreaterThanThreshold \
    --alarm-actions "$BILLING_SNS" \
    --profile "$PROFILE" --region "us-east-1"

echo "\n✅ Déploiement Quick Win terminé."
echo "   Score estimé : ~65/100 (vs 24/100 avant)."
