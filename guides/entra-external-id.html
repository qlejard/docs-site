Solutions écartées

💡
Pourquoi Microsoft Entra External ID et pas une solution tierce ?
  • Auth0 / Okta (CIAM) → Écartés : coût supplémentaire significatif, intégration M365 native inexistante, double gestion identités. Pour les tenants déjà M365, External ID est intégré nativement.
  • Fédération AD FS manuelle → Écartée : infrastructure lourde (serveurs AD FS + WAP), maintenance certificats, pas de cloud-native. Cross-tenant access + B2B sont la solution moderne.
  • Comptes locaux manuels pour partenaires → Écartés : pas de lifecycle automatisé, risque de comptes orphelins, pas d'audit centralisé.

🎯 Contexte — Pourquoi maîtriser le B2B ?

Microsoft Entra External ID B2B permet d'inviter des utilisateurs externes (partenaires, prestataires, clients) dans votre tenant pour collaborer sur les ressources Microsoft 365 et Azure. Par défaut, tout le monde peut inviter des guests — c'est le paramètre le plus permissif et le plus dangereux à la création d'un tenant.

Les guests sont des objets User de type #EXT# dans votre annuaire. Ils apparaissent dans les groupes, les applications, les sites SharePoint. Sans contrôle, un tenant peut accumuler des centaines de comptes guests orphelins en quelques mois, augmentant la surface d'attaque et le bruit dans les logs.

💡
Analogie — Le badge visiteur : Inviter un guest B2B, c'est comme délivrer un badge visiteur dans un immeuble de bureaux. Sans contrôle à l'accueil (restriction des invitations), n'importe quel employé pourrait fabriquer des badges. Sans date d'expiration (lifecycle management), les badges restent actifs indéfiniment. Et sans registre des entrées (audit logs), vous ne savez pas qui est dans le bâtiment.

⚠️ Piège classique

🚫
Piège n°1 — « Les guests existants ne sont pas affectés » : Quand vous changez « Guest invite settings » de « Everyone » à « Only admins », cela ne bloque que les nouvelles invitations. Les centaines de guests déjà présents restent actifs avec tous leurs accès. Il faut impérativement lancer un nettoyage des guests existants (voir S5 — Guest Lifecycle Management).
🚫
Piège n°2 — « Bloquer les domaines consumer ne suffit pas » : Un attaquant peut créer un tenant Microsoft Entra ID gratuit avec n'importe quel domaine personnalisé. La blocklist domaines consumer (gmail, hotmail) est un minimum mais ne remplace pas une allowlist stricte de domaines partenaires couplée à des cross-tenant access settings (S2).

🎯 Contexte — Pourquoi configurer le cross-tenant access ?

Les Cross-Tenant Access Settings (CTA) contrôlent comment votre tenant interagit avec les autres tenants Microsoft Entra ID / Microsoft Entra pour la collaboration B2B. Ils définissent deux flux : Inbound (qui peut entrer chez vous en tant que guest) et Outbound (quels utilisateurs de chez vous peuvent aller en guest chez les autres).

Depuis 2023, les CTA remplacent l'ancien modèle « tout ou rien » des External Collaboration Settings pour offrir un contrôle par tenant partenaire, par utilisateur, par application, et avec des options de trust (accepter le MFA et le device compliance du tenant source).

💡
Analogie — L'accord diplomatique : Les cross-tenant access settings sont comme des accords diplomatiques entre pays. Le paramètre « Default » est votre politique générale (visa requis par défaut). Les settings par organisation sont des accords bilatéraux (libre circulation avec l'UE, visa strict avec d'autres). Le « Trust » MFA, c'est comme reconnaître le passeport d'un pays partenaire sans redemander une vérification d'identité.

⚠️ Piège classique

🚫
Piège n°1 — « Bloquer inbound default sans ajouter les partenaires » : Si vous passez l'inbound default à « Block all » sans avoir ajouté les partenaires en « Organizational settings », tous les guests existants perdent leur accès immédiatement. Procédure : (1) inventorier les tenants sources des guests actuels, (2) ajouter chaque tenant en org-specific, (3) ensuite bloquer le default.
🚫
Piège n°2 — « Trust MFA sans vérifier le partenaire » : Activer « Trust MFA » signifie que vous acceptez le MFA effectué dans le tenant source. Si ce tenant a un MFA faible (SMS seul, pas de number matching), vos ressources sont exposées. Ne trustez le MFA que pour les tenants partenaires dont vous connaissez la politique MFA.

🔧 Application — CTA Default + Org-Specific (PowerShell)

PowerShell — Microsoft.Graph (Cross-Tenant Access)
# Connexion
Connect-MgGraph -Scopes "Policy.ReadWrite.CrossTenantAccess"

# ── Configurer le DEFAULT inbound : Block all ──
$defaultPolicy = @{
    b2bCollaborationInbound = @{
        usersAndGroups = @{
            accessType = "blocked"
            targets    = @(@{ target = "AllUsers"; targetType = "user" })
        }
        applications = @{
            accessType = "blocked"
            targets    = @(@{ target = "AllApplications"; targetType = "application" })
        }
    }
    inboundTrust = @{
        isMfaAccepted              = $true
        isCompliantDeviceAccepted   = $false
        isHybridAzureADJoinedDeviceAccepted = $false
    }
}

Invoke-MgGraphRequest -Method PATCH `
    -Uri "https://graph.microsoft.com/v1.0/policies/crossTenantAccessPolicy/default" `
    -Body ($defaultPolicy | ConvertTo-Json -Depth 10)

# ── Ajouter un partenaire (org-specific) ──
$partnerTenantId = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"  # Tenant ID Contoso

$orgSettings = @{
    tenantId = $partnerTenantId
    b2bCollaborationInbound = @{
        usersAndGroups = @{
            accessType = "allowed"
            targets    = @(@{ target = "AllUsers"; targetType = "user" })
        }
        applications = @{
            accessType = "allowed"
            targets    = @(
                @{ target = "Office365"; targetType = "application" }
            )
        }
    }
    inboundTrust = @{
        isMfaAccepted            = $true
        isCompliantDeviceAccepted = $true
    }
}

Invoke-MgGraphRequest -Method POST `
    -Uri "https://graph.microsoft.com/v1.0/policies/crossTenantAccessPolicy/partners" `
    -Body ($orgSettings | ConvertTo-Json -Depth 10)

🔧 Application — Inventorier les tenants sources des guests existants

PowerShell — Pré-requis avant de bloquer le default
# Lister les tenants d'origine de tous les guests actuels
Connect-MgGraph -Scopes "User.Read.All"

$guests = Get-MgUser -Filter "userType eq 'Guest'" -All `
    -Property DisplayName, Mail, UserPrincipalName, CreationType, ExternalUserState `
    -ConsistencyLevel eventual

# Extraire le domaine source de chaque guest
$tenantDomains = $guests | ForEach-Object {
    if ($_.Mail) { ($_.Mail -split "@")[1] }
    elseif ($_.UserPrincipalName -match "#EXT#") {
        (($_.UserPrincipalName -split "_")[-1] -split "#")[0]
    }
} | Sort-Object -Unique

Write-Host "Domaines sources uniques :"
$tenantDomains | ForEach-Object { Write-Host "  - $_" }
Write-Host "Total : $($tenantDomains.Count) domaines"

PowerShell — Script de nettoyage guests inactifs
# ── Script de nettoyage des guests dormants ──
# Phase 1 : Désactiver (grace period) | Phase 2 : Supprimer après 30j
Connect-MgGraph -Scopes "User.ReadWrite.All","AuditLog.Read.All"

$staleThreshold  = (Get-Date).AddDays(-90)   # Inactif depuis 90 jours
$deleteThreshold = (Get-Date).AddDays(-120)  # Désactivé depuis 30j = 90+30

# Récupérer tous les guests avec leur dernière connexion
$guests = Get-MgUser -Filter "userType eq 'Guest'" -All `
    -Property Id, DisplayName, Mail, AccountEnabled, `
              SignInActivity, ExternalUserState, CreatedDateTime `
    -ConsistencyLevel eventual

Write-Host "Total guests : $($guests.Count)"

# ── Phase 1 : Désactiver les guests dormants (>90j sans sign-in) ──
$staleGuests = $guests | Where-Object {
    $_.AccountEnabled -eq $true -and (
        $_.SignInActivity.LastSignInDateTime -lt $staleThreshold -or
        ($_.SignInActivity.LastSignInDateTime -eq $null -and
         $_.CreatedDateTime -lt $staleThreshold)
    )
}

Write-Host "`n=== PHASE 1 : Désactivation ($($staleGuests.Count) guests dormants >90j) ==="
$staleGuests | ForEach-Object {
    Write-Host "  Désactivation : $($_.DisplayName) ($($_.Mail))"
    # Décommenter pour exécuter :
    # Update-MgUser -UserId $_.Id -AccountEnabled:$false
}

# ── Phase 2 : Supprimer les guests désactivés depuis >30j ──
$disabledGuests = $guests | Where-Object {
    $_.AccountEnabled -eq $false -and (
        $_.SignInActivity.LastSignInDateTime -lt $deleteThreshold -or
        ($_.SignInActivity.LastSignInDateTime -eq $null -and
         $_.CreatedDateTime -lt $deleteThreshold)
    )
}

Write-Host "`n=== PHASE 2 : Suppression ($($disabledGuests.Count) guests désactivés >30j) ==="
$disabledGuests | ForEach-Object {
    Write-Host "  Suppression : $($_.DisplayName) ($($_.Mail))"
    # Décommenter pour exécuter :
    # Remove-MgUser -UserId $_.Id
}

# ── Nettoyer les invitations pendantes >30j ──
$pendingThreshold = (Get-Date).AddDays(-30)
$pendingGuests = $guests | Where-Object {
    $_.ExternalUserState -eq "PendingAcceptance" -and

Migration & Modernization B2C
Migration B2C classic → External ID for customers, AD FS → B2C, coéxistence, pricing — 🔵 Criticité standard

🎯 Contexte — Migrer vers External ID for Customers

Microsoft a annoncé External ID for Customers comme successeur d'Azure AD B2C. Les tenants B2C classiques restent supportés, mais les nouveaux projets doivent utiliser External ID for Customers (CIAM). Pour les organisations utilisant AD FS pour l'authentification partenaire, la migration vers B2B/B2C natif est également recommandée.

💡
Analogie — La rénovation de l'hôtel : L'hôtel (B2C classique) fonctionne bien, mais une nouvelle chaîne (External ID for Customers) offre des équipements modernes : réservation en ligne simplifiée, branding native, intégration fluide avec le siège (tenant workforce). Migrer = transférer les clients vers le nouvel hôtel sans perdre leur historique.

🔄 Comparaison — B2C Classic vs External ID for Customers

⚠️ Piège classique

PowerShell — Inventaire B2C pour décision migration
# Inventaire du tenant B2C pour planifier la migration
Connect-MgGraph -TenantId "monentrepriseb2c.onmicrosoft.com" `
    -Scopes "User.Read.All","IdentityUserFlow.Read.All","IdentityProvider.Read.All","Application.Read.All"

Write-Host "=== INVENTAIRE B2C POUR MIGRATION ==="

# 1. Nombre d'utilisateurs B2C
$users = Get-MgUser -All -CountVariable userCount -ConsistencyLevel eventual
Write-Host "`n📊 Utilisateurs : $userCount"

# 2. User flows
$flows = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/identity/b2cUserFlows"
Write-Host "`n📋 User Flows : $($flows.value.Count)"
$flows.value | ForEach-Object { Write-Host "  - $($_.id)" }

# 3. Custom policies (IEF)
try {
    $policies = Invoke-MgGraphRequest -Method GET `
        -Uri "https://graph.microsoft.com/beta/trustFramework/policies"
    Write-Host "`n⚙️ Custom Policies IEF : $($policies.value.Count)"
    if ($policies.value.Count -gt 0) {
        Write-Host "  ⚠️ Custom policies détectées → migration vers External ID for Customers BLOQUÉE" -ForegroundColor Red
        $policies.value | ForEach-Object { Write-Host "  - $($_.id)" }
    } else {
        Write-Host "  ✅ Aucune custom policy → migration possible" -ForegroundColor Green
    }
} catch {
    Write-Host "  ℹ️ Impossible de lire les IEF policies"
}

# 4. Identity Providers
$idps = Get-MgIdentityProvider -All
Write-Host "`n🔑 Identity Providers : $($idps.Count)"
$idps | ForEach-Object { Write-Host "  - $($_.DisplayName) ($($_.IdentityProviderType))" }

# 5. Applications enregistrées
$apps = Get-MgApplication -All
Write-Host "`n📱 Applications : $($apps.Count)"
$apps | ForEach-Object { Write-Host "  - $($_.DisplayName)" }

# Decision tree
Write-Host "`n=== RECOMMANDATION ==="
if ($policies.value.Count -gt 0) {
    Write-Host "  🔴 RESTER sur B2C classique (custom policies non supportées dans External ID)"
} else {
    Write-Host "  🟢 MIGRATION POSSIBLE vers External ID for Customers"
    Write-Host "  → Planifier : export users, recréer user flows, migrer app registrations"
}

Script All-in-One — Hardening External ID
Script exécutable en une seule fois pour appliquer toutes les best practices B2B & External ID
⚠️
Prérequis : Licence Entra ID P2 pour les fonctions Identity Governance. Exécuter en mode -WhatIf d'abord. Le script ne modifie rien sans confirmation explicite.
PowerShell — Script All-in-One Audit External ID
#Requires -Modules Microsoft.Graph.Authentication, Microsoft.Graph.Identity.SignIns, Microsoft.Graph.Users
# ═══════════════════════════════════════════════════════════════════
# Script All-in-One — Audit External ID (B2B & identités externes)
# Version  : 1.1 — Mai 2026
# Prérequis: Microsoft.Graph PowerShell SDK v2+
# Scopes   : User.Read.All, Policy.Read.All, CrossTenantAccess
#            IdentityProvider.Read.All, AuditLog.Read.All
# ═══════════════════════════════════════════════════════════════════

param(
    [string]$OutputFolder = ".\ExternalID-Audit-$(Get-Date -Format 'yyyyMMdd')",
    [switch]$IncludeB2C
)

# ─── Connexion ───
$scopes = @(
    "User.Read.All",
    "Policy.Read.All",
    "Policy.ReadWrite.CrossTenantAccess",
    "IdentityProvider.Read.All",
    "AuditLog.Read.All",
    "Directory.Read.All"
)
Connect-MgGraph -Scopes $scopes -NoWelcome
New-Item -Path $OutputFolder -ItemType Directory -Force | Out-Null

Write-Host "═══ AUDIT EXTERNAL ID — $(Get-Date -Format 'dd/MM/yyyy HH:mm') ═══" -ForegroundColor Cyan

# ─── S1 : Inventaire Guests B2B ───
Write-Host "`n[S1] Inventaire des guests B2B..."
$guests = Get-MgUser -Filter "userType eq 'Guest'" -All `
    -Property DisplayName,Mail,UserPrincipalName,CreatedDateTime,SignInActivity,ExternalUserState
Write-Host "  📊 Guests totaux : $($guests.Count)"
$guests | Select-Object DisplayName,Mail,ExternalUserState,CreatedDateTime | `
    Export-Csv "$OutputFolder\S1-guests-inventory.csv" -NoTypeInformation

# ─── S2 : Cross-Tenant Access ───
Write-Host "`n[S2] Cross-Tenant Access Policies..."
$ctaPolicies = Get-MgPolicyCrossTenantAccessPolicyPartner -All
Write-Host "  🔗 Partenaires configurés : $($ctaPolicies.Count)"
$ctaPolicies | ConvertTo-Json -Depth 5 | `
    Set-Content "$OutputFolder\S2-cross-tenant-access.json"

# ─── S3 : Identity Providers ───
Write-Host "`n[S3] Identity Providers..."
$idps = Get-MgIdentityProvider -All
Write-Host "  🔑 IdPs configurés : $($idps.Count)"
$idps | Select-Object DisplayName,IdentityProviderType | `
    Export-Csv "$OutputFolder\S3-identity-providers.csv" -NoTypeInformation

# ─── S4 : CA Policies ciblant les guests ───
Write-Host "`n[S4] Conditional Access Policies (guests)..."
$allCA = Get-MgIdentityConditionalAccessPolicy -All
$guestCA = $allCA | Where-Object {
    $_.Conditions.Users.IncludeGuestsOrExternalUsers -ne $null -or
    $_.Conditions.Users.IncludeUsers -contains "GuestsOrExternalUsers"
}
Write-Host "  🛡 CA policies guests : $($guestCA.Count) / $($allCA.Count)"
$guestCA | Select-Object DisplayName,State | `
    Export-Csv "$OutputFolder\S4-ca-guest-policies.csv" -NoTypeInformation

# ─── S5 : Guests dormants ───
Write-Host "`n[S5] Guests dormants (> 90 jours)..."
$threshold = (Get-Date).AddDays(-90)
$dormant = $guests | Where-Object {
    $_.SignInActivity.LastSignInDateTime -lt $threshold -or
    $_.SignInActivity.LastSignInDateTime -eq $null
}
Write-Host "  ⚠️ Guests dormants : $($dormant.Count)"
$dormant | Select-Object DisplayName,Mail,@{N='LastSignIn';E={$_.SignInActivity.LastSignInDateTime}} | `
    Export-Csv "$OutputFolder\S5-dormant-guests.csv" -NoTypeInformation

# ─── S9 : Guest Access Restriction Level ───
Write-Host "`n[S9] Authorization Policy (guest restrictions)..."
$authPol = Get-MgPolicyAuthorizationPolicy
Write-Host "  🔒 GuestUserRoleId : $($authPol.GuestUserRoleId)"
Write-Host "  📧 AllowInvitesFrom : $($authPol.AllowInvitesFrom)"

# ─── Rapport final ───
Write-Host "`n═══ AUDIT TERMINÉ ═══" -ForegroundColor Green
Write-Host "📁 Exports dans : $OutputFolder"
Write-Host "  - S1-guests-inventory.csv"
Write-Host "  - S2-cross-tenant-access.json"
Write-Host "  - S3-identity-providers.csv"
Write-Host "  - S4-ca-guest-policies.csv"
Write-Host "  - S5-dormant-guests.csv"
