Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel EKS Security
📋
Ce document est un référentiel d'audit et de hardening Amazon EKS couvrant la sécurité complète d'un cluster Kubernetes managé sur AWS. Il cible les 3 audiences : décideur (scénarios concrets, analogies), admin/implémenteur (commandes CLI + Terraform), architecte (sources CIS, justifications).

Analogie générale : Amazon EKS est un immeuble de bureaux partagé — l'infrastructure commune (ascenseurs, entrée, réseau électrique) est gérée par AWS, mais chaque étage (namespace) doit être isolé, sécurisé et surveillé par le locataire.

Création du cluster, endpoints, versions Kubernetes, encryption etcd, add-ons
Réf. CIS Amazon EKS Benchmark v1.8.0 §3.x (Worker Nodes), §5.3 (KMS)
🚨
Scénario concret : Par défaut, le API server d'un cluster EKS est accessible depuis Internet (endpoint public activé). Un attaquant qui découvre votre endpoint peut tenter des attaques brute-force sur l'API Kubernetes, exploiter des vulnérabilités 0-day du kube-apiserver, ou utiliser des credentials volées pour prendre le contrôle du cluster.

Analogie : L'endpoint public EKS, c'est comme laisser la porte d'entrée de l'immeuble de bureaux grande ouverte sur la rue — n'importe qui peut entrer dans le hall et tenter d'accéder aux étages. Le passer en privé, c'est mettre un sas à badge à l'entrée.

De même, sans encryption KMS d'etcd, les Secrets Kubernetes (mots de passe, tokens, clés API) sont stockés en clair dans etcd. Toute personne ayant accès au stockage sous-jacent peut les lire.

Pourquoi chaque paramètre compte :
  • endpointPublicAccess = false : l'API server exposé sur Internet a historiquement été un vecteur d'attaque majeur sur Kubernetes (cf. CVE-2018-1002105 et CVE-2020-8559 sur kube-apiserver, exploitables à distance). Le passer en privé réduit drastiquement la surface d'attaque externe. Source : CIS Amazon EKS Benchmark v1.8.0 §5.4.2 (Private Endpoint Enabled, Public Access Disabled)
  • encryptionConfig (KMS) : sans encryption enveloppe, les Secrets K8s (tokens de service account, mots de passe applicatifs, clés TLS) sont en base64 dans etcd — PAS chiffrés. Tout accès au stockage etcd (backup, snapshot) expose les secrets en clair. Source : CIS Amazon EKS Benchmark v1.8.0 §5.3.1 (Secrets encrypted with CMK)
  • Kubernetes version latest : Amazon EKS supporte chaque version mineure pendant 14 mois. Après, la version entre en Extended Support (coût supplémentaire). Les versions anciennes ne reçoivent pas les patchs CVE critiques. Source : AWS EKS Version Lifecycle
  • Add-ons managés : sans EKS managed add-ons, les composants CoreDNS, kube-proxy et VPC CNI ne sont pas mis à jour automatiquement → vulnérabilités connues persistent. Source : AWS FSBP
  • Cluster logging (5 types) : sans logs control plane, aucune traçabilité des appels API Kubernetes (création de pods, modifications RBAC, tentatives d'authentification). Obligation RGPD Art. 5 (traçabilité) et ISO 27001 A.12.4. Source : CIS Amazon EKS Benchmark v1.8.0 §2.1 (Logging)
  • STS endpoint régional : l'endpoint global sts.amazonaws.com est déconseillé par AWS (non déprécié formellement, mais non recommandé). L'utilisation de sts.<region>.amazonaws.com réduit la latence, améliore la résilience et limite le rayon d'impact en cas d'incident régional. Source : AWS Best Practices
⚠️
Pièges classiques :
  • Encryption etcd non rétroactive : activer le KMS encryption après la création du cluster ne chiffre pas les secrets existants. Il faut recréer ou re-pousher chaque secret pour qu'il soit chiffré.
  • Endpoint privé = besoin d'un accès VPC : couper l'endpoint public sans avoir configuré un VPN, un bastion, ou AWS CloudShell dans le VPC vous verrouille hors du cluster.
  • eksctl crée des clusters publics par défaut : eksctl create cluster sans flags crée un cluster public, sans encryption, sans logging. Toujours utiliser un fichier ClusterConfig YAML.
  • Extended Support coûte cher : après les 14 mois de Standard Support, EKS facture +0,60 $/h/cluster en sus du tarif standard (soit ~0,70 $/h au total, x7) en Extended Support. Planifier les upgrades.

S1.2 — Configuration

Chemin 1 : AWS CLI

Bash — Création cluster EKS sécurisé (AWS CLI)
# Variables — ADAPTER avant exécution
CLUSTER_NAME="prod-cluster"
REGION="eu-west-1"
K8S_VERSION="1.33"
KMS_KEY_ARN="arn:aws:kms:eu-west-1:123456789012:key/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
SUBNET_IDS="subnet-private-1,subnet-private-2,subnet-private-3"
SG_ID="sg-xxxxxxxxxxxxxxxxx"
ROLE_ARN="arn:aws:iam::123456789012:role/EKSClusterRole"
PROFILE="admin-prod"

# Créer le cluster avec encryption et endpoint privé
aws eks create-cluster \
    --name "$CLUSTER_NAME" \
    --region "$REGION" \
    --profile "$PROFILE" \
    --kubernetes-version "$K8S_VERSION" \
    --role-arn "$ROLE_ARN" \
    --resources-vpc-config subnetIds="$SUBNET_IDS",securityGroupIds="$SG_ID",endpointPublicAccess=false,endpointPrivateAccess=true \
    --encryption-config '[{"resources":["secrets"],"provider":{"keyArn":"'"$KMS_KEY_ARN"'"}}]' \
    --logging '{"clusterLogging":[{"types":["api","audit","authenticator","controllerManager","scheduler"],"enabled":true}]}' \
    --output json

# Activer les add-ons managés
for addon in vpc-cni coredns kube-proxy; do
    aws eks create-addon \
        --cluster-name "$CLUSTER_NAME" \
        --addon-name "$addon" \
        --resolve-conflicts OVERWRITE \
        --region "$REGION" \
        --profile "$PROFILE"
done

S1.4 — État des lieux

S2
Authentication & IRSA IAM gratuit
IAM Roles for Service Accounts, EKS Pod Identity, OIDC, access entries, least privilege
Réf. CIS Amazon EKS Benchmark v1.8.0 §4.x (RBAC & Policies), AWS IRSA guidance
🚨
Scénario concret : Par défaut, les pods Kubernetes tournent avec le service account default sans aucun rôle IAM. Si un pod a besoin d'accéder à S3 ou DynamoDB, les développeurs stockent des access keys longue durée en variables d'environnement — clés qui ne tournent jamais, sont partagées entre pods, et se retrouvent souvent dans les dépôts Git.

Analogie : IRSA/Pod Identity, c'est comme donner un badge nominatif temporaire à chaque employé (pod) de l'immeuble, au lieu de leur partager tous la même clé passe-partout. Si un badge est compromis, seul cet employé est affecté, et le badge expire automatiquement.

Sans IRSA, un pod compromis avec des access keys peut pivoter vers n'importe quelle ressource AWS associée à ces clés. Avec IRSA, les credentials sont temporaires (STS), scopées au service account, et révoquées automatiquement à l'expiration.

Pourquoi chaque paramètre compte :
  • OIDC provider : prérequis obligatoire pour IRSA. Sans lui, impossible d'associer un rôle IAM à un service account K8s. Pas de fédération OIDC = pas de credentials temporaires. Source : AWS IAM Roles for Service Accounts (IRSA) — prérequis non-CIS mais recommandé AWS Best Practices
  • Access entries (mode API) : le aws-auth ConfigMap est fragile — une erreur YAML verrouille tout le cluster. Les access entries via l'API EKS sont auditables, versionnables, et intégrées IAM. Source : AWS Best Practices
  • automountServiceAccountToken = false : par défaut, chaque pod monte un token K8s. Un pod compromis peut utiliser ce token pour interroger l'API server. Désactiver le montage automatique sur le SA default limite l'exposition. Source : CIS Amazon EKS Benchmark v1.8.0 §4.1.6 (Service Account Tokens only mounted where necessary)
  • Node IAM Role minimal : un node role trop large (ex : AdministratorAccess) donne à tous les pods du node un accès admin AWS via l'IMDS (Instance Metadata Service). Chaque workload doit avoir son propre rôle via IRSA. Source : AWS FSBP
⚠️
Pièges classiques :
  • IMDS hop limit : par défaut, les pods sur EC2 peuvent accéder à l'IMDS du node et récupérer les credentials du node role. Fixer httpPutResponseHopLimit=1 sur le launch template ou utiliser le VPC CNI avec AWS_EC2_METADATA_DISABLED=true.
  • aws-auth + access entries : en mode API_AND_CONFIG_MAP, les deux sont évalués. En mode API, seules les access entries comptent. Attention aux accès orphelins dans aws-auth après migration.
  • Trust policy OIDC trop large : ne jamais utiliser "*" dans le champ sub de la trust policy — cela permet à tous les service accounts du cluster d'assumer le rôle. Toujours spécifier system:serviceaccount:<ns>:<sa>.
  • Access key oubliées : après migration vers IRSA, les anciennes access keys dans les Secrets/ConfigMaps continuent de fonctionner. Les supprimer et révoquer dans IAM.

S2.3 — Configuration IRSA

Chemin 1 : AWS CLI

Bash — Configuration IRSA complète (AWS CLI + kubectl)
# Variables
CLUSTER_NAME="prod-cluster"
REGION="eu-west-1"
PROFILE="admin-prod"
NAMESPACE="app-backend"
SA_NAME="backend-s3-reader"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text --profile "$PROFILE")

# 1. Associer l'OIDC provider (une seule fois par cluster)
eksctl utils associate-iam-oidc-provider \
    --cluster "$CLUSTER_NAME" \
    --region "$REGION" \
    --approve

# Récupérer l'OIDC issuer
OIDC_ISSUER=$(aws eks describe-cluster \
    --name "$CLUSTER_NAME" \
    --region "$REGION" \
    --profile "$PROFILE" \
    --query 'cluster.identity.oidc.issuer' \
    --output text | sed 's|https://||')

# 2. Créer le rôle IAM avec trust policy scoped
cat <<EOF > trust-policy.json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"Federated": "arn:aws:iam::${ACCOUNT_ID}:oidc-provider/${OIDC_ISSUER}"},
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "${OIDC_ISSUER}:aud": "sts.${REGION}.amazonaws.com",
        "${OIDC_ISSUER}:sub": "system:serviceaccount:${NAMESPACE}:${SA_NAME}"
      }
    }
  }]
}
EOF

aws iam create-role \
    --role-name "eks-${CLUSTER_NAME}-${SA_NAME}" \
    --assume-role-policy-document file://trust-policy.json \
    --profile "$PROFILE"

# 3. Attacher une policy minimale (exemple : lecture S3)
aws iam attach-role-policy \
    --role-name "eks-${CLUSTER_NAME}-${SA_NAME}" \
    --policy-arn "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess" \
    --profile "$PROFILE"

# 4. Créer le service account K8s annoté
kubectl create namespace "$NAMESPACE" --dry-run=client -o yaml | kubectl apply -f -
kubectl create serviceaccount "$SA_NAME" -n "$NAMESPACE" --dry-run=client -o yaml | kubectl apply -f -
kubectl annotate serviceaccount "$SA_NAME" -n "$NAMESPACE" \
    eks.amazonaws.com/role-arn=arn:aws:iam::${ACCOUNT_ID}:role/eks-${CLUSTER_NAME}-${SA_NAME} \
    --overwrite

# 5. Désactiver automount sur le SA default
kubectl patch serviceaccount default -n "$NAMESPACE" \
    -p '{"automountServiceAccountToken": false}'

S3
Network Security VPC CNI gratuit
VPC CNI, Security Groups for Pods, Network Policies (Calico/Cilium), private cluster, custom networking
Réf. CIS Amazon EKS Benchmark v1.8.0 §5.x (Managed Services — KMS, Cluster Networking, Auth)
🚨
Scénario concret : Par défaut, Kubernetes est un réseau plat : tous les pods peuvent communiquer avec tous les autres pods sans restriction. Un pod compromis dans le namespace frontend peut scanner et atteindre directement la base de données dans le namespace database, exfiltrer des données, ou pivoter latéralement vers d'autres workloads.

Analogie : Sans Network Policies, c'est un immeuble sans cloisons ni portes — tous les étages (namespaces) communiquent librement, les visiteurs (pods publics) accèdent directement au coffre-fort (database). Les Network Policies sont les murs coupe-feu et portes verrouillées entre zones de l'immeuble.

EKS + VPC CNI : chaque pod reçoit une adresse IP VPC réelle (ENI secondaire). Avantage : intégration native Security Groups, ALB target-type=ip. Inconvénient : consommation rapide d'IPs privées.

Pourquoi chaque paramètre compte :
  • Network Policies default-deny : sans elles, tout trafic intra-cluster est permis. Un pod compromis peut scanner 100% du cluster en quelques secondes. Le default-deny force la déclaration explicite de chaque flux autorisé. Source : CIS Amazon EKS Benchmark v1.8.0 §4.3.2 (Namespaces have Network Policies defined)
  • VPC CNI NetworkPolicy support : depuis le VPC CNI v1.14, Amazon supporte nativement les Network Policies Kubernetes sans add-on tiers (Calico). Il faut activer le flag ENABLE_NETWORK_POLICY=true dans le ConfigMap amazon-vpc-cni. Source : AWS VPC CNI docs
  • Security Groups for Pods : permet d'appliquer des SG AWS directement aux pods (pas seulement aux nodes). Essentiel pour l'accès direct à RDS, ElastiCache, etc. sans exposer tout le node. Source : AWS Best Practices
  • Custom networking : évite l'exhaustion d'IPs dans les subnets de nodes. Les pods utilisent des subnets dédiés avec des plages CIDR séparées. Source : AWS EKS Networking Best Practices
⚠️
Pièges classiques :
  • Network Policy sans controller = no-op : créer des Network Policies sans avoir activé ENABLE_NETWORK_POLICY=true dans le VPC CNI (ou installé Calico/Cilium) est silencieusement ignoré. Zéro protection.
  • Oublier le DNS dans egress deny : un default-deny egress bloque aussi le DNS (port 53). Tous les pods perdent la résolution de noms. Toujours créer une policy egress vers kube-system/kube-dns:53.
  • SG for Pods désactive Calico : sur les pods avec Security Groups, les Network Policies Calico ne s'appliquent pas (le trafic bypasse iptables). Choisir l'un ou l'autre par workload.
  • IP exhaustion : avec le VPC CNI, chaque pod consomme une IP VPC. Sans prefix delegation ou custom networking, un cluster de 100 pods peut consommer 100+ IPs dans le subnet. Prévoir des subnets /20 ou plus grands.

S4
Pod Security PSA gratuit Gatekeeper/Kyverno
Pod Security Standards, PSA, OPA Gatekeeper, Kyverno, security contexts, image scanning
Réf. CIS Amazon EKS Benchmark v1.8.0 §4.2 (Pod Security Standards)
🚨
Scénario concret : Un développeur déploie un pod avec privileged: true et hostNetwork: true pour « débugger un problème réseau ». Ce pod a un accès complet au node : système de fichiers host, interfaces réseau, tous les namespaces Linux. Si ce pod est compromis, l'attaquant obtient un accès root au node et peut pivoter vers tous les autres pods via l'IMDS.

Analogie : Pod Security, c'est le détecteur de métaux à l'entrée de l'aéroport. Chaque passager (pod) est inspecté avant l'embarquement (scheduling). Un pod qui tente de monter des armes (hostPID, hostNetwork, privileged) est rejeté à la porte, pas une fois en vol.

Pod Security Standards (PSS) définit 3 profils : Privileged (tout permis), Baseline (bloque les attaques triviales), Restricted (hardening complet). Pod Security Admission (PSA) est le contrôleur d'admission natif K8s qui enforce ces profils.

S4.1 — Pod Security Standards : 3 profils

Profil PSS Ce qui est bloqué Cas d'usage Mode recommandé
Privileged Rien — tout est permis kube-system, monitoring agents audit (uniquement sur namespaces system)
Baseline hostNetwork, hostPID, hostIPC, privileged, hostPath dangereux Minimum viable pour workloads applicatifs enforce (tous les namespaces non-system)
Restricted Baseline + runAsNonRoot obligatoire, drop ALL capabilities, readOnlyRootFilesystem, seccompProfile runtime/default Production hardened enforce (namespaces critiques : PCI, SOC2)

Pourquoi chaque paramètre compte :
  • PSA Baseline enforce : bloque les vecteurs d'évasion de conteneur les plus courants (privileged, hostPID, hostNetwork). Sans PSA, n'importe quel déploiement peut créer un conteneur privilégié. Source : CIS EKS §4.2
  • runAsNonRoot : par défaut, les conteneurs tournent en root (UID 0). Un processus root peut exploiter des vulnérabilités kernel pour s'échapper. Forcer runAsNonRoot + UID > 10000. Source : CIS EKS §4.2.6
  • readOnlyRootFilesystem : empêche les attaquants d'écrire des backdoors, webshells, ou cron malveillants dans le filesystem du conteneur. Source : CIS EKS §4.2.7
  • drop: [ALL] capabilities : les capabilities Linux par défaut incluent CAP_NET_RAW (ARP spoofing), CAP_SYS_CHROOT. Les supprimer toutes et n'ajouter que le strict nécessaire. Source : PSS Restricted
⚠️
Pièges classiques :
  • PSA enforce sur kube-system : ne jamais appliquer restricted enforce sur kube-system ! Les composants system (kube-proxy, aws-node, coredns) nécessitent des capabilities élevées. Utiliser audit ou warn uniquement.
  • Images root-only : certaines images officielles (nginx, mysql) tournent par défaut en root. Utiliser les variantes non-root ou ajouter runAsUser/runAsGroup explicites.
  • Kyverno/Gatekeeper en audit-only indéfiniment : démarrer en audit est correct pour la migration, mais planifier la bascule en enforce avec une date cible. L'audit seul ne bloque rien.
  • emptyDir oublié : avec readOnlyRootFilesystem=true, les applications qui écrivent dans /tmp crashent. Toujours monter un emptyDir sur les chemins d'écriture.

Kubernetes Secrets, AWS Secrets Manager CSI, External Secrets Operator, KMS encryption, rotation
Réf. CIS Amazon EKS Benchmark v1.8.0 §5.3 (KMS & Secrets)
🚨
Scénario concret : Les Secrets Kubernetes sont stockés en base64 dans etcd — ce n'est pas du chiffrement. Quiconque ayant accès en lecture à l'API server ou à un backup etcd voit tous les secrets en clair. Par défaut sur EKS, les secrets etcd sont chiffrés par AWS avec une clé managée, mais les RBAC sont souvent trop larges : un développeur avec get secrets sur le cluster voit les credentials de production.

Risques : secrets en clair dans les manifests Git, pas de rotation, RBAC trop large sur les secrets, pas d'audit trail sur l'accès aux secrets.

Pourquoi chaque paramètre compte :
  • KMS CMK pour etcd : les secrets etcd sont chiffrés par défaut avec une clé AWS-managed. Avec une CMK, vous contrôlez les policies d'accès, la rotation, et l'audit via CloudTrail. Source : CIS EKS §3.1
  • Secrets Manager + CSI : le CSI Secrets Store driver monte les secrets AWS directement dans les pods comme fichiers. Avantage : le secret n'est jamais stocké dans etcd K8s, la rotation est transparente. Source : AWS Best Practices
  • Rotation automatique : un secret qui ne tourne pas est un secret qui s'accumule dans les backups, les logs, les screenshots. La rotation réduit la fenêtre d'exploitation d'un secret compromis. Source : NIST SP 800-53
⚠️
Pièges classiques :
  • CSI driver ≠ sync K8s Secret : par défaut, le CSI driver monte le secret comme fichier dans le pod mais ne crée pas de K8s Secret. Si votre app lit $DB_PASSWORD en variable d'env, activer syncSecret dans le SecretProviderClass.
  • Pod restart nécessaire après rotation : le CSI driver ne notifie pas le pod quand le secret change. Configurer rotationPollInterval (ex : 120s) et vérifier que l'app relit le fichier.
  • RBAC wildcard sur secrets : un ClusterRole avec resources: ["secrets"], verbs: ["*"] donne accès à tous les secrets du cluster (y compris les tokens SA). Toujours limiter par namespace et par resourceNames.
  • SealedSecrets + namespace différent : un SealedSecret est scopé au namespace cible. Déplacer un SealedSecret dans un autre namespace ne fonctionne pas sans re-scellement.

S8

ECR scanning (basic + enhanced), image signing (Cosign), admission controllers, lifecycle policies, distroless
Réf. AWS Well-Architected — Security Pillar
⚠️
Scénario concret : Un développeur pousse une image node:latest en production. Cette image contient 247 CVE dont 12 critiques (Log4Shell, OpenSSL heap overflow…). Sans scanning ECR ni admission controller, l'image est déployée sans aucun contrôle. Pire : le tag latest est mutable — il peut pointer vers une image différente à chaque pull (supply chain attack).

Analogie : Déployer une image non scannée, c'est accepter un colis sans vérifier l'expéditeur ni passer au détecteur à rayons X. Le scan ECR est le détecteur, l'admission controller est le douanier qui refuse les colis suspects, et la signature d'image (Cosign) est le sceau d'authenticité.

Chaîne de sécurité complète :
  • Build : image de base minimale (distroless/Alpine), multi-stage build, pas de secrets dans le Dockerfile.
  • Push : scan ECR automatique (basic ClamAV ou enhanced Inspector), signer avec Cosign.
  • Deploy : admission controller (Kyverno/OPA) qui refuse les images non scannées, non signées, ou avec CVE critiques.

  • Tag immutability : empêche l'écrasement d'un tag existant. Sans cela, un v1.0 peut être remplacé par une image malveillante. Source : SLSA Framework Level 2
  • Image signing (Cosign) : atteste que l'image a été produite par votre CI/CD (sous réserve d'une chaîne de clés correctement gérée). L'admission controller vérifie la signature avant le déploiement. Source : NIST SSDF

  • Basic scan ≠ Enhanced scan : le scan basic ne détecte que les CVE au niveau OS (APT/YUM). Les vulnérabilités dans node_modules ou requirements.txt passent à travers.

  • Lifecycle policy trop agressive : purger toutes les images >30j peut supprimer des images utilisées par des pods en cours. Toujours garder au minimum les N dernières versions taguées.
  • Pull depuis Docker Hub sans proxy : les images publiques (nginx, redis) sont soumises au rate limit Docker Hub. Utiliser un ECR pull-through cache pour proxy et scanner automatiquement.

ℹ️

Script All-in-One — Hardening EKS Complet
Script exécutable pour appliquer toutes les best practices (AWS CLI + kubectl + Terraform)
⚠️
Ce script est un template. Adaptez les variables, testez sur un cluster non-prod, et exécutez section par section. Ne lancez jamais l'intégralité en aveugle sur un cluster de production.
Bash — eks-hardening.sh
#!/usr/bin/env bash
# ═══════════════════════════════════════════════════════════
# EKS Hardening All-in-One — Template
# Adapter les variables puis exécuter section par section
# ═══════════════════════════════════════════════════════════
set -euo pipefail

# ── Variables ──
export CLUSTER_NAME="my-eks-cluster"
export REGION="eu-west-1"
export PROFILE="my-aws-profile"
export ACCOUNT_ID="$(aws sts get-caller-identity --query Account --output text --profile $PROFILE)"
export VPC_ID="$(aws eks describe-cluster --name $CLUSTER_NAME --query 'cluster.resourcesVpcConfig.vpcId' --output text --region $REGION --profile $PROFILE)"

echo "🚀 EKS Hardening — Cluster: $CLUSTER_NAME | Region: $REGION"

# ── S1 : Vérifier private endpoint ──
echo -e "\n🔴 [S1] Cluster Setup & IAM"
aws eks describe-cluster --name "$CLUSTER_NAME" \
    --query 'cluster.resourcesVpcConfig.{PublicAccess:endpointPublicAccess,PrivateAccess:endpointPrivateAccess}' \
    --region "$REGION" --profile "$PROFILE"

# Activer private endpoint
aws eks update-cluster-config --name "$CLUSTER_NAME" \
    --resources-vpc-config endpointPublicAccess=false,endpointPrivateAccess=true \
    --region "$REGION" --profile "$PROFILE" || echo "SKIP: déjà configuré ou VPN requis"

# ── S2 : Vérifier IRSA ──
echo -e "\n🔴 [S2] Authentication & IRSA"
OIDC_URL=$(aws eks describe-cluster --name "$CLUSTER_NAME" --query 'cluster.identity.oidc.issuer' --output text --region "$REGION" --profile "$PROFILE")
echo "OIDC Provider: $OIDC_URL"
aws iam list-open-id-connect-providers --profile "$PROFILE" | grep "$(echo $OIDC_URL | cut -d/ -f5)" && echo "✅ OIDC provider exists" || echo "❌ OIDC provider manquant"

# ── S3 : Vérifier NetworkPolicies ──
echo -e "\n🔴 [S3] Network Security"
kubectl get netpol -A --no-headers 2>/dev/null | wc -l | xargs -I{} echo "NetworkPolicies actives: {}"
kubectl get ns --no-headers | wc -l | xargs -I{} echo "Namespaces totaux: {}"

# ── S4 : Vérifier PSA labels ──
echo -e "\n🟠 [S4] Pod Security"
kubectl get ns -o custom-columns='NS:.metadata.name,PSA_ENFORCE:.metadata.labels.pod-security\.kubernetes\.io/enforce' 2>/dev/null

# ── S5 : Vérifier les secrets en clair ──
echo -e "\n🟠 [S5] Secrets & Configuration"
kubectl get secrets -A -o json | jq '[.items[] | select(.type=="Opaque")] | length' | xargs -I{} echo "Secrets Opaque (potentiellement non chiffrés): {}"

# ── S6 : Vérifier GuardDuty ──
echo -e "\n🟠 [S6] GuardDuty EKS Protection"
DETECTOR_ID=$(aws guardduty list-detectors --query 'DetectorIds[0]' --output text --region "$REGION" --profile "$PROFILE")
if [ "$DETECTOR_ID" != "None" ] && [ -n "$DETECTOR_ID" ]; then
  aws guardduty get-detector --detector-id "$DETECTOR_ID" \
      --query 'Features[?Name==`EKS_AUDIT_LOGS` || Name==`EKS_RUNTIME_MONITORING`].{Feature:Name,Status:Status}' \
      --region "$REGION" --profile "$PROFILE"
else
  echo "❌ GuardDuty non activé"
fi

# ── S7 : Vérifier le logging ──
echo -e "\n🟡 [S7] Logging & Monitoring"
aws eks describe-cluster --name "$CLUSTER_NAME" \
    --query 'cluster.logging.clusterLogging[0].{Types:types,Enabled:enabled}' \
    --region "$REGION" --profile "$PROFILE"

# ── S8 : Vérifier ECR scan ──
echo -e "\n🟡 [S8] Image Security"
aws ecr describe-registry --query 'replicationConfiguration' --region "$REGION" --profile "$PROFILE" 2>/dev/null
aws ecr get-registry-scanning-configuration --query 'scanningConfiguration.scanType' --region "$REGION" --profile "$PROFILE" 2>/dev/null

# ── S9 : Version du cluster ──
echo -e "\n🔵 [S9] Upgrades & Maintenance"
aws eks describe-cluster --name "$CLUSTER_NAME" \
    --query '{Version:cluster.version,Platform:cluster.platformVersion}' \
    --region "$REGION" --profile "$PROFILE"
kubectl get pdb -A --no-headers 2>/dev/null | wc -l | xargs -I{} echo "PodDisruptionBudgets actifs: {}"

# ── S10 : Coûts ──
echo -e "\n🔵 [S10] Cost Optimization"
kubectl get nodes -L karpenter.sh/capacity-type --no-headers 2>/dev/null | awk '{print $NF}' | sort | uniq -c || echo "Karpenter non détecté"
kubectl top nodes --no-headers 2>/dev/null | awk '{print "Node:", $1, "CPU:", $3, "MEM:", $5}' || echo "metrics-server non installé"

echo -e "\n✅ Audit terminé. Corrigez les ❌ ci-dessus."
