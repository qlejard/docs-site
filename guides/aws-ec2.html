Instance Metadata Service (IMDSv2) Gratuit
🔴 Critique — Migration IMDSv1 → IMDSv2 obligatoire — 15 pts — Score défaut : 0/15
🚨
Scénario d’attaque : Un attaquant compromet une application web vulnérable à une attaque SSRF (Server-Side Request Forgery). Via IMDSv1, il envoie une simple requête http://169.254.169.254/latest/meta-data/iam/security-credentials/ et récupère les credentials IAM temporaires de l’instance — sans authentification, sans token, en une seule requête HTTP. Avec ces credentials, il peut accéder à S3, appeler des APIs AWS, escalader les privilèges.

Analogie : IMDSv1, c’est comme un interphone d’immeuble sans digicode — n’importe qui dans le hall peut appuyer sur « ouvrir ». IMDSv2 est l’interphone sécurisé : vous devez d’abord sonner avec un token avant d’être entendu.

Cas réel (2019) : La brèche Capital One a exposé 100 millions de dossiers clients via une attaque SSRF exploitant IMDSv1 sur EC2. Source : AWS Security Blog.

💡
Pourquoi chaque paramètre compte :
• HttpTokens: required — C’est le paramètre central : sans lui, IMDSv1 reste accessible et toute requête non authentifiée peut lire les credentials. Recommandé par CIS AWS Benchmark v7.0.0 contrôle 5.6.3.
• HttpPutResponseHopLimit: 1 — Limite la portée du token IMDSv2 à l’instance seule (TTL=1). Avec hop=2 (défaut), un container Docker mal isolé sur l’instance peut accéder au métadat. Hop=1 bloque cette escalade.
• LaunchTemplate par défaut — Sans template imposé au niveau account/OU, chaque nouvelle instance lancée manuellement hérite de l’ancien défaut « optional ». Le template force la conformité dès la création.
• SCP de blocage — Même un admin peut revenir à IMDSv1 si aucune SCP ne l’en empêche. La SCP est la dernière ligne de défense contre une régression involontaire. Source : AWS Security Blog.
⚠️
Piège : migrer HttpTokens à required sur une instance dont l’application lit encore l’IMDS via IMDSv1 (appels sans token) provoque une rupture de service immédiate. Toujours détecter d’abord les appels IMDSv1 actifs via CloudWatch metric MetadataNoToken avant de basculer. Les SDKs AWS récents (boto3 ≥1.9, SDK Java v2, Node.js AWS SDK v3) supportent IMDSv2 nativement.

1.2 — Playbook de migration IMDSv1 → IMDSv2

1
Détecter — Identifier les instances utilisant encore IMDSv1 via AWS Config ou CloudWatch
2
Analyser — Monitorer la métrique MetadataNoToken (appels IMDSv1 actifs) pendant 7-14 jours
3
Migrer — Passer HttpTokens=required sur les instances déjà sans appels IMDSv1 actifs
4

1.3 — Détection & Configuration

AWS CLI — Détecter les instances IMDSv1 (HttpTokens=optional)
# Lister toutes les instances avec IMDSv1 encore actif dans la région
aws ec2 describe-instances \
  --filters "Name=metadata-options.http-tokens,Values=optional" \
  --query "Reservations[*].Instances[*].{ID:InstanceId,Name:Tags[?Key=='Name']|[0].Value,State:State.Name,HttpTokens:MetadataOptions.HttpTokens}" \
  --output table \
  --profile audit-readonly \
  --region eu-west-1

# Vérifier la métrique MetadataNoToken (appels IMDSv1 actifs) sur les 7 derniers jours
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name MetadataNoToken \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --start-time $(date -u -d '7 days ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 86400 --statistics Sum \
  --profile audit-readonly --region eu-west-1
AWS CLI — Migrer une instance vers IMDSv2 (HttpTokens=required)
# Migrer une instance spécifique vers IMDSv2 obligatoire
aws ec2 modify-instance-metadata-options \
  --instance-id i-0123456789abcdef0 \
  --http-tokens required \
  --http-put-response-hop-limit 1 \
  --http-endpoint enabled \
  --profile ec2-admin \
  --region eu-west-1

# Migrer TOUTES les instances d'une région en masse
aws ec2 describe-instances \
  --query "Reservations[*].Instances[?MetadataOptions.HttpTokens=='optional'].InstanceId" \
  --output text --region eu-west-1 --profile ec2-admin | \
  tr '\t' '\n' | \
  xargs -I{} aws ec2 modify-instance-metadata-options \
    --instance-id {} \
    --http-tokens required \
    --http-put-response-hop-limit 1 \
    --region eu-west-1 --profile ec2-admin
Terraform — Enforcer IMDSv2 dans un LaunchTemplate
# Terraform — LaunchTemplate avec IMDSv2 obligatoire
resource "aws_launch_template" "secure_ec2" {
  name_prefix   = "secure-"
  image_id      = data.aws_ami.amazon_linux_2023.id
  instance_type = "t3.medium"

  metadata_options {
    http_endpoint               = "enabled"
    http_tokens                 = "required"    # IMDSv2 obligatoire
    http_put_response_hop_limit = 1              # Pas d'accès depuis containers
    instance_metadata_tags      = "enabled"
  }

  tag_specifications {
    resource_type = "instance"
    tags = {
      Environment = "production"
      ManagedBy   = "terraform"
    }
  }
}

# SCP Organizations — Bloquer le retour à IMDSv1
resource "aws_organizations_policy" "deny_imdsv1" {
  name        = "DenyIMDSv1"
  description = "Interdire la modification des options IMDS vers IMDSv1"
  type        = "SERVICE_CONTROL_POLICY"

  content = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid    = "DenyIMDSv1"
      Effect = "Deny"
      Action = "ec2:ModifyInstanceMetadataOptions"
      Resource = "*"
      Condition = {
        StringEquals = {
          "ec2:MetadataHttpTokens" = "optional"
        }
      }
    }]
  })
}

Security Groups & Network ACLs Gratuit
🔴 Critique — Least privilege réseau, SSH/RDP 0.0.0.0/0 interdit — 15 pts — Score défaut : 5/15
🚨
Scénario d’attaque : Un Security Group avec la règle 0.0.0.0/0:22 expose l’instance SSH à l’Internet entier. Des scanners automatiques (Shodan, Masscan) détectent ce port en quelques minutes. Un attaquant peut lancer une attaque par force brute ou exploiter une vulnérabilité OpenSSH (ex : CVE-2024-6387 « regreSSHion ») sans aucune friction.

Analogie : le Security Group est le vigile personnel de chaque instance — contrairement à un firewall périmétrique qui protège tout le bâtiment, chaque instance a son propre videur qui inspecte les entrées ET les sorties. Laisser 0.0.0.0/0:22, c’est dire au vigile « laisse passer tout le monde ».

Contrôles officiels : AWS Security Hub CSPM fournit des contrôles dédiés pour détecter les groupes de sécurité par défaut ouverts et l’exposition SSH/RDP ou d’autres ports à risque sur Internet (EC2.2, EC2.13, EC2.14, EC2.18, EC2.19, EC2.53).

💡
Pourquoi chaque règle compte :
• Supprimer SSH/RDP 0.0.0.0/0 — Ce n’est pas optionnel : CIS AWS Foundations Benchmark v7.0.0 §6.3 et §6.4 classent ces règles comme failing dès leur présence. Security Hub EC2.22 et EC2.6 les détectent automatiquement. La meilleure solution n’est pas de restreindre à une IP — c’est de supprimer le port et d’utiliser SSM.
• Règle outbound restreinte — Par défaut 0.0.0.0/0 allow all en sortie permet à un malware de « call home » sans restriction (C2 traffic, exfiltration DNS). Restreindre les destinations légitimes limite la blast radius en cas de compromission.
• ALB devant EC2 — Ne jamais exposer directement une instance EC2 sur Internet. L’ALB isole l’instance, absorbe les DDoS L7, permet le WAF, et masque l’IP réelle de l’instance.
• Références SG-to-SG — Utiliser l’ID du Security Group comme source (plutôt qu’une plage IP) permet une maintenance dynamique : quand une instance est remplacée, la règle reste valide sans modification.
⚠️
Piège des Security Groups par défaut : le VPC default Security Group autorise tout le trafic intra-SG (toutes les instances dans ce SG se voient). Ne jamais utiliser le default SG — créer des Security Groups dédiés par rôle (web, app, db, mgmt). AWS Config rule default-security-group-closed vérifie que le SG par défaut n’a aucune règle. Source : Security Hub EC2.2.

🔴 Critique — Jamais de credentials hardcodés, instance profiles, least privilege — 15 pts — Score défaut : 3/15
🚨
Scénario d’attaque : Un développeur hardcode des credentials IAM (AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY) dans le code source d’une application deployée sur EC2. Le code est poussé sur GitHub (même privé). Des outils comme TruffleHog, GitGuardian ou des bots AWS « honeypot » détectent les clés en quelques minutes. L’attaquant crée des centaines d’instances EC2 en région ap-northeast-1 pour du cryptomining — facture AWS : 50 000 $ en 48h.

Analogie : les credentials hardcodés, c’est écrire votre code de carte bancaire sur votre bureau au travail — visible par tous, copié en douce sans que vous vous en rendiez compte. L’instance profile, c’est votre carte d’entreprise qui vous donne automatiquement accès aux ressources autorisées sans que vous ayez à mémoriser un code.

Source : ce scénario représente la cause n°1 de facturation anormale AWS. AWS rembourse généralement une fois, pas deux.

Pourquoi chaque paramètre compte :
• Instance Profile vs credentials hardcodés — L’instance profile permet à l’application de récupérer des tokens temporaires STS via IMDSv2. Ces tokens expirent toutes les 1h et sont automatiquement renouvelés par AWS. Zéro rotation manuelle, zéro risque de leak dans le code.
• Least privilege — Un role ec2:* ou AdministratorAccess attaché à une instance web signifie que la compromission de l’appli = accès admin AWS complet. Chaque instance ne doit avoir que les permissions dont elle a strictement besoin.
• GuardDuty finding InstanceCredentialExfiltration — Si des credentials temporaires de l’instance sont utilisés depuis une IP externe (exfiltration), GuardDuty détecte l’anomalie dans les minutes qui suivent. Sans GuardDuty, cette exfiltration peut passer inaperçue pendant des jours.
• PermissionBoundary — Mécanisme qui définit le plafond de droits qu’un rôle peut avoir, même si une policy permissive lui est attachée ultérieurement par erreur. Source : AWS IAM Documentation.
⚠️
Piège : attacher un instance profile à une instance déjà lancée ne remplace pas les credentials hardcodés dans les variables d’environnement ou fichiers ~/.aws/credentials. Les SDKs utilisent la credential provider chain avec un ordre de priorité strict : 1) Variables d’env → 2) Fichier credentials → 3) Instance Profile. Les credentials hardcodés prennent toujours la priorité sur l’instance profile — supprimer explicitement les fichiers ~/.aws/credentials après migration.

Key Pairs & Accès SSH/RDP Gratuit SSM inclus
🟠 Haut — EC2 Instance Connect, Session Manager zéro bastion — 10 pts — Score défaut : 4/10
🚨
Scénario d’attaque : Une clé privée SSH RSA est stockée dans le répertoire Downloads d’un développeur. Son poste est compromis. L’attaquant exfiltre la clé et se connecte en SSH directement à l’instance EC2 de production — sans aucun log, sans alerte, sans MFA.

SSM Session Manager vs SSH — Pourquoi c’est un game changer :
• Zéro port ouvert — L’instance n’a pas besoin du port 22 ni du port 3389. La communication passe par l’API SSM (HTTPS sortant depuis l’instance vers les endpoints SSM).
• Audit trail complet — Chaque session est loggée dans CloudTrail (StartSession), les logs de session peuvent être exportés vers S3 ou CloudWatch Logs. Impossible de se connecter "en douce".
• Authentification IAM — L’accès est contrôlé par des permissions IAM (ex : ssm:StartSession) + conditions (horaires, tags de ressource, MFA). Pas de clé SSH à gérer.
• EC2 Instance Connect Endpoint — Depuis 2023, EIC Endpoint permet une connexion SSH native (client ssh local) via l’API AWS, sans port 22 ouvert — le meilleur des deux mondes si SSH est indispensable. Source : AWS SSM Session Manager docs.
⚠️
Pièges :
• SSM Agent non installé — Amazon Linux 2023, Windows Server 2016+ et Ubuntu 16.04+ ont SSM Agent préinstallé. Les AMIs personnalisées doivent l’installer manuellement via yum install amazon-ssm-agent.
• Instance sans instance profile SSM — Session Manager nécessite que l’instance profile inclue la managed policy AmazonSSMManagedInstanceCore. Sans ce rôle, SSM ne peut pas établir la connexion même si l’agent est actif.
• EC2 Serial Console — En cas de crash système, la Serial Console permet un accès de récupération sans SSH — vérifier qu’elle est activée uniquement pour les administrateurs nécessaires.

Hardening des AMIs Image Builder inclus Inspector payant
🟠 Haut — Golden AMIs, CIS Benchmarks, EC2 Image Builder, Inspector post-build — 10 pts — Défaut : 5/10
🚨
Scénario d’attaque : Une équipe déploie des instances EC2 à partir d’AMIs publiques non mises à jour. L’AMI utilisée a 8 mois de retard en patches — elle embarque le kernel vulnérable à CVE-2022-0847 (« Dirty Pipe », LPE Linux). Point officiel : Amazon Inspector scanne en continu les instances EC2 pour les vulnérabilités logicielles et l’exposition réseau non intentionnelle ; le coût dépend ensuite du nombre moyen d’instances évaluées et du mode de scan retenu.

Chiffrement EBS & Snapshots Gratuit KMS CMK payant
🟠 Haut — Chiffrement par défaut au niveau account, KMS CMK, snapshots, Delete on Termination — 10 pts — Défaut : 0/10
🚨
Scénario d’attaque : Un snapshot EBS de production est partagé publiquement par erreur (--group-ids all). Les données client (PII, logs applicatifs, base de données) sont accessibles à n’importe quel compte AWS. Sans chiffrement : les données sont lisibles directement après aws ec2 create-volume --snapshot-id snap-xxx.

Analogie : un volume EBS non chiffré partagé, c’est une clé USB de données confidentielles posée dans un couloir public. Avec chiffrement + CMK KMS, même si quelqu’un copie la clé USB, les données restent illisibles sans la clé de déchiffrement — et cette clé n’est accessible qu’à votre compte via le contrôle KMS.

Point officiel : AWS recommande d’activer le chiffrement EBS par défaut au niveau Région : il s’applique aux nouveaux volumes et aux copies de snapshots, mais n’a aucun effet sur les volumes ou snapshots existants.

🟡 Moyen — Sous-réseaux privés, NAT Gateway, VPC Flow Logs, PrivateLink, NACLs — 8 pts — Défaut : 5/8
🚨

Pourquoi chaque paramètre compte :
• Subnet privé vs public — Le défaut VPC AWS assigne des IP publiques automatiquement sur les subnets du VPC par défaut. Dans un VPC personnalisé, désactiver mapPublicIpOnLaunch sur tous les subnets privés empêche tout accès Internet direct, même en cas d’erreur de SG.
• VPC Flow Logs — Indépendants des Security Groups, les Flow Logs capturent tout le trafic réseau (ACCEPT et REJECT). Essentiels pour la détection des scans de ports, mouvements latéraux et exfiltration. Le coût CloudWatch Logs peut être réduit en filtrant uniquement les flux REJECT ou en utilisant S3 + Athena.
• VPC Endpoint Gateway (S3) — Gratuit, route le trafic vers S3 directement dans le backbone AWS sans passer par Internet. Inévitable si les instances sont en subnet privé sans NAT Gateway.
• Defense in depth — NACLs + Security Groups = double couche stateless + stateful. Si une règle SG est mal configurée, le NACL du subnet peut bloquer le trafic anormal.

• NAT Gateway vs NAT Instance — NAT Gateway est géré AWS, HA natif, sans maintenance. NAT Instance reste plus flexible mais nécessite de la gestion. Côté coût, NAT Gateway est facturée à l’heure de provisionnement et au Go traité, avec des frais de transfert de données séparés et variables par région. Pour S3 ou DynamoDB, préférer des gateway endpoints afin d’éviter le data processing d’un NAT.

• Conformance Pack CIS Level 1 — AWS fournit des conformance packs prêts à l’emploi ; vérifier la version exacte du template et le nombre de règles associé avant déploiement.
• Auto-remediation + approbation — Pour les actions à risque (suppression de SG, modification d’IAM), configurer une remediation action avec approbation manuelle via SNS + Automation + workflow ITSM plutôt qu’en full-auto.
⚠️
Pièges :
• Security Hub score ≠ score de sécurité réel — Security Hub note chaque contrôle sur une échelle 0-100, mais un score de 80% peut masquer des contrôles CRITICAL non conformes qui valent individuellement plus que 10 contrôles LOW. Toujours filtrer par sévérité CRITICAL en priorité.
• Coût AWS Config — AWS Config facture séparément les configuration items enregistrés, les rule evaluations et les conformance pack evaluations. Dimensionner le service selon le volume réel de ressources, les modes d’évaluation et la périodicité.

Script All-in-One — Hardening EC2 Complet
Script AWS CLI idémpotent couvrant les 10 phases — Quick Win applicables en <15 min
⚠️
Important : Tester chaque phase sur un compte non-production avant la mise en œuvre. Les actions marquées [QW] sont les Quick Win (≤15 min, +27 pts). Le script est idémpotent : exécutez-le plusieurs fois sans risque.
AWS CLI — Script All-in-One Hardening EC2 (10 phases)
#!/usr/bin/env bash
# AWS EC2 Security Hardening - All-in-One Script
# Idempotent - safe to run multiple times
# Requires: AWS CLI v2, jq

# ============================================================
# VARIABLES - adapter avant exécution
# ============================================================
PROFILE="ec2-hardening"         # Profil AWS CLI
REGION="eu-west-1"              # Région cible
ACCOUNT_ID="$(aws sts get-caller-identity --query Account --output text --profile $PROFILE --region $REGION)"

echo "=== AWS EC2 Security Hardening === Account: $ACCOUNT_ID Region: $REGION ==="

# ============================================================
# PHASE 1 [QW] - IMDSv2 Enforce sur TOUTES les instances
# ============================================================
echo "[Phase 1] Enforce IMDSv2..."
aws ec2 describe-instances \
  --query "Reservations[*].Instances[*].InstanceId" \
  --output text \
  --profile $PROFILE --region $REGION | tr "\t" "\n" | while read id; do
  aws ec2 modify-instance-metadata-options \
    --instance-id "$id" \
    --http-tokens required \
    --http-endpoint enabled \
    --http-put-response-hop-limit 1 \
    --profile $PROFILE --region $REGION --output json > /dev/null
  echo "  IMDSv2 required: $id"
done

# ============================================================
# PHASE 2 [QW] - EBS Default Encryption
# ============================================================
echo "[Phase 2] Enable EBS default encryption..."
aws ec2 enable-ebs-encryption-by-default \
  --profile $PROFILE --region $REGION --output json
echo "  EBS default encryption: ENABLED"

# ============================================================
# PHASE 3 [QW] - Audit et révocation SG avec 0.0.0.0/0
# ============================================================
echo "[Phase 3] Audit Security Groups open to 0.0.0.0/0..."
aws ec2 describe-security-groups \
  --filters "Name=ip-permission.cidr,Values=0.0.0.0/0" \
  --query "SecurityGroups[*].{SG:GroupId,Name:GroupName,VPC:VpcId}" \
  --output table \
  --profile $PROFILE --region $REGION
echo "  [ACTION MANUELLE] Révoquer manuellement les règles SSH/RDP 0.0.0.0/0 listées ci-dessus"

# ============================================================
# PHASE 4 - GuardDuty activation
# ============================================================
echo "[Phase 4] Enable GuardDuty..."
GD_DETECTOR="$(aws guardduty list-detectors --query 'DetectorIds[0]' --output text --profile $PROFILE --region $REGION)"
if [ "$GD_DETECTOR" = "None" ] || [ -z "$GD_DETECTOR" ]; then
  aws guardduty create-detector \
    --enable \
    --profile $PROFILE --region $REGION --output json
  echo "  GuardDuty: CREATED"
else
  echo "  GuardDuty: already enabled (detector: $GD_DETECTOR)"
fi

# ============================================================
# PHASE 5 - VPC Flow Logs (sur tous les VPCs)
# ============================================================
echo "[Phase 5] Enable VPC Flow Logs..."
aws ec2 describe-vpcs \
  --query "Vpcs[*].VpcId" \
  --output text \
  --profile $PROFILE --region $REGION | tr "\t" "\n" | while read vpc; do
  EXISTING="$(aws ec2 describe-flow-logs --filter "Name=resource-id,Values=$vpc" --query 'FlowLogs[0].FlowLogId' --output text --profile $PROFILE --region $REGION)"
  if [ "$EXISTING" = "None" ] || [ -z "$EXISTING" ]; then
    echo "  [ACTION MANUELLE] Créer Flow Logs pour VPC: $vpc (destination S3 ou CWL requise)"
  else
    echo "  VPC $vpc: Flow Logs already exist ($EXISTING)"
  fi
done

# ============================================================
# PHASE 6 - Amazon Inspector v2
# ============================================================
echo "[Phase 6] Enable Amazon Inspector v2..."
aws inspector2 enable \
  --resource-types EC2 ECR \
  --profile $PROFILE --region $REGION --output json 2>&1 | grep -v "ALREADY_ENABLED"
echo "  Inspector v2: ENABLED (EC2 + ECR)"

# ============================================================
# PHASE 7 - Security Hub (FSBP + CIS)
# ============================================================
echo "[Phase 7] Enable Security Hub..."
aws securityhub enable-security-hub \
  --enable-default-standards \
  --profile $PROFILE --region $REGION --output json 2>&1 | grep -v "already"
echo "  Security Hub: ENABLED"

# ============================================================
# PHASE 8 - CloudTrail multi-régions
# ============================================================
echo "[Phase 8] Verify CloudTrail..."
aws cloudtrail describe-trails \
  --include-shadow-trails false \
  --query "trailList[*].{Name:Name,MultiRegion:IsMultiRegionTrail,Logging:HasCustomEventSelectors}" \
  --output table \
  --profile $PROFILE --region $REGION

# ============================================================
# PHASE 9 - Instances sans instance profile IAM
# ============================================================
echo "[Phase 9] Audit instances without IAM instance profile..."
aws ec2 describe-instances \
  --query "Reservations[*].Instances[?!IamInstanceProfile].{ID:InstanceId,State:State.Name}" \
  --output table \
  --profile $PROFILE --region $REGION

# ============================================================
# PHASE 10 - Vérification finale
# ============================================================
echo "=== Vérification finale ==="
echo "IMDSv2 required: $(aws ec2 describe-instances --query "Reservations[*].Instances[?MetadataOptions.HttpTokens=='required'].InstanceId" --output text --profile $PROFILE --region $REGION | wc -w) instances"
echo "EBS default enc: $(aws ec2 get-ebs-encryption-by-default --query 'EbsEncryptionByDefault' --output text --profile $PROFILE --region $REGION)"
echo "GuardDuty:       $(aws guardduty list-detectors --query 'DetectorIds[0]' --output text --profile $PROFILE --region $REGION)"
echo "Inspector:       $(aws inspector2 batch-get-account-status --account-ids $ACCOUNT_ID --query 'accounts[0].state.status' --output text --profile $PROFILE --region $REGION)"
echo "=== DONE ==="

Glossaire
Définitions des acronymes et termes techniques utilisés dans ce guide
TermeDéfinition
AMIAmazon Machine Image — Modèle d’instance EC2 contenant l’OS, les applications et la configuration. Point de départ de chaque instance.
CIS BenchmarkCenter for Internet Security — Référentiel de configuration sécurisée. Level 1 = recommandations minimales; Level 2 = durcissement avancé pouvant impacter les services.
CMKCustomer Managed Key — Clé KMS créée et gérée par le client. Différent des clés AWS managed (aws/ebs, aws/ssm) qui ne peuvent pas être restreintes.
eBPFextended Berkeley Packet Filter — Technologie kernel Linux permettant d’instrumenter le noyau sans module kernel dédié. Utilisé par GuardDuty Runtime Monitoring pour la surveillance comportementale.
EBSElastic Block Store — Stockage par blocs persistant pour EC2. Survit à l’arrêt d’instance (contrairement au stockage instance store).
FSBPAWS Foundational Security Best Practices — Standard Security Hub AWS. Pour EC2, contrôles EC2.1 (à EC2.25 couvrant IMDSv2, SG, EBS, etc.
IAMIdentity and Access Management — Service AWS de gestion des identités et des permissions. Un Instance Profile est le mécanisme qui attache un rôle IAM à une instance EC2.
IMDSInstance Metadata Service — Service HTTP accessible depuis l’instance EC2 à l’adresse 169.254.169.254. Fournit les métadonnées et credentials temporaires IAM.
IMDSv2Instance Metadata Service version 2 — Version sécurisée de IMDS utilisant des sessions orientées (token PUT avant GET). Protège contre les attaques SSRF.
KMSAWS Key Management Service — Service géré de gestion des clés de chiffrement. Intégré nativement avec EBS, S3, SSM Parameter Store, Secrets Manager.
NACLNetwork Access Control List — Pare-feu stateless au niveau subnet. Complémentaire aux Security Groups (qui sont stateful).
SCPService Control Policy — Politique AWS Organizations appliquant des restrictions au niveau account ou OU. Un SCP peut bloquer des actions même pour les admins du compte.
SGSecurity Group — Pare-feu stateful au niveau instance EC2. Contrôle le trafic entrant/sortant par protocole, port et source/destination.
SSMAWS Systems Manager — Suite de services AWS pour la gestion opérationnelle des instances. Inclut Session Manager, Run Command, Patch Manager, Parameter Store, State Manager.
SSRFServer-Side Request Forgery — Vulnérabilité permettant à un attaquant de faire envoyer des requêtes HTTP par le serveur victime. Exploité pour accéder à IMDS et voler les credentials IAM (Capital One breach 2019).
VPCVirtual Private Cloud — Réseau privé virtuel isolé dans AWS. Les instances EC2 sont déployées dans des subnets d’un VPC.
