VNets & Subnets Inclus Azure
Address space planning, segmentation, peering, topologie hub-spoke — le socle de toute architecture réseau Azure
Réf. CIS Azure Foundations v6.0.0 §7 (Network) & §7.11 (Subnets associated with NSG)
🚨
Scénario concret : Sans segmentation VNet, toutes les ressources partagent un réseau plat. Un attaquant qui compromet une VM web peut scanner l'intégralité du réseau : bases de données, Key Vaults, APIs internes — mouvement latéral libre.

Analogie : un VNet, c'est comme un quartier de la ville — chaque bâtiment (subnet) a sa fonction (résidentiel, commerce, entrepôt). Sans murs entre les quartiers, n'importe qui circule partout. La topologie hub-spoke, c'est une étoile routière : le hub est le carrefour central (firewall, gateway), les spokes sont les branches (prod, dev, DMZ). Tout le trafic passe par le carrefour = contrôle centralisé.

💡
Pourquoi chaque paramètre compte :
  • Topologie hub-spoke : un réseau plat = mouvement latéral libre. Le hub centralise le contrôle (firewall, gateway). Les spokes isolent les workloads. C'est le modèle de référence Microsoft (Cloud Adoption Framework).
  • Address space /16 par VNet : un /24 semble suffisant aujourd'hui mais bloque l'extension demain. Re-adresser un VNet en production impose une migration complète (pas de resize en place). Le /16 coûte 0 € de plus.
  • Subnet segmentation (front/app/data) : permet d'attacher un NSG par tier → front n'accepte que HTTPS, data n'accepte que depuis app. Sans ça, un seul NSG trop permissif protège tout.
  • VNet Peering avec AllowForwardedTraffic : sans ce flag, le trafic hub → spoke ne peut pas transiter via le firewall. UseRemoteGateways permet aux spokes d'utiliser le VPN/ER du hub sans gateway dédiée.
  • UDR 0.0.0.0/0 → Firewall : sans UDR, le trafic sortant des spokes part directement vers Internet sans inspection. Le UDR force tout le trafic par le firewall central → filtrage, logging, DLP.
  • Private Endpoints > Service Endpoints : les SE restreignent l'accès par subnet mais le trafic reste sur le backbone Microsoft (pas dans votre VNet). Les PE créent une IP privée dans votre subnet → trafic entièrement privé, contrôlable par NSG/Firewall.
  • Subnets dédiés : AzureFirewallSubnet et GatewaySubnet sont des noms réservés par Azure. Si vous les nommez autrement ou les sous-dimensionnez (</26 pour FW), le déploiement échoue. Créez-les dès le hub initial.
  • Diagnostic logs : sans logs VNet, aucune visibilité sur le trafic accepté/refusé → impossible de détecter une exfiltration ou un scan interne. Microsoft recommande désormais les Virtual Network Flow Logs pour les nouveaux déploiements, avec les anciens NSG Flow Logs surtout en contexte de migration.
⚠️
Pièges classiques VNets :
  • Chevauchement d'adresses : si le hub (10.0.0.0/16) et un spoke utilisent la même plage, le peering échoue silencieusement. Vérifier aussi les plages on-prem avant de configurer le VPN.
  • Peering non transitif : Spoke-A peeré au Hub et Spoke-B peeré au Hub ne communiquent pas directement. Il faut une NVA/Firewall dans le hub + UDR pour forcer le transit.
  • Oubli du UDR : le peering seul ne suffit pas à forcer le trafic par le firewall. Sans table de routes avec 0.0.0.0/0 → FW private IP, les spokes contournent le firewall.
  • default subnet : Azure crée un subnet default qui couvre tout l'espace d'adressage. Cela bloque la création de subnets dédiés. Supprimer default et recréer les subnets ciblés.
  • AzureFirewallSubnet sans /26 : Azure exige un minimum de /26 pour ce subnet. Un /28 échoue au déploiement du firewall avec une erreur peu explicite.

1.2 — Configuration

Az CLI — Création hub-spoke complète
# ── Variables ──
RG="rg-network-hub"
LOCATION="francecentral"

# ── Hub VNet ──
az network vnet create \
    --resource-group $RG \
    --name "vnet-hub" \
    --location $LOCATION \
    --address-prefixes "10.0.0.0/16"

# Subnets dédiés du hub
az network vnet subnet create --resource-group $RG --vnet-name "vnet-hub" \
    --name "AzureFirewallSubnet" --address-prefixes "10.0.1.0/26"
az network vnet subnet create --resource-group $RG --vnet-name "vnet-hub" \
    --name "GatewaySubnet" --address-prefixes "10.0.2.0/27"
az network vnet subnet create --resource-group $RG --vnet-name "vnet-hub" \
    --name "AzureBastionSubnet" --address-prefixes "10.0.3.0/26"
az network vnet subnet create --resource-group $RG --vnet-name "vnet-hub" \
    --name "ManagementSubnet" --address-prefixes "10.0.4.0/24"

# ── Spoke-Prod VNet ──
az network vnet create \
    --resource-group "rg-network-prod" \
    --name "vnet-spoke-prod" \
    --location $LOCATION \
    --address-prefixes "10.1.0.0/16"

az network vnet subnet create --resource-group "rg-network-prod" --vnet-name "vnet-spoke-prod" \
    --name "FrontSubnet" --address-prefixes "10.1.1.0/24"
az network vnet subnet create --resource-group "rg-network-prod" --vnet-name "vnet-spoke-prod" \
    --name "AppSubnet" --address-prefixes "10.1.2.0/24"
az network vnet subnet create --resource-group "rg-network-prod" --vnet-name "vnet-spoke-prod" \
    --name "DataSubnet" --address-prefixes "10.1.3.0/24"
az network vnet subnet create --resource-group "rg-network-prod" --vnet-name "vnet-spoke-prod" \
    --name "PrivateEndpointSubnet" --address-prefixes "10.1.4.0/24"

# ── Peering Hub ↔ Spoke-Prod ──
az network vnet peering create \
    --resource-group $RG \
    --name "hub-to-spoke-prod" \
    --vnet-name "vnet-hub" \
    --remote-vnet "vnet-spoke-prod" \
    --allow-vnet-access true \
    --allow-forwarded-traffic true \
    --allow-gateway-transit true

az network vnet peering create \
    --resource-group "rg-network-prod" \
    --name "spoke-prod-to-hub" \
    --vnet-name "vnet-spoke-prod" \
    --remote-vnet "vnet-hub" \
    --allow-vnet-access true \
    --allow-forwarded-traffic true \
    --use-remote-gateways true

# ── UDR : forcer le trafic spoke → Firewall ──
FW_PRIVATE_IP="10.0.1.4"  # IP privée du Azure Firewall

az network route-table create \
    --resource-group "rg-network-prod" \
    --name "rt-spoke-prod-to-fw" \
    --location $LOCATION \
    --disable-bgp-route-propagation true

az network route-table route create \
    --resource-group "rg-network-prod" \
    --route-table-name "rt-spoke-prod-to-fw" \
    --name "default-to-firewall" \
    --address-prefix "0.0.0.0/0" \
    --next-hop-type VirtualAppliance \
    --next-hop-ip-address $FW_PRIVATE_IP

# Associer le UDR aux subnets du spoke
for SN in FrontSubnet AppSubnet DataSubnet; do
    az network vnet subnet update \
        --resource-group "rg-network-prod" \
        --vnet-name "vnet-spoke-prod" \
        --name $SN \
        --route-table "rt-spoke-prod-to-fw"
done

resource routeTable 'Microsoft.Network/routeTables@2024-01-01' = {

    disableBgpRoutePropagation: true

      {
        name: 'default-to-firewall'
        properties: {
          addressPrefix: '0.0.0.0/0'
          nextHopType: 'VirtualAppliance'

        }
      }
    ]
  }
}

🚨
Scénario concret : Sans NSG, tout le trafic inbound/outbound est autorisé par défaut entre subnets. Un attaquant qui compromet un serveur web peut immédiatement scanner le port SQL (1433) du data tier, accéder au management (SSH/RDP) et pivoter vers d'autres spokes.

Analogie : un NSG, c'est le vigile à l'entrée de chaque bâtiment (subnet). Sans vigile, n'importe qui entre librement dans n'importe quel étage. Avec un NSG deny-all inbound + règles explicites, seuls les flux autorisés passent. Les ASGs (Application Security Groups) sont les badges d'accès par rôle — au lieu de gérer des IPs, vous gérez des groupes logiques (Web, App, DB).

Pourquoi chaque paramètre compte :
  • 1 NSG par subnet : un NSG au niveau NIC est trop granulaire et impossible à maintenir. Au niveau subnet, une seule règle protège toutes les VMs du tier. Sans NSG, la règle implicite Azure AllowVNetInBound (priorité 65000) autorise tout le trafic intra-VNet.
  • Deny-all inbound explicite : la règle implicite AllowVNetInBound autorise tout trafic entre subnets du même VNet. Un deny-all en priorité 4096 (avant 65000) bloque tout sauf les règles explicites → zéro trust au niveau subnet.
  • Flow logs réseau : Microsoft recommande désormais les Virtual Network Flow Logs, plus larges et mieux adaptés que les anciens NSG Flow Logs. Sans flow logs, aucune investigation forensic n'est possible après un incident réseau ; les NSG Flow Logs restent surtout un sujet de migration des déploiements existants.
  • ASGs : au lieu de source: 10.1.1.0/24, vous écrivez source: ASG-Web. Quand vous ajoutez une VM web, vous l'attachez à l'ASG → les règles s'appliquent automatiquement. Moins d'erreurs, maintenance simplifiée.
  • Ports 22/3389 fermés : les ports d'admin ouverts sur Internet sont le vecteur #1 de compromission Azure (Microsoft Digital Defense Report 2024). Bastion fournit SSH/RDP via le portail sans IP publique sur la VM.
  • Effective Security Rules : un NSG peut avoir des règles contradictoires (allow + deny sur le même port avec des priorités différentes). az network nic show-effective-nsg montre le résultat net → indispensable pour l'audit.
⚠️
Pièges classiques NSGs :
  • AllowVNetInBound implicite : cette règle cachée (priorité 65000) autorise tout le trafic intra-VNet. Elle n'est pas visible dans le portail sauf dans « Effective Security Rules ». Le deny-all en 4096 la surcharge.
  • NSG sur NIC vs. subnet : si vous avez un NSG sur le subnet ET sur la NIC, les deux sont évalués — inbound : subnet d'abord, puis NIC. Un allow au subnet mais deny à la NIC = deny. Privilégier le NSG subnet uniquement pour éviter la confusion.
  • Oubli de l'outbound : la règle implicite AllowInternetOutBound (65001) autorise tout le trafic sortant vers Internet. Un malware peut exfiltrer des données sans obstacle. Le deny-all outbound empêche cela.
  • Portée de flow logs inadaptée : certains subnets délégués (ex. AzureFirewallSubnet) n'accueillent pas de NSG. Privilégier alors un flow log au niveau VNet, subnet ou NIC selon le besoin, et compléter avec les logs natifs du firewall quand le trafic transite par lui.
  • Règle 65000 vs. 4096 : les priorités vont de 100 (plus haute) à 4096 (plus basse user). Les règles 65000+ sont système et non modifiables. Votre deny-all doit être en 4096 pour couvrir tout votre espace.

Pourquoi chaque paramètre compte :
  • Firewall dans le hub : les NSGs filtrent au niveau subnet (L3/L4) mais ne savent pas inspecter les URLs, FQDNs ou le contenu applicatif. Le firewall centralisé complète en L7 et contrôle le trafic inter-spoke (Est-Ouest) + sortant (Nord-Sud).
  • Firewall Policy (pas Classic) : les Classic Rules sont dépréciées et ne supportent pas la hiérarchie parent/enfant. La Policy permet de définir une base policy (corporate) héritée par les policies enfants (par spoke/env). Gestion centralisée.
  • Règles Application FQDN : au lieu de lister des IPs (qui changent), vous autorisez *.windows.net, *.ubuntu.com. Le firewall résout le FQDN à la volée et filtre. Sans FQDN rules, vous devez maintenir des listes d'IPs impossibles à tenir à jour.
  • Threat Intelligence = Alert & Deny : bloque automatiquement le trafic vers/depuis les IPs connues malveillantes (botnet C2, mining pools). En Alert only (défaut Basic), le trafic passe et vous recevez une alerte — trop tard si c'est un C2.
  • DNS Proxy activé : obligatoire pour que les règles FQDN fonctionnent. Sans DNS proxy, le firewall ne voit que les IPs destina-taires et ne peut pas résoudre les FQDNs dans les règles Application. Activez-le avant de créer les règles FQDN.
  • Diagnostic logs : sans logs firewall, impossible de savoir quel trafic est bloqué/autorisé. Les Structured Logs (format Resource-specific) permettent des requêtes KQL efficaces dans Log Analytics.
⚠️
Pièges classiques Azure Firewall :
  • DNS Proxy oublié : si DNS Proxy = false, les règles Application avec FQDN ne fonctionnent pas. Le firewall log « No rule matched » alors que la règle existe.
  • UDR manquant → contournement : déployer le firewall sans UDR 0.0.0.0/0 → FW IP sur les subnets spoke = le trafic ne passe jamais par le firewall. L'argent est dépensé mais rien n'est filtré.
  • AzureFirewallSubnet trop petit : Azure recommande /26 minimum. Un /28 fonctionne pour le déploiement initial mais bloque le scale-out si le trafic augmente. Prévoir /26 dès le départ.
  • SNAT exhaustion : le firewall a ~2 496 ports SNAT par IP publique. Pour un gros volume sortant, ajoutez des IPs publiques supplémentaires ou un NAT Gateway en frontal.
  • TLS Inspection + certificat interne : en Premium, la TLS inspection nécessite un CA certificate dans Key Vault. Sans lui, le déploiement Premium fonctionne mais l'inspection TLS reste inactive. Les clients internes doivent aussi faire confiance au CA.
  • Coût surprise : Azure Firewall facture à l'heure même à l'arrêt (allocated). Utilisez az network firewall deallocate en DEV pour stopper la facturation.

  • Private DNS Zone : sans elle, nslookup renvoie l'IP publique du service PaaS. Le PE existe mais le trafic ne l'utilise pas. La zone privatelink.<service>.core.windows.net doit être liée au VNet.
  • Désactiver l'accès public : un PE seul ne bloque pas l'accès public. Quelqu'un peut toujours atteindre votre Storage via l'endpoint public. Il faut explicitement désactiver l'accès public après création du PE.
  • NSG sur subnet PE : depuis 2023, Azure supporte les NSGs sur les subnets Private Endpoint. Activez-les pour contrôler qui dans votre VNet peut atteindre le PE (pas seulement le firewall PaaS).

⚠️
Pièges classiques Private Endpoints :
  • DNS non configuré : piège #1. Le PE est créé, le subnet est correct, mais la résolution DNS renvoie toujours l'IP publique. Vérifiez avec nslookup depuis une VM du VNet.
  • Accès public non désactivé : l'administrateur crée le PE et croit le service « sécurisé ». Mais l'endpoint public reste ouvert. Désactivez-le avec --public-network-access Disabled.
  • Private DNS Zone non liée au bon VNet : la zone existe mais n'est linkée qu'au hub. Les VMs du spoke ne résolvent pas. Linkez la zone à tous les VNets (ou centralisez via DNS forwarder/Private Resolver dans le hub).
  • Cross-region PE : un PE peut être dans une région différente du service PaaS, mais la latence augmente et les coûts de transfert inter-région s'appliquent.
  • Service Endpoint ≠ Private Endpoint : Service Endpoint route le trafic vers le PaaS via le backbone mais l'IP reste publique. Private Endpoint attribue une IP privée. Ne confondez pas les deux.

  • Private DNS Zones : sans elles, nslookup mystorage.blob.core.windows.net renvoie l'IP publique. Tous vos Private Endpoints sont inutiles. Chaque type PaaS a sa propre zone (privatelink.blob.core.windows.net, privatelink.database.windows.net, etc.).
  • VNet Links : une Private DNS Zone non liée à un VNet est invisible par ce VNet. Linkez chaque zone à tous les VNets concernés, ou centralisez via un Private Resolver dans le hub.
  • Private Resolver Inbound : permet aux machines on-prem de résoudre les zones Private DNS Azure. Le DC on-prem forward privatelink.* vers l'IP de l'Inbound Endpoint. Sans ça, l'hybride DNS est cassé.
  • Private Resolver Outbound : permet aux VMs Azure de résoudre les zones AD on-prem (contoso.local). Le Forwarding Ruleset définit quels suffixes sont envoyés vers quel DNS on-prem.
  • DNS Proxy (Firewall) : alternative légère au Resolver. Le Firewall agit comme DNS proxy et résout via Azure DNS. Plus simple mais ne gère pas le forwarding on-prem aussi bien.
⚠️
Pièges classiques DNS Azure :
  • Zone link oublié sur un spoke : la Private DNS Zone est liée uniquement au hub. Les VMs du spoke-dev résolvent l'IP publique du Storage au lieu du PE. Symptôme : « connection timeout » car le trafic sort par Internet alors que l'accès public est désactivé.
  • DNS servers custom + pas de redirection : vous configurez un DNS custom sur le VNet (ex: AD DC). Si ce DC ne forward pas vers 168.63.129.16, les Private DNS Zones deviennent invisibles. Toujours ajouter un conditional forwarder ou Private Resolver.
  • Resolver Inbound ≠ Outbound : Inbound = on-prem qui interroge Azure. Outbound = Azure qui interroge on-prem. Les deux nécessitent des subnets dédiés distincts (délégués à Microsoft.Network/dnsResolvers). Ne les mettez pas dans le même subnet.
  • TTL trop long après migration : si vous migrez un endpoint public vers un PE, le cache DNS (TTL) peut garder l'ancienne IP publique pendant des heures. Baissez le TTL à 60s avant migration, puis remontez après validation.

Pourquoi chaque paramètre compte :
  • WAF en mode Prevention : en Detection, le WAF journalise les attaques mais les laisse passer. En Prevention, il bloque activement. Commencez en Detection pour évaluer les faux positifs, puis passez en Prevention après tuning.
  • DRS 2.1 (Default Rule Set) : remplace OWASP CRS 3.x avec de meilleures détections et moins de faux positifs. Couvre OWASP Top 10, bots, et scanners.
  • TLS 1.2 minimum : TLS 1.0/1.1 ont des vulnérabilités connues. La policy AppGwSslPolicy20220101 force TLS 1.2+ avec des ciphers modernes.

⚠️

  • Subnet trop petit : l'App Gateway v2 a besoin d'un subnet dédié /24 recommandé (minimum /26). Chaque instance autoscalée consomme une IP privée.
  • WAF Detection ≠ Protection : une erreur fréquente est de rester en Detection en production en pensant être protégé. Les logs montrent les attaques, mais elles passent toutes.
  • NSG bloque la health probe : l'App Gateway utilise le tag GatewayManager sur les ports 65503-65534 (v1) ou 65200-65535 (v2). Si le NSG du subnet bloque ces ports, le gateway passe en Unhealthy.
  • Backend pool en IP publique : si le backend est dans le même VNet, utilisez l'IP privée. L'App Gateway via IP publique sort et re-rentre dans le VNet inutilement.
  • Certificate expiré : le certificat TLS sur le listener expire silencieusement. Configurez une alerte Key Vault ou un check automatisé.

Scénario concret : Une attaque DDoS volumétrique cible l'IP publique de votre Application Gateway. Sans protection, Azure absorbe ce qu'il peut au niveau plateforme (Basic), mais vos ressources saturent, le WAF ne répond plus, et l'application plante.

Pourquoi chaque paramètre compte :
  • Network Protection vs Basic : Basic ne protège qu'au niveau infrastructure Azure (couche 3/4 grossière). Network Protection ajoute l'auto-tuning adaptatif par IP publique, la télémétrie d'attaque en temps réel, le cost protection (remboursement scale-out pendant une attaque) et l'accès au DDoS Rapid Response Team.
  • Auto-tuning : apprend le profil de trafic normal de chaque IP publique et ajuste les seuils de détection automatiquement. Sans cela, les seuils génériques peuvent être trop hauts (attaque non détectée) ou trop bas (faux positifs).
  • Alertes « Under DDoS Attack » : la métrique IfUnderDDoSAttack passe à 1 dès détection. Sans alerte, vous ne savez pas que vous êtes attaqué jusqu'à ce que les utilisateurs signalent des problèmes.
  • Diagnostic logs : les logs DDoSProtectionNotifications, DDoSMitigationFlowLogs et DDoSMitigationReports sont essentiels pour le post-mortem et la conformité.
⚠️
Pièges classiques DDoS Protection :
  • Coût surprise : DDoS Network Protection ajoute un plan mensuel et peut facturer au-delà du périmètre couvert. Il se justifie surtout pour des workloads publics critiques ; pour quelques IPs seulement, comparer avec l'option DDoS IP Protection sur la page pricing live.
  • Basic ≠ protection applicative : Basic protège l'infrastructure Azure, pas votre application. Les attaques L7 (HTTP flood, slowloris) nécessitent un WAF en complément.
  • Plan non associé au VNet : créer un DDoS Protection Plan ne suffit pas — il faut l'associer explicitement à chaque VNet à protéger.
  • IPs publiques non rattachées : seules les IPs publiques sur un VNet associé au plan sont protégées. Une IP orpheline (non associée à une resource sur le VNet) n'est pas couverte.

Az CLI — Script All-in-One Hardening Réseau Azure
#!/bin/bash
# ═══════════════════════════════════════════════════════
# Script All-in-One — Hardening Réseau Azure
# Guide de Configuration & Audit — v1.0
# ═══════════════════════════════════════════════════════

set -euo pipefail

# ── VARIABLES GLOBALES ──
LOCATION="francecentral"
RG_HUB="rg-network-hub-prod"
RG_SPOKE="rg-network-spoke-prod"
VNET_HUB="vnet-hub-prod"
VNET_SPOKE="vnet-spoke-prod"
LOG_WORKSPACE="log-hub-prod"

# Créer les Resource Groups
az group create --name $RG_HUB --location $LOCATION
az group create --name $RG_SPOKE --location $LOCATION

# Log Analytics Workspace
az monitor log-analytics workspace create \
    --resource-group $RG_HUB --workspace-name $LOG_WORKSPACE \
    --retention-time 90

# ═══ S1 — VNets & Subnets ═══
echo "[S1] Création VNets Hub & Spoke..."
az network vnet create --resource-group $RG_HUB --name $VNET_HUB \
    --address-prefixes 10.0.0.0/16
az network vnet subnet create --resource-group $RG_HUB --vnet-name $VNET_HUB \
    --name AzureFirewallSubnet --address-prefixes 10.0.1.0/26
az network vnet subnet create --resource-group $RG_HUB --vnet-name $VNET_HUB \
    --name GatewaySubnet --address-prefixes 10.0.2.0/27

az network vnet create --resource-group $RG_SPOKE --name $VNET_SPOKE \
    --address-prefixes 10.1.0.0/16
az network vnet subnet create --resource-group $RG_SPOKE --vnet-name $VNET_SPOKE \
    --name snet-app --address-prefixes 10.1.1.0/24

# Peering
az network vnet peering create --resource-group $RG_HUB \
    --name hub-to-spoke --vnet-name $VNET_HUB \
    --remote-vnet $VNET_SPOKE --allow-forwarded-traffic --allow-gateway-transit
az network vnet peering create --resource-group $RG_SPOKE \
    --name spoke-to-hub --vnet-name $VNET_SPOKE \
    --remote-vnet $VNET_HUB --use-remote-gateways

# ═══ S2 — NSGs ═══
echo "[S2] Création NSGs..."
az network nsg create --resource-group $RG_SPOKE --name "nsg-app"
az network nsg rule create --resource-group $RG_SPOKE --nsg-name "nsg-app" \
    --name DenyAllInbound --priority 4096 --direction Inbound \
    --access Deny --protocol '*' --source-address-prefixes '*' \
    --destination-address-prefixes '*' --destination-port-ranges '*'

# ═══ S3 — Azure Firewall ═══
echo "[S3] Déploiement Azure Firewall..."
az network public-ip create --resource-group $RG_HUB --name pip-fw --sku Standard --zone 1 2 3
az network firewall policy create --resource-group $RG_HUB --name policy-fw-prod \
    --sku Standard --threat-intel-mode Deny --enable-dns-proxy true
az network firewall create --resource-group $RG_HUB --name fw-hub-prod \
    --sku AZFW_VNet --tier Standard --vnet-name $VNET_HUB \
    --public-ip pip-fw --firewall-policy policy-fw-prod

# ═══ S8 — Load Balancer Standard ═══
echo "[S8] Déploiement Load Balancer Standard..."
az network public-ip create --resource-group $RG_SPOKE --name pip-lb-prod --sku Standard --zone 1 2 3
az network lb create --resource-group $RG_SPOKE --name lb-prod --sku Standard \
    --frontend-ip-name frontend --public-ip-address pip-lb-prod --backend-pool-name backend-pool
az network lb probe create --resource-group $RG_SPOKE --lb-name lb-prod \
    --name probe-https --protocol Https --port 443 --path /health

# ═══ S9 — DDoS Protection ═══
echo "[S9] Création DDoS Protection Plan..."
az network ddos-protection create --resource-group $RG_HUB --name ddos-plan-prod
az network vnet update --resource-group $RG_HUB --name $VNET_HUB \
    --ddos-protection-plan ddos-plan-prod --ddos-protection true

# ═══ S10 — VNet Flow Logs ═══
echo "[S10] Activation Virtual Network Flow Logs..."
az network watcher flow-log create --resource-group NetworkWatcherRG \
    --name flowlog-vnet-hub \
    --vnet $(az network vnet show --resource-group $RG_HUB --name $VNET_HUB --query id -o tsv) \
    --storage-account stflowlogs --retention 90 \
    --traffic-analytics true --workspace $LOG_WORKSPACE --interval 10

echo "✅ Hardening réseau terminé. Vérifiez les résultats dans le portail Azure."
