📋
Ce document est un référentiel de gouvernance Power Platform couvrant la sécurisation des environnements, la prévention de fuite de données (DLP), la sécurité des applications, et le monitoring. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un tenant.

Analogie globale : Power Platform est une ville ouverte par défaut — tout le monde peut construire, connecter des services et publier. Ce guide transforme cette ville ouverte en zone réglementée avec permis de construire (environnements), douanes (DLP), contrôle d'accès (sécurité) et vidéosurveillance (monitoring).

Pourquoi chaque paramètre compte :
  • Who can create environments : par défaut, tout utilisateur M365 peut créer un environnement Production ou Sandbox. Chaque environnement créé consomme de la capacité Dataverse et échappe au monitoring si non gouverné. Source : Microsoft Learn — Control environment creation.
  • Maker welcome content : sans message d'accueil personnalisé, les makers créent des apps dans l'environnement par défaut sans savoir qu'un environnement dédié existe. Le message redirige vers le bon processus.
  • Sharing controls (Managed Env) : sans limite, un maker peut partager une app canvas avec l'organisation entière (Everyone). Avec une limite nominative définie par l'organisation (par exemple 20 utilisateurs), le partage large nécessite une approbation — principe de moindre privilège. Source : Microsoft Learn — Sharing limits.
  • Managed Environments : c'est le pilier de la gouvernance — sans activation, pas de sharing limits, pas de solution checker forcé, pas de weekly digest, pas de pipelines. C'est la « résidence surveillée » : l'environnement fonctionne, mais avec des règles et un monitoring renforcés.
  • Solution checker enforcement : sans enforcement, les solutions importées peuvent contenir des vulnérabilités (XSS, appels HTTP non sécurisés). En mode « Block », aucune solution avec erreur critique ne peut être importée.
  • Naming convention : sans convention, les environnements s'appellent « Test », « Mon app », « Projet X » — impossible d'identifier le propriétaire, le type, ou la business unit. La convention {Org}-{Type}-{Nom} rend l'inventaire lisible.
  • Rename default environment : le nom par défaut n'indique pas que c'est un environnement non-production. Renommer en « {Org}-Default-NePasUtiliser » décourage les makers d'y travailler.
⚠️
Piège classique : restreindre la création d'environnements ne supprime pas les environnements existants. Après activation, il faut auditer et nettoyer les environnements orphelins déjà créés. De plus, Managed Environments peut être désactivé via PowerShell, mais uniquement après avoir vérifié qu'aucune capacité de gouvernance gérée n'est encore requise ; tester d'abord sur un sandbox reste prudent.

Attention : l'environnement par défaut ne peut pas être supprimé. Il faut le verrouiller (DLP restrictive + rename + sharing limits) et rediriger les makers vers des environnements dédiés.

PowerShell — Configuration

PowerShell — Restreindre la création d'environnements
# Prérequis : module Microsoft.PowerApps.Administration.PowerShell
# Install-Module -Name Microsoft.PowerApps.Administration.PowerShell -Force
Add-PowerAppsAccount

# Désactiver la création d'environnements pour les non-admins
$tenantSettings = Get-TenantSettings
$tenantSettings.powerPlatform.environments.disableEnvironmentCreationByNonAdminUsers = $true
$tenantSettings.powerPlatform.environments.disableTrialEnvironmentCreationByNonAdminUsers = $true
$tenantSettings.powerPlatform.environments.disableDeveloperEnvironmentCreationByNonAdminUsers = $true
Set-TenantSettings -RequestBody $tenantSettings

# Renommer l'environnement par défaut
$defaultEnv = Get-AdminPowerAppEnvironment -Default
Set-AdminPowerAppEnvironmentDisplayName -EnvironmentName $defaultEnv.EnvironmentName `
    -NewDisplayName "Contoso-Default-NePasUtiliser"

🚨
Scénario concret : Sans DLP, un employé crée un flow qui copie les données SharePoint interne vers Dropbox ou Gmail via un connecteur standard. Pourquoi chaque paramètre compte :
  • DLP tenant-wide : c'est le filet de sécurité global. Sans politique tenant-wide, tout nouvel environnement créé n'a aucune restriction DLP — les connecteurs y sont tous utilisables ensemble. Source : Microsoft Learn — Data loss prevention policies.
  • Business / Non-Business groups : les connecteurs du groupe Business peuvent échanger des données entre eux. Un flow ne peut PAS combiner un connecteur Business et un Non-Business. C'est le mécanisme central de DLP Power Platform.
  • HTTP connector : c'est la faille #1. Le connecteur HTTP permet d'envoyer des données vers n'importe quelle URL externe — contournant toute protection par défaut. Bloquer HTTP est le Quick Win le plus impactant. Source : Microsoft Learn — Block HTTP requests.
  • Custom connectors : par défaut, les custom connectors sont dans le groupe Business et peuvent interagir avec SharePoint, Dataverse, etc. Un custom connector malveillant pourrait exfiltrer des données. Les forcer en Non-Business par défaut empêche l'exfiltration.
  • Connector action control : un connecteur SharePoint autorisé ne signifie pas que toutes ses actions sont nécessaires. Bloquer « Delete item » ou « Delete file » réduit le risque de suppression massive accidentelle ou malveillante.
  • Endpoint filtering : si HTTP doit être autorisé dans certains environnements, l'endpoint filtering restreint les URLs appelables à une liste blanche. Fonctionnalité plus récente, à activer systématiquement. Source : Microsoft Learn — DLP connector configuration.
⚠️
Piège classique : créer une DLP qui couvre « Add multiple environments » au lieu de « All environments ». Tout nouvel environnement créé après la politique ne sera pas couvert. Toujours utiliser le scope tenant-wide pour la politique restrictive de base.

Autre piège : lors d'un changement de DLP, Power Platform propage la politique, réévalue les apps, flows et chatbots existants, peut les placer en suspension / quarantaine, puis désactive les connexions devenues non conformes. Pendant cette propagation, certains usages échouent ensuite aussi à l'exécution. Prévoir un audit post-DLP avec le CoE Starter Kit et une fenêtre de propagation pouvant aller jusqu'à 24 h dans les cas les plus volumineux.

PowerShell — Configuration

PowerShell — Créer la politique DLP restrictive
# Prérequis : module Microsoft.PowerApps.Administration.PowerShell
Add-PowerAppsAccount

# Créer la politique DLP tenant-wide restrictive
$policy = New-DlpPolicy `
    -DisplayName "Contoso-DLP-Restrictive-TenantWide" `
    -EnvironmentType "AllEnvironments"

# Déplacer les connecteurs M365 dans le groupe Business
$businessConnectors = @(
    "shared_sharepointonline",
    "shared_office365",
    "shared_teams",
    "shared_onedriveforbusiness",
    "shared_excelonlinebusiness",
    "shared_planner",
    "shared_approvals"
)
foreach ($connector in $businessConnectors) {
    Add-ConnectorToBusinessDataGroup -PolicyName $policy.Name `
        -ConnectorName $connector
}

# Bloquer les connecteurs HTTP (HTTP, HTTP with Microsoft Entra ID, HTTP Webhook)
$connectorsToBlock = @(
    [pscustomobject]@{
        id = "/providers/Microsoft.PowerApps/apis/shared_sendhttp"
        name = "HTTP"
        type = "providers/Microsoft.PowerApps/apis"
    },
    [pscustomobject]@{
        id = "/providers/Microsoft.PowerApps/apis/shared_webcontents"
        name = "HTTP with Microsoft Entra ID"
        type = "providers/Microsoft.PowerApps/apis"
    },
    [pscustomobject]@{
        id = "/providers/Microsoft.PowerApps/apis/shared_httpwebhook"
        name = "HTTP Webhook"
        type = "providers/Microsoft.PowerApps/apis"
    }
)
Add-ConnectorsToPolicy -PolicyName $policy.Name `
    -Connectors $connectorsToBlock `
    -Classification "Blocked"

# Configurer les custom connectors en Non-Business par défaut
Set-DlpPolicy -PolicyName $policy.Name `
    -DefaultConnectorsClassification "NonBusiness"

  • Tenant isolation : sans isolation, un utilisateur externe (tenant B) peut créer un flow qui accède aux données SharePoint de votre tenant A via un connecteur partagé.   • App sharing : sans Managed Environments, il n'y a aucun contrôle natif sur le nombre de personnes avec qui un maker partage une app. La seule protection est la DLP (qui contrôle les données, pas l'accès à l'app).
  • App quarantine : permet de désactiver instantanément une app signalée comme non conforme sans la supprimer (réversible). Source : Microsoft Learn — Quarantine apps.
  • Connexions implicites : quand un maker partage une app canvas utilisant SharePoint, la connexion du maker est partagée — les utilisateurs accèdent aux données avec les permissions du maker, pas les leurs. Passer en connexion explicite force chaque utilisateur à s'authentifier avec ses propres permissions.
  • Guest access : Power Apps peut être partagé avec des guests Microsoft Entra ID. Sans contrôle, un maker pourrait partager une app métier avec un sous-traitant externe. Vérifier la politique guest access de votre tenant.
⚠️
Piège classique : activer le tenant isolation casse immédiatement toutes les connexions cross-tenant existantes. Vérifier d'abord avec Get-PowerAppTenantIsolationPolicy s'il y a des connexions actives vers d'autres tenants. Ajouter les tenants partenaires légitimes en exception avant d'activer.
