🚨
Scénario : Un attaquant compromet une VM dans votre VNet Azure. Sans Azure Firewall, il pivote librement vers toutes les autres VMs, accède à Internet pour exfiltrer des données, et contacte ses serveurs C2 (Command & Control) via HTTPS — le tout sans aucun contrôle ni journalisation.

Analogie : Azure Firewall est le "douanier central" de votre réseau cloud. Tout le trafic entre zones (Internet, entre VNets, vers on-premises) doit passer par lui — il décide qui entre, qui sort, et journalise chaque décision.

Pourquoi chaque paramètre compte :
• FQDN Tags : permettent d'autoriser certains services Microsoft via une seule règle, sans maintenir manuellement des listes d'adresses qui évoluent.
• Threat Intelligence : alimenté par Microsoft Threat Intelligence (MSTIC), il bloque les IPs et FQDNs connus comme malveillants. En mode "Alert seul" (Standard), il journalise sans bloquer — passer en Deny via Firewall Policy pour une protection active.
• Zone Redundancy : les SKUs Azure Firewall supportent les Availability Zones et la documentation précise qu'il n'y a pas de surcoût spécifique à ce déploiement.
• Forced Tunneling : si activé, le plan de gestion du Firewall utilise une IP de management dédiée et le trafic Internet peut être redirigé vers un autre next hop ou bloqué.

1.3 — Déploiement Azure Firewall Standard avec Zone Redundancy

☁️ Azure CLI — Déploiement Azure Firewall Standard + Zone Redundancy 
# Variables
$RG         = "rg-hub-prod"
$LOCATION   = "westeurope"
$VNET_NAME  = "vnet-hub"
$FW_NAME    = "fw-hub-prod"
$FW_POLICY  = "fwpol-hub-prod"
$FW_PIP     = "pip-fw-hub-prod"

# Créer la Public IP (Standard SKU requis pour Zone Redundancy)
az network public-ip create \
  --resource-group $RG \
  --name $FW_PIP \
  --sku Standard \
  --zone 1 2 3 \
  --location $LOCATION

# Créer la Firewall Policy (méthode recommandée)
az network firewall policy create \
  --resource-group $RG \
  --name $FW_POLICY \
  --location $LOCATION \
  --sku Standard \
  --threat-intel-mode Alert

# Créer Azure Firewall Standard avec Zone Redundancy
az network firewall create \
  --resource-group $RG \
  --name $FW_NAME \
  --location $LOCATION \
  --sku AZFW_VNet \
  --tier Standard \
  --vnet-name $VNET_NAME \
  --public-ip $FW_PIP \
  --firewall-policy $FW_POLICY \
  --zones 1 2 3

# Obtenir l'IP privée du Firewall (pour les UDRs)
az network firewall show \
  --resource-group $RG \
  --name $FW_NAME \
  --query "ipConfigurations[0].privateIPAddress" \

💻 PowerShell (Az.Network) — Déploiement Azure Firewall Standard 
# Variables
$rgName      = "rg-hub-prod"
$location    = "westeurope"
$vnetName    = "vnet-hub"
$fwName      = "fw-hub-prod"
$fwPolicyName= "fwpol-hub-prod"
$pipName     = "pip-fw-hub-prod"

# Créer la Public IP avec Zone Redundancy
$pip = New-AzPublicIpAddress -ResourceGroupName $rgName -Name $pipName `
    -Location $location -Sku Standard -Zone 1,2,3 -AllocationMethod Static

# Créer la Firewall Policy
$fwPolicy = New-AzFirewallPolicy -ResourceGroupName $rgName -Name $fwPolicyName `
    -Location $location -SkuTier Standard -ThreatIntelMode Alert

# Obtenir le VNet et le sous-réseau AzureFirewallSubnet
$vnet    = Get-AzVirtualNetwork -ResourceGroupName $rgName -Name $vnetName
$fwSubnet= Get-AzVirtualNetworkSubnetConfig -VirtualNetwork $vnet -Name "AzureFirewallSubnet"

# Configurer l'IP du Firewall
$fwIpConfig = New-AzFirewallIpConfiguration -Name "IpConf" -PublicIpAddress $pip `
    -Subnet $fwSubnet

# Créer Azure Firewall Standard avec Zone Redundancy
$fw = New-AzFirewall -ResourceGroupName $rgName -Name $fwName `
    -Location $location -IpConfiguration $fwIpConfig `
    -FirewallPolicyId $fwPolicy.Id -Zone 1,2,3

# Vérification
Get-AzFirewall -ResourceGroupName $rgName -Name $fwName | `
    Select-Object Name, Location, Zones, ProvisioningState

Firewall Policy pay-per-hour
Ressource centralisée recommandée — Rule Collection Groups, hiérarchie, RBAC, migration depuis classic rules
🚨
Scénario : Sans Firewall Policy centralisée, chaque équipe définit ses propres règles par Firewall. Résultat : incohérences entre environnements (Dev autorise ce que Prod bloque), pas de gouvernance, impossible d'auditer l'ensemble des règles en vigueur, et la migration vers un deuxième Firewall nécessite de recréer toutes les règles manuellement.

Analogie : La Firewall Policy est le "règlement intérieur" de votre réseau — la liste partageable des autorisations et interdictions. Un seul règlement peut s'appliquer à plusieurs firewalls (multi-régions, multi-environments).

2.1 — Structure d'une Firewall Policy

Firewall Policy (fwpol-hub-prod) │ ├── Rule Collection Group : RCG-Infra [Priority: 100] │ ├── Rule Collection : RC-DNAT-JumpBox [DNAT, Priority: 100] │ └── Rule Collection : RC-Network-Azure [Network, Priority: 200] │ ├── Rule Collection Group : RCG-Workloads [Priority: 200] │ ├── Rule Collection : RC-App-Egress [Application, Priority: 100] │ └── Rule Collection : RC-Network-Spoke [Network, Priority: 200] │ └── Rule Collection Group : RCG-Security [Priority: 300] ├── Rule Collection : RC-Block-Malicious [Network, Priority: 100 — Deny] └── Rule Collection : RC-Block-BadFQDN [Application, Priority: 200 — Deny]

💡
Pourquoi chaque paramètre compte :
• ThreatIntelMode = Deny : bloque en temps réel les communications vers des IPs/domaines connus comme malveillants selon MSTIC. En mode Alert seul, un malware peut quand même contacter son C2.
• DNS Proxy = Enabled : obligatoire pour que les FQDN fonctionnent dans les Network Rules. Sans DNS Proxy, seules les Application Rules supportent les FQDN — ce qui limite les scénarios de filtrage non-HTTP.
• Hiérarchie de policies : une policy parent ("globale") peut imposer des règles non modifiables aux policies enfants (régionales, par business unit). Utile pour les grandes organisations avec plusieurs équipes.
• SNAT PrivateRanges : par défaut, le Firewall SNATe tout le trafic vers Internet. Si vous avez des plages IP privées non-RFC1918 (ex : 100.64.0.0/10), déclarez-les explicitement pour éviter un SNAT non désiré.
⚠️
Piège migration Classic Rules → Firewall Policy : si vous avez créé votre Firewall avec des "Classic Rules" (avant Firewall Policy), vous ne pouvez pas mélanger les deux modèles. La migration se fait via Azure CLI : az network firewall policy create + importer les règles manuellement. Prévoir une fenêtre de maintenance.

2.3 — Créer une Firewall Policy avec hiérarchie et Threat Intel

☁️ Azure CLI — Firewall Policy + Rule Collection Group 
# Créer la Firewall Policy Standard avec Threat Intel en mode Deny
az network firewall policy create \
  --resource-group rg-hub-prod \
  --name fwpol-hub-prod \
  --location westeurope \
  --sku Standard \
  --threat-intel-mode Deny \
  --enable-dns-proxy true \
  --dns-servers 168.63.129.16

# Créer le Rule Collection Group "Infrastructure" (priorité 100)
az network firewall policy rule-collection-group create \
  --resource-group rg-hub-prod \
  --policy-name fwpol-hub-prod \

  --priority 100

az network firewall policy rule-collection-group create \
  --resource-group rg-hub-prod \

Règles : DNAT, Network, Application pay-per-hour
Ordre de traitement, DNAT (inbound), Network (L4), Application (FQDN/L7), FQDN Tags, Default Deny
🚨
Scénario : Sans règles explicites, Azure Firewall applique un Default Deny total — tout le trafic est bloqué. Un mauvais ordre de règles (Network avant Application pour le même flux) peut faire passer du trafic HTTPS non inspecté, ou au contraire bloquer des services Azure légitimes comme Windows Update ou Azure Monitor.

Analogies : DNAT = "standard téléphonique" (redirige les appels entrants vers le bon interlocuteur) | Network Rules = "liste blanche IP/port" | Application Rules = "contrat de navigation web" (autorise par nom de domaine, pas par IP).

3.1 — Ordre de traitement des règles (CRITIQUE)

Trafic entrant/sortant │ ▼ 1. DNAT Rules → Si match : traduire la destination et transmettre │ (Si pas de match → passe à l'étape suivante) ▼ 2. Network Rules → Si match Allow : autoriser | Si match Deny : bloquer │ (Si pas de match → passe à l'étape suivante) ▼ 3. Application Rules → Si match Allow : autoriser | Si match Deny : bloquer │ (Si pas de match) ▼ 4. Default Deny → BLOQUER & journaliser (comportement par défaut)

Pourquoi chaque type de règle compte :
• FQDN Tags (WindowsUpdate, AzureKubernetesService…) : Microsoft maintient ces listes à jour automatiquement. Sans eux, vous devez gérer manuellement des centaines d'IPs et FQDNs Microsoft qui changent régulièrement — et rater une mise à jour peut bloquer Windows Update sur l'ensemble de votre parc.
• Application Rules pour HTTPS : à la différence des Network Rules, elles autorisent par nom de domaine (FQDN). Une IP peut changer (CDN, load balancer), un FQDN est stable — c'est la bonne maille de filtrage pour Internet egress.
• Ordre DNAT → Network → Application : ce n'est pas configurable. Comprendre cet ordre évite des règles redondantes ou conflictuelles. Une Network Rule "allow all" en priorité haute court-circuite toutes les Application Rules.
⚠️
Piège : Network Rules vs Application Rules pour HTTPS
Une Network Rule allow * → * :443 TCP autorise tout le HTTPS sans inspection, bypasse les Application Rules, et empêche la TLS Inspection (Premium). Utilisez toujours des Application Rules pour le trafic HTTPS sortant vers Internet — et réservez les Network Rules aux flux TCP/UDP non-HTTP (SQL, DNS, NTP, RDP...).

3.4 — Déploiement des règles de base

☁️ Azure CLI — Règles DNAT + Network + Application 
$RG     = "rg-hub-prod"
$POLICY = "fwpol-hub-prod"
$RCG    = "RCG-Infra"
$FW_PIP = $(az network public-ip show -g $RG -n pip-fw-hub-prod --query ipAddress -o tsv)

# ── DNAT : RDP vers JumpBox ──────────────────────────────────────────
az network firewall policy rule-collection-group collection add-nat-collection \
  --resource-group $RG --policy-name $POLICY \
  --rule-collection-group-name $RCG \
  --name RC-DNAT-JumpBox --collection-priority 100 --action DNAT \
  --rule-name dnat-jumpbox-rdp \
  --translated-address 10.0.1.10 --translated-port 3389 \
  --ip-protocols TCP \
  --source-addresses "*" \
  --destination-addresses $FW_PIP \
  --destination-ports 3389

# ── Network : DNS + NTP + inter-spoke ───────────────────────────────
az network firewall policy rule-collection-group collection add-filter-collection \
  --resource-group $RG --policy-name $POLICY \
  --rule-collection-group-name $RCG \
  --name RC-Network-Core --collection-priority 200 \
  --action Allow --rule-type NetworkRule \
  --rule-name net-dns-azure \
  --ip-protocols UDP \
  --source-addresses "10.0.0.0/8" \
  --destination-addresses "168.63.129.16" \

Scénario : Sans architecture Hub-Spoke, chaque VNet "Spoke" a sa propre passerelle Internet, ses propres NSGs à gérer, et les équipes de sécurité ne voient pas le trafic entre Spokes. Une VM compromise dans Spoke-A peut atteindre directement toutes les VMs de Spoke-B sans passer par aucun contrôle.

Analogie : Le Hub est le "carrefour giratoire" de votre réseau — tout le trafic inter-Spoke et vers Internet passe par lui. Azure Firewall dans le Hub est le radar de ce carrefour.

5.1 — Architecture Hub-Spoke avec Azure Firewall

Internet │ ▼ Hub VNet (10.0.0.0/16) │ ├── AzureFirewallSubnet (10.0.1.0/26) ← Azure Firewall Premium │ ├── GatewaySubnet (10.0.2.0/27) ← VPN/ExpressRoute Gateway │ └── AzureFirewallManagementSubnet ← (si Forced Tunneling) │ ├──[VNet Peering]──> Spoke-1 VNet (10.1.0.0/16) — Workloads Production │ UDR: 0.0.0.0/0 → Firewall IP privée │ ├──[VNet Peering]──> Spoke-2 VNet (10.2.0.0/16) — Développement │ UDR: 0.0.0.0/0 → Firewall IP privée │ └──[VPN/ExpressRoute]──> On-Premises
💡
Règle critique — le Peering doit avoir AllowGatewayTransit / UseRemoteGateways désactivé séparément des UDRs. Le trafic inter-Spoke passe par le Firewall grâce aux User Defined Routes (UDRs) sur les subnets Spoke — et non par une configuration de Peering. Sans UDR 0.0.0.0/0 → IP_Firewall sur chaque subnet Spoke, le trafic bypasse complètement le Firewall.

DNS Proxy, Custom DNS, FQDN dans Network Rules, Private DNS Zones, split-horizon DNS, Private Endpoints
🚨
Scénario : Sans DNS Proxy, les Network Rules avec des noms FQDN (*.storage.azure.com) ne fonctionnent pas — elles ne sont évaluées que si le Firewall résout lui-même le nom. Les VMs résolvent les FQDNs via leur propre DNS, les IPs obtenues ne correspondent pas à ce que le Firewall attendait, et les règles ne correspondent jamais. Résultat : certains flux passent en dehors des règles FQDN.

Analogie : Le DNS Proxy est l'"annuaire centralisé" du Firewall — toutes les VMs appellent cet annuaire (le Firewall), qui résout à leur place et peut ainsi filtrer par FQDN dans les Network Rules.

💡
Pourquoi le DNS Proxy est critique :
• FQDN dans Network Rules : sans DNS Proxy, les FQDN dans les Network Rules ne matchent que les IPs résolues par le Firewall lui-même. Si une VM résout api.example.com via un DNS différent, le match échoue.
• Private DNS Zones : les Private Endpoint DNS Zones (privatelink.blob.core.windows.net) nécessitent que le DNS Proxy soit la passerelle entre les VMs et Azure DNS (168.63.129.16) — qui est uniquement accessible depuis le VNet Azure.
• Audit DNS : le DNS Proxy journalise toutes les requêtes dans la table AZFWDnsQuery — utile pour détecter des DGA (Domain Generation Algorithms) utilisés par les malwares.

  --resource-group rg-spoke-prod \

  --resource-group rg-spoke-prod \

  --firewall-policy fwpol-spoke-prod

Auto-scaling & Haute Disponibilité pay-per-hour
Autoscale par SKU, Availability Zones, résilience, capacity planning et maintenance
🚨
Scénario : Sans Zone Redundancy, Azure Firewall est deployé dans une seule Availability Zone. Si cette zone subit une panne, votre Firewall tombe — et avec lui, l'intégralité du trafic Internet + inter-Spoke de votre infrastructure. Azure Firewall s'adapte automatiquement à la charge dans les limites documentées du SKU choisi. La borne maximale diffère selon le SKU et doit être validée contre la page de performance / features courante.

Script All-in-One — Azure CLI
Déploiement complet Azure Firewall Standard + Policy + règles de base + logging
💡
Ce script reprend les commandes Quick Win de toutes les sections. Il déploie un Azure Firewall Premium zone-redondant avec Policy, DNS Proxy, Threat Intel Deny, Diagnostic Settings et une UDR Spoke de base. Adapter les variables en haut du script avant exécution.
☁️ Azure CLI — Script All-in-One (Quick Win complet) 
#!/bin/bash
# ================================================================
# AZURE FIREWALL — SCRIPT ALL-IN-ONE (Quick Wins)
# Adapter les variables ci-dessous avant exécution
# ================================================================

# === Variables ===
RG="rg-hub-prod"
LOCATION="francecentral"
VNET_HUB="vnet-hub-prod"
FW_NAME="fw-hub-prod"
POL_NAME="fwpol-hub-prod"
PIP_NAME="pip-fw-hub-prod"
LAW_NAME="law-fw-hub-prod"
RCG_NAME="rcg-baseline"

# === [S1] Déployer Azure Firewall Premium avec Zones ===
az network public-ip create \
  --resource-group $RG --name $PIP_NAME \
  --sku Standard --zone 1 2 3 --location $LOCATION

az network firewall policy create \
  --resource-group $RG --name $POL_NAME \
  --sku Premium --location $LOCATION \
  --threat-intel-mode Deny \
  --enable-dns-proxy true \
  --dns-servers 168.63.129.16

az network firewall create \
  --resource-group $RG --name $FW_NAME \
  --location $LOCATION \
  --sku AZFW_VNet --tier Premium \
  --vnet-name $VNET_HUB \
  --public-ip $PIP_NAME \
  --firewall-policy $POL_NAME \
  --zones 1 2 3

# === [S3] Rule Collection Group + App Rule FQDN (Quick Win) ===
az network firewall policy rule-collection-group create \
  --resource-group $RG --policy-name $POL_NAME \
  --name $RCG_NAME --priority 300

az network firewall policy rule-collection-group collection add-filter-collection \
  --resource-group $RG --policy-name $POL_NAME \
  --rule-collection-group-name $RCG_NAME \
  --name allow-internet-outbound --priority 300 \
  --action Allow --rule-type ApplicationRule \
  --rule-name allow-windows-update \
  --protocols Http=80 Https=443 \
  --source-addresses 10.0.0.0/8 \
  --target-fqdns "*.windowsupdate.microsoft.com" "*.update.microsoft.com"

# === [S6] Activer Diagnostic Settings vers Log Analytics ===
# Exemple de rétention ; adaptez la valeur à votre politique de conservation
az monitor log-analytics workspace create \
  --resource-group $RG --workspace-name $LAW_NAME \
  --location $LOCATION --retention-time 90

FW_ID=$(az network firewall show -g $RG -n $FW_NAME --query id -o tsv)
LAW_ID=$(az monitor log-analytics workspace show -g $RG -n $LAW_NAME --query id -o tsv)

az monitor diagnostic-settings create \
  --resource "$FW_ID" \
  --name "diag-fw-law" \
  --workspace "$LAW_ID" \
  --export-to-resource-specific true \
  --logs '[
    {"category":"AZFWApplicationRule","enabled":true},
    {"category":"AZFWNetworkRule","enabled":true},
    {"category":"AZFWNatRule","enabled":true},
    {"category":"AZFWThreatIntel","enabled":true},
    {"category":"AZFWIdpsSignature","enabled":true},
    {"category":"AZFWDnsQuery","enabled":true}
  ]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'

# === [S9] Vérification finale ===
echo "=== Vérification Zones ==="
az network firewall show -g $RG -n $FW_NAME --query zones -o tsv
echo "=== Vérification Tier ==="
az network firewall show -g $RG -n $FW_NAME --query sku.tier -o tsv
echo "=== Vérification Threat Intel ==="
az network firewall policy show -g $RG -n $POL_NAME --query threatIntelMode -o tsv
echo "=== Vérification DNS Proxy ==="
az network firewall policy show -g $RG -n $POL_NAME --query "dnsSettings.enableProxy" -o tsv
echo "=== Diagnostic Settings actifs ==="
az monitor diagnostic-settings list --resource "$FW_ID" --query "length(value)" -o tsv
