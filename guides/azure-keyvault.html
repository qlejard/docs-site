Analogie fondamentale

Azure Key Vault = coffre-fort numérique. Imaginez un coffre-fort dans une banque : vous y stockez vos objets de valeur (secrets, clés, certificats), seules les personnes autorisées y accèdent (RBAC), il est protégé physiquement (HSM), surveillé par caméras (monitoring), et vous avez un plan en cas de sinistre (backup/DR). Ce guide couvre chaque couche de protection de ce coffre-fort.

1
Vault Creation & Configuration Inclus Azure
Création du Key Vault, soft delete, purge protection, stratégie vault-per-environment — 15 pts
🚨
Scénario : Un administrateur supprime accidentellement un Key Vault contenant les secrets de production (connection strings, clés API, certificats TLS). Sans soft delete et purge protection, la suppression est irréversible : toutes les applications perdent l'accès à leurs secrets, les certificats TLS expirent sans backup, et la restauration nécessite une recréation manuelle de chaque secret. Sans soft delete, la perte est définitive. Avec soft delete, la récupération reste possible pendant la période de rétention ; avec purge protection, la suppression définitive n'est plus possible avant son échéance.

Analogie : le coffre-fort numérique — Azure Key Vault est le coffre-fort de votre infrastructure. Soft Delete = la corbeille de sécurité (suppression réversible). Purge Protection = le cadenas anti-destruction (même un admin ne peut pas vider la corbeille pendant la période de rétention).

💡
Pourquoi chaque paramètre Vault Creation compte :
  • enablePurgeProtection = true : sans purge protection, un principal ayant les droits de suppression et de purge peut supprimer définitivement un vault ou un objet supprimé. La fonctionnalité reste optionnelle, mais Microsoft la recommande pour les usages de chiffrement et plusieurs services intégrés l'exigent pour éviter la perte de données. Attention : l'activation est irréversible. Source : Microsoft Learn — Soft Delete.
  • enableRbacAuthorization = true : Azure RBAC est le modèle recommandé et devient le modèle d'accès par défaut pour les nouveaux vaults créés avec l'API 2026-02-01 ou ultérieure. Les access policies restent supportées, mais doivent être explicitement demandées sur les nouveaux déploiements. Source : Microsoft Learn — RBAC vs access policies, Microsoft Learn — RBAC default access control.
  • softDeleteRetentionInDays = 90 : 90 jours est la valeur par défaut et laisse le plus de temps pour récupérer une suppression accidentelle. La valeur se choisit uniquement à la création, dans la plage 7-90 jours, puis devient immutable.
  • Vault per environment : un vault partagé entre dev et prod signifie qu'un développeur peut accéder à la prod (CIS Azure v6.0.0 §8.3 Key Vault);veloppeur ayant accès aux secrets de dev peut potentiellement accéder à ceux de prod. L'isolation par environnement est un principe de least privilege fondamental. Source : CIS Azure Foundations Benchmark v6.0.0 — 8.3.
  • publicNetworkAccess = Disabled : un Key Vault exposé publiquement est atteignable depuis Internet. Même avec authentification, cela augmente la surface d'attaque (brute force, credential stuffing). Le private endpoint garantit que seul le trafic du VNet accède au vault.
  • Naming convention : le nom du Key Vault est globalement unique dans Azure (DNS : {name}.vault.azure.net). Une convention structurée évite les conflits et facilite l'audit. Max 24 caractères, alphanumérique + tirets.
⚠️
Pièges des défauts :
  • Purge protection irréversible : une fois activée, elle ne peut jamais être désactivée. Planifier avant d'activer sur les vaults de dev (où vous pourriez vouloir purger rapidement).
  • softDeleteRetentionInDays immutable : la valeur est fixée à la création. Si vous créez un vault avec 7 jours, vous ne pourrez pas la passer ensuite à 90. Choisir la valeur cible selon votre politique interne avant la création.
  • Nom DNS global : le nom du vault est un enregistrement DNS public (kv-xxx.vault.azure.net). Un attaquant peut énumérer les noms de vault par DNS. Ne mettez pas d'informations sensibles dans le nom (pas de kv-password-prod).
  • Access Policies → RBAC : migration non réversible simplement : le passage de Access Policies à RBAC supprime toutes les access policies existantes. Préparer les role assignments RBAC avant la migration.
  • Managed HSM vs Key Vault Premium : Managed HSM doit être réservé aux besoins HSM single-tenant avec local RBAC et security domain. Key Vault Premium couvre les clés HSM-protected dans un vault multi-tenant. Vérifier le pricing live au moment du choix de l'architecture.

1.2 — Configuration (Az CLI)

Az CLI — Création d'un Key Vault sécurisé
# Variables à adapter
$rgName="rg-myapp-prod-westeurope"
$kvName="kv-myapp-prod-we"
$location="westeurope"

# Création du Key Vault avec toutes les protections
az keyvault create \
  --name $kvName \
  --resource-group $rgName \
  --location $location \
  --sku "standard" \
  --enable-rbac-authorization true \
  --enable-purge-protection true \
  --retention-days 90 \
  --tags "Environment=Production" "Owner=InfraTeam" "CostCenter=IT-001"

# Activer purge protection sur un vault existant
az keyvault update \
  --name $kvName \
  --resource-group $rgName \
  --enable-purge-protection true

1.4 — État des lieux

2
Secrets Management KV Standard
Création, rotation, expiration, référencement depuis App Service/Functions — 15 pts
🚨
Scénario : Un secret de base de données en production n'a jamais été roté. Il est compromis dans un breach tiers. L'attaquant a accès illimité aux données de production tant que le secret n'est pas changé. Sans automatisation, la détection et la mise à jour du service source reposent sur des opérations manuelles. Avec une rotation pilotée par Event Grid + Function, Key Vault publie l'événement SecretNearExpiry 30 jours avant l'expiration ; l'automate crée une nouvelle version puis met à jour le service source. Prévoir un décalage possible entre ces étapes et les retries Event Grid en cas d'échec.

Analogie : les secrets sont comme des mots de passe WiFi d'hôtel. Un mot de passe qui ne change jamais finit par être connu de tous les anciens clients. La rotation automatique = un nouveau mot de passe distribué automatiquement à chaque nouveau check-in.

Pourquoi chaque paramètre Secrets Management compte :
  • Expiration date : un secret sans expiration est un risque permanent (CIS Azure v6.0.0 §8.3.1-8.3.4 Expiration Date for Keys/Secrets)compromis, il reste valide indéfiniment. L'expiration force la rotation et réduit la fenêtre d'exposition. Azure Policy Secrets should have expiration date set permet d'enforcer ce contrôle. Source : CIS Azure Foundations Benchmark v6.0.0 — 8.3.1.
  • KV References App Service : copier un secret en clair dans une app setting est un anti-pattern majeur. Le secret est visible en clair dans le portail, dans les déploiements ARM, dans les logs de diagnostic. La référence KV (@Microsoft.KeyVault(SecretUri=...)) résout le secret au runtime via Managed Identity — zéro secret en clair. Source : Microsoft Learn — KV References.
  • Rotation automatique : la rotation manuelle est oubliée, reportée, ou ratée. Event Grid déclenche un event SecretNearExpiry qui invoque une Azure Function pour renouveler le secret auprès du service source (SQL, Storage, API tierce). Source : Microsoft Learn — Secret Rotation.
  • Content Type : facilite la découverte et le filtrage. Un secret sans content type est impossible à distinguer dans un audit (est-ce un mot de passe ? une connection string ? un certificat PFX ?).
⚠️
Pièges des défauts :
  • Secret expiré = erreur silencieuse : un secret expiré retourne un 403 Forbidden, pas un message clair « secret expiré ». L'application plante sans message explicite. Solution : configurer une alerte 30 jours avant expiration via Event Grid SecretNearExpiry.
  • KV Reference sans Managed Identity : la syntaxe @Microsoft.KeyVault(...) nécessite une System-assigned ou User-assigned Managed Identity avec le rôle Key Vault Secrets User. Sans cela, l'App Service ne peut pas résoudre la référence et retourne la syntaxe brute comme valeur.
  • Rotation et downtime : si l'application met en cache le secret au démarrage (ex: connection string SQL), la rotation du secret dans le Key Vault ne prend effet qu'au prochain redémarrage. Solution : utiliser SecretClient avec refresh ou App Service slot swap.
  • Taille maximale : un secret Key Vault est limité à 25 Ko. Pour des fichiers plus gros (certificats PFX, JSON complexes), utiliser un Certificate ou un Blob Storage chiffré.
  • Versions non purgées : chaque az keyvault secret set crée une nouvelle version. Après 100 rotations, vous avez 100 versions. Seule la dernière est « latest » mais les anciennes consomment du stockage et sont potentiellement sensibles. Purger les anciennes versions.

$expiry=$(date -u -d "+90 days" +%Y-%m-%dT%H:%M:%SZ)

  --vault-name $kvName \
  --name "sql-connection-string" \

2.6 — État des lieux

3
Keys & Cryptographic Operations KV Standard KV Premium
Types de clés, rotation policy, BYOK, opérations cryptographiques, CMK — 10 pts
🚨
Scénario : Un service utilise une clé RSA-2048 software pour chiffrer ses données (Customer Managed Key). La clé n'a jamais été rotée depuis 3 ans. Une vulnérabilité cryptographique est découverte. Sans rotation policy, il faut manuellement identifier tous les services utilisant cette clé, générer une nouvelle version, et redéployer — risque d'interruption de service si la clé est invalide. Avec une rotation policy automatique, la nouvelle version est générée et les services utilisant « latest » migrent automatiquement.

Analogie : la serrure de coffre-fort. Une clé cryptographique est comme la clé physique du coffre. Le HSM est l'armoire blindée où est stockée la clé maîtresse — la clé ne sort jamais, les opérations se font à l'intérieur. La rotation = changer la serrure régulièrement sans perdre accès au contenu.

Pourquoi chaque paramètre Keys compte :
  • Rotation Policy native : depuis 2023, Key Vault supporte la rotation automatique des clés nativement (sans Event Grid). Configurer rotationPolicy avec autoRotate = la clé est renouvelée automatiquement et les services CMK (Storage, SQL, Disk Encryption) migrent vers la nouvelle version. Source : Microsoft Learn — Key Rotation.
  • HSM vs Software : les clés software sont protégées par les mécanismes du service, tandis que les clés HSM restent confinées dans des HSM validés FIPS 140-3 Level 3 pour Key Vault Premium et Managed HSM. Si une exigence impose des clés HSM non extractibles, utiliser Premium ou Managed HSM selon le besoin multi-tenant ou single-tenant.
  • Key Operations restriction : une clé autorisant toutes les opérations (encrypt + sign + wrap) viole le principe de separation of duties. Une clé de chiffrement ne devrait pas pouvoir signer, et vice-versa. Source : NIST SP 800-57.
  • RSA-2048 vs RSA-3072+ : RSA-2048 est acceptable jusqu'en 2030 (NIST). Pour une durée de protection > 2030, privilégier RSA-3072 ou EC P-256 (même sécurité, meilleures performances).

Purge protection est fortement recommandée, et souvent exigée par les services Azure qui utilisent ces clés pour le chiffrement, afin d'éviter la perte de données.
  • Latence HSM : les opérations HSM sont légèrement plus lentes que les opérations software. Pour du TLS offloading à haut débit, tester les performances avant de migrer vers HSM.
  • EC vs RSA pour CMK : de nombreux services Azure CMK ne supportent que RSA, pas EC (Elliptic Curve). Vérifier la compatibilité du service cible avant de choisir EC.

🚨
Scénario : Un développeur reçoit une Access Policy « all permissions » sur le Key Vault de production pour débugger un problème. L'Access Policy n'est jamais révoquée.   • Propagation RBAC ~5-10 min : les role assignments RBAC peuvent prendre jusqu'à 10 minutes pour se propager. Ne pas conclure à une erreur de configuration trop vite.
  • Owner ≠ accès aux secrets : le rôle Owner sur un Key Vault donne le droit de gérer le vault (supprimer, modifier RBAC) mais pas de lire les secrets. C'est un piège classique : un admin Owner ne peut pas lire un secret sans un rôle data plane (Key Vault Secrets User).
  • Workload Identity Federation : pour les pipelines GitHub Actions / Azure DevOps, utiliser Workload Identity Federation au lieu d'un client_secret. Plus sécurisé (pas de secret à stocker) et recommandé par Microsoft.

Disaster Recovery & Backup Inclus Azure
Backup/restore, limites de résilience, réplication Microsoft-managed et restauration — 5 pts
🚨
Scénario : Un administrateur supprime accidentellement un secret critique puis purge le vault. La région Azure subit une panne majeure la semaine suivante. Sans backup et sans réplication cross-région, les secrets sont définitivement perdus. Avec les fonctions natives de résilience Key Vault, une partie de la disponibilité est assurée par la réplication Microsoft-managed, mais le failover régional reste un processus best effort déclenché par Microsoft, pouvant prendre plusieurs heures et laisser le vault en lecture seule. Pour un RTO/RPO strict, il faut prévoir une stratégie multi-région sur mesure et des backups exportés.

Analogie : le backup Key Vault est comme la copie du plan du coffre-fort stockée dans un autre bâtiment. Si l'immeuble brûle, vous pouvez reconstruire le coffre avec la copie. La réplication cross-région = un coffre miroir dans un autre pays, synchronisé automatiquement.

  • Backup individuel : la commande az keyvault secret backup crée un blob chiffré restaurable uniquement dans la même géographie Azure (même souveraineté). Un backup France ne peut pas être restauré aux USA.

⚠️
Pièges :
  • Backup = blob chiffré non lisible : le backup Key Vault est un blob chiffré par Microsoft. Vous ne pouvez pas lire le contenu du backup, uniquement le restaurer dans un vault de la même géographie.
  • Restore dans le même vault uniquement ? : le restore doit se faire dans un vault de la même souscription Azure et de la même géographie. Vous ne pouvez pas restaurer un backup d'un vault France dans un vault US.
  • Failover = lecture seule : pendant un failover régional, le vault est en lecture seule. Les écritures (création/modification de secrets) échouent. Les applications doivent tolérer ce mode.
  • Backup ne sauvegarde pas les policies/RBAC : la commande backup sauvegarde le secret/clé/certificat uniquement. Les Access Policies, role assignments RBAC, network rules, et diagnostic settings ne sont pas sauvegardés. Utiliser Bicep/Terraform pour l'infrastructure.

Private Endpoint Tarification réseau séparée Exposition Internet, interception À prioriser pour les vaults critiques ou internal-only
Diagnostic Settings Ingestion et rétention variables Pas d'audit, pas de forensique Dimensionner le workspace selon les besoins d'audit et de conformité
Defender for Key Vault Tarification Defender for Cloud Menaces comportementales non détectées À chiffrer selon le volume protégé et le niveau de surveillance attendu
Secret Rotation Transactions + Event Grid / Function selon l'implémentation Secret compromis actif indéfiniment À réserver en priorité aux identifiants critiques
Azure Policy (Deny) Inclus dans le service Vaults non conformes déployés Appliquer progressivement après phase Audit et exemptions planifiées
Backup automatisé Stockage + opérations variables Perte définitive après purge protection expirée Conserver la contrainte de même souscription et même géographie

💰 Utiliser les pages pricing Azure et le calculateur pour convertir ces modèles en montants réels selon la topologie, la région, la devise et le volume.

💡
Conclusion : Les mesures de base (RBAC, soft delete, purge protection, Azure Policy) sont majoritairement incluses. Les coûts variables viennent surtout du réseau, des logs, de Defender, des rotations planifiées et du volume d'opérations. Le bon niveau d'investissement doit se calculer à partir du trafic, de la topologie et des exigences d'audit, pas à partir d'un montant figé ou d'un ROI générique.

Script All-in-One — Hardening Key Vault
Script exécutable en une seule fois pour appliquer toutes les best practices
⚠️
Avertissement : Ce script applique les best practices en production. Tester d'abord sur un vault de développement. Certaines actions sont irréversibles (purge protection, RBAC activation). Lire chaque section du guide avant exécution.
Bash — Script Hardening Key Vault All-in-One
#!/bin/bash
# ═══════════════════════════════════════════════════════════
# Key Vault Hardening — Script All-in-One
# Prérequis : az login, subscription sélectionnée
# ═══════════════════════════════════════════════════════════

set -euo pipefail

# ——— VARIABLES (À ADAPTER) ———
RG="rg-keyvault-prod"
KV_NAME="kv-prod-001"
LOCATION="francecentral"
LAW_NAME="law-security-prod"
VNET_NAME="vnet-prod"
SUBNET_PE="snet-privateendpoints"
ACTION_GROUP_ID=""  # ID du Action Group pour les alertes

echo "🔒 [1/8] Création du Key Vault avec hardening..."
az keyvault create \
  --name $KV_NAME \
  --resource-group $RG \
  --location $LOCATION \
  --sku "standard" \
  --enable-rbac-authorization true \
  --enable-soft-delete true \
  --retention-days 90 \
  --enable-purge-protection true \
  --public-network-access "Disabled"

KV_ID=$(az keyvault show --name $KV_NAME --query "id" -o tsv)

echo "🌐 [2/8] Configuration du Private Endpoint..."
az network private-endpoint create \
  --name "pe-${KV_NAME}" \
  --resource-group $RG \
  --vnet-name $VNET_NAME \
  --subnet $SUBNET_PE \
  --private-connection-resource-id $KV_ID \
  --group-ids "vault" \
  --connection-name "plsc-${KV_NAME}"

echo "📊 [3/8] Configuration Diagnostic Settings..."
LAW_ID=$(az monitor log-analytics workspace show \
  --resource-group $RG --workspace-name $LAW_NAME --query "id" -o tsv)

az monitor diagnostic-settings create \
  --name "diag-${KV_NAME}" \
  --resource $KV_ID \
  --workspace $LAW_ID \
  --logs '[{"categoryGroup":"allLogs","enabled":true,"retentionPolicy":{"enabled":true,"days":365}}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'

echo "🛡️ [4/8] Activation Defender for Key Vault..."
az security pricing create --name "KeyVaults" --tier "Standard"

echo "👥 [5/8] Configuration RBAC — Managed Identity..."
# Exemple : assigner Key Vault Secrets User à une Managed Identity
# MI_PRINCIPAL_ID=$(az webapp identity show --name myapp --resource-group $RG --query principalId -o tsv)
# az role assignment create --role "Key Vault Secrets User" --assignee $MI_PRINCIPAL_ID --scope $KV_ID

echo "📜 [6/8] Assignation Azure Policies (Deny)..."
SUB_ID=$(az account show --query "id" -o tsv)

az policy assignment create \
  --name "enforce-kv-rbac" \
  --policy "12d4fa5e-1f9f-4c21-97a9-b99b3c6611b5" \
  --scope "/subscriptions/$SUB_ID" \
  --params '{"effect":{"value":"Deny"}}'

az policy assignment create \
  --name "enforce-kv-purge" \
  --policy "0b60c0b2-2dc2-4e1c-b5c9-abbed971de53" \
  --scope "/subscriptions/$SUB_ID" \
  --params '{"effect":{"value":"Deny"}}'

echo "💾 [7/8] Backup initial des secrets et clés..."
mkdir -p "./backups-${KV_NAME}"
for secret in $(az keyvault secret list --vault-name $KV_NAME --query "[].name" -o tsv 2>/dev/null); do
  az keyvault secret backup --vault-name $KV_NAME --name "$secret" --file "./backups-${KV_NAME}/${secret}.bak"
done

echo "✅ [8/8] Vérification finale..."
az keyvault show --name $KV_NAME \
  --query "{Name:name, RBAC:properties.enableRbacAuthorization, SoftDelete:properties.enableSoftDelete, PurgeProtection:properties.enablePurgeProtection, PublicAccess:properties.publicNetworkAccess}" \
  -o table

echo ""
echo "🎉 Hardening terminé pour $KV_NAME"
echo "📋 Prochaines étapes :"
echo "  - Configurer les DNS privés (Private DNS Zone)"
echo "  - Assigner les rôles RBAC aux Managed Identities"
echo "  - Configurer les alertes dans Action Groups"
echo "  - Créer les secrets avec expiration + rotation"
✅
