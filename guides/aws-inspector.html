Mode d'emploi de ce guide
Comment utiliser ce référentiel Amazon Inspector pour configurer et auditer la gestion des vulnérabilités
📋
Ce document est un référentiel complet couvrant toutes les configurations Amazon Inspector v2 avec les best practices AWS. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un compte AWS.
Analogie : Amazon Inspector est le médecin légiste de vos workloads — il examine chaque instance EC2, chaque image de container, chaque fonction Lambda pour détecter les faiblesses (CVE). Sans Inspector, c'est comme exploiter une salle de réanimation sans infirmière : les pathologies se développent silencieusement jusqu'à l'urgence vitale.

Pourquoi chaque paramètre :
• EC2 scanning : couvre ~80% de la surface CVE (OS packages, kernel). Principal vecteur d'attaque ransomware (Eternal Blue, Log4Shell sur hosts).
• ECR scanning : les containers embarquent souvent des bases images vulnérables (Ubuntu 20.04). Détecté au push, avant déploiement Kubernetes ou ECS.
• Lambda standard : packages Python/Node.js dans les zips ont des CVE (ex: requests < 2.31.0). Lambda n'a pas d'OS vu par l'admin — sans scan = angle mort total.
• Lambda code (SAST) : détecte secrets hardcodés, command injection, path traversal. Requis si code custom critique.
• Auto-enable Organizations : garantit que chaque nouveau compte AWS créé soit couvert automatiquement sans action manuelle.
⚠️
Piège Organizations : Activer Inspector v2 sur le Management Account sans configurer auto-enable laisse les comptes membres non couverts. Vous avez l'illusion d'une couverture globale alors que 80% de vos comptes AWS sont aveugles. Configurer le delegated administrator (section S7) en priorité dans un contexte multi-comptes.

S2
EC2 Vulnerability Scanning
Scan des instances Amazon EC2 — packages OS, network reachability, EPSS scoring — 20 pts
🚨
Contexte : Une instance EC2 accumule en moyenne 150+ CVE connues dans les 6 mois suivant sa création sans patch. Sans Inspector, la détection n'est possible que lors d'un incident. Analogie : Chaque CVE non corrigée est une maladie connue avec un traitement documentd — le rapport médical (finding Inspector) indique le diagnostic (CVE ID), la gravité (Inspector Score), et l'ordonnance (SSM Patch Manager, kernel update).

Comparatif modes de scan EC2

ParamètreDéfaut AWS⚡ Quick Win🔒 Avancé🎯 Optimal
Mode de scan EC2DésactivéAgent SSM (EC2_SSM_AGENT_BASED)Agent + agentless previewAgent + agentless + scheduled
Packages OS scannés—RPM (RHEL/Amazon Linux), DEB (Ubuntu/Debian)+ packages overlay containers+ scan de packages custom
Network reachability—✅ Activé automatiquement✅ + contexte port/sg✅ + reporting par VPC
EPSS scoring—✅ Inclus dans Inspector Score✅ Filtre EPSS > 0.1✅ Alerte EPSS > 0.5 auto
Fréquence de scan— Continu (event-based: au démarrage + nouvelle CVE publiée NVD) + scan forcé mensuel+ scan post-patch SSM
Score S2 (pts)0/2010/2016/2020/20
💡
Pourquoi chaque paramètre :
• SSM Agent : requis pour le scan agent-based. Sans SSM, l'instance est en statut NO_INVENTORY. Installation via User Data ou Systems Manager Run Command.
• Network reachability : type de finding dédié qui détecte les chemins réseau ouverts depuis Internet (0.0.0.0/0 dans Security Group, NACL, route IGW). Combiné aux findings de vulnérabilité packages, il enrichit le contexte d'exposition du finding et pèse dans l'Inspector Score : un port exposé sur une CVE CVSSv3 7.0 sera priorité supérieure à la même CVE sur un port interne.
• EPSS (Exploit Prediction Scoring System) : probabilité qu'une CVE soit exploitée dans les 30 prochains jours. CVE avec CVSS 9.8 mais EPSS 0.001 = théorique. CVE CVSS 7.0 EPSS 0.95 = urgente.
• Scan continu event-based : Inspector re-scanne automatiquement quand le NVD publie une nouvelle CVE matchant les packages inventoriés. Pas de scan périodique à planifier.
⚠️
Piège SSM : Inspector ne peut pas scanner une EC2 sans SSM Agent actif ET avec le rôle IAM AmazonSSMManagedInstanceCore. Les instances lancées depuis des AMI communautaires ou anciennes n'ont pas SSM préinstallé. Vérifier le coverage avec la liste des instances en statut NO_INVENTORY avant de considérer la couverture EC2 comme complète.

Pourquoi chaque paramètre :
• Enhanced vs Basic : Basic scanning utilise Clair (moteur open source limité). Enhanced utilise Amazon Inspector v2 avec base NVD complète, EPSS, contexte réseau. Passage à Enhanced = détection 3x plus de CVE.
• Continuous scan : une image pushée propre en janvier peut devenir vulnérable en mars si Log4Shell est publié. Continuous re-scan garantit détection rétroactive sans nouveau push.
• Pipeline gate : bloque le déploiement si une CVE Critical est détectée. Le développeur est notifié immédiatement après le docker push, avant que l'image atteigne la production.
• Filtres de repo : exclure les registres d'images archivées ou les environnements dev pour réduire le coût de scan.
⚠️
Piège continuous scan + registre énorme : Si votre registre ECR contient 5 000 images actives et qu'un lot de 20 CVE est publié le vendredi soir, Inspector va re-scanner toutes les images concernées. Cela peut générer 100 000 scans en quelques heures. Configurer des règles d'inclusion (continuous scan uniquement sur repos prod, pas sur repos dev/test) pour maîtriser le coût.

Pipeline CI/CD Gate (GitHub Actions)

# === GITHUB ACTIONS - Pipeline gate ECR Inspector ===
# .github/workflows/deploy.yml

name: Build, Push, and Inspector Gate
on:
  push:
    branches: [main]

jobs:
  build-and-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_DEPLOY_ROLE_ARN }}
          aws-region: eu-west-1

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push image
        id: build
        run: |
          IMAGE_URI="${{ steps.login-ecr.outputs.registry }}/my-app:${{ github.sha }}"
          docker build -t $IMAGE_URI .
          docker push $IMAGE_URI
          echo "image_uri=$IMAGE_URI" >> $GITHUB_OUTPUT

      - name: Wait for Inspector scan completion
        run: |
          echo "Waiting 60s for Inspector to scan the pushed image..."
          sleep 60

      - name: Check Inspector findings - Block on Critical
        run: |
          IMAGE_DIGEST=$(aws ecr describe-images \
            --repository-name my-app \
            --image-ids imageTag=${{ github.sha }} \
            --query 'imageDetails[0].imageDigest' --output text)

          CRITICAL_COUNT=$(aws inspector2 list-findings \
            --filter-criteria "{
              \"resourceType\":[{\"comparison\":\"EQUALS\",\"value\":\"AWS_ECR_CONTAINER_IMAGE\"}],
              \"ecrImageHash\":[{\"comparison\":\"EQUALS\",\"value\":\"$IMAGE_DIGEST\"}],
              \"severity\":[{\"comparison\":\"EQUALS\",\"value\":\"CRITICAL\"}]
            }" \
            --query 'length(findings)' --output text)

          echo "Critical CVE count: $CRITICAL_COUNT"
          if [ "$CRITICAL_COUNT" -gt "0" ]; then
            echo "::error::DEPLOYMENT BLOCKED - $CRITICAL_COUNT Critical CVE(s) found by Amazon Inspector"
            exit 1
          fi
          echo "Inspector gate PASSED - no Critical CVE found"

      - name: Deploy to ECS (only if gate passed)
        run: |
          echo "Deploying ${{ steps.build.outputs.image_uri }} to ECS..."
          # aws ecs update-service --cluster my-cluster --service my-service ...

S5
Findings & Prioritization
Inspector Score, EPSS, filtres et suppression des faux positifs — 15 pts
🚨
Contexte : Un environnement AWS typique génère des milliers de findings Inspector. Sans priorisation intelligente, les équipes SecOps sont submergées et traitent les CVE dans le mauvais ordre. Analogie : L'Inspector Score est un indice de gravité médicale (comme l'ISS en traumatologie) : il combine la pathologie connue (CVSS), la probabilité de contamination (EPSS), et la proximité du patient avec la zone d'infection (network reachability). Un score 9.5 = soins intensifs immédiats. Un score 2.0 = consultation planifiée.

Formule Inspector Score

📊
Inspector Score = f(CVSS v3.1 base + EPSS weight + network reachability weight + exploit-in-wild weight)

• CVSS v3.1 base : score NVD officiel (0-10). Composante principale.
• EPSS (Exploit Prediction Scoring System) : probabilité d'exploitation dans les 30 jours (0-1). EPSS 0.97 = 97% de probabilité. Augmente le score Inspector.
• Network reachability : si le port vulnérable est exposé sur Internet (0.0.0.0/0 dans Security Group), le score est majoré.
• Exploit in the wild : si la CVE a un exploit connu et actif (CISA KEV, Metasploit), le score est majoré.

Exemple Log4Shell : CVE-2021-44228 — CVSS 10.0 + EPSS 0.97 + port 8080 ouvert + exploit in wild → Inspector Score 9.5 Critical.
Contre-exemple : CVE-2023-XXXXX — CVSS 9.8 + EPSS 0.001 + port non exposé + no exploit in wild → Inspector Score 5.0 Medium (théorique, non prioritaire).

🔧 Script de diagnostic global

💡
Utilisation : Ce script vérifie l'état complet d'Amazon Inspector v2 dans la région courante. Sortie structurée pour audit ou intégration dans un pipeline de conformité. Pré-requis : AWS CLI v2, droits inspector2:* en lecture. 
#!/usr/bin/env bash
# ============================================================
# inspector_v2_diagnostic.sh
# Diagnostic complet Amazon Inspector v2 - région courante
# Usage : ./inspector_v2_diagnostic.sh [--region eu-west-1]
# ============================================================
set -euo pipefail

REGION="${AWS_DEFAULT_REGION:-$(aws configure get region)}"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
SEPARATOR="════════════════════════════════════════════════════════"

echo ""
echo "AMAZON INSPECTOR v2 — DIAGNOSTIC"
echo "$SEPARATOR"
echo "Compte  : $ACCOUNT_ID"
echo "Région  : $REGION"
echo "Date    : $(date -u '+%Y-%m-%d %H:%M:%S UTC')"
echo "$SEPARATOR"

# ─── 1. STATUT D'ACTIVATION ───────────────────────────────
echo ""
echo "▶ [1/6] Statut d'activation des scan types"
aws inspector2 batch-get-account-status \
  --account-ids "$ACCOUNT_ID" \
  --region "$REGION" \
  --query 'accounts[0].{EC2:resourceState.ec2.status,ECR:resourceState.ecr.status,Lambda:resourceState.lambda.status,LambdaCode:resourceState.lambdaCode.status}' \
  --output table

# ─── 2. COVERAGE ──────────────────────────────────────────
echo ""
echo "▶ [2/6] Coverage (ressources scannées / non scannées)"
COVERED=$(aws inspector2 get-coverage-statistics \
  --region "$REGION" \
  --query 'countsByResourceType' \
  --output json)
echo "$COVERED"

# ─── 3. FINDINGS CRITIQUES ACTIFS ─────────────────────────
echo ""
echo "▶ [3/6] Findings CRITICAL ACTIVE (top 10)"
aws inspector2 list-findings \
  --region "$REGION" \
  --filter-criteria '{
    "severity":[{"comparison":"EQUALS","value":"CRITICAL"}],
    "findingStatus":[{"comparison":"EQUALS","value":"ACTIVE"}]
  }' \
  --sort-criteria '{STR0:STR1,STR2:STR3}' \
  --max-results 10 \
  --query 'findings[*].{CVE:packageVulnerabilityDetails.vulnerabilityId,Score:inspectorScore,Resource:resources[0].id,EPSS:packageVulnerabilityDetails.cvss[0].baseScore}' \
  --output table 2>/dev/null || echo "  Aucun finding CRITICAL ACTIVE"

# ─── 4. COUNT PAR SÉVÉRITÉ ────────────────────────────────
echo ""
echo "▶ [4/6] Nombre de findings ACTIVE par sévérité"
for SEVERITY in CRITICAL HIGH MEDIUM LOW INFORMATIONAL; do
  COUNT=$(aws inspector2 list-findings \
    --region "$REGION" \
    --filter-criteria "{\"severity\":[{\"comparison\":\"EQUALS\",\"value\":\"$SEVERITY\"}],\"findingStatus\":[{\"comparison\":\"EQUALS\",\"value\":\"ACTIVE\"}]}" \
    --query 'length(findings)' \
    --output text 2>/dev/null || echo "0")
  printf "  %-20s : %s\n" "$SEVERITY" "$COUNT"
done

# ─── 5. IMAGES ECR AVEC CRITICAL ──────────────────────────
echo ""
echo "▶ [5/6] Images ECR avec findings CRITICAL ACTIVE"
aws inspector2 list-findings \
  --region "$REGION" \
  --filter-criteria '{
    "severity":[{"comparison":"EQUALS","value":"CRITICAL"}],
    "findingStatus":[{"comparison":"EQUALS","value":"ACTIVE"}],
    "resourceType":[{"comparison":"EQUALS","value":"AWS_ECR_CONTAINER_IMAGE"}]
  }' \
  --query 'findings[*].resources[0].id' \
  --output text 2>/dev/null | sort | uniq | head -20 || echo "  Aucune image ECR avec Critical"

# ─── 6. VÉRIFICATION DELEGATED ADMIN ──────────────────────
echo ""
echo "▶ [6/6] Delegated Administrator Inspector"
aws inspector2 list-delegated-admin-accounts \
  --region "$REGION" \
  --query 'delegatedAdminAccounts[*].{Account:accountId,Status:status}' \
  --output table 2>/dev/null || echo "  Non configuré (compte standalone ou pas d'org)"

echo ""
echo "$SEPARATOR"
echo "Diagnostic terminé."
echo ""

📖 Glossaire

TermeDéfinition
Amazon Inspector v2Service AWS de gestion des vulnérabilités (EC2, ECR, Lambda). Successeur d'Inspector Classic (déprécié novembre 2023). Préfixe CLI : aws inspector2.
CVECommon Vulnerabilities and Exposures : identifiant unique d'une vulnérabilité connue (ex: CVE-2021-44228 = Log4Shell). La "maladie connue" du logiciel.
CVSSCommon Vulnerability Scoring System : score de sévérité intrinsèque d'une CVE (0-10). CVSS v3.1 = standard actuel. Calcule : vecteur d'attaque + complexité + privilèges + interaction utilisateur + impact.
Inspector ScoreScore contextuel Amazon Inspector (0-10). Formule : CVSS v3.1 base + poids EPSS + poids Network Reachability + poids Exploit-in-Wild. Plus pertinent que le CVSS seul car intègre le contexte d'exposition.
EPSSExploit Prediction Scoring System : probabilité qu'une CVE soit exploitée dans les 30 jours (0-1). Produit par FIRST.org. EPSS > 0.1 (10%) = risque élevé d'exploitation imminente.
KEVCISA Known Exploited Vulnerabilities : catalogue des CVE activement exploitées dans la nature, maintenu par la CISA américaine. Une CVE dans KEV = exploitation confirmée.
Network ReachabilityContexte d'exposition réseau d'une instance EC2 : port ouvert vers Internet, vers le VPC, ou isolé. Une CVE sur une instance Internet-facing a un Inspector Score plus élevé que sur une instance privée.
SBOMSoftware Bill of Materials : liste exhaustive des composants logiciels (librairies, packages) d'une application. Analogue à la liste des ingrédients d'un produit alimentaire. Formats : CycloneDX 1.4, SPDX 2.3.
CycloneDXFormat SBOM open-source maintenu par OWASP. Version 1.4 = standard industrie le plus adopté pour l'analyse de sécurité supply chain.
SPDXSoftware Package Data Exchange : format SBOM Linux Foundation. Version 2.3. Requis dans certains contextes réglementaires (US Executive Order 14028, NIS2).
Enhanced Scanning (ECR)Scan ECR via Amazon Inspector v2 (vs Basic Scanning via Clair/OS uniquement). Enhanced Scanning détecte les vulnérabilités OS ET les packages applicatifs (npm, pip, Maven, etc.).
SSM AgentAWS Systems Manager Agent : agent installé sur les instances EC2, requis par Inspector v2 pour le scan agentless EC2. Déjà inclus dans les AMI Amazon Linux 2/2023, Windows Server 2016+.
Suppression RuleRègle Inspector qui marque automatiquement certains findings comme SUPPRESSED (ex: faux positif confirmé, vulnérabilité non applicable). Nécessite une justification. Les findings supprimés ne comptent pas dans les métriques par défaut.
Delegated AdministratorCompte AWS désigné pour gérer Inspector v2 au niveau d'une organisation AWS Organizations. Généralement le Security/Audit Account. Concentre tous les findings des comptes membres.
ASFFAmazon Security Finding Format : format JSON standardisé pour les findings de sécurité AWS. Utilisé par Inspector, GuardDuty, Macie, Config pour l'intégration avec Security Hub.
NVDNational Vulnerability Database : base de données américaine (NIST) des CVE avec scores CVSS. Source principale des données de vulnérabilité utilisées par Inspector.
