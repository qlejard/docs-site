Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de sécurisation AWS Lambda
📋
Ce document est un référentiel technique complet couvrant toutes les configurations de sécurité AWS Lambda avec les best practices AWS Well-Architected Framework, CIS AWS Benchmark v7.0.0 et AWS Foundational Security Best Practices. Triple chemin : AWS CLI + Terraform + AWS SAM.

Analogies clés — Lambda comme employé intérimaire

👥
Lambda = employé intérimaire. Il fait sa mission puis disparaît — il ne doit avoir que les droits pour cette mission.
Execution Role = badge d'accès limité. L'intérimaire a accès qu'aux pièces dont il a besoin, pas à tout l'immeuble.
AWS Secrets Manager = coffre-fort. Les secrets ne voyagent pas dans les poches de l'intérimaire (variables d'env).
Lambda dans VPC = bureau sécurisé. La fonction travaille dans un espace isolé du reste du réseau.
Function URL sans auth = interphone sans code. N'importe qui peut déclencher la fonction depuis Internet.
Code Signing = signature du contrat. Seul le code approuvé et signé peut être exécuté.

1
IAM Execution Role & Least Privilege Gratuit
L'identité de la fonction Lambda — principe du moindre privilège, confused deputy, permission boundaries
🚨
Scénario d'attaque : Une fonction Lambda avec un execution role ayant "Action": "*", "Resource": "*" est compromise via une injection de code dans son event source (SQS, S3). L'attaquant utilise les credentials STS de la fonction pour pivoter sur l'ensemble du compte AWS : lire tous les secrets, exfiltrer des données S3, créer des utilisateurs IAM backdoor. C'est l'attaque de l'employé intérimaire qui copie les clés de tout l'immeuble.
Statistique : 73% des fonctions Lambda en production ont un execution role trop permissif (source : rapport Datadog State of Serverless 2024).

💡
Pourquoi chaque paramètre compte :
Role dédié : Un role partagé entre plusieurs fonctions signifie qu'une compromission d'une fonction donne accès aux permissions de toutes. Principe d'isolation.
Resource ARN spécifique : "Resource": "*" sur S3 = accès à tous les buckets du compte. Un ARN spécifique limite l'impact d'une compromission au seul bucket nécessaire.
Confused Deputy : Sans aws:SourceArn, n'importe quel service AWS peut assumer votre role en se faisant passer pour Lambda. Avec la condition, seule votre fonction spécifique peut l'assumer (CWE-441).
Permission Boundary : Empêche qu'un développeur puisse créer un role Lambda avec plus de permissions que ce que le boundary autorise. Contre l'escalade de privilèges interne.
IAM Access Analyzer : Détecte automatiquement les policies qui permettent un accès externe non intentionnel (cross-account, public). Source : AWS IAM Access Analyzer docs.
⚠️
Piège fréquent : La policy gérée AWSLambdaBasicExecutionRole est souvent considérée comme "minimale" mais elle autorise logs:CreateLogGroup, logs:CreateLogStream et logs:PutLogEvents sur "Resource": "*". En pratique, cela signifie que la fonction peut écrire dans n'importe quel log group AWS de votre compte — y compris les logs CloudTrail ou de sécurité. Toujours restreindre à arn:aws:logs:<region>:<account>:log-group:/aws/lambda/<function-name>:*.

1.2 — Configuration — Policy Least Privilege

Exemple 1 : Lecture S3 spécifique

AWS CLI — Policy S3 ReadOnly sur un bucket spécifique
# 1. Créer le rôle d'exécution dédié à la fonction
aws iam create-role \
  --role-name "lambda-my-function-role" \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"lambda.amazonaws.com"},"Action":"sts:AssumeRole","Condition":{"StringEquals":{"aws:SourceAccount":"<ACCOUNT_ID>"},"ArnLike":{"aws:SourceArn":"arn:aws:lambda:eu-west-1:<ACCOUNT_ID>:function:my-function"}}}]}' \
  --profile audit --region eu-west-1

# 2. Attacher une policy CloudWatch Logs minimale (log group dédié)
aws iam put-role-policy \
  --role-name "lambda-my-function-role" \
  --policy-name "CloudWatchLogsMinimal" \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":["logs:CreateLogStream","logs:PutLogEvents"],"Resource":"arn:aws:logs:eu-west-1:<ACCOUNT_ID>:log-group:/aws/lambda/my-function:*"}]}' \
  --profile audit --region eu-west-1

# 3. Attacher la policy S3 ReadOnly spécifique
aws iam put-role-policy \
  --role-name "lambda-my-function-role" \
  --policy-name "S3ReadSpecificBucket" \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":["s3:GetObject","s3:ListBucket"],"Resource":["arn:aws:s3:::my-bucket","arn:aws:s3:::my-bucket/*"]}]}' \
  --profile audit --region eu-west-1

Terraform — Execution Role avec least privilege

Terraform — aws_iam_role + aws_iam_role_policy
# terraform/lambda_iam.tf
resource "aws_iam_role" "lambda_exec" {
  name = "lambda-my-function-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "lambda.amazonaws.com" }
      Action    = "sts:AssumeRole"
      Condition = {
        StringEquals = { "aws:SourceAccount" = var.account_id }
        ArnLike = { "aws:SourceArn" = "arn:aws:lambda:${var.region}:${var.account_id}:function:my-function" }
      }
    }]
  })
}

1.5 — Checklist état des lieux

Chaque fonction Lambda a son propre execution role dédié (aucun role partagé)
Aucune policy ne contient "Resource": "*" (sauf exception justifiée et documentée)
La trust policy inclut aws:SourceAccount (protection confused deputy)
IAM Access Analyzer activé dans toutes les régions utilisées
Permission boundaries appliquées aux roles Lambda (niveaux Avancé/Optimal)
Aucune policy AWSLambdaFullAccess ni AdministratorAccess attachée à un execution role

Secrets & Variables d'Environnement Payant
Gestion sécurisée des credentials, secrets et configuration — Secrets Manager, KMS CMK, rotation
🚨
Scénario d'attaque : Un développeur injecte une clé API de base de données dans une variable d'environnement Lambda (pratique "rapide" très courante). Un attaquant qui obtient un accès en lecture aux configurations Lambda (via lambda:GetFunctionConfiguration) récupère instantanément tous les secrets en clair. Les variables d'environnement Lambda sont chiffrées at rest par AWS par défaut, mais visibles en clair dans la console et via l'API — comme des notes sous le clavier du bureau.
Statistique : Les credentials exposés représentent le vecteur #1 des incidents cloud (Verizon DBIR 2024, 80% des breaches cloud).

💡
Pourquoi chaque paramètre compte :
Secrets Manager vs env vars : Une variable d'environnement est visible par quiconque peut appeler GetFunctionConfiguration ou voir la console Lambda. Secrets Manager impose une policy IAM dédiée, auditée, et supporte la rotation sans redéploiement.
KMS CMK : La clé AWS managed (alias aws/lambda) est partagée entre tous les clients AWS de la région. Une CMK dédiée permet un contrôle total : audit, révocation, rotation. Source : AWS KMS + Lambda docs.
Rotation : Un secret statique qui ne tourne jamais devient une bombe à retardement. Si un ancien employé ou une dépendance compromise l'a vu, le secret est compromis pour toujours sans rotation.
Caching (Powertools) : Sans cache, 1M d'invocations Lambda = 1M d'appels à Secrets Manager (0,05 $/10 000 API calls = 5 $ supplémentaires/M invocations). Avec cache 300s, le coût devient négligeable.
⚠️
Piège : AWS Parameter Store avec SecureString n'est pas équivalent à Secrets Manager. SecureString chiffre la valeur mais ne supporte pas la rotation native automatique ni les versions (AWSCURRENT / AWSPREVIOUS). Pour des credentials de base de données ou des API keys, toujours utiliser AWS Secrets Manager. Pour la configuration applicative non-sensible (URLs, feature flags), SSM Parameter Store Standard est suffisant et gratuit.

2.2 — Configuration — Secrets Manager + KMS CMK

AWS CLI — Créer un secret + activer la rotation
# 1. Créer une clé KMS CMK dédiée à la fonction
KMS_KEY_ID=$(aws kms create-key \
  --description "CMK Lambda my-function secrets" \
  --key-usage ENCRYPT_DECRYPT \
  --profile audit --region eu-west-1 \
  --query 'KeyMetadata.KeyId' --output text)

# 2. Créer le secret dans Secrets Manager
aws secretsmanager create-secret \
  --name "lambda/my-function/db-credentials" \
  --description "DB credentials for my-function" \
  --secret-string '{"username":"app_user","password":"CHANGE_ME"}' \
  --kms-key-id $KMS_KEY_ID \
  --profile audit --region eu-west-1

# 3. Activer la rotation automatique (rotation tous les 30 jours)
aws secretsmanager rotate-secret \
  --secret-id "lambda/my-function/db-credentials" \
  --rotation-rules 'AutomaticallyAfterDays=30' \
  --profile audit --region eu-west-1

# 4. Chiffrer les env vars Lambda avec la CMK (pour la config non-sensible)
aws lambda update-function-configuration \
  --function-name "my-function" \
  --kms-key-arn "arn:aws:kms:eu-west-1:<ACCOUNT>:key/$KMS_KEY_ID" \
  --profile audit --region eu-west-1
Python — Fetch runtime avec Lambda Powertools (caching)
# requirements.txt : aws-lambda-powertools[all]
from aws_lambda_powertools.utilities import parameters

# Cache 300 secondes (5 min) — évite 1 appel API par invocation
DB_SECRET = parameters.get_secret("lambda/my-function/db-credentials", max_age=300)
db_config = json.loads(DB_SECRET)

# Alternative : boto3 direct (sans cache)
import boto3, json
client = boto3.client('secretsmanager', region_name='eu-west-1')
response = client.get_secret_value(SecretId='lambda/my-function/db-credentials')
db_config = json.loads(response['SecretString'])

Event Source Security Gratuit
Sécurité des sources d’événements — SQS DLQ, SNS cross-account, S3 triggers, Kinesis
🚨
Scénario d'attaque : Une file SQS configurée comme event source Lambda n'a pas de Dead Letter Queue (DLQ). Un message malveillant (payload malformé ou exploité) déclenche une exception dans Lambda. Sans DLQ, le message entre en boucle de retry infinie — le lambda est invoqué des centaines de fois, consommant toute la concurrence disponible et bloquant les autres fonctions. Avec DLQ : le message empoisonné est isolé après N tentatives, la file principale reste disponible.

7.1 — Tableau de configuration — Event Sources

Source Risque principal Configuration sécurisée Vérification
Amazon SQS Messages empoisonnés (poison pill), DLQ absente DLQ configurée + maxReceiveCount = 3-5 + chiffrement SSE-KMS aws sqs get-queue-attributes --attribute-names RedrivePolicy
Amazon S3 (notifications) Boucle infinie (Lambda écrit dans le même bucket qu'elle surveille) Surveiller un bucket d’entrée différent du bucket de sortie + filtre préfixe/suffixe Vérifier que notification bucket IN ≠ bucket OUT
Amazon SNS (cross-account) Invocation Lambda depuis un compte non autorisé Resource policy Lambda avec condition aws:SourceAccount aws lambda get-policy --function-name
Amazon Kinesis Starting position TRIM_HORIZON = retraitement de tout l'historique Starting position LATEST + bisect-on-batch-item-failure activé aws lambda list-event-source-mappings
Amazon DynamoDB Streams Réessais infinis sur les erreurs, streams non chiffrés DLQ configurée + streams chiffrés KMS + maximum retry = 3 Inspecter event source mapping DestinationConfig
💡
DLQ vs On-Failure Destination — Quelle différence ?
DLQ (SQS/SNS) : Configurée sur la fonction pour les invocations asynchrones (S3, SNS, EventBridge). Le message en échec est envoyé à la DLQ après 2 tentatives par défaut.
On-Failure Destination : Plus puissant — vous pouvez envoyer vers SQS, SNS, EventBridge, ou une autre Lambda. Inclut le payload complet de la requête + la réponse d'erreur.
Pour les event source mappings (SQS, Kinesis, DynamoDB) : La DLQ est configurée sur le mapping lui-même (DestinationConfig.OnFailure), pas sur la fonction.

Concurrence, Timeout & Résilience Gratuit
Reserved & provisioned concurrency, timeout, idempotency, protection contre le DoS financier
💡
Pourquoi la concurrence est un sujet de sécurité :
Lambda facture à l'invocation et à la durée d'exécution. Sans limite de concurrence, une attaque de type flood ou une boucle infinie peut consommer des millions d'invocations en quelques minutes. AWS ne coupe pas automatiquement une fonction surchargée : la facture monte, et les autres fonctions du compte sont throttlées. Définir une ReservedConcurrency constitue un contrôle de sécurité financière autant qu'une protection de disponibilité.
⚠️
Scénario DoS financier : Par défaut, chaque compte AWS a une limite de 1 000 exécutions concurrentes pour toutes les fonctions Lambda réunies. Si une seule fonction reçoit un pic de trafic (ou une attaque de type flood), elle peut consommer toute la concurrence disponible et provoquer l'échec throttling de toutes les autres fonctions du compte. Réserver une concurrence limitée isole les fonctions critiques.

Script d'Audit Lambda — Bash / AWS CLI
Script complet pour auditer une fonction Lambda ou toutes les fonctions d'un compte AWS
audit-lambda.sh — Script d'audit complet
#!/usr/bin/env bash
# =============================================================================
# audit-lambda.sh — Audit sécurité AWS Lambda
# Usage : ./audit-lambda.sh [function-name] (sans argument = toutes les fonctions)
# =============================================================================
set -euo pipefail

AWS_PROFILE="audit"
AWS_REGION="eu-west-1"
FUNCTION_FILTER="${1:-ALL}"  # Nom d'une fonction ou ALL
SCORE=0; MAX_SCORE=0
REPORT="lambda-audit-$(date +%Y%m%d-%H%M).txt"

aws_cmd() { aws "$@" --profile $AWS_PROFILE --region $AWS_REGION; }

check() {
  local pts=$1 label="$2" result="$3"
  MAX_SCORE=$((MAX_SCORE + pts))
  if [ "$result" = "OK" ]; then
    SCORE=$((SCORE + pts)); echo "  [✅ +${pts}pts] $label" | tee -a $REPORT
  else
    echo "  [❌  0pts] $label — $result" | tee -a $REPORT
  fi
}

audit_function() {
  local fn="$1"
  echo "\n=== Audit : $fn ===" | tee -a $REPORT
  local conf
  conf=$(aws_cmd lambda get-function-configuration --function-name "$fn" 2>/dev/null) || { echo "ERR: Fonction introuvable"; return; }

  # Phase 1 — IAM Execution Role
  local role; role=$(echo "$conf" | python3 -c "import sys,json; print(json.load(sys.stdin)['Role'])")
  local policy; policy=$(aws_cmd iam list-role-policies --role-name "$(basename $role)" 2>/dev/null)
  local inline_count; inline_count=$(echo "$policy" | python3 -c "import sys,json; print(len(json.load(sys.stdin).get('PolicyNames',[])))")
  [ "$inline_count" -eq 0 ] && check 20 "IAM : Pas de politique inline" "OK" || check 20 "IAM : Pas de politique inline" "$inline_count politique(s) inline détectée(s)"

  # Phase 2 — Variables d'environnement (secrets)
  local envvars; envvars=$(echo "$conf" | python3 -c "import sys,json; e=json.load(sys.stdin).get('Environment',{}).get('Variables',{}); bad=[k for k in e if any(s in k.upper() for s in ['SECRET','PASSWORD','TOKEN','KEY','PASSWD'])]; print('\n'.join(bad))")
  [ -z "$envvars" ] && check 20 "Secrets : Aucun secret en env var" "OK" || check 20 "Secrets : Aucun secret en env var" "Suspectes: $envvars"

  # Phase 3 — VPC
  local vpc; vpc=$(echo "$conf" | python3 -c "import sys,json; print(json.load(sys.stdin).get('VpcConfig',{}).get('VpcId',''))")
  [ -n "$vpc" ] && check 15 "VPC : Fonction dans un VPC" "OK" || check 15 "VPC : Fonction dans un VPC" "ALERTE — Fonction hors VPC"

  # Phase 4 — Function URL auth
  local furl; furl=$(aws_cmd lambda get-function-url-config --function-name "$fn" 2>/dev/null || echo "none")
  if [ "$furl" != "none" ]; then
    local auth; auth=$(echo "$furl" | python3 -c "import sys,json; print(json.load(sys.stdin).get('AuthType',''))")
    [ "$auth" = "AWS_IAM" ] && check 10 "Function URL : AuthType=AWS_IAM" "OK" || check 10 "Function URL : AuthType=AWS_IAM" "CRITIQUE — AuthType=$auth"
  else
    check 10 "Function URL : Pas de Function URL (OK)" "OK"
  fi

  # Phase 5 — Code Signing
  local csc; csc=$(aws_cmd lambda get-function-code-signing-config --function-name "$fn" 2>/dev/null || echo "none")
  [ "$csc" != "none" ] && check 10 "Code Signing : Code Signing Config définie" "OK" || check 10 "Code Signing : Code Signing Config définie" "ABSENT"

  # Phase 6 — Rétention logs
  local lg; lg=$(aws_cmd logs describe-log-groups --log-group-name-prefix "/aws/lambda/$fn" --query 'logGroups[0].retentionInDays' --output text 2>/dev/null)
  [ "$lg" != "None" ] && [ -n "$lg" ] && check 10 "Logs : Rétention définie ($lg jours)" "OK" || check 10 "Logs : Rétention définie" "ABSENT — Never Expire"

  # Phase 7 — Runtime
  local rt; rt=$(echo "$conf" | python3 -c "import sys,json; print(json.load(sys.stdin).get('Runtime',''))")
  local deprecated=(python3.8 python3.9 nodejs14.x nodejs16.x nodejs18.x)
  local is_deprecated=0
  for d in "${deprecated[@]}"; do [ "$rt" = "$d" ] && is_deprecated=1; done
  [ "$is_deprecated" -eq 0 ] && check 4 "Runtime : Actif ($rt)" "OK" || check 4 "Runtime : Actif" "DÉPRÉCIÉ — $rt"

  # Phase 8 — X-Ray
  local xray; xray=$(echo "$conf" | python3 -c "import sys,json; print(json.load(sys.stdin).get('TracingConfig',{}).get('Mode',''))")
  [ "$xray" = "Active" ] && check 3 "Tracing : X-Ray Active" "OK" || check 3 "Tracing : X-Ray Active" "PassThrough"
}

# Main
echo "Lambda Security Audit — $(date)" | tee $REPORT
if [ "$FUNCTION_FILTER" = "ALL" ]; then
  fns=$(aws_cmd lambda list-functions --query 'Functions[*].FunctionName' --output text)
  for fn in $fns; do audit_function "$fn"; done
else
  audit_function "$FUNCTION_FILTER"
fi
echo "\n===== SCORE FINAL : $SCORE / $MAX_SCORE =====" | tee -a $REPORT
echo "Rapport écrit dans : $REPORT"

CORSCross-Origin Resource Sharing — mécanisme navigateur autorisant ou bloquant les requêtes HTTP cross-domain
CSCCode Signing Configuration — configuration Lambda qui exige que le code déployé soit signé par un profil AWS Signer autorisé
DLQDead Letter Queue — file SQS ou topic SNS recevant les messages/invocations échoués après N tentatives
ENIElastic Network Interface — interface réseau virtuelle créée par Lambda dans votre VPC pour l'accès aux ressources privées
ESMEvent Source Mapping — configuration reliant une source d'événements (SQS, Kinesis, DynamoDB) à une fonction Lambda
IAMIdentity and Access Management — service AWS de contrôle d'accès (rôles, politiques, utilisateurs)
KMSKey Management Service — service AWS de gestion des clés de chiffrement (CMK et AWS managed keys)
OWASPOpen Web Application Security Project — organisation édite le Top 10 des vulnérabilités web les plus critiques
SASTStatic Application Security Testing — analyse du code source pour détecter des vulnérabilités sans exécuter le programme (ex : Bandit, Semgrep)
SBOMSoftware Bill of Materials — inventaire complet des dépendances d'un logiciel avec leurs versions et licences
SCPService Control Policy — politique AWS Organizations qui s'applique à tous les comptes d'une OU ou de l'organisation
SigV4AWS Signature Version 4 — protocole de signature des requêtes HTTP vers les services AWS (authentification des appels API)
SSMAWS Systems Manager — service incluant Parameter Store (stockage de configurations et secrets) et Session Manager
VPCVirtual Private Cloud — réseau isolé dans AWS avec sous-réseaux privés/publics, tables de routage et Security Groups
WAFWeb Application Firewall — pare-feu applicatif AWS protégeant contre les attaques OWASP Top 10, SQL injection, XSS
X-RayAWS X-Ray — service de tracé distribué permettant de visualiser les flux de requêtes end-to-end à travers les services AWS
