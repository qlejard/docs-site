💰 Coûts "cachés" : Apple & Google

Apple Business Manager (ABM) et Managed Google Play sont gratuits. Les coûts suivants peuvent s'appliquer :

  • Apple Developer Enterprise Program : NON requis pour MDM/ADE. À considérer uniquement pour certains scénarios de distribution d'apps internes hors App Store, et à vérifier sur la tarification Apple en vigueur.
  • Apps iOS payantes via Apple Business Manager : coût variable selon le catalogue Apple et les contrats éditeurs. Les applications gratuites restent à 0.
  • Android Zero-Touch Enrollment : gratuit pour OEM participants (Samsung, Google Pixel, Motorola, etc.). Vérifier compatibilité device sur androidenterprise.dev
  • Samsung Knox Mobile Enrollment (KME) : vérifier le niveau de service Samsung/Knox actif et les options premium éventuelles, hors socle Intune.

Infrastructure MDM Mobile Critique 10 pts
APNs, Apple Business Manager, Managed Google Play — prérequis absolus avant tout enrollment
🚨
Risque maximal : Un certificat APNs expiré entraîne la perte totale du contrôle de tous les appareils iOS/iPadOS. Un iPhone volé non géré donne accès aux e-mails, Teams, SharePoint sans aucun blocage possible. Des ports réseau fermés bloquent toutes les commandes Intune vers les appareils.
💡
Pourquoi chaque composant est obligatoire : APNs est le canal exclusif pour envoyer des commandes iOS/iPadOS depuis Intune. Sans APNs valide, aucune politique, aucun wipe, aucun profil ne peut être déployé. ABM (Apple Business Manager) active le mode supervisé et l'ADE (Automated Device Enrollment), prérequis pour 90 % des restrictions de sécurité iOS avancées. Managed Google Play est le prérequis absolu pour Android Enterprise : il lie le tenant Intune à un compte Google Pro et permet la distribution silencieuse d'applications.

    $exp = [DateTime]::Parse($_.tokenExpirationDateTime)
    Write-Host "ADE '$($_.tokenName)': expire $exp ($($_.enrolledDeviceCount) devices)"

2
Enrollment iOS / iPadOS Critique 12 pts
ADE/ABM, BYOD, User Enrollment — méthodes, comparaison et scripts de vérification
🚨
Token ADE à renouvellement annuel : Le token ADE (Automated Device Enrollment) expire chaque année. Sans renouvellement, les nouveaux appareils iOS/iPadOS ne peuvent plus être enrollés via ABM/ADE. Le renouvellement doit être effectué dans Apple Business Manager ET dans Intune avec le même Apple ID.
💡
Pourquoi le mode supervisé est déterminant : l'ADE (Automated Device Enrollment) est le seul moyen d'inscrire automatiquement un appareil iOS sans interaction physique de l'utilisateur. Le mode supervisé (activé via ABM) déverrouille de nombreuses restrictions de sécurité supplémentaires (USB Restricted Mode, blocage iCloud, filtrage DNS, etc.) non disponibles en mode non supervisé. Pour les flottes corporate, l'ADE supervisé est l'architecture recommandée.

Enrollment Android Enterprise Critique 10 pts
COBO, COPE, BYOD Work Profile, COSU/Kiosk — modes, flux et Zero-Touch Enrollment
🚨
Android Device Administrator (DA) fin de support Intune : Le mode Android Device Administrator est déprécié depuis août 2024 dans Microsoft Intune. Tous les appareils encore en mode DA doivent être migrés vers Android Enterprise. Sans migration, les politiques de conformité et les profils de configuration ne peuvent plus être appliqués.
💡
Pourquoi Android Enterprise est requis : Android Enterprise (AE) est le framework Google pour la gestion d'entreprise sur Android. Il requiert une liaison Managed Google Play obligatoire entre Intune et un compte Google Pro. AE sépare proprement les données personnelles et professionnelles (Work Profile) et permet le déploiement silencieux d'applications via Managed Google Play Store, sans compte Google personnel sur l'appareil.

Configuration Profiles Mobile Critique 12 pts
Restrictions iOS/iPadOS et Android Enterprise — paramètres de sécurité, déploiement GUI et PowerShell
🚨
De nombreuses restrictions iOS nécessitent le mode supervisé : des paramètres critiques comme USB Restricted Mode, blocage iCloud Backup, filtrage DNS, blocage App Store ou désactivation de Siri ne sont disponibles que sur les appareils iOS supervisés via ADE/ABM. Sur un appareil non supervisé (BYOD), ces restrictions sont ignorées silencieusement par le système.
💡
Distinction Configuration Profiles / Compliance Policies : Les Configuration Profiles imposent des paramètres à l'appareil (push actif — l'appareil est forcé d'appliquer le paramètre). Les Compliance Policies vérifient l'état de l'appareil (pull passif — Intune interroge et rapporte). Un profil de configuration qui bloque l'App Store empêche l'utilisateur d'y accéder. Une politique de conformité qui vérifie l'absence de jailbreak se contente de marquer l'appareil non conforme.

⚠ Solutions écartées

L'approche MAM-only sans enrollment MDM (protection des apps sans gestion de l'appareil) a été évaluée comme solution alternative. Elle a été écartée pour les flottes Corporate car elle ne permet pas d'appliquer de restrictions au niveau OS (USB, App Store, iCloud), ni de déclencher un wipe en cas de perte/vol. Elle reste pertinente pour les scénarios BYOD purs (Section 6). Le mode Android Device Administrator a également été écarté suite à sa dépréciation par Google et Microsoft Intune (août 2024).

Compliance Policies Mobile Important 10 pts
Conformité iOS/Android, intégration Conditional Access, actions en cas de non-conformité
🚨
Sans Conditional Access, la compliance ne bloque RIEN : Les politiques de conformité seules sont purement informatives. Un appareil marqué "Non conforme" continue d'accéder aux ressources (Exchange, SharePoint, Teams) jusqu'à ce qu'une règle Conditional Access basée sur la compliance soit activée. L'intégration CA est indispensable pour que la compliance ait un effet de sécurité réel.
💡
Pourquoi la compliance est la clé du Zero Trust mobile : La compliance policy évalue en continu l'état de l'appareil (OS à jour, pas de jailbreak/root, chiffrement activé, code PIN défini). Ce signal de conformité est transmis à Microsoft Entra ID via le connecteur Intune, et les règles de Conditional Access peuvent conditionner l'accès aux ressources cloud à cet état. C'est le fondement du modèle Zero Trust pour les appareils mobiles.

Certificats & Accès Réseau Standard 8 pts
SCEP vs PKCS, Wi-Fi et VPN — déploiement de certificats d'authentification sur mobile
🚨
Ordre de déploiement obligatoire : Le déploiement de certificats sur mobile suit un ordre strict : 1. Trusted Certificate (CA racine) → 2. SCEP ou PKCS (certificat utilisateur/device) → 3. Wi-Fi / VPN (profil qui référence le certificat). Tout déploiement hors ordre échoue silencieusement : le profil Wi-Fi ou VPN s'installe sans le certificat requis.
💡
Pourquoi les certificats remplacent les mots de passe pour Wi-Fi/VPN : L'authentification par certificat (802.1X EAP-TLS) élimine la nécessité de stocker des mots de passe Wi-Fi dans les profils Intune. Le certificat est lié à l'identité de l'appareil ou de l'utilisateur, ne peut pas être partagé, et est révocable instantanément. SCEP nécessite un serveur NDES sur site. PKCS fonctionne avec le seul Intune Certificate Connector (sans NDES) — plus simple pour les environnements cloud-only (Microsoft Entra ID).

Mobile Threat Defense (MTD) Standard 8 pts
Microsoft Defender for Endpoint, connecteurs MTD partenaires — intégration compliance et Conditional Access
🚨
MTD sans Compliance Policy = détection sans blocage : Un connecteur MTD activé qui détecte un malware sur un appareil Android ne bloque rien si aucune Compliance Policy ne référence le niveau de risque MTD. Le flux requis est : Connecteur MTD actif → Compliance Policy avec critère "niveau risque MTD ≤ Medium" → Conditional Access "Require compliance" → blocage effectif.
💡
Connecteurs MTD pris en charge par Intune : Microsoft Defender for Endpoint (recommandé — inclus M365 E5), Lookout for Work, Zimperium zIPS, Check Point Harmony Mobile, Pradeo, SentinelOne, et d'autres. Le connecteur envoie un signal de risque (heartbeat obligatoire toutes les 24h) à Intune, qui le transmet à Microsoft Entra ID pour évaluation dans les règles Conditional Access. Si le heartbeat dépasse 24h, l'appareil peut être automatiquement marqué non conforme.

Les Scope Tags associent chaque rôle RBAC à un sous-ensemble d'objets Intune (appareils, politiques, apps).

PowerShell — Audit MDM Mobile All-in-One (S1–S10) 
#Requires -Modules Microsoft.Graph.Authentication
# Connexion : Connect-MgGraph -Scopes "DeviceManagementServiceConfig.Read.All","DeviceManagementConfiguration.Read.All","DeviceManagementManagedDevices.Read.All","DeviceManagementApps.Read.All","DeviceManagementRBAC.Read.All"

function Add-Finding {
    param([string]$Section, [string]$Check, [string]$Status, [string]$Detail)
    [PSCustomObject]@{ Section = $Section; Check = $Check; Status = $Status; Detail = $Detail }
}

$findings = [System.Collections.Generic.List[PSCustomObject]]::new()

# ─── S1 : Infrastructure ───────────────────────────────────────────────────
$apns = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/applePushNotificationCertificate"
$daysLeft = ([DateTime]::Parse($apns.expirationDateTime) - (Get-Date)).Days
$findings.Add((Add-Finding "S1" "APNs expiration" `
    $(if ($daysLeft -gt 30) { "OK" } else { "WARNING" }) "$daysLeft jours restants"))

$mgp = Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/androidManagedStoreAccountEnterpriseSettings"
$findings.Add((Add-Finding "S1" "Managed Google Play" `
    $(if ($mgp.bindStatus -eq "bound") { "OK" } else { "FAIL" }) $mgp.bindStatus))

# ─── S5 : Compliance ───────────────────────────────────────────────────────
$policies = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/deviceCompliancePolicies").value
$iosPol  = $policies | Where-Object { $_.'@odata.type' -like "*ios*" }
$andPol  = $policies | Where-Object { $_.'@odata.type' -like "*android*" }
$findings.Add((Add-Finding "S5" "Compliance iOS" `
    $(if ($iosPol) { "OK" } else { "MISSING" }) "$($iosPol.Count) politique(s)"))
$findings.Add((Add-Finding "S5" "Compliance Android" `
    $(if ($andPol) { "OK" } else { "MISSING" }) "$($andPol.Count) politique(s)"))

# ─── S9 : MTD ──────────────────────────────────────────────────────────────
$mtd = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/mobileThreatDefenseConnectors").value
foreach ($c in $mtd) {
    $findings.Add((Add-Finding "S9" "MTD $($c.partnerType)" `
        $c.partnerState "heartbeat: $($c.lastHeartbeatDateTime)"))
}

# ─── S2 : Enrollment iOS ───────────────────────────────────────────────────
$adeTokens = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/beta/deviceManagement/depOnboardingSettings").value
$findings.Add((Add-Finding "S2" "Tokens ADE iOS" `
    $(if ($adeTokens.Count -gt 0) { "OK" } else { "MISSING" }) "$($adeTokens.Count) token(s) ADE"))

# ─── S3 : Enrollment Android Enterprise ────────────────────────────────────
$findings.Add((Add-Finding "S3" "AE Device Owner activé" `
    $(if ($mgp.androidDeviceOwnerFullyManagedEnrollmentEnabled) { "OK" } else { "WARNING" }) `
    "deviceOwner=$($mgp.androidDeviceOwnerFullyManagedEnrollmentEnabled)"))

# ─── S4 : Config Profiles ──────────────────────────────────────────────────
$configProfiles = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/deviceConfigurations").value
$iosP = ($configProfiles | Where-Object { $_.'@odata.type' -like "*ios*" }).Count
$andP = ($configProfiles | Where-Object { $_.'@odata.type' -like "*android*" }).Count
$findings.Add((Add-Finding "S4" "Profils config iOS" `
    $(if ($iosP -gt 0) { "OK" } else { "MISSING" }) "$iosP profil(s)"))
$findings.Add((Add-Finding "S4" "Profils config Android" `
    $(if ($andP -gt 0) { "OK" } else { "MISSING" }) "$andP profil(s)"))

# ─── S6 : MAM ──────────────────────────────────────────────────────────────
$mamAll = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceAppManagement/managedAppPolicies").value
$mamIos = ($mamAll | Where-Object { $_.'@odata.type' -like "*ios*" }).Count
$mamAnd = ($mamAll | Where-Object { $_.'@odata.type' -like "*android*" }).Count
$findings.Add((Add-Finding "S6" "App Protection Policy iOS" `
    $(if ($mamIos -gt 0) { "OK" } else { "MISSING" }) "$mamIos policy(ies)"))
$findings.Add((Add-Finding "S6" "App Protection Policy Android" `
    $(if ($mamAnd -gt 0) { "OK" } else { "MISSING" }) "$mamAnd policy(ies)"))

# ─── S7 : Stale Devices ────────────────────────────────────────────────────
$staleDate = (Get-Date).AddDays(-90).ToString("yyyy-MM-ddT00:00:00Z")
$stale = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/managedDevices?`$filter=lastSyncDateTime le $staleDate&`$top=100").value
$findings.Add((Add-Finding "S7" "Stale devices (>90j sans sync)" `
    $(if ($stale.Count -eq 0) { "OK" } else { "WARNING" }) "$($stale.Count)+ appareil(s) inactif(s)"))

# ─── S8 : Certificats ──────────────────────────────────────────────────────
$certP = ($configProfiles | Where-Object { `
    $_.'@odata.type' -match 'Scep|Pkcs|TrustedCertificate' }).Count
$findings.Add((Add-Finding "S8" "Profils certificats SCEP/PKCS/CA" `
    $(if ($certP -gt 0) { "OK" } else { "MISSING" }) "$certP profil(s)"))

# ─── S10 : Governance / Scope Tags ─────────────────────────────────────────
$scopeTagCount = (Invoke-MgGraphRequest -Method GET `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/roleScopeTags").value.Count
$findings.Add((Add-Finding "S10" "Scope Tags (hors Default)" `
    $(if ($scopeTagCount -gt 1) { "OK" } else { "WARNING" }) `
    "$scopeTagCount tag(s) — 1 = Default Scope seul"))

# ─── Rapport final ─────────────────────────────────────────────────────────
Write-Host "`n═══════ AUDIT MDM MOBILE — RAPPORT ═══════" -ForegroundColor Cyan
$findings | Format-Table Section, Check, Status, Detail -AutoSize
$ok   = ($findings | Where-Object Status -eq "OK").Count
$fail = ($findings | Where-Object Status -in @("FAIL", "MISSING")).Count
$warn = ($findings | Where-Object Status -eq "WARNING").Count
Write-Host "RÉSULTAT : $ok OK — $warn WARNING — $fail FAIL" `
    -ForegroundColor $(if ($fail -gt 0) { 'Red' } elseif ($warn -gt 0) { 'Yellow' } else { 'Green' })
