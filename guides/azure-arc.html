Mode d'emploi de ce guide
Périmètre, conventions et stratégie de déploiement
📋
Ce guide couvre l'intégralité de la chaîne Azure Arc : onboarding des machines physiques/VMs non-Azure, sécurisation réseau, RBAC, Policy, protection Defender, observabilité Monitor, SQL Arc, Azure Update Manager et Kubernetes. Chaque section est autonome — un admin peut appliquer uniquement les sections pertinentes à son périmètre.

Serveurs Windows/Linux on-premises connectés via Connected Machine AgentVMs Azure natives (gérées nativement)
Serveurs dans d'autres clouds (AWS EC2, GCP Compute Engine)Azure Stack HCI (guide dédié)
Clusters Kubernetes non-Azure (EKS, GKE, on-prem)AKS (voir azure-aks.html)
SQL Server on-premises et multicloud (Arc-enabled SQL)Azure SQL Database / Managed Instance (natifs)

1
Onboarding des Serveurs (Connected Machine Agent) Gratuit
Installation et enregistrement de l'agent Arc — condition préalable à toutes les sections suivantes
🚨
Scénario : Un serveur on-premises n'est pas visible dans Azure : aucune politique de conformité ne s'applique, aucun Defender ne le protège, aucun patch ne peut être orchestré depuis le cloud. C'est une machine aveugle — comme une caméra de surveillance débranchée. L'installation de l'agent Connected Machine prend <5 min et change radicalement la visibilité.

1.1 — Tableau de configuration agent

ParamètreDéfaut⚡ Quick Win🔒 Durci🎯 OptimalLicence
Agent installéNonOui (manuel)Oui (policy/script)Oui (at-scale)Gratuit
Mode connexion (--cloud)Non connectéPublic endpointProxy authentifiéPrivate LinkPrivate Link
Service principal dédié (onboarding)Compte interactifCompte interactifSP dédiéSP + certGratuit
Resource group & tagsNon structurésRG par siteTags Env/Owner/SitePolicy enforce tagsGratuit
Auto-upgrade agent (--enable-automatic-upgrade, preview)DésactivéDésactivéActivéActivé + suiviGratuit
Extensions installéesAucuneAMA (Azure Monitor)AMA + MDEAMA + MDE + KeyVaultDefender P1
💡
Pourquoi chaque paramètre compte :
• Service principal dédié : l'onboarding interactif expose des credentials admin. Un SP limité au rôle Azure Connected Machine Onboarding respecte le moindre privilège (Microsoft Cloud Security Benchmark IM-1 / PA-7).
• Auto-upgrade : Microsoft publie des mises à jour mensuelles et ne supporte officiellement que les versions de l'agent publiées dans l'année glissante ; activer l'upgrade automatique (preview) réduit le risque de dérive.
• Tags : sans tags, la facturation Defender est impossible à ventiler et les Policy d'exclusion deviennent ingouvernables.
⚠️
Piège fréquent : le script généré depuis le portail (Add servers > Generate script) contient un token valide 1 heure. En déploiement masse, utiliser un service principal sans expiration de token.

1.2 — Installation manuelle (Linux/Windows)

Bash / PowerShell — Onboarding manuel avec Service Principal
# === VARIABLES ===
$subscriptionId = "<subscription-id>"
$resourceGroup  = "rg-arc-servers-prod"
$location        = "westeurope"
$tenantId        = "<tenant-id>"
$spClientId      = "<service-principal-client-id>"
$spClientSecret  = "<service-principal-secret>"

# === WINDOWS — Téléchargement et installation agent ===
Invoke-WebRequest -Uri "https://aka.ms/AzureConnectedMachineAgent" -OutFile "AzureConnectedMachineAgent.msi"
msiexec.exe /i AzureConnectedMachineAgent.msi /l*v installationlog.txt /qn

# === Connexion avec Service Principal ===
azcmagent connect --service-principal-id $spClientId `
  --service-principal-secret $spClientSecret `
  --tenant-id $tenantId `
  --subscription-id $subscriptionId `
  --resource-group $resourceGroup `
  --location $location `
  --tags "Env=Prod;Owner=IT;Site=Paris" `
  --enable-automatic-upgrade

Sécurité Réseau — Connectivité, Proxy & Private Link Gratuit Private Link
Contrôler les flux sortants de l'agent et éliminer l'exposition Internet
🚨
Scénario : Par défaut, l'agent Arc communique avec les endpoints Microsoft via Internet public. Dans un environnement sensible (banque, santé, industrie), il faut réduire cette exposition au strict nécessaire. Azure Arc Private Link Scope fait passer les flux Arc compatibles Private Link par un endpoint privé ; Microsoft Entra ID, Azure Resource Manager et les autres services associés (ex: Azure Monitor) demandent encore leurs propres autorisations ou private endpoints.

2.2 — Configuration proxy

Bash/PowerShell — Configuration proxy sur l'agent Arc
# Configurer un proxy authentifié
azcmagent config set proxy.url "http://proxy.corp.local:8080"  # éviter d'inclure des credentials dans la commande — préférer un proxy d'authentification Kerberos/NTLM transparent

# Exclure des hosts du proxy (ex: endpoints internes)
azcmagent config set proxy.bypass "169.254.169.254,corp.local"

# Vérification
azcmagent config get proxy.url

2.3 — Configuration Azure Arc Private Link Scope

Azure CLI — Création du Private Link Scope + Private Endpoint
# 1. Créer le Private Link Scope
az resource create --resource-group rg-arc-network `
  --resource-type "Microsoft.HybridCompute/privateLinkScopes" `
  --name "arc-pls-prod" `
  --location westeurope `
  --properties '{"publicNetworkAccess":"Disabled"}'

# 2. Associer les serveurs au scope
# - Nouveau serveur : générer le script d'onboarding Azure Arc avec le mode de connectivité "Private endpoint"
# - Serveur existant : associer la machine depuis Azure Arc Private Link Scope > Azure Arc resources > Add

# 3. Créer le Private Endpoint dans le VNet
az network private-endpoint create `
  --resource-group rg-arc-network `
  --name pe-arc-pls `
  --vnet-name vnet-prod `
  --subnet snet-endpoints `
  --private-connection-resource-id /subscriptions/<sub>/resourceGroups/rg-arc-network/providers/Microsoft.HybridCompute/privateLinkScopes/arc-pls-prod `
  --group-id hybridcompute `
  --connection-name arc-pls-conn

RBAC & Gouvernance des Identités Arc Gratuit
Contrôler qui peut voir, modifier et gérer les machines Arc et leurs extensions
🚨
Scénario : Sans RBAC granulaire, tout contributeur Azure peut installer des extensions (scripts, agents) sur vos serveurs on-premises — équivalent à donner les clés de votre datacenter à n'importe quel administrateur Azure. Le rôle Azure Arc ScVmm VM Contributor mal attribué permet l'exécution de commandes arbitraires sur les machines.

3.1 — Rôles RBAC Arc natifs

RôlePermissions clésUsage recommandéRisque si sur-attribué
Azure Connected Machine OnboardingEnregistrer des machines (write limité)SP d'onboarding uniquementFaible
Azure Connected Machine Resource AdministratorLecture/écriture machines + extensionsAdmins Arc dédiésMoyen (install extensions)
Azure Arc VMware VM ContributorStart/Stop/Delete VMs Arc VMwareOps niveau 2/3Élevé (delete machines)
Contributor (abonnement)Tout — y compris extensions Run CommandNe pas utiliser sur le scope ArcCritique (RCE via extensions)
ReaderLecture seule inventaireAuditeurs, RSSIAucun
Azure Arc Kubernetes Cluster Userkubeconfig pour clusters Arc K8sDéveloppeurs (namespace limité)Moyen
💡
Pourquoi pas Contributor : le rôle Contributor sur une machine Arc permet l'installation d'extensions (dont CustomScriptExtension) qui exécutent des scripts arbitraires sur le serveur on-premises. C'est une escalade de privilèges cloud → on-premises. Utiliser Azure Connected Machine Resource Administrator au scope Resource Group uniquement.

3.2 — Configuration RBAC

Azure CLI — Attribution RBAC granulaire sur Arc
# Rôle onboarding pour le SP de déploiement (scope abonnement)
az role assignment create `
  --assignee "<sp-client-id>" `
  --role "Azure Connected Machine Onboarding" `
  --scope "/subscriptions/<subscription-id>"

# Admins Arc : Resource Administrator au scope RG uniquement
az role assignment create `
  --assignee "arc-admins@corp.com" `
  --role "Azure Connected Machine Resource Administrator" `
  --scope "/subscriptions/<sub>/resourceGroups/rg-arc-servers-prod"

# Auditeurs : Reader au scope RG
az role assignment create `
  --assignee "rssi@corp.com" `
  --role "Reader" `
  --scope "/subscriptions/<sub>/resourceGroups/rg-arc-servers-prod"

4
Azure Policy & Guest Configuration Policy Guest Config
Appliquer et auditer automatiquement la conformité sur toutes les machines Arc
🚨
Scénario : Sans Policy, un serveur on-premises peut dériver de la configuration de référence en silence — mot de passe admin local réutilisé, audit Windows désactivé, TLS 1.0 encore actif. Azure Policy + Guest Configuration transforme chaque machine Arc en objet continuellement audité, comme un scanner de conformité permanent.
💡
Pourquoi c'est critique : Azure Policy + Guest Configuration est le seul mécanisme transformant un serveur Arc en ressource continuellement vérifiée. Sans Policy, l'onboarding crée de la visibilité mais aucune conformité n'est appliquée — le serveur peut dériver silencieusement sans alerte.

Piège : les Policy avec effet DeployIfNotExists nécessitent une managed identity assignée à la Policy Assignment avec le rôle Contributor. Sans cela, la rémédiation échoue silencieusement et le tableau de bord Compliance affiche 0% de conformité sans erreur visible.

4.2 — Assignation des initiatives via Azure CLI

Azure CLI — Assignation Policy CIS Benchmark sur les serveurs Arc
# Récupérer l'ID de la Policy CIS Windows
$policyId = az policy definition list `
  --query "[?contains(displayName,'Windows machines should meet CIS')].id | [0]" -o tsv

# Assigner la Policy au scope RG avec identité managée
az policy assignment create `
  --name "arc-cis-windows" `
  --display-name "Arc - CIS Windows Benchmark" `
  --policy $policyId `
  --scope "/subscriptions/<sub>/resourceGroups/rg-arc-servers-prod" `
  --mi-user-assigned "/subscriptions/<sub>/resourceGroups/rg-arc-network/providers/Microsoft.ManagedIdentity/userAssignedIdentities/mi-policy-arc" `
  --location westeurope

5
Microsoft Defender for Servers — Protection Arc Defender P1 Defender P2
Détection des menaces, évaluation de vulnérabilités et accès JIT sur les serveurs hybrides
🚨
Scénario : Un serveur on-premises connecté via Arc mais sans Defender est une cible sans défense EDR. Une attaque par ransomware (ex: double extorsion via RDP brute-force) ne sera détectée que lors de l'impact. Defender for Servers Plan 1 active l'intégration MDE rapidement après activation du plan ; pour le coût, s'appuyer sur la page pricing live Microsoft Defender for Cloud plutôt que sur un montant figé dans le guide.

Azure Monitor & Log Analytics — Observabilité Hybride Gratuit Pay/Go LA
Collecte des logs, métriques et alertes sur toutes les machines Arc
🚨
Scénario : Sans Azure Monitor, les logs Windows Event et syslog Linux restent enfermés sur la machine. Lors d'un incident de sécurité, l'investigation rétrospective est impossible (>72h de travail d'investigation perdu). L'agent Azure Monitor (AMA) centralise tous les logs dans Log Analytics en moins d'une heure — base de toute investigation SIEM.

6.2 — Déploiement AMA + DCR

# Installer extension AMA sur une machine Arc
az connectedmachine extension create `
  --resource-group rg-arc-servers-prod `
  --machine-name <machine-name> `
  --name AzureMonitorWindowsAgent `
  --type AzureMonitorWindowsAgent `
  --publisher Microsoft.Azure.Monitor `
  --location westeurope

Arc-enabled SQL Server — Gouvernance & Sécurité SQL Gratuit Defender SQL
Visibilité, conformité et protection des instances SQL Server on-premises via Azure
🚨
Scénario : Des instances SQL Server 2012 en fin de vie, non patchées, sur des serveurs on-premises invisibles — CVE critiques non corrigées. Arc-enabled SQL Server permet de détecter toutes ces instances automatiquement, d'appliquer la conformité CIS et d'activer Defender for SQL en quelques clics.
💡
Pourquoi c'est utile : Arc-enabled SQL Server détecte automatiquement toutes les instances SQL on-premises via les machines Arc connectées — sans agent SCOM ni System Center. C'est la base d'un inventaire SQL entreprise exhaustif avec évaluation Best Practices ; la protection menace SQL s'active ensuite via le plan séparé Defender for SQL Servers on Machines.

⚡ Script All-in-One — Déploiement Azure Arc Complet

⚠️
Prérequis : Az CLI connecté (az login), Azure PowerShell module, droits Owner sur l'abonnement, serveur cible avec accès Internet ou proxy configuré.
PowerShell — Arc Deploy All-in-One (10 étapes)
# ============================================================
# AZURE ARC — SCRIPT DE DÉPLOIEMENT ALL-IN-ONE
# Adapte les variables de configuration ci-dessous
# ============================================================

$SUB   = "<subscription-id>"
$RG    = "rg-arc-servers-prod"
$LOC   = "westeurope"
$RGMON = "rg-arc-monitoring"
$LAW   = "law-arc-prod"
$TENANT= "<tenant-id>"

# [1/10] — Sélectionner l'abonnement
az account set --subscription $SUB

# [2/10] — Créer les Resource Groups
az group create --name $RG    --location $LOC
az group create --name $RGMON --location $LOC

# [3/10] — Créer le Service Principal d'onboarding
$SP = az ad sp create-for-rbac --name "arc-onboarding-sp" `
  --role "Azure Connected Machine Onboarding" `
  --scopes "/subscriptions/$SUB" | ConvertFrom-Json

# [4/10] — Créer le workspace Log Analytics (90j rétention)
az monitor log-analytics workspace create `
  --resource-group $RGMON `
  --workspace-name $LAW `
  --location $LOC `
  --retention-time 90

# [5/10] — Activer Defender for Servers Plan 1
az security pricing create --name VirtualMachines --tier Standard --subplan P1

# [6/10] — Activer auto-provisioning MDE + AMA
az security auto-provisioning-setting update --name MicrosoftDefenderForEndpoint --auto-provision On
az security auto-provisioning-setting update --name MicrosoftMonitoringAgent --auto-provision On

# [7/10] — Assigner la Policy CIS Azure Benchmark
az policy assignment create `
  --name "cis-azure-v2" `
  --policy-set-definition "/providers/Microsoft.Authorization/policySetDefinitions/06f19060-9e68-4070-92ca-f15cc126059e" `
  --scope "/subscriptions/$SUB"

# [8/10] — Attribuer RBAC admins Arc sur le RG
az role assignment create `
  --assignee "arc-admins@corp.com" `
  --role "Azure Connected Machine Resource Administrator" `
  --scope "/subscriptions/$SUB/resourceGroups/$RG"

# [9/10] — Sur la machine cible : installer l'agent Arc
# (Commande à exécuter sur le serveur on-premises)
# Invoke-WebRequest -Uri "https://aka.ms/AzureConnectedMachineAgent" -OutFile "AzureConnectedMachineAgent.msi"
# msiexec.exe /i AzureConnectedMachineAgent.msi /qn
# azcmagent connect --service-principal-id $SP.appId --service-principal-secret $SP.password --tenant-id $TENANT --subscription-id $SUB --resource-group $RG --location $LOC

# [10/10] — Vérifier le déploiement
az connectedmachine list `
  --resource-group $RG `
  --query "[].{Name:name, Status:status, Agent:agentVersion}" `
  --output table

Write-Host "[OK] Déploiement Arc terminé. Vérifiez le portail : portal.azure.com > Azure Arc > Servers"
