🚨
Scénario d'attaque : Un pipeline CodePipeline V1 est configuré avec un webhook GitHub public, sans restriction de branche. Un attaquant forke le repo, insère un buildspec.yml malveillant qui ex-filtre les variables d'environnement (secrets AWS) vers un serveur externe. Le pipeline se déclenche automatiquement sur le PR, le CodeBuild exécute le buildspec avec le rôle IAM de production. Résultat : credentials AWS exposés en 90 secondes, déploiement d'un reverse shell en production, pivot vers d'autres comptes via les rôles cross-account.

Analogie : 🏭 Le pipeline CI/CD est une chaîne de montage automobile. Chaque étape (source, build, test, approval, deploy) vérifie la qualité avant de passer à la suivante. Si une pièce défectueuse (code malveillant) entre sur la ligne sans contrôle, elle se retrouve dans le véhicule livré au client. Les approval gates sont le chef d'atelier qui signe le bon de sortie.

💡
Pourquoi chaque paramètre Pipeline Architecture compte :
  • Pipeline V2 vs V1 : V1 facture 1 $/pipeline/mois même inactif. V2 = gratuit + paiement par action (0,002 $). V2 ajoute les triggers avec filtres (branches, tags, file paths), les variables pipeline, et l'execution mode (QUEUED/PARALLEL). V1 est en maintenance-only depuis 2023. Source : AWS CodePipeline Pipeline Types.
  • Execution mode QUEUED : en mode SUPERSEDED (défaut V1), un nouveau commit annule l'exécution en cours. Si un déploiement prod est en cours, il est interrompu. QUEUED sérialise les exécutions — chaque commit est traité dans l'ordre. Source : AWS Well-Architected — Reliability Pillar.
  • CodeConnections vs OAuth : les OAuth tokens sont liés à un utilisateur personnel — si l'employé part, le pipeline casse. CodeConnections utilise une app installation (org-level), indépendante des utilisateurs. De plus, OAuth tokens ont souvent des scopes trop larges. Source : AWS Blog « Migrating from CodeStar Connections ».
  • Trigger filters : sans filtre de branche, un push sur feature/test-123 déclenche le pipeline prod. Sans filtre de fichier, un changement dans README.md lance un build complet (>10 min, >0,50 $). Les filtres éliminent 60-80% des exécutions inutiles. Source : retour terrain audit.
  • Artifact encryption CMK : avec SSE-S3 (défaut), tout utilisateur ayant accès au bucket S3 peut lire les artifacts. Avec une CMK dédiée, l'accès requiert à la fois le droit S3 et le droit KMS — double barrière. Indispensable pour le cross-account. Source : CIS AWS Foundations v7.0.0 §2.1.
  • Approval gate : sans approval, un commit mergé dans main se retrouve en production en ~3-5 minutes. L'approval gate impose une validation humaine (4-eyes principle, SOC 2 CC8.1). Le timeout ≤24h évite les pipelines bloqués indéfiniment.
  • Pipeline IAM role : le rôle par défaut créé par la console a des permissions excessives (accès à tous les buckets S3, toutes les fonctions Lambda). Un rôle custom avec ARN spécifiques limite le blast radius en cas de compromission. Source : AWS SEC02-BP04 — Least Privilege.
  • Cross-account : déployer depuis le même compte que le code source viole la séparation des environnements. Le pattern recommandé : compte CI/CD dédié, AssumeRole vers les comptes cibles (dev, staging, prod). Source : AWS Organizations Best Practices, Well-Architected SEC02.

1.3 — Anti-patterns pipeline courants

⚠️
Les 7 anti-patterns pipeline les plus fréquents :
  1. Rester en Pipeline V1 — pas de triggers filés, pas de variables, coût fixe 1 $/mois même inactif. Migrer vers V2 immédiatement.
  2. Pas d'approval gate avant production — un merge dans main arrive en prod en 3 min sans validation humaine. SOC 2 CC8.1 exige le 4-eyes principle.
  3. Pipeline IAM role avec * — le rôle généré par la console a accès à tous les buckets S3. Réécrire avec ARN spécifiques.
  4. Source via OAuth token personnel — lié à un employé qui partira. Utiliser CodeConnections (app installation org-level).
  5. Pas de filtre de déclenchement — un README.md modifié lance un build complet de 15 min. Filtrer par branche + file path.
  6. Artifacts non chiffrés avec CMK — SSE-S3 par défaut = pas de contrôle d'accès KMS, pas d'audit Decrypt dans CloudTrail.
  7. Pipeline mono-compte — le code source, le build et la prod dans le même compte. Un attaquant compromet tout. Séparer CI/CD et workload accounts.

1.5 — Configuration — Terraform

HCL — Terraform — Pipeline V2 DevSecOps complète
# --- KMS Key pour artifacts pipeline ---
resource "aws_kms_key" "pipeline" {
  description         = "KMS key for CodePipeline artifacts"
  enable_key_rotation = true
  deletion_window_in_days = 14
}

# --- S3 Bucket pour artifacts ---
resource "aws_s3_bucket" "pipeline_artifacts" {
  bucket = "myapp-pipeline-artifacts-${data.aws_region.current.name}"
}

resource "aws_s3_bucket_server_side_encryption_configuration" "pipeline_artifacts" {
  bucket = aws_s3_bucket.pipeline_artifacts.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.pipeline.arn
    }
  }
}

# --- CodePipeline V2 ---
resource "aws_codepipeline" "main" {
  name           = "myapp-prod-pipeline"
  role_arn       = aws_iam_role.codepipeline.arn
  pipeline_type  = "V2"
  execution_mode = "QUEUED"

  artifact_store {
    location = aws_s3_bucket.pipeline_artifacts.id
    type     = "S3"
    encryption_key {
      id   = aws_kms_key.pipeline.arn
      type = "KMS"
    }
  }

  # Stage 1 — Source
  stage {
    name = "Source"
    action {
      name             = "GitHub-Source"
      category         = "Source"
      owner            = "AWS"
      provider         = "CodeStarSourceConnection"
      version          = "1"
      output_artifacts = ["SourceOutput"]
      namespace        = "SourceVariables"
      configuration = {
        ConnectionArn        = aws_codestarconnections_connection.github.arn
        FullRepositoryId     = "myorg/myapp"
        BranchName           = "main"
        DetectChanges        = "true"
        OutputArtifactFormat = "CODE_ZIP"
      }
    }
  }

  # Stage 2 — SAST
  stage {
    name = "SAST"
    action {
      name             = "SecurityScan"
      category         = "Build"
      owner            = "AWS"
      provider         = "CodeBuild"
      version          = "1"
      input_artifacts  = ["SourceOutput"]
      output_artifacts = ["SASTOutput"]
      configuration = {
        ProjectName = aws_codebuild_project.sast.name
      }
    }
  }

  # Stage 3 — Build
  stage {
    name = "Build"
    action {
      name             = "BuildAndPush"
      category         = "Build"
      owner            = "AWS"
      provider         = "CodeBuild"
      version          = "1"
      input_artifacts  = ["SourceOutput"]
      output_artifacts = ["BuildOutput"]
      namespace        = "BuildVariables"
      configuration = {
        ProjectName = aws_codebuild_project.build.name
      }
    }
  }

  # Stage 4 — Test
  stage {
    name = "Test"
    action {
      name            = "IntegrationTests"
      category        = "Build"
      owner           = "AWS"
      provider        = "CodeBuild"
      version         = "1"
      input_artifacts = ["BuildOutput"]
      configuration = {
        ProjectName = aws_codebuild_project.test.name
      }
    }
  }

  # Stage 5 — Approval
  stage {
    name = "Approval"
    action {
      name     = "ManualApproval"
      category = "Approval"
      owner    = "AWS"
      provider = "Manual"
      version  = "1"
      configuration = {
        NotificationArn = aws_sns_topic.pipeline_approvals.arn
        CustomData      = "Approve deployment to production?"
      }
    }
  }

  # Stage 6 — Deploy
  stage {
    name = "Deploy"
    action {
      name            = "DeployECS"
      category        = "Deploy"
      owner           = "AWS"
      provider        = "ECS"
      version         = "1"
      input_artifacts = ["BuildOutput"]
      configuration = {
        ClusterName = "myapp-prod-cluster"
        ServiceName = "myapp-prod-service"
        FileName    = "imagedefinitions.json"
      }
    }
  }

  # Trigger — filtré par branche et file paths
  trigger {
    provider_type = "CodeStarSourceConnection"
    git_configuration {
      source_action_name = "GitHub-Source"
      push {
        branches { includes = ["main"] }
        file_paths { excludes = ["docs/*", "*.md", ".gitignore"] }
      }
    }
  }
}

Pourquoi chaque paramètre CodeBuild Security compte :
  • Image build managed vs publique : les images Docker publiques (ubuntu:latest, node:18) peuvent contenir des vulnérabilités connues ou des backdoors dans des layers intermédiaires.   • IAM service role least privilege : le rôle CodeBuild est actif pendant toute la durée du build (~5-30 min). Si le buildspec est compromis, l'attaquant dispose de credentials temporaires avec toutes les permissions du rôle. Un rôle avec s3:* permet d'ex-filtrer tous les buckets. Réduire aux ARN spécifiques.   • Privileged mode : donne accès au Docker daemon du host. Un buildspec malveillant peut monter le filesystem host, accéder aux metadata EC2 (169.254.169.254), et pivoter vers d'autres containers. N'activer que pour les builds Docker-in-Docker. Source : CIS Docker Benchmark 5.4.

En VPC avec des VPC Endpoints (S3, ECR, Secrets Manager), le trafic reste sur le réseau AWS privé. Source : AWS PrivateLink Best Practices.
  • Secrets dans le buildspec : les variables PLAINTEXT apparaissent dans la console et les logs CloudWatch. PARAMETER_STORE et SECRETS_MANAGER sont injectés au runtime sans exposition dans les logs. Secrets Manager ajoute la rotation automatique (Lambda rotation function). Source : AWS SEC02-BP05 — Audit and Rotate Credentials.
  • Build timeout : un build légitime prend 5-15 min. Si un build tourne >30 min, c'est potentiellement du crypto-mining (utilisation des vCPUs pour miner des cryptomonnaies). Facture surprise de >1000 $/jour. Le timeout est la première défense. Source : retour terrain audit AWS.
  • Cache : sans cache, chaque build télécharge les dépendances (npm install, pip install) depuis Internet. Risque de dependency confusion si un package malveillant porte le même nom. Le cache S3 + local réduit la surface d'attaque et le temps de build. Source : AWS CodeBuild Caching Best Practices.

2.3 — Anti-patterns CodeBuild courants

⚠️
Les 8 anti-patterns CodeBuild les plus fréquents :
  1. env | sort ou printenv dans le buildspec — imprime TOUTES les variables d'environnement dans les logs CloudWatch, incluant les credentials temporaires AWS. Supprimer immédiatement.
  2. Secrets en PLAINTEXT — visibles dans la console CodeBuild et les logs. Utiliser PARAMETER_STORE ou SECRETS_MANAGER exclusivement.
  3. Image :latest — non reproductible, potentiellement différente entre deux builds. Utiliser des tags immutables (SHA ou date).
  4. Rôle IAM avec "Resource": "*" — le build a accès à tous les buckets, tous les secrets, tous les ECR repos. Spécifier les ARN exacts.
  5. Privileged mode activé sans raison — accès au Docker daemon host. Risque d'évasion du container. Désactiver sauf pour les builds Docker-in-Docker.
  6. Timeout par défaut (60 min) — un build compromis (crypto-mining) peut tourner 1h avant d'être arrêté. Réduire à 10-30 min selon le type.
  7. Pas de cache — chaque build télécharge les dépendances depuis des registries publics. Risque de dependency confusion + lenteur. Activer S3 cache + Docker layer cache.
  8. Builds sans VPC mode pour les projets accédant à des ressources privées — les credentials de la BDD passent par Internet public. Configurer VPC + security groups.

💡
Pourquoi chaque paramètre SAST & Code Analysis compte :
  • SAST obligatoire dans le pipeline : 70% des vulnérabilités en production auraient été détectées par un scan SAST (source : Veracode State of Software Security 2024). Le coût de correction en production est 30x plus élevé qu'en phase de build. Le stage SAST dans le pipeline est le meilleur ROI sécurité.
  • IaC scanning : 60% des incidents cloud proviennent de mauvaises configurations (Gartner). Un Security Group avec 0.0.0.0/0:22 dans un template CloudFormation/Terraform est un vecteur d'attaque direct. cfn-lint, tfsec et checkov détectent ces problèmes avant le déploiement.
  • Quality gate bloquante : sans exit 1 sur les findings critiques, le scan SAST est un « rapport qu'on ignore ». Le pipeline doit échouer pour forcer la correction. Les équipes qui rendent le scan non-bloquant retrouvent les mêmes findings 6 mois plus tard (source : retour terrain audit).
  • SARIF + Security Hub : sans centralisation, chaque outil produit son propre rapport dans un format différent. SARIF (Static Analysis Results Interchange Format) est le standard OASIS pour les rapports SAST. Security Hub agrège les findings de CodeGuru, Inspector, et les outils tiers importés via SARIF.
  • Nightly full scan : les CVE sont publiées en continu. Du code écrit il y a 6 mois peut devenir vulnérable aujourd'hui (à cause d'une nouvelle CVE). Le scan différentiel sur commit ne détecte que les nouvelles lignes. Le scan nightly complet relève les vulnérabilités sur le code existant.
  • Custom rules Semgrep : les règles génériques ne connaissent pas les conventions métier. Exemple : interdire pickle.loads() (déserialisation non sûre), forcer parameterized queries au lieu de f-strings SQL, interdire les imports non autorisés. Semgrep permet d'écrire des règles YAML custom en 5 min.
  • Secrets scanning : une clé AWS poussée sur GitHub est détectée et exploitée en <5 minutes par des bots automatiques (Truffle Security Research 2023). git-secrets est un pre-commit hook qui bloque le commit avant le push. C'est la première ligne de défense.

3.3 — Anti-patterns SAST courants

⚠️
Les 7 anti-patterns SAST les plus fréquents :
  1. Pas de SAST dans le pipeline — « on fait la sécurité manuellement avant la release ». En pratique, ce review n'a jamais lieu. Intégrer le SAST dans le stage 2 du pipeline.
  2. SAST non bloquant — « on génère un rapport mais on ne bloque pas ». Après 3 sprints, plus personne ne lit le rapport. exit 1 sur High/Critical obligatoire.
  3. Trop de faux positifs → désactivation — mauvaise calibration de l'outil. Commencer avec un petit jeu de règles (Semgrep --config auto) et ajouter progressivement.
  4. Scanner IaC après le déploiement — AWS Config détecte les dérives, mais le mal est fait. Scanner avant le terraform apply / cfn deploy.
  5. Un seul outil pour tout — aucun outil ne couvre tous les langages et patterns. Combiner SAST code (Semgrep) + SAST IaC (checkov) + SCA (Inspector).
  6. Pas de secrets scanning — les développeurs commettent accidentellement des clés AWS, tokens GitHub, mots de passe DB. git-secrets + pipeline scan = double barrière.
  7. Ignorer le scan IaC — « c'est juste de l'infra, pas du code ». 60% des incidents cloud = mauvaise config. L'IaC est du code et doit être scanné.

  • Pull-through cache : Docker Hub limite les pulls anonymes à 100/6h. Un cluster EKS avec 50 nodes qui scale un déploiement atteint ce rate limit en minutes. Le pull-through cache ECR stocke les images localement et élimine cette dépendance.

4.2 — Anti-patterns Container Security courants

⚠️
Les 8 anti-patterns Container Security les plus fréquents :
  1. Pas de scan ECR — « on scanne en local avec Docker Scout ». Le scan local est bypassed si le développeur ne le lance pas. Le scan ECR est automatique (au push ou continu).
  2. Tags mutables (:latest) — impossible de savoir quelle version tourne en production. Impossible de détecter un tag override malveillant. Toujours utiliser le commit SHA comme tag.
  3. Image full OS (ubuntu:22.04) — 700 MB, 1200 packages, 50+ CVE high. Utiliser Alpine (5 MB base) ou distroless. « Mais on a besoin de bash pour debug » — utilisez kubectl debug avec une image ephemeral.
  4. USER root en production — si le container est compromis, l'attaquant a root. Avec USER appuser, il n'a accès qu'aux fichiers de l'app.
  5. COPY .env /app/ dans le Dockerfile — les secrets se retrouvent dans le layer Docker, accessible via docker history. Utiliser --mount=type=secret (BuildKit) ou des variables d'environnement injectées au runtime.
  6. Pas de lifecycle policy — 3000 images dans un repo ECR, les plus vieilles ont 2 ans avec des CVE Log4Shell non corrigées. Configurer le nettoyage automatique.
  7. Pull direct Docker Hub en production — rate limit + dépendance sur un service externe. Si Docker Hub est down, vos déploiements échouent. Pull-through cache ECR.
  8. Single-stage Dockerfile — le compilateur, les dev-dependencies, les fichiers source sont dans l'image de production. Surface d'attaque x3. Multi-stage obligatoire.

Pourquoi chaque paramètre Supply Chain compte :
  • CodeArtifact proxy : en 2024, les attaques supply chain ont augmenté de 300% (Sonatype State of the Software Supply Chain). Le registre public est un vecteur d'attaque direct. CodeArtifact agit comme un firewall pour vos dépendances : il cache les packages, permet de bloquer des versions, et élimine la dépendance au registre public en production.
  • Lock file + hash : sans lock file, npm install peut résoudre une version différente entre deux builds. Avec npm ci, le build utilise exactement les versions du package-lock.json. --require-hashes (pip) vérifie l'intégrité de chaque package téléchargé (protection contre MITM).
  • SBOM : quand Log4Shell (CVE-2021-44228) a frappé, les organisations sans SBOM ont mis des semaines pour identifier quelles applications utilisaient Log4j. Avec un SBOM, la recherche prend 5 minutes. Le SBOM est exigé par le US Executive Order 14028 et le NIST SP 800-218 (SSDF).
  • Dependency confusion : l'attaquant publie un package myorg-internal-utils sur npm public avec une version supérieure (99.0.0). Si le pipeline résout depuis npm public avant CodeArtifact, il installe le package malveillant. Solution : scoper les packages internes (@myorg/) + configurer CodeArtifact comme seul upstream.
  • Licence compliance : utiliser un package GPL-3.0 dans du code propriétaire oblige à publier le code source. Les risques juridiques sont réels. SSPL (MongoDB) interdit l'utilisation en SaaS. Vérifier les licences avant d'ajouter une dépendance, pas 6 mois après en production.
  • --ignore-scripts : les postinstall scripts npm sont le vecteur #1 des supply chain attacks. event-stream, ua-parser-js, node-ipc — tous utilisaient des postinstall malveillants. Désactiver par défaut dans le build, activer au cas par cas si nécessaire.
  • Dépendances obsolètes : 40% des dépendances dans les projets enterprise sont obsolètes de >1 version majeure (Snyk Open Source Security Report). Plus la dette est vieille, plus la mise à jour est coûteuse. Automatiser les patches avec Dependabot/Renovate réduit cette dette en continu.

5.2 — Anti-patterns Supply Chain courants

⚠️
Les 8 anti-patterns Supply Chain les plus fréquents :
  1. npm install au lieu de npm ci — npm install peut modifier le lock file et résoudre des versions différentes. npm ci est déterministe et échoue si le lock file est incohérent.
  2. Pas de CodeArtifact (install direct public) — dépendance au registre public (downtime, rate limit, supply chain attack). CodeArtifact = cache + contrôle + audit trail.
  3. Pas de SBOM — « on ne sait pas quelles dépendances sont dans nos images ». Quand la prochaine Log4Shell arrive, vous perdez des jours à chercher.
  4. Lock file non committé — le package-lock.json / poetry.lock / pom.xml avec versions [1.0,2.0) n'est pas versionné. Chaque build peut produire un résultat différent.
  5. Pas de scan de vulnérabilités — npm audit existe depuis npm 6 (2018), et pourtant 60% des pipelines ne l'exécutent pas. 1 commande, 0 coût, détecte les CVE connues.
  6. Dependency confusion non protégé — packages internes sans scope (@myorg/), pas de filtrage upstream dans CodeArtifact. L'attaquant publie sur npm public → installation automatique.
  7. Scripts postinstall non désactivés — npm install exécute les postinstall scripts par défaut. Vecteur d'attaque supply chain. --ignore-scripts en build, sauf exception documentée.
  8. Dépendances non mises à jour (>2 ans) — CVE non corrigées, incompatibilités futures, coût de migration exponentiel. Automatiser les mises à jour patch/minor.

Pourquoi chaque paramètre Secrets compte :
  • Secrets Manager vs variables d'env : les variables d'environnement CodeBuild sont chiffrées au repos, mais visibles dans la console AWS et dans les logs si echo. Secrets Manager injecte le secret en runtime uniquement, sans le stocker dans la config du projet. L'accès est loggé dans CloudTrail.
  • Parameter Store SecureString : idéal pour les configurations semi-sensibles (endpoints, feature flags chiffrés, certificates). Gratuit jusqu'à 10 000 paramètres (Standard tier). SecureString chiffre avec KMS — on peut auditer l'accès via la key policy.
  • git-secrets (pré-commit) : l'erreur humaine est inévitable. En 2023, 12,8 millions de secrets ont été détectés dans les commits GitHub publics (GitGuardian). Le hook pré-commit est la première barrière : il empêche le secret de quitter le poste dev. Même avec le hook, on scanne aussi dans le pipeline (défense en profondeur).
  • Rotation automatique : un secret non roté est un secret compromis à terme. La rotation automatique réduit la fenêtre d'exploitation si le secret fuite. Secrets Manager fournit des templates Lambda pour RDS/Aurora/Redshift — la rotation se fait sans interruption de service (staging label + version).
  • detect-secrets dans le pipeline : même si le dev bypass le hook (force push, autre machine), le scan dans le pipeline est la deuxième barrière. Le baseline file (.secrets.baseline) permet de whitelister les faux positifs. L'entrée du pipeline est le dernier checkpoint avant que le code n'atteigne l'artifact.
  • Cross-account : dans une organisation AWS multi-comptes (landing zone), le pipeline CICD est dans un compte dédié. Les secrets de production sont dans le compte production. La resource-based policy de Secrets Manager permet un accès cross-account sans dupliquer le secret. Ajouter un VPC endpoint pour éviter que le trafic transite par internet.
  • Alerte sur GetSecretValue : si un rôle non autorisé tente de lire un secret, CloudTrail capture l'événement, EventBridge déclenche une alerte SNS, et une Lambda peut révoquer automatiquement la clé compromise. C'est la détection et réponse automatique (SOAR) pour les secrets.

6.2 — Anti-patterns Secrets courants

⚠️
Les 8 anti-patterns Secrets les plus dangereux :
  1. Secret en clair dans le buildspec — env: variables: DB_PASSWORD: "P@ssw0rd". Visible dans la console, les logs, et le repo Git. Solution : env: secrets-manager: DB_PASSWORD: "prod/myapp/db:password".
  2. Secret dans le code source — AWS_SECRET_ACCESS_KEY=AKIAxxx dans un fichier .env, config.py, ou terraform.tfvars. Le secret reste dans l'historique Git même après suppression.
  3. Pas de hook pré-commit — sans git-secrets ou detect-secrets, le développeur n'a aucun filet de sécurité. L'erreur humaine est garantie à échelle.
  4. Secrets non rotés — un API key créée il y a 3 ans, jamais changée. Si elle a fuité (et statistiquement, elle l'a), l'attaquant a 3 ans d'accès. Rotation 30-90 jours minimum.
  5. Secrets Manager sans CMK dédiée — la clé AWS managed ne permet pas de contrôler qui peut déchiffrer via la key policy. Avec une CMK dédiée, on peut restreindre kms:Decrypt aux rôles CI/CD uniquement.
  6. Duplication des secrets entre comptes — copier le secret du compte prod vers le compte CI/CD. Drift garanti : quand on roté côté prod, la copie CI/CD n'est pas mise à jour → incident. Utiliser la resource-based policy cross-account.
  7. echo $SECRET dans le buildspec — pour « debug », le dev ajoute un echo du secret. Le secret se retrouve dans les logs CloudWatch. Linter le buildspec pour interdire echo sur les variables sensibles.
  8. Pas d'audit trail — Secrets Manager loggé dans CloudTrail par défaut, mais si personne ne regarde les logs, c'est inutile. Créer des alertes EventBridge sur GetSecretValue par des rôles non attendus.

Pourquoi chaque paramètre Deployment compte :
  • Blue/Green vs AllAtOnce : avec AllAtOnce, un bug touche 100% des utilisateurs instantanément. Le rollback nécessite un nouveau déploiement (5-30 min). En Blue/Green, l'ancienne version tourne toujours — le rollback est un re-routing ALB/DNS en <60 secondes. Le coût est zéro supplémentaire (les deux target groups existent déjà).
  • Auto-rollback sur alarme : le problème n'est pas de détecter l'erreur — c'est le temps de réaction. Un humain met 5-15 min à être notifié, comprendre, et décider. Une alarme CloudWatch déclenche le rollback en 30 secondes. Règle d'or : si l'alarme 5xx rate >5% se déclenche dans les 5 premières minutes post-déploiement → rollback automatique.
  • Canary deployments : envoyer 10% du trafic pendant 5 minutes sur la nouvelle version. Si les métriques sont OK → basculer à 100%. Si les métriques sont KO → rollback. L'impact est limité à 10% des utilisateurs pendant 5 minutes, vs 100% en AllAtOnce.

  • AppSpec hooks : la validation automatique à chaque étape du déploiement. BeforeAllowTraffic = « la nouvelle version démarre-t-elle correctement ? » AfterAllowTraffic = « les requêtes sont-elles traitées sans erreur ? » Si un hook échoue, CodeDeploy rollback automatiquement.
  • CloudFormation change sets : un UPDATE_REPLACE_CLEANUP sur une base RDS = suppression de la base + recréation (perte de données). Le change set montre exactement ce qui va être modifié/remplacé/supprimé avant l'exécution. Toujours reviewer, surtout les Replace et Delete.
  • Freeze periods : déployer le vendredi soir = debugging le week-end. Déployer pendant Black Friday = risquer le pic de CA. Définir des fenêtres claires et automatiser l'enforcement (Lambda gate qui refuse les déploiements hors fenêtre).

7.3 — Anti-patterns Deployment courants

⚠️
Les 8 anti-patterns Deployment les plus coûteux :
  1. AllAtOnce en production — 100% du trafic bascule instantanément. Un bug = downtime total. Aucune possibilité de limiter l'impact. Blue/Green ou Canary = zéro coût supplémentaire avec CodeDeploy.
  2. Pas d'auto-rollback — le rollback dépend d'un humain qui doit être disponible, diagnostiquer, et agir. MTTR = 15-45 min au lieu de <60 sec. Configurer autoRollbackConfiguration.enabled=true + alarmes.
  3. Pas d'approval gate — push sur main → déploiement automatique en production. Un merge accidentel ou un code non validé passe directement en prod. Ajouter un stage Approval dans CodePipeline.
  4. Déployer le vendredi — si le déploiement échoue, l'équipe est en week-end. Implémenter une Lambda gate qui refuse les déploiements hors fenêtre autorisée.

Blue/Green ou Canary utilisé pour tous les déploiements production (jamais AllAtOnce)

CloudFormation change sets utilisés systématiquement (review des Replace/Delete)
ECS circuit breaker activé avec rollback=true

Fenêtre de déploiement définie (mardi-jeudi, pas le vendredi)
Freeze periods configurés (Black Friday, fêtes, migrations critiques)
Ancienne version conservée 60 min après déploiement (Blue instances termination wait)
Drift detection activée sur les stacks CloudFormation production
🚨
Scénario d'incident : Vendredi 17h — un développeur pousse un changement de schéma de base de données incompatible avec le code existant. Le pipeline déploie directement en production (déploiement AllAtOnce, pas d'approval gate, pas de rollback automatique). 100% du trafic bascule immédiatement sur la nouvelle version. Les requêtes SQL échouent, le taux d'erreur HTTP 500 passe à 80%. L'équipe met 45 minutes à identifier le problème et 30 minutes supplémentaires pour un rollback manuel (re-déployer l'ancienne version). Résultat : 1h15 d'indisponibilité complète. Avec un Blue/Green + auto-rollback sur alarme CloudWatch, le rollback aurait pris <60 secondes.

Analogie : 🚦 Le déploiement sécurisé, c'est le frein d'urgence du TGV. Le Blue/Green est l'aiguillage ferroviaire : vous avez deux voies (ancienne et nouvelle version), et vous basculez le trafic progressivement. Si un signal d'alarme retentit (CloudWatch alarm), le frein automatique (auto-rollback) ramène tout le trafic sur la voie sûre en secondes, sans intervention humaine. Le déploiement Canary est le canari dans la mine : on envoie 10% du trafic d'abord, et si le canari meurt (erreurs), on arrête immédiatement.

Pourquoi chaque paramètre Deployment compte :
  • Blue/Green vs AllAtOnce : avec AllAtOnce, un bug touche 100% des utilisateurs instantanément. Le rollback nécessite un nouveau déploiement (5-30 min). En Blue/Green, l'ancienne version tourne toujours — le rollback est un re-routing ALB/DNS en <60 secondes. Le coût est zéro supplémentaire (les deux target groups existent déjà).
  • Auto-rollback sur alarme : le problème n'est pas de détecter l'erreur — c'est le temps de réaction. Un humain met 5-15 min à être notifié, comprendre, et décider. Une alarme CloudWatch déclenche le rollback en 30 secondes. Règle d'or : si l'alarme 5xx rate >5% se déclenche dans les 5 premières minutes post-déploiement → rollback automatique.
  • Canary deployments : envoyer 10% du trafic pendant 5 minutes sur la nouvelle version. Si les métriques sont OK → basculer à 100%. Si les métriques sont KO → rollback. L'impact est limité à 10% des utilisateurs pendant 5 minutes, vs 100% en AllAtOnce.
  • Approval gate : la dernière barrière humaine avant la production. Empêche les déploiements accidentels (merge sur main → pipeline automatique → prod). Configurer une notification SNS (Slack/email) pour que l'approbateur soit prévenu immédiatement.
  • AppSpec hooks : la validation automatique à chaque étape du déploiement. BeforeAllowTraffic = « la nouvelle version démarre-t-elle correctement ? » AfterAllowTraffic = « les requêtes sont-elles traitées sans erreur ? » Si un hook échoue, CodeDeploy rollback automatiquement.
  • CloudFormation change sets : un UPDATE_REPLACE_CLEANUP sur une base RDS = suppression de la base + recréation (perte de données). Le change set montre exactement ce qui va être modifié/remplacé/supprimé avant l'exécution. Toujours reviewer, surtout les Replace et Delete.
  • Freeze periods : déployer le vendredi soir = debugging le week-end. Déployer pendant Black Friday = risquer le pic de CA. Définir des fenêtres claires et automatiser l'enforcement (Lambda gate qui refuse les déploiements hors fenêtre).

7.3 — Anti-patterns Deployment courants

⚠️
Les 8 anti-patterns Deployment les plus coûteux :
  1. AllAtOnce en production — 100% du trafic bascule instantanément. Un bug = downtime total. Aucune possibilité de limiter l'impact. Blue/Green ou Canary = zéro coût supplémentaire avec CodeDeploy.
  2. Pas d'auto-rollback — le rollback dépend d'un humain qui doit être disponible, diagnostiquer, et agir. MTTR = 15-45 min au lieu de <60 sec. Configurer autoRollbackConfiguration.enabled=true + alarmes.
  3. Pas d'approval gate — push sur main → déploiement automatique en production. Un merge accidentel ou un code non validé passe directement en prod. Ajouter un stage Approval dans CodePipeline.
  4. Déployer le vendredi — si le déploiement échoue, l'équipe est en week-end. Implémenter une Lambda gate qui refuse les déploiements hors fenêtre autorisée.
  5. Pas de health check dans AppSpec — CodeDeploy considère le déploiement réussi même si l'application ne démarre pas (pas de hook de validation). Ajouter BeforeAllowTraffic avec un curl sur /health.
  6. CloudFormation deploy direct (sans change set) — aws cloudformation update-stack applique les changements sans review. Un Replace sur une base RDS = perte de données. Toujours créer un change set d'abord.
  7. ECS rolling update sans circuit breaker — le service ECS tente indéfiniment de déployer une image qui crashe. Activer deploymentCircuitBreaker avec rollback=true pour arrêter après N échecs.
  8. Pas de timeout sur le déploiement — un déploiement bloqué (health check en boucle) reste en cours pendant des heures. Configurer un timeout (15-30 min) pour forcer l'échec et le rollback.

Analogie : 📖 La gouvernance CI/CD, c'est le registre du notaire. Chaque acte (déploiement) est daté, signé (approbateur), et archivé. On peut remonter à n'importe quel moment pour prouver qui a fait quoi, quand, et pourquoi. L'approval gate est la signature du notaire : sans elle, l'acte n'est pas valide. CloudTrail est le livre de comptes certifié : infalsifiable et audité.

💡
Pourquoi chaque paramètre Compliance compte :
  • CloudTrail multi-région : sans CloudTrail, il est impossible de prouver qui a déployé quoi et quand. C'est la base de tout audit (SOC 2, ISO 27001, PCI-DSS). Multi-région car un attaquant peut créer des ressources dans une région non monitorée. Object Lock (WORM) empêche la suppression/modification des logs même par un admin compromis.
  • Archivage long terme : SOC 2 Type II exige 1 an de rétention minimum. RGPD/HDS peut exiger 3-7 ans. L'historique CodePipeline natif ne couvre que 12 mois. EventBridge → S3 + Athena permet de requêter l'historique sur des années (« qui a déployé en prod le 15 mars 2024 ? »).
  • Ségrégation des rôles : le principe de separation of duties (SOC 2 CC6.1) : celui qui écrit le code ne doit pas pouvoir déployer en production. Celui qui déploie ne doit pas pouvoir modifier le pipeline. IAM policies + SCP enforcés au niveau Organizations.
  • Approval actions : la preuve que quelqu'un a validé le déploiement. L'approbateur est loggé dans l'historique CodePipeline et CloudTrail. C'est l'équivalent de la double signature bancaire pour les montants importants.
  • Cross-account : la blast radius reduction — si le compte dev est compromis, l'attaquant ne peut pas accéder à la production (pas de credentials, pas de rôles). Le pipeline CICD est le seul chemin vers la production, et il passe par les stages de sécurité.
  • Service Catalog : empêche les équipes de créer des pipelines « sauvages » sans SAST, sans approval, sans rollback. Le template standardisé inclut tous les stages de sécurité par défaut. Les contraintes empêchent de supprimer ces stages.
  • Pipeline-as-Code : les modifications de pipeline sont auditées via Git (qui, quand, quoi). Les PR reviews détectent les changements risqués (suppression d'un stage SAST, modification des rôles IAM). Le state Terraform locké empêche les modifications concurrentes.

8.2 — Anti-patterns Compliance courants

⚠️
Les 7 anti-patterns Compliance les plus risqués :
  1. Pas de CloudTrail multi-région — les événements ne sont loggés que dans la région principale. Un attaquant crée un pipeline malveillant dans ap-southeast-1 → invisible. Activer le trail dans toutes les régions.
  2. Pipelines créés via la console — aucune trace Git de la configuration, aucune review. Si quelqu'un supprime un stage SAST, personne ne le sait. Pipeline-as-Code (Terraform) + PR obligatoire.
  3. Même rôle pour dev et deploy — un développeur peut modifier le pipeline ET déployer en prod. Violation de la separation of duties. Rôles séparés + SCP au niveau Organizations.
  4. Pas d'archivage long terme — l'historique CodePipeline expire après 12 mois. L'audit SOC 2 demande les déploiements de l'année précédente → données manquantes → finding critique.
  5. Déploiement manuel (aws ecs update-service) — contourne tous les stages de sécurité (SAST, approval, tests). Interdire via IAM + SCP les actions de déploiement direct sauf depuis le pipeline.
  6. Pas de notification sur les événements pipeline — un pipeline échoue silencieusement. Personne ne sait que le déploiement de sécurité n'a pas passé. SNS + Slack pour tous les événements critiques.
  7. Tout dans un seul compte AWS — dev, staging, prod dans le même compte. Si le compte est compromis, tout est compromis. Architecture multi-comptes (Organizations) + pipeline cross-account via AssumeRole.

💡 Pourquoi chaque type de test est nécessaire :
• Unit tests : détectent les erreurs de logique métier au plus tôt (coût de correction ×1)
• Integration tests : valident les contrats d'API et les interactions entre services (coût de correction ×10 si tardif)
• DAST : identifie les vulnérabilités runtime (XSS, IDOR, injection) invisibles en analyse statique
• Load tests : préviennent les régressions de performance avant qu'elles impactent les SLA
• Quality gate : garantit qu'aucun code ne passe en production sans validation automatisée

💡 Pourquoi migrer vers CodePipeline V2 :
• V2 est gratuit au niveau pipeline — seules les exécutions d'actions sont facturées ($0.002/action)
• V2 supporte les déclencheurs avancés (filtres de branches, chemins, tags Git)
• V2 intègre les variables de pipeline pour paramétrisation sans duplication
• Le cache S3 CodeBuild réduit les temps de build de 40–70% sur des projets avec dépendances volumineuses
• Les builds parallèles dans un stage réduisent le chemin critique du pipeline
⚠️ Pièges à éviter :
• Utiliser BUILD_GENERAL1_LARGE systématiquement — commencer par SMALL, scaler uniquement si nécessaire
• Cache S3 sans invalidation de clé correcte — un cache périmé peut introduire des régressions
• Migrer vers V2 sans tester les filtres de déclenchement — risque de pipelines qui ne se déclenchent plus
• Conserver des artifacts S3 indéfiniment — coût croissant et non visible
• Ne pas surveiller les coûts avec AWS Cost Explorer par tag projet

function Write-Check($label, $ok, $detail = "") {
    if ($ok) { Write-Host "  [OK] $label" -ForegroundColor Green }
    else      { Write-Host "  [KO] $label — $detail" -ForegroundColor Red }
}
$score = 0; $total = 0

# ── [1/7] S1 — Pipeline Architecture ──
Write-Host "`n[1/7] S1 — Pipeline Architecture" -ForegroundColor Cyan
$pipe = aws codepipeline get-pipeline --name $Pipeline --profile $Profile --region $Region | ConvertFrom-Json
$isV2 = $pipe.pipeline.pipelineType -eq "V2"; $total++
Write-Check "Pipeline type V2" $isV2 "Migrer vers V2 (S10)"
if ($isV2) { $score++ }

$encKey = $pipe.pipeline.artifactStore.encryptionKey.id; $total++
Write-Check "Artifact store chiffré (KMS CMK)" ($encKey -ne $null) "Configurer CMK KMS (S1)"
if ($encKey) { $score++ }

# ── [2/7] S2 — CodeBuild Security ──
Write-Host "`n[2/7] S2 — CodeBuild Security" -ForegroundColor Cyan
$proj = aws codebuild batch-get-projects --names $Project --profile $Profile --region $Region | ConvertFrom-Json
$env = $proj.projects[0].environment
$total++
Write-Check "privilegedMode désactivé" (-not $env.privilegedMode) "Désactiver privilegedMode (S2)"
if (-not $env.privilegedMode) { $score++ }

$hasVpc = $proj.projects[0].vpcConfig.vpcId -ne $null; $total++
Write-Check "CodeBuild dans VPC privé" $hasVpc "Configurer VPC CodeBuild (S2)"
if ($hasVpc) { $score++ }

# ── [3/7] S3 — SAST ──
Write-Host "`n[3/7] S3 — SAST & Code Analysis" -ForegroundColor Cyan
$csIntegrations = aws inspector2 list-code-security-integrations --profile $Profile --region $Region 2>$null | ConvertFrom-Json
$csConfigs = aws inspector2 list-code-security-scan-configurations --profile $Profile --region $Region 2>$null | ConvertFrom-Json
$hasIntegration = (($csIntegrations.integrations | Where-Object { $_.status -eq "ACTIVE" }).Count -gt 0)
$hasScan = $hasIntegration -and ($csConfigs.configurations.Count -gt 0); $total++
Write-Check "Amazon Inspector Code Security — intégration + scan config" $hasScan "Configurer Inspector Code Security (S3)"
if ($hasScan) { $score++ }

# ── [4/7] S4 — Container Security ──
Write-Host "`n[4/7] S4 — Container Security" -ForegroundColor Cyan
$repos = aws ecr describe-repositories --profile $Profile --region $Region | ConvertFrom-Json
foreach ($repo in $repos.repositories) {
    $scanOnPush = $repo.imageScanningConfiguration.scanOnPush; $total++
    Write-Check "ECR scanOnPush [$($repo.repositoryName)]" $scanOnPush "Activer scanOnPush (S4)"
    if ($scanOnPush) { $score++ }
}

# ── [5/7] S6 — Secrets ──
Write-Host "`n[5/7] S6 — Secrets in Pipelines" -ForegroundColor Cyan
$secrets = aws secretsmanager list-secrets --profile $Profile --region $Region | ConvertFrom-Json
$noRotation = $secrets.SecretList | Where-Object { -not $_.RotationEnabled }
$total++
Write-Check "Tous les secrets avec rotation activée" ($noRotation.Count -eq 0) "$($noRotation.Count) secret(s) sans rotation (S6)"
if ($noRotation.Count -eq 0) { $score++ }

# ── [6/7] S9 — Tests ──
Write-Host "`n[6/7] S9 — Testing Integration" -ForegroundColor Cyan
$rg = aws codebuild list-report-groups --profile $Profile --region $Region | ConvertFrom-Json
$total++
Write-Check "Report Group CodeBuild configuré" ($rg.reportGroups.Count -gt 0) "Créer un report group (S9)"
if ($rg.reportGroups.Count -gt 0) { $score++ }

# ── [7/7] Rapport final ──
Write-Host "`n═══════════ RAPPORT FINAL ═══════════" -ForegroundColor Yellow
$pct = [math]::Round(($score / $total) * 100)
Write-Host "Score : $score / $total ($pct%)"
if ($pct -ge 80) { Write-Host "Niveau : OPTIMAL ✅" -ForegroundColor Green }
elseif ($pct -ge 50) { Write-Host "Niveau : AVANCÉ ⚡" -ForegroundColor Yellow }
else { Write-Host "Niveau : INSUFFISANT ❌ — Appliquer les Quick Wins" -ForegroundColor Red }
