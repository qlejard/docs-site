Mode d'emploi de ce guide

📋
Ce guide couvre le déploiement complet de Microsoft Defender for Identity (MDI) : installation des capteurs sur les contrôleurs de domaine, configuration des détections d'attaques identité (lateral movement, vol de credentials, persistence Kerberos), tuning des faux positifs, et intégration avec Sentinel / Defender XDR. Chaque section est scorée /100 pour mesurer votre niveau de maturité.
🚨
Particularité MDI : contrairement aux protections cloud (EOP, Entra ID), MDI est quasi inopérant sans déploiement actif. Les capteurs doivent être installés sur chaque DC, les audit policies Windows configurées, et la résolution de noms vérifiée. Score par défaut attendu : ~10/100 (MDI est licencié mais ne détecte rien tant que les capteurs ne sont pas installés). Analogie : MDI est une caméra de vidéosurveillance, mais elle est livrée dans son carton — il faut l'installer, la brancher et orienter l'objectif.

Stratégie de déploiement recommandée

1
Quick Win — Installer les capteurs — ~2h, +40 pts. Déployer le sensor MDI sur chaque DC (silent install MSI), créer le compte gMSA, vérifier la communication vers le cloud MDI (port 443).
2
Configurer les audits Windows — ~1h, +22 pts. Déployer les Advanced Audit Policies via GPO (Logon/Logoff, Kerberos, Directory Service Access). Sans ces audits, MDI ne voit que le trafic réseau.
3
Configurer NNR + Tuning — ~2h, +16 pts. Vérifier la résolution de noms (NTLM over RPC, DNS reverse), configurer les exclusions, déployer des honeypot accounts.
4
Intégration & Posture — ~4h, +12 pts. Connecter à Sentinel (IdentityLogonEvents), résoudre les Identity Security Posture Assessments, mettre en place les routines de maintenance.

1
Prérequis & Architecture MDI AD On-Prem
Ports réseau, sizing capteurs, compte gMSA, capacity planning — le socle sans lequel MDI ne fonctionne pas
🚨
Point de blocage #1 des déploiements MDI : les ports réseau. Le capteur MDI doit pouvoir communiquer avec le cloud Microsoft (port 443) ET résoudre les noms via des protocoles spécifiques (RPC, DNS, LDAP). Si un firewall bloque un seul de ces flux, le capteur apparaît « Not communicating » dans le portail et aucune détection ne fonctionne.
ℹ️
Analogie : le capteur MDI est une caméra de vidéosurveillance installée directement sur le contrôleur de domaine. Il voit tout le trafic d'authentification (Kerberos, NTLM, LDAP) qui transite par le DC. Sans la caméra (capteur) et sans alimentation (électrique = ports réseau), vous avez payé le système de surveillance mais vous ne voyez rien.

1.4 — Création du compte gMSA

PowerShell — Création du gMSA pour MDI
# Prérequis : KDS Root Key (si pas déjà créée)
# Vérifier si une clé existe déjà
Get-KdsRootKey

# Si aucune clé : créer (attendre 10h de réplication en prod)
# En lab uniquement : -EffectiveImmediately (ne PAS utiliser en production)
Add-KdsRootKey -EffectiveTime ((Get-Date).AddHours(-10))

# Créer le gMSA pour MDI
New-ADServiceAccount -Name "gMSA-MDI" `
    -DNSHostName "gMSA-MDI.domaine.local" `
    -Description "Compte gMSA pour Microsoft Defender for Identity" `
    -PrincipalsAllowedToRetrieveManagedPassword "Domain Controllers" `
    -KerberosEncryptionType AES128, AES256

# Vérifier l'installation sur chaque DC
Test-ADServiceAccount -Identity "gMSA-MDI"

⚠️
Pourquoi chaque prérequis compte :
  • Port 443 : sans lui, le capteur ne peut pas envoyer les données au cloud MDI → aucune détection
  • gMSA : sans compte dédié, MDI ne peut pas interroger l'AD pour résoudre les SID/noms → alertes incomplètes
  • .NET 4.7+ : le capteur MDI est une application .NET — sans la bonne version il ne démarre pas
  • Sizing CPU/RAM : un capteur sous-dimensionné droppe des paquets → événements d'authentification manqués → faux négatifs

1.6 — RBAC & Permissions MDI

MDI utilise un modèle RBAC intégré au portail Microsoft Defender XDR. Les rôles sont attribués via Entra ID (anciennement Azure AD).

Rôle Entra IDAccès MDICas d'usage
Security Administrator Full access — configuration, exclusions, tags, capteurs Admin MDI principal, responsable du déploiement
Security Reader Lecture seule — alertes, timeline, health Analyste SOC L1/L2, auditeur
Global Administrator Full access (hérité) À éviter — utiliser Security Administrator
Custom RBAC (Unified RBAC) Granulaire par workload (preview) Organisations matures souhaitant séparer MDI/MDE/MDO
⚠️
Principe du moindre privilège : ne pas attribuer Global Administrator pour administrer MDI. Le rôle Security Administrator suffit pour toutes les opérations MDI (déploiement capteurs, configuration exclusions, gestion des tags). Pour le SOC, Security Reader est suffisant pour l'investigation quotidienne.

2.4 — Capteur AD CS (Active Directory Certificate Services)

Si votre environnement utilise une PKI interne (AD CS), MDI peut déployer un capteur spécifique sur les serveurs CA pour détecter les attaques de type ESC1-ESC8 (abus de modèles de certificats).

Prérequis AD CS sensorDétail
Serveur cibleChaque serveur hébergeant le rôle AD CS (Enterprise CA)
InstallationMême MSI que le capteur DC — MDI détecte automatiquement le rôle AD CS
Audits requisEvent 4886 (Certificate Services received a certificate request) — activer via certutil -setreg policy\EditFlags +EDITF_AUDITCERTTEMPLATELOAD
Détections couvertesESC1 (misconfigured template), ESC8 (NTLM relay to CA), demande de certificat suspecte, modification de template
ℹ️
Si votre organisation n'utilise pas AD CS, cette étape est non applicable. Vérifiez avec certutil -config - -ping — si la commande échoue, il n'y a pas de CA Enterprise active.
⚠️
Troubleshoot « Sensor not communicating » :
  • Port 443 bloqué : vérifier le firewall → Test-NetConnection *.atp.azure.com -Port 443
  • Proxy non configuré : si le DC passe par un proxy, ajouter ProxyUrl dans la config ou réinstaller avec le paramètre
  • TLS inspection : les proxies avec TLS inspection cassent le certificat MDI → exclure *.atp.azure.com
  • Clé d'accès expirée : régénérer la clé dans le portail et réinstaller le capteur
  • NTP désync : si l'horloge du DC dérive de > 5 min, le token d'auth expire → vérifier w32tm /query /status
⏳
Période d'apprentissage (~30 jours) : après le déploiement des capteurs, MDI entre en phase de Learning Period. Pendant cette période, le moteur comportemental construit les baselines (horaires de connexion, machines utilisées, ressources accédées par chaque compte).

Conséquences opérationnelles :
  • Alertes comportementales absentes : les détections de type « Abnormal » (connexion inhabituelle, accès anormal à une ressource) ne se déclenchent pas tant que la baseline n'est pas construite
  • Détections signature actives immédiatement : les alertes basées sur des patterns connus (DCSync, Pass-the-Hash, Golden Ticket) fonctionnent dès J+0
  • Ne pas tuner pendant l'apprentissage : les faux positifs des 30 premiers jours ne sont pas représentatifs — attendre la fin de la période avant d'ajuster les exclusions
  • Couverture complète requise : la baseline n'est fiable que si 100% des DC sont couverts. Un capteur ajouté plus tard relance l'apprentissage pour les comptes de ce DC
💡
Pourquoi chaque paramètre compte :
  • Capteur direct vs standalone : le capteur direct est préféré car il ne nécessite pas de port mirroring physique — installation plus simple, maintenance réduite, couverture immédiate
  • gMSA vs compte classique : le gMSA élimine le risque de mot de passe expiré/faible qui interromprait les détections MDI sans alerte visible
  • Auto-update sensor : Microsoft publie ~2 mises à jour/mois. Sans auto-update, le capteur accumule du retard et perd la capacité de détecter les nouvelles techniques
  • Couverture 100% DC : un seul DC non couvert est un angle mort total — un attaquant DCSync depuis ce DC ne sera pas détecté
  • Delayed update 48h en prod : retarder permet que les capteurs lab détectent les régressions avant le déploiement sur les DC de production

Résolution de noms (NNR) MDI AD On-Prem
Network Name Resolution — la capacité de MDI à associer des IP à des comptes. Impact direct sur la qualité des détections.
🚨
NNR défaillant = détections dégradées. MDI capture le trafic réseau (Kerberos, NTLM, LDAP) et voit des adresses IP. Pour générer des alertes exploitables, il doit résoudre ces IP en noms de machines et comptes utilisateurs. Si la NNR échoue, les alertes affichent des IP brutes au lieu de noms → impossible d'investiguer efficacement. Analogie : c'est comme une caméra de sécurité qui filme des silhouettes floues au lieu de visages identifiables.

3.1 — Méthodes de résolution (ordre de priorité)

Méthode NNR Protocole / Port Défaut MDI ⚡ Recommandé 🎯 Best Practice Impact si absent
NTLM over RPC RPC (port 135 + dynamiques) Désactivé Activé Activé — meilleure précision Perte de résolution pour les sessions NTLM
DNS reverse lookup DNS (port 53) Actif Actif Actif + zones reverse (PTR) à jour Résolution partielle uniquement
NetBIOS (NBNS) UDP 137 Actif Actif Actif (fallback) Pas de fallback pour les anciens clients
RDP Event Parsing Event ID 4624 (type 10) Actif (si audits OK) Actif Actif + audits configurés Pas de résolution pour connexions RDP
ETW Events Event Tracing for Windows Inactif si audits manquants Actif (audit policies) Actif Perte de contexte pour les détections avancées
⚠️
Piège DNS reverse : les zones de recherche inversée (PTR) ne sont pas toujours maintenues à jour dans les environnements AD. Vérifier que les enregistrements PTR sont créés automatiquement (option « Register this connection's addresses in DNS » sur les clients + DHCP « Always dynamically update DNS records »). Sans PTR, MDI ne peut pas résoudre les IP des postes clients en noms de machines.

3.2 — Configuration

PowerShell — Vérifier et configurer NNR
# Vérifier les zones DNS reverse existantes
Get-DnsServerZone | Where-Object { $_.IsReverseLookupZone } | Format-Table ZoneName, ZoneType, IsDsIntegrated

# Créer une zone reverse si manquante (exemple pour 10.0.0.0/24)
Add-DnsServerPrimaryZone -NetworkId "10.0.0.0/24" -ReplicationScope "Domain"

# Vérifier la résolution reverse pour une IP spécifique
Resolve-DnsName -Name "10.0.0.50" -Type PTR

# Vérifier que le DHCP met à jour les PTR automatiquement
Get-DhcpServerv4Scope | Get-DhcpServerv4DnsSetting | Format-List DynamicUpdates, DeleteDnsRROnLeaseExpiry

# Tester la résolution RPC (NTLM over RPC)
# Depuis le DC où le capteur est installé :
Test-NetConnection -ComputerName "poste-client.domaine.local" -Port 135

3.3 — Vérification de la santé NNR

1
Ouvrir security.microsoft.com → Settings → Identities → Health issues
2
Chercher les alertes « Network Name Resolution failure » ou « Some network activities could not be mapped to identities »
3
Vérifier le pourcentage de résolution : > 90 % = bon, < 70 % = problème réseau critique à investiguer
ℹ️
Pourquoi chaque méthode NNR compte :
  • NTLM over RPC : seule méthode qui résout l'IP en utilisateur (pas juste en machine). Critique pour le lateral movement path.
  • DNS reverse : résolution rapide IP → hostname. Fiable si les PTR sont à jour.
  • NetBIOS : fallback pour les anciens systèmes qui n'ont pas de PTR DNS. Bruité mais nécessaire.
  • ETW Events : enrich les détections avec les Event IDs Windows (4624 type 10 = RDP logon).

Le mouvement latéral est la phase où l'attaquant, après avoir compromis un poste, se déplace de machine en machine pour atteindre ses objectifs (DC, serveur de fichiers, tier 0). Analogie : un cambrioleur qui passe de pièce en pièce dans un bâtiment — MDI détecte chaque porte forcée.

4.2 — Comprendre Pass-the-Hash vs Pass-the-Ticket

Comparaison PtH / PtT / Overpass-the-Hash
Pass-the-Hash (PtH)
L'attaquant vole le hash NTLM d'un compte (via mimikatz, secretsdump). Il l'utilise pour s'authentifier en NTLM sans connaître le mot de passe.
🔍 MDI détecte l'utilisation anormale du hash : même hash utilisé depuis une IP différente.
Pass-the-Ticket (PtT)
L'attaquant vole un ticket Kerberos (TGT ou TGS) en mémoire. Il l'injecte dans sa session pour accéder aux ressources sans s'authentifier à nouveau.
🔍 MDI détecte le ticket utilisé depuis une machine différente de celle qui l'a demandé.
Overpass-the-Hash
L'attaquant utilise un hash NTLM pour demander un ticket Kerberos (passage de NTLM à Kerberos).
🔍 MDI détecte le downgrade de chiffrement : TGT demandé avec RC4 alors que le compte supporte AES.

4.3 — Lateral Movement Paths (LMP)

ℹ️
Qu'est-ce qu'un chemin de mouvement latéral ?
MDI analyse en permanence les connexions Logon (Event 4624) pour construire un graphe des machines accessibles depuis chaque compte. Si un attaquant compromet un poste utilisateur, le LMP montre le chemin le plus court jusqu'au DC (« combien de sauts pour atteindre le Tier 0 ? »).

Prérequis : NNR fonctionnel (S3) + audits Logon complets (événements 4624/4625 activés sur tous les DC).

6.2 — Comprendre DCSync

DCSync — Réplication malveillante de l'AD
1. Compromission
Attaquant obtient un compte avec droits Replicating Directory Changes (All) (Domain Admin, compte de réplication, ACL déléguée).
2. DRS GetNCChanges
L'attaquant envoie une requête DRS GetNCChanges (protocole de réplication AD) depuis une machine non-DC.
3. Extraction hashes
Le DC répond en envoyant les hashes NTLM de tous les comptes demandés (y compris krbtgt).
4. Détection MDI
MDI détecte la requête DRS provenant d'une machine qui n'est pas un DC → alerte immédiate.

7.2 — Golden Ticket vs Silver Ticket

Comparaison Golden Ticket / Silver Ticket
👑 Golden Ticket
Qu'est-ce que c'est ? Un TGT Kerberos forgé en utilisant le hash du compte krbtgt.

Impact : L'attaquant peut demander un TGS pour n'importe quel service dans le domaine, avec n'importe quelle identité et n'importe quels privilèges.

Durée de vie : Par défaut 10 heures (mais l'attaquant peut mettre 10 ans dans le ticket forgé).

Remédiation : Reset du mot de passe krbtgt 2 fois (car AD conserve N et N-1).
🪙 Silver Ticket
Qu'est-ce que c'est ? Un TGS Kerberos forgé en utilisant le hash d'un compte de service (SPN).

Impact : Accès limité à un seul service (ex: CIFS sur un file server, HTTP sur un web server).

Avantage attaquant : Ne passe pas par le KDC → pas de log Kerberos côté DC. Plus furtif que le Golden Ticket.

Remédiation : Reset du mot de passe du compte de service ciblé, idéalement migration vers gMSA.

7.3 — Skeleton Key — le backdoor ultime

💀
Skeleton Key est un malware qui patche le processus LSASS en mémoire sur un DC. Après injection, n'importe quel mot de passe est accepté pour n'importe quel compte, tout en laissant les vrais mots de passe fonctionner.

MDI détecte : le chiffrement anormal dans les réponses d'authentification (downgrade forcé en RC4).
Particularité : ne survit pas au redémarrage du DC (en mémoire uniquement). Redémarrer le DC suffit à supprimer le backdoor, mais il faut d'abord identifier comment l'attaquant a obtenu l'accès au DC.

7.4 — Vérifications et remédiation

PowerShell — Vérifications anti-persistence
# Vérifier l'âge du mot de passe krbtgt (Golden Ticket mitigation)
Get-ADUser krbtgt -Properties PasswordLastSet |
    Select-Object Name, PasswordLastSet,
    @{N="AgeDays";E={(Get-Date) - $_.PasswordLastSet | Select-Object -Expand Days}}

# Si >180 jours, planifier un double reset :
# Reset 1 → attendre la réplication complète → Reset 2
# Set-ADAccountPassword -Identity krbtgt -Reset -NewPassword (ConvertTo-SecureString "..." -AsPlainText -Force)

# Détecter les comptes avec SID-History suspect
Get-ADUser -Filter {SIDHistory -ne "$null"} -Properties SIDHistory |
    Select-Object Name, SamAccountName, SIDHistory

# Vérifier les ACL du container AdminSDHolder
$adminSDHolder = "CN=AdminSDHolder,CN=System," + (Get-ADRootDSE).defaultNamingContext
Get-Acl "AD:\$adminSDHolder" |
    Select-Object -ExpandProperty Access |
    Where-Object { $_.IdentityReference -notmatch "BUILTIN|NT AUTHORITY|S-1-5-32" } |
    Format-Table IdentityReference, ActiveDirectoryRights, AccessControlType

# Vérifier les Event IDs Skeleton Key (service installé par mimikatz)
Get-WinEvent -FilterHashtable @{LogName='System'; Id=7045} -MaxEvents 20 |
    Where-Object { $_.Message -match "misspell|DVDL|mskelper" }
⚠️
Piège krbtgt reset : le double reset est obligatoire car Active Directory conserve le hash actuel (N) et le précédent (N-1). Si vous ne faites qu'un seul reset, les Golden Tickets forgés avec l'ancien hash fonctionnent encore via N-1.
Espacer les deux resets de 12-24h minimum pour laisser le temps à la réplication de converger.
Attention : le premier reset peut invalider temporairement des tickets légitimes → planifier en heures creuses.
💡
Pourquoi chaque paramètre compte :
  • Âge krbtgt ≤ 180 jours : plus le hash krbtgt est ancien, plus la fenêtre d'exploitation d'un Golden Ticket est large. Un double reset annuel est le minimum
  • Audit Account Management : sans cet audit, les modifications de SID-History passent inaperçues — l'attaquant s'octroie des privilèges silencieusement
  • Audit DS Access pour AdminSDHolder : les ACL modifiées sur AdminSDHolder se propagent en 60 min à tous les groupes protégés — l'impact est massif et discret
  • Monitoring Event 7045 : l'installation de Skeleton Key crée un service — sans surveillance du log System, ce backdoor passe inaperçu

  • SPN sur le honeytoken : piège les attaquants qui font du Kerberoasting — technique très courante en phase de credential access

9.3 — Remédiations rapides (Quick Wins)

PowerShell — Remédiations posture identité
# 1. Comptes avec pré-auth désactivée (AS-REP Roasting)
Get-ADUser -Filter {DoesNotRequirePreAuth -eq $true} -Properties DoesNotRequirePreAuth |
    ForEach-Object {
        Set-ADAccountControl -Identity $_ -DoesNotRequirePreAuth $false
        Write-Host "[FIX] Pre-auth activée pour $($_.SamAccountName)"
    }

# 2. Délégation Kerberos non contrainte (hors DC)
Get-ADComputer -Filter {TrustedForDelegation -eq $true} -Properties TrustedForDelegation |
    Where-Object { $_.DistinguishedName -notmatch "Domain Controllers" } |
    Select-Object Name, DNSHostName, TrustedForDelegation

# 3. Print Spooler actif sur les DC
$DCs = Get-ADDomainController -Filter * | Select-Object -Expand HostName
foreach ($dc in $DCs) {
    $svc = Get-Service -Name Spooler -ComputerName $dc -ErrorAction SilentlyContinue
    if ($svc.Status -eq "Running") {
        Write-Host "[WARN] Spooler actif sur $dc - désactiver !" -ForegroundColor Red
    }
}

# 4. Comptes dormants dans les groupes sensibles
$sensitiveGroups = @("Domain Admins", "Enterprise Admins", "Schema Admins", "Administrators")
foreach ($group in $sensitiveGroups) {
    Get-ADGroupMember $group | Get-ADUser -Properties LastLogonDate |
        Where-Object { $_.LastLogonDate -lt (Get-Date).AddDays(-180) } |
        Select-Object @{N="Group";E={$group}}, Name, LastLogonDate, Enabled
}
✅
Impact sur le Secure Score : les assessments MDI contribuent directement au Microsoft Secure Score. Chaque remédiation appliquée augmente le score global. MDI réévalue automatiquement toutes les 24h. L'objectif est de n'avoir aucune recommandation « Not started » dans les assessments MDI.
💡
Pourquoi chaque paramètre compte :
  • Unconstrained delegation : un serveur avec délégation non contrainte peut impersonner n'importe quel utilisateur — c'est un chemin direct vers le Tier 0
  • Comptes dormants dans groupes sensibles : un compte inactif dans Domain Admins est une cible idéale — personne ne surveille ses connexions
  • Print Spooler sur DC : le service Spooler est le vecteur de PrintNightmare et PetitPotam (coercion d'authentification) — le désactiver ne coûte rien
  • Weak cipher (RC4/DES) : les chiffrements faibles facilitent le cracking offline des tickets Kerberos — forcer AES rallonge le temps de crack de minutes à siècles
⚠️
Piège courant : les Identity Security Posture Assessments sont souvent ignorés car non bloquants. Pourtant, chaque assessment non résolu (ex : unconstrained delegation, comptes dormants Tier 0) représente un chemin d'attaque exploitable. Traiter ces recommandations comme des vulnérabilités, pas comme des suggestions.

KQL — Détection d'anomalies d'authentification
// Authentifications NTLM suspectes (devrait être rare si Kerberos fonctionne)
IdentityLogonEvents
| where Timestamp > ago(7d)
| where Protocol == "NTLM"
| summarize NTLMCount=count() by AccountName, DeviceName
| where NTLMCount > 50
| sort by NTLMCount desc

// Modifications de groupes sensibles (tier 0)
IdentityDirectoryEvents
| where Timestamp > ago(30d)
| where ActionType == "Group Membership changed"
| where TargetAccountDisplayName in ("Domain Admins", "Enterprise Admins", "Schema Admins")
| project Timestamp, AccountName, TargetAccountDisplayName, AdditionalFields

// Enumérations LDAP massives (reconnaissance)
IdentityQueryEvents
| where Timestamp > ago(7d)
| where QueryType == "LDAP"
| summarize QueryCount=count() by DeviceName, AccountName, bin(Timestamp, 1h)
| where QueryCount > 500
| sort by QueryCount desc

// Corrélation cross-domain : alerte MDI + activité MDE sur le même device
AlertInfo
| where ServiceSource == "Microsoft Defender for Identity"
| join kind=inner (AlertEvidence | where EntityType == "Machine") on AlertId
| join kind=inner (DeviceEvents | where Timestamp > ago(1h)) on $left.DeviceName == $right.DeviceName
| project AlertTime=Timestamp, AlertTitle=Title, DeviceName, DeviceAction=ActionType

# ---- S3 : DNS Reverse (NNR) ----

# ---- S4/S6 : Audit Policies ----
Write-Host "`n[S4/S6] Audit Policies" -ForegroundColor Yellow
$auditChecks = @(
    @{Name="Logon"; Expected="Success and Failure"},
    @{Name="Kerberos Authentication Service"; Expected="Success and Failure"},
    @{Name="Kerberos Service Ticket Operations"; Expected="Success and Failure"},
    @{Name="Credential Validation"; Expected="Success and Failure"},
    @{Name="Directory Service Access"; Expected="Success and Failure"}
)
$auditOutput = auditpol /get /category:* 2>&1 | Out-String
foreach ($check in $auditChecks) {
    if ($auditOutput -match "$($check.Name)\s+$($check.Expected)") {
        Write-Host "  [OK] $($check.Name) = $($check.Expected)" -ForegroundColor Green
    } else {
        Write-Host "  [KO] $($check.Name) ≠ $($check.Expected)" -ForegroundColor Red
    }
}

if ($age -le 180) {
    Write-Host "  [OK] krbtgt password age = $age jours" -ForegroundColor Green
} else {
    Write-Host "  [WARN] krbtgt password age = $age jours (>180 - planifier un reset)" -ForegroundColor Red
