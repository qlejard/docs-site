S1
Activation & Identity Source
🔴 Critique — Activation IAM Identity Center, sources d’identité, SCIM provisioning Gratuit Orgs requis
⚠️
Scénario d’attaque : Sans IAM Identity Center, chaque développeur dispose d’un IAM user avec ses propres credentials (access key + mot de passe). Quand l’employé quitte l’entreprise, l’administrateur doit penser à révoquer manuellement chaque compte IAM dans chaque compte AWS. Dans un environnement avec 20+ comptes AWS, cette opération est omise dans 40% des cas (AWS Well-Architected Framework — Security Pillar). Résultat : d’anciens employés gardent un accès actif pendant des mois.

Analogie : IAM Identity Center = badge d’entreprise universel. Un seul badge pour accéder à tous les bâtiments AWS. Quand quelqu’un quitte l’entreprise, on révoque un seul badge — tous les accès sont coupés instantanément.

Paramètres d’activation & Identity Source

Paramètre Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal Licence
IAM Identity Center activé Non activé par défaut Activé (1 seule région home) Activé Activé Gratuit
Identity Source IAM Identity Center directory (interne) IAM IC directory Active Directory (AWS Managed AD) External IdP (Okta / Entra ID) via SAML 2.0 Gratuit
SCIM provisioning Désactivé — gestion manuelle des utilisateurs Désactivé Activé (avec AD ou IdP SCIM) Activé + groupe-based access control Gratuit
Région home IAM IC Non configuré Région principale (ex: eu-west-1) Région principale Région principale + DR plan Gratuit
User portal URL URL générée aléatoirement URL générée Sous-domaine personnalisé Sous-domaine personnalisé + CNAME corporate Gratuit
Session duration (portail) 8 heures (480 min) 8 heures 4 heures (travail standard) 1 heure avec re-auth SAML automatique Gratuit
💡
Pourquoi chaque paramètre compte :
• External IdP (SAML 2.0) : l’IdP externe est l’annuaire maître. Quand un employé est désactivé dans Okta ou Entra ID, son accès AWS est immédiatement coupé. Sans External IdP, il faut gérer les comptes manuellement dans deux systèmes.
• SCIM : synchronisation automatique RH. Quand quelqu’un rejoint ou quitte l’entreprise, les accès AWS sont mis à jour automatiquement en quelques minutes (vs heures/jours en manuel).
• Région unique obligatoire : IAM Identity Center ne peut être activé que dans une seule région (home region). Ce choix est irréversible sans réinitialisation complète. Choisir la région principale de l’organisation.
• Session duration : une session de 8h laisse un token actif toute la journée même si l’employé quitte son poste. 4h est le compromis standard (CIS AWS Foundations Benchmark v7.0.0 recommande ≤4h).
🚧
Piège fréquent : IAM Identity Center est désactivé par défaut. De nombreuses organisations continuent d’utiliser des IAM users pour les accès console humains pendant des années faute d’avoir connaissance de ce service. Autre piège : activer IAM IC dans la mauvaise région (us-east-1 par habitude) alors que l’infrastructure est en Europe — la latence d’authentification sera perceptible.

Activation via AWS CLI

bash — Activation IAM Identity Center 
# ─── Prérequis : AWS Organizations doit être activé ───────────────────────────
# Vérifier que Organizations est actif
aws organizations describe-organization \
  --profile management-account \
  --region eu-west-1 \
  --output json

# Activer IAM Identity Center (via la console OU via CloudFormation/Terraform)
# L'activation CLI directe n'existe pas — utiliser le portail AWS Management Console
# Console : https://console.aws.amazon.com/singlesignon/
# Cliquer "Enable" → choisir la région (irréversible)

# ─── Après activation : vérifier l'instance IAM IC ───────────────────────────
aws sso-admin list-instances \
  --profile management-account \
  --region eu-west-1 \
  --output json

# Récupérer l'ARN de l'instance et l'Identity Store ID (nécessaires pour toutes les opérations)
INSTANCE_ARN=$(aws sso-admin list-instances \
  --profile management-account \
  --region eu-west-1 \
  --query 'Instances[0].InstanceArn' \
  --output text)

IDENTITY_STORE_ID=$(aws sso-admin list-instances \
  --profile management-account \
  --region eu-west-1 \
  --query 'Instances[0].IdentityStoreId' \
  --output text)

echo "Instance ARN : $INSTANCE_ARN"
echo "Identity Store ID : $IDENTITY_STORE_ID"

S2
Permission Sets
🔴 Critique — Définition des accès, catalogue de 10 Permission Sets types, least privilege Gratuit
⚠️
Scénario d’attaque : Un développeur avec un Permission Set AdministratorAccess pour « faciliter son travail » peut créer des IAM users avec des access keys non traçables, exfiltrer des données S3, ou provisionner des ressources coûteuses dans n’importe quelle région. En 2023, 68% des incidents cloud liés aux identités impliquaient des permissions excessives (Verizon DBIR 2024).

Analogie : Un Permission Set = niveau d’habilitation. Comme dans un bâtiment sécurisé, chaque badge n’ouvre que les portes correspondant à la fonction : le stagiaire n’accède pas au datacenter, même si c’est plus pratique pour lui.

Paramètres clés d’un Permission Set

Paramètre Défaut AWS ⚡ Quick Win 🎯 Optimal Licence
Session Duration 1 heure (3600s) Adapter par rôle (4h standard, 1h admin) 1h admin, 4h standard, 8h read-only Gratuit
Managed Policies Aucune assignée AWS Managed Policies pour les rôles standards Custom inline policies pour least privilege Gratuit
Permissions Boundary Non configuré Non requis Quick Win Boundary IAM bloquant creation de nouveaux IAM users/roles pour les admins Gratuit
Relay State (URL post-login) Non configuré Non requis URL console ciblant le service principal du rôle (ex: RDS console pour DatabaseAdmin) Gratuit
Inline Policy Aucune Non requis pour démarrer Deny IAM user/key creation pour tous les Permission Sets non-admin Gratuit
💡
Pourquoi chaque paramètre compte :
• Session Duration : un token de session actif peut être volé et réutilisé jusqu’à son expiration. CIS AWS Foundations Benchmark v7.0.0 §1.16 recommande ≤1h pour les accès administrateur.
• Permissions Boundary : sans boundary, un admin via IAM IC peut créer des IAM users avec des access keys permanentes, contournant complètement IAM Identity Center. La boundary empêche ce vecteur de privilège escalation.
• AWS Managed vs Custom inline : les AWS Managed Policies (« AdministratorAccess ») sont trop larges pour la production. Custom inline permet le principle of least privilege strict, mais nécessite un inventaire des API calls utilisés (AWS IAM Access Analyzer peut générer des policies depuis les logs).
• Relay State : Améliore l’expérience utilisateur et réduit les erreurs — le DBA arrive directement sur la console RDS sans devoir naviguer.
🚧
Piège fréquent : Utiliser AdministratorAccess comme Permission Set par défaut « pour aller vite ». Ce Permission Set donne l’accès à tout, y compris la création d’access keys IAM permanentes, la modification des SCPs Organizations, et l’accès à tous les secrets. Même pour les admins, ajouter une Permissions Boundary bloquant iam:CreateUser, iam:CreateAccessKey, et organizations:*.

Création d’un Permission Set via AWS CLI

bash — Créer le Permission Set SecurityAudit 
# Variables
INSTANCE_ARN="arn:aws:sso:::instance/ssoins-xxxxxxxxxx"
REGION="eu-west-1"
PROFILE="management-account"

# ─── Créer le Permission Set SecurityAudit ────────────────────────────────────
PS_ARN=$(aws sso-admin create-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --name "SecurityAudit" \
  --description "Security team - read-only security audit access" \
  --session-duration "PT4H" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --query 'PermissionSet.PermissionSetArn' \
  --output text)

echo "Permission Set ARN : $PS_ARN"

# Attacher la AWS Managed Policy
aws sso-admin attach-managed-policy-to-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --permission-set-arn "$PS_ARN" \
  --managed-policy-arn "arn:aws:iam::aws:policy/SecurityAudit" \
  --profile "$PROFILE" \
  --region "$REGION"

# ─── Créer le Permission Set ReadOnlyAccess ───────────────────────────────────
RO_ARN=$(aws sso-admin create-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --name "ReadOnlyAccess" \
  --description "Developers - read-only access to production" \
  --session-duration "PT8H" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --query 'PermissionSet.PermissionSetArn' \
  --output text)

aws sso-admin attach-managed-policy-to-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --permission-set-arn "$RO_ARN" \
  --managed-policy-arn "arn:aws:iam::aws:policy/ReadOnlyAccess" \
  --profile "$PROFILE" \
  --region "$REGION"

# ─── Créer le Permission Set PowerUserAccess-Dev avec inline Deny IAM ─────────
PU_ARN=$(aws sso-admin create-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --name "PowerUserAccess-Dev" \
  --description "Developers - power user on dev/staging only" \
  --session-duration "PT8H" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --query 'PermissionSet.PermissionSetArn' \
  --output text)

aws sso-admin attach-managed-policy-to-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --permission-set-arn "$PU_ARN" \
  --managed-policy-arn "arn:aws:iam::aws:policy/PowerUserAccess" \
  --profile "$PROFILE" \
  --region "$REGION"

# Ajouter une inline policy de restriction IAM
aws sso-admin put-inline-policy-to-permission-set \
  --instance-arn "$INSTANCE_ARN" \
  --permission-set-arn "$PU_ARN" \
  --inline-policy '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Sid": "DenyIAMUserAndKeyCreation",
        "Effect": "Deny",
        "Action": [
          "iam:CreateUser",
          "iam:CreateAccessKey",
          "iam:CreateLoginProfile",
          "organizations:*"
        ],
        "Resource": "*"
      }
    ]
  }' \
  --profile "$PROFILE" \
  --region "$REGION"

# permission_sets.tf

# ─── Définition des Permission Sets via une map ───────────────────────────────
locals {
  permission_sets = {
    SecurityAudit = {
      description      = "Security team — read-only security audit"
      session_duration = "PT4H"
      managed_policies = ["arn:aws:iam::aws:policy/SecurityAudit"]
      inline_policy    = null
    }
    ReadOnlyAccess = {
      description      = "Developers — read-only production"
      session_duration = "PT8H"
      managed_policies = ["arn:aws:iam::aws:policy/ReadOnlyAccess"]
      inline_policy    = null
    }
    PowerUserAccessDev = {
      description      = "Developers — power user dev/staging"
      session_duration = "PT8H"
      managed_policies = ["arn:aws:iam::aws:policy/PowerUserAccess"]
      inline_policy    = file("policies/deny-iam-user-creation.json")
    }
    BillingViewer = {
      description      = "Finance — billing read-only"
      session_duration = "PT8H"
      managed_policies = [
        "arn:aws:iam::aws:policy/AWSBillingReadOnlyAccess",
        "arn:aws:iam::aws:policy/job-function/Billing"
      ]
      inline_policy    = null
    }
    AdministratorRestricted = {
      description      = "Platform engineers — admin with boundary"
      session_duration = "PT1H"
      managed_policies = ["arn:aws:iam::aws:policy/AdministratorAccess"]
      inline_policy    = file("policies/deny-iam-user-creation.json")
    }
  }
}

# Créer les Permission Sets
resource "aws_ssoadmin_permission_set" "this" {
  for_each     = local.permission_sets
  instance_arn = local.instance_arn
  name         = each.key
  description  = each.value.description
  session_duration = each.value.session_duration
}

# Attacher les AWS Managed Policies
resource "aws_ssoadmin_managed_policy_attachment" "this" {
  for_each = {
    for pair in flatten([
      for ps_name, ps in local.permission_sets : [
        for policy_arn in ps.managed_policies : {
          key        = "${ps_name}__${policy_arn}"
          ps_name    = ps_name
          policy_arn = policy_arn
        }
      ]
    ]) : pair.key => pair
  }
  instance_arn       = local.instance_arn
  permission_set_arn = aws_ssoadmin_permission_set.this[each.value.ps_name].arn
  managed_policy_arn = each.value.policy_arn
}

# Attacher les inline policies si définies
resource "aws_ssoadmin_permission_set_inline_policy" "this" {
  for_each = {
    for ps_name, ps in local.permission_sets :
    ps_name => ps if ps.inline_policy != null
  }
  instance_arn       = local.instance_arn
  permission_set_arn = aws_ssoadmin_permission_set.this[each.key].arn
  inline_policy      = each.value.inline_policy
}

Account Assignments
🔴 Critique — Assignation Groupe → Permission Set → Compte, gestion à l’échelle, off-boarding Gratuit
⚠️
Scénario d’attaque : Sans gestion des Account Assignments via groupes, les accès sont assignés individuellement. Quand un employé change d’équipe, ses anciens accès restent actifs car personne n’a de vision globale. Un auditeur sécurité a documenté 34% d’accès « zombies » dans des organisations sans gestion groupée des assignments (AWS Security Reference Architecture, 2024).

Analogie : Gérer les assignments via des groupes = badge par niveau d’habilitation, pas par individu. Quand quelqu’un rejoint l’équipe sécurité, il reçoit automatiquement tous les accès de l’équipe. Quand il part, retirer du groupe révoque tout.

Création d’un Account Assignment via AWS CLI

bash — Account Assignment : groupe → Permission Set → compte 
# Variables
INSTANCE_ARN="arn:aws:sso:::instance/ssoins-xxxxxxxxxx"
IDENTITY_STORE_ID="d-xxxxxxxxxx"
TARGET_ACCOUNT="123456789012"
REGION="eu-west-1"
PROFILE="management-account"

# Récupérer l'ID du groupe "security-auditors"
GROUP_ID=$(aws identitystore list-groups \
  --identity-store-id "$IDENTITY_STORE_ID" \
  --filters "AttributePath=DisplayName,AttributeValue=security-auditors" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --query 'Groups[0].GroupId' \
  --output text)

# Récupérer l'ARN du Permission Set "SecurityAudit"
PS_ARN=$(aws sso-admin list-permission-sets \
  --instance-arn "$INSTANCE_ARN" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --query 'PermissionSets[]' \
  --output text | while read arn; do
    NAME=$(aws sso-admin describe-permission-set \
      --instance-arn "$INSTANCE_ARN" \
      --permission-set-arn "$arn" \
      --profile "$PROFILE" --region "$REGION" \
      --query 'PermissionSet.Name' --output text)
    if [ "$NAME" = "SecurityAudit" ]; then echo "$arn"; fi
  done)

# Créer l'assignment : groupe → Permission Set → compte AWS
aws sso-admin create-account-assignment \
  --instance-arn "$INSTANCE_ARN" \
  --target-id "$TARGET_ACCOUNT" \
  --target-type "AWS_ACCOUNT" \
  --permission-set-arn "$PS_ARN" \
  --principal-type "GROUP" \
  --principal-id "$GROUP_ID" \
  --profile "$PROFILE" \
  --region "$REGION" \
  --output json

MFA & Conditional Access
🟠 Élevé — Enforcement MFA, FIDO2/WebAuthn, context-aware, rapports de conformité Gratuit
⚠️
Scénario d’attaque : Sans MFA enforcé, le vol d’un mot de passe suffit pour accéder à tous les comptes AWS de l’entreprise via le portail IAM Identity Center. En 2024, 80% des intrusions cloud réussies impliquaient l’absence de MFA ou un MFA contournable (TOTP via phishing). Cisco Duo et Microsoft Entra ID ont rapporté une hausse de 146% des attaques MFA bypass (AiTM) en 2023–2024.

Analogie : MFA = double verrou sur la porte d’entrée. Voler la clé (mot de passe) ne suffit plus — il faut aussi le badge physique (token FIDO2). FIDO2/WebAuthn est résistant au phishing car lié au domaine de l’application.

Pourquoi chaque paramètre compte :
• FIDO2/WebAuthn vs TOTP : les attaques AiTM (Adversary-in-the-Middle) interceptent les tokens TOTP en temps réel. FIDO2/WebAuthn est phishing-resistant car la clé cryptographique est liée à l’URL — une fausse page ne peut pas obtenir le token.
• Context-aware vs Always-On : Context-aware améliore l’expérience utilisateur (pas de MFA depuis le bureau) mais réduit la sécurité. CIS AWS Foundations v7.0.0 §1.14 recommande Always-On pour les accès admin.
• SMS MFA à bannir : les attaques SIM-swap (transfert de numéro) permettent de recevoir les SMS d’un employé. NIST SP 800-63B déconseille le SMS MFA depuis 2017.
• MFA via External IdP : si l’IdP (Okta, Entra ID) est configuré comme Identity Source, le MFA doit être enforcé dans l’IdP ET dans IAM IC (defense in depth).
🚧
Piège fréquent : Activer « Context-aware MFA » en croyant que c’est suffisant. Context-aware signifie que le MFA n’est pas demandé depuis les IP de confiance (bureau, VPN). Un employé dont le laptop est volé hors du bureau sera protégé, mais une compromission interne (réseau bureau) ne l’activera pas. Pour les admins, Always-On est obligatoire.

    AND eventTime > DATEADD(DAY, -7, NOW())
ORDER BY eventTime DESC;

    eventTime,
    userIdentity.arn AS actor,

    json_extract(requestParameters, '$.principalId') AS principal
FROM $EDS_ID
WHERE
    eventSource = 'sso.amazonaws.com'

CLI & SDK Access (Token Provider)
🟡 Moyen — AWS CLI v2 SSO, Token Provider, Boto3, SDK, scripts d’automation Gratuit
⚠️
Scénario d’attaque : Un développeur configure aws configure avec des Access Keys statiques pour son compte prod. Ces credentials sont accidentellement commités dans un repo GitHub public. Des outils automatiques (TruffleHog, GitGuardian) les détectent en moins de 5 secondes et des bots les exploitent immédiatement. Avec AWS CLI v2 SSO, les credentials sont temporaires (1h) et générés à la demandée — aucune clé statique à compromettre.

Pourquoi migrer vers les profils SSO :
• Credentials temporaires : les tokens SSO expirent après 1h (configurable). Un token volé a une fenêtre d’exploitation limitée vs une Access Key statique (valide pour toujours jusqu’à suppression manuelle).
• Multi-comptes centralisé : un seul aws sso login permet d’accéder à tous les comptes AWS de l’organisation via des profils distincts dans ~/.aws/config.
• Audit centralisé : toutes les authentifications CLI apparaissent dans CloudTrail sous l’identité IAM IC — traçabilité complète qui et quand vs les Access Keys où l’identité est opaque.
• Token Provider : l’AWS CLI v2 renouvelle automatiquement le token SSO si la session portail est active — expérience développeur fluide sans sso login répété.
🚧
Piège fréquent : Utiliser l’ancienne commande aws configure sso sans spécifier le sso-session. Avec AWS CLI v2, la bonne approche est de définir une [sso-session] dans ~/.aws/config pour partager la session SSO entre plusieurs profils. Sans cela, chaque profil a sa propre session et l’utilisateur doit se connecter séparément pour chaque profil.

🟡 Moyen — TIP, Amazon Q, Redshift, QuickSight, audit trails enrichis Service dépendant
ℹ️
Trusted Identity Propagation (TIP) — Nouveauté 2023 : AWS IAM Identity Center permet de propaguer l’identité de l’utilisateur final (pas seulement le rôle IAM) aux services AWS sous-jacents. Amazon Redshift, Amazon QuickSight et Amazon Q Business peuvent ainsi appliquer des politiques d’accès basées sur l’identité spécifique de l’utilisateur — row-level security, audit enrichi avec le vrai nom d’utilisateur.

💡
Pourquoi Trusted Identity Propagation change la donne :
• Avant TIP : un utilisateur alice et un utilisateur bob avec le même Permission Set DataEngineerAccess accèdent à Redshift avec la même identité IAM. Impossible de filtrer les données par utilisateur.
• Avec TIP : Redshift reçoit un token signé par IAM IC avec l’identité alice@company.com. Les politiques RLS Redshift peuvent filtrer par current_user = nom IAM IC.
• Audit enrichi : les logs CloudTrail Redshift montrent le vrai nom d’utilisateur, pas juste le rôle IAM — traçabilité complète qui a accédé à quelle donnée.
• Conformité RGPD/SOC2 : prouver que seul alice a accédé aux données PII (pas juste « quelqu’un avec le rôle DataEngineer »).
🚧
Piège : TIP n’est disponible que si l’application est intégrée avec IAM Identity Center (elle doit être référencée comme trusted token issuer). Les applications tierces ne bénéficient pas automatiquement de TIP. Vérifier la documentation de chaque service AWS pour confirmer la compatibilité TIP.

Procédure Break-Glass

🚧
Break-glass account : Compte IAM user d’urgence dans le compte management, avec AdministratorAccess direct (pas via IAM IC). À utiliser uniquement si IAM Identity Center est injoignable (panne AWS, External IdP en panne). Le compte break-glass doit avoir une MFA hardware (YubiKey), son mot de passe dans un coffre-fort physique, et toute utilisation doit déclencher une alerte CloudWatch immédiate.
bash — Alerte CloudWatch sur utilisation break-glass 
# Alerte si le compte break-glass est utilisé
aws cloudwatch put-metric-alarm \
  --alarm-name "breakglass-account-used" \
  --alarm-description "CRITICAL: Break-glass IAM account has been used" \
  --metric-name ConsoleLogin \
  --namespace CloudTrailMetrics \
  --statistic Sum \
  --period 300 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --dimensions Name=Username,Value=breakglass-emergency \
  --alarm-actions "arn:aws:sns:eu-west-1:111111111111:security-critical-alerts" \
  --profile management-account \
  --region eu-west-1
💡
Pourquoi chaque pratique compte :
• Naming convention : sans convention, en 6 mois vous avez 200 Permission Sets aux noms cryptiques. L’audit devient impossible.
• Break-glass : l’External IdP peut tomber en panne. Sans break-glass, plus personne ne peut accéder à AWS en urgence. CIS AWS Foundations v7.0.0 §2.7 (Eliminate use of root) recommande explicitement les break-glass accounts.
• Access review trimestrielle : des études montrent que 34% des assignations deviennent obsolètes dans les 6 mois (mutations, départs, changements de projet). Une review trimestrielle maintient l’hygiene des accès.
• IaC uniquement : forcer la création de Permission Sets via Terraform garantit une peer review (Pull Request) avant tout changement de privilège.

AiTM (Adversary-in-the-Middle)Attaque de phishing avancé qui intercepte les tokens MFA en temps réel via un proxy. Contourne les MFA OTP/TOTP mais pas FIDO2/WebAuthn.
AWS OrganizationsService AWS permettant de regrouper plusieurs comptes AWS dans une hiérarchie gérée (racine, OUs, comptes). IAM Identity Center est activé dans le compte management d’une Organization.
FIDO2 / WebAuthnStandard d’authentification sans mot de passe basé sur la cryptographie asymmétrique. Résistant au phishing car la clé est liée au domaine de l’application. Implémenté par YubiKey, Touch ID, Windows Hello.
IAM Identity CenterService AWS de gestion des identités et des accès centralisé pour les organisations multi-comptes. Anciennement appelé AWS SSO. Fournit un portail web, une API et une intégration SAML/SCIM.
IdP (Identity Provider)Système qui authentifie les utilisateurs et fournit des assertions d’identité (Microsoft Entra ID, Okta, Google Workspace). Dans SAML, l’IdP émet les assertions SAML.
JIT ProvisioningJust-in-Time Provisioning — création automatique d’un utilisateur dans IAM IC lors de sa première connexion SAML. Alternative moins précise au SCIM.
Permission SetCollection de policies IAM (gérées et/ou inline) définissant ce qu’un utilisateur peut faire dans un compte AWS. Comparable à un rôle IAM mais géré centralement dans IAM IC.
SAML 2.0Security Assertion Markup Language — standard XML pour l’échange d’informations d’authentification et d’autorisation entre IdP et SP. Utilisé pour la fédération d’identité.
SCIM 2.0System for Cross-domain Identity Management — standard REST pour la synchronisation automatique des utilisateurs et groupes entre un IdP et une application (IAM IC). Remplace la création manuelle des comptes.
SP (Service Provider)Application qui reçoit et valide les assertions SAML. Dans la fédération IAM IC, AWS IAM Identity Center est le SP qui reçoit les assertions de l’IdP.
TOTP (Time-based OTP)Algorithme générant un code à 6 chiffres valid 30 secondes (RFC 6238). Utilisé par Google Authenticator, Authy. Vulnérable aux attaques AiTM contrairement à FIDO2.
Trusted Identity PropagationFonctionnalité IAM IC (2023) permettant de propager l’identité réelle de l’utilisateur (pas juste le rôle IAM) aux services AWS sous-jacents (Redshift, QuickSight, Q Business).

Script All-in-One — Audit & Configuration IAM Identity Center
Script AWS CLI complet pour auditer et configurer IAM Identity Center en une passe
bash — iam_ic_audit.sh — Script complet d'audit et vérification 
#!/bin/bash
# ─────────────────────────────────────────────────────────────────────────────
# iam_ic_audit.sh — Audit & configuration AWS IAM Identity Center
# Usage : ./iam_ic_audit.sh
# Prérequis : AWS CLI v2, jq, profil "management-account" configuré
# ─────────────────────────────────────────────────────────────────────────────

set -euo pipefail

# ─── Variables — À ADAPTER ────────────────────────────────────────────────────
PROFILE="management-account"
REGION="eu-west-1"
OUTPUT_FILE="iam_ic_audit_$(date +%Y%m%d_%H%M%S).txt"

# ─── Couleurs terminal ────────────────────────────────────────────────────────
RED='\033[0;31m'; GREEN='\033[0;32m'; YELLOW='\033[1;33m'; NC='\033[0m'

log() { echo -e "$1" | tee -a "$OUTPUT_FILE"; }
ok()  { log "${GREEN}  ✓ $1${NC}"; }
warn(){ log "${YELLOW}  ⚠ $1${NC}"; }
err() { log "${RED}  ✗ $1${NC}"; }
sep() { log "\n═══════════════════════════════════════════════════════════════"; }

sep
log "AWS IAM Identity Center — Audit & Configuration"
log "Date    : $(date)"
log "Profil  : $PROFILE | Région : $REGION"
sep

# ─── [1/8] Vérifier l'instance IAM Identity Center ───────────────────────────
log "\n[1/8] Instance IAM Identity Center"
INSTANCE=$(aws sso-admin list-instances \
  --profile "$PROFILE" --region "$REGION" --output json 2>/dev/null)

INSTANCE_COUNT=$(echo "$INSTANCE" | jq '.Instances | length')
if [ "$INSTANCE_COUNT" -eq 0 ]; then
  err "Aucune instance IAM IC trouvée — IAM Identity Center non activé!"
  err "Activer via : Console AWS → IAM Identity Center → Enable"
  exit 1
fi

INSTANCE_ARN=$(echo "$INSTANCE" | jq -r '.Instances[0].InstanceArn')
IDENTITY_STORE_ID=$(echo "$INSTANCE" | jq -r '.Instances[0].IdentityStoreId')
ok "Instance trouvée : $INSTANCE_ARN"
ok "Identity Store   : $IDENTITY_STORE_ID"

# ─── [2/8] Compter les Permission Sets ───────────────────────────────────────
log "\n[2/8] Permission Sets"
PS_LIST=$(aws sso-admin list-permission-sets \
  --instance-arn "$INSTANCE_ARN" \
  --profile "$PROFILE" --region "$REGION" \
  --output json)
PS_COUNT=$(echo "$PS_LIST" | jq '.PermissionSets | length')

if [ "$PS_COUNT" -eq 0 ]; then
  err "Aucun Permission Set configuré — accès impossible!"
elif [ "$PS_COUNT" -lt 3 ]; then
  warn "$PS_COUNT Permission Set(s) — vérifier ReadOnly, Admin, PowerUser minimum"
else
  ok "$PS_COUNT Permission Sets configurés"
fi

# Afficher les noms des Permission Sets
log "  Permission Sets existants :"
for ps_arn in $(echo "$PS_LIST" | jq -r '.PermissionSets[]'); do
  PS_NAME=$(aws sso-admin describe-permission-set \
    --instance-arn "$INSTANCE_ARN" \
    --permission-set-arn "$ps_arn" \
    --profile "$PROFILE" --region "$REGION" \
    --query 'PermissionSet.Name' --output text)
  log "    - $PS_NAME ($ps_arn)"
done

# ─── [3/8] Compter les Account Assignments ───────────────────────────────────
log "\n[3/8] Account Assignments"
ACCOUNTS=$(aws organizations list-accounts \
  --profile "$PROFILE" --region "us-east-1" \
  --output json 2>/dev/null | jq '.Accounts | length' || echo "0")
ok "$ACCOUNTS comptes AWS dans l'organisation"

# ─── [4/8] Vérifier la configuration MFA ─────────────────────────────────────
log "\n[4/8] Configuration MFA"
MFA_CONFIG=$(aws sso-admin describe-instance-access-control-attribute-configuration \
  --instance-arn "$INSTANCE_ARN" \
  --profile "$PROFILE" --region "$REGION" \
  --output json 2>/dev/null || echo '{}')
log "  MFA Config : $MFA_CONFIG"
warn "Vérifier manuellement dans la console : IAM IC → Settings → Authentication → MFA (Always-On + FIDO2)"

# ─── [5/8] Vérifier l'Identity Source ────────────────────────────────────────
log "\n[5/8] Identity Source"
IDENTITY_SOURCE=$(aws sso-admin describe-instance \
  --instance-arn "$INSTANCE_ARN" \
  --profile "$PROFILE" --region "$REGION" \
  --output json 2>/dev/null | jq -r '.Instance.IdentityStoreId // "unknown"')

if echo "$IDENTITY_SOURCE" | grep -q "d-"; then
  ok "External IdP configuré (Identity Store ID : $IDENTITY_SOURCE)"
else
  warn "IAM IC Directory local — recommander migration vers External IdP (Entra ID/Okta)"
fi

# ─── [6/8] Vérifier CloudTrail ───────────────────────────────────────────────
log "\n[6/8] Amazon CloudTrail"
TRAILS=$(aws cloudtrail list-trails \
  --profile "$PROFILE" --region "$REGION" \
  --output json | jq '.Trails | length')

if [ "$TRAILS" -eq 0 ]; then
  err "Aucun trail CloudTrail configuré — logs d'audit absents!"
else
  ok "$TRAILS trail(s) CloudTrail configuré(s)"
  aws cloudtrail describe-trails \
    --profile "$PROFILE" --region "$REGION" \
    --output json | jq -r '.trailList[] | "\(.Name) (\(.HomeRegion))"'
fi

# ─── [7/8] Utilisateurs et Groupes ───────────────────────────────────────────
log "\n[7/8] Utilisateurs & Groupes IAM IC"
USER_COUNT=$(aws identitystore list-users \
  --identity-store-id "$IDENTITY_STORE_ID" \
  --profile "$PROFILE" --region "$REGION" \
  --output json | jq '.Users | length')
GROUP_COUNT=$(aws identitystore list-groups \
  --identity-store-id "$IDENTITY_STORE_ID" \
  --profile "$PROFILE" --region "$REGION" \
  --output json | jq '.Groups | length')

ok "$USER_COUNT utilisateur(s) dans IAM IC"
ok "$GROUP_COUNT groupe(s) dans IAM IC"

if [ "$GROUP_COUNT" -lt 3 ]; then
  warn "Peu de groupes — recommander min 3 groupes (Admin, PowerUser, ReadOnly)"
fi

# ─── [8/8] Résumé ────────────────────────────────────────────────────────────
sep
log "\n[8/8] RÉSUMÉ DE L'AUDIT"
log "  Instance IAM IC    : $INSTANCE_ARN"
log "  Permission Sets    : $PS_COUNT"
log "  Comptes AWS        : $ACCOUNTS"
log "  Utilisateurs IAM IC: $USER_COUNT"
log "  Groupes IAM IC     : $GROUP_COUNT"
log "  Trails CloudTrail  : $TRAILS"
sep
log "Rapport sauvegardé dans : $OUTPUT_FILE"
log "Prochaine étape : vérifier MFA (console), configurer SCIM si External IdP, activer CloudTrail Lake"
sep
