Solutions écartées

💡
Pourquoi Azure natif et pas des outils tiers ?
  • Terraform Cloud / Spacelift → utiles pour le provisioning et les workflows IaC, mais distincts d'Azure Policy, qui évalue les ressources Azure aux scopes ARM et peut appliquer des effets comme Deny, Modify ou DeployIfNotExists.
  • Pulumi / Crossplane → outils de provisioning et d'orchestration complémentaires ; ils ne remplacent pas le moteur natif d'évaluation et de compliance d'Azure Policy sur les ressources Azure.
  • Cloud Custodian / OPA/Gatekeeper → utiles sur d'autres plans de contrôle ou pour Kubernetes, tandis qu'Azure Policy reste le contrôle natif pour les ressources ARM et leurs assignments hiérarchiques.
  • Outils CSPM tiers (Wiz, Prisma) → renvoi vers defender-cloud.html. Ce guide se concentre sur la gouvernance structurelle, pas la posture de sécurité.

1
Management Groups & Hiérarchie Inclus Azure
Structure organisationnelle des subscriptions — fondation de toute la gouvernance
Réf. CIS Azure Foundations Benchmark v2.1 §1.1
🚨
Scénario : Sans hiérarchie de Management Groups, chaque subscription est isolée. Un admin assigne une policy sur la Sub A, oublie la Sub B → la Sub B n'a aucun guardrail. Un développeur crée un Storage Account public dans la Sub B → fuite de données.
Analogie : les Management Groups sont l'organigramme de votre entreprise Azure. Chaque branche (entité) hérite automatiquement des règles du niveau supérieur — comme un règlement intérieur qui s'applique à tous les départements sans avoir à le répéter dans chaque bureau.
ℹ️
Concepts clés : Azure supporte une hiérarchie de Management Groups jusqu'à 6 niveaux de profondeur (hors root et subscriptions). Les Azure Policies et les RBAC role assignments associés à un MG sont hérités par tous les MG enfants et toutes les subscriptions contenues. Le pattern recommandé par Microsoft est l'Azure Landing Zone (ALZ) : Root → Platform / Workloads / Decommissioned / Sandbox.

💡
Pourquoi chaque paramètre compte :
  • Hiérarchie MG : sans elle, chaque subscription est un îlot — impossible d'appliquer une policy à toutes les subscriptions en une seule opération. Source : Microsoft CAF — Organize your resources.
  • Nombre de niveaux : Azure supporte jusqu'à 6 niveaux de MG. En pratique, beaucoup d'organisations gardent une hiérarchie plus courte pour conserver une lecture claire des héritages et des responsabilités.
  • Policies au Root MG : ce qui est assigné au root management group s'applique à toute la hiérarchie du tenant. Ce scope doit donc rester réservé aux guardrails vraiment globaux.
  • RBAC au Root MG : le rôle Owner au Root MG donne le contrôle total sur TOUTES les subscriptions. Le CIS Benchmark recommande de limiter ce accès à 1-2 comptes break-glass (CIS v2.1 §1.1).
  • Require authorization : par défaut, un tenant autorise la création de nouveaux Management Groups. Activer cette protection oblige à disposer des permissions d'écriture appropriées sur le root management group pour créer de nouveaux MG.
  • Default MG pour nouvelles subs : par défaut, une nouvelle subscription arrive sous le root management group. Définir un default management group permet de faire atterrir les nouvelles subscriptions dans une branche plus adaptée à vos guardrails initiaux.

1.2 — Hiérarchie Azure Landing Zone (ALZ) de référence

Tenant Root Group ├── Platform │ ├── Identity (AD DS, DNS, Entra Connect) │ ├── Management (Log Analytics, Automation, Monitor) │ └── Connectivity (Hub VNet, Firewall, ExpressRoute, VPN) ├── Landing Zones │ ├── Corp (workloads connectés au hub — accès on-prem) │ └── Online (workloads publics — pas de VPN vers on-prem) ├── Decommissioned (subs en cours de démantèlement — policies deny all) └── Sandbox (expérimentation — policies légères, pas de peering hub)
⚠️
Pièges classiques :
  • Elevate Access : un Global Admin Entra peut activer “Access management for Azure resources” et recevoir le rôle User Access Administrator à la racine (/). C'est un mécanisme d'élévation puissant, prévu pour des besoins ponctuels, qui doit être remis sur No après usage et surveillé dans les logs.
  • Policies trop permissives au Root : assigner une policy Deny au Root sans exemption bloque toutes les subscriptions. Commencer par Audit, pas Deny.
  • Déplacer un MG avec enfants : déplacer un MG réapplique l'héritage RBAC + Policy du nouveau parent → risque de rupture d'accès si le nouveau parent est plus restrictif.

1.3 — Configuration

Az CLI — Création de la hiérarchie ALZ
# Variables
$tenantRootId="$(az account show --query tenantId -o tsv)"

# Créer les Management Groups de niveau 1
az account management-group create --name "mg-platform" --display-name "Platform"
az account management-group create --name "mg-landingzones" --display-name "Landing Zones"
az account management-group create --name "mg-decommissioned" --display-name "Decommissioned"
az account management-group create --name "mg-sandbox" --display-name "Sandbox"

# Créer les MG enfants de Platform
az account management-group create --name "mg-identity" --display-name "Identity" --parent "mg-platform"
az account management-group create --name "mg-management" --display-name "Management" --parent "mg-platform"
az account management-group create --name "mg-connectivity" --display-name "Connectivity" --parent "mg-platform"

# Créer les MG enfants de Landing Zones
az account management-group create --name "mg-corp" --display-name "Corp" --parent "mg-landingzones"
az account management-group create --name "mg-online" --display-name "Online" --parent "mg-landingzones"

# Déplacer une subscription existante sous le bon MG
az account management-group subscription add --name "mg-corp" --subscription "<subscription-id>"

# Configurer le Default MG pour les nouvelles subscriptions

1.5 — État des lieux

2
Azure Policy Inclus Azure
Règles de gouvernance automatisées — ce qui est autorisé, obligatoire ou interdit
Réf. CIS Azure Foundations Benchmark v2.1 §2
🚨
Scénario : Aucune policy Azure assignée. Un développeur crée un SQL Database dans East US sans TDE, un Storage Account en HTTP sans chiffrement, des VMs sans backup. Lors de l'audit, plusieurs non-conformités sont découvertes tardivement et doivent être corrigées dans l'urgence au lieu d'avoir été prévenues en amont.
Analogie : Azure Policy est le règlement intérieur de votre entreprise Azure. Il définit ce qui est interdit (Deny), ce qui est surveillé (Audit), ce qui est corrigé automatiquement (DeployIfNotExists) et ce qui est ajouté (Append). Sans lui, chaque employé invente ses propres règles.
ℹ️
Concepts clés :
  • Policy definition : une règle unitaire (ex : “Require HTTPS on Storage Accounts”).
  • Policy initiative (= Policy Set) : un ensemble de policies groupées (ex : “CIS Microsoft Azure Foundations Benchmark v2.1”).
  • Assignment : l'application d'une policy ou initiative à un scope (MG, subscription, resource group).
  • Effets : Audit, Deny, AuditIfNotExists, DeployIfNotExists (DINE), Modify, Append, Manual, Disabled.
  • Exemptions : dérogation temporaire (Waiver) ou définitive (Mitigated) pour une ressource spécifique.

Pourquoi chaque paramètre compte :
  • Initiative CIS : fournit 200+ contrôles alignés sur un standard reconnu par les auditeurs. L'assigner en Audit donne immédiatement une vue complète de la posture sans rien bloquer.
  • Allowed Locations : répond aux exigences RGPD / souveraineté des données. Empêche un développeur de déployer dans eastus par erreur quand la politique impose francecentral.
  • Require tags : sans tags, impossible de ventiler les coûts par projet, équipe ou environnement. Interaction directe avec S3 Naming/Tagging et S7 Cost Management.
  • HTTPS/TLS : une policy Deny aide à empêcher la création ou la mise à jour de ressources non conformes au standard attendu, y compris lorsque le service permet encore une configuration moins sécurisée.
  • Diagnostic Settings (DINE) : DeployIfNotExists déploie automatiquement les diagnostics sur chaque nouvelle ressource → aucune ressource “muette” en monitoring. Interaction avec S10 Monitoring.
  • Network policies : bloquer les IP publiques non nécessaires est la mesure la plus efficace contre l'exposition accidentelle. Interaction avec le guide réseau (azure-networking.html).
  • Exemptions : sans documentation, on découvre des Deny bypass sans justification lors de l'audit. Les exemptions doivent avoir une date d'expiration pour éviter qu'elles deviennent permanentes.

Subscription Design Inclus Azure
Modèles d'organisation des subscriptions et automatisation du vending
Réf. CAF — Subscription democratization
🚨
Scénario : Tout dans une seule subscription. Plusieurs équipes et ressources mélangent prod, dev, sandbox et POCs. Un développeur supprime accidentellement un resource group de production car il avait Contributor sur toute la sub, avec interruption de service et restauration à gérer en urgence. Impossible de limiter les quotas, les coûts se mélangent, les RBAC sont trop larges.
Analogie : une subscription Azure est un contrat de location. Mettre tous les départements dans le même bail rend impossible de connaître qui consomme quoi, et si un département fait une bêtise, tous les autres sont impactés.
ℹ️
Concepts clés :
  • Subscription as a unit of scale : chaque sub a ses propres quotas, limites et billing scope.
  • Subscription vending : processus automatisé (IaC) pour créer une sub pré-configurée (tags, policies, RBAC, budget) via un pipeline CI/CD.
  • Modèles : Workload-based (1 sub par workload), Environment-based (1 sub par env), Hybrid (1 sub par workload×env).

  • Séparation Platform/Workloads : les services partagés (DNS, Firewall, Log Analytics) doivent être dans des subs dédiées pour éviter qu'un développeur modifie l'infrastructure commune.
  • Budget par sub : sans budget ni alerte, une dérive de consommation peut passer inaperçue jusqu'au cycle de facturation. Interaction directe avec S7 Cost Management.
  • Subscription vending : automatiser la création des subscriptions aide à appliquer dès le départ les conventions de tags, les policies et les attributions RBAC attendues (“guardrails by design”).
⚠️
Pièges classiques :
  • Trop de subscriptions sans automatisation : plus le nombre de subscriptions augmente, plus les écarts de configuration se multiplient si la création reste manuelle. L'automatisation devient alors le moyen le plus fiable de garder une base homogène.
  • Limites de subscription : certaines ressources ont des quotas par sub (ex : 250 Storage Accounts, 25k vCPUs par région). Ne pas ignorer ces limites dans le design.
  • Billing account prérequis : pour créer des subs par API, il faut des droits sur le Billing Account (EA Enrollment, MCA Billing Profile). Souvent bloqué par le service finance/IT → planifier en amont.

Azure RBAC Inclus Azure PIM = Entra P2
Contrôle d'accès basé sur les rôles — least privilege à chaque niveau de scope
Réf. CIS Azure Foundations Benchmark v2.1 §1.2
🚨
Scénario : Trop d'utilisateurs disposent du rôle Owner au niveau subscription. Un compte surprovisionné supprime un Resource Group de production. Un autre compte avec Contributor lance des ressources très coûteuses hors cadre attendu, ce qui provoque une dérive budgétaire rapide.

ℹ️
Concepts clés :
  • Role Assignment = Security Principal + Role Definition + Scope.
  • Scope hierarchy : Management Group → Subscription → Resource Group → Resource. Les permissions sont héritées du parent vers l'enfant.

  • PIM (Privileged Identity Management) : activation just-in-time des rôles privilégiés. Nécessite Entra ID P2.
  • Custom roles : rôles sur mesure quand les built-in sont trop larges ou trop restrictifs.

Pourquoi chaque paramètre compte :
  • Limiter les Owners : le CIS Benchmark recommande ≤3 Owners par subscription (CIS v2.1 §1.22). Chaque Owner peut tout faire, y compris supprimer la sub elle-même.
  • Scope le plus précis : Contributor à la sub = droits sur TOUTES les ressources. Contributor au RG = droits limités à ce RG. Le principle of least privilege commence par le scope.
  • Groupes vs directs : gérer les accès via groupes simplifie l'offboarding et rend les revues d'accès plus lisibles qu'une accumulation d'assignments individuels.
  • Managed Identity : quand le service la supporte, la Managed Identity évite la gestion directe de secrets applicatifs. Sinon, un Service Principal doit rester fortement borné et gouverné.
  • PIM : 0 Owner permanent signifie que même si un compte est compromis, l'attaquant n'a pas les droits Owner sans activer PIM (+ MFA + justification).
⚠️
Pièges classiques :
  • Classic Administrator rôles : les rôles “Co-Administrator” et “Service Administrator” sont des legacy d'Azure Classic. Ils donnent des droits Owner et ne sont PAS visibles dans les RBAC assignments standards. Vérifier et supprimer.
  • Contributor ≠ sans risque : Contributor ne peut pas gérer RBAC mais peut supprimer des ressources, créer des VMs GPU, exfiltrer des données. Ce n'est PAS un rôle “sûr”.
  • Assignments orphelins : quand un utilisateur est supprimé d'Entra ID, son role assignment reste avec un ObjectId orphelin. Nettoyer régulièrement.

🚨

Scénario : Pas de budget configuré. Un développeur lance des VMs GPU pour des tests ML un vendredi soir, oublie de les éteindre. Lundi matin : 8 000€ de consommation week-end. Le CFO découvre la facture à M+1 → trop tard pour corriger.
Analogie : Cost Management est le contrôleur de gestion de votre cloud. Sans lui, c'est comme donner une carte bancaire sans plafond à 50 employés et ne regarder le relevé qu'une fois par trimestre.
ℹ️
Concepts clés :
  • Budget : montant mensuel ou trimestriel avec seuils d'alerte configurables selon votre tolérance au risque et votre cycle de pilotage.
  • Action Group : notifications (email, SMS, webhook) et actions automatiques (Azure Function, Logic App pour arrêter des VMs).
  • Cost exports : export automatique des données de coût vers un Storage Account pour analyse BI.
  • Cost allocation : répartition des coûts partagés (VNet hub, Firewall) entre les équipes via tags.
  • Azure Advisor : recommandations de réduction (right-sizing, reserved instances, shutdown).

  • Tags pour allocation : le CFO demande “combien coûte le projet Alpha ?” → sans tag CostCenter, la réponse est “on ne sait pas”. Interaction directe avec S3 Naming/Tagging.
  • Cost exports : exporter les données de coûts vers un Storage Account facilite l'historisation, les analyses externes et les usages BI au-delà des vues portail.
  • Advisor : fournit des recommandations d'optimisation comme le rightsizing, l'identification de ressources sous-utilisées ou certaines opportunités d'engagement. Le gain réel dépend fortement du parc et du niveau de maturité FinOps.
⚠️
Pièges classiques :
  • Budget ≠ plafond : un budget Azure est une ALERTE, pas un blocage. Atteindre 100% n'arrête pas les ressources. Pour bloquer, il faut coupler avec une Action Group qui déclenche un script (ou utiliser spending limits sur les sub MSDN/dev).
  • Cost exports sans nettoyage : les exports quotidiens vers Storage finissent par accumuler un volume significatif de données. Prévoir une lifecycle management policy alignée sur votre politique de rétention.
  • RI sur le mauvais scope : une Reserved Instance achetée au scope “Single subscription” ne bénéficie pas aux autres subs. Préférer scope “Shared” pour maximiser l'utilisation.

  • Phase Ready vs phase Govern : la LZ (Ready) fournit la fondation ; la gouvernance (Govern) ajoute les guardrails au fil du temps. Ne pas essayer de tout faire dans la LZ initiale — itérer.

  • Limite de 1000 résultats : Resource Graph pagine à 1000 lignes par défaut. Toujours utiliser | summarize count() ou gérer la pagination avec $skipToken pour les gros tenants.

RBACRole-Based Access Control — modèle d'autorisation Azure basé sur des rôles (Owner, Contributor, Reader, custom) attribués à des scopes (MG, Sub, RG, ressource).
RIReserved Instance — engagement de 1 ou 3 ans sur des ressources Azure pour bénéficier de réductions (jusqu'à 72%).
SoDSeparation of Duties — principe de sécurité imposant que différentes personnes gèrent différentes fonctions (ex : governance vs workload).
Deployment StackRessource Azure remplaçant les Blueprints (dépréciés) pour déployer des templates avec protection deny-settings contre les modifications non autorisées.
MDCMicrosoft Defender for Cloud — service de sécurité cloud incluant CSPM (posture) et CWP (protection des workloads).
Secure ScoreScore de posture de sécurité calculé par MDC (0-100%) basé sur les recommandations implémentées.
VWANVirtual WAN — service Azure de networking hub managé pour connecter VNets, VPN, ExpressRoute dans une architecture hub-spoke simplifiée.

Script All-in-One — Governance Baseline
Script exécutable pour déployer la gouvernance complète en une seule fois
⚠️
Précaution : Ce script est un point de départ. Adaptez les noms, régions et paramètres à votre contexte avant exécution. Testez sur une subscription sandbox / non-prod d'abord. Les commandes sont idémpotentes (relancer ne crée pas de doublon).
PowerShell / Az CLI — Governance Baseline (Quick Win → Avancé)
# ═══════════════════════════════════════════════════════════════
# GOVERNANCE BASELINE — Azure All-in-One
# Prérequis : Az CLI installé, connecté (az login), droits Owner au Root MG
# ═══════════════════════════════════════════════════════════════

# ─── Variables à personnaliser ───
$tenantId    = "$(az account show --query tenantId -o tsv)"
$location    = "francecentral"
$mgmtSubId   = "<SUBSCRIPTION_ID_MANAGEMENT>"
$mgmtRg      = "rg-management-prod-frc"
$lawName     = "law-governance-prod-frc"
$budgetEmail = "cloud-governance@contoso.com"

# ═══════════ S1 — Management Groups ═══════════
Write-Host "[S1] Création de la hiérarchie Management Groups..."
az account management-group create --name "mg-platform" --display-name "Platform"
az account management-group create --name "mg-landingzones" --display-name "Landing Zones"
az account management-group create --name "mg-decommissioned" --display-name "Decommissioned"
az account management-group create --name "mg-sandbox" --display-name "Sandbox"
az account management-group create --name "mg-identity" --display-name "Identity" --parent "mg-platform"
az account management-group create --name "mg-management" --display-name "Management" --parent "mg-platform"
az account management-group create --name "mg-connectivity" --display-name "Connectivity" --parent "mg-platform"
az account management-group create --name "mg-corp" --display-name "Corp" --parent "mg-landingzones"
az account management-group create --name "mg-online" --display-name "Online" --parent "mg-landingzones"

# ═══════════ S2 — Policies essentielles ═══════════
Write-Host "[S2] Assignation des policies de base..."
# Require tag “Env” sur les Resource Groups
az policy assignment create \
  --name "require-tag-env" \
  --display-name "Require Env tag on Resource Groups" \
  --policy "96670d01-0a4d-4649-9c89-2d3abc0a5025" \
  --scope "/providers/Microsoft.Management/managementGroups/mg-landingzones" \
  --params '{"tagName":{"value":"Env"}}'

# Allowed Locations (France Central + France South)
az policy assignment create \
  --name "allowed-locations" \
  --display-name "Restrict to France regions" \
  --policy "e56962a6-4747-49cd-b67b-bf8b01975c4c" \
  --scope "/providers/Microsoft.Management/managementGroups/mg-landingzones" \
  --params '{"listOfAllowedLocations":{"value":["francecentral","francesouth"]}}'

# ═══════════ S5 — RBAC baseline ═══════════
Write-Host "[S5] Configuration RBAC de base..."
# Supprimer les Owner inutiles au scope subscription (garder uniquement break-glass)
# Lister les assignments Owner pour revue :
az role assignment list \
  --role "Owner" \
  --scope "/subscriptions/$mgmtSubId" \
  --output table

# ═══════════ S6 — Locks sur ressources critiques ═══════════
Write-Host "[S6] Création des Resource Locks..."
az lock create \
  --name "CanNotDelete-LAW" \
  --resource-group $mgmtRg \
  --resource-name $lawName \
  --resource-type "Microsoft.OperationalInsights/workspaces" \
  --lock-type CanNotDelete

# ═══════════ S7 — Budget & alertes ═══════════
Write-Host "[S7] Création du budget et des alertes coûts..."
az consumption budget create \
  --budget-name "budget-governance-monthly" \
  --amount 5000 \
  --time-grain Monthly \
  --start-date "$(Get-Date -Format 'yyyy-MM-01')" \
  --end-date "2027-12-31" \
  --category Cost

# ═══════════ S10 — Monitoring governance ═══════════
Write-Host "[S10] Configuration du monitoring de gouvernance..."
az monitor action-group create \
  --resource-group $mgmtRg \
  --name "ag-governance-alerts" \
  --short-name "GovAlerts" \
  --action email governance-team $budgetEmail

az monitor activity-log alert create \
  --resource-group $mgmtRg \
  --name "alert-policy-compliance-change" \
  --description "Alerte sur changement de compliance" \
  --condition category=Policy \
  --action-group "ag-governance-alerts"

Write-Host "\n[OK] Governance baseline déployée. Exécutez la checklist ci-dessous pour valider."
