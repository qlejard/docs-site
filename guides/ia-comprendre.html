1.2 Qu'est-ce qu'un LLM, vraiment ?

Un LLM est un programme qui a lu énormément de texte (des milliers de milliards de mots : Wikipédia, GitHub, livres, sites web, forums…) et qui a appris une chose toute simple : prédire le mot suivant.

Imaginez un serveur qui a passé sa vie dans des milliers de restaurants. Il a écouté toutes les conversations, lu tous les menus, vu toutes les recettes. Si vous dites « Bonjour, je voudrais un steak… », il sait avec une probabilité énorme que le mot suivant sera « frites », « tartare » ou « saignant ». Pas parce qu'il « comprend » la viande — mais parce qu'il a entendu cette suite des millions de fois.

Un LLM, c'est ça, mais avec tout le savoir écrit de l'humanité.

1.3 Ce que l'IA n'est PAS

❌ Idées fausses

  • « Elle réfléchit comme un humain » → non, elle calcule des probabilités
  • « Elle dit la vérité » → non, elle dit ce qui est probable
  • « Elle apprend en continu » → non, son savoir est figé à sa date d'entraînement
  • « Elle est consciente » → non, c'est une fonction mathématique

✅ Ce qu'elle est

  • Un excellent moteur de complétion de texte
  • Un résumeur, traducteur, reformulateur hors-pair
  • Un générateur de code (parce que le code est du texte structuré)
  • Un assistant qu'il faut superviser et vérifier

Comment ça marche sous le capot ?
Tokens, contexte, hallucinations — les concepts qui expliquent tous les comportements bizarres.

2.1 Les tokens : la « langue » du LLM

Le LLM ne lit pas des mots, il lit des tokens. Un token, c'est un fragment de mot. Par exemple :

PhraseDécoupage en tokensNb tokens
BonjourBon + jour2
GitHub CopilotGit + Hub + Co + pilot4
function getUserById(id) {function + get + User + ById + ( + id + ) + {8

👉 Règle de pouce : 1 token ≈ 4 caractères de texte. En nombre de mots, le ratio varie selon la langue et le contenu : c'est utile pour estimer, pas pour facturer au centime.

2.2 La fenêtre de contexte : la mémoire courte

Chaque modèle a une fenêtre de contexte : la quantité maximale de tokens qu'il peut « voir » d'un coup. C'est sa mémoire de travail.

2.4 Les hallucinations : pourquoi il invente

⚠️ La règle d'or à ne JAMAIS oublier Un LLM ne sait pas qu'il ne sait pas. Quand il ne connaît pas la réponse, il complète quand même avec ce qui est statistiquement plausible. C'est ce qu'on appelle une hallucination : une réponse qui sonne juste mais qui est fausse.

Exemples typiques :

  • Inventer un nom de fonction qui n'existe pas dans une librairie
  • Citer un article ou un RFC qui n'a jamais été publié
  • Donner une commande PowerShell avec un paramètre imaginaire

👉 Toujours vérifier les noms de fonctions, paramètres, URLs, et faits techniques. C'est pour ça que les outils (tools, MCP, recherche web) sont devenus indispensables : ils donnent au LLM accès au réel.

3.1 Modèles « raisonneurs » vs « réactifs »

Depuis 2024, on distingue deux familles :

⚡

Modèles réactifs

Répondent vite, parfaits pour du code court, des questions simples et de l'itération rapide. Ex : familles mini, haiku, flash, petits modèles locaux.

🧠

Modèles raisonneurs

Prennent un temps pour « réfléchir » avant de répondre. Excellents en math, debug profond, planification et workflows agentiques. Ex : familles GPT frontier/reasoning, Claude Opus, Gemini Pro, DeepSeek-R1 selon disponibilité.

💡 Astuce VS Code Dans Copilot, la liste exacte des modèles dépend de votre plan et du catalogue actif. Pour une question rapide, choisissez un modèle rapide/mini/flash. Pour un refactoring complexe, prenez un modèle frontier/code/reasoning plus coûteux.

Schéma 3 — Les 3 niveaux d'autonomie d'une IA.

4.1 Le Chat (niveau 1)

Vous parlez, il répond. Aucune autonomie. Vous copiez-collez le code. Exemples : ChatGPT, Claude.ai, Gemini.

4.2 L'assistant (niveau 2)

Il a accès à votre code et peut proposer des modifications, mais c'est vous qui validez chaque changement. Exemple : GitHub Copilot en mode Ask ou Edit.

4.3 L'agent (niveau 3)

Il agit pour vous : lit des fichiers, modifie du code, lance des commandes, appelle des APIs. Vous donnez un objectif, il décompose en sous-tâches et exécute. Exemples : GitHub Copilot Agent, Cursor Agent, Cline, Claude Code.

Au restaurant :

  • Chat = vous appelez le chef au téléphone pour lui demander la recette, vous cuisinez chez vous.
  • Assistant = le chef vient dans votre cuisine et vous guide étape par étape, mais vous tenez la casserole.
  • Agent = vous donnez l'objectif (« je veux un menu 3 plats pour 4 personnes »), le chef fait les courses, prépare, dresse, vous appelle quand c'est prêt.
⚠️ Plus d'autonomie = plus de risque Un agent peut supprimer des fichiers, pousser sur Git, dépenser de l'argent (appels d'API payants). C'est pour ça que VS Code demande confirmation avant les actions destructives, et que vous devez avoir un AGENTS.md bien clair qui définit ce qui est interdit.

Les outils (tools / function calling)
Comment l'IA fait des actions concrètes, et pas seulement du texte.

Par défaut, un LLM ne sait que générer du texte. Il ne peut pas vraiment lire un fichier, exécuter une commande, ou appeler une API. Les tools (parfois appelés function calling) sont le mécanisme qui change ça.

7.1 Le principe

On donne au modèle une liste de fonctions disponibles avec leur description. Quand il a besoin de faire une action, il répond en demandant d'appeler une fonction. Le moteur (VS Code) exécute la fonction et lui renvoie le résultat. Puis il continue.

MCP — Model Context Protocol ★ clé en 2026
Le standard universel pour brancher des outils externes à n'importe quelle IA.

8.1 Le problème que MCP résout

Avant 2024, si OpenAI voulait que GPT puisse lire votre Outlook, il fallait coder un connecteur OpenAI ↔ Outlook. Et si Anthropic voulait la même chose avec Claude, tout réécrire. Multipliez par 100 applis et 10 modèles = 1000 connecteurs à maintenir.

Le Model Context Protocol (MCP), proposé par Anthropic fin 2024 puis repris progressivement par l'écosystème, est un standard ouvert. Vous écrivez un seul serveur MCP pour votre service, et tous les clients IA qui implémentent MCP peuvent l'utiliser.

MCP, c'est la prise électrique standardisée du restaurant. Avant, chaque fournisseur livrait avec sa propre prise (gaz, 110V, 220V, USB). Maintenant tout le monde livre en « prise MCP ». Le chef branche le fournisseur qu'il veut sans changer son matériel.

9
Les Skills
Des « fiches recettes » spécialisées que l'IA charge à la demande.

9.1 Le concept

Un Skill est un document markdown qui contient des instructions expertes sur un domaine précis. Quand l'IA détecte que votre demande tombe dans ce domaine, elle charge le skill et suit ses consignes.

Un skill, c'est la fiche recette plastifiée du chef. Le chef connaît plein de plats par cœur, mais pour la blanquette de la grand-mère, il sort la fiche. Idem pour l'IA : pour un cas standard elle improvise, pour « déployer une fonction Azure » elle ouvre la fiche azure-deploy.

9.2 Anatomie d'un skill

Chaque skill a deux parties :

  1. Une description (visible par l'IA) qui dit quand l'utiliser : mots-clés, intentions, contre-exemples.
  2. Le contenu (chargé sur demande) : la procédure détaillée, commandes, pièges.
SKILL.md (extrait simplifié)
---
name: azure-deploy
description: Déploiement Azure avec azd up, terraform apply...
WHEN: "déployer sur Azure", "azd up", "push prod"
DO NOT USE WHEN: "créer une app" (utiliser azure-prepare)
---

# Skill : Déploiement Azure

## Étape 1 — Vérifier la connexion
az login --tenant $TENANT_ID

## Étape 2 — Lancer le déploiement
azd up --no-prompt

Les Subagents (agents custom)
Déléguer une tâche à un « second couteau » spécialisé.

Un subagent, c'est une instance d'IA lancée par l'agent principal pour traiter une sous-tâche spécifique, avec son propre contexte et ses propres outils.

Le chef exécutif (agent principal) ne pèle pas les patates lui-même. Il appelle le commis « épluche-moi 5 kilos », le commis fait son boulot dans son coin, et revient avec le résultat. Le chef ne s'embarrasse pas des détails — il garde sa tête claire pour le menu global.

10.1 À quoi ça sert vraiment ?

  • Préserver le contexte principal : explorer 50 fichiers dans une recherche, c'est 50 fichiers de pollués dans la fenêtre. Un subagent fait le sale boulot et renvoie juste un résumé.
  • Paralléliser : lancer 3 explorations en même temps.
  • Spécialiser : un subagent java-modernizer avec ses propres skills, un autre azure-cost-optimizer, etc.

RAG & embeddings — donner de la mémoire à l'IA
Comment Copilot trouve « les bons fichiers » dans votre repo de 90 000 lignes.

15.1 Le problème

Votre projet Audit_EntraID_Intune fait ~90k lignes. Impossible de tout envoyer au modèle (la fenêtre de contexte explose, et le coût grimpe vite). Comment le modèle sait quels fichiers sont pertinents pour votre question ?

Réponse : RAG = Retrieval-Augmented Generation. Avant d'envoyer la question au modèle, on cherche les morceaux pertinents et on les ajoute au prompt.

15.2 Les embeddings : transformer le sens en chiffres

Un embedding, c'est un vecteur (une liste de ~1500 nombres) qui représente le sens d'un texte. Deux textes proches sémantiquement ont des vecteurs proches géométriquement.

TexteEmbedding simplifié
« lister les utilisateurs Entra ID »[0.12, 0.88, -0.34, ...]
« afficher les comptes du tenant »[0.14, 0.86, -0.31, ...] ✅ proche
« recette de tarte aux pommes »[-0.91, 0.02, 0.55, ...] ❌ très loin

Imaginez la bibliothèque infinie du restaurant. Chaque recette est rangée selon ses ingrédients et techniques (température au four, type de viande, durée…). Quand vous demandez « un plat mijoté à base de bœuf », le sommelier ne lit pas les 50 000 recettes : il va dans le rayon « mijoté + bœuf » et en sort 5 candidates. L'embedding, c'est le système de classement géométrique.

15.5 RAG vs Fine-tuning : la confusion classique

✅ RAG (souvent le bon premier choix)

  • Pas de réentraînement : on ajoute du contexte récupéré au moment de la requête
  • Mise à jour plus simple : modifier la base documentaire suffit souvent à changer la réponse
  • Sources citables (« d'après le fichier X ») si le pipeline les remonte
  • Coût : indexation/stockage + appels modèle, à dimensionner selon le volume et la fréquence

⚠️ Fine-tuning (cas pointus)

  • Réentraîner le modèle sur vos données
  • Connaissance plus « intégrée » au comportement du modèle, mais moins simple à mettre à jour
  • Pas de citation de source native à lui seul
  • Coût : préparation des données, entraînement/évaluation et exploitation
  • Utile pour : style d'écriture, format strict, langage métier rare

19.1 Chain-of-Thought (CoT)

Au lieu de répondre directement, le modèle verbalise son raisonnement étape par étape. Pour les problèmes math/logique, ça améliore la précision de 20-40 %.

Mauvais : réponse directe
Q: J'ai 17 utilisateurs. 3 sont admin, 5 ont MFA désactivé,
   1 est admin sans MFA. Combien d'utilisateurs standards ont MFA ?
R: 9
Bon : avec chain-of-thought
Q: ... (même question)
R: Réfléchissons étape par étape.
   - Total : 17
   - Admins : 3
   - Standards : 17 - 3 = 14
   - Sans MFA total : 5
   - Admins sans MFA : 1
   - Standards sans MFA : 5 - 1 = 4
   - Standards AVEC MFA : 14 - 4 = 10
   Réponse : 10

👉 En 2026, les modèles raisonneurs intègrent souvent cette étape en interne, avec un raisonnement partiellement résumé ou masqué selon la surface.

Comment évaluer la qualité d'une réponse IA ?
Méthodes pour ne pas vous faire avoir par une jolie hallucination.

22.1 Les 5 questions à se poser

📎

1. Source citée ?

Pour un fait technique, demandez la source. Si le modèle ne peut pas la citer → suspect.

🔁

2. Reproductible ?

Posez la même question 2 fois. Si les réponses divergent fortement → faible confiance.

🧪

3. Testé ?

Pour du code : lancez-le. Pour une commande : --dry-run ou -WhatIf.

🎯

4. Précis ?

Une réponse trop vague (« en général », « il se peut que ») cache souvent un manque de certitude.

⚖️

5. Cohérent ?

Demandez au modèle d'expliquer son raisonnement et cherchez les contradictions.

22.2 Le « second avis »

Sur une réponse importante, demandez à un autre modèle. Si Copilot vous dit X et que Claude dit Y, creusez. C'est le principe du LLM-as-a-judge utilisé en production.

Tendances 2026-2027 — où va l'IA ?
Ce qui se prépare et ce qu'il faut surveiller.

23.1 Ce qui est déjà acté

  • MCP s'étend vite : de plus en plus de clients IA majeurs l'implémentent. C'est une couche d'interop de plus en plus importante, sans être encore universelle partout de la même façon.
  • Les assistants deviennent plus agentiques : les simples chats restent utiles, mais davantage de produits savent maintenant agir, planifier et itérer.
  • Multimodalité native : texte, image, audio et parfois vidéo convergent dans les grandes familles actuelles.
  • Contextes très larges : la limite devient souvent autant économique qu'architecturale.
  • Models on-device : des SLM comme Phi, Llama, Gemma ou Qwen deviennent plus réalistes en local selon le matériel disponible.

23.2 Ce qui émerge fort

  • Computer Use : l'agent contrôle votre souris/clavier pour piloter n'importe quelle app (Claude Computer Use, Operator d'OpenAI).
  • Voice agents : conversation vocale temps réel (Realtime API OpenAI, GPT-5 Voice).
  • World models : modèles qui simulent une réalité physique (Sora, Genie 3).
  • Reasoning chains plus visibles ou mieux contrôlées : selon la surface, le raisonnement peut être résumé, partiellement exposé ou masqué.
  • Agent identity : votre skill entra-agent-id est en plein dans cette tendance — chaque agent a sa propre identité Entra avec ses permissions.
  • Marketplace de skills/MCP : un « npm » pour les capacités d'agents.

23.3 Ce qu'il faut surveiller (sur 12 mois)

🟠 Côté légal EU AI Act : obligations déployées progressivement depuis 2024 et encore en train de se concrétiser selon les usages. Si vous déployez de l'IA pour des clients, anticipez la documentation de conformité et faites valider le statut réglementaire actuel.
🟠 Côté sécurité Prompt injection devient un vrai sujet en entreprise. Microsoft Defender for AI et autres solutions de « AI security posture management » émergent.
💡 Côté skills Le format « skill markdown + frontmatter » se standardise (Claude Skills, GitHub Copilot Skills, AGENTS.md). Investir dans vos skills custom = investissement durable.

23.4 Ce qu'il faut arrêter de croire

  • « L'IA va remplacer les devs » → non, elle les rend 2-5× plus rapides. Le métier évolue, pas disparaît.
  • « Un très bon modèle généraliste suffit, plus besoin de RAG/MCP/skills » → faux. Un contexte explicite + outils restent souvent supérieurs à un modèle « brut ».
  • « Plus le modèle est gros, mieux c'est » → les SLM bien fine-tunés battent les gros modèles sur des tâches précises.
  • « On peut faire confiance aux benchmarks » → la plupart sont contaminés. Ne croyez que vos évaluations sur vos données.

FAQ — les questions qu'on me pose le plus souvent
Réponses courtes aux interrogations fréquentes.

« Copilot peut-il vraiment lire tout mon repo ? »

Non. Il indexe les fichiers (embeddings) et lit à la demande via les outils (read_file, semantic_search). Il n'envoie jamais tout d'un coup. En mode Enterprise, l'index est privé et chiffré.

« Si je désactive Copilot, il continue à envoyer mes données ? »

Non. Quand l'extension est désactivée, aucune communication. Ce qui est déjà transité ne peut pas être récupéré, par contre (rétention selon le plan).

« Comment savoir si le modèle hallucine ? »

3 réflexes : (1) demandez la source, (2) testez le code/la commande, (3) vérifiez les noms de fonctions dans la doc officielle (via Microsoft Learn MCP par ex.).

« Pourquoi Copilot me propose parfois du code obsolète ? »

Les modèles ont une date de cutoff, souvent décalée de plusieurs mois par rapport à l'actualité des librairies. Pour les libs qui bougent vite (Azure SDK, React), précisez la version dans le prompt ou utilisez le MCP Microsoft Learn / Context7.

« Mon AGENTS.md est trop long, ça pose problème ? »

Oui à partir de ~10-15k tokens (chaque conversation paie ce coût). Le vôtre est bien dimensionné. Au-delà, déportez les détails vers des .instructions.md ciblés via applyTo.

« Quelle différence entre Copilot et Microsoft 365 Copilot ? »

GitHub Copilot = dans VS Code, pour le code. Microsoft 365 Copilot = dans Word/Excel/Outlook/Teams, pour les docs et mails. Deux produits, deux licences, deux UX. Ils partagent peu de choses.

« Peut-on rendre Copilot déterministe (même réponse à chaque fois) ? »

Quasiment. Avec temperature=0 et le même prompt, la réponse est ~stable. Mais Copilot ne vous expose pas ce paramètre. En API directe (Azure OpenAI), oui.

« Faut-il apprendre LangChain / LlamaIndex / Semantic Kernel ? »

Si vous voulez construire des agents : oui (Semantic Kernel ou Microsoft Agent Framework sont les choix Microsoft). Si vous voulez juste utiliser Copilot dans VS Code : non, c'est superflu.

« L'IA va-t-elle continuer à progresser à ce rythme ? »

Plus probablement par améliorations incrémentales (raisonnement, outils, multimodalité) que par sauts de capacité brute. La course aux paramètres ralentit ; la course à l'usage s'accélère.

« Combien de temps avant d'être autonome avec Copilot Agent ? »

~1 semaine pour utiliser, ~1 mois pour bien prompter, ~3 mois pour customiser (skills, MCP, AGENTS.md). Vous êtes déjà au stade 3, d'où ce guide qui consolide.

⚠️ Anti-pattern Garder la même conversation pour 5 tâches différentes (« et tant qu'on y est, ajoute X, ah et corrige Y, et… »). Le contexte se pollue, le modèle oublie, les coûts explosent.

✅ Correction : Ctrl+L (new chat) entre chaque tâche indépendante. Une conversation = un objectif.

26.6 « L'AGENTS.md négligé »

⚠️ Anti-pattern Un AGENTS.md générique copié d'un template, jamais relu. L'agent ignore vos vraies conventions.

✅ Correction : votre AGENTS.md actuel est exemplaire — il liste les projets, les règles inviolables, les commandes, les anti-patterns à refuser. Relisez-le tous les 2-3 mois pour ajuster.
