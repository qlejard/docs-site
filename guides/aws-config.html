Contexte, périmètre, convention des colonnes et stratégie de déploiement
📋
AWS Config est l'inspecteur permanent de votre infrastructure : il photographie l'état de chaque ressource en temps réel, détecte les dérives de configuration, évalue la conformité et peut corriger automatiquement les ressources non conformes. Ce guide couvre le cycle complet : activation, règles, conformance packs, remédiation automatique et supervision multi-comptes.

Optimiser les coûts dès le départ

1
Sélectionner les resource types pertinents — Enregistrer uniquement les types critiques (EC2, S3, IAM, RDS, Lambda) plutôt que tous les 300+ types supportés. Réduction coût estimée : 30-60%.
2
Préférer les règles event-driven aux règles périodiques — Les règles déclenchées par changement de configuration sont moins coûteuses que les règles périodiques (moins d'évaluations inutiles).
3
Utiliser les Conformance Packs plutôt que des règles individuelles — Un conformance pack regroupe plusieurs règles en un seul déploiement, simplifié et moins cher en gestion.
4
Un seul Aggregator par Organisation — Ne pas déployer plusieurs aggregators par région. Un aggregator Organization-level suffit généralement pour centraliser la vue et évite surtout les vues doublonnées et la complexité opératoire.
1
Activation & Configuration Recorder Pay-per-use
Configuration Recorder, Delivery Channel (S3 requis, SNS optionnel), multi-région, IAM service-linked role — 20 pts
🚨
Scénario d'attaque : Un attaquant compromet un compte IAM et crée une instance EC2 dans une région secondaire non surveillée, désactive CloudTrail, et exfiltre des données via un bucket S3 mal configuré. Sans AWS Config activé, aucune de ces modifications n'est tracée — l'entreprise ne découvre la brèche qu'en recevant une facture AWS anormale 30 jours plus tard.

Analogie : AWS Config est l'inspecteur permanent de votre infrastructure. Sans lui, votre datacenter cloud n'a ni badge de sécurité, ni registre des accès, ni historique des modifications. Chaque ressource est une pièce sans camera de surveillance.

1.1 — Composants du Configuration Recorder

ℹ️
AWS Config repose sur un Configuration Recorder (enregistre les changements de configuration), un Delivery Channel requis pour livrer l'historique et les snapshots vers S3, et un Service-Linked Role (permissions IAM nécessaires pour lire l'état des ressources). Le topic SNS du delivery channel est optionnel : utile pour diffuser des notifications temps réel, mais non requis pour faire fonctionner AWS Config.

💡
Pourquoi chaque paramètre compte :
Configuration Recorder (CONTINUOUS) : le mode CONTINUOUS enregistre chaque changement en temps réel. Le mode DAILY (snapshot quotidien) manque les modifications éphémères — une instance compromise créée et supprimée dans la même journée est invisible.
SNS Topic : le topic SNS est optionnel, mais recommandé si vous voulez diffuser les changements hors de la console Config vers email, Lambda, EventBridge ou vos workflows internes.
Rétention 7 ans : le RGPD Art. 30 (registre des traitements) et le Code de commerce Art. L123-22 imposent une conservation des traces d'audit pendant 6 à 10 ans selon le secteur. Une rétention de 30 jours (défaut) est juridiquement insuffisante pour la plupart des entreprises.
Multi-région : les attaquants ciblent délibérément les régions AWS non surveillées (ap-southeast, sa-east) pour déployer des ressources non détectées. Config doit être activé dans toutes les régions, même celles considérées comme inactives.
⚠️
Piège fréquent : Activer Config dans la région principale uniquement et supposer que include-global-resource-types: true couvre partout les ressources IAM globales. La doc AWS précise que les IAM globales onboardées avant février 2022 ne peuvent être enregistrées que dans les régions AWS Config supportées avant février 2022, et recommande de les enregistrer une seule fois dans une région supportée pour éviter évaluations inutiles et throttling. Les ressources globales ajoutées après février 2022 doivent, elles, être traitées dans leur home Region.

Autre piège : Le bucket S3 de livraison doit avoir le chiffrement SSE activé et les accès publics bloqués — sinon les snapshots de configuration (qui contiennent les détails de toutes vos ressources) sont potentiellement accessibles.

1.3 — Activation via AWS Console (GUI)

1
Accéder à AWS Config : console.aws.amazon.com/config → Get started (premier accès) ou Settings
2
Configuration Recorder : Sélectionner Record all resources supported in this region. N'activer Include global resources (IAM resources) que dans une région AWS Config legacy si vous devez enregistrer les IAM globales ; sinon traiter les ressources globales dans leur home Region.
3
Delivery Channel — S3 : Choisir ou créer un bucket S3 dédié. Activer le chiffrement SSE-S3 ou SSE-KMS. Format préconisé : config-bucket-{account-id}-{region}
4
Delivery Channel — SNS (optionnel) : Créer ou sélectionner un SNS Topic si vous voulez diffuser les changements vers email, Lambda ou EventBridge ; sinon laisser cette partie non configurée.
5
IAM Role : Laisser AWS créer automatiquement AWSServiceRoleForConfig (recommandé). Ce rôle a les permissions minimales pour lire toutes les ressources.
6
Cliquer sur Save : Config démarre immédiatement l'enregistrement. Vérifier dans Resource inventory que des ressources apparaissent dans les minutes suivantes.

1.4 — Configuration via AWS CLI

AWS CLI — Activation Configuration Recorder (Quick Win)
#=============================================================
# VARIABLES (adapter avant exécution)
#=============================================================
ACCOUNT_ID="$(aws sts get-caller-identity --query Account --output text)"
REGION="eu-west-1"
BUCKET_NAME="config-bucket-${ACCOUNT_ID}-${REGION}"
SNS_TOPIC="arn:aws:sns:${REGION}:${ACCOUNT_ID}:config-notifications"

#=============================================================
# ÉTAPE 1 : Créer le bucket S3 avec chiffrement
#=============================================================
aws s3api create-bucket \
    --bucket "${BUCKET_NAME}" \
    --region "${REGION}" \
    --create-bucket-configuration LocationConstraint=${REGION}

# Bloquer les accès publics (OBLIGATOIRE)
aws s3api put-public-access-block \
    --bucket "${BUCKET_NAME}" \
    --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Chiffrement SSE-S3
aws s3api put-bucket-encryption \
    --bucket "${BUCKET_NAME}" \
    --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# Policy S3 pour autoriser AWS Config à livrer les snapshots
aws s3api put-bucket-policy \
    --bucket "${BUCKET_NAME}" \
    --policy "{
  \"Version\": \"2012-10-17\",
  \"Statement\": [
    {
      \"Sid\": \"AWSConfigBucketPermissionsCheck\",
      \"Effect\": \"Allow\",
      \"Principal\": {\"Service\": \"config.amazonaws.com\"},
      \"Action\": \"s3:GetBucketAcl\",
      \"Resource\": \"arn:aws:s3:::${BUCKET_NAME}\",
      \"Condition\": {\"StringEquals\": {\"AWS:SourceAccount\": \"${ACCOUNT_ID}\"}}
    },
    {
      \"Sid\": \"AWSConfigBucketDelivery\",
      \"Effect\": \"Allow\",
      \"Principal\": {\"Service\": \"config.amazonaws.com\"},
      \"Action\": \"s3:PutObject\",
      \"Resource\": \"arn:aws:s3:::${BUCKET_NAME}/AWSLogs/${ACCOUNT_ID}/Config/*\",
      \"Condition\": {
        \"StringEquals\": {
          \"s3:x-amz-acl\": \"bucket-owner-full-control\",
          \"AWS:SourceAccount\": \"${ACCOUNT_ID}\"
        }
      }
    }
  ]
}"

#=============================================================
# ÉTAPE 2 (optionnelle) : Créer le SNS Topic pour les notifications
#=============================================================
aws sns create-topic \
    --name "config-notifications" \
    --region "${REGION}"

# Autoriser AWS Config à publier sur le SNS Topic
aws sns set-topic-attributes \
    --topic-arn "${SNS_TOPIC}" \
    --attribute-name Policy \
    --attribute-value "{
  \"Version\": \"2012-10-17\",
  \"Statement\": [{
    \"Sid\": \"AWSConfigSNSPolicy\",
    \"Effect\": \"Allow\",
    \"Principal\": {\"Service\": \"config.amazonaws.com\"},
    \"Action\": \"SNS:Publish\",
    \"Resource\": \"${SNS_TOPIC}\",
    \"Condition\": {\"StringEquals\": {\"AWS:SourceAccount\": \"${ACCOUNT_ID}\"}}
  }]
}"

#=============================================================
# ÉTAPE 3 : Créer le Configuration Recorder
#=============================================================
# Pour les IAM globales, n'activer includeGlobalResourceTypes que dans une région AWS Config legacy
aws configservice put-configuration-recorder \
    --configuration-recorder "{
  \"name\": \"default\",
  \"roleARN\": \"arn:aws:iam::${ACCOUNT_ID}:role/aws-service-role/config.amazonaws.com/AWSServiceRoleForConfig\",
  \"recordingGroup\": {
    \"allSupported\": true,
    \"includeGlobalResourceTypes\": false
  },
  \"recordingMode\": {
    \"recordingFrequency\": \"CONTINUOUS\"
  }
}"

#=============================================================
# ÉTAPE 4 : Créer le Delivery Channel
#=============================================================
# Si vous n'utilisez pas SNS, supprimez la ligne snsTopicARN ci-dessous et ignorez l'étape 2
aws configservice put-delivery-channel \
    --delivery-channel "{
  \"name\": \"default\",
  \"s3BucketName\": \"${BUCKET_NAME}\",
  \"snsTopicARN\": \"${SNS_TOPIC}\",
  \"configSnapshotDeliveryProperties\": {
    \"deliveryFrequency\": \"TwentyFour_Hours\"
  }
}"

#=============================================================
# ÉTAPE 5 : Démarrer le Recorder
#=============================================================
aws configservice start-configuration-recorder \
    --configuration-recorder-name "default"

#=============================================================
# ÉTAPE 6 : Configurer la rétention (7 ans = 2557 jours)
#=============================================================
aws configservice put-retention-configuration \
    --retention-period-in-days 2557

Terraform — Configuration Recorder complet (main.tf)
# ============================================================
# Terraform — AWS Config : Recorder + Delivery Channel
# ============================================================

data "aws_caller_identity" "current" {}
data "aws_region" "current" {}

# Bucket S3 pour la livraison des configuration items
resource "aws_s3_bucket" "config_bucket" {
  bucket = "config-bucket-${data.aws_caller_identity.current.account_id}-${data.aws_region.current.name}"
  force_destroy = false
}

resource "aws_s3_bucket_public_access_block" "config_bucket_pab" {
  bucket                  = aws_s3_bucket.config_bucket.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "config_bucket_sse" {
  bucket = aws_s3_bucket.config_bucket.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_s3_bucket_versioning" "config_bucket_versioning" {
  bucket = aws_s3_bucket.config_bucket.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Policy S3 autorisant AWS Config à livrer les snapshots
resource "aws_s3_bucket_policy" "config_bucket_policy" {
  bucket = aws_s3_bucket.config_bucket.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "AWSConfigBucketPermissionsCheck"
        Effect    = "Allow"
        Principal = { Service = "config.amazonaws.com" }
        Action    = "s3:GetBucketAcl"
        Resource  = aws_s3_bucket.config_bucket.arn
        Condition = { StringEquals = { "AWS:SourceAccount" = data.aws_caller_identity.current.account_id } }
      },
      {
        Sid       = "AWSConfigBucketDelivery"
        Effect    = "Allow"
        Principal = { Service = "config.amazonaws.com" }
        Action    = "s3:PutObject"
        Resource  = "${aws_s3_bucket.config_bucket.arn}/AWSLogs/${data.aws_caller_identity.current.account_id}/Config/*"
        Condition = {
          StringEquals = {
            "s3:x-amz-acl"       = "bucket-owner-full-control"
            "AWS:SourceAccount"  = data.aws_caller_identity.current.account_id
          }
        }
      }
    ]
  })
}

# SNS Topic pour les notifications Config
resource "aws_sns_topic" "config_notifications" {
  name = "config-notifications"
}

resource "aws_sns_topic_policy" "config_sns_policy" {

    Version = "2012-10-17"
    Statement = [{
      Sid       = "AWSConfigSNSPolicy"
      Effect    = "Allow"

⚠️

Si la Lambda lève une exception avant l'appel à PutEvaluations, la règle reste en état d'évaluation pendant 1h avant de passer en erreur — créant un angle mort de conformité.

Timeout : la Lambda doit s'exécuter en moins du timeout Config (défaut : 10 secondes). Une règle trop lente → AWS Config time-out → statut d'erreur → faux négatif.

3.2 — Exemple 1 : Custom Rule Lambda — Tag DataClassification obligatoire sur S3

Python — Lambda Custom Rule : S3 must have DataClassification tag
"""
Custom AWS Config Rule — Lambda
Vérifie que chaque bucket S3 a un tag 'DataClassification'
avec une valeur dans : PUBLIC, INTERNAL, CONFIDENTIAL, SECRET
"""
import json
import boto3

config_client = boto3.client('config')
s3_client     = boto3.client('s3')

VALID_CLASSIFICATIONS = {'PUBLIC', 'INTERNAL', 'CONFIDENTIAL', 'SECRET'}

def lambda_handler(event, context):
    invoking_event = json.loads(event['invokingEvent'])
    config_item    = invoking_event.get('configurationItem')

    # Cas : snapshot de configuration (plusieurs items)
    if invoking_event.get('messageType') == 'OversizedConfigurationItemChangeNotification':
        config_item = get_config_item_from_s3(invoking_event)

    if config_item is None or config_item.get('configurationItemStatus') in ('ResourceDeleted', 'ResourceNotRecorded'):
        return put_evaluations(event, [], config_item)

    bucket_name = config_item['resourceName']
    compliance  = evaluate_bucket(bucket_name, config_item)

    evaluation = {
        'ComplianceResourceType': 'AWS::S3::Bucket',
        'ComplianceResourceId'  : bucket_name,
        'ComplianceType'        : compliance,
        'OrderingTimestamp'     : config_item['configurationItemCaptureTime']
    }
    return put_evaluations(event, [evaluation], config_item)

def evaluate_bucket(bucket_name, config_item):
    # Récupérer les tags depuis le configuration item (sans appel S3 supplémentaire)
    tags = config_item.get('tags', {})
    classification = tags.get('DataClassification')
    if not classification:
        return 'NON_COMPLIANT'  # Tag absent
    if classification.upper() not in VALID_CLASSIFICATIONS:
        return 'NON_COMPLIANT'  # Valeur invalide
    return 'COMPLIANT'

def put_evaluations(event, evaluations, config_item):
    config_client.put_evaluations(
        Evaluations  = evaluations,
        ResultToken  = event['resultToken'],
        TestMode     = False
    )
    return {'status': 'evaluated', 'count': len(evaluations)}

5.1 — Tableau de configuration : Auto-Remediation

Paramètre Défaut AWS ⚡ Quick Win 🔒 Avancé 🎯 Optimal
Type de remédiation Aucune (manuel) Manuelle (notification SNS uniquement) Automatique sur les règles critiques S3/SG Automatique sur toutes les règles Critique + Élevé

Mode d'exécution
(automatique vs manuel)		 Exécution manuelle uniquement Manuelle (approval required) Automatique pour règles non-disruptives Auto pour Critique, Manuel pour Élevé (sauf exceptions)
Retry Logic
(nombre de tentatives)		 Aucun retry N/A Maximum 3 tentatives automatiques 3 tentatives + alerte SNS si échec
IAM Role de remédiation
(permissions SSM Automation)		 N/A N/A Rôle dédié avec permissions minimales Rôle dédié + audit trail CloudTrail des actions
Exceptions de remédiation
(ResourceIdScope)		 N/A N/A Exceptions via tag ConfigRemediationExempt=true Exceptions documentées + révisées tous les 90 jours
💡
Pourquoi chaque paramètre est critique :
Auto vs Manuel : la remédiation automatique doit être réservée aux actions non-disruptives (bloquer un accès public S3, fermer un port 22 ouvert vers 0.0.0.0/0). Pour les actions potentiellement disruptives (supprimer une instance, désactiver une clé IAM active), préférer la remédiation manuelle avec approbation.
Retry à 3 : certains Documents SSM échouent en cas de race condition (ressource modifiée pendant la remédiation). 3 tentatives avec backoff exponentiel couvrent 99% des cas transitoires sans créer de boucle infinie.
Rôle dédié : éviter d'utiliser le rôle Admin ou Config lui-même — le rôle SSM Automation doit avoir les permissions minimales pour les actions de remédiation spécifiques uniquement.
⚠️
Piège d'architecture : Activer l'auto-remediation sur restricted-ssh sans exception peut bloquer les pipelines CI/CD qui utilisent SSH. Toujours tester la remédiation en mode Manuel en premier, puis passer en Automatique après validation.

Boucle infinie : Une remédiation qui modifie une ressource déclenchant une nouvelle évaluation Config → nouvelle non-conformité → nouvelle remédiation. Concevoir les remédiation pour être idempotentes (même résultat si exécuté N fois).

Aggregators & Multi-Account Sans coût additionnel direct
Config Aggregator (multi-compte + multi-région), Organization Aggregator, vue centralisée — 10 pts
🚨
Scénario d'attaque : L'organisation dispose de 50 comptes AWS. Sans Aggregator, chaque compte doit être audité individuellement — un attaquant compromet un compte de développement avec des règles Config moins strictes, pivote vers un compte production via des rôles cross-account, et aucun tableau de bord centralisé ne corrèle les événements. Avec un Aggregator, la vue de conformité de l'ensemble de l'organisation est visible depuis un seul endroit.

Analogie : L'Aggregator est la tour de contrôle centrale — il collecte les données de conformité de toutes les régions et tous les comptes dans une vue unique.

6.1 — Tableau de configuration : Aggregators

💡
Pourquoi chaque paramètre compte :
Organization vs Account Aggregator : l'ORGANIZATION Aggregator inclut automatiquement tout nouveau compte AWS créé dans l'organisation — pas besoin de mettre à jour la liste manuellement. Pour une organisation qui crée de nouveaux comptes régulièrement, c'est essentiel.
Delegated Admin : le compte Management AWS Organizations ne doit pas être utilisé pour les opérations quotidiennes (best practice). Déléguer la gestion Config à un compte "Audit" dédié permet d'isoler les privilèges et de respecter la séparation des responsabilités.
Toutes les régions : l'Aggregator collecte les données Config depuis les régions sources — si une région n'est pas incluse, ses ressources ne sont pas visibles dans la vue consolidée. Les régions secondaires sont souvent les moins surveillées.
⚠️
Prérequis souvent manqué : Pour déployer un Organization Aggregator, AWS Config doit être activé dans chaque compte membre. Si un compte n'a pas de Configuration Recorder actif, ses données ne remontent pas dans l'Aggregator.

Coût de l'Aggregator : selon la doc AWS, l'utilisation des aggregators n'entraîne pas de coût AWS Config additionnel direct. Les coûts restent liés aux configuration items, règles, packs, S3/SNS/Lambda et autres services dans les comptes source.

AWS CLI — Advanced Queries : Inventaire sécurité complet

# Advanced Queries — Inventaire Sécurité AWS Config
# Syntaxe SQL-like (SELECT, WHERE, GROUP BY)
# ============================================================

# Q1 : Instances EC2 sans IMDSv2 (vecteur SSRF)
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, awsRegion, accountId,
    configuration.metadataOptions.httpTokens
    WHERE resourceType = 'AWS::EC2::Instance'
    AND configuration.metadataOptions.httpTokens != 'required'"

# Q2 : Buckets S3 publics (lecture ou écriture)
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, awsRegion, accountId,
    configuration.publicAccessBlockConfiguration
    WHERE resourceType = 'AWS::S3::Bucket'
    AND (configuration.publicAccessBlockConfiguration.blockPublicAcls = false
    OR configuration.publicAccessBlockConfiguration.blockPublicPolicy = false)"

# Q3 : Volumes EBS non chiffrés
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, awsRegion, accountId,
    configuration.encrypted, configuration.size
    WHERE resourceType = 'AWS::EC2::Volume'
    AND configuration.encrypted = false"

# Q4 : Instances RDS publiquement accessibles
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, awsRegion, accountId,
    configuration.publiclyAccessible, configuration.engine, configuration.engineVersion
    WHERE resourceType = 'AWS::RDS::DBInstance'
    AND configuration.publiclyAccessible = true"

# Q5 : Clés IAM Access Keys non utilisées depuis 90 jours
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, configuration.userName,
    configuration.accessKeyMetadata
    WHERE resourceType = 'AWS::IAM::User'
    AND configuration.accessKeyMetadata.status = 'Active'"

# Q6 : Compter les ressources par type et région
aws configservice select-resource-config \
    --expression "SELECT resourceType, awsRegion, COUNT(*) AS count
    WHERE resourceType IN ('AWS::EC2::Instance', 'AWS::S3::Bucket', 'AWS::RDS::DBInstance', 'AWS::Lambda::Function')
    GROUP BY resourceType, awsRegion"

# Q7 : Security Groups autorisant SSH depuis internet
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceName, awsRegion, accountId, tags
    WHERE resourceType = 'AWS::EC2::SecurityGroup'
    AND configuration.ipPermissions.fromPort = 22
    AND configuration.ipPermissions.ipRanges = '0.0.0.0/0'"

# Q8 : Requête cross-account via Aggregator
aws configservice select-aggregate-resource-config \
    --configuration-aggregator-name "org-config-aggregator" \
    --expression "SELECT accountId, awsRegion, COUNT(*) AS public_buckets
    WHERE resourceType = 'AWS::S3::Bucket'
    AND configuration.publicAccessBlockConfiguration.blockPublicAcls = false
    GROUP BY accountId, awsRegion
    ORDER BY public_buckets DESC"

# Q9 : Ressources sans tag 'Environment' (gouvernance tagging)
aws configservice select-resource-config \
    --expression "SELECT resourceId, resourceType, awsRegion, tags
    WHERE resourceType IN ('AWS::EC2::Instance', 'AWS::S3::Bucket', 'AWS::RDS::DBInstance')
    AND tags.Environment NOT EXISTS"

Script All-in-One — Hardening AWS Config Complet
Script AWS CLI exécutable en une seule fois pour activer et configurer AWS Config selon les best practices
💡
Script idempotent : Ce script peut être exécuté plusieurs fois sans créer de doublons. Définir les variables d'environnement avant exécution. Testé sur AWS CLI v2.x.
Bash — AWS Config Hardening All-in-One (config_hardening.sh)
#!/usr/bin/env bash
# =============================================================================
# AWS Config Hardening — Script All-in-One
# Idempotent : peut être ré-exécuté sans impact sur l'existant
# Prérequis : aws-cli v2, jq, droits IAM Config + S3 + SNS
# =============================================================================
set -euo pipefail

# Variables — À adapter
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
AWS_REGION=${AWS_DEFAULT_REGION:-us-east-1}
DELIVERY_BUCKET="aws-config-logs-${ACCOUNT_ID}-${AWS_REGION}"
ROLE_ARN="arn:aws:iam::${ACCOUNT_ID}:role/aws-service-role/config.amazonaws.com/AWSServiceRoleForConfig"

echo "[Phase 1/4] Activation et Configuration Recorder..."

# Bucket S3 avec chiffrement AES256
if ! aws s3api head-bucket --bucket "${DELIVERY_BUCKET}" 2>/dev/null; then
  aws s3api create-bucket --bucket "${DELIVERY_BUCKET}" --region "${AWS_REGION}" \
    $([ "${AWS_REGION}" != "us-east-1" ] && echo "--create-bucket-configuration LocationConstraint=${AWS_REGION}")
  aws s3api put-bucket-encryption --bucket "${DELIVERY_BUCKET}" \
    --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'
  aws s3api put-public-access-block --bucket "${DELIVERY_BUCKET}" \
    --public-access-block-configuration '{"BlockPublicAcls":true,"IgnorePublicAcls":true,"BlockPublicPolicy":true,"RestrictPublicBuckets":true}'
  echo "  ✔ Bucket créé : ${DELIVERY_BUCKET}"
else
  echo "  ↩ Bucket déjà existant : ${DELIVERY_BUCKET}"
fi

# Configuration Recorder
# Activer includeGlobalResourceTypes uniquement dans une région AWS Config legacy si les IAM globales sont requises
aws configservice put-configuration-recorder --configuration-recorder \
    "{\"name\":\"default\",\"roleARN\":\"${ROLE_ARN}\",\"recordingGroup\":{\"allSupported\":true,\"includeGlobalResourceTypes\":false}}"

# Delivery Channel
aws configservice put-delivery-channel --delivery-channel \
  "{\"name\":\"default\",\"s3BucketName\":\"${DELIVERY_BUCKET}\",\"configSnapshotDeliveryProperties\":{\"deliveryFrequency\":\"TwentyFour_Hours\"}}"

aws configservice start-configuration-recorder --configuration-recorder-name default
echo "  ✔ Configuration Recorder démarré"

echo "[Phase 2/4] Managed Rules Quick Win (15 règles essentielles)..."
QW_RULES=(
  "root-account-mfa-enabled:ROOT_ACCOUNT_MFA_ENABLED:TwentyFour_Hours"
  "cloud-trail-enabled:CLOUD_TRAIL_ENABLED:TwentyFour_Hours"
  "s3-bucket-public-read-prohibited:S3_BUCKET_PUBLIC_READ_PROHIBITED:CONFIGURATION_CHANGE"
  "s3-bucket-public-write-prohibited:S3_BUCKET_PUBLIC_WRITE_PROHIBITED:CONFIGURATION_CHANGE"
  "s3-bucket-server-side-encryption-enabled:S3_BUCKET_SERVER_SIDE_ENCRYPTION_ENABLED:CONFIGURATION_CHANGE"
  "ec2-ebs-encryption-by-default:EC2_EBS_ENCRYPTION_BY_DEFAULT:TwentyFour_Hours"
  "vpc-flow-logs-enabled:VPC_FLOW_LOGS_ENABLED:TwentyFour_Hours"
  "vpc-default-security-group-closed:VPC_DEFAULT_SECURITY_GROUP_CLOSED:CONFIGURATION_CHANGE"
  "restricted-ssh:INCOMING_SSH_DISABLED:CONFIGURATION_CHANGE"
  "restricted-common-ports:RESTRICTED_INCOMING_TRAFFIC:CONFIGURATION_CHANGE"
  "ec2-imdsv2-check:EC2_IMDSV2_CHECK:CONFIGURATION_CHANGE"
  "access-keys-rotated:ACCESS_KEYS_ROTATED:TwentyFour_Hours"
  "iam-root-access-key-check:IAM_ROOT_ACCESS_KEY_CHECK:TwentyFour_Hours"
  "s3-bucket-logging-enabled:S3_BUCKET_LOGGING_ENABLED:CONFIGURATION_CHANGE"
  "multi-region-cloudtrail-enabled:MULTI_REGION_CLOUD_TRAIL_ENABLED:TwentyFour_Hours"
)

for entry in "${QW_RULES[@]}"; do
  IFS=: read -r name source_id frequency <<< "${entry}"
  aws configservice put-config-rule --config-rule \
    "{\"ConfigRuleName\":\"${name}\",\"Source\":{\"Owner\":\"AWS\",\"SourceIdentifier\":\"${source_id}\"},\"MaximumExecutionFrequency\":\"${frequency}\"}" 2>/dev/null || true
  echo "  ✔ ${name}"
done

echo "[Phase 3/4] Conformance Pack CIS AWS Foundations Level 1..."
aws configservice put-conformance-pack \
  --conformance-pack-name "cis-foundations-level1" \
  --template-s3-uri "s3://aws-config-rules-packages-${AWS_REGION}/aws-cis-foundations-benchmark-level-1.yaml" \
  --delivery-s3-bucket "${DELIVERY_BUCKET}" 2>/dev/null || true
echo "  ✔ Conformance Pack CIS Level 1 déployé (status async)"

echo "[Phase 4/4] Remédiation automatique S3 Block Public Access..."
REMEDIATION_ROLE="arn:aws:iam::${ACCOUNT_ID}:role/config-remediation-role"
aws configservice put-remediation-configurations \
  --remediation-configurations "[{
    \"ConfigRuleName\": \"s3-bucket-public-read-prohibited\",
    \"TargetType\": \"SSM_DOCUMENT\",
    \"TargetId\": \"AWS-DisablePublicAccessForS3Bucket\",
    \"Parameters\": {
      \"AutomationAssumeRole\": {\"StaticValue\": {\"Values\": [\"${REMEDIATION_ROLE}\"]}},
      \"S3BucketName\": {\"ResourceValue\": {\"Value\": \"RESOURCE_ID\"}}
    },
    \"Automatic\": true,
    \"MaximumAutomaticAttempts\": 3,
    \"RetryAttemptSeconds\": 60
  }]" 2>/dev/null || true

echo ""
echo "✅ AWS Config Hardening complet — $(date)"
echo "   Compte       : ${ACCOUNT_ID}"
echo "   Région        : ${AWS_REGION}"
echo "   Bucket S3     : ${DELIVERY_BUCKET}"
echo "   Règles actives : $(aws configservice describe-config-rules --query 'length(ConfigRules)' --output text)"

Glossaire
Définitions des termes clés AWS Config
Terme Définition
Configuration Item (CI) Snapshot de la configuration d'une ressource AWS à un instant T. Contient les attributs, les relations, et les métadonnées. Base de toutes les évaluations Config.
Configuration Recorder Composant AWS Config qui détecte les changements de configuration et enregistre les Configuration Items. Doit être démarré explicitement (start-configuration-recorder). Un seul par région et par compte.
Delivery Channel Canal par lequel AWS Config envoie les Configuration Items et les évaluations vers un bucket S3 (et optionnellement une queue SNS). Configure la fréquence de snapshot (1h à 24h).
Managed Rule Règle d'évaluation pré-construite et maintenue par AWS. Exempt des coûts Lambda. Plus de 200 règles disponibles dans le catalogue. Évaluation CONFIGURATION_CHANGE ou périodique (TwentyFour_Hours).
Custom Rule Règle définie par l'utilisateur, évaluée par une fonction Lambda (Python/Node.js) ou par AWS CloudFormation Guard (DSL déclaratif). Permet des contrôles métier non couverts par les Managed Rules.
Conformance Pack Collection de règles Config et d'actions de remédiation packagées ensemble dans un template YAML. Disponible en packs AWS managés (CIS, PCI DSS, NIST, HIPAA) ou custom. Déployable en organisation entière.
Auto-Remediation Action automatique déclenchée lorsqu'une ressource devient non-conforme. Basée sur des documents SSM Automation. Configurable : manuelle, automatique, avec retry et seuil de tentatives maximum.
Config Aggregator Vue centralisée des données Config de plusieurs comptes et/ou régions. Deux types : Account Aggregator (liste explicite de comptes) et Organization Aggregator (tous les comptes de l'org AWS Organizations).
Config Timeline Historique chronologique des changements de configuration d'une ressource. Permet de visualiser l'état passé et de corréler les changements avec les évaluations de conformité.
Advanced Queries Fonctionnalité permettant d'interroger le Configuration State de l'inventaire AWS Config via un SQL-like (AWS Config Query Language). Supporte les requêtes cross-account via les Aggregators.
Organization Aggregator Aggregator utilisant AWS Organizations pour collecter automatiquement les données Config de tous les comptes membres. Nécessite un Delegated Administrator et le trust AWS Config → Organizations.
Compliance Score Pourcentage de ressources conformes par rapport au total des ressources évaluées par un Conformance Pack ou une règle. Consultable via get-conformance-pack-compliance-summary.
Service Control Policy (SCP) Politique AWS Organizations qui restreint les actions IAM disponibles dans les comptes membres. Complémentaire à Config : les SCP empêchent les actions non autorisées, Config détecte les états non conformes.
