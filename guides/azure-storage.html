Solutions écartées

🚫
Pourquoi pas les Access Keys en production ? Les clés d'accès Storage (Shared Key) donnent un accès total à l'ensemble du compte — Blob, Files, Queues, Tables — sans possibilité de restriction RBAC. En cas de fuite, la seule révocation est la rotation de clé qui coupe toutes les applications simultanément. Alternative retenue : Managed Identity + RBAC granulaire.

Pourquoi pas NFS v3 sur Azure Files pour Windows ? NFS v4.1 (Azure Files Premium) ne supporte pas l'authentification Kerberos / Microsoft Entra ID. Alternative retenue : SMB 3.1.1 avec Entra ID Kerberos pour les environnements Windows, NFS uniquement pour les workloads Linux sur VNET privé.

Pourquoi pas les Account SAS au lieu des User Delegation SAS ? Les Account SAS sont signés avec la clé Storage Account (Shared Key) — si la clé est compromise, tous les SAS le sont. Les User Delegation SAS sont signés par une identité Microsoft Entra ID et révocables individuellement. Alternative retenue : User Delegation SAS + Stored Access Policy.

Authentification & Contrôle d'Accès Pay-as-you-go
Managed Identity + RBAC vs SAS Token vs Access Keys — éliminer le passe-partout universel
🚨
Scénario d'attaque : Un développeur commit accidentellement une Access Key dans un dépôt GitHub public. Un attaquant la récupère en quelques minutes via des scanners automatisés, exfiltre tous les blobs du Storage Account, puis efface les données. La clé accès donne un accès total — sans expiration, sans IP restriction, sans audit trail précis.
🎰 Analogie : l'Access Key est le passe-partout universel de l'entrepôt. Si quelqu'un le copie, il entre dans toutes les pièces à toute heure, pour toujours.

Source : Microsoft Learn — Prevent Shared Key authorization • CIS Microsoft Azure Foundations Benchmark v3.0 §3.1, §3.3

💡
Pourquoi chaque paramètre compte :
• allowSharedKeyAccess = false : force toutes les applications à utiliser Entra ID (recommandation Microsoft Cloud Security Benchmark IM-1, non couvert nominativement par CIS Azure v6.0.0 §9.3)D ou SAS signé par clé — pas possible de contourner par brute force sur la clé. CIS Azure §3.3 : "Ensure that 'Allow storage account key access' is set to 'Disabled'". Sans ce paramètre, une application compromisent peut utiliser l'Access Key pour accéder à tous les blobs.
• SAS expiration courte : un SAS volé (dans les logs, un email, un commit) est inutilisable après expiration. Verizon DBIR 2024 : 68% des attaques cloud exploitent des crédentiels exposés.
• signedProtocol = https : empêche l'interception du SAS en clair sur un réseau non chiffré (attaque man-in-the-middle).
⚠️
Piège fréquent : Désactiver allowSharedKeyAccess casse les applications qui utilisent encore les Access Keys ou les SAS Account signés (y compris AzCopy avec clé). Migrer toutes les applications vers Managed Identity ou User Delegation SAS AVANT de désactiver. Vérifier via Azure Monitor : filtrer les logs sur AuthenticationType = SharedKey avant de basculer.

1.3 — Configuration (Azure CLI + PowerShell + Portail)

Azure CLI — Désactiver Access Keys + configurer SAS Policy
# Désactiver l'accès par clés partagées (Shared Key)
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --allow-shared-key-access false

# Configurer la politique SAS : expiration max 1h, action Log
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --sas-expiration-period "0.01:00:00" \
  --key-expiration-period-in-days 90

# Assigner le rôle Storage Blob Data Contributor à une Managed Identity
$STORAGE_ID=$(az storage account show \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --query id --output tsv)

az role assignment create \
  --role "Storage Blob Data Contributor" \
  --assignee "<ManagedIdentityClientId>" \
  --scope $STORAGE_ID
PowerShell (Az.Storage) — Désactiver Access Keys + SAS Policy
Set-AzStorageAccount `
  -ResourceGroupName "<ResourceGroup>" `
  -Name "<StorageAccountName>" `
  -AllowSharedKeyAccess $false

# Politique SAS : expiration max 1 heure
Set-AzStorageAccount `
  -ResourceGroupName "<ResourceGroup>" `
  -Name "<StorageAccountName>" `
  -SasExpirationPeriod "0.01:00:00" `
  -KeyExpirationPeriodInDay 90
Terraform (azurerm_storage_account) — Désactiver Access Keys
resource "azurerm_storage_account" "example" {
  name                     = "<storage_account_name>"
  resource_group_name      = var.resource_group_name
  location                 = var.location
  account_tier             = "Standard"
  account_replication_type = "LRS"

  # Désactiver Shared Key Access
  shared_access_key_enabled = false

  # SAS Policy : expiration max 1h
  sas_policy {
    expiration_period = "0.01:00:00"
    expiration_action = "Log"
  }
}

Chiffrement Pay-as-you-go
SSE AES-256 at-rest, Customer-Managed Keys, double chiffrement, TLS 1.2 in-transit
🚨
Scénario d'attaque : Un disque physique d'un datacenter Azure est volé (scénario hypothétique documenté par les fournisseurs cloud). Sans chiffrement at-rest, les données sont lisibles directement. Sans TLS enforced, un accès HTTP expose les données en transit (attaque MITM sur réseaux non sécurisés).
🔒 Analogie : le chiffrement at-rest, c'est le coffre-fort dans l'entrepôt — même si quelqu'un entre dans la pièce, il ne peut pas lire le contenu sans la clé. Le TLS, c'est le transport blindé entre l'entrepôt et son destinataire.

💡
Pourquoi chaque paramètre compte :
• supportsHttpsTrafficOnly = true : sans ce paramètre, une application peut se connecter en HTTP. CIS Azure v6.0.0 §9.3.4 (Secure transfer required) & §9.3.6 (Min TLS 1.2)nnecter en HTTP clair — les données transitent non chiffrées sur le réseau. CIS Azure §3.15 : recommandation Level 1.
• minimumTlsVersion = TLS1_2 : TLS 1.0 et 1.1 ont des vulnérabilités connues (POODLE, BEAST). NIST SP 800-52r2 : TLS 1.2 minimum depuis 2020.
• Customer-Managed Keys (CMK) : avec les Microsoft-managed keys, Microsoft peut techniquement accéder aux clés. CMK place le contrôle dans votre Key Vault — révocation de la clé = aucun accès aux données. Requis ISO 27001 Annexe A.10, RGPD Art. 25 (protection by design).
• requireInfrastructureEncryption : double couche de chiffrement pour les secteurs réglementés (santé, finance). Exigence possible HIPAA, PCI-DSS. Configurable uniquement à la création du Storage Account — impossible d'activer après coup.
⚠️
Piège CMK : Si la clé CMK dans Key Vault est supprimée ou expirée, le Storage Account devient inaccessible (toutes les lectures/écritures échouent). Activer obligatoirement le Soft Delete sur Key Vault (90 jours min) et la purge protection avant de configurer CMK.

2.3 — Configuration Customer-Managed Keys (CMK)

Azure CLI — Activer CMK via Key Vault
# Prérequis : Key Vault avec Soft Delete + Purge Protection activés
# Étape 1 : Activer Managed Identity sur le Storage Account
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --assign-identity

# Étape 2 : Récupérer le Principal ID de la Managed Identity
$PRINCIPAL_ID=$(az storage account show \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --query "identity.principalId" --output tsv)

# Étape 3 : Donner accès Key Vault au Storage Account
az keyvault set-policy \
  --name "<KeyVaultName>" \
  --object-id $PRINCIPAL_ID \
  --key-permissions get unwrapKey wrapKey

# Étape 4 : Récupérer la clé CMK
$KEY_ID=$(az keyvault key show \
  --vault-name "<KeyVaultName>" \
  --name "<KeyName>" \
  --query "key.kid" --output tsv)

# Étape 5 : Activer CMK sur le Storage Account
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --encryption-key-source Microsoft.Keyvault \
  --encryption-key-vault "https://<KeyVaultName>.vault.azure.net" \
  --encryption-key-name "<KeyName>" \
  --encryption-key-version ""
Terraform — HTTPS + TLS 1.2 + Infrastructure Encryption (à la création)
resource "azurerm_storage_account" "secure" {
  name                     = "<storage_account_name>"
  resource_group_name      = var.resource_group_name

  account_replication_type = "LRS"

Network Access Control Pay-as-you-go
Firewall Storage Account, Private Endpoints, Service Endpoints — fermer le stockage à Internet
🚨
Scénario d'attaque : Par défaut, un Storage Account est accessible depuis Internet. Un attaquant avec une Access Key volée (ou un SAS exposé) peut exfiltrer des térabytes depuis n'importe quel PC dans le monde. Sans Private Endpoint ni Firewall, même un Storage Account bien authentifié est exposé au brute-force, aux scanners automatisés et aux accès depuis des réseaux compromis.
🛣️ Analogie : le Private Endpoint, c'est le couloir privé — la livraison entre ton VNET et le stockage ne passe jamais par la rue (Internet), même si quelqu'un surveille le trafic extérieur.

💡
Pourquoi chaque paramètre compte :
• defaultAction = Deny : passe d'une posture "tout autorisé sauf liste noire" à "tout refusé sauf liste blanche" (CIS Azure v6.0.0 §9.3 Storage Accounts — Networking)fusé sauf liste blanche". CIS Azure §3.7 : recommandation Level 2. Sans ce paramètre, tout changement d'application ouvre automatiquement l'accès.
• Private Endpoint vs Service Endpoint : le Service Endpoint route le trafic via le backbone Azure mais depuis une IP publique du VNET — un Storage Account avec SE est toujours visible sur Internet. Le Private Endpoint attribue une IP privée dans le VNET — le stockage n'a plus d'IP publique accessible.
• bypass = AzureServices : permet à Azure Monitor, Azure Backup, Azure Site Recovery d'accéder au stockage même avec Firewall = Deny. À garder pour les services Microsoft, à supprimer si tout est en Private Endpoint.
⚠️
Piège : Service Endpoint ≠ Private Endpoint. Le Service Endpoint est plus simple à configurer mais moins sécurisé : le trafic reste sur le backbone Azure mais le Storage Account reste accessible depuis Internet via son endpoint public. Pour une sécurité optimale, utiliser les Private Endpoints et désactiver publicNetworkAccess.

3.2 — Configuration Firewall (Azure CLI + PowerShell + Terraform)

Azure CLI — Firewall : Default Deny + IP autorisées
# Définir le Firewall : Default Deny + autoriser IP bureau
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --default-action Deny \
  --bypass AzureServices Logging Metrics

# Ajouter une plage IP autorisée
az storage account network-rule add \
  --account-name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --ip-address "<IPOrRange>"

# Ajouter un sous-réseau VNET (Service Endpoint)
az storage account network-rule add \
  --account-name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --vnet-name "<VNetName>" \
  --subnet "<SubnetName>"
Azure CLI — Créer un Private Endpoint pour Blob
$STORAGE_ID=$(az storage account show \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --query id -o tsv)

az network private-endpoint create \
  --name "pe-<StorageAccountName>-blob" \
  --resource-group "<ResourceGroup>" \
  --vnet-name "<VNetName>" \
  --subnet "<SubnetName>" \
  --private-connection-resource-id $STORAGE_ID \
  --group-id blob \
  --connection-name "conn-<StorageAccountName>-blob"

# Désactiver l'accès public après création du Private Endpoint
az storage account update \
  --name "<StorageAccountName>" \
  --resource-group "<ResourceGroup>" \
  --public-network-access Disabled

Update-AzStorageAccountNetworkRuleSet `
  -ResourceGroupName "<ResourceGroup>" `
  -Name "<StorageAccountName>" `
  -DefaultAction Deny `

• Identity-based auth (Kerberos) : avec la clé de stockage, tout utilisateur qui obtient la clé monte le partage avec droits complets. Avec Kerberos, l'accès est lié à l'identité Active Directory — RBAC fin par dossier/fichier, audit trail Entra ID.
• SMB 3.0 minimum : SMB 2.1 ne supporte pas le chiffrement SMB. Forcer SMB 3.0 garantit que toutes les connexions sont chiffrées en transit, même sur un réseau compromis.
• Private Endpoint pour Files : le port 445 (SMB) est souvent bloqué par les FAI sur les connexions domestiques. Un Private Endpoint via VNET/VPN évite d'ouvrir le port 445 à Internet.
⚠️
Piège NFS v4.1 : Les partages Azure Files NFS (Linux) ne supportent pas l'authentification Kerberos — ils utilisent les IP VNET pour le contrôle d'accès. Les partages NFS doivent obligatoirement être dans un VNET avec Private Endpoint et ne jamais être exposés à Internet.

Audit vs Deny — quelle stratégie :
• Mode Audit : détecte les non-conformités sans bloquer. Idéal pour la phase initiale d'inventaire. Defender for Cloud affiche les non-conformes dans le Secure Score.

💡
Rotation de clés sans interruption de service :
La stratégie "Key1 → Key2" permet une rotation sans downtime :
1⃣ Basculer toutes les apps sur key2 (ou Key Vault référence)
2⃣ Régénérer key1 (l'ancienne clé compromise est maintenant invalide)
3⃣ Basculer les apps sur key1 (nouvelle valeur)
4⃣ Régénérer key2 (hygiène)
→ Zéro downtime, clés compromises invalides dès l'étape 2.
⚠️
Piège User Delegation SAS : La commande revoke-delegation-keys invalide tous les User Delegation SAS du compte, y compris les SAS légitimes distribués aux clients. Glossaire
Acronymes et termes techniques utilisés dans ce guide
Acronyme / TermeDéfinition
ACLAccess Control List — liste de contrôle d'accès définissant les permissions d'accès aux ressources (ex: Azure Data Lake Storage Gen2)
ADLSAzure Data Lake Storage Gen2 — stockage optimisé pour l'analyse de big data, basé sur Azure Blob Storage avec support des ACL hiérarchiques
AESAdvanced Encryption Standard — algorithme de chiffrement symétrique. AES-256-GCM est le standard actuel pour SMB et chiffrement Azure Storage
CMKCustomer-Managed Key — clé de chiffrement gérée par le client dans Azure Key Vault, par opposition aux clés gérées par Microsoft (SSE-M)
CORSCross-Origin Resource Sharing — mécanisme permettant à un navigateur web d'accéder à des ressources sur un domaine différent. À restreindre sur Azure Storage
GRSGeo-Redundant Storage — réplication synchrone dans la région primaire + réplication asynchrone vers une région secondaire distante
HTTPSHyperText Transfer Protocol Secure — HTTP avec couche TLS. Obligatoire pour tout accès Azure Storage (paramètre supportsHttpsTrafficOnly)
KQLKusto Query Language — langage de requête utilisé dans Azure Monitor Log Analytics, Application Insights et Microsoft Sentinel
LAWLog Analytics Workspace — espace de travail Azure Monitor pour stocker et analyser les logs envoyés via Diagnostic Settings
LRSLocally Redundant Storage — réplication synchrone dans 3 copies au sein d'un seul datacenter de la région primaire
MDC / MDFCMicrosoft Defender for Cloud (anciennement Azure Security Center + Azure Defender) — plateforme CSPM et protection des workloads cloud
MDCAMicrosoft Defender for Cloud Apps (anciennement MCAS) — solution CASB de Microsoft pour la sécurité des applications cloud
NFSNetwork File System — protocole de partage de fichiers réseau pour Linux/Unix. Supporté par Azure Files Premium (NFS v4.1, sans Kerberos)
NSGNetwork Security Group — pare-feu réseau Azure contrôlant le trafic entrant/sortant d'une subnet ou d'une interface réseau
RBACRole-Based Access Control — contrôle d'accès basé sur les rôles. Pour Azure Storage : Storage Blob Data Owner/Contributor/Reader
RGPD / GDPRRèglement Général sur la Protection des Données (EU 2016/679) — réglement européen encadrant la collecte et le traitement des données personnelles
SASShared Access Signature — token d'accès temporaire et délégué à des ressources Azure Storage. Types : Service SAS, Account SAS, User Delegation SAS
SMBServer Message Block — protocole de partage de fichiers réseau Windows. Azure Files supporte SMB 2.1, 3.0, 3.1.1 (SMB 3.x avec chiffrement obligatoire)
SSEStorage Service Encryption — chiffrement au repos automatique de toutes les données Azure Storage (activé par défaut, AES-256)
TLSTransport Layer Security — protocole de sécurité réseau remplaçant SSL. Azure Storage exige TLS 1.2 minimum (TLS 1.0/1.1 dépréciés)
VNETVirtual Network — réseau virtuel privé Azure isolé logiquement. Les Service Endpoints et Private Endpoints permettent de restreindre l'accès Storage à un VNET
WORMWrite Once Read Many — politique d'immutabilité empêchant la modification ou suppression de blobs pendant une durée définie (Azure Immutable Blob Storage)
ZRSZone-Redundant Storage — réplication synchrone dans 3 zones de disponibilité de la région primaire. Haute disponibilité sans indisponibilité en cas de panne de zone
