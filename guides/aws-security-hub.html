Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de configuration AWS Security Hub
📋
Ce document est un référentiel complet couvrant toutes les configurations AWS Security Hub avec les best practices AWS, CIS et NIST. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un environnement AWS.

Analogie : Security Hub est le tableau de bord central de sécurité de votre environnement AWS — une vue unifiée de tous les problèmes de sécurité, comme un centre de commande qui agrège les alertes de toutes les caméras de surveillance (GuardDuty, Inspector, Macie, Config).

Solutions écartées

🔄
Pourquoi Security Hub et pas un outil tiers ?
  • Prisma Cloud (Palo Alto) — Couverture multi-cloud supérieure, mais coût ~15-25 $/ressource/mois vs ~0,001 $/finding pour Security Hub. Non natif AWS, latence sur les nouveaux services.
  • Wiz — Excellent CSPM agentless, mais overlay supplémentaire, pas d'intégration native EventBridge/Organizations. Coût significativement plus élevé.
  • Prowler (open source) — Gratuit, bon pour les audits ponctuels CIS/FSBP, mais pas de dashboard centralisé, pas d'automation native, maintenance manuelle.
  • Choix retenu : Security Hub — Natif AWS, intégration directe avec Organizations/EventBridge/GuardDuty/Inspector, coût proportionnel à l'usage, 30 jours gratuits pour évaluation.

💡
Pourquoi activer Security Hub dans TOUTES les régions ? Même si vous n'utilisez que eu-west-1, un attaquant peut lancer des ressources dans ap-southeast-1 (crypto-mining, exfiltration). Sans Security Hub activé dans cette région, aucun finding ne sera généré et l'attaque restera invisible. Le coût dans les régions inutilisées est quasi nul (0 finding = 0 $).
⚠️
Piège classique — Oublier AWS Config : Security Hub requiert AWS Config pour les security checks (standards FSBP, CIS, etc.). Si Config n'est pas activé dans une région, Security Hub s'active mais tous les contrôles restent en état NO_DATA. Vous pensez être protégé mais aucune évaluation n'est effectuée — faux sentiment de sécurité.
Vérifiez systématiquement : aws configservice describe-configuration-recorders

Configuration — AWS CLI

Bash — Activation Security Hub + AWS Config multi-région
#!/bin/bash
# === S1 — Activation Security Hub multi-région ===
# Prérequis : AWS CLI v2, credentials admin, AWS Config actif

# Variables
AGGREGATION_REGION="eu-west-1"
REGIONS=$(aws ec2 describe-regions --query "Regions[].RegionName" --output text)
DELEGATED_ADMIN_ACCOUNT="123456789012"  # Compte Security dédié

# 1. Activer AWS Config dans chaque région (prérequis obligatoire)
for region in $REGIONS; do
    echo "[Config] Activation dans $region..."
    aws configservice put-configuration-recorder \
        --configuration-recorder name=default,roleARN=arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):role/aws-service-role/config.amazonaws.com/AWSServiceRoleForConfig,recordingGroup='{allSupported:true,includeGlobalResourceTypes:true}' \
        --region "$region"
    aws configservice start-configuration-recorder \
        --configuration-recorder-name default \
        --region "$region"
done

# 2. Désigner le delegated admin (depuis le management account)
aws organizations register-delegated-administrator \
    --account-id "$DELEGATED_ADMIN_ACCOUNT" \
    --service-principal securityhub.amazonaws.com

# 3. Activer Security Hub dans chaque région
for region in $REGIONS; do
    echo "[SecurityHub] Activation dans $region..."
    aws securityhub enable-security-hub \
        --enable-default-standards \
        --control-finding-generator SECURITY_CONTROL \
        --region "$region" 2>/dev/null || echo "  → Déjà activé dans $region"
done

# 4. Configurer l'agrégation cross-region
echo "[SecurityHub] Configuration agrégation dans $AGGREGATION_REGION..."
aws securityhub create-finding-aggregator \
    --region "$AGGREGATION_REGION" \
    --region-linking-mode ALL_REGIONS

# 5. Auto-enable pour les nouveaux comptes membres
aws securityhub update-organization-configuration \
    --auto-enable \
    --auto-enable-standards DEFAULT \
    --region "$AGGREGATION_REGION"

echo "✅ Security Hub activé dans toutes les régions avec agrégation vers $AGGREGATION_REGION"

🚨
Contexte critique : Les standards de sécurité sont le cœur de Security Hub — ce sont eux qui génèrent les security checks automatisés contre vos ressources AWS. Par défaut, seul FSBP (AWS Foundational Security Best Practices) est activé. Sans activer CIS, PCI ou NIST, vous manquez des centaines de contrôles de sécurité critiques.
Analogie : Un standard est comme un référentiel d'inspection immobilière — chaque contrôle est un point vérifié (isolation, électricité, plomberie). Un finding est le ticket de non-conformité généré quand un point ne passe pas l'inspection.

💡
Pourquoi activer FSBP + CIS v3.0 au minimum ? FSBP couvre les best practices spécifiques AWS (IAM, S3, EC2, RDS, Lambda, etc.) tandis que CIS couvre un référentiel reconnu internationalement. Les contrôles CIS sont complémentaires, pas redondants — CIS ajoute des vérifications sur CloudTrail, VPC Flow Logs, MFA root, rotation des clés, etc. Avec le mode SECURITY_CONTROL, les contrôles identiques entre standards sont dédupliqués automatiquement (1 finding au lieu de N).
⚠️
Piège classique — Ne jamais désactiver un contrôle sans documenter : Désactiver un contrôle FSBP (DisableSecurityHubControl) le supprime des évaluations dans TOUTES les régions et tous les comptes si la fonctionnalité « central configuration » est active. Conservez un registre des exceptions avec justification, propriétaire et date de revue.
De plus, les contrôles désactivés ne diminuent pas le score — ils sont simplement exclus du calcul, ce qui peut faussement gonfler votre score global.

Configuration — AWS CLI

Bash — Activation des standards & gestion des contrôles
#!/bin/bash
# === S2 — Activation standards Security Hub ===

REGION="eu-west-1"

# 1. Lister les standards disponibles
aws securityhub describe-standards \
    --region "$REGION" \
    --query "Standards[*].{Name:Name,Arn:StandardsArn,Status:StandardsStatus}" \
    --output table

# 2. Activer CIS AWS Foundations Benchmark v7.0.0
aws securityhub batch-enable-standards \
    --region "$REGION" \
    --standards-subscription-requests '[
        {"StandardsArn": "arn:aws:securityhub:'"$REGION"'::standards/cis-aws-foundations-benchmark/v/3.0.0"}
    ]'

# 3. Activer NIST 800-53 Rev. 5 (si applicable)
aws securityhub batch-enable-standards \
    --region "$REGION" \
    --standards-subscription-requests '[
        {"StandardsArn": "arn:aws:securityhub:'"$REGION"'::standards/nist-800-53/v/5.0.0"}
    ]'

# 4. Désactiver un contrôle non pertinent (avec justification documentée)
# Exemple : EC2.8 — IMDSv2 déjà forcé par SCP, le contrôle est redondant
aws securityhub update-standards-control \
    --standards-control-arn "arn:aws:securityhub:$REGION:123456789012:control/aws-foundational-security-best-practices/v/1.0.0/EC2.8" \
    --control-status DISABLED \
    --disabled-reason "IMDSv2 enforced via SCP org-wide — control redundant. Review: 2026-07-01"

# 5. Personnaliser les paramètres d'un contrôle (ex: max age des clés)
aws securityhub update-security-control \
    --security-control-id IAM.3 \
    --parameters '{"maxCredentialUsageAge": {"ValueType": "CUSTOM", "Value": {"Integer": 90}}}' \
    --last-update-reason "Policy interne : rotation clés à 90 jours"

# 6. Désactiver le standard legacy CIS v1.2 si activé
# D'abord récupérer l'ARN de souscription
CIS12_SUB=$(aws securityhub get-enabled-standards \
    --region "$REGION" \
    --query "StandardsSubscriptions[?starts_with(StandardsArn, 'arn:aws:securityhub:::ruleset/cis')].StandardsSubscriptionArn" \
    --output text)
if [ -n "$CIS12_SUB" ]; then
    aws securityhub batch-disable-standards \
        --standards-subscription-arns "$CIS12_SUB" \
        --region "$REGION"
    echo "✅ CIS v1.2 legacy désactivé (remplacé par v3.0)"
fi

Bash — Audit des standards & scores
#!/bin/bash
# === S2 — Vérification standards Security Hub ===

REGION="eu-west-1"

echo "=== 1. Standards activés ==="
aws securityhub get-enabled-standards \
    --region "$REGION" \
    --query "StandardsSubscriptions[*].{Arn:StandardsArn,Status:StandardsStatus}" \
    --output table

echo "=== 2. Score de sécurité par standard ==="
for sub in $(aws securityhub get-enabled-standards --region "$REGION" --query "StandardsSubscriptions[*].StandardsSubscriptionArn" --output text); do
    echo "--- $sub ---"
    aws securityhub get-security-control-definitions \
        --standards-arn "$sub" \
        --query "length(SecurityControlDefinitions)" \
        --output text --region "$REGION" 2>/dev/null && echo " contrôles"
done

echo "=== 3. Contrôles en échec (FAILED) — top 20 ==="
aws securityhub get-findings \
    --region "$REGION" \
    --filters '{
        "ComplianceStatus": [{"Value": "FAILED", "Comparison": "EQUALS"}],
        "WorkflowStatus": [{"Value": "NEW", "Comparison": "EQUALS"}],
        "RecordState": [{"Value": "ACTIVE", "Comparison": "EQUALS"}]
    }' \
    --sort-criteria '{"Field": "SeverityLabel", "SortOrder": "desc"}' \
    --max-items 20 \
    --query "Findings[*].{Control:Compliance.SecurityControlId,Severity:Severity.Label,Title:Title}" \
    --output table

echo "=== 4. Contrôles désactivés ==="
aws securityhub list-security-control-definitions \
    --region "$REGION" \
    --query "SecurityControlDefinitions[?CurrentStatus=='DISABLED'].{Id:SecurityControlId,Title:Title}" \
    --output table 2>/dev/null || echo "  ⚠️ Commande non supportée — vérifier manuellement dans la console"

echo "=== 5. Résumé compliance ==="
aws securityhub get-findings \
    --region "$REGION" \
    --filters '{"RecordState": [{"Value": "ACTIVE", "Comparison": "EQUALS"}]}' \
    --query "length(Findings)" \
    --output text
echo " findings actifs au total"

Pourquoi structurer le workflow des findings ? Le format ASFF (AWS Security Finding Format) est commun à tous les services intégrés. Un finding passe par les états : NEW → NOTIFIED → RESOLVED / SUPPRESSED. Sans workflow défini : les findings restent indéfiniment en NEW, les équipes ignorent les alertes, et lors d'un audit, il est impossible de prouver que les problèmes ont été traités. Avec le workflow : chaque finding a un propriétaire, un SLA, et une trace d'audit.
⚠️
Piège classique — Archiver sans traiter : L'action SUPPRESSED ou RESOLVED peut être utilisée pour « faire disparaître » les findings sans véritable remediation. Un auditeur vérifiera les findings archivés — un finding CRITICAL passé en SUPPRESSED sans Note de justification est un red flag. Ajoutez systématiquement un Note.Text avec la justification et la référence au ticket de remediation.

for sev in CRITICAL HIGH MEDIUM LOW INFORMATIONAL; do

Dans un environnement multi-comptes avec tous les standards actifs, les coûts peuvent atteindre plusieurs milliers de $/mois. L'optimisation consiste à réduire le volume de findings sans réduire la couverture de sécurité.
Analogie : Chaque finding est un ticket de non-conformité. Payer pour des tickets sur des contrôles non pertinents, c'est comme payer un inspecteur pour vérifier des pièces qui n'existent pas dans votre bâtiment.

Script All-in-One — Hardening AWS Security Hub
Script Bash/AWS CLI exécutable pour appliquer toutes les best practices
Bash — Script All-in-One Hardening Security Hub
#!/bin/bash
# ============================================================
# AWS Security Hub — Hardening All-in-One Script
# Version: 1.0 | Avril 2026
# Prérequis: AWS CLI v2, jq, droits admin Security Hub
# Usage: ./securityhub-hardening.sh [REGION] [SECURITY_ACCOUNT_ID]
# ============================================================

set -euo pipefail

REGION="${1:-eu-west-1}"
SECURITY_ACCOUNT_ID="${2:-}"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
SNS_TOPIC="arn:aws:sns:$REGION:$ACCOUNT_ID:security-alerts"

echo "╔══════════════════════════════════════════════════╗"
echo "║  AWS Security Hub — Hardening Script             ║"
echo "║  Region: $REGION | Account: $ACCOUNT_ID      ║"
echo "╚══════════════════════════════════════════════════╝"

# === S1 — Activation ===
echo "[S1] Activation de Security Hub..."
aws securityhub enable-security-hub \
    --region "$REGION" \
    --enable-default-standards \
    --control-finding-generator SECURITY_CONTROL 2>/dev/null || echo "  Déjà activé"

# === S2 — Standards ===
echo "[S2] Activation des standards..."
aws securityhub batch-enable-standards \
    --region "$REGION" \
    --standards-subscription-requests \
        '[{"StandardsArn":"arn:aws:securityhub:::standards/aws-foundational-security-best-practices/v/1.0.0"},
          {"StandardsArn":"arn:aws:securityhub:::standards/cis-aws-foundations-benchmark/v/3.0.0"}]' 2>/dev/null

# === S3 — Cross-region aggregation ===
echo "[S3] Configuration cross-region aggregation..."
aws securityhub create-finding-aggregator \
    --region "$REGION" \
    --region-linking-mode ALL_REGIONS 2>/dev/null || echo "  Déjà configuré"

# === S5 — Automation rules ===
echo "[S5] Création automation rules..."
aws securityhub create-automation-rule \
    --region "$REGION" \
    --rule-name "AutoSuppressInformational" \
    --rule-order 1 \
    --description "Auto-suppress INFORMATIONAL findings" \
    --criteria '{"SeverityLabel":[{"Comparison":"EQUALS","Value":"INFORMATIONAL"}],"RecordState":[{"Comparison":"EQUALS","Value":"ACTIVE"}]}' \
    --actions '[{"Type":"FINDING_FIELDS_UPDATE","FindingFieldsUpdate":{"Workflow":{"Status":"SUPPRESSED"},"Note":{"Text":"Auto-suppressed","UpdatedBy":"hardening-script"}}}]' 2>/dev/null || echo "  Rule déjà existante"

# === S6 — Delegated admin (si SECURITY_ACCOUNT_ID fourni) ===
if [ -n "$SECURITY_ACCOUNT_ID" ]; then
    echo "[S6] Désignation delegated admin: $SECURITY_ACCOUNT_ID"
    aws securityhub enable-organization-admin-account \
        --admin-account-id "$SECURITY_ACCOUNT_ID" \
        --region "$REGION" 2>/dev/null || echo "  Déjà désigné"
fi

# === S9 — Alerting ===
echo "[S9] Configuration alerting..."
aws sns create-topic --name security-alerts --region "$REGION" 2>/dev/null
aws events put-rule \
    --name "securityhub-critical-alert" \
    --region "$REGION" \
    --event-pattern '{"source":["aws.securityhub"],"detail-type":["Security Hub Findings - Imported"],"detail":{"findings":{"Severity":{"Label":["CRITICAL"]},"Workflow":{"Status":["NEW"]},"RecordState":["ACTIVE"]}}}' 2>/dev/null
aws events put-targets \
    --rule "securityhub-critical-alert" \
    --region "$REGION" \
    --targets "Id=sns-target,Arn=$SNS_TOPIC" 2>/dev/null

echo ""
echo "╔══════════════════════════════════════════════════╗"
echo "║  ✅ Hardening terminé — Niveau: Hardened (82pts) ║"
echo "╚══════════════════════════════════════════════════╝"
echo "Prochaines étapes :"
echo "  1. Souscrire à SNS: aws sns subscribe --topic-arn $SNS_TOPIC --protocol email --notification-endpoint VOTRE_EMAIL"
echo "  2. Vérifier les scores: aws securityhub get-findings --region $REGION --query 'length(Findings)'"
echo "  3. Consulter ce guide pour les Custom Insights (S7) et le Compliance Reporting (S8)"
