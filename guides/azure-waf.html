💡
Pourquoi chaque paramètre compte :
  • SKU Premium : Front Door Standard n'est pas « sans WAF » au sens strict, mais il reste limité aux custom rules. Le SKU Premium ajoute les managed rule sets, Bot Manager et Private Link, donc c'est le vrai choix par défaut pour une protection applicative globale complète.
  • WAF Policy associée : Standard comme Premium peuvent avoir une WAF policy, mais sans association active, aucune protection WAF ne s'applique. Le mode Detection reste l'étape 1 avant Prevention.
  • Routing LatencyBased : dirige l'utilisateur vers l'origin la plus rapide. Sans routing, Front Door ne fait que du round-robin — un utilisateur à Paris peut être routé vers un backend US East.
  • Health probes HTTPS + /health : sans health probe, Front Door ne détecte pas un backend en panne et continue à lui envoyer du trafic → erreurs 502/503 pour les utilisateurs. Le protocole HTTPS évite les faux positifs (redirects HTTP→HTTPS).
  • Private Link : sans Private Link, l'origin (App Service, Storage) reste accessible publiquement. Un attaquant qui découvre l'IP du backend contourne entièrement Front Door et le WAF. C'est l'équivalent de l'« entrée des artistes » — scelle l'accès direct.
  • Origin response timeout à 16s : le défaut 60s maintient les connexions trop longtemps. En cas de slowloris ou de backend lent, 60s multiplie les connexions actives et le coût. 16s est la recommandation Microsoft pour les APIs.
  • Migration Classic : Azure Front Door Classic est sur trajectoire de retrait, avec retraite au 31/03/2027. Les nouvelles fonctionnalités et la création de nouveaux profils/domaines basculent désormais vers Standard/Premium.
⚠️
Pièges classiques :
  • Front Door Premium sans WAF policy : le SKU Premium inclut la capacité WAF, mais il faut créer et associer une WAF policy manuellement. Sans cette étape, zéro protection.
  • Health probes en HTTP : si votre backend redirige HTTP → HTTPS (301/302), Front Door interprète la redirection comme un échec → le backend est marqué unhealthy → 503 pour tous les utilisateurs.
  • Origin accessible publiquement : même avec Front Door en façade, si l'App Service n'a pas de restriction d'accès (IP filtering ou Private Link), le WAF est contournable.
  • Classic → Standard migration : la migration ne peut pas être annulée (one-way). Tester en staging d'abord. Les WAF policies Classic ne sont pas compatibles 1:1.

Az CLI — Déploiement Front Door Premium complet
# Variables à adapter
$rgName       = "rg-frontdoor-prod"
$location     = "global"
$profileName  = "afd-prod-001"
$endpointName = "ep-webapp-prod"
$originHost   = "webapp-prod.azurewebsites.net"

# 1. Créer le profil Front Door Premium
az afd profile create `
    --resource-group $rgName `
    --profile-name $profileName `
    --sku "Premium_AzureFrontDoor"

# 2. Créer l'endpoint
az afd endpoint create `
    --resource-group $rgName `
    --profile-name $profileName `
    --endpoint-name $endpointName `
    --enabled-state Enabled

# 3. Créer l'origin group avec health probe HTTPS
az afd origin-group create `
    --resource-group $rgName `
    --profile-name $profileName `
    --origin-group-name "og-webapp" `
    --probe-request-type HEAD `
    --probe-protocol Https `
    --probe-interval-in-seconds 30 `
    --probe-path "/health" `
    --sample-size 4 `
    --successful-samples-required 3

# 4. Ajouter l'origin avec Private Link (Premium)
az afd origin create `
    --resource-group $rgName `
    --profile-name $profileName `
    --origin-group-name "og-webapp" `
    --origin-name "origin-webapp" `
    --host-name $originHost `
    --origin-host-header $originHost `
    --http-port 80 `
    --https-port 443 `
    --priority 1 `
    --weight 1000 `
    --enable-private-link true `
    --private-link-location "westeurope" `
    --private-link-resource "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Web/sites/webapp-prod" `
    --private-link-request-message "Front Door Private Link"

# 5. Créer la route (HTTPS only, LatencyBased)
az afd route create `
    --resource-group $rgName `
    --profile-name $profileName `
    --endpoint-name $endpointName `
    --route-name "route-default" `
    --origin-group "og-webapp" `
    --supported-protocols Https `
    --https-redirect Enabled `
    --patterns-to-match "/*" `
    --forwarding-protocol HttpsOnly

Scénario concret : Votre application e-commerce est en ligne derrière Front Door Premium, mais aucune WAF policy n'est associée. 💡
Pourquoi chaque paramètre compte :
  • Mode Detection d'abord : activer directement Prevention sans analyse préalable des logs = faux positifs garantis. Les formulaires légitimes contenant des balises HTML (CMS, éditeurs rich text) seront bloqués. Detection logge sans bloquer → vous identifiez les faux positifs → vous configurez les exclusions → vous passez en Prevention.
  • DRS 2.2 vs anciennes versions : DRS 2.2 apporte une meilleure couverture que DRS 2.1, davantage de signatures Microsoft Threat Intelligence et moins de faux positifs. Rester sur une ancienne version laisse des protections plus récentes de côté.
  • Paranoia level plutôt que seuil custom : avec DRS 2.x, le score d'anomalie de référence reste à 5. Le bon levier de durcissement consiste à activer progressivement des règles PL2 en mode Log puis Block, pas à inventer un seuil à 3 dans le design.
  • Exclusions per-rule : une exclusion globale (RequestBody contains "script") désactive le contrôle XSS pour TOUTES les requêtes. Une exclusion per-rule (rule 942100 seulement, sur le paramètre richtext) est chirurgicale et maintient la protection partout ailleurs.
  • Request body inspection : sans inspection du body, les attaques POST (SQLi dans un formulaire de login, XSS dans un commentaire) passent sans contrôle. La limite de 128 KB couvre 99% des cas — ajuster uniquement si votre API accepte des payloads > 128 KB.
⚠️
Pièges classiques :
  • Passer en Prevention sans tuning : la cause #1 de rollback WAF. Un CMS WordPress avec TinyMCE déclenche systématiquement les règles XSS (942100-942440). Toujours analyser 24-48h de logs Detection d'abord.
  • Désactiver des rule groups entiers : plutôt que désactiver tout le group SQLi (942xxx) pour un faux positif, créer une exclusion ciblée sur le paramètre concerné.
  • WAF policy non associée : créer une WAF policy ne suffit pas — il faut la lier via une security-policy. Vérifier avec az afd security-policy list.
  • Multiples WAF policies : une seule WAF policy par endpoint. Si vous avez plusieurs domains sur un endpoint, les règles s'appliquent à tous. Utiliser des conditions match dans les custom rules pour différencier.

2.2 — Configuration

Chemin portail : portal.azure.com > Web Application Firewall policies (WAF) > + Create > Scope = Front Door Premium

Az CLI — Créer WAF Policy + Managed Rules + Associer
# Variables
$wafPolicyName = "wafprod001"
$rgName        = "rg-frontdoor-prod"
$profileName   = "afd-prod-001"

# 1. Créer la WAF policy en mode Detection
az afd waf-policy create `
    --resource-group $rgName `
    --name $wafPolicyName `
    --sku Premium_AzureFrontDoor `
    --mode Detection `
    --redirect-url "https://www.contoso.com/blocked.html"

# 2. Ajouter le managed rule set DRS 2.2
az afd waf-policy managed-rule-set add `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --type DefaultRuleSet `
    --version "2.2" `
    --action Block

# 3. Ajouter le Bot Manager rule set
az afd waf-policy managed-rule-set add `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --type Microsoft_BotManagerRuleSet `
    --version "1.1" `
    --action Block

# 4. Ajouter une exclusion ciblée (exemple : param "richtext" pour éditeur CMS)
az afd waf-policy managed-rule-set exclusion add `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --type DefaultRuleSet `
    --rule-group-id "SQLI" `
    --match-variable RequestBodyPostArgNames `
    --selector-match-operator Equals `
    --selector "richtext"

# 5. Associer la WAF policy au profil Front Door (security-policy)
az afd security-policy create `
    --resource-group $rgName `
    --profile-name $profileName `
    --security-policy-name "secpol-waf" `
    --domains "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Cdn/profiles/{profile}/afdEndpoints/{endpoint}" `
    --waf-policy "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Network/FrontDoorWebApplicationFirewallPolicies/{wafPolicyName}"

🚨
Scénario concret : Votre API d'authentification (/api/login) subit une attaque brute-force — un bot essaie 10 000 combinaisons username/password par minute depuis la même IP. Les managed rules OWASP ne bloquent pas cette attaque (les requêtes sont individuellement valides). Sans rate limiting, l'attaquant finit par trouver un mot de passe faible. Analogie : Les custom rules sont des règles de la maison spécifiques à votre établissement. Le rate limiting est la file d'attente — si un visiteur revient trop souvent en trop peu de temps, c'est suspect et il est temporairement refusé.

  • Priorité des rules : les custom rules s'exécutent dans l'ordre de priorité (plus bas = premier). Si votre allow-list IP a une priorité 50 et votre geo-block une priorité 10, vos partenaires étrangers seront bloqués avant d'être autorisés.

⚠️
Pièges classiques :
  • Rate limit trop agressif sur les assets statiques : un seul chargement de page peut générer 50-100 requêtes (CSS, JS, images). Un rate limit global à 100/min bloque les utilisateurs légitimes. Exclure les patterns /static/*, /assets/* du rate limiting ou ajuster le seuil.

  • Confondre rate limiting Front Door et App Gateway : Front Door supporte le rate limiting natif (req/min par IP/socket). App Gateway WAF ne supporte pas le rate limiting natif — il faut utiliser des custom rules avec match conditions (moins flexible).
  • Oublier les headers X-Forwarded-For : si votre origin est derrière un autre reverse proxy (Nginx, etc.), le rate limiting par SocketAddr rate-limite le proxy, pas le client. Utiliser RemoteAddr (= l'IP vue par Front Door).

⚠️

  • Cache poisoning avec des query strings : si le cache inclut les query strings, un attaquant peut polluer le cache avec /page?evil=payload. Utiliser "Include specified parameters" (whitelist) plutôt que "Include all".
  • Compression déjà faite par l'origin : si votre origin compresse déjà (Nginx gzip), Front Door ne re-compresse pas. Vérifier que la compression est faite par Front Door OU l'origin, pas les deux (doublon inutile).

Cost Management Azure
Modèle de facturation Front Door / WAF / DDoS Protection, estimation et optimisation multi-apps
🚨
Scénario concret : Votre CFO reçoit une facture Azure réseau nettement supérieure au budget prévu et demande des explications. L'équipe sécurité a déployé Front Door Premium, plusieurs policies WAF et un tier DDoS avancé pour une application dont le trafic reste pourtant modéré. C'est comme louer un convoi blindé pour transporter une baguette de pain — la protection est disproportionnée par rapport à la charge et au risque.

Analogie : Le coût WAF est un contrat d'assurance. Trop peu = vulnérable (pas d'assurance incendie). Trop = surpayer pour un risque inexistant (assurance tremblement de terre en Bretagne). L'objectif est de dimensionner la protection au risque réel et de mutualiser les coûts fixes quand c'est possible.

💡
Stratégies d'optimisation :
  • Mutualiser quand la politique de sécurité le permet : une même WAF policy ou un même tier DDoS Network Protection peuvent couvrir plusieurs surfaces homogènes. Le gain vient de la mutualisation, pas d'un tarif magique figé.
  • Standard vs Premium : Premium doit être choisi pour un besoin clair de managed rules complètes, Bot Manager ou Private Link. Sans ces besoins, Standard reste le point de départ rationnel.
  • DDoS IP vs Network Protection : IP Protection convient à quelques IPs Standard isolées. Network Protection devient pertinent quand il faut mutualiser, obtenir DRR, cost protection ou couvrir un parc plus large.
  • Cache hit ratio = réduction de transfert : plus le cache hit ratio est haut, plus le trafic sortant depuis l'origin diminue et plus la facture réseau se détend.
  • Réduire les logs inutiles : FrontDoorHealthProbeLog génère du volume (sonde toutes les 30 sec). Si vous n'analysez pas les health probes, désactiver ce log pour réduire les coûts Log Analytics.
⚠️
Pièges coûts classiques :
  • Network Protection pour une seule app non critique : si vous n'avez que quelques IPs exposées et pas d'exigence DRR/cost protection, IP Protection ou la protection d'infrastructure peuvent suffire.
  • Premium "par défaut" : déployer Premium sans utiliser Bot Manager, Private Link ni managed WAF avancé ajoute un surcoût non négligeable sans valeur claire. Commencer par le besoin, pas par le tier le plus haut.
  • Logs non surveillés mais facturés : Log Analytics facture l'ingestion. Un volume WAF soutenu peut vite devenir matériel. Si personne n'exploite ces logs, c'est du gaspillage.
  • Oublier le coût de complexité des custom rules : multiplier les règles augmente la maintenance et peut aussi augmenter la facture selon les mètres actifs. Consolider les règles reste le bon réflexe.

Script All-in-One — Déploiement WAF & Front Door
Script exécutable pour déployer et configurer l'ensemble des best practices
⚠️
Prérequis : Az CLI ≥ 2.61, extension front-door installée, droits Contributor sur le resource group. Ce script déploie Front Door Standard + WAF Prevention + custom rules + HTTPS redirect + monitoring. Adapter les variables avant exécution. Tester d'abord en Detection mode.
PowerShell / Az CLI — Déploiement complet WAF & Front Door
#!/usr/bin/env pwsh
# ═══════════════════════════════════════════════════════════════
# SCRIPT ALL-IN-ONE — Azure WAF & Front Door (Standard)
# Adapté du guide WAF & Front Door — Protection Applicative
# ═══════════════════════════════════════════════════════════════

# ── VARIABLES ──────────────────────────────────────────────────
$rgName         = "rg-waf-prod"
$location       = "westeurope"
$profileName    = "afd-prod"
$endpointName   = "afd-endpoint-prod"
$wafPolicyName  = "wafpol-prod"
$originHost     = "app-prod.azurewebsites.net"
$customDomain   = "www.contoso.com"
$logWorkspace   = "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.OperationalInsights/workspaces/{ws}"

# ── S1 : FRONT DOOR PROFILE ────────────────────────────────────
Write-Host "[S1] Création du profil Front Door Premium..."
az afd profile create `
    --resource-group $rgName `
    --profile-name $profileName `
    --sku Premium_AzureFrontDoor

az afd endpoint create `
    --resource-group $rgName `
    --profile-name $profileName `
    --endpoint-name $endpointName `
    --enabled-state Enabled

az afd origin-group create `
    --resource-group $rgName `
    --profile-name $profileName `
    --origin-group-name "og-backend" `
    --probe-request-type HEAD `
    --probe-protocol Https `
    --probe-interval-in-seconds 30 `
    --probe-path "/health"

az afd origin create `
    --resource-group $rgName `
    --profile-name $profileName `
    --origin-group-name "og-backend" `
    --origin-name "origin-app" `
    --host-name $originHost `
    --origin-host-header $originHost `
    --http-port 80 `
    --https-port 443 `
    --priority 1 `
    --weight 1000 `
    --enabled-state Enabled

az afd route create `
    --resource-group $rgName `
    --profile-name $profileName `
    --endpoint-name $endpointName `
    --route-name "route-default" `
    --origin-group "og-backend" `
    --supported-protocols Https `
    --https-redirect Enabled `
    --forwarding-protocol HttpsOnly `
    --link-to-default-domain Enabled `
    --enable-caching true

# ── S2 : WAF POLICY (Prevention + DRS 2.2) ────────────────────
Write-Host "[S2] Création de la WAF policy..."
az afd waf-policy create `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --sku Premium_AzureFrontDoor `
    --mode Prevention `
    --redirect-url "https://${customDomain}/blocked"

az afd waf-policy managed-rule-set add `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --type DefaultRuleSet `
    --version 2.2 `
    --action Block

# ── S3 : CUSTOM RULES ──────────────────────────────────────────
Write-Host "[S3] Ajout des custom rules..."
# Rate limiting — /login — 50 req / 1 min
az afd waf-policy custom-rule create `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --rule-name "RateLimitLogin" `
    --priority 100 `
    --rule-type RateLimitRule `
    --rate-limit-threshold 50 `
    --rate-limit-duration-in-minutes 1 `
    --action Block `
    --match-condition "match-variable=RequestUri operator=Contains match-value=/login"

# Geo-filter — Autoriser uniquement Europe
az afd waf-policy custom-rule create `
    --resource-group $rgName `
    --policy-name $wafPolicyName `
    --rule-name "GeoFilterEU" `
    --priority 200 `
    --rule-type MatchRule `
    --action Block `
    --match-condition "match-variable=RemoteAddr operator=GeoMatch negate=true match-value=FR DE NL BE LU CH AT IT ES PT"

# ── S6 : HTTPS REDIRECT (via route, déjà configuré ci-dessus) ─
Write-Host "[S6] HTTPS redirect activé via la route..."

# ── S8 : MONITORING ────────────────────────────────────────────
Write-Host "[S8] Configuration des Diagnostic Settings..."
$profileId = $(az afd profile show `
    --resource-group $rgName `
    --profile-name $profileName `
    --query id -o tsv)

az monitor diagnostic-settings create `
    --name "diag-afd-all" `
    --resource $profileId `
    --workspace $logWorkspace `
    --logs '[{"category":"FrontDoorAccessLog","enabled":true},{"category":"FrontDoorWebApplicationFirewallLog","enabled":true}]' `
    --metrics '[{"category":"AllMetrics","enabled":true}]'

# ── ASSOCIATION WAF → Endpoint ──────────────────────────────────
Write-Host "[WAF] Association de la policy au profil..."
az afd security-policy create `
    --resource-group $rgName `
    --profile-name $profileName `
    --security-policy-name "secpol-waf" `
    --domains $endpointName `
    --waf-policy $wafPolicyName

Write-Host "✅ Déploiement terminé. Vérifier avec :"
Write-Host "   az afd profile show -g $rgName -n $profileName"
Write-Host "   az afd waf-policy show -g $rgName -n $wafPolicyName"
