Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel d'audit et de hardening MDCA
📋
Ce document est un référentiel générique couvrant toutes les configurations de Microsoft Defender for Cloud Apps (MDCA, anciennement MCAS). Il couvre le shadow IT, la gouvernance des apps OAuth, le contrôle de session, la détection d'anomalies et le SSPM. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'un tenant.
⚠️
Spécificité MDCA : contrairement à Exchange Online ou Entra ID, MDCA n'a pas de module PowerShell officiel. Les commandes de ce guide utilisent l'API REST MDCA (via Invoke-RestMethod) et les étapes portail en parallèle. Le portail principal est security.microsoft.com/cloudapps.

Cloud Discovery — Shadow IT MDCA
Découverte et gouvernance des applications cloud utilisées dans l'organisation
🚨
Vecteur de risque : sans Cloud Discovery, vous ne savez pas que vos utilisateurs utilisent 300+ applications cloud non sanctionées. Un scénario typique : on découvre que 340 apps cloud sont utilisées, dont 45 avec un score de risque < 5/10 (stockage de fichiers sensibles sur des services sans chiffrement, sans RGPD, sans MFA).
ℹ️
Cloud Discovery fonctionne en 2 modes :
• Snapshot report : upload manuel d'un fichier log firewall/proxy → analyse ponctuelle (idéal pour un premier audit)
• Continuous report : log collector déployé sur Docker/VM qui envoie les logs en continu → tableau de bord permanent
Le catalogue MDCA référence 25 000+ applications avec un score de risque de 1 à 10 basé sur 90+ facteurs (chiffrement, conformité, fiabilité).

⚠️
Piège : le snapshot report donne une photo à un instant T mais ne détecte pas les nouvelles apps utilisées après l'upload. Pour une protection continue, le log collector Docker est indispensable. Sans lui, le Cloud Discovery est un audit ponctuel, pas une protection.
💡
Pourquoi chaque paramètre Cloud Discovery compte :
  • Continuous vs Snapshot : un snapshot ne détecte que le passé. Le mode continu détecte en temps réel l'apparition d'une app de transfert de fichiers utilisée pour exfiltrer des données.
  • Sanctioning : taguer une app comme « Unsanctioned » permet de la bloquer via l'intégration Defender for Endpoint (MDE génère un custom indicator qui bloque le domaine).
  • Score de risque : le score 1-10 est calculé sur 90+ facteurs (certifications SOC 2/ISO 27001, chiffrement at-rest/in-transit, localisation des données, support MFA, conformité RGPD). Un score < 5 indique un risque élevé.
  • App Discovery Policy : sans policy, les nouvelles apps apparaissent dans le dashboard mais personne n'est notifié. Avec une policy, le SOC reçoit une alerte dès qu'une app risquée est détectée.

1.4 — Configuration API — Sanctionner des apps

PowerShell — API REST MDCA — Sanctionner / Bloquer des apps
# --- PRÉ-REQUIS : Générer un API Token dans Settings > Security extensions > API tokens ---
# ⚠ API MDCA standalone deprecated — migrer vers Microsoft Graph Security API (graph.microsoft.com/beta/security/...)
$mdcaUrl   = "https://<tenant>.portal.cloudappsecurity.com"
$apiToken  = "<VOTRE_API_TOKEN>"
$headers   = @{ "Authorization" = "Token $apiToken"; "Content-Type" = "application/json" }

# Lister les apps découvertes (top 50 par trafic)
$body = @{
    filters = @{}
    sortField = "traffic"
    sortDirection = "desc"
    limit = 50
} | ConvertTo-Json

$apps = Invoke-RestMethod -Uri "$mdcaUrl/api/v1/discovery/discovered_apps/" `
    -Method POST -Headers $headers -Body $body
$apps.data | Select-Object appId, appName, risk, category, trafficTotalBytes

# Taguer une app comme UNSANCTIONED (bloquée)
$appIdToBlock = 12345  # Remplacer par l'appId réelle
Invoke-RestMethod -Uri "$mdcaUrl/api/v1/discovery/discovered_apps/tag/" `
    -Method POST -Headers $headers `
    -Body (@{ appId = $appIdToBlock; tagType = "unsanctioned" } | ConvertTo-Json)

# Taguer une app comme SANCTIONED (autorisée)
$appIdToAllow = 11161  # Ex: Microsoft 365
Invoke-RestMethod -Uri "$mdcaUrl/api/v1/discovery/discovered_apps/tag/" `
    -Method POST -Headers $headers `
    -Body (@{ appId = $appIdToAllow; tagType = "sanctioned" } | ConvertTo-Json)

Gouvernance OAuth — Contrôle des consentements MDCA
Audit, gouvernance et révocation des applications OAuth tierces connectées à votre tenant
🚨
Vecteur d'attaque n°1 : Consent Phishing — un attaquant crée une app malveillante « Document Viewer » qui demande Mail.Read + Files.ReadWrite.All. L'utilisateur clique « Accepter » → l'attaquant a un accès persistant aux mails et fichiers même si le mot de passe est changé (le token OAuth reste valide). Sans gouvernance OAuth dans MDCA, ces apps passent sous le radar.
ℹ️
Interaction avec Entra ID : la restriction du consentement utilisateur (guide Entra ID, S2) est la première ligne de défense. MDCA ajoute une deuxième couche : visibilité complète sur les apps déjà consenties, scoring de risque, et capacité de révocation en masse. Les deux sont complémentaires et non substituables.

⚠️

💡
Pourquoi chaque paramètre OAuth compte :
  • High permissions policy : une app avec Mail.ReadWrite peut lire, modifier et supprimer tous les mails d'un utilisateur. C'est le vecteur principal du BEC (Business Email Compromise) via OAuth.
  • Community use : une app utilisée par < 3 tenants est suspecte — c'est souvent une app créée spécifiquement pour une attaque de consent phishing.
  • Tags Approved/Banned : permettent de créer un catalogue d'apps autorisées. Les policies peuvent ensuite cibler « toute app non-Approved avec permissions élevées ».
  • Auto-revoke : sans automatisation, le temps entre la détection et la révocation peut être de plusieurs jours — pendant lesquels l'attaquant exfiltre des données.

2.2 — Configuration API — Audit et gouvernance OAuth

PowerShell — API REST MDCA — Audit des apps OAuth
# --- Lister toutes les apps OAuth avec leurs permissions ---
$body = @{
    filters = @{}
    sortField = "severity"
    sortDirection = "desc"
    limit = 200
} | ConvertTo-Json

$oauthApps = Invoke-RestMethod -Uri "$mdcaUrl/api/v1/app_permissions/" `
    -Method POST -Headers $headers -Body $body

# Afficher les apps avec permissions élevées
$oauthApps.data | Where-Object { $_.severity -ge 2 } |
    Select-Object appId, appName, severity, userCount,
        @{N='Permissions'; E={ ($_.permissions | Select-Object -First 5).permissionName -join ', ' }} |
    Format-Table -AutoSize

# --- Bannir une app OAuth (révoquer les tokens) ---
$appIdToBan = "<APP_OBJECT_ID>"
Invoke-RestMethod -Uri "$mdcaUrl/api/v1/app_permissions/$appIdToBan/ban/" `
    -Method POST -Headers $headers

# --- Approuver une app OAuth légitime ---
$appIdToApprove = "<APP_OBJECT_ID>"
Invoke-RestMethod -Uri "$mdcaUrl/api/v1/app_permissions/$appIdToApprove/approve/" `
    -Method POST -Headers $headers

# --- Révocation en masse des apps à haut risque non-approuvées ---
$riskyApps = $oauthApps.data | Where-Object {
    $_.severity -ge 2 -and $_.tag -ne "approved"
}
foreach ($app in $riskyApps) {
    Write-Host "Banning: $($app.appName) (severity: $($app.severity), users: $($app.userCount))"
    Invoke-RestMethod -Uri "$mdcaUrl/api/v1/app_permissions/$($app.appId)/ban/" `
        -Method POST -Headers $headers
}

Politiques de Session — Conditional Access App Control MDCA Entra P1
Contrôle en temps réel des sessions utilisateurs dans les applications cloud
🚨
Scénario critique : un employé se connecte à SharePoint Online depuis un appareil personnel non géré (BYOD). Sans policy de session, il peut télécharger l'intégralité des documents confidentiels sur son poste. Avec le Conditional Access App Control, vous pouvez autoriser la lecture en ligne mais bloquer le téléchargement, le copier-coller, et l'impression.
ℹ️
Pré-requis Entra ID : les session policies reposent sur le Conditional Access d'Entra ID. Il faut créer une CA Policy qui redirige le trafic vers MDCA (Use Conditional Access App Control). Le flux est : Utilisateur → Entra CA → Reverse Proxy MDCA → App — l'URL devient *.mcas.ms (domaine d'origine, anciennement MCAS).

⚠️
Piège proxy : l'activation du Conditional Access App Control redirige le trafic via le reverse proxy MDCA (*.mcas.ms — domaine historique, anciennement MCAS). Certaines apps ne supportent pas cette interception (problèmes CORS, apps thick-client, certificate pinning). Tester en mode Monitor first avant de bloquer, et commencer par un groupe pilote.
💡
Pourquoi chaque paramètre Session Policy compte :
  • Block download (unmanaged) : sans cette policy, un BYOD peut télécharger toute la base documentaire. C'est le risque DLP n°1 sur les devices non gérés.
  • Real-time monitoring : détecte les comportements anormaux en session (téléchargement massif, accès inhabituel) même sans bloquer.
  • Protect download : alternative au block — on chiffre le fichier téléchargé avec un label Purview. L'utilisateur peut télécharger mais le fichier est protégé hors du tenant.
  • CA Policy : c'est le « switch » qui active le proxy. Sans CA Policy ciblant MDCA, les session policies ne s'appliquent jamais.

3.4 — Access Policies vs Session Policies

ℹ️

  • Access Policy : décision binaire avant l'accès à l'app. Bloquer ou autoriser selon : appareil (géré/non géré), localisation (pays), groupe utilisateur, client app (navigateur/desktop). Exemple : « Bloquer Salesforce depuis un pays hors UE ».

  • Unusual admin activity : détecte les actions administratives anormales (création de règles de transport, modification de permissions) — signe de privilege escalation ou de compte admin compromis.

Piège : les alertes MDCA non traitées s'accumulent rapidement. Après quelques semaines, l'équipe cesse de les regarder (« alert fatigue »). Règle d'or : chaque alerte doit être soit investiguée (True Positive), soit supprimée avec une règle (False Positive). Ne jamais laisser des alertes en statut « New » indéfiniment.
💡
Pourquoi chaque paramètre d'alerte compte :
  • Email notification : sans notification, les alertes restent dans la console que personne ne consulte. Le délai moyen de détection d'une brèche est de 204 jours (IBM 2023) — les notifications réduisent ce délai à quelques heures.
  • Sévérité alignée : si tout est « Medium », le SOC ne sait pas prioriser. Aligner les sévérités (High = action immédiate, Medium = 24h, Low = batch hebdo) réduit l'alert fatigue de 60 % (Gartner).
  • Triage structuré : sans processus, les alertes s'accumulent jusqu'à être ignorées. Un workflow triage → investigate → contain → resolve garantit que chaque alerte a un propriétaire et un SLA.
  • Suppression des faux positifs : chaque FP non traité génère du bruit à chaque occurrence. 10 FP/jour = 3 650/an de bruit. Les règles de suppression documentées transforment le bruit en signal actionnable.
  • Rétention Activity Log : les 180 jours MDCA ne suffisent pas pour la conformité réglementaire (RGPD Art. 5, PCI-DSS). L'export vers Sentinel ou un Data Lake garantit la rétention longue et la corrélation cross-source.
