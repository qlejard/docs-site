Mode d'emploi de ce guide
Comment utiliser ce document pour piloter un déploiement Microsoft Security Copilot.
Microsoft Security Copilot se comporte comme un assistant SOC orienté workflows avec une expérience de plus en plus centrée sur les agents. En 2026, deux parcours coexistent : les tenants Microsoft 365 E5 / E7 éligibles peuvent être auto-provisionnés par Microsoft ; les autres clients doivent passer par un onboarding manuel avec capacité SCU.

Stratégie de déploiement recommandée

Quick Win : confirmer d'abord si le tenant est inclus et auto-provisionné ; sinon provisionner une petite capacité initiale, activer les plugins natifs essentiels et standardiser quelques promptbooks opérationnels.

Solutions écartées

AlternativeRaison d'élimination
Déployer uniquement Microsoft Sentinel sans CopilotNe couvre pas l'accélération conversationnelle SOC ni la standardisation promptbook.

Conserver un fonctionnement 100% manuel L1/L2Temps de triage plus long, variabilité forte des analyses et difficulté à scaler.

Matrice des licences et capacité
Microsoft Security Copilot combine un parcours d'inclusion éligible et un parcours de capacité SCU.
ElémentDétailRègle opérationnelle
Parcours clientM365 E5 / E7 éligible ou onboarding manuel par SCUVérifier d'abord l'éligibilité et l'état d'auto-provisionnement du tenant
CapacitéParcours inclusion : 400 SCUs / mois par 1 000 licences payantes, jusqu'à 10 000 SCUs / mois. Parcours manuel : capacité provisionnée (minimum 1 SCU) avec overage facultatif.Identifier le modèle applicable avant tout dimensionnement
ProvisioningAzure resource type microsoft.securitycopilot/capacitiesAutomatiser si vous suivez le parcours manuel SCU
PricingParcours manuel : capacité provisionnée facturée par heure pleine, overage facturé à l'usage. Parcours inclusion : consommation déduite du quota mensuel inclus.Éviter les montants figés ; documenter la méthode de contrôle des coûts et vérifier le pricing courant

Point de vigilance 2026 : l'inclusion dans Microsoft 365 E5 / E7 ne signifie pas un accès immédiat sur tous les tenants : Microsoft parle d'éligibilité, de notification et d'auto-provisionnement progressif. Les tenants hors inclusion doivent provisionner des SCUs avant usage.

Architecture & SCU Provisioning SCU
Modèles de capacité, workspace, data residency et monitoring.
2 / 15
Contexte : sans cadrage du modèle de capacité, vous mélangez facilement tenant auto-provisionné, capacité provisionnée et overage, avec un risque simultané de saturation et de surcoût.
ParamètreDéfautQuick WinAvancéOptimal
Modèle de capacitéNon identifiéConfirmer inclusion E5/E7 ou parcours manuel SCUProvisioned + overage documentésRight-sizing piloté par usage réel
Capacité initialeFixée sans baselineMinimum 1 SCU si parcours manuelProvisioned capacity + overage plafonnéAjustement sur usage monitoring et charge horaire
Workspace / géoChoix ad hocValider géo de stockage et prompt evaluationRégion conforme residencyRégion primaire + gouvernance de création workspace
Overage policyNon documentéeLimiter ou désactiver l'overageSeuils 75/90/100%Runbook de throttling + budget
Monitoring SCUPas de dashboardUsage monitoring actifExports réguliersCorrélation incident/SCU et tendance mensuelle
Pourquoi chaque paramètre compte : la capacité provisionnée se recharge par heure pleine, l'overage absorbe les pics, et l'inclusion E5/E7 fonctionne avec un quota mensuel partagé. Le bon pilotage consiste donc à distinguer clairement le modèle appliqué avant de parler de sizing.
Piège : conserver des snippets ARM/CLI figés sans revalider l'API courante, ou surprovisionner sans baseline, crée plus de dette que de capacité utile.

Configuration (squelette IaC pour parcours manuel)

Terraform / AzAPI — Capacité Microsoft Security Copilot
terraform {
  required_version = ">= 1.6.0"
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = ">= 3.100.0"
    }
  }
}

provider "azurerm" {
  features {}
}

locals {
  # Revalider le type ARM exact et l'API version dans la doc Microsoft en vigueur.
  security_copilot_capacity_type = "Microsoft.SecurityCopilot/capacities@<API_VERSION_A_VALIDER>"
}

variable "location" { type = string }
variable "resource_group_name" { type = string }
variable "capacity_name" { type = string }
variable "scu_capacity" { type = number }
variable "tags" { type = map(string) default = {} }

resource "azurerm_resource_group" "rg" {
  name     = var.resource_group_name
  location = var.location
  tags     = var.tags
}

resource "azapi_resource" "security_copilot_capacity" {
  type      = local.security_copilot_capacity_type
  name      = var.capacity_name
  parent_id = azurerm_resource_group.rg.id
  location  = var.location

  # Ce bloc ne concerne que le parcours manuel SCU.
  body = jsonencode({
    sku = {
      name = "SecurityCopilot"
    }
    properties = {
      numberOfUnits = var.scu_capacity
    }
    tags = var.tags
  })
}

output "capacity_id" {
  value = azapi_resource.security_copilot_capacity.id
}

Configuration (PowerShell + API, parcours manuel)

PowerShell — Provisioning et scaling SCU
$subscriptionId = "00000000-0000-0000-0000-000000000000"
$resourceGroup  = "rg-copilot-security-prod"
$location       = "westeurope"
$capacityName   = "cpsoc-prod"
$apiVersion     = "<API_VERSION_A_VALIDER>"

# Revalider l'API version et le schéma ARM avant exécution.
# Ne pas utiliser ce bloc pour un tenant E5/E7 auto-provisionné.

az account set --subscription $subscriptionId
az group create -n $resourceGroup -l $location

# Provision 2 SCUs (capacité manuelle)
az resource create --resource-group $resourceGroup --name $capacityName `
  --resource-type "Microsoft.SecurityCopilot/capacities" --api-version $apiVersion `
  --location $location --properties "{\"numberOfUnits\":2}" --sku "{\"name\":\"SecurityCopilot\"}"

# Scale to 3 SCUs
$body = @{ properties = @{ numberOfUnits = 3 } } | ConvertTo-Json -Depth 5
$resourceId = "/subscriptions/$subscriptionId/resourceGroups/$resourceGroup/providers/Microsoft.SecurityCopilot/capacities/$capacityName"
Invoke-AzRestMethod -Method PATCH -Path "$resourceId?api-version=$apiVersion" -Payload $body

PowerShell — Gabarit de calcul interne
$analystHourlyCost = 0
$hoursSaved        = 0
$analysts          = 0
$observedScuCost   = 0
$capacityModel     = "Included|Provisioned|Overage"

$estimatedValue  = $analystHourlyCost * $hoursSaved * $analysts
$estimatedImpact = $estimatedValue - $observedScuCost

[pscustomobject]@{
  CapacityModel   = $capacityModel
  EstimatedValue  = [math]::Round($estimatedValue,2)
  ObservedScuCost = [math]::Round($observedScuCost,2)
  NetImpact       = [math]::Round($estimatedImpact,2)
}
