Mode d’emploi de ce guide
Stratégie de sécurisation Azure App Service de A à Z
📋
Ce guide couvre la sécurisation de la plateforme Azure App Service (Web Apps, Azure Functions, API Apps). Il ne couvre pas la sécurité du code applicatif ni APIM ou WAF (guides dédiés). Utilisable tel quel en présentation client ou pour audit d’une souscription Azure.

1
Authentication & Authorization (Easy Auth) Inclus
Authentification intégrée sans code applicatif via Microsoft Entra ID — protection contre tout accès non authentifié
🚨
Scénario d’attaque : Une Web App exposée sur Internet sans authentification est accessible par n’importe quel acteur malveillant : scraping de données, tentatives de brute-force, exploitation de vulnérabilités 0-day sur le framework applicatif. Sans Easy Auth, chaque requête atteint votre code applicatif — y compris les attaques. Avec Easy Auth, les requêtes non authentifiées sont bloquées au niveau de la plateforme Azure avant d’atteindre votre application.
Analogie : agent de sécurité à l’entrée de l’immeuble — vérifie le badge avant de laisser entrer, quelle que soit la porte.

1.1 — Tableau de configuration

ParamètreDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence

login / tokenRefreshExtensionHours72 h72 h8 h1 h (apps sensibles)Inclus
💡
Pourquoi chaque paramètre compte :
• unauthenticatedClientAction: AllowAnonymous (défaut) = Easy Auth activé mais laisse passer les requêtes non authentifiées — protection illusoire.
• Return401 pour les APIs (vs RedirectToLoginPage pour les Web Apps) évite les redirections OAuth inappropriées dans les flux programmatiques.
• tokenStore: true stocke les tokens OAuth côté serveur, évite de les exposer au client JavaScript (XSS). Avec Easy Auth activé et un provider configuré, le token store est disponible par défaut ; ne le couper que si l’application n’en a pas besoin.
• requireHttps: true est le comportement sûr par défaut avec Easy Auth ; conserver la redirection HTTPS automatique, puis renforcer aussi le paramètre global httpsOnly côté App Service.
• tokenRefreshExtensionHours: 8 réduit la fenêtre d’exposition en cas de vol de token. Source : Microsoft Learn — App Service Authentication.
⚠️
Piège : Easy Auth ne remplace pas une autorisation fine (rôles, groupes). Il authentifie mais n’autorise pas — implémenter les allowedApplications et allowedGroups dans la configuration Entra ID pour restreindre l’accès à des utilisateurs spécifiques.

1.2 — Configuration (portail + CLI)

Portail Azure — Activer Easy Auth Entra ID
# Portail : App Service > Authentication > Add identity provider
# Identity provider : Microsoft
# App registration : Create new (ou existing)
# Unauthenticated requests : HTTP 302 Redirect (Web App) ou HTTP 401 (API)
# Token store : Enable
Azure CLI — Easy Auth Entra ID
$APP="mon-webapp"; $RG="mon-rg"; $TENANT="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"; $CLIENTID="yyyyyyyy-..."

az webapp auth update \
  --name $APP --resource-group $RG \
  --enabled true \
  --action RedirectToLoginPage \
  --aad-allowed-token-audiences "api://$CLIENTID"

# Configurer le provider V2 (authsettingsV2)
az webapp auth microsoft update \
  --name $APP --resource-group $RG \
  --client-id $CLIENTID \
  --tenant-id $TENANT \
  --client-secret-setting-name "MICROSOFT_PROVIDER_AUTHENTICATION_SECRET"

Azure CLI — Vérification Easy Auth
az webapp auth show --name $APP --resource-group $RG \

1.4 — Checklist Easy Auth

Easy Auth activé : az webapp auth show --name $APP --resource-group $RG --query enabled retourne true
unauthenticatedClientAction configuré sur RedirectToLoginPage (Web App) ou Return401 (API)

requireHttps: true dans les paramètres Easy Auth
2
Network Isolation Basic+
VNet Integration, Private Endpoints, Access Restrictions — rendre l’application inatteignable depuis Internet si nécessaire
🚨
Scénario d’attaque : Une application exposée publiquement sur Internet est scannée en permanence par des bots. Sans restrictions réseau, un attaquant peut tenter d’exploiter directement l’application, bypass le WAF via l’IP directe de l’App Service, ou accéder à Kudu (SCM). VNet Integration + Private Endpoint + accès public désactivé = l’app n’est joignable que depuis le réseau privé.
Analogie : accès au réseau interne — l’application cloud se comporte comme si elle était dans votre datacenter, injoignable depuis l’extérieur.

2.1 — Tableau de configuration

Paramètre / FeatureDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence
Access Restrictions (IP allowlist)Tout Internet (0.0.0.0/0)Restreindre IPs connues ou Service Tags (AzureFrontDoor)Allowlist stricte + deny allAccès public désactivé si Private EndpointInclus
SCM (Kudu) Access RestrictionsTout InternetRestreindre aux IPs adminRestreindre + MFADésactiver SCM basic authInclus
VNet Integration (outbound)Non configuré—Activer (Basic+)VNet Intégré + NSG + Route TableBasic+
Private Endpoint (inbound)Non configuré—Configurer si app internePrivate Endpoint + DNS privé + accès public offBasic+ + Private Link
WEBSITE_VNET_ROUTE_ALL0 (désactivé)—1 (tout le trafic sortant passe par VNet)1 + UDR vers firewall AzureBasic+
Service Tags (AzureFrontDoor.Backend)Non configuréAjouter si AFD utiliséAFD + header X-Azure-FDIDAFD + header + WAFInclus
💡
Pourquoi chaque paramètre compte :
• Access Restrictions = 0.0.0.0/0 (défaut) : l’App Service est exposé sur une IP publique Azure connue, scannée en permanence.
• VNet Integration (outbound) permet à l’app d’accéder aux ressources privées (SQL, Service Bus, Storage) sans les exposer sur Internet.
• Private Endpoint (inbound) ajoute une entrée privée à l’app, mais l’accès public peut coexister tant que publicNetworkAccess n’est pas désactivé. Les Access Restrictions ne s’appliquent pas au trafic qui entre par le Private Endpoint ; piloter le subnet PE avec DNS privé et NSG.
• WEBSITE_VNET_ROUTE_ALL=1 force le trafic sortant à passer par le VNet (et potentiellement un Azure Firewall) — évite l’exfiltration de données par Internet. Source : Microsoft Learn — VNet Integration.
⚠️
Piège : VNet Integration (outbound) ≠ Private Endpoint (inbound). VNet Integration permet à l’app d’appeler des ressources privées, mais l’app reste accessible depuis Internet. Le Private Endpoint crée une entrée privée, sans couper automatiquement l’endpoint public : pour une app interne-only, désactiver aussi publicNetworkAccess.

2.2 — Configuration

Azure CLI — Access Restrictions + VNet Integration
# Access restriction : autoriser seulement Azure Front Door
az webapp config access-restriction add \
  --name $APP --resource-group $RG \
  --rule-name "AllowAFD" --action Allow \
  --service-tag "AzureFrontDoor.Backend" --priority 100

# Bloquer tout le reste
az webapp config access-restriction add \
  --name $APP --resource-group $RG \
  --rule-name "DenyAll" --action Deny \
  --ip-address "0.0.0.0/0" --priority 200

# VNet Integration (outbound)
$SUBNET_ID="/subscriptions/.../subnets/app-subnet"
az webapp vnet-integration add \
  --name $APP --resource-group $RG \
  --vnet "mon-vnet" --subnet "app-subnet"

# Forcer tout le trafic sortant dans le VNet
az webapp config appsettings set \
  --name $APP --resource-group $RG \
  --settings WEBSITE_VNET_ROUTE_ALL=1
PowerShell — Restreindre l’accès SCM (Kudu)
Add-AzWebAppAccessRestrictionRule -ResourceGroupName $RG -WebAppName $APP `
  -Name "AllowAdminIP" -Priority 100 -Action Allow `
  -IpAddress "203.0.113.0/24" -SlotName "scm"

Add-AzWebAppAccessRestrictionRule -ResourceGroupName $RG -WebAppName $APP `
  -Name "DenyAll" -Priority 200 -Action Deny `
  -IpAddress "0.0.0.0/0" -SlotName "scm"

2.3 — Vérification

Azure CLI — Vérification réseau
az webapp config access-restriction show --name $APP --resource-group $RG
az webapp vnet-integration list --name $APP --resource-group $RG
az webapp config appsettings list --name $APP --resource-group $RG \
  --query "[?name=='WEBSITE_VNET_ROUTE_ALL']"

2.4 — Checklist Réseau

Access Restrictions configurées : seules les IPs/Service Tags légitimes autorisées
SCM (Kudu) restreint aux IPs admin uniquement
VNet Integration activée si l’app doit accéder à des ressources privées
WEBSITE_VNET_ROUTE_ALL=1 si tout le trafic sortant doit passer par le VNet
Private Endpoint configuré + accès public désactivé si l’application doit être joignable uniquement en privé
3
TLS, HTTPS & Custom Domains Inclus
HTTPS obligatoire, TLS 1.2 minimum, mTLS, désactivation FTP, certificats managés
🚨
Scénario d’attaque : Sans HTTPS forcé, une requête HTTP transite en clair sur Internet. Un attaquant MITM (Man-In-The-Middle) peut intercepter des tokens, cookies de session, données personnelles. Sans bloquer TLS 1.0/1.1, des attaques POODLE, BEAST restent possibles. httpsOnly + minTlsVersion 1.2 = base non-négociable.
Analogie : enveloppe scellée vs carte postale — HTTP = carte postale lisible par tous les facteurs entre expéditeur et destinataire.

3.1 — Tableau de configuration

ParamètreDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence
httpsOnlyfalsetruetruetrue + HSTSInclus
minTlsVersion1.2 (défaut actuel)Conserver 1.21.21.2 + revue périodique des versions TLSInclus
ftpsStateAllAllowedDisabledDisabledDisabledInclus
clientCertEnabled (mTLS)false—true si B2Btrue + clientCertExclusionPathsInclus
Certificat managé App ServiceNon configuréApp Service Managed CertificateKey Vault certKey Vault cert + auto-renewBasic+ (cert gratuit)
HSTS headerNon configuré—Strict-Transport-Security: max-age=31536000+ includeSubDomains; preloadInclus
💡
App Service Managed Certificate (gratuit) : Azure gère le renouvellement automatique des certificats TLS pour les custom domains. Limitations confirmées : pas de wildcard, pas d’export, pas de private DNS, et un seul certificat managé par domaine supporté. Pour un root/apex domain, l’émission et le renouvellement exigent que l’application reste joignable depuis Internet. Pour les certificats wildcard ou les scénarios avancés, utiliser Key Vault — voir Guide Key Vault.
Source : Microsoft Learn — TLS/SSL certificates.
⚠️
Piège : httpsOnly: true redirige HTTP vers HTTPS mais ne désactive pas FTP. Désactiver FTP (ftpsState: Disabled) séparément. FTP transmet les credentials en clair — vecteur d’attaque classique même avec un mot de passe fort.

3.2 — Configuration

Azure CLI — HTTPS + TLS 1.2 + FTP Disabled
az webapp update \
  --name $APP --resource-group $RG \
  --https-only true

az webapp config set \
  --name $APP --resource-group $RG \
  --min-tls-version 1.2 \
  --ftps-state Disabled

# HSTS via App Settings (ajouter un custom header via startup ou web.config)
az webapp config appsettings set \
  --name $APP --resource-group $RG \
  --settings WEBSITE_ADD_SITENAME_BINDINGS_IN_APPHOST_CONFIG=1

3.3 — Vérification

Azure CLI — Vérification TLS
az webapp show --name $APP --resource-group $RG \
  --query "{httpsOnly:httpsOnly}"
az webapp config show --name $APP --resource-group $RG \
  --query "{minTls:minTlsVersion, ftps:ftpsState}"

3.4 — Checklist TLS

httpsOnly: true configuré sur toutes les App Services
minTlsVersion: 1.2 (TLS 1.0 et 1.1 désactivés)
ftpsState: Disabled (FTP complètement désactivé)
Certificat TLS valide sur tous les custom domains (Managed Certificate ou Key Vault)
mTLS activé si l’app reçoit des connexions de systèmes partenaires (B2B)
4
Managed Identity & Key Vault References Inclus
Identité sans secret — l’app s’authentifie auprès des ressources Azure sans stocker de mot de passe
🚨
Scénario d’attaque : Un développeur stocke un connection string SQL Server en dur dans appsettings.json (ou dans les App Settings en clair). Ce secret est compromis via un repo public, une fuite de logs, ou un accès non autorisé au portail. Managed Identity = zéro secret à stocker, à renouveler, ou à gérer.
Analogie : badge entreprise — l’application a une carte d’identité reconnue par Azure. Plus besoin de passe-partout (password).

Syntaxe Key Vault References :
• Par URI complet (version fixée) : @Microsoft.KeyVault(SecretUri=https://monkv.vault.azure.net/secrets/db-password/abc123)
• Par nom (version latest, recommandé) : @Microsoft.KeyVault(VaultName=monkv;SecretName=db-password)
• Statut de résolution visible dans App Service → Configuration → icone « clé » à gauche de la valeur.
Source : Guide Key Vault — Références App Service.
⚠️
Piège : L’App Service doit avoir un droit de lecture sur les secrets du Key Vault. En modèle RBAC, attribuer Key Vault Secrets User ; en modèle Access Policies, attribuer au minimum Get sur les secrets. Sans ce droit, le démarrage de l’app peut échouer avec une erreur Microsoft.KeyVault reference resolution failed.

4.2 — Configuration

Azure CLI — Managed Identity + RBAC Key Vault
# Activer Managed Identity System-Assigned
az webapp identity assign \
  --name $APP --resource-group $RG

# Récupérer le principal ID
$PRINCIPAL_ID=$(az webapp identity show \
  --name $APP --resource-group $RG \
  --query principalId --output tsv)

# Exemple RBAC : assigner Key Vault Secrets User sur le Key Vault
$KV_ID=$(az keyvault show --name "mon-keyvault" --query id --output tsv)
az role assignment create \
  --assignee $PRINCIPAL_ID \
  --role "Key Vault Secrets User" \
  --scope $KV_ID

# App Setting avec Key Vault Reference
az webapp config appsettings set \
  --name $APP --resource-group $RG \
  --settings "DB_PASSWORD=@Microsoft.KeyVault(VaultName=mon-keyvault;SecretName=db-password)"
Bicep — Managed Identity + Key Vault Reference
resource webApp 'Microsoft.Web/sites@2023-01-01' = {
  name: appName
  location: location
  identity: {
    type: 'SystemAssigned'  // Managed Identity
  }
  properties: {
    siteConfig: {
      appSettings: [
        {
          name: 'DB_PASSWORD'
          value: '@Microsoft.KeyVault(VaultName=${keyVaultName};SecretName=db-password)'
        }
      ]
    }
  }
}

resource kvRoleAssignment 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
  name: guid(keyVault.id, webApp.id, 'Key Vault Secrets User')
  scope: keyVault
  properties: {
    roleDefinitionId: subscriptionResourceId('Microsoft.Authorization/roleDefinitions', '4633458b-17de-408a-b874-0445c86b69e6')
    principalId: webApp.identity.principalId
    principalType: 'ServicePrincipal'
  }
}

Managed Identity activée (SystemAssigned ou UserAssigned) sur toutes les App Services

5
Deployment Security Inclus
Désactivation FTP basic auth, Workload Identity Federation (OIDC), restrictions SCM, slots de déploiement
🚨
Scénario d’attaque : Un secret GitHub Actions stockant les credentials Azure est compromis via une PR malveillante ou un fork. L’attaquant peut déployer du code arbitraire sur l’App Service. Workload Identity Federation (OIDC) élimine complètement les secrets de déploiement — GitHub/Azure DevOps s’authentifient via token OIDC temporaire, sans client_secret ni certificate à stocker.
Analogie : badge temporaire visiteur — valable seulement le temps du déploiement, se désactive automatiquement.

5.1 — Tableau de configuration

ParamètreDéfaut Azure⚡ Quick Win🔒 Avancé🎯 OptimalLicence

💡
Workload Identity Federation (WIF) — flux :
1. GitHub Actions génère un token OIDC signé par GitHub.
2. Azure valide le token contre le endpoint OIDC de GitHub.
3. Azure émet un token de courte durée pour le Service Principal.
4. Le déploiement se fait avec ce token — aucun secret permanent.
Source : Guide Azure DevOps — Workload Identity Federation.
⚠️
Piège : Désactiver SCM basic auth coupe les méthodes de déploiement basées sur les publishing credentials SCM (local Git, WebDeploy et autres flux basiques), mais ne restreint pas l’accès IP au site SCM (monapp.scm.azurewebsites.net). Le flag FTP est distinct. Combiner avec les Access Restrictions sur le SCM (voir S2).

5.2 — Configuration

Azure CLI — Désactiver FTP + SCM Basic Auth
# Désactiver FTP basic auth
az resource update \
  --resource-group $RG \
  --name "ftp" \
  --namespace Microsoft.Web \
  --resource-type basicPublishingCredentialsPolicies \
  --parent "sites/$APP" \
  --set properties.allow=false

# Désactiver SCM (Kudu) basic auth
az resource update \
  --resource-group $RG \
  --name "scm" \
  --namespace Microsoft.Web \
  --resource-type basicPublishingCredentialsPolicies \
  --parent "sites/$APP" \
  --set properties.allow=false
GitHub Actions — OIDC (Workload Identity Federation)
# .github/workflows/deploy.yml
permissions:
  id-token: write   # obligatoire pour OIDC
  contents: read

steps:
  - name: Azure Login (OIDC - sans secret)
    uses: azure/login@v2
    with:

Script All-in-One — Sécurisation App Service
Script idempotent combiné Azure CLI + PowerShell — variables en haut, phases numérotées, vérification finale
⚠️
Pré-requis : Az CLI 2.50+, PowerShell 7+ avec module Az.Websites, droits Contributor sur le RG + Security Admin pour Defender. Adapter les variables VARIABLES en haut du script avant exécution.
Bash/PowerShell — Script All-in-One App Service Security
#!/usr/bin/env bash
# Script : Sécurisation Azure App Service - All-in-One
# Sections : S1 Easy Auth | S2 Network | S3 TLS | S4 Identity | S5 Deploy | S6 Defender | S7 Hardening | S8 Logs | S10 Policy
# Idempotent : oui | Prérequis : az CLI 2.50+ | Droits : Contributor + Security Admin
#================================================================

# ============================================================
# VARIABLES - Adapter avant exécution
# ============================================================
APP="mon-webapp"
RG="mon-resource-group"
KV_NAME="mon-keyvault"
LAW_NAME="mon-log-analytics-workspace"
TENANT_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
CLIENT_ID="yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy"
ADMIN_IP="203.0.113.0/24"       # IP admin pour SCM
ALLOWED_ORIGIN="https://monfrontend.exemple.fr"

echo "=== [1/8] TLS + HTTPS + FTP Disabled ==="
az webapp update --name $APP --resource-group $RG --https-only true
az webapp config set --name $APP --resource-group $RG \
  --min-tls-version 1.2 --ftps-state Disabled \
  --remote-debugging-enabled false --http20-enabled true

echo "=== [2/8] Managed Identity + Key Vault RBAC ==="
az webapp identity assign --name $APP --resource-group $RG
PRINCIPAL_ID=$(az webapp identity show --name $APP --resource-group $RG \
  --query principalId -o tsv)
KV_ID=$(az keyvault show --name $KV_NAME --query id -o tsv)
az role assignment create --assignee $PRINCIPAL_ID \
  --role "Key Vault Secrets User" --scope $KV_ID 2>/dev/null || echo "Role déjà assigné"

echo "=== [3/8] Easy Auth (Microsoft Entra ID) ==="
az webapp auth microsoft update --name $APP --resource-group $RG \
  --client-id $CLIENT_ID --tenant-id $TENANT_ID \
  --client-secret-setting-name "MICROSOFT_PROVIDER_AUTHENTICATION_SECRET"
az webapp auth update --name $APP --resource-group $RG \
  --enabled true --action RedirectToLoginPage

echo "=== [4/8] Network - Access Restrictions + SCM ==="
az webapp config access-restriction add --name $APP --resource-group $RG \
  --rule-name DenyAll --action Deny --ip-address "0.0.0.0/0" --priority 900 2>/dev/null || echo "Règle déjà présente"
az webapp config access-restriction add --name $APP --resource-group $RG \
  --rule-name AllowAdminScm --action Allow \
  --ip-address $ADMIN_IP --priority 100 --scm-site true 2>/dev/null || echo "Règle SCM déjà présente"

echo "=== [5/8] Deployment Security - FTP + SCM Basic Auth Off ==="
az resource update --resource-group $RG --name ftp \
  --namespace Microsoft.Web --resource-type basicPublishingCredentialsPolicies \
  --parent "sites/$APP" --set properties.allow=false
az resource update --resource-group $RG --name scm \
  --namespace Microsoft.Web --resource-type basicPublishingCredentialsPolicies \
  --parent "sites/$APP" --set properties.allow=false

echo "=== [6/8] Runtime Hardening ==="
az webapp update --name $APP --resource-group $RG --client-affinity-enabled false
az webapp cors remove --name $APP --resource-group $RG --allowed-origins "*" 2>/dev/null
az webapp cors add --name $APP --resource-group $RG --allowed-origins $ALLOWED_ORIGIN

echo "=== [7/8] Logging - Diagnostic Settings ==="
APP_ID=$(az webapp show --name $APP --resource-group $RG --query id -o tsv)
LAW_ID=$(az monitor log-analytics workspace show --workspace-name $LAW_NAME \
  --resource-group $RG --query id -o tsv)
az monitor diagnostic-settings create --name "appservice-diag" \
  --resource $APP_ID --workspace $LAW_ID \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceAuditLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]' 2>/dev/null || echo "Diagnostic settings déjà configuré"

echo "=== [8/8] Defender for App Services ==="
az security pricing create --name AppServices --tier Standard 2>/dev/null || echo "Defender déjà activé"

echo "=== VÉRIFICATION FINALE ==="
az webapp show --name $APP --resource-group $RG \
  --query "{httpsOnly:httpsOnly, clientAffinity:clientAffinityEnabled}"
az webapp config show --name $APP --resource-group $RG \
  --query "{minTls:minTlsVersion, ftps:ftpsState, debug:remoteDebuggingEnabled, http2:http20Enabled}"
az webapp auth show --name $APP --resource-group $RG \
  --query "{authEnabled:enabled, action:unauthenticatedClientAction}"
echo "✅ Script terminé"
