API Types & Architecture Pay-per-request
REST API vs HTTP API vs WebSocket — choix d'architecture, endpoints, stages et versioning — 8 pts
Réf. AWS Well-Architected Framework — Security Pillar
🚨
Scénario : Votre équipe crée une REST API « par défaut » pour chaque microservice. Résultat : la facture peut être nettement plus élevée qu'avec une HTTP API équivalente si les fonctionnalités avancées ne sont pas utilisées, les stages ne sont pas isolés, et l'API est exposée en edge-optimized (CloudFront) alors qu'elle ne sert que la région eu-west-1.
Analogie : choisir le type d'API, c'est comme choisir le type de route — REST API = autoroute à péage avec toutes les options (caching, WAF, VTL), HTTP API = nationale rapide et économique, WebSocket = ligne de téléphone ouverte en continu. Prendre l'autoroute pour 2 km coûte cher et n'apporte rien.

Custom Domain Oui Oui Oui Toujours utiliser un custom domain
💡
Pourquoi chaque critère d'architecture compte :
  • Type d'API (REST vs HTTP) : ce choix détermine fortement le modèle de coût et la surface de sécurité disponible. REST API apporte WAF, caching, API keys, request validation et private endpoints ; HTTP API vise un périmètre plus léger et un coût plus bas si ces fonctions ne sont pas requises.
  • Endpoint type (Regional vs Edge) : Edge-optimized route via CloudFront — utile uniquement pour une API consommée mondialement. Pour une API régionale ou interne, Regional évite la latence CloudFront et simplifie le troubleshooting. AWS recommande Regional par défaut (FSBP).
  • Stages : sans stages séparés (dev/staging/prod), un déploiement casse directement la production. Chaque stage a ses propres variables, throttling et logs — c'est l'isolation minimale.
  • Versioning : sans stratégie (path-based /v1/, header X-API-Version, ou stage), les breaking changes cassent les clients existants. La convention /v1/ est la plus adoptée (Google, Stripe, GitHub).
  • Timeout d'intégration : HTTP API est limité à 30s, WebSocket à 29s, et REST API reste à 29s par défaut, avec possibilité de relèvement pour certaines REST régionales/privées au prix d'un compromis potentiel sur le throttle régional. Pour les traitements longs, prévoir un pattern async.

Paramètre Défaut AWS ⚡ Quick Win 🔒 Renforcé 🎯 Optimal Type API

Canary deployment Désactivé Désactivé 10% canary 10% canary + rollback auto REST
⚠️
Pièges courants :
  • Edge-optimized par défaut : AWS crée une distribution CloudFront invisible. Vous payez la latence CloudFront sans le vouloir, et le troubleshooting IP source est compliqué (IP CloudFront, pas le client).
  • Pas de rollback natif REST API : un déploiement de stage écrase l'ancien. Sans canary ou multi-stage, un bug en prod = downtime jusqu'au fix.
  • HTTP API n'intègre pas AWS WAF nativement via API Gateway : si vous avez besoin d'une association directe de web ACL au stage API Gateway, choisissez REST API. Avec HTTP API, la protection WAF passe généralement par CloudFront + AWS WAF en frontal.

    --name "my-api-v1" \
    --description "API de production - microservice orders" \

Authentication & Authorization IAM Cognito
Cognito, Lambda, IAM, JWT authorizers, resource policies, mTLS — 25 pts
Réf. AWS Well-Architected — SEC05, SEC06 • CIS AWS Foundations 1.x
🚨
Scénario : Une API publique REST est déployée sans authorizer. Les développeurs utilisent des API Keys comme seul mécanisme d'accès. Résultat : n'importe qui peut appeler l'API — les API Keys ne sont PAS de l'authentification (elles servent uniquement au throttling/quota via usage plans). En 48h, un bot a exfiltré 100 000 enregistrements.

Analogie : un authorizer est un vigile qui vérifie les papiers d'identité (token JWT, credentials IAM). Une API Key, c'est juste un badge visiteur numéroté — il identifie le quota consommé, mais ne prouve pas l'identité du porteur. Sans vigile, le badge ne sert à rien.
💡
Pourquoi chaque critère d'authentification compte :
  • Authorizer type : une API sans authorizer est ouverte à tous. Le choix (Cognito, Lambda, JWT, IAM) détermine la complexité et le coût — le JWT natif sur HTTP API n'ajoute pas d'invocation Lambda authorizer dédiée, alors que Lambda Authorizer ajoute une invocation Lambda par requête.
  • Token validation : sans validation de l'audience, de l'issuer et de l'expiration, un token volé d'un autre service est accepté. C'est la première cause d'escalade de privilèges sur les APIs.
  • Scopes / Groups : l'authentification seule (qui êtes-vous ?) ne suffit pas — l'autorisation (qu'avez-vous le droit de faire ?) via les scopes OAuth2 ou les groupes Cognito est indispensable pour le least privilege.
  • Resource Policy : même avec un authorizer, sans resource policy un compte AWS compromis peut appeler l'API. La resource policy ajoute une couche de filtrage IP/VPC/account en amont de tout authorizer.
  • mTLS : pour les APIs B2B, le mutual TLS garantit l'identité du client via certificat X.509 — plus robuste qu'un token JWT pouvant être exfiltré depuis un navigateur.

⚠️
Pièges courants :
  • API Key comme seule auth : la clé est transmise en clair dans le header x-api-key. Elle peut être interceptée, partagée ou commitée dans le code. Ce n'est PAS de l'authentification.
  • Lambda Authorizer sans cache : chaque requête invoque Lambda (~50-300ms de latence + coût). Activer le cache TTL (min 60s, max 3600s) pour réduire latence et coût de 90%.
  • Wildcard * dans resource policy : "Principal": "*" ouvre l'API à tous les comptes AWS. Toujours restreindre au compte, VPC ou IP source.
  • COGNITO_USER_POOLS sans scopes : tout utilisateur authentifié peut appeler toutes les routes. Ajouter des scopes OAuth2 pour différencier lecture/écriture/admin.
  • JWT sans validation aud : un token JWT valide pour une autre application peut être réutilisé. Toujours valider l'audience (aud) et l'issuer (iss).

2.5 — Configuration JWT Authorizer (HTTP API)

AWS CLI — JWT Authorizer avec Cognito
# Créer un JWT authorizer sur HTTP API (Cognito)
aws apigatewayv2 create-authorizer \
    --api-id HTTP_API_ID \
    --name "cognito-jwt-auth" \
    --authorizer-type "JWT" \
    --identity-source '$request.header.Authorization' \
    --jwt-configuration '{
        "Audience":["YOUR_COGNITO_APP_CLIENT_ID"],
        "Issuer":"https://cognito-idp.eu-west-1.amazonaws.com/YOUR_USER_POOL_ID"
    }' \
    --profile audit-admin \
    --region eu-west-1

# Attacher l'authorizer à une route avec scopes
aws apigatewayv2 update-route \
    --api-id HTTP_API_ID \
    --route-id ROUTE_ID \
    --authorization-type "JWT" \
    --authorizer-id AUTHORIZER_ID \
    --authorization-scopes "orders/read" \
    --profile audit-admin \
    --region eu-west-1

3.1 — Hiérarchie des limites de throttling

┌──────────────────────────────────────────────────────┐ │ LIMITES DU COMPTE AWS (RÉGIONALES) │ │ Steady-state : quota régional du compte │ │ Par défaut 10 000 RPS dans la plupart des │ │ régions, 2 500 RPS dans certaines régions │ │ récentes ; burst géré par token bucket │ │ ────────────────────────────────────────────── │ │ ┌────────────────────────────────────────────┐ │ │ │ STAGE-LEVEL (par API, par stage) │ │ │ │ Configurable par l'administrateur │ │ │ │ ──────────────────────────────────────── │ │ │ │ ┌───────────────────┐ ┌────────────────┐ │ │ │ │ │ METHOD-LEVEL │ │ USAGE PLAN │ │ │ │ │ │ GET /orders: │ │ Premium: │ │ │ │ │ │ 100 RPS │ │ 5000 RPS │ │ │ │ │ │ POST /orders: │ │ 100k/mois │ │ │ │ │ │ 50 RPS │ │ Free: │ │ │ │ │ │ │ │ 100 RPS │ │ │ │ │ │ │ │ 1k/mois │ │ │ │ │ └───────────────────┘ └────────────────┘ │ │ │ └────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────┘ → La limite la PLUS restrictive gagne toujours. → La limite effective reste la plus restrictive entre le quota régional du compte, le stage, la méthode et le plan d'usage.
💡
Pourquoi chaque niveau compte :
  • Account-level : limite partagée par toutes les APIs d'un compte dans une région. Le quota par défaut est souvent 10 000 RPS, mais certaines régions récentes démarrent à 2 500 RPS ; une API mal configurée peut saturer ce plafond global.
  • Stage-level : isole le trafic par environnement. Le stage dev ne doit pas impacter prod.
  • Method-level : les endpoints d'écriture (POST, DELETE) doivent être plus restrictifs que la lecture (GET).
  • Usage Plan : différencie les consommateurs (free tier vs premium). Sans usage plan, tous les clients partagent la même bande passante.
  • Burst vs Steady-state : le burst autorise un pic court via le token bucket, puis la limite redescend au steady-state. Le burst de compte est géré par API Gateway ; les limites stage/méthode restent votre principal levier de cadrage.

Scénario : Une API REST accepte n'importe quel JSON dans le body sans validation. Un attaquant envoie un payload de 10 MB avec des champs imbriqués, provoquant un denial-of-wallet (Lambda exécute pendant 30s × milliers de requêtes = facture exorbitante). 💡
Pourquoi chaque critère de validation compte :
  • Request Validation (body) : rejette les payloads malformés avant invocation Lambda — économise 100% du coût Lambda sur les requêtes invalides et bloque les injections au niveau gateway.

  • Gateway Responses : par défaut, les erreurs 4xx/5xx exposent des détails internes (stack traces). Personnaliser les réponses masque l'implémentation et améliore l'expérience client.

  • VPC Link : sans VPC Link, le trafic entre API Gateway et un backend privé passe par Internet. Le VPC Link crée un tunnel privé via un NLB/ALB/Cloud Map.

🚨
Scénario : Une API REST publique sans WAF est ciblée par un botnet. Un volume massif de payloads SQLi est envoyé. Le throttle API Gateway au niveau compte ne bloque rien car chaque IP envoie < 100 RPS. Lambda s'exécute 10 000 fois/sec, la facture explose, et la base DynamoDB subit un hot-partitioning. Un WAF avec managed rules, éventuellement complété par des rate-based rules ou Bot Control, aurait pu filtrer une partie significative du trafic avant API Gateway, sous réserve d'un tuning adapté.

Analogie : si le throttling (S3) est le limiteur de débit qui contrôle le flux, le WAF est le videur à l'entrée du club : il inspecte chaque requête et refuse celles qui portent des armes (SQLi), de faux papiers (bots) ou qui viennent de pays restreints (geo-blocking).
💡
Pourquoi chaque critère WAF compte :
  • Web ACL association : sans WAF, l'API reste plus exposée aux attaques web courantes. Les AWS Managed Rules fournissent des rule groups baseline, use-case specific et IP reputation, mais AWS recommande de les tester et de les ajuster avant la mise en production.
  • Rate-based rules : le throttling API Gateway (S3) limite le débit global, mais le WAF rate-based rule bloque par IP source — essentiel contre les attaques distribuées lentes.
  • Bot Control : les bots représentent 30-50% du trafic API. Sans détection, le scraping et le credential stuffing passent sous le radar du throttling classique.
  • IP Set / Geo restrictions : réduire la surface d'attaque en bloquant les pays non pertinents ou en whitelistant uniquement les IPs partenaires.
  • Custom rules (regex/size) : les managed rules ne couvrent pas les validations métier. Les rules custom (taille body, patterns regex) ajoutent une couche de protection spécifique à votre API.
⚠️
Limitation importante : AWS WAF ne peut être attaché qu'aux REST APIs (API Gateway v1). Les HTTP APIs (v2) ne supportent pas WAF nativement. Solution de contournement : placer un CloudFront devant l'HTTP API, puis attacher WAF à CloudFront.

Custom domain names, certificats ACM, Route 53, base path mappings, TLS 1.2 minimum — 5 pts
Réf. AWS Well-Architected — SEC09 • FSBP APIGateway.2
🚨
Scénario : L'API est accessible via l'URL par défaut abc123.execute-api.eu-west-1.amazonaws.com. Un développeur l'intègre en dur dans l'application mobile. Le jour où l'API est recréée (migration, changement de région), l'URL change et l'app est cassée pour tous les utilisateurs. Un custom domain api.example.com aurait permis de migrer le backend de manière transparente. De plus, l'URL execute-api supporte encore TLS 1.0 par défaut — non conforme PCI DSS.

Analogie : l'URL execute-api est l'adresse IP de votre serveur. Le custom domain est votre nom de domaine professionnel (carte de visite). Changer d'hébergeur ne change pas votre adresse — vous redirigez le DNS.
💡
Pourquoi chaque critère de domaine personnalisé compte :
  • Custom Domain Name : sans domaine personnalisé, l'URL execute-api est couplée à l'API. Toute recréation/migration casse les clients. Le custom domain découple le DNS du backend.
  • TLS Security Policy : la policy par défaut accepte TLS 1.0 (vulnérable POODLE/BEAST). Forcer TLS_1_2 est obligatoire PCI DSS et recommandé FSBP (APIGateway.2).
  • Certificat ACM : ACM génère et renouvelle automatiquement les certificats TLS — zéro risque d'expiration oubliée. La région du certificat doit correspondre au type d'endpoint.
  • Base Path Mapping : permet de router /v1 vers une API et /v2 vers une autre derrière le même domaine. Indispensable pour le versioning transparent.
  • Disable Default Endpoint : même avec un custom domain, l'URL execute-api reste active par défaut. La désactiver force tout le trafic via le domaine sécurisé (TLS 1.2 + WAF).

Private APIs & VPC REST API HTTP API
Private REST APIs, VPC endpoints, resource policies VPC, VPC Link v2, intégrations privées — 4 pts
Réf. AWS Well-Architected — SEC05 • FSBP EC2.10
🚨
Scénario : Une API interne de gestion des paies est déployée en REST API publique avec un API key comme seule protection. Un attaquant scanne les endpoints execute-api depuis Internet, récupère l'API key dans le code source d'un frontend, et accède aux données salariales. Une Private API + VPC Endpoint aurait rendu l'API totalement invisible depuis Internet.

Analogie : une API publique est un guichet ouvert sur la rue. Une Private API est un bureau accessible uniquement depuis le couloir interne de l'immeuble (VPC).
💡
Pourquoi chaque critère d'API privée compte :
  • Private API (REST) : rend l'API totalement invisible depuis Internet — aucun DNS public, aucun chemin réseau. Le seul accès passe par le VPC Endpoint Interface dans le même VPC (ou via peering/Transit Gateway).
  • VPC Endpoint (Interface) : crée une ENI privée dans votre VPC pour accéder au service API Gateway. Le Security Group de cette ENI contrôle qui peut appeler l'API — firewall L4 avant toute logique applicative.
  • Resource Policy VPC : restreint l'accès à un VPC/VPC Endpoint spécifique. Sans resource policy, n'importe quel compte AWS avec le bon VPC Endpoint peut appeler votre Private API.
  • VPC Link (v1/v2) : le pont entre une API publique et un backend privé (ALB, NLB, Cloud Map). Sans VPC Link, le backend doit être publiquement accessible — ce qui annule l'intérêt du VPC.
  • DNS privé : avec le DNS privé activé sur le VPC Endpoint, le hostname execute-api résout vers l'IP privée — les appels restent dans le réseau AWS sans transiter par Internet.

  • Private API sans DNS privé : si privateDnsEnabled=false sur le VPC Endpoint, vous devez utiliser l'URL spéciale vpce-xxx.execute-api.region.vpce.amazonaws.com + header Host: api-id.execute-api.region.amazonaws.com. Activer le DNS privé simplifie l'appel.
  • VPC Link v1 vs v2 : v1 = REST API + NLB uniquement. v2 = HTTP API + ALB/NLB/Cloud Map. Ils ne sont pas interchangeables.
  • Resource Policy vide : une resource policy mal formée ou vide bloque TOUT accès à l'API. Toujours tester après déploiement.

Scénario : Une équipe utilise une REST API avec Lambda pour un simple proxy CRUD. 💡
Pourquoi chaque critère de coût compte :
  • HTTP API vs REST API : HTTP API suit un modèle tarifaire plus léger que REST API, mais le gain réel dépend surtout des fonctions requises (WAF, API keys, validation, private endpoints, etc.).

  • Compression (gzip) : activer minimumCompressionSize peut réduire le volume de transfert sortant, à condition de mesurer l'impact réel sur les payloads servis.
  • Intégrations directes AWS : selon le type d'API et le service cible, une intégration sans Lambda peut réduire la latence et la surface opérationnelle, mais la couverture fonctionnelle doit être validée au cas par cas.
  • Request Validation : filtrer plus tôt les requêtes invalides peut éviter de consommer inutilement des ressources backend, en particulier sur REST API où la validation est plus complète.

Script All-in-One — Audit & Hardening API Gateway
Script AWS CLI exécutable pour auditer et sécuriser toutes les APIs d'un compte
Bash — Script All-in-One Audit API Gateway
#!/bin/bash
# ============================================================
# API Gateway Security Audit Script
# Usage: ./audit-apigw.sh --profile audit-readonly --region eu-west-1
# ============================================================

set -euo pipefail

PROFILE="${1:---profile audit-readonly}"
REGION="${2:---region eu-west-1}"

echo "╔══════════════════════════════════════════════════╗"
echo "║  API Gateway Security Audit                      ║"
echo "╚══════════════════════════════════════════════════╝"

# --- S1: Lister toutes les APIs ---
echo "\n[S1] REST APIs:"
aws apigateway get-rest-apis $PROFILE $REGION \
    --query "items[].{Name:name,Id:id,Type:endpointConfiguration.types[0],DisableDefault:disableExecuteApiEndpoint}" \
    --output table

echo "\n[S1] HTTP APIs:"
aws apigatewayv2 get-apis $PROFILE $REGION \
    --query "Items[].{Name:Name,Id:ApiId,Protocol:ProtocolType}" \
    --output table

# --- S2: Authorizers ---
echo "\n[S2] Authorizers par API:"
for API_ID in $(aws apigateway get-rest-apis $PROFILE $REGION --query "items[].id" --output text); do
    echo "  API: $API_ID"
    aws apigateway get-authorizers --rest-api-id "$API_ID" $PROFILE $REGION \
        --query "items[].{Name:name,Type:type,TTL:authorizerResultTtlInSeconds}" \
        --output table 2>/dev/null || echo "    (aucun authorizer)"
done

# --- S3: Throttling (account level) ---
echo "\n[S3] Account-level throttling:"
aws apigateway get-account $PROFILE $REGION \
    --query "{RateLimit:throttleSettings.rateLimit,BurstLimit:throttleSettings.burstLimit}"

# --- S6: WAF associations ---
echo "\n[S6] WAF Web ACLs:"
aws wafv2 list-web-acls --scope REGIONAL $PROFILE $REGION \
    --query "WebACLs[].{Name:Name,Id:Id}" --output table

for API_ID in $(aws apigateway get-rest-apis $PROFILE $REGION --query "items[].id" --output text); do
    STAGE_ARNS=$(aws apigateway get-stages --rest-api-id "$API_ID" $PROFILE $REGION \
        --query "item[].{Stage:stageName}" --output text 2>/dev/null)
    for STAGE in $STAGE_ARNS; do
        ARN="arn:aws:apigateway:${REGION#--region }::/restapis/$API_ID/stages/$STAGE"
        WAF=$(aws wafv2 get-web-acl-for-resource --resource-arn "$ARN" $PROFILE $REGION \
            --query "WebACL.Name" --output text 2>/dev/null || echo "NONE")
        echo "  $API_ID/$STAGE -> WAF: $WAF"
    done
done

# --- S7: Logging ---
echo "\n[S7] Stages sans access logs:"
for API_ID in $(aws apigateway get-rest-apis $PROFILE $REGION --query "items[].id" --output text); do
    aws apigateway get-stages --rest-api-id "$API_ID" $PROFILE $REGION \
        --query "item[?!accessLogSettings].{API:'$API_ID',Stage:stageName}" \
        --output table 2>/dev/null
done

# --- S8: Custom Domains & TLS ---
echo "\n[S8] Custom Domains & Security Policy:"
aws apigateway get-domain-names $PROFILE $REGION \
    --query "items[].{Domain:domainName,TLS:securityPolicy,Type:endpointConfiguration.types[0]}" \
    --output table

# --- S9: VPC Endpoints ---
echo "\n[S9] VPC Endpoints execute-api:"
aws ec2 describe-vpc-endpoints \
    --filters "Name=service-name,Values=com.amazonaws.*.execute-api" \
    $PROFILE $REGION \
    --query "VpcEndpoints[].{Id:VpcEndpointId,VPC:VpcId,DNS:PrivateDnsEnabled,State:State}" \
    --output table

echo "\n╔══════════════════════════════════════════════════╗"
echo "║  Audit terminé - Vérifier les résultats ci-dessus  ║"
echo "╚══════════════════════════════════════════════════╝"
