Scénario d'attaque : Un cluster AKS créé avec les paramètres par défaut expose son API server sur Internet (--enable-private-cluster non activé). Un attaquant découvre l'endpoint via Shodan, exploite une vulnérabilité Kubernetes non patchée (CVE), obtient un accès cluster-admin et déploie un cryptominer sur tous les nodes. Le cluster utilisant un service principal (SPN) au lieu d'une managed identity, l'attaquant récupère aussi les credentials Azure pour pivoter vers d'autres ressources du tenant.

Analogie : Un cluster AKS public, c'est un immeuble avec une entrée directe sur rue — n'importe qui peut tenter d'ouvrir la porte. Un private cluster déplace l'entrée à l'intérieur d'une cour fermée : seuls ceux déjà dans le réseau interne peuvent y accéder.

Pourquoi chaque paramètre Cluster Security compte :

• --enable-private-cluster : Sans private cluster, l'API server Kubernetes est accessible depuis Internet. C'est la surface d'attaque n°1 — toute vulnérabilité K8s (CVE) est directement exploitable à distance. CIS AKS Benchmark v1.8.0 §5.4.1 (Restrict Access to the Control Plane Endpoint) & §5.4.2 (Private Endpoint Enabled).

• --api-server-authorized-ip-ranges : Quand le private cluster n'est pas possible (coût du VPN, complexité), les authorized IP ranges sont le minimum vital. Sans restriction, les 4.3 milliards d'adresses IPv4 peuvent atteindre votre API server.

• --enable-aad / --enable-azure-rbac : Les comptes locaux Kubernetes (certificats client) n'ont pas de MFA, pas d'audit centralisé, pas de Conditional Access. Avec Entra ID, chaque accès kubectl passe par l'authentification Azure (MFA, CA, PIM). CIS AKS Benchmark v1.8.0 §5.5.1 (Manage Kubernetes RBAC users with Microsoft Entra ID, ex-Azure AD) & §5.5.2 (Use Azure RBAC for Kubernetes Authorization).

• --disable-local-accounts : Même avec Entra ID activé, les comptes locaux restent fonctionnels (az aks get-credentials --admin). Un attaquant disposant du kubeconfig admin bypass complètement Entra ID. Désactiver les comptes locaux force 100% du trafic par Entra ID.

• --assign-identity (user-assigned si nécessaire) : AKS crée aujourd'hui une managed identity system-assigned par défaut. Une user-assigned identity reste utile quand le cycle de vie de l'identité doit survivre à la recréation du cluster, ou quand sa gouvernance RBAC doit être pilotée séparément.

• --tier Standard/Premium : Le tier Free n'a pas de SLA control plane. Standard est le tier de production avec SLA control plane. Premium ajoute surtout le support LTS Microsoft pour les versions Kubernetes concernées ; il ne faut pas le confondre avec AKS Automatic, qui repose sur le tier Standard.

• --os-sku : Ubuntu et Azure Linux sont tous deux supportés dans AKS, avec des versions par défaut qui évoluent selon la version Kubernetes. Le bon choix est un OS SKU explicite validé contre vos besoins de compatibilité, FIPS/CVM, taille de VM et fenêtre de support. Évitez les comparatifs figés de type « X a n% de CVE en moins que Y » : la position officielle actuelle est de suivre la matrice de support et les annonces de retraite live.

• --auto-upgrade-channel : none désactive les mises à jour automatiques ; patch, stable et rapid pilotent des cadences différentes. Pour rester dans la fenêtre de support AKS sans surprise, les docs Microsoft recommandent généralement stable ou rapid, à combiner avec une maintenance window.

• --node-os-upgrade-channel : Ce canal est distinct de l'upgrade Kubernetes du cluster. SecurityPatch privilégie des correctifs sécurité testés et peut limiter les reimages ; NodeImage applique des images nodes AKS testées avec correctifs sécurité et bugfixes sur une cadence hebdomadaire. Sur les nouveaux clusters, NodeImage est le comportement par défaut ; en production, le plus important est de choisir explicitement le canal et de le cadrer par maintenance window.
⚠️
Pièges courants — ce que les défauts ne couvrent PAS :

• Private cluster ≠ tout est privé : Le private cluster rend l'API server privé, mais les load balancers de type LoadBalancer restent publics par défaut. Chaque Service type=LoadBalancer crée une IP publique. Utiliser service.beta.kubernetes.io/azure-load-balancer-internal: "true" pour les forcer en interne.

• az aks get-credentials --admin : Même avec Entra ID activé, cette commande génère un kubeconfig cluster-admin qui bypass Entra ID. Solution : --disable-local-accounts.

• AKS Automatic ≠ tout est sécurisé : AKS Automatic préconfigure plusieurs briques (tier Standard, identité managée, posture opérationnelle), mais il ne dispense pas de valider vos add-ons de sécurité, votre stratégie de secrets, l'exposition réseau et les contrôles d'egress.

• Migration managed identity : Certains clusters historiques utilisent encore un Service Principal (SPN) avec un secret qui expire. Vérifier avec az aks show --query servicePrincipalProfile. Si non-null, migrer vers managed identity.

Scénario d'attaque : Un pod compromis dans le namespace app-frontend effectue un mouvement latéral vers la base de données dans app-backend car aucune network policy n'est définie : par défaut, tous les pods d'un cluster AKS peuvent communiquer entre eux sans restriction (flat network). Analogie : Sans network policies, un cluster AKS est un open space sans cloisons — n'importe quel poste (pod) peut parler à n'importe quel autre. Les network policies ajoutent des portes coupe-feu entre chaque zone : le frontend ne peut parler qu'au backend, le backend qu'à la BDD, et rien ne sort sans autorisation explicite.

Pourquoi chaque paramètre Network Security compte :

• --network-plugin azure : Azure CNI est aujourd'hui la base réseau recommandée pour les nouveaux clusters AKS. En mode Overlay, les pods utilisent un CIDR privé distinct du VNet, ce qui réduit la pression sur les IPs du réseau Azure tout en gardant un modèle supporté à grande échelle. Kubenet reste encore visible sur des clusters existants, mais Microsoft a annoncé sa retraite sur AKS : ce n'est plus une cible saine pour un nouveau design.

• --network-policy / Cilium Network Policy : Sans network policies, tous les pods peuvent communiquer entre eux (port 80, 443, 5432, 27017… tout est ouvert). Un pod compromis peut scanner l'intégralité du cluster. AKS supporte plusieurs moteurs selon le mode réseau et l'OS, mais la documentation Microsoft recommande aujourd'hui Cilium pour les nouveaux déploiements Linux. Les network policies implémentent la micro-segmentation : seul le trafic explicitement autorisé passe.

• Egress lockdown : Un pod compromis peut exfiltrer des données vers n'importe quelle destination Internet si le trafic sortant n'est pas filtré. Azure Firewall avec FQDN filtering autorise uniquement les destinations nécessaires (registres de conteneurs, APIs, mises à jour). Réduit le risque de C2 (Command & Control) et d'exfiltration.

• Ingress Controller + WAF : Exposer des services directement via LoadBalancer sans WAF = pas de protection contre les attaques web (SQLi, XSS, DDoS L7). Application Gateway avec WAF v2 offre une couche de protection applicative avant même d'atteindre les pods.

• Internal load balancer : Chaque Service type=LoadBalancer crée une IP publique Azure par défaut. Un cluster avec 10 services expose 10 IPs publiques. L'annotation internal: "true" force les services en interne sur le VNet.
⚠️
Pièges courants — réseau :

• kubenet n'est plus une cible d'avenir : Kubenet peut encore exister sur des clusters AKS historiques et Calico peut couvrir certains cas Linux, mais il ne supporte pas la trajectoire recommandée Azure CNI + Cilium. Microsoft a annoncé la fin de support AKS de kubenet ; pour tout nouveau design ou plan de migration, ciblez Azure CNI, idéalement Overlay si la pression IP VNet est un sujet.

• Network policy deny-all oubliée sur kube-system : Appliquer une deny-all sur kube-system sans exceptions casse CoreDNS, kube-proxy, et les composants système. Toujours créer des exceptions pour les pods système.

• Azure CNI et plan d'adressage : En mode Azure CNI traditionnel, chaque pod consomme une IP du plan réseau Azure ; en mode Overlay, chaque node reçoit un bloc /24 prélevé dans le pod CIDR. Le point critique n'est pas d'imposer un /16 universel, mais de dimensionner le subnet nodes, le pod CIDR et le service CIDR sans chevauchement avec peering, ExpressRoute, VPN et réseaux on-prem.

• Egress lockdown partiel : Un UDR (User Defined Route) qui redirige le trafic vers Azure Firewall ou un autre point de sortie sans toutes les destinations requises casse rapidement le cluster. Ne figez pas une allowlist d'endpoints dans un guide long terme : vérifiez toujours la documentation live AKS sur les flux sortants requis pour votre mode réseau, votre OS et vos add-ons.

# Deny-all ingress & egress par défaut pour le namespace app
cat <<EOF | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
  namespace: app-production
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
---
# Autoriser le frontend à recevoir le trafic du load balancer
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-frontend
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
    - Ingress
  ingress:
    - from: []  # depuis l'extérieur (ingress controller)
      ports:
        - protocol: TCP
          port: 8080
---

  name: allow-frontend-to-backend
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP

---

Pourquoi chaque paramètre Identity compte :

• Azure RBAC for Kubernetes : Le RBAC natif K8s (Roles/ClusterRoles) est géré indépendamment d'Azure — pas de Conditional Access, pas de PIM, pas d'audit centralisé dans Entra ID. Azure RBAC for K8s unifie l'autorisation : un seul système RBAC pour Azure ET Kubernetes, avec des rôles prédéfinis (Azure Kubernetes Service RBAC Reader/Writer/Admin/Cluster Admin). CIS AKS Benchmark v1.8.0 §5.5.2 (Use Azure RBAC for Kubernetes Authorization).

• Workload Identity : Avant Workload Identity, les pods accédaient aux services Azure via : (1) des secrets en dur dans des ConfigMaps/Secrets (🚨 fuite de credentials), (2) la managed identity du node pool entier (tous les pods du node ont les mêmes permissions). Workload Identity attribue une identité unique à chaque pod via la fédération OIDC — le pod reçoit un token à durée limitée, sans secret stocké. Remplaçant officiel de AAD Pod Identity (déprécié depuis oct. 2023).

• ServiceAccount auto-mount = false : Par défaut, chaque pod reçoit un token Kubernetes monté à /var/run/secrets/kubernetes.io/serviceaccount/token. Si le pod est compromis, ce token permet de faire des appels API Kubernetes avec les permissions du ServiceAccount. Désactiver l'auto-mount et n'activer que quand nécessaire réduit la surface d'attaque. CIS Kubernetes 5.1.6.

• PIM JIT pour cluster-admin : Un accès permanent cluster-admin est le vecteur d'attaque n°1 sur un cluster K8s. PIM (Privileged Identity Management) permet un accès temporaire (1h), avec approbation et justification, audité dans Entra ID.
⚠️
Pièges courants — identité :

• AAD Pod Identity (déprécié) : Si vous utilisez encore aadpodidentity (NMI + MIC), migrez vers Workload Identity immédiatement. AAD Pod Identity est déprécié depuis octobre 2023 et ne reçoit plus de correctifs sécurité. Il présente aussi des vulnérabilités connues (NMI TOCTOU race condition).

• AcrPull via --admin-enabled sur ACR : Activer l'admin account sur ACR (az acr update --admin-enabled true) crée un compte avec username/password statique. Préférer az aks update --attach-acr qui utilise la managed identity du cluster.

• Kubelet identity trop permissive : La managed identity du kubelet (node pool) a par défaut le rôle Contributor sur le resource group des nodes (MC_*). Cela signifie que n'importe quel pod peut, via IMDS (169.254.169.254), obtenir un token avec des permissions Contributor. Workload Identity élimine ce risque car chaque pod n'a que ses propres permissions.

• ClusterRoleBinding cluster-admin trop large : Vérifier avec kubectl get clusterrolebindings -o json | jq '.items[] | select(.roleRef.name=="cluster-admin")'. Chaque binding doit être justifié.

Pod Security Gratuit
Pod Security Standards (PSS), Security Context, AppArmor, Seccomp, readOnlyRootFilesystem, non-root — 10 pts
🚨
Scénario d'attaque : Un conteneur déployé avec privileged: true et hostPID: true. Un attaquant exploite une RCE dans l'application, obtient un shell dans le conteneur, puis utilise nsenter --target 1 --mount --uts --ipc --net --pid -- /bin/bash pour s'échapper vers le host node. Pourquoi c'est critique : Un conteneur privileged n'a aucune isolation du host — il a accès à tous les devices (/dev), tous les namespaces du host, et toutes les capabilities Linux. C'est équivalent à exécuter le processus directement sur le node en tant que root.

Pourquoi chaque paramètre Pod Security compte :

• Pod Security Standards (PSS) : Remplaçant officiel de PodSecurityPolicy (retiré K8s 1.25). Trois niveaux : Privileged (tout autorisé), Baseline (bloque les élévations dangereuses), Restricted (hardening complet). S'applique par namespace via des labels. CIS Kubernetes 5.2.*.

• runAsNonRoot : Par défaut, la plupart des images Docker tournent en root (UID 0). Un processus root dans un conteneur peut exploiter des vulnérabilités kernel pour s'échapper. runAsNonRoot: true force K8s à rejeter le pod si l'image spécifie USER root.

• readOnlyRootFilesystem : Empêche un attaquant d'écrire des binaires malveillants dans le conteneur (webshells, cryptominers). Le conteneur ne peut écrire que dans les emptyDir / tmpfs explicitement montés.

• Capabilities drop ALL : Le jeu de capabilities par défaut inclut CAP_NET_RAW (ARP spoofing), CAP_SYS_CHROOT (jail escape), CAP_SETUID (privilege escalation). drop: ALL les supprime toutes.

• Seccomp RuntimeDefault : Réduit l'exposition à des syscalls sensibles sans exiger d'écrire un profil custom pour chaque workload. C'est une bonne base de hardening pour la majorité des applications, à valider en staging avant généralisation.
⚠️
Pièges courants — pod security :

• PSS enforce immédiat = pods qui crashent : Appliquer Restricted enforce sur un namespace existant peut casser des pods qui tournent en root (Redis, Nginx par défaut). Commencez par warn + audit, corrigez les workloads, puis passez en enforce.

• InitContainers oubliés : Les initContainers nécessitent parfois des capabilities que les containers n'ont pas (ex: NET_ADMIN pour Istio init). Vérifiez chaque initContainer individuellement.

• readOnlyRootFilesystem + applications stateful : Certaines applications écrivent dans /tmp, /var/cache, /var/log. Montez un emptyDir pour ces chemins.

• PodSecurityPolicy (retirée) : PSP retirées en K8s 1.25. Migrez vers Pod Security Standards + Azure Policy.

Pourquoi chaque paramètre Secrets & Configuration compte :

• Base64 ≠ chiffrement : Les K8s Secrets sont encodés en Base64 (pas chiffrés). Toute personne avec get secrets dans le namespace peut les lire. C'est le piège le plus courant pour les débutants K8s. CIS Kubernetes 5.4.1.

• CSI Secrets Store + Key Vault : Le driver CSI monte les secrets Key Vault directement comme fichiers dans le pod. Le pod ne voit jamais le secret en clair dans un objet K8s. Avantages : audit centralisé dans Key Vault, rotation sans redéploiement, RBAC Azure fin (Key Vault Secrets User par secret).

• Rotation automatique : Le CSI driver peut polluer Key Vault périodiquement (rotationPollInterval) pour détecter les changements. Le pod voit le nouveau secret sans restart. Critique pour les certificats TLS et les tokens API.

• Workload Identity + Key Vault : Combine S3 (Workload Identity) + S8 (Key Vault). Chaque pod a sa propre identité Azure avec accès uniquement aux secrets qui le concernent. Zéro credential stocké.
⚠️
Pièges courants — secrets :

• syncSecret.enabled = true : L'option secretObjects du CSI driver synchronise les secrets Key Vault en K8s Secrets. Cela revient à stocker le secret dans etcd — annulant en partie l'avantage de Key Vault. N'utilisez cette option que si nécessaire (ex: imagePullSecret, ingress TLS).

• Key Vault Firewall : Si Key Vault a un firewall activé et un private endpoint, les nodes AKS doivent pouvoir le joindre via le VNet. Vérifiez que le subnet AKS est autorisé ou qu'un private endpoint est configuré.

• Secrets dans les logs : Les env vars sont visibles dans kubectl describe pod. Préférez les volume mounts aux env vars pour les secrets sensibles.

Upgrades & Maintenance Gratuit AKS Premium (LTS)

🚨
Scénario d'attaque : Un cluster AKS reste plusieurs versions mineures derrière et sort de la fenêtre de support normale (N, N-1, N-2), voire tombe en platform support (N-3). Une CVE critique sur kubelet ou un composant du control plane est corrigée sur les versions supportées, mais le cluster en retard ne bénéficie plus des mêmes garanties de correctifs. L'attaquant exploite la faille, prend le contrôle des nodes et pivote vers les workloads et secrets du cluster.

Analogie : Un cluster AKS non mis à jour, c'est un système d'alarme dont la batterie est morte. Il a l'air d'être là, mais quand un intrus entre, rien ne se déclenche. La mise à jour, c'est remplacer la batterie avant qu'elle ne soit vide.

Pourquoi chaque paramètre Upgrades compte :

• Support Microsoft : AKS supporte les 3 dernières versions GA (N, N-1, N-2). Une version N-3 peut encore bénéficier d'un platform support limité, mais sans les mêmes garanties sur les composants Kubernetes et leurs correctifs. Pour les clusters de production, rester dans la fenêtre N / N-1 / N-2 doit être la norme.

• Auto-upgrade channels : none désactive les upgrades automatiques, patch garde la minor actuelle, stable suit une minor supportée stable, et rapid suit la dernière minor supportée. Les docs Microsoft recommandent généralement stable ou rapid si vous voulez rester automatiquement dans la fenêtre de support ; patch reste utile si vous souhaitez figer la minor version tout en gardant les correctifs.

• Node OS auto-upgrade : Ce canal est distinct de l'upgrade Kubernetes. SecurityPatch privilégie les correctifs sécurité testés ; NodeImage applique des images nodes AKS testées avec correctifs sécurité et bugfixes. Sur les nouveaux clusters, NodeImage est le comportement par défaut ; le point important est de choisir explicitement votre canal et de le cadrer par maintenance window.

• Maintenance window + PDB : Une maintenance window contrôle quand l'upgrade démarre, pas l'absence d'impact. Microsoft recommande une fenêtre d'au moins 4 heures. Des PDB cohérents et un maxSurge adapté restent indispensables pour limiter les interruptions.
⚠️
Pièges courants — upgrades :

• Skip versions : Les upgrades de control plane restent séquentiels entre versions mineures supportées. Si vous êtes très en retard, plusieurs opérations successives peuvent être nécessaires pour revenir dans la fenêtre supportée. Les règles de version skew des node pools n'éliminent pas cette contrainte sur le control plane.

• API deprecations : Chaque version Kubernetes retire ou déprécie des APIs. Avant d'upgrader, vérifiez la page AKS des breaking changes, les release notes, et utilisez si besoin un outil comme kubent pour détecter les manifests à risque.

• Maintenance window = pas de garantie zero-downtime : La maintenance window définit quand l'upgrade commence, pas sa durée. Avec des PDB correctement configurés, un surge suffisant et une fenêtre d'au moins 4 heures, l'impact est réduit mais pas supprimé.

• Node pools système : Le node pool système (mode System) héberge CoreDNS, kube-proxy, etc. Upgradez-le avec précaution. Idéal : créez un 2e node pool système, migrez, puis supprimez l'ancien (blue-green).

Interaction Sections Dépendance Impact
Workload Identity → Key Vault S3 ↔ S8 Le CSI Secrets Store driver utilise Workload Identity pour s'authentifier auprès de Key Vault sans credentials. Si WI est mal configuré, aucun secret Key Vault n'est accessible aux pods. Configurer S3 avant S8.
Network Policies → Defender S2 ↔ S6 Defender for Containers utilise un DaemonSet (sensor eBPF) qui communique avec le service cloud Defender. Les Network Policies deny-all bloquent ce trafic. Ajouter des Network Policies allow pour gatekeeper-system, kube-system (egress vers l'API Microsoft) lors de la configuration S2.
Azure Policy → Pod Security S7 ↔ S5 Les Azure Policies "no privileged containers" et "allowed capabilities" renforcent les PSS du namespace. Les deux mécanismes doivent être alignés. Si S7 dit Deny privileged mais S5 n'a pas le label PSS Restricted, les pods existants peuvent contourner la policy (Gatekeeper = admission only, PSS = label check).
Container Images → Azure Policy S4 ↔ S7 La policy "allowed images" de S7 est le mécanisme d'enforcement de la restriction de registres définie en S4. Définir la liste des registres autorisés en S4, puis appliquer le Deny via S7. Ordre : S4 (stratégie) → S7 (enforcement).
Monitoring → Secrets S9 ↔ S8 Les diagnostic logs de Key Vault (S8) doivent être envoyés au Log Analytics workspace configuré en S9 pour la corrélation d'incidents. Configurer S9 (Log Analytics workspace) avant S8 (Key Vault diagnostic settings).
Cluster Security → Upgrades S1 ↔ S10 Un cluster privé (S1) avec API server privé nécessite un jump box ou un command invoke pour lancer les upgrades (S10). Prévoir l'accès à l'API server pour les opérations de maintenance. Tester az aks command invoke avant le premier upgrade.
Identity → ACR S3 ↔ S4 L'attach ACR (S4) utilise la kubelet identity (S3) pour puller les images. Un changement de managed identity casse le pull. Toujours ré-attacher l'ACR après un changement d'identité cluster : az aks update --attach-acr.

Script All-in-One — Hardening AKS Complet
Script exécutable pour appliquer toutes les best practices de sécurité AKS

Script idempotent — peut être exécuté plusieurs fois sans effet de bord. Variables au début, phases numérotées, vérification incluse.

Bash — Script All-in-One AKS Hardening
#!/bin/bash
# ═══════════════════════════════════════════════════════════════
# AKS Security Hardening — Script All-in-One
# Idempotent : peut être exécuté plusieurs fois sans risque
# ═══════════════════════════════════════════════════════════════
set -euo pipefail

# ──── VARIABLES (À ADAPTER) ────
RG="rg-aks-production"
CLUSTER="aks-prod-001"
LOCATION="westeurope"
ACR_NAME="acrprod001"
KV_NAME="kv-aks-prod-001"
LAW_NAME="law-aks-prod"
SUB_ID=$(az account show --query id -o tsv)
TENANT_ID=$(az account show --query tenantId -o tsv)

echo "══════════════════════════════════════════"
echo "AKS Security Hardening — $CLUSTER"
echo "══════════════════════════════════════════"

# ──── PHASE 1 : Entra ID + RBAC (S1, S3) ────
echo "[Phase 1/7] Entra ID integration + RBAC..."
az aks update \
    --resource-group $RG --name $CLUSTER \
    --enable-aad --enable-azure-rbac \
    --aad-admin-group-object-ids "$(az ad group show -g 'AKS-Admins' --query id -o tsv)" \
    2>/dev/null || echo "  → Déjà activé"

# Authorized IP ranges (adapter les IPs)
az aks update \
    --resource-group $RG --name $CLUSTER \
    --api-server-authorized-ip-ranges "203.0.113.0/24,198.51.100.0/24" \
    2>/dev/null || echo "  → IP ranges déjà configurées"

# ──── PHASE 2 : Network Policies (S2) ────
echo "[Phase 2/7] Vérification Network Policies..."
NP_COUNT=$(kubectl get netpol --all-namespaces --no-headers 2>/dev/null | wc -l)
if [ "$NP_COUNT" -eq 0 ]; then
    echo "  ⚠ Aucune NetworkPolicy détectée — créez des deny-all par namespace"
else
    echo "  ✓ $NP_COUNT NetworkPolicies détectées"
fi

# ──── PHASE 3 : Container Image Security (S4) ────
echo "[Phase 3/7] ACR attach + Defender for Containers..."
az aks update \
    --resource-group $RG --name $CLUSTER \
    --attach-acr $ACR_NAME \
    2>/dev/null || echo "  → ACR déjà attaché"

# ──── PHASE 4 : Defender for Containers (S6) ────
echo "[Phase 4/7] Activation Defender for Containers..."
az security pricing create \
    --name Containers --tier Standard \
    2>/dev/null || echo "  → Defender déjà activé"

# ──── PHASE 5 : Azure Policy (S7) ────
echo "[Phase 5/7] Azure Policy addon + initiative..."
az aks enable-addons \
    --resource-group $RG --name $CLUSTER \
    --addons azure-policy \
    2>/dev/null || echo "  → Policy addon déjà activé"

az policy assignment create \
    --name "aks-baseline-audit" \
    --display-name "AKS Baseline Security (Audit)" \
    --scope "/subscriptions/$SUB_ID/resourceGroups/$RG" \
    --policy-set-definition "a8640138-9b0a-4a28-b8cb-1666c838647d" \
    --params '{"effect":{"value":"Audit"}}' \
    2>/dev/null || echo "  → Initiative déjà assignée"

# ──── PHASE 6 : Secrets (S8) ────
echo "[Phase 6/7] CSI Secrets Store addon..."
az aks enable-addons \
    --resource-group $RG --name $CLUSTER \
    --addons azure-keyvault-secrets-provider \
    2>/dev/null || echo "  → CSI driver déjà activé"

az aks update \
    --resource-group $RG --name $CLUSTER \
    --enable-secret-rotation --rotation-poll-interval 2m \
    2>/dev/null || echo "  → Rotation déjà activée"

# ──── PHASE 7 : Monitoring + Auto-upgrade (S9, S10) ────
echo "[Phase 7/7] Monitoring + Auto-upgrade..."
LAW_ID=$(az monitor log-analytics workspace show \
    --resource-group $RG --workspace-name $LAW_NAME \
    --query id -o tsv 2>/dev/null)

if [ -n "$LAW_ID" ]; then
    az aks enable-addons \
        --resource-group $RG --name $CLUSTER \
        --addons monitoring --workspace-resource-id $LAW_ID \
        2>/dev/null || echo "  → Monitoring déjà activé"
fi

az aks update \
    --resource-group $RG --name $CLUSTER \
    --auto-upgrade-channel patch \
    --node-os-upgrade-channel SecurityPatch \
    2>/dev/null || echo "  → Auto-upgrade déjà configuré"

# ──── VÉRIFICATION FINALE ────
echo ""
echo "══════════════════════════════════════════"
echo "VÉRIFICATION FINALE"
echo "══════════════════════════════════════════"
az aks show --resource-group $RG --name $CLUSTER \
    --query "{
        EntraID: aadProfile.managed,
        RBAC: enableRbac,
        AzureRBAC: aadProfile.enableAzureRbac,
        AuthorizedIPs: apiServerAccessProfile.authorizedIpRanges,
        PolicyAddon: addonProfiles.azurepolicy.enabled,
        CSISecrets: addonProfiles.azureKeyvaultSecretsProvider.enabled,
        Monitoring: addonProfiles.omsagent.enabled,
        AutoUpgrade: autoUpgradeProfile.upgradeChannel,
        NodeOSUpgrade: autoUpgradeProfile.nodeOSUpgradeChannel,
        Version: kubernetesVersion
    }"

echo ""
echo "✓ Script terminé. Vérifiez les résultats ci-dessus."
echo "⚠ Actions manuelles restantes :"
echo "  - Créer les NetworkPolicies deny-all par namespace (S2)"
echo "  - Configurer les PSS labels sur les namespaces (S5)"
echo "  - Créer les SecretProviderClass pour chaque workload (S8)"
echo "  - Configurer la maintenance window (S10)"
echo "  - Créer les PodDisruptionBudgets (S10)"
