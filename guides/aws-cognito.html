UserPoolAddOns.AdvancedSecurityMode OFF OFF AUDIT ENFORCED Plus

💡
Pourquoi chaque paramètre compte :
  • MinimumLength = 12+ : un mot de passe de 8 caractères est crackable par brute force en quelques heures (NIST SP 800-63B recommande ≥8 avec vérification de dictionnaire, CIS recommande ≥14). 12 est un compromis UX/sécurité raisonnable.
  • RequireUppercase/Numbers/Symbols : désactivés par défaut ! Sans complexité, les mots de passe courants (« password », « 12345678 ») sont acceptés.
  • TemporaryPasswordValidityDays = 1-3 : un mot de passe temporaire valide 7 jours laisse une fenêtre d'attaque large si l'email d'invitation est intercepté.
  • AllowAdminCreateUserOnly : si false (défaut), n'importe qui peut créer un compte sur votre User Pool via l'API SignUp. Pour les applications internes ou B2B, c'est un risque majeur d'abus.
  • UsernameAttributes = email : utiliser un username texte libre sans email vérifié empêche la récupération de compte et facilite la création de comptes fantômes.
  • AutoVerifiedAttributes : sans auto-vérification, les adresses email ne sont jamais confirmées → impossible de distinguer un vrai utilisateur d'un faux.
  • DeletionProtection : désactivé par défaut ! Un DeleteUserPool accidentel ou malveillant supprime tous les utilisateurs sans possibilité de récupération.
⚠️
Pièges classiques :
  • Schema immuable : les attributs custom (custom:company, custom:role) ne peuvent pas être supprimés ni rendus non-required après création du pool. Planifiez le schema avant la mise en production.
  • Username case sensitivity : par défaut CaseSensitive = true → « John » et « john » sont deux comptes différents. Toujours mettre false.
  • Pas de password history : Amazon Cognito ne vérifie pas que le nouveau mot de passe diffère de l'ancien. Mitigé via un Lambda trigger Pre Authentication.
  • Self sign-up + pas de MFA : combinaison critique → création de masse de comptes pour abus (DDoS applicatif, spam).

1.2 — Configuration

AWS CLI — Création d'un User Pool sécurisé
# Créer un User Pool avec password policy durcie et protection suppression
aws cognito-idp create-user-pool \
    --pool-name "MyApp-UserPool-Prod" \
    --policies '{
        "PasswordPolicy": {
            "MinimumLength": 12,
            "RequireUppercase": true,
            "RequireLowercase": true,
            "RequireNumbers": true,
            "RequireSymbols": true,
            "TemporaryPasswordValidityDays": 3
        }
    }' \
    --auto-verified-attributes "email" \
    --username-attributes "email" \
    --username-configuration '{"CaseSensitive": false}' \
    --account-recovery-setting '{
        "RecoveryMechanisms": [
            {"Priority": 1, "Name": "verified_email"}
        ]
    }' \
    --admin-create-user-config '{"AllowAdminCreateUserOnly": false}' \
    --deletion-protection "ACTIVE" \
    --user-pool-add-ons '{"AdvancedSecurityMode": "AUDIT"}' \
    --profile $AWS_PROFILE \
    --region $AWS_REGION \
    --output json
Terraform — User Pool sécurisé
resource "aws_cognito_user_pool" "main" {
  name = "MyApp-UserPool-Prod"

  # Password policy durcie
  password_policy {
    minimum_length                   = 12
    require_uppercase                 = true
    require_lowercase                 = true
    require_numbers                   = true
    require_symbols                   = true
    temporary_password_validity_days  = 3
  }

  # Sign-in par email, case insensitive
  username_attributes      = ["email"]
  auto_verified_attributes = ["email"]

  username_configuration {
    case_sensitive = false
  }

  # Récupération par email vérifié
  account_recovery_setting {
    recovery_mechanism {
      name     = "verified_email"
      priority = 1
    }
  }

  # Protection contre la suppression accidentelle
  deletion_protection = "ACTIVE"

    # Threat Protection via AdvancedSecurityMode en AUDIT (passer à ENFORCED en prod si justifié)
  user_pool_add_ons {
    advanced_security_mode = "AUDIT"
  }

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

💡
Pourquoi chaque paramètre compte :
  • Authorization Code + PKCE (S256) : seul flow sécurisé pour SPA/mobile. L'Implicit Flow expose le token dans l'URL et est déprécié par OAuth 2.1.
  • Tokens à 1h (ID+Access) : des tokens de 24h (défaut !) laissent une fenêtre d'exploitation énorme si un token est compromis. Les bonnes pratiques recommandent 15-60 min.
  • Refresh token à 7j : 30 jours par défaut est excessif. Un refresh token volé donne un accès silencieux pendant un mois.
  • GenerateSecret = false pour SPA : un SPA ne peut pas stocker un secret de manière sécurisée (code source visible). PKCE remplace le secret.
  • CallbackURLs strictes : un wildcard (*) ou localhost en production permet une redirection vers un domaine attaquant (open redirect vulnerability).
  • PreventUserExistenceErrors = ENABLED : en mode LEGACY, l'API répond différemment selon que l'utilisateur existe ou non → énumération de comptes.
⚠️
Pièges classiques :
  • Implicit Flow encore actif : beaucoup de tutoriels anciens utilisent response_type=token. Vérifier que seul code est autorisé dans l'app client.
  • Token validation côté client uniquement : le JWT doit être validé côté serveur (signature, issuer, audience, expiration). Ne jamais faire confiance à un token non vérifié.
  • Plusieurs App Clients pour un même frontend : chaque type de client (SPA, backend, M2M) doit avoir son propre app client avec des scopes et des durées de token adaptés.
  • Custom scopes non créés : sans Resource Server et custom scopes, le Client Credentials flow ne renvoie aucun scope utile.

2.3 — Configuration App Client (SPA avec PKCE)

AWS CLI — Création d'un App Client SPA avec PKCE
# Créer un app client pour SPA (pas de secret, PKCE enforced)
aws cognito-idp create-user-pool-client \
    --user-pool-id $USER_POOL_ID \
    --client-name "MyApp-SPA-Client" \
    --no-generate-secret \
    --explicit-auth-flows "ALLOW_REFRESH_TOKEN_AUTH" \
    --allowed-o-auth-flows "code" \
    --allowed-o-auth-flows-user-pool-client \
    --allowed-o-auth-scopes "openid" "email" "profile" \
    --callback-urls "https://app.example.com/callback" \
    --logout-urls "https://app.example.com/logout" \
    --id-token-validity 1 \
    --access-token-validity 1 \
    --refresh-token-validity 7 \
    --token-validity-units '{
        "IdToken": "hours",
        "AccessToken": "hours",
        "RefreshToken": "days"
    }' \
    --prevent-user-existence-errors "ENABLED" \
    --enable-token-revocation \
    --profile $AWS_PROFILE \
    --region $AWS_REGION \
    --output json

🚨
Scénario concret : Sans MFA, un attaquant qui obtient le mot de passe d'un utilisateur (phishing, credential stuffing, data breach) accède immédiatement au compte. En mode Optional, Amazon Cognito n'impose pas l'enrôlement MFA dans managed login ; votre application doit donc proposer un parcours de configuration utilisateur. En mode Required, managed login guide l'utilisateur vers l'enrôlement d'un second facteur. Si vous voulez utiliser l'adaptive authentication de Threat Protection, le MFA doit rester Optional.

  • Coût du tier Plus : vérifier le pricing live par région et par MAU ; ne l'activer que si la threat protection est réellement requise.

💡
Stratégies d'optimisation :
  • Nettoyage des users inactifs : seuls les utilisateurs avec activité d'identité dans le mois comptent comme MAU, mais un annuaire dormant complexifie l'administration et l'audit.
  • Choix du tier : si vous n'utilisez ni managed login, ni email MFA, ni passkeys, ni access token customization, Lite peut suffire ; sinon Essentials est le socle 2026. Réserver Plus à la threat protection.
  • SAML/OIDC fédération : les MAU fédérés ont un free tier dédié (50 MAU / mois) et une facturation distincte des connexions directes/sociales, quel que soit le tier.
  • SMS MFA : facturé via Amazon SNS avec un coût variable par pays. Préférer TOTP ou Email MFA quand l'UX et la conformité le permettent.
⚠️
Pièges classiques :
  • Tier par défaut : les nouveaux user pools sont en Essentials par défaut. Le passage à Plus doit être explicitement justifié.
  • MAU ≠ users totaux : seuls les utilisateurs actifs dans le mois sont facturés. Un User Pool de 1 million d'users dont 10 000 actifs = 10 000 MAU.
  • Lambda triggers coûtent séparément : chaque invocation Lambda est facturée à part (Lambda pricing, pas Cognito). Un trigger sur chaque login = N invocations/mois.
  • SMS en Europe : le coût SMS varie fortement par pays et par route ; vérifier la page SNS Pricing du pays ciblé.

Script All-in-One — Hardening Amazon Cognito
Script exécutable (AWS CLI) pour auditer et appliquer les best practices en une passe
⚠️
Prérequis : AWS CLI v2 configuré avec un profil possédant cognito-idp:*, wafv2:*, lambda:* et cognito-identity:*. Tester d'abord en mode audit (lecture seule) avant d'appliquer les corrections.
Bash — cognito-hardening-audit.sh
#!/usr/bin/env bash
# Amazon Cognito Hardening Audit Script
# Usage: ./cognito-hardening-audit.sh <user-pool-id> [region]
set -euo pipefail

USER_POOL_ID="${1:?Usage: $0 <user-pool-id> [region]}"
REGION="${2:-eu-west-1}"
SCORE=0
MAX=100
ISSUES=()

echo "=== Amazon Cognito Hardening Audit ==="
echo "User Pool: $USER_POOL_ID | Region: $REGION"
echo "$(date -u '+%Y-%m-%dT%H:%M:%SZ')"
echo "======================================="

# Récupérer la config du User Pool
POOL_JSON=$(aws cognito-idp describe-user-pool \
    --user-pool-id "$USER_POOL_ID" \
    --region "$REGION" \
    --output json)

# S1 — Password Policy
MIN_LEN=$(echo "$POOL_JSON" | jq -r '.UserPool.Policies.PasswordPolicy.MinimumLength // 8')
if [ "$MIN_LEN" -ge 12 ]; then
    echo "[OK] S1 Password MinLength=$MIN_LEN (>=12)"; SCORE=$((SCORE+5))
else
    echo "[FAIL] S1 Password MinLength=$MIN_LEN (<12)"; ISSUES+=("S1: MinimumLength < 12")
fi

# S1 — DeletionProtection
DEL_PROT=$(echo "$POOL_JSON" | jq -r '.UserPool.DeletionProtection // "INACTIVE"')
if [ "$DEL_PROT" = "ACTIVE" ]; then
    echo "[OK] S1 DeletionProtection=ACTIVE"; SCORE=$((SCORE+5))
else
    echo "[FAIL] S1 DeletionProtection=INACTIVE"; ISSUES+=("S1: DeletionProtection off")
fi

# S3 — MFA Configuration
MFA=$(aws cognito-idp get-user-pool-mfa-config \
    --user-pool-id "$USER_POOL_ID" \
    --region "$REGION" \
    --query "MfaConfiguration" --output text)
if [ "$MFA" = "ON" ]; then
    echo "[OK] S3 MFA=ON (enforced)"; SCORE=$((SCORE+15))
elif [ "$MFA" = "OPTIONAL" ]; then
    echo "[WARN] S3 MFA=OPTIONAL"; SCORE=$((SCORE+5)); ISSUES+=("S3: MFA should be ON")
else
    echo "[FAIL] S3 MFA=OFF"; ISSUES+=("S3: MFA disabled")
fi

# S2 — Check App Clients token validity
CLIENTS=$(aws cognito-idp list-user-pool-clients \
    --user-pool-id "$USER_POOL_ID" \
    --region "$REGION" \
    --query "UserPoolClients[].ClientId" --output text)
TOKEN_OK=true
for CID in $CLIENTS; do
    ACCESS_TTL=$(aws cognito-idp describe-user-pool-client \
        --user-pool-id "$USER_POOL_ID" --client-id "$CID" \
        --region "$REGION" \
        --query "UserPoolClient.AccessTokenValidity" --output text)
    if [ "$ACCESS_TTL" != "None" ] && [ "$ACCESS_TTL" -le 30 ] 2>/dev/null; then
        echo "[OK] S2 Client $CID AccessTokenTTL=${ACCESS_TTL}min"
    else
        echo "[WARN] S2 Client $CID AccessTokenTTL=${ACCESS_TTL} (should be ≤30min)"
        TOKEN_OK=false
    fi
done
if $TOKEN_OK; then SCORE=$((SCORE+10)); else SCORE=$((SCORE+3)); ISSUES+=("S2: Access tokens too long"); fi

# S7 — WAF Association
WAF_ARN=$(aws cognito-idp describe-user-pool \
    --user-pool-id "$USER_POOL_ID" \
    --region "$REGION" \
    --query "UserPool.UserPoolAddOns.AdvancedSecurityAdditionalFlows // 'NONE'" --output text 2>/dev/null || echo "NONE")
WAF_ASSOC=$(aws wafv2 get-web-acl-for-resource \
    --resource-arn "arn:aws:cognito-idp:${REGION}:$(aws sts get-caller-identity --query Account --output text):userpool/${USER_POOL_ID}" \
    --region "$REGION" \
    --query "WebACL.ARN" --output text 2>/dev/null || echo "NONE")
if [ "$WAF_ASSOC" != "NONE" ] && [ "$WAF_ASSOC" != "None" ]; then
    echo "[OK] S7 WAF associated: $WAF_ASSOC"; SCORE=$((SCORE+10))
else
    echo "[FAIL] S7 No WAF associated"; ISSUES+=("S7: No WAF on User Pool")
fi

# S8 — Custom Domain
DOMAIN=$(echo "$POOL_JSON" | jq -r '.UserPool.CustomDomain // .UserPool.Domain // "NONE"')
if [ "$DOMAIN" != "NONE" ] && [ "$DOMAIN" != "null" ]; then
    echo "[OK] S8 Domain=$DOMAIN"; SCORE=$((SCORE+4))
else
    echo "[WARN] S8 No custom domain"; ISSUES+=("S8: No custom domain")
fi

# Résultat final
echo ""
echo "======================================="
echo "SCORE: $SCORE / $MAX"
if [ ${#ISSUES[@]} -gt 0 ]; then
    echo "ISSUES (${#ISSUES[@]}):"
    for issue in "${ISSUES[@]}"; do
        echo "  - $issue"
    done
else
    echo "Aucun problème détecté."
fi
echo "======================================="
