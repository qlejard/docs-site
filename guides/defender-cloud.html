Mode d'emploi de ce guide
Comment utiliser ce document comme référentiel de posture cloud Azure
📋
Ce document est un référentiel générique couvrant toutes les configurations Microsoft Defender for Cloud (MDC) — CSPM, Defender Plans, Regulatory Compliance, Workflow Automation. Il peut être utilisé tel quel en présentation client ou adapté en ajoutant les données spécifiques d'une souscription Azure.
⚠️
Analogie clé : Defender for Cloud est le « bilan médical de votre infrastructure Azure ». Le CSPM gratuit donne un Secure Score (note de santé) avec des recommandations. Les Defender Plans sont comme des « alarmes par pièce » — on active la protection pièce par pièce : serveurs, bases, stockage, conteneurs. La Regulatory Compliance est un « audit blanc continu » qui vérifie en permanence si vous êtes conforme aux normes (CIS, ISO 27001, PCI-DSS).

S1
CSPM & Secure Score CSPM Gratuit Defender CSPM
Posture de sécurité cloud, Secure Score, recommandations, governance rules, auto-provisioning
ℹ️
Le Secure Score est la « note de santé » de votre infrastructure Azure. Chaque recommandation non traitée fait baisser le score. Par défaut, sans Defender Plans activés, le score reflète uniquement les recommandations CSPM gratuites (~25-35 %). Activer Defender CSPM ajoute l'attack path analysis, le cloud security graph et l'agentless scanning.

⚠️
Piège : Le Secure Score est calculé sur les recommandations applicables uniquement. Exempter une recommandation la retire du calcul — cela fait monter le score artificiellement. Utilisez les exemptions avec parcimonie et documentez la justification. Azure Policy Exempt est audité dans le compliance dashboard.
💡
Pourquoi chaque paramètre compte :
  • Secure Score : score pondéré (max points par contrôle). Les contrôles les plus lourds sont : « Enable MFA », « Apply system updates », « Remediate vulnerabilities ». Traiter les 5 plus gros contrôles peut faire passer le score de 30 % à 60 %.
  • Auto-provisioning AMA : sans agent, de nombreuses recommandations restent « Not applicable » — le score semble bon mais la couverture est fausse. L'agent AMA (Azure Monitor Agent) remplace le MMA (déprécié).
  • Attack path analysis : identifie les chemins d'attaque exploitables combinant plusieurs faiblesses (ex: VM exposée + managed identity avec accès Key Vault). Invisible sans Defender CSPM.
  • Governance rules : assignent un propriétaire et un délai à chaque recommandation. Sans gouvernance, les recommandations s'accumulent sans responsable.
  • Continuous export : permet l'intégration avec Sentinel (voir S9) et la rétention longue durée dans Log Analytics.

    --rule-name "Critical-Reco-Owner" \
    --display-name "Assign critical recommendations" \

    --grace-period "7.00:00:00" \

Piège : Assigner un standard via Azure Policy ne signifie pas que vous êtes conforme. Le dashboard montre les gaps — c'est le point de départ, pas l'arrivée. De plus, certains contrôles sont marqués Manual : MDC ne peut pas les vérifier automatiquement (ex: « formation des employés »).

  • MCSB : le benchmark par défaut de Microsoft couvre ~200 contrôles regroupés en 12 domaines (Network, Identity, Data Protection, etc.). C'est la base gratuite — ne pas le désactiver.
  • CIS Azure Foundations : standard reconnu mondialement pour le hardening Azure. L'ajouter coûte 0 € (c'est une initiative Azure Policy). Exigé fréquemment en audit.
  • PCI-DSS / ISO 27001 : souvent requis contractuellement. Le dashboard permet de générer des rapports PDF pour les auditeurs. Sans ces standards assignés, impossible de prouver la conformité.
  • Custom initiatives : pour les exigences internes spécifiques (ex: « tous les Storage accounts doivent être en HTTPS only »). Permet de capturer les règles métier non couvertes par CIS.
  • Export des rapports : le dashboard exporte en PDF/CSV, utilisable directement lors d'audits formels. Sans export, les preuves de conformité doivent être capturées manuellement.

Pourquoi chaque fonctionnalité compte :
  • JIT VM Access : ferme les ports de gestion (RDP 3389, SSH 22) par défaut et les ouvre temporairement sur demande. Réduit la surface d'attaque de ~70 % sur les VMs exposées.
  • FIM (File Integrity Monitoring) : détecte les modifications non autorisées sur les fichiers système critiques (registre Windows, /etc/passwd Linux). Exigence PCI-DSS section 11.5.
  • Adaptive Application Controls : whitelisting applicatif — seules les apps autorisées peuvent s'exécuter. Bloque les malwares et les outils offensifs.
  • Vulnerability Assessment (MDVM) : scan intégré Microsoft Defender Vulnerability Management. Agentless ou agent-based. Rapports de vulnérabilités directement dans MDC.
  • 500 Mo/jour gratuit : P2 inclut 500 Mo/jour/serveur d'ingestion gratuite dans Log Analytics — significatif dans le calcul de coût total.

S8.5 — État des lieux

S9
Intégration Sentinel Sentinel
Data connector bi-directionnel, alertes MDC → incidents Sentinel, SecurityAlert / SecurityRecommendation tables
🚨
Scénario d'attaque : Un attaquant compromet une managed identity sur une VM Azure exposée. Il utilise cette identité pour lire tous les secrets du Key Vault, puis pivote vers les bases SQL. Sans intégration Sentinel, l'alerte MDC « Suspicious access to Key Vault » reste isolée dans le dashboard MDC. Personne ne corrèle avec les sign-ins suspects dans Microsoft Entra ID (anciennement Azure AD) ni l'activité ARM anormale. Résultat : l'attaquant exfiltre 200 secrets avant réaction du SOC — 48h de délai au lieu de 15 min avec l'auto-création d'incidents Sentinel et les analytics rules de corrélation.
ℹ️
L'intégration MDC → Sentinel transforme les alertes de sécurité cloud en incidents investigables dans le SIEM. Le data connector est bi-directionnel : les alertes MDC deviennent des incidents Sentinel, et les statuts d'incident Sentinel se synchronisent avec MDC. Pour la configuration complète de Sentinel, voir sentinel.html.
💡
Pourquoi chaque paramètre compte :
  • Data connector MDC → Sentinel : sans ce connecteur, les alertes MDC restent isolées dans le portail MDC. Avec Sentinel, elles deviennent des incidents corrélés avec les sign-ins Microsoft Entra ID (anciennement Azure AD), les logs Office 365 et les alertes MDE — vue 360° de l'attaque.
  • Bi-directional sync : quand un analyste SOC ferme un incident dans Sentinel, le statut se propage dans MDC (et inversement). Sans sync, les équipes travaillent en silo avec des statuts contradictoires.
  • SecurityAlert table : alimentée automatiquement dès que le connecteur est actif. Permet les requêtes KQL de corrélation (ex: alerte MDC + AzureActivity du même IP).
  • SecurityRecommendation table : nécessite le continuous export (S8). Permet de suivre l'évolution des recommandations dans le temps et de détecter les régressions de posture.
  • Analytics rules MDC : les templates pré-construits détectent les patterns courants (brute force SQL + exfiltration, escalation de privilèges + accès Key Vault). À activer dès la connexion.
  • Incident auto-creation : sans auto-création, les alertes MDC restent de simples alertes sans workflow d'investigation. L'auto-création d'incidents déclenche les playbooks et les assignments SOC.

Verdict

🎯
MDC retenu pour les raisons suivantes :
  • Azure-first : notre infrastructure est à >90 % sur Azure. L'intégration native évite un outil supplémentaire à maintenir.
  • Consolidation MDE + MDC + Sentinel : un seul data model pour endpoint, cloud et SIEM. Pas de mapping d'alertes entre systèmes hétérogènes.
  • Coût : CSPM gratuit par défaut, plans Defender inclus dans l'Enterprise Agreement. Pas de ligne budgétaire séparée.
  • Prisma Cloud écarté : excellent en multi-cloud mais ajoute un silo d'alertes découplé de Sentinel. Coût additionnel non justifié sur un périmètre mono-Azure.
  • Wiz écarté : CSPM agentless très performant mais CWPP encore immature. Pas d'intégration native MDE. Premium pricing.
